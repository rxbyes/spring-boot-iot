# 10 智能助手技能与任务选型指南

> 文档定位：`spring-boot-iot` 项目内智能助手技能选型、组合顺序和典型任务映射指南。
> 适用角色：研发、测试、文档维护者、智能助手协作人员。
> 权威级别：二级权威，负责技能选型与协作方法，不替代原始 `SKILL.md`。
> 上游来源：`AGENTS.md`、`README.md`、`docs/README.md`、当前可用技能说明、当前项目模块边界与真实环境规则。

## 1. 先记住的 6 条规则

1. 不知道先用什么时，先按 `Using Superpowers` 的思路做技能判断，而不是直接让助手开始改代码。
2. 只要任务涉及新增能力、页面结构、交互方式、模块职责或行为变化，优先先走 `Brainstorming`。
3. 任务一旦跨模块、跨文档、跨验收或超过一轮对话能稳定 hold 住的复杂度，就应进入 `Writing Plans`。
4. 只要出现测试失败、接口异常、MQTT 不回流、TDengine 不写入、页面表现和预期不一致，就先走 `Systematic Debugging`，不要直接猜修复方案。
5. 任何“已修复 / 已完成 / 已通过 / 可以提测”的说法，都必须先用 `Verification Before Completion` 对应到真实命令和最新证据。
6. 本项目所有验收最终都要回到 `application-dev.yml` 对应的真实环境，不允许用 H2 或旧浏览器自动化链路代替结论。

## 2. 技能总览

| 类别 | 核心技能 | 解决什么问题 | 默认下一步 |
|---|---|---|---|
| 总控与入口 | `Using Superpowers` | 先判断该走设计、计划、调试还是执行路径 | 进入最合适的流程技能 |
| 需求澄清与方案 | `Brainstorming` | 把模糊需求变成可确认的设计 | `Writing Plans` |
| 计划与拆解 | `Writing Plans` | 把设计拆成文件、测试、命令级别的实施计划 | `Using Git Worktrees` + 执行类技能 |
| 工作区隔离 | `Using Git Worktrees` | 在干净分支/工作树里推进任务 | `Subagent Driven Development` 或 `Executing Plans` |
| 计划执行 | `Subagent Driven Development`、`Executing Plans` | 真正按计划实现代码、文档和验证 | `Requesting Code Review` |
| 并行协作 | `Dispatching Parallel Agents` | 把多个独立问题或独立子任务并发处理 | 汇总后统一验证 |
| 质量保障 | `Test Driven Development`、`Systematic Debugging` | 防止盲写代码、盲修 bug | `Verification Before Completion` |
| 评审协作 | `Requesting Code Review`、`Receiving Code Review` | 主动发起评审，或正确处理评审意见 | 修复后再次验证 |
| 交付收尾 | `Verification Before Completion`、`Finishing a Development Branch` | 用证据确认完成，并规范处理分支 | 合并、本地保留或提 PR |
| 生态扩展 | `OpenAI Docs`、`Skill Installer`、`Skill Creator`、`Writing Skills` | 查官方资料、安装技能、建设项目私有技能 | 进入集成或技能沉淀 |

## 3. 各技能在本项目里的用法

### 3.1 总控与规划类

| 技能 | 核心定位 | 什么时候用 | 本项目高频场景 | 常见误用 |
|---|---|---|---|---|
| `Using Superpowers` | 所有技能的总入口 | 每次接到新任务、准备行动前 | 你还没判断这是“先分析”“先设计”“先修 bug”还是“直接执行计划”时 | 跳过它，直接让助手改代码或全仓库扫描 |
| `Brainstorming` | 先设计后实现 | 新增功能、页面优化、流程改造、行为变更、模块职责调整 | 新增告警规则能力、重做 `/reporting` 页交互、设计新的消息处理环节 | 把它当“可选项”，在需求没确认前就开写 |
| `Writing Plans` | 从设计到实施清单 | 任务超过单次实现复杂度，或跨模块 / 跨文档 / 跨验收 | 设备接入链路改造、Phase 4 子模块复验、前后端联动需求 | 没有设计或需求还模糊时硬写计划 |
| `Using Git Worktrees` | 隔离工作空间 | 当前仓库已有未提交修改，或要并行推进多个任务 | 一边修 MQTT，一边做前端治理，一边整理文档 | 在脏工作区直接开工，导致变更互相污染 |

补充说明：
- `Brainstorming` 在本项目里尤其适合“改业务语义、改页面结构、改协作流程、改模块边界”类任务，因为项目对模块边界和文档同步要求很硬。
- `Writing Plans` 适合把“代码、SQL、脚本、文档、验收”一次性拆清楚，避免改到一半才发现忘了更新 `docs/` 或真实环境步骤。

### 3.2 执行与协作类

| 技能 | 核心定位 | 什么时候用 | 本项目高频场景 | 常见误用 |
|---|---|---|---|---|
| `Subagent Driven Development` | 在当前会话中按计划高质量推进 | 已经有书面计划，且任务可按子任务逐个实现 | 分阶段完成接口、页面、文档和验收脚本 | 在还没计划时直接派子代理乱做 |
| `Executing Plans` | 线性执行现成计划 | 已有完整计划，但不想走强子代理评审流程 | 执行中等复杂度的功能或重构任务 | 没有计划就开始执行，或跳过计划审查 |
| `Dispatching Parallel Agents` | 并行处理多个独立问题 | 同时有 2 个以上互不干扰的问题域 | 前端样式漂移、协议单测失败、告警状态 bug 同时存在 | 问题其实共享根因，却被拆成多个并行任务互相打架 |

补充说明：
- 在 `spring-boot-iot` 里，只有“文件改动范围几乎不重叠、问题根因明显独立”时才适合 `Dispatching Parallel Agents`。
- 如果任务本身是一个连续链路，例如 MQTT 发布 -> consumer -> pipeline -> telemetry，不要先并行拆；应先判断根因在哪一层。

### 3.3 质量与交付类

| 技能 | 核心定位 | 什么时候用 | 本项目高频场景 | 常见误用 |
|---|---|---|---|---|
| `Systematic Debugging` | 先找根因再修 | 任何 bug、测试失败、接口异常、联调问题、环境异常 | MQTT 不回流、`traceId` 绑定失败、TDengine 无数据、JWT 鉴权异常 | 先拍脑袋改代码，再回头找原因 |
| `Test Driven Development` | 先写失败测试，再写实现 | 功能新增、bug 修复、行为变化、重构 | 修 `GET /api/telemetry/latest`、补接口边界、修权限判断 | 先写实现再补测试，或者只做手工验证 |
| `Requesting Code Review` | 主动请求代码审查 | 完成一轮功能、修完复杂问题、准备合并前 | Phase 4 一轮子功能完成、复杂链路修复完成 | 觉得“改动不大”就跳过审查 |
| `Receiving Code Review` | 正确处理评审意见 | 收到同事、代理或平台 review 后 | reviewer 指出 service 边界、命名、权限、测试或 SQL 问题 | 不验证就盲改，或只做社交式附和 |
| `Verification Before Completion` | 没证据不宣称完成 | 任何准备说“通过了”“修好了”“可以提测了”时 | 真实环境验收、脚本执行、构建通过、页面回归通过 | 用“应该好了”“本地看起来可以”代替正式验证 |
| `Finishing a Development Branch` | 规范收尾分支 | 实现和验证都完成后 | 决定本地合并、推 PR、暂存分支或丢弃工作树 | 测试没过就收尾，或没确认就删分支 |

补充说明：
- `Systematic Debugging` 对本项目价值很高，因为你们的关键问题经常跨 HTTP/MQTT、Redis、MySQL、TDengine、前端代理和权限链路，不适合靠猜。
- `Verification Before Completion` 与本项目“真实环境才算验收”的规则完全一致，是交付阶段的高优先技能。

### 3.4 生态扩展类

| 技能 | 核心定位 | 什么时候用 | 本项目高频场景 | 常见误用 |
|---|---|---|---|---|
| `OpenAI Docs` | 查 OpenAI 官方最新资料 | 要接 OpenAI API、模型选型、Agents/Responses 集成时 | 以后若要做告警总结、AI 助手、知识问答、智能工单辅助 | 用记忆或二手博客替代官方文档 |
| `Skill Installer` | 安装更多可用技能 | 当前技能不够，需要新增专用能力 | 想补 SQL、前端设计、文档治理或其他工作流技能 | 还没搞清楚需求就盲装一堆技能 |
| `Skill Creator` | 设计一个可复用技能 | 想做项目私有技能包时 | 制作“真实环境验收技能”“MQTT 排障技能” | 把一次性经验直接塞成长期技能 |
| `Writing Skills` | 用 TDD 思路打磨技能 | 对新技能做压力测试和迭代 | 把项目协作方法沉淀为稳定技能 | 只写 `SKILL.md`，不做实际验证 |

补充说明：
- `Skill Creator` 更偏“怎么构造一个技能包”。
- `Writing Skills` 更偏“怎么确认这个技能真的能约束代理行为、经得起压力场景”。

## 4. 按项目任务类型选技能

### 4.1 新增后端业务能力

推荐链路：

`Using Superpowers -> Brainstorming -> Writing Plans -> Using Git Worktrees -> Subagent Driven Development -> Requesting Code Review -> Verification Before Completion -> Finishing a Development Branch`

适用场景：
- 新增设备、告警、事件、规则、报表、平台治理等业务能力
- 涉及 Controller / Service / Mapper / SQL / 文档同步更新

为什么这样选：
- 先用 `Brainstorming` 把业务边界、模块归属、验收标准讲清楚。
- 再用 `Writing Plans` 把代码、SQL、验收和文档拆成可执行步骤。
- 最后用 `Subagent Driven Development` 在当前会话内持续推进并做中间评审。

### 4.2 新增前端页面或做前端优化

推荐链路：

`Using Superpowers -> Brainstorming -> Writing Plans（任务较大时） -> Test Driven Development -> Requesting Code Review -> Verification Before Completion`

适用场景：
- 新增管理页、详情抽屉、报表面板
- 优化 `/reporting`、`/message-trace`、`/in-app-message`、平台治理列表页

本项目额外要求：
- 优先复用 `PanelCard`、`StandardPagination`、`StandardTableToolbar`、`StandardTableTextColumn`、`StandardFormDrawer`、`StandardDetailDrawer`、`confirmAction`
- 保持 UTF-8，可读可检
- 如暴露了样式漂移或治理问题，结束前回写 [15-前端优化与治理计划.md](./15-前端优化与治理计划.md)

### 4.3 修接口异常、业务 bug 或测试失败

推荐链路：

`Using Superpowers -> Systematic Debugging -> Test Driven Development -> Requesting Code Review（复杂修复时） -> Verification Before Completion`

适用场景：
- 接口 `500`
- 权限错误
- 业务状态更新异常
- 单测、集成测试、构建失败

为什么这样选：
- 先用 `Systematic Debugging` 定位真正根因。
- 根因确认后，再用 `Test Driven Development` 固化回归测试。

### 4.4 修 MQTT / 协议 / telemetry / message-flow 链路问题

推荐链路：

`Using Superpowers -> Systematic Debugging -> Dispatching Parallel Agents（仅当问题独立） -> Test Driven Development -> Verification Before Completion`

适用场景：
- `/api/message/mqtt/report/publish` 返回成功但未回流
- `sessionId` 有了但 `traceId` 绑定失败
- timeline 不完整
- `TELEMETRY_PERSIST` 阶段失败
- TDengine 未写入或 `latest` 查询异常

项目内排障顺序建议：
1. 先判断问题属于哪一层：发布、consumer、pipeline、持久化还是查询。
2. 再看具体阶段：`PROTOCOL_DECODE`、`PAYLOAD_APPLY`、`TELEMETRY_PERSIST`、`DEVICE_STATE`、`RISK_DISPATCH`。
3. 只有当多个异常明显独立时，才用 `Dispatching Parallel Agents`。

### 4.5 做真实环境复验、提测或交付核对

推荐链路：

`Using Superpowers -> Verification Before Completion`

若失败则立刻切换为：

`Systematic Debugging`

适用场景：
- 运行 [真实环境测试与验收手册.md](./真实环境测试与验收手册.md) 中的流程
- 回归 `application-dev.yml` 对应环境
- 输出“本轮通过 / 阻塞 / 部分通过”结论

项目内硬约束：
- 不允许回退 H2
- 环境访问受阻时必须明确写“环境阻塞”
- 构建通过、单测通过、脚本看起来正常，都不能代替真实环境结论

### 4.6 处理 code review 意见

推荐链路：

`Using Superpowers -> Receiving Code Review -> Test Driven Development（涉及行为变化时） -> Verification Before Completion`

适用场景：
- reviewer 指出 service / mapper 边界不清
- reviewer 指出接口命名、权限、事务、测试或 SQL 问题

项目内注意：
- 先验证该意见是否符合当前模块边界和业务背景
- 不要因为 reviewer 说“更规范”就把持久化逻辑塞到错误模块

### 4.7 做跨模块的大任务

推荐链路：

`Using Superpowers -> Brainstorming -> Writing Plans -> Using Git Worktrees -> Subagent Driven Development -> Requesting Code Review -> Verification Before Completion`

适用场景：
- Phase 4 某项能力的完整收口
- 横跨后端、前端、SQL、脚本、文档和验收的任务
- “实现 + 联调 + 文档 + 复验”一体化需求

### 4.8 做 OpenAI 集成或 AI 能力接入

推荐链路：

`Using Superpowers -> OpenAI Docs -> Brainstorming -> Writing Plans`

适用场景：
- 后续若要做智能告警摘要、辅助问答、工单建议、AI 运维助手

为什么这样选：
- `OpenAI Docs` 先确保信息来自官方最新文档。
- `Brainstorming` 再决定是做同步调用、异步任务，还是工具型 agent。

### 4.9 沉淀项目私有技能

推荐链路：

`Using Superpowers -> Skill Creator -> Writing Skills -> Skill Installer`

适用场景：
- 把“真实环境验收步骤”“MQTT 排障流程”“文档原位更新规则”沉淀成项目长期能力

推荐优先沉淀的技能主题：
- `spring-boot-iot-real-env-acceptance`
- `spring-boot-iot-mqtt-debugging`
- `spring-boot-iot-doc-update-checklist`

## 5. 本项目最值得固定下来的 5 套组合

### 5.1 新功能开发默认组合

`Brainstorming -> Writing Plans -> Using Git Worktrees -> Subagent Driven Development -> Requesting Code Review -> Verification Before Completion`

适合：
- 功能新增
- 大幅行为变化
- 前后端联动需求

### 5.2 修 bug 默认组合

`Systematic Debugging -> Test Driven Development -> Verification Before Completion`

适合：
- 接口错误
- 权限 bug
- 数据异常
- 联调失败

### 5.3 复杂联调默认组合

`Systematic Debugging -> Dispatching Parallel Agents（仅独立问题） -> Verification Before Completion`

适合：
- MQTT / Redis / TDengine / message-flow 多层联调
- 多个独立报错同时出现

### 5.4 评审闭环默认组合

`Requesting Code Review -> Receiving Code Review -> Verification Before Completion`

适合：
- 大功能完成后
- 合并前
- 复杂问题修复后

### 5.5 技能建设默认组合

`Skill Creator -> Writing Skills -> Skill Installer`

适合：
- 你准备把重复工作方法沉淀成团队长期可复用资产时

## 6. 与项目规则绑定的硬约束

### 6.1 真实环境优先

- 所有系统验收最终都必须回到 `spring-boot-iot-admin/src/main/resources/application-dev.yml`
- 任何技能组合都不能绕过真实环境结论
- `Verification Before Completion` 是验收阶段强绑定技能

### 6.2 模块边界优先

- `message` 只做入口和调度
- `protocol` 只做协议编解码和适配
- `device` 负责设备、产品、属性、消息日志
- `telemetry` 负责时序写入与查询抽象
- `alarm` 负责告警、事件、风险点、风险监测

因此：
- 设计任务先用 `Brainstorming` 明确边界
- review 阶段用 `Receiving Code Review` 防止被“看起来更规范”的建议带偏

### 6.3 文档原位更新优先

以下任务只要改动了行为或口径，都要同步更新文档：
- 接口
- SQL / 表结构
- 页面结构
- 启动方式
- 验收步骤
- 角色可见性
- 协作规则

因此：
- `Writing Plans` 里应明确写出文档更新步骤
- `Verification Before Completion` 里应包含“文档是否已更新”的检查

### 6.4 前端治理优先

前端相关任务不只是“把页面改对”，还要保持：
- 共享组件优先
- 统一品牌配色
- UTF-8 无乱码
- 列表 / 分页 / 抽屉 / 详情模式一致

因此：
- 任何前端优化都建议至少走 `Brainstorming`
- 如涉及治理规则变化，还应同步回写 [15-前端优化与治理计划.md](./15-前端优化与治理计划.md)

## 7. 快速决策表

| 如果你的当前状态是… | 先用哪个技能 | 下一步 |
|---|---|---|
| 我不知道这算设计、修 bug 还是直接改 | `Using Superpowers` | 进入对应流程技能 |
| 我知道要做什么，但边界没想清楚 | `Brainstorming` | `Writing Plans` |
| 我已经有明确设计，任务又大又杂 | `Writing Plans` | `Using Git Worktrees` |
| 我已经有计划，要在当前会话里高质量推进 | `Subagent Driven Development` | 每任务审查与验证 |
| 我只想按现成步骤稳定执行 | `Executing Plans` | 完成后进入分支收尾 |
| 我遇到 bug 或测试失败 | `Systematic Debugging` | 根因确认后 `Test Driven Development` |
| 我正在修功能或改行为 | `Test Driven Development` | 通过后走验证 |
| 我做完一轮功能，想让助手再审一遍 | `Requesting Code Review` | 修复后验证 |
| 我收到了评审意见 | `Receiving Code Review` | 逐项验证和处理 |
