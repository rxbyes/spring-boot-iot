## 1. 业务域划分

后端业务仍保持“三大业务域”，前端导航则按角色工作场景重组为“五大工作台”：

1. 后端设备接入域：产品、设备、协议、消息、属性、下行指令。
2. 后端风险处置域：告警、事件、工单、风险点、阈值规则、联动规则、应急预案、分析报表。
3. 后端系统治理域：组织、用户、角色、菜单、区域、字典、通知渠道、审计日志。
4. 前端五大工作台：`接入智维`、`风险运营`、`风险策略`、`平台治理`、`质量工场`。

### 1.1 三域关系总览

三大后端业务域的关系不是并列平铺，而是稳定的“数据源 -> 业务闭环 -> 治理支撑”关系：

1. `IoT 主链路`
   - 负责产品、设备、物模型、属性/遥测、消息日志，是平台上游数据源。
2. `风险平台`
   - 消费 IoT 主链路结果，完成风险点、规则、告警、事件、工单闭环。
3. `系统管理`
   - 提供组织、区域、字典、通知渠道、审计日志等治理底座。

主关系固定为：

1. `IoT 主链路 -> 风险平台`
2. `系统管理 -> IoT 主链路`
3. `系统管理 -> 风险平台`
4. `风险平台 -> IoT 主链路`

#### 2.1.0 IoT 五层治理模型

`产品 -> 设备 -> 物模型 -> 属性/遥测 -> 消息日志` 是当前设备接入域的固定治理顺序，不应被理解为彼此独立的并列对象：

1. `产品`
   - 定义一类设备的接入模板和业务身份。
2. `设备`
   - 定义某台具体资产实例。
3. `物模型`
   - 定义平台认可的正式字段标准。
4. `属性/遥测`
   - 定义运行期最新值和历史值。
5. `消息日志`
   - 定义原始报文证据和处理留痕。

最小治理规则：

1. 先建产品，再建设备。
2. 同一产品下正式字段 `identifier` 必须唯一。
3. 核心判定字段必须同时明确 latest 与 telemetry 口径。
4. 成功与失败上报都必须保留消息日志。
5. 父设备 / 子设备场景必须明确风险责任归属。

详细字段治理表与业务建档联调步骤，参考 [appendix/iot-field-governance-and-sop.md](./appendix/iot-field-governance-and-sop.md)。

补充口径：
- `/reporting` 当前已升级为“结果复盘优先”的链路验证中心：默认先进入 `结果复盘`，通过共享诊断状态头统一表达 `尚未验证 / 验证成功 / 等待回流 / 验证失败 / 复盘受限`，并继续保留 `模拟验证 / 结果复盘 / 最近记录` 三个真实业务页签。
- `deviceCode` 仍是链路验证中心唯一主查询入口；查询成功后前端只读反显 `productKey`、`protocolCode`、`clientId=deviceCode`。
- 页面保留 `tenantId` 与 `topic` 可编辑，其中 `tenantId` 默认 `1` 且仅 HTTP 提交生效；`topic` 默认 `$dp`，HTTP / MQTT 均允许手工覆盖。
- 明文模式下，只有可识别 JSON 才参与 C.1 / C.2 / C.3 判定与帧构造；XML 与普通文本仅提供格式化和原样发送，不新增 XML 协议解析。
- 当传输方式为 MQTT、`topic=$dp` 且明文内容为普通属性 JSON 且缺少设备身份字段时，前端会在实际发送 payload 中自动补入 `deviceCode`，保证默认兼容链路可用。
- 发送成功后，页面会自动切回 `结果复盘` 页签，并在共享诊断状态头和“处理时间线”区域同时表达同一次提交的固定 Pipeline 复盘：HTTP 直接展示 timeline，MQTT 先显示“等待消费回流”，绑定 `traceId` 后再展示完整时间线。
- `最近记录` 当前已从“恢复入口”升级为诊断清单：每条记录优先展示 verdict、当前卡点和建议动作，再提供恢复当前复盘上下文的入口。
- `/message-trace` 当前详情抽屉已定稿为客户视角收纳后的 `A 完整内容平衡版`：抽屉标题行承担轻刊头，但不再重复展示 `消息类型 / TraceId / Topic` 标签，避免和正文重复；这些识别信息统一由 `消息概览 lead-sheet` 与 `链路信息` 承接。正文完整保留 `消息概览 lead-sheet`、`链路信息`、`Payload 对照`、`处理时间线` 四个章节；其中 `链路信息` 章内必须继续拆成 `链路标识 / 接入上下文` 双分组，避免链路字段散排，首屏视觉以附件终版的大标题 lead-sheet 和右侧指标列为准，不再回退到设备详情 hero 语法，也不再堆叠说明墙、排查建议或跨页导流文案。`消息概览` 右侧的 `上报时间 / Topic 节点 / 产品标识 / 日志 ID` 当前固定按 `2 x 2` 指标块对齐，避免长值在首屏换行打散；其中显式带 `UTC / offset` 的上报时间会统一换算为中国时区 `Asia/Shanghai` 展示，无时区的历史时间串继续按原值显示，避免已是本地时间的日志再次偏移。`链路标识` 当前固定为 `日志 ID / 创建时间 / TraceId`，不再重复展示消息类型；`Topic` 固定保留在 `接入上下文` 分组内，例如 `$dp`。`Payload 对照` 优先根据消息日志详情接口恢复 `原始 Payload / 解密后明文 / 解析结果`，但展示形态已改为三条纵向折叠板块：默认收起、按需展开，并为每个板块补齐复制动作。即使 Redis 时间线已过期，也仍可继续查看恢复后的明文与解析结果；若恢复接口给出空字符串或空对象，前端仍需回退到原始 Payload 兜底显示，避免用户误判为业务未解析。`处理时间线` 当前默认折叠，只保留状态摘要和过期/存储异常提示；展开后才进入固定 Pipeline 复盘，不再影响 Payload 对照的可读性。
- `/message-trace` 的 `失败归档` 模式当前会优先保留阻塞型 MQTT Pipeline 失败的真实上下文：当异常发生在 `TOPIC_ROUTE / PROTOCOL_DECODE / DEVICE_CONTRACT / MESSAGE_LOG / PAYLOAD_APPLY / DEVICE_STATE / RISK_DISPATCH` 等固定阶段时，归档记录会尽量沿用当次链路已解析出的 `traceId / deviceCode / productKey / protocolCode / messageType / topic`，并把 `failureStage` 优先对齐到真实 Pipeline 阶段；若 `contractSnapshot` 组装过程中再次命中脏 schema 或产品查询异常，平台会跳过快照但继续保留基础失败样本，不再因为附加快照恢复失败把整条失败归档吞掉。
- `接入智维` 当前已通过共享诊断上下文把 `/reporting`、`/message-trace`、`/system-log`、`/file-debug` 与失败归档模式串成首批排查流；跨页只透传高价值定位字段与来源页，不新增后端聚合接口或第二套路由骨架。
- 当诊断结果指向契约治理或资产治理时，页面可直接跳入 `产品定义中心` 或 `设备资产中心`；治理页当前只在工作台内联提示“来自哪个诊断页 + 优先核对什么”，不再额外堆介绍墙、概况卡或重复导流文案。
- 2026-03-28 起，`接入智维` 7 个入口当前统一收口为“轻量页面壳 + 真业务页签/单主内容区”结构：总览=`资产底座 / 诊断排障`，产品=`单主列表页（物模型设计器与关联设备回收到行操作和抽屉）`，设备=`单主列表页（未登记资产与拓扑治理回收到同一主台账）`，链路验证=`模拟验证 / 结果复盘 / 最近记录`，异常观测=`单主列表页（通过顶部动作回跳链路追踪和失败归档）`，链路追踪=`链路追踪 / 失败归档`，数据校验=`设备校验 / 原始响应`。

#### 2.1.1 产品与设备的逻辑关系

- 产品是平台里的“接入模板”，不是某一台具体硬件。一个产品定义了一组设备共同遵循的 `productKey`、`protocolCode`、`nodeType`、`dataFormat`、`manufacturer` 和物模型边界。
- 设备是某个产品下的具体资产实例。设备补充的是 `deviceCode`、鉴权凭据、固件版本、IP、安装位置、在线状态、最近上报时间等实例信息。
- 平台关系可以简化为：`1 个产品 -> N 台设备 -> N 条消息 / 属性 / 命令记录`。同一产品下可以有很多设备，但设备不能脱离产品单独定义接入协议。
- 是否应该拆成新产品，不看“是不是同一个项目”，而看“接入契约是否一致”。协议编码、Topic/报文结构、物模型、认证方式、下行指令、节点类型或生命周期策略任一项发生变化，都应评估拆成不同产品。

#### 2.1.2 建档顺序与治理口径

| 规则 | 说明 | 当前状态 |
|---|---|---|
| 先定义产品身份与协议，再允许设备建档 | 设备建档时必须先通过 `productKey` 找到产品，再继承产品的协议编码与节点类型 | 已落地 |
| `productKey` 创建后不允许修改 | `productKey` 是接入身份，变更后会影响建档、Topic 和日志检索 | 已落地 |
| 删除产品前先清理库存设备 | 产品删除前会校验是否还有关联设备库存 | 已落地 |
| 停用产品前先核查库存设备是否仍在使用 | 停用前会校验该产品下是否仍存在启用中的关联设备，避免存量在用设备继续挂在停用产品下 | 已落地 |

补充说明：
- 当前“删除产品”会校验全部未删除库存设备；“停用产品”会校验仍处于启用状态的关联设备。
- 停用产品后，系统会拒绝新增设备、批量导入、设备更换目标建档、设备上报接入以及 MQTT 下行指令下发；`GET /api/device/product/list` 也只返回启用产品。
- 若当前设备需要纳入父子拓扑，新增/编辑/设备更换场景可直接维护 `parentDeviceId`；批量导入场景可通过 `parentDeviceCode` 建立父子关系。设备详情、列表和导出会同步展示父设备与关联网关信息，便于资产中心直接核对拓扑归属。
- 设备资产中心当前新增 `登记状态` 口径：`已登记` 行来自 `iot_device` 主档；`未登记` 行来自最近一次未建档上报失败归档，默认优先取 `iot_device_access_error_log` 最近一条失败记录，并按设备补齐仅存在 `iot_device_message_log(messageType=dispatch_failed)` 的最近上报；若共享环境尚未补齐 access-error 表或列结构，则整体回退到 `dispatch_failed` 失败轨迹。
- 未登记上报名单当前只用于“发现设备、补档建账、定位失败阶段”，因此在 `/devices` 中保持只读：可查看详情、失败阶段、Topic、TraceId 与原始载荷，但不支持编辑、设备更换、删除或跳转对象洞察。
- `/devices` 详情抽屉当前固定采用独立 `DeviceDetailWorkbench` 扁平工作台：外层抽屉继续保留大标题和说明，用于先交代当前设备上下文；正文已登记设备按“资产概览与运行概览 -> 身份与部署台账 -> 运行与认证台账 -> 关系与建档补充 -> 扩展元数据快照”展开，未登记设备按“来源档案与失败摘要 -> 最近载荷”展开，不再保留内部 hero 区、右上状态带、附件区或重复字段说明文案。

#### 2.1.3 产品字段定义口径

| 字段 | 本质定义 | 建议写法 |
|---|---|---|
| `productKey` | 机器侧接入身份标识，用于产品唯一识别、设备建档、MQTT Topic、消息日志和审计追踪 | 使用固定英文小写编码，推荐模式：`{vendorCode}-{scene}-{category}[-{protocolOrVersion}]`；不要包含中文、空格、项目名、站点名、客户名等易变信息 |
| `productName` | 面向人读的人类可理解名称，用于列表、详情、运营协作与报表识别 | 推荐模式：`厂商 + 场景/用途 + 品类 [+ 协议/系列/版本]`，例如 `南方测试 监测型 GNSS`、`中海达 预警型 爆闪灯终端` |
| `manufacturer` | 厂商品牌/制造商的标准名称，用于主数据归类 | 只填厂商，如 `南方测试`、`中海达`；不要写成 `中海达GNSS`、`南方测试雨量计项目` 这类混合值 |

补充规则：
- `vendorCode` 需要团队先约定并保持稳定，例如统一使用 `nf` 表示南方测试、`zhd` 表示中海达；不要今天用 `south`，明天改成 `nf`。
- 如果同一厂商、同一品类下存在多个协议版本或 payload 版本，建议在 `productKey` 末尾加版本后缀，如 `-v1`、`-v2` 或 `-mqttjson-v1`。
- 同一项目、同一站点里的多台同类设备，应共用一个产品，把序列号、安装点位、区域、设备编号放到设备维度维护。
- 如果只是换了项目名称、客户名称、安装位置，不应新建产品；如果换了协议、物模型或下行指令，则应评估新建产品。

- 联动规则：CRUD + 自动闭环命中留痕来源。
- 应急预案：CRUD + 自动闭环场景化匹配来源。
- 报表分析：风险趋势、告警统计、事件闭环、设备健康。

## 3. 核心业务流程

### 3.1 设备接入主流程（HTTP）

1. 先创建产品与物模型，明确 `productKey`、协议、节点类型、数据格式和厂商。
2. 创建设备并绑定产品。
3. 设备经 `/api/message/http/report` 上报。
4. 协议层标准化上报消息。
5. 写入消息日志与最新属性。
6. 更新设备在线/最近上报时间。

### 3.2 设备接入主流程（MQTT）

1. 设备按标准 Topic 或 `$dp` 上报。
2. 解析 Topic 中的 `productKey` 与 `deviceCode`，鉴别直连/子设备/兼容模式。
3. 协议适配器完成解码、安全校验、必要解密。
4. 写入日志与属性，刷新在线状态。
5. 异常链路记录为 `system_error` 审计，并可触发现有 webhook/企业消息通知。

### 3.3 告警与事件闭环流程

1. 规则触发告警（`iot_alarm_record`）。
2. 告警确认/抑制/关闭。
3. 必要时转事件（`iot_event_record`）并派发工单（`iot_event_work_order`）。
4. 工单执行并反馈，事件关闭形成闭环。
5. 报表聚合输出闭环指标。

补充口径：
- 2026-03-21 起，深部位移 `$dp` 子设备链路在属性落库后会发布风险判定事件，由 `alarm` 模块按 `risk_point_device` 绑定自动完成平台内闭环。
- 2026-04-02 起，自动闭环判级优先消费启用中的 `rule_definition`；`2026-04-06` 起，规则匹配会优先按 `riskMetricId` 命中风险指标目录，缺失目录时再回退到 `metricIdentifier`，仅当当前测点仍未命中可执行阈值策略时，才回退到 `iot.alarm.auto-closure.*` 默认阈值。
- 默认回退判级仍按毫米绝对值执行：蓝 `< 5`、黄 `[5, 10)`、橙 `[10, 20)`、红 `>= 20`。
- 黄色仅建告警；橙色、红色会继续自动转事件并派发工单；蓝色只更新风险点等级，不生成告警/事件。
- 自动闭环默认按 `deviceCode + metricIdentifier + level` 做 30 分钟冷却去重，风险点等级按全部绑定测点的最新值取最高等级后回写。
- 告警/事件留痕当前会写入 `policySource`、阈值文本、联动命中结果与预案命中结果，供风险运营与对象洞察统一回看。
- 自动闭环命中应急预案时，会优先筛选与“深部位移/位移/边坡/滑坡”及风险点上下文相关的启用预案；若只有无关通用预案，则留痕中的 `emergencyPlan` 为空，不再误挂无关预案。
- 已升级实例之间会额外使用 Redis 分布式锁抑制同一 `deviceCode + metricIdentifier + level` 的并发重复建单；若共享环境仍运行旧版 consumer，验收时可能出现新旧留痕并存，应先统一升级再判最终结果。

### 3.4 系统治理流程

1. `POST /api/auth/login` 登录，前端持久化 token。
2. 后端按用户角色返回动态菜单与按钮权限。
3. 前端请求自动附加 `Authorization: Bearer <token>`。
4. 业务页面按权限渲染按钮并调用对应接口。
5. 审计过滤器记录操作结果，失败和异常进入排障链路。
