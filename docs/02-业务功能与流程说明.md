# 02 业务功能与流程说明

更新时间：2026-03-19

## 1. 业务域划分

后端业务仍保持“三大业务域”，前端导航则按角色工作场景重组为“五大工作台”：

1. 后端设备接入域：产品、设备、协议、消息、属性、下行指令。
2. 后端风险处置域：告警、事件、工单、风险点、阈值规则、联动规则、应急预案、分析报表。
3. 后端系统治理域：组织、用户、角色、菜单、区域、字典、通知渠道、审计日志。
4. 前端五大工作台：`接入智维`、`风险运营`、`风险策略`、`平台治理`、`质量工场`。

## 2. 当前可用能力清单

### 2.1 设备接入域

- 产品定义中心首个闭环：分页筛选、详情查看、新增、编辑、单删、CSV 导出、按产品查看关联设备库存。
- 设备新增/查询（按 `id`、按 `deviceCode`）。
- 设备资产中心第二轮闭环：分页筛选、详情查看、新增、编辑、批量导入、设备更换、单删、批量删除、CSV 导出。
- HTTP 模拟上报。
- MQTT 上行接入（标准 Topic + `$dp` 兼容）。
- `mqtt-json` 协议解码。
- 消息日志持久化与查询。
- 最新属性更新。
- 设备在线状态更新。
- 下行发布最小能力（MQTT publish）。
- 消息追踪（按 TraceId/设备/Topic 检索）。

#### 2.1.1 产品与设备的逻辑关系

- 产品是平台里的“接入模板”，不是某一台具体硬件。一个产品定义了一组设备共同遵循的 `productKey`、`protocolCode`、`nodeType`、`dataFormat`、`manufacturer` 和物模型边界。
- 设备是某个产品下的具体资产实例。设备补充的是 `deviceCode`、鉴权凭据、固件版本、IP、安装位置、在线状态、最近上报时间等实例信息。
- 平台关系可以简化为：`1 个产品 -> N 台设备 -> N 条消息 / 属性 / 命令记录`。同一产品下可以有很多设备，但设备不能脱离产品单独定义接入协议。
- 是否应该拆成新产品，不看“是不是同一个项目”，而看“接入契约是否一致”。协议编码、Topic/报文结构、物模型、认证方式、下行指令、节点类型或生命周期策略任一项发生变化，都应评估拆成不同产品。

#### 2.1.2 建档顺序与治理口径

| 规则 | 说明 | 当前状态 |
|---|---|---|
| 先定义产品身份与协议，再允许设备建档 | 设备建档时必须先通过 `productKey` 找到产品，再继承产品的协议编码与节点类型 | 已落地 |
| `productKey` 创建后不允许修改 | `productKey` 是接入身份，变更后会影响建档、Topic 和日志检索 | 已落地 |
| 删除产品前先清理库存设备 | 产品删除前会校验是否还有关联设备库存 | 已落地 |
| 停用产品前先核查库存设备是否仍在使用 | 停用前会校验该产品下是否仍存在启用中的关联设备，避免存量在用设备继续挂在停用产品下 | 已落地 |

补充说明：
- 当前“删除产品”会校验全部未删除库存设备；“停用产品”会校验仍处于启用状态的关联设备。
- 停用产品后，系统会拒绝新增设备、批量导入、设备更换目标建档、设备上报接入以及 MQTT 下行指令下发；`GET /api/device/product/list` 也只返回启用产品。

#### 2.1.3 产品字段定义口径

| 字段 | 本质定义 | 建议写法 |
|---|---|---|
| `productKey` | 机器侧接入身份标识，用于产品唯一识别、设备建档、MQTT Topic、消息日志和审计追踪 | 使用固定英文小写编码，推荐模式：`{vendorCode}-{scene}-{category}[-{protocolOrVersion}]`；不要包含中文、空格、项目名、站点名、客户名等易变信息 |
| `productName` | 面向人读的人类可理解名称，用于列表、详情、运营协作与报表识别 | 推荐模式：`厂商 + 场景/用途 + 品类 [+ 协议/系列/版本]`，例如 `南方测试 监测型 GNSS`、`中海达 预警型 爆闪灯终端` |
| `manufacturer` | 厂商品牌/制造商的标准名称，用于主数据归类 | 只填厂商，如 `南方测试`、`中海达`；不要写成 `中海达GNSS`、`南方测试雨量计项目` 这类混合值 |

补充规则：
- `vendorCode` 需要团队先约定并保持稳定，例如统一使用 `nf` 表示南方测试、`zhd` 表示中海达；不要今天用 `south`，明天改成 `nf`。
- 如果同一厂商、同一品类下存在多个协议版本或 payload 版本，建议在 `productKey` 末尾加版本后缀，如 `-v1`、`-v2` 或 `-mqttjson-v1`。
- 同一项目、同一站点里的多台同类设备，应共用一个产品，把序列号、安装点位、区域、设备编号放到设备维度维护。
- 如果只是换了项目名称、客户名称、安装位置，不应新建产品；如果换了协议、物模型或下行指令，则应评估新建产品。

#### 2.1.4 结合你当前场景的建议拆分方式

| 厂商 | 场景 | 品类 | 建议 `productKey` | 建议 `productName` | 建议 `manufacturer` | 拆分理由 |
|---|---|---|---|---|---|---|
| 南方测试 | 监测 | GNSS | `nf-monitor-gnss-v1` | `南方测试 监测型 GNSS` | `南方测试` | GNSS 监测上报字段、告警阈值和物模型通常独立 |
| 南方测试 | 监测 | 深部位移 | `nf-monitor-deep-displacement-v1` | `南方测试 监测型 深部位移仪` | `南方测试` | 深部位移与 GNSS 的测点结构、属性模型不同 |
| 南方测试 | 监测 | 雨量计 | `nf-monitor-rain-gauge-v1` | `南方测试 监测型 雨量计` | `南方测试` | 雨量采集周期、指标含义和告警口径不同 |
| 中海达 | 监测 | GNSS | `zhd-monitor-gnss-v1` | `中海达 监测型 GNSS` | `中海达` | 与其他品类相比，GNSS 物模型和接入语义独立 |
| 中海达 | 监测 | 多维 | `zhd-monitor-multidim-v1` | `中海达 监测型 多维位移仪` | `中海达` | 多维位移设备通常拥有独立测轴和属性集合 |
| 中海达 | 监测 | 雨量计 | `zhd-monitor-rain-gauge-v1` | `中海达 监测型 雨量计` | `中海达` | 与监测 GNSS、多维设备的物模型和规则不同 |
| 中海达 | 预警 | 广播 | `zhd-warning-broadcast-v1` | `中海达 预警型 广播终端` | `中海达` | 广播设备更偏控制/播报指令，不应与监测类产品混在一起 |
| 中海达 | 预警 | 爆闪灯 | `zhd-warning-strobe-v1` | `中海达 预警型 爆闪灯终端` | `中海达` | 爆闪灯的下行控制能力和状态字段独立 |
| 中海达 | 预警 | 情报板 | `zhd-warning-info-board-v1` | `中海达 预警型 情报板` | `中海达` | 情报板通常包含文本/版式下发，与广播、爆闪灯不同 |

进一步建议：
- 如果“中海达监测型 GNSS”有多种硬件型号，但协议、Topic、字段和命令完全一致，可以继续共用一个产品，在 `description` 或设备备注里区分型号。
- 如果同一品类后来切换了新的协议版本、密钥体系或物模型，请新建 `v2` 产品，不要在已有设备存量上直接改 `productKey`。
- 项目名称、库区名称、边坡名称、监测点名称都不建议进入 `productKey` 和 `manufacturer`；这些信息应维护在设备、组织、区域或风险点维度。

## 3. 核心业务流程

### 3.1 设备接入主流程（HTTP）

1. 先创建产品与物模型，明确 `productKey`、协议、节点类型、数据格式和厂商。
2. 创建设备并绑定产品。
3. 设备经 `/api/message/http/report` 上报。
4. 协议层标准化上报消息。
5. 写入消息日志与最新属性。
6. 更新设备在线/最近上报时间。

### 3.2 设备接入主流程（MQTT）

1. 设备按标准 Topic 或 `$dp` 上报。
2. 解析 Topic 中的 `productKey` 与 `deviceCode`，鉴别直连/子设备/兼容模式。
3. 协议适配器完成解码、安全校验、必要解密。
4. 写入日志与属性，刷新在线状态。
5. 异常链路记录为 `system_error` 审计，并可触发 webhook 通知。

### 3.3 告警与事件闭环流程

1. 规则触发告警（`iot_alarm_record`）。
2. 告警确认/抑制/关闭。
3. 必要时转事件（`iot_event_record`）并派发工单（`iot_event_work_order`）。
4. 工单执行并反馈，事件关闭形成闭环。
5. 报表聚合输出闭环指标。

### 3.4 系统治理流程

1. `POST /api/auth/login` 登录，前端持久化 token。
2. 后端按用户角色返回动态菜单与按钮权限。
3. 前端请求自动附加 `Authorization: Bearer <token>`。
4. 业务页面按权限渲染按钮并调用对应接口。
5. 审计过滤器记录操作结果，失败和异常进入排障链路。

## 5. 当前菜单结构与角色化工作台基线

### 5.1 已落地的一级工作台

当前前端导航与菜单种子已统一为五个一级工作台：

| 一级工作台 | 当前挂载功能 | 主要角色 |
|---|---|---|
| 接入智维 | `产品定义中心`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` | 运维人员、开发人员 |
| 风险运营 | `实时监测台`、`GIS态势图`、`告警运营台`、`事件协同台`、`对象洞察台`、`运营分析中心` | 业务人员、管理人员 |
| 风险策略 | `风险对象中心`、`阈值策略`、`联动编排`、`应急预案库` | 管理人员、运维人员、开发人员 |
| 平台治理 | `组织架构`、`账号中心`、`角色权限`、`导航编排`、`区域版图`、`数据字典`、`通知编排`、`审计中心` | 管理人员、超级管理员 |
| 质量工场 | `自动化工场` | 开发人员、测试人员、超级管理员 |

### 5.2 结构调整结果

本轮实际完成的结构调整如下：

1. `对象洞察台` 已从接入侧移入 `风险运营`，用于承接监测对象属性、日志与研判线索。
2. `实时监测台`、`GIS态势图` 不再长期独立为“风险增强”一级菜单，而是收口到 `风险运营`。
3. `自动化工场` 已从 `系统治理` 剥离，独立进入 `质量工场`，避免工程工具稀释治理入口。
4. `设备接入 / 预警处置 / 系统治理` 的旧命名已分别升级为 `接入智维 / 风险运营 / 平台治理`，并新增 `风险策略 / 质量工场` 两个顶层工作台。

### 5.3 角色首页与默认落点

后端 `/api/auth/me` 返回的 `homePath` 已按角色收口为工作台首页，默认规则如下：

| 角色 | 默认首页 | 说明 |
|---|---|---|
| 业务人员 | `/risk-disposal` | 默认先进入风险运营总览 |
| 管理人员 | `/risk-disposal` | 先看风险运营，再进入风险策略和平台治理 |
| 运维人员 | `/device-access` | 默认先进入接入智维总览 |
| 开发人员 | `/device-access` | 默认先进入接入智维，总览后继续到风险策略和质量工场 |
| 超级管理员 | `/system-management` | 默认先进入平台治理总览 |

若目标工作台根菜单未授权，后端会退回到首个可访问业务页面，避免空首页。

## 6. 待确认与缺失

- `【待确认】` `实时监测`、`GIS 风险态势` 完成真实环境验收后，是直接并入正式 `风险运营`，还是继续保留灰度标签。
- `【文档缺失】` 当前尚无正式的“角色首页/默认路由/默认可见菜单”配置说明，首页 preset 与数据库角色授权还未形成统一产品规则。
- `【文档缺失】` 多租户跨组织数据权限模型（只看到 `tenant_id` 和角色菜单关系）。
