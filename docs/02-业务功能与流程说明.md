## 1. 业务域划分

后端业务仍保持“三大业务域”，前端导航则按角色工作场景重组为“五大工作台”：

1. 后端设备接入域：产品、设备、协议、消息、属性、下行指令。
2. 后端风险处置域：告警、事件、工单、风险点、阈值规则、联动规则、应急预案、分析报表。
3. 后端系统治理域：组织、用户、角色、菜单、区域、字典、通知渠道、审计日志。
4. 前端五大工作台：`接入智维`、`风险运营`、`风险策略`、`平台治理`、`质量工场`。

## 2. 当前可用能力清单

### 2.1 设备接入域

- 产品定义中心当前闭环：分页筛选、详情查看、新增、编辑、单删、CSV 导出、按产品查看关联设备库存。
- 产品物模型设计器当前继续挂在 `/products` 产品定义中心内，不新增一级路由；同一抽屉内分为“候选提炼”和“正式模型”两种模式。
- 设计器抽屉当前已升级为大工作台结构：统一采用“舞台头部层 + 总览指标层 + 主体工作区层”，候选提炼模式固定为“真实证据概览 + 候选目录 + 候选主列表 + 确认写入 rail”，正式模型模式固定为“正式契约总览 + 类型概览条 + 统一资产卡列表”。
- 候选提炼当前按产品逐个执行：属性候选优先来自 `iot_device_property` 并辅以 `iot_device_message_log` 最近真实上报，事件 / 服务只有存在真实证据时才会生成候选；无证据时直接展示空态提示，不伪造契约。
- 当候选提炼接口暂不可用、后端尚未升级到候选提炼版本，或真实环境实例短暂失联时，抽屉当前会自动降级到“正式模型”模式，保证既有 `iot_product_model` 维护不被阻断。
- 正式模型维护当前继续提供 `property / event / service` 三类物模型的列表、新增、编辑、删除、类型切换和 JSON 合法性提示；正式卡片与候选卡复用同一套资产骨架，只有人工确认过的候选才会写入 `iot_product_model`。
- 产品详情当前额外提供设备活跃度统计：今日活跃、7 日活跃、30 日活跃，以及平均在线时长、最长在线时长。
- 设备新增/查询（按 `id`、按 `deviceCode`）。
- 设备资产中心第三轮闭环：分页筛选、详情查看、新增、编辑、批量导入、设备更换、单删、批量删除、CSV 导出、父设备/子设备拓扑维护，以及“已登记设备 + 未登记上报名单”同页联查。
- 链路验证中心模拟上报（HTTP / MQTT 双通道，设备编码反查接入契约）。
- MQTT 上行接入（标准 Topic + `$dp` 兼容）。
- `mqtt-json` 协议解码。
- 消息日志持久化与查询。
- 最新属性更新。
- 设备在线状态更新。
- 下行发布最小能力（MQTT publish）。
- 消息追踪（按 TraceId/设备/Topic 检索）。
- 接入处理时间线复盘（按 `sessionId / traceId` 查看固定 Pipeline 阶段、耗时、处理类/方法与摘要）。

补充口径：
- `/reporting` 当前已收口为“设备编码 -> 查询设备 -> 发送模拟 -> 诊断复盘”交互。`deviceCode` 是唯一主查询入口；查询成功后前端只读反显 `productKey`、`protocolCode`、`clientId=deviceCode`。
- 页面保留 `tenantId` 与 `topic` 可编辑，其中 `tenantId` 默认 `1` 且仅 HTTP 提交生效；`topic` 默认 `$dp`，HTTP / MQTT 均允许手工覆盖。
- 明文模式下，只有可识别 JSON 才参与 C.1 / C.2 / C.3 判定与帧构造；XML 与普通文本仅提供格式化和原样发送，不新增 XML 协议解析。
- 当传输方式为 MQTT、`topic=$dp` 且明文内容为普通属性 JSON 且缺少设备身份字段时，前端会在实际发送 payload 中自动补入 `deviceCode`，保证默认兼容链路可用。
- 发送成功后，页面会在“处理时间线”面板中展示同一次提交的固定 Pipeline 复盘：HTTP 直接展示 timeline，MQTT 先显示“等待消费回流”，绑定 `traceId` 后再展示完整时间线。
- `/message-trace` 当前详情抽屉会按 `traceId` 异步拉取同一条 Redis 时间线；若时间线已过期，页面明确提示“时间线已过期，仅保留消息日志”。

#### 2.1.1 产品与设备的逻辑关系

- 产品是平台里的“接入模板”，不是某一台具体硬件。一个产品定义了一组设备共同遵循的 `productKey`、`protocolCode`、`nodeType`、`dataFormat`、`manufacturer` 和物模型边界。
- 设备是某个产品下的具体资产实例。设备补充的是 `deviceCode`、鉴权凭据、固件版本、IP、安装位置、在线状态、最近上报时间等实例信息。
- 平台关系可以简化为：`1 个产品 -> N 台设备 -> N 条消息 / 属性 / 命令记录`。同一产品下可以有很多设备，但设备不能脱离产品单独定义接入协议。
- 是否应该拆成新产品，不看“是不是同一个项目”，而看“接入契约是否一致”。协议编码、Topic/报文结构、物模型、认证方式、下行指令、节点类型或生命周期策略任一项发生变化，都应评估拆成不同产品。

#### 2.1.2 建档顺序与治理口径

| 规则 | 说明 | 当前状态 |
|---|---|---|
| 先定义产品身份与协议，再允许设备建档 | 设备建档时必须先通过 `productKey` 找到产品，再继承产品的协议编码与节点类型 | 已落地 |
| `productKey` 创建后不允许修改 | `productKey` 是接入身份，变更后会影响建档、Topic 和日志检索 | 已落地 |
| 删除产品前先清理库存设备 | 产品删除前会校验是否还有关联设备库存 | 已落地 |
| 停用产品前先核查库存设备是否仍在使用 | 停用前会校验该产品下是否仍存在启用中的关联设备，避免存量在用设备继续挂在停用产品下 | 已落地 |

补充说明：
- 当前“删除产品”会校验全部未删除库存设备；“停用产品”会校验仍处于启用状态的关联设备。
- 停用产品后，系统会拒绝新增设备、批量导入、设备更换目标建档、设备上报接入以及 MQTT 下行指令下发；`GET /api/device/product/list` 也只返回启用产品。
- 若当前设备需要纳入父子拓扑，新增/编辑/设备更换场景可直接维护 `parentDeviceId`；批量导入场景可通过 `parentDeviceCode` 建立父子关系。设备详情、列表和导出会同步展示父设备与关联网关信息，便于资产中心直接核对拓扑归属。
- 设备资产中心当前新增 `登记状态` 口径：`已登记` 行来自 `iot_device` 主档；`未登记` 行来自最近一次未建档上报失败归档，默认优先取 `iot_device_access_error_log` 最近一条失败记录，并按设备补齐仅存在 `iot_device_message_log(messageType=dispatch_failed)` 的最近上报；若共享环境尚未补齐 access-error 表或列结构，则整体回退到 `dispatch_failed` 失败轨迹。
- 未登记上报名单当前只用于“发现设备、补档建账、定位失败阶段”，因此在 `/devices` 中保持只读：可查看详情、失败阶段、Topic、TraceId 与原始载荷，但不支持编辑、设备更换、删除或跳转对象洞察。

#### 2.1.3 产品字段定义口径

| 字段 | 本质定义 | 建议写法 |
|---|---|---|
| `productKey` | 机器侧接入身份标识，用于产品唯一识别、设备建档、MQTT Topic、消息日志和审计追踪 | 使用固定英文小写编码，推荐模式：`{vendorCode}-{scene}-{category}[-{protocolOrVersion}]`；不要包含中文、空格、项目名、站点名、客户名等易变信息 |
| `productName` | 面向人读的人类可理解名称，用于列表、详情、运营协作与报表识别 | 推荐模式：`厂商 + 场景/用途 + 品类 [+ 协议/系列/版本]`，例如 `南方测试 监测型 GNSS`、`中海达 预警型 爆闪灯终端` |
| `manufacturer` | 厂商品牌/制造商的标准名称，用于主数据归类 | 只填厂商，如 `南方测试`、`中海达`；不要写成 `中海达GNSS`、`南方测试雨量计项目` 这类混合值 |

补充规则：
- `vendorCode` 需要团队先约定并保持稳定，例如统一使用 `nf` 表示南方测试、`zhd` 表示中海达；不要今天用 `south`，明天改成 `nf`。
- 如果同一厂商、同一品类下存在多个协议版本或 payload 版本，建议在 `productKey` 末尾加版本后缀，如 `-v1`、`-v2` 或 `-mqttjson-v1`。
- 同一项目、同一站点里的多台同类设备，应共用一个产品，把序列号、安装点位、区域、设备编号放到设备维度维护。
- 如果只是换了项目名称、客户名称、安装位置，不应新建产品；如果换了协议、物模型或下行指令，则应评估新建产品。

进一步建议：
- 如果“中海达监测型 GNSS”有多种硬件型号，但协议、Topic、字段和命令完全一致，可以继续共用一个产品，在 `description` 或设备备注里区分型号。

## 3. 核心业务流程

### 3.1 设备接入主流程（HTTP）

1. 先创建产品与物模型，明确 `productKey`、协议、节点类型、数据格式和厂商。
2. 创建设备并绑定产品。
3. 设备经 `/api/message/http/report` 上报。
4. 协议层标准化上报消息。
5. 写入消息日志与最新属性。
6. 更新设备在线/最近上报时间。

### 3.2 设备接入主流程（MQTT）

1. 设备按标准 Topic 或 `$dp` 上报。
2. 解析 Topic 中的 `productKey` 与 `deviceCode`，鉴别直连/子设备/兼容模式。
3. 协议适配器完成解码、安全校验、必要解密。
4. 写入日志与属性，刷新在线状态。
5. 异常链路记录为 `system_error` 审计，并可触发现有 webhook/企业消息通知。

### 3.3 告警与事件闭环流程

1. 规则触发告警（`iot_alarm_record`）。
2. 告警确认/抑制/关闭。
3. 必要时转事件（`iot_event_record`）并派发工单（`iot_event_work_order`）。
4. 工单执行并反馈，事件关闭形成闭环。
5. 报表聚合输出闭环指标。

补充口径：
- 2026-03-21 起，深部位移 `$dp` 子设备链路在属性落库后会发布风险判定事件，由 `alarm` 模块按 `risk_point_device` 绑定自动完成平台内闭环。
- 判级按毫米绝对值执行：蓝 `< 5`、黄 `[5, 10)`、橙 `[10, 20)`、红 `>= 20`。
- 黄色仅建告警；橙色、红色会继续自动转事件并派发工单；蓝色只更新风险点等级，不生成告警/事件。
- 自动闭环默认按 `deviceCode + metricIdentifier + level` 做 30 分钟冷却去重，风险点等级按全部绑定测点的最新值取最高等级后回写。
- 自动闭环命中应急预案时，会优先筛选与“深部位移/位移/边坡/滑坡”及风险点上下文相关的启用预案；若只有无关通用预案，则留痕中的 `emergencyPlan` 为空，不再误挂无关预案。
- 已升级实例之间会额外使用 Redis 分布式锁抑制同一 `deviceCode + metricIdentifier + level` 的并发重复建单；若共享环境仍运行旧版 consumer，验收时可能出现新旧留痕并存，应先统一升级再判最终结果。

### 3.4 系统治理流程

1. `POST /api/auth/login` 登录，前端持久化 token。
2. 后端按用户角色返回动态菜单与按钮权限。
3. 前端请求自动附加 `Authorization: Bearer <token>`。
4. 业务页面按权限渲染按钮并调用对应接口。
5. 审计过滤器记录操作结果，失败和异常进入排障链路。

## 4. 业务语义边界（非交付判定）

### 4.1 当前稳定业务语义

- Phase 1~3 主链路继续作为设备接入与平台闭环的稳定业务语义。
- Phase 4 当前稳定业务语义覆盖实时监测、GIS 态势图、告警、事件、风险策略、报表分析、系统治理和系统内容治理。
- 产品定义中心当前语义定位为“产品主数据、接入契约与产品物模型治理入口”。
- 设备资产中心当前语义定位为“设备库存、父子拓扑、运行状态和未登记上报名单联查”。

### 4.2 当前已冻结并启动实现的下一阶段单一增强语义

- 产品定义中心下一子项：产品物模型设计器（独立任务卡见 [2026-03-24-product-model-designer-implementation-plan.md](./superpowers/plans/2026-03-24-product-model-designer-implementation-plan.md)）

冻结依据：

## 5. 当前菜单结构与角色化工作台基线

### 5.1 已落地的一级工作台

当前前端导航与菜单种子已统一为五个一级工作台：

| 一级工作台 | 当前挂载功能 | 主要角色 |
|---|---|---|
| 接入智维 | `产品定义中心`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` | 运维人员、开发人员 |
| 风险运营 | `实时监测台`、`GIS态势图`、`告警运营台`、`事件协同台`、`对象洞察台`、`运营分析中心` | 业务人员、管理人员 |
| 风险策略 | `风险对象中心`、`阈值策略`、`联动编排`、`应急预案库` | 管理人员、运维人员、开发人员 |
| 平台治理 | `组织架构`、`账号中心`、`角色权限`、`导航编排`、`区域版图`、`数据字典`、`通知编排`、`站内消息`、`帮助文档`、`审计中心` | 管理人员、超级管理员 |
| 质量工场 | `自动化工场` | 开发人员、测试人员、超级管理员 |

补充口径：

- `接入智维总览` 当前定位为“中枢台”，只负责跨模块判断、最近使用恢复、推荐处理顺序和 6 个专台入口，不再在总览页重复子页正文。
- `产品定义中心`、`设备资产中心` 当前定位为“资产治理专台”：页面上方只保留一句治理判断和必要入口，主战区仍然是台账、详情抽屉和治理动作。
- `链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 当前定位为“诊断专台”：页面上方只保留紧凑状态条，主战区分别是模拟验证、异常台账、追踪台账和单设备校验结果。

### 5.2 结构调整结果

本轮实际完成的结构调整如下：

1. `对象洞察台` 已从接入侧移入 `风险运营`，用于承接监测对象属性、日志与研判线索。
2. `实时监测台`、`GIS态势图` 已于 2026-03-24 完成真实环境复验，正式收口到 `风险运营`，不再保留灰度标签或独立“风险增强”口径。
3. `自动化工场` 已从 `系统治理` 剥离，独立进入 `质量工场`，避免工程工具稀释治理入口。
4. `设备接入 / 预警处置 / 系统治理` 的旧命名已分别升级为 `接入智维 / 风险运营 / 平台治理`，并新增 `风险策略 / 质量工场` 两个顶层工作台。

### 5.3 角色首页与默认落点

后端 `/api/auth/me` 返回的 `homePath` 已按角色收口为工作台首页；当前前端也已补齐同口径 fallback 规则，在历史菜单数据缺失或 `homePath` 为空时，仍按同一套角色策略回落到对应工作台总览。默认规则如下：

| 角色 | 默认首页 | 说明 |
|---|---|---|
| 业务人员 | `/risk-disposal` | 默认先进入风险运营总览 |
| 管理人员 | `/risk-disposal` | 先看风险运营，再进入风险策略和平台治理 |
| 运维人员 | `/device-access` | 默认先进入接入智维总览 |
| 开发人员 | `/device-access` | 默认先进入接入智维，总览后继续到风险策略和质量工场 |
| 超级管理员 | `/system-management` | 默认先进入平台治理总览 |

若目标工作台根菜单未授权，后端会退回到首个可访问业务页面，避免空首页。

### 5.4 导航与搜索工作方式

- 一级导航、二级导航、分组首页、路由 `title/description`、角色默认落点当前统一由前端共享 workspace schema 派生，避免 `AppShell`、分组首页与路由表各自维护一套菜单语义。
- 顶部搜索已升级为全局命令面板，支持 `Ctrl/Cmd + K` 快捷唤起，按当前账号可见菜单在五个工作台内检索功能入口。
- 右上角 `通知中心 / 帮助中心` 已升级为角色化壳层入口：通知中心按当前角色、工作台、最近操作和失败记录聚合 `系统事件 / 业务事件 / 错误事件`；帮助中心按当前菜单权限筛选 `业务类 / 技术类 / FAQ` 资料。
- 2026-03-21 起，`spring-boot-iot-system` 已提供与壳层一致的后端支撑；2026-03-21 晚间起，前端壳层已优先消费站内消息与帮助文档接口，本地内容构造仅在接口不可用时作为兜底，分类口径继续保持 `系统事件 / 业务事件 / 错误事件` 与 `业务类 / 技术类 / FAQ`。
- 壳层摘要弹层当前只展示少量分组卡片，并在底部统一提供“查看更多”；通知摘要额外提供“全部已读”，帮助摘要继续只做消费入口，不新增独立菜单或私有页面模块。
- 通知消费闭环当前已收口为“摘要 -> 列表抽屉 -> 详情抽屉 -> 显式已读”：打开摘要或详情都不会自动清空未读，只有点击“标记已读”或“全部已读”时才更新未读徽标。
- 2026-03-22 起，通知中心生产侧也已收口到统一内部发布服务：`system_error`、事件派工、工单接收/开始/完成都会自动生成站内消息，并继续由全局壳层消费，不新增独立通知页面。
- 通知中心当前固定来源类型为 `manual / system_error / event_dispatch / work_order / governance`；治理侧可筛选来源、查看统计，并对系统自动消息执行“查看或停用”，不再直接改正文或删除。
- 2026-03-22 深夜起，通知中心已新增“高优未读阈值后桥接通知渠道”闭环：
  - 只对 `high / critical` 站内消息生效。
  - 超过阈值仍未读时，后端按调度扫描并复用 `/api/system/channel` 既有渠道配置桥接到外部群通知，不新增新的多渠道通知产品页。
  - 桥接仍按渠道 `config.scenes` 过滤，场景名固定为 `in_app_unread_bridge`。
- 2026-03-22 晚间起，平台治理 `/in-app-message` 已在原消息治理列表下方补齐“桥接效果运营”专区：
  - 不新增独立菜单或路由，继续留在同页折叠区内完成桥接统计和日志治理。
  - 管理侧可查看桥接记录数、桥接成功数、待重试数、桥接成功率、渠道/来源分布，以及桥接日志分页。
  - 单条桥接记录可打开独立桥接详情抽屉，同时查看消息原文和按尝试序号倒序排列的逐次尝试明细。
- 帮助消费闭环当前已收口为“摘要 -> 列表抽屉 -> 详情抽屉”：列表抽屉支持关键字搜索、服务端分页、当前页相关资料优先排序，以及标题/摘要/关键词/正文四处安全高亮。
- 2026-03-21 夜间起，平台治理工作台已新增 `/in-app-message` 与 `/help-doc` 前端编排页，管理/超管可直接维护消息分类、投放范围、角色/用户定向、帮助分类、可见角色和关联页面路径。
- 原“最近访问标签栏”已移除，功能页顶部仅保留面包屑；页面定位继续由一级工作台、左侧二级菜单和命令面板承担。
