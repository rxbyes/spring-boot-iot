## 1. 业务域划分

后端业务仍保持“三大业务域”，前端导航则按角色工作场景重组为“五大工作台”：

1. 后端设备接入域：产品、设备、协议、消息、属性、下行指令。
2. 后端风险处置域：告警、事件、工单、风险点、阈值规则、联动规则、应急预案、分析报表。
3. 后端系统治理域：组织、用户、角色、菜单、区域、字典、通知渠道、审计日志。
4. 前端五大工作台：`接入智维`、`风险运营`、`风险策略`、`平台治理`、`质量工场`。

### 1.1 三域关系总览

三大后端业务域的关系不是并列平铺，而是稳定的“数据源 -> 业务闭环 -> 治理支撑”关系：

1. `IoT 主链路`
   - 负责产品、设备、物模型、属性/遥测、消息日志，是平台上游数据源。
2. `风险平台`
   - 消费 IoT 主链路结果，完成风险点、规则、告警、事件、工单闭环。
3. `系统管理`
   - 提供组织、区域、字典、通知渠道、审计日志等治理底座。

主关系固定为：

1. `IoT 主链路 -> 风险平台`
2. `系统管理 -> IoT 主链路`
3. `系统管理 -> 风险平台`
4. `风险平台 -> IoT 主链路`

- `2026-04-23` 起，对象洞察进入“按设备拓扑角色分场景展示”改造执行阶段，目标是先收口读侧真相，避免同一产品下 `单台直报` 与 `复合子设备` 共用同一套快照/趋势模板。统一口径如下：
  1. `/insight` 设备角色解析固定收口为 `collector_parent / collector_child / standalone`，并成为后端与前端共同真相；不再允许一处只按 `nodeType=2` 判断、另一处再单独兼容 `nf-collect-rtu-v1 + nodeType=1`。
  2. 父采集器快照只展示父设备自有运行参数（如 `signal_4g / battery / temp / humidity / ext_power_volt / lon / lat / sw_version`），不再把子设备业务值（如 `dispsX / dispsY / value`）作为父快照主体展示。
  3. 子设备与单台设备继续共用 canonical 契约（深部位移 `dispsX / dispsY / sensor_state`，激光测距 `value / sensor_state`），但对象洞察展示按角色过滤：`collector_child` 仅展示子设备业务值 + `sensor_state`，`standalone` 可额外展示该设备本机运行参数。
  4. 子设备状态语义从“有值/无值”升级为 `REPORTED_NORMAL / REPORTED_ABNORMAL / MISSING / STALE`；若同包业务报文已覆盖 `L1_SW_1~8` 但状态仅回报 `L1_SW_1`，其余通道必须显式标记 `MISSING`，不再显示为 `--`。

#### 2.1.0 IoT 五层治理模型

`产品 -> 设备 -> 物模型 -> 属性/遥测 -> 消息日志` 是当前设备接入域的固定治理顺序，不应被理解为彼此独立的并列对象：

1. `产品`
   - 定义一类设备的接入模板和业务身份。
2. `设备`
   - 定义某台具体资产实例。
3. `物模型`
   - 定义平台认可的正式字段标准。
4. `属性/遥测`
   - 定义运行期最新值和历史值。
5. `消息日志`
   - 定义原始报文证据和处理留痕。

最小治理规则：

1. 先建产品，再建设备。
2. 同一产品下正式字段 `identifier` 必须唯一。
3. 核心判定字段必须同时明确 latest 与 telemetry 口径。
4. 成功与失败上报都必须保留消息日志。
5. 父设备 / 子设备场景必须明确风险责任归属。

详细字段治理表与业务建档联调步骤，参考 [appendix/iot-field-governance-and-sop.md](./appendix/iot-field-governance-and-sop.md)。

补充口径：
- `/reporting` 当前已升级为“排障起点 + 结果复盘优先”的链路验证中心：默认先进入 `结果复盘`，共享诊断状态头会先回答 `当前节点：链路验证`，再统一表达 `尚未验证 / 验证成功 / 等待回流 / 验证失败 / 复盘受限` 与下一步诊断分支，并继续保留 `模拟验证 / 结果复盘 / 最近记录` 三个真实业务页签。
- `deviceCode` 仍是链路验证中心唯一主查询入口；查询成功后前端只读反显 `productKey`、`protocolCode`、`clientId=deviceCode`。
- 页面保留 `tenantId` 与 `topic` 可编辑，其中 `tenantId` 默认 `1` 且仅 HTTP 提交生效；`topic` 默认 `$dp`，HTTP / MQTT 均允许手工覆盖。
- 明文模式下，只有可识别 JSON 才参与 C.1 / C.2 / C.3 判定与帧构造；XML 与普通文本仅提供格式化和原样发送，不新增 XML 协议解析。
- 当传输方式为 MQTT、`topic=$dp` 且明文内容为普通属性 JSON 且缺少设备身份字段时，前端会在实际发送 payload 中自动补入 `deviceCode`，保证默认兼容链路可用。
- 发送成功后，页面会自动切回 `结果复盘` 页签，并在共享诊断状态头和“处理时间线”区域同时表达同一次提交的固定 Pipeline 复盘：HTTP 直接展示 timeline，MQTT 先显示“等待消费回流”，绑定 `traceId` 后再展示完整时间线。
- `最近记录` 当前已从“恢复入口”升级为诊断清单：每条记录优先展示 verdict、当前卡点和建议动作，再提供恢复当前复盘上下文的入口。
- `/message-trace` 当前详情抽屉已定稿为客户视角收纳后的扁平工作台语法：抽屉标题行承担轻刊头，但不再重复展示 `消息类型 / TraceId / Topic` 标签，避免和正文重复；这些识别信息统一由 `消息态势与处理概况` 和 `链路与接入台账` 承接。正文完整保留 `消息态势与处理概况`、`链路与接入台账`、`Payload 对照`、`处理时间线` 四个章节；命中过协议模板化拆分时，允许在 `链路与接入台账` 与 `Payload 对照` 之间插入轻量 `协议模板证据`。其中显式带 `UTC / offset` 的上报时间会统一换算为中国时区 `Asia/Shanghai` 展示，无时区的历史时间串继续按原值显示，避免已是本地时间的日志再次偏移。`链路与接入台账` 当前固定回答 `TraceId / 日志 ID / 创建时间 / 设备编码 / 路由类型 / 产品标识 / Topic`，统一使用纵向台账行，不再保留 `链路标识 / 接入上下文` 左右分组。`Payload 对照` 优先根据消息日志详情接口恢复 `原始 Payload / 解密后明文 / 解析结果`，展示形态固定为三条纵向折叠板块：默认收起、按需展开，并为每个板块补齐复制动作。即使 Redis 时间线已过期，也仍可继续查看恢复后的明文与解析结果；若恢复接口给出空字符串或空对象，前端仍需回退到原始 Payload 兜底显示，避免用户误判为业务未解析。`处理时间线` 当前默认折叠，但首屏必须固定展示 `当前状态 / 处理节点 / Trace 归属 / 存储提示` 四张摘要卡；展开后才进入固定 Pipeline 复盘，不再影响 Payload 对照的可读性。
- `/message-trace` 的 `失败归档` 模式当前会优先保留阻塞型 MQTT Pipeline 失败的真实上下文：当异常发生在 `TOPIC_ROUTE / PROTOCOL_DECODE / DEVICE_CONTRACT / MESSAGE_LOG / PAYLOAD_APPLY / DEVICE_STATE / RISK_DISPATCH` 等固定阶段时，归档记录会尽量沿用当次链路已解析出的 `traceId / deviceCode / productKey / protocolCode / messageType / topic`，并把 `failureStage` 优先对齐到真实 Pipeline 阶段；若 `contractSnapshot` 组装过程中再次命中脏 schema 或产品查询异常，平台会跳过快照但继续保留基础失败样本，不再因为附加快照恢复失败把整条失败归档吞掉。
- `/message-trace` 下的 `接入失败归档台` 当前继续复用同一路由工作区和共享列表骨架；行级动作固定直出 `详情 / 追踪 / 观测` 三项，不再保留行内 `更多`。失败详情抽屉固定按 `失败态势与处理概况 / 链路与主体台账 / 异常诊断与回跳 / 契约与报文快照` 纵向排布，并通过台账行与快照块承接 `TraceId / Topic / 异常摘要 / contractSnapshot / rawPayload` 等排障证据。
- `接入智维` 当前已通过共享诊断上下文把 `/reporting`、`/message-trace`、`/system-log`、`/file-debug` 与治理页串成标准排障树：`链路验证中心 -> 链路追踪台 / 异常观测台 / 数据校验台 -> 产品定义中心 / 设备资产中心`。完整决策树只在 `/device-access` 总览显式展示；各诊断子页只透传高价值定位字段与来源页，并通过 `当前节点 + 下一步` 回答本页职责，不新增后端聚合接口或第二套路由骨架。
- 当诊断结果指向契约治理或资产治理时，页面可直接跳入 `产品定义中心` 或 `设备资产中心`；治理页当前只在工作台内联提示“来自哪个诊断页 + 优先核对什么”，不再额外堆介绍墙、概况卡或重复导流文案。
- 2026-04-18 起，`接入智维` 9 个入口当前统一收口为“轻量页面壳 + 真业务页签/单主内容区”结构：总览=`资产底座 / 诊断排障`，无代码接入=`单主工作台（统一接入案例、阻塞原因与下一步动作）`，产品=`单主列表页（物模型设计器与关联设备回收到行操作和抽屉）`，协议治理=`单主工作台（协议族 / 解密档案 / 协议模板同页治理）`，设备=`单主列表页（未登记资产与拓扑治理回收到同一主台账）`，链路验证=`模拟验证 / 结果复盘 / 最近记录`，异常观测=`单主列表页（跨页协同下沉到行级动作与详情抽屉）`，链路追踪=`链路追踪 / 失败归档`，数据校验=`设备校验 / 原始响应`。

#### 2.1.0A 监测编码扩展模板（L1~L4）

- `2026-04-23` 起，`契约字段` 的扩展接入识别口径固定为“编码前缀 + 字段叶子”双维识别：优先按既有正式字段和专用场景识别，未命中时再按原始编码兜底，最后由人工提炼确认。`2026-04-25` 起，规范兜底结果统一表达为 `MATCHED / AMBIGUOUS / MISSED`，不再只回答“命中/未命中”。
- 编码结构统一按 `Lx_XX_n`（监测类）或 `Sx_XX_n`（状态类）：`x` 为监测内容编码（如 `1/2/3/4`），`XX` 为监测类型编码（如 `LF/NW/QW/SW/JS/QJ`），`n` 为同类测点序号。
- 同一测点允许两类上报形态并存：`标量值`（如 `"L4_NW_1": 0`）与 `对象叶子`（如 `"L3_QW_1":{"value":24.6}`）；对象叶子中的 `value/temp/totalValue/...` 作为字段叶子参与识别。
- 自动识别优先级固定为：`已发布正式字段完全命中 > 专用规范场景命中 > 编码前缀兜底判定（Lx/Sx + XX + 叶子） > 手动提炼确认`；编码兜底只在 `MATCHED` 时给出唯一 canonical，`AMBIGUOUS` 只给候选与建议，`MISSED` 继续保留原始字段。
- 正式字段收口规则固定为“同一语义只保留一个正式标识符”：可切换展示中文名，但 `identifier` 不允许在 `L4_NW_1` 与 `L4_NW_1.value` 之间来回漂移。
- 风险闭环边界固定为规范目录 `riskEnabled=1`：`riskEnabled=0` 字段可用于对象洞察/属性快照/历史趋势展示，但不进入风险闭环。

|---|---|---|---|---|

- 这份模板用于“未来设备接入前的治理底座”，不代表所有类型设备已接入完成。
- 新设备未入专用场景前，优先依赖 `Lx/Sx + XX + 叶子` 做自动识别；仍无法命中时继续走人工提炼，并沉淀映射规则/显示规则。
- 状态镜像字段（如 `S1_ZT_1.sensor_state.<logicalChannelCode>`）继续按既有状态治理规则执行，不与业务字段模板混用。
- `phase3-water-surface` 与 `phase6-radar` 已纳入共享基线，分别覆盖 `L3/DB/temp/value` 与 `L4/LD/X/Y/Z/speed` 的 compare/apply 和运行时归一回归；`2026-04-25` 起，附件规范中的 `phase1-vibration`、`phase2-acoustic-emission`、`phase3-settlement`、`phase3-air-pressure`、`phase4-surface-flow-speed` 也已纳入自动兜底种子，默认只做识别与展示，不进入风险闭环。
- compare 行展示口径固定为：展示 `MATCHED / AMBIGUOUS / MISSED`、命中来源、命中依据和候选项；冲突候选进入人工确认或映射规则建议，不得自动归一，也不得默认进入待生效或提交审批。
- compare 行治理动作固定为：`AMBIGUOUS` 优先跳转映射规则建议并携带 raw identifier；`MISSED` 优先跳转运行态名称/单位治理并预填 raw identifier；两类动作都只辅助治理，不直接创建或发布正式字段。
- 运行态规范库允许使用轻量缓存降低重复查询，但只作为 published snapshot 和映射规则均未命中后的兜底读侧/写侧辅助，不改变 `iot_product_model`、合同发布批次或 resolver snapshot 的正式真相。

#### 2.1.1 产品与设备的逻辑关系

- 产品是平台里的“接入模板”，不是某一台具体硬件。一个产品定义了一组设备共同遵循的 `productKey`、`protocolCode`、`nodeType`、`dataFormat`、`manufacturer` 和物模型边界。
- 设备是某个产品下的具体资产实例。设备补充的是 `deviceCode`、鉴权凭据、固件版本、IP、安装位置、在线状态、最近上报时间等实例信息。
- 平台关系可以简化为：`1 个产品 -> N 台设备 -> N 条消息 / 属性 / 命令记录`。同一产品下可以有很多设备，但设备不能脱离产品单独定义接入协议。
- 是否应该拆成新产品，不看“是不是同一个项目”，而看“接入契约是否一致”。协议编码、Topic/报文结构、物模型、认证方式、下行指令、节点类型或生命周期策略任一项发生变化，都应评估拆成不同产品。

#### 2.1.2 建档顺序与治理口径

| 规则 | 说明 | 当前状态 |
|---|---|---|
| 先定义产品身份与协议，再允许设备建档 | 设备建档时必须先通过 `productKey` 找到产品，再继承产品的协议编码与节点类型 | 已落地 |
| `productKey` 创建后不允许修改 | `productKey` 是接入身份，变更后会影响建档、Topic 和日志检索 | 已落地 |
| 删除产品前先清理库存设备 | 产品删除前会校验是否还有关联设备库存 | 已落地 |
| 停用产品前先核查库存设备是否仍在使用 | 停用前会校验该产品下是否仍存在启用中的关联设备，避免存量在用设备继续挂在停用产品下 | 已落地 |

补充说明：
- 当前“删除产品”会校验全部未删除库存设备；“停用产品”会校验仍处于启用状态的关联设备。
- 停用产品后，系统会拒绝新增设备、批量导入、设备更换目标建档、设备上报接入以及 MQTT 下行指令下发；`GET /api/device/product/list` 也只返回启用产品。
- 若当前设备需要纳入父子拓扑，新增/编辑/设备更换场景可直接维护 `parentDeviceId`；批量导入场景可通过 `parentDeviceCode` 建立父子关系。设备详情、列表和导出会同步展示父设备与关联网关信息，便于资产中心直接核对拓扑归属。
- 设备资产中心当前新增 `登记状态` 口径：`已登记` 行来自 `iot_device` 主档；`未登记` 行默认优先取 `iot_device_invalid_report_state` 中 `DEVICE_NOT_FOUND` 的最新态，并按设备补齐 `iot_device_access_error_log` 与 `iot_message_log(messageType=dispatch_failed)` 的失败摘要；若共享环境尚未补齐 invalid-report-state 表或列结构，则整体回退到 `access_error`，再回退 `dispatch_failed` 失败轨迹。
- 未登记上报名单当前仍用于“发现设备、补档建账、定位失败阶段”，但不再是只读名单：`/devices` 中可继续查看详情、失败阶段、Topic、TraceId 与原始载荷，同时也可直接通过 `编辑` 打开与已登记设备相同的完整表单完成正式建档；提交成功后，该记录会转成 `已登记` 设备，未登记来源不会继续回流。
- `2026-04-18` 起，`/devices` 又补齐“建议优先、人工确认”的半自动接入切片：带 `lastTraceId` 的未登记线索当前新增行级 `接入建议`，会按 `traceId` 读取推荐产品、协议族、解密档案、协议模板与 `ruleGaps[]`；工具条新增 `批量转正式设备`，只允许选中同批“未登记 + 带 Trace”线索执行。批量转正时后端会逐条复用当前建议校验 `suggestionStatus=READY` 且无缺口，再沿用既有设备建档真相链创建设备；不满足条件的线索会按 trace 原样拦截并返回原因，因此当前能力仍是“建议优先 + 人工确认”的半自动闭环，不是无人值守自动建档。
- `/devices` 详情抽屉当前固定采用独立 `DeviceDetailWorkbench` 扁平工作台：外层抽屉继续保留大标题和说明，用于先交代当前设备上下文；正文已登记设备按“资产概览与运行概览 -> 身份与部署台账 -> 运行与认证台账 -> 关系与建档补充 -> 扩展元数据快照”展开，未登记设备按“来源档案与失败摘要 -> 最近载荷”展开，不再保留内部 hero 区、右上状态带、附件区或重复字段说明文案。

#### 2.1.3 产品字段定义口径

| 字段 | 本质定义 | 建议写法 |
|---|---|---|
| `productKey` | 机器侧接入身份标识，用于产品唯一识别、设备建档、MQTT Topic、消息日志和审计追踪 | 使用固定英文小写编码，推荐模式：`{vendorCode}-{scene}-{category}[-{protocolOrVersion}]`；不要包含中文、空格、项目名、站点名、客户名等易变信息 |
| `productName` | 面向人读的人类可理解名称，用于列表、详情、运营协作与报表识别 | 推荐模式：`厂商 + 场景/用途 + 品类 [+ 协议/系列/版本]`，例如 `南方测试 监测型 GNSS`、`中海达 预警型 爆闪灯终端` |
| `manufacturer` | 厂商品牌/制造商的标准名称，用于主数据归类 | 只填厂商，如 `南方测试`、`中海达`；不要写成 `中海达GNSS`、`南方测试雨量计项目` 这类混合值 |

补充规则：
- `vendorCode` 需要团队先约定并保持稳定，例如统一使用 `nf` 表示南方测试、`zhd` 表示中海达；不要今天用 `south`，明天改成 `nf`。
- 如果同一厂商、同一品类下存在多个协议版本或 payload 版本，建议在 `productKey` 末尾加版本后缀，如 `-v1`、`-v2` 或 `-mqttjson-v1`。
- 同一项目、同一站点里的多台同类设备，应共用一个产品，把序列号、安装点位、区域、设备编号放到设备维度维护。
- 如果只是换了项目名称、客户名称、安装位置，不应新建产品；如果换了协议、物模型或下行指令，则应评估新建产品。

### 2.2 风险预警处置域

## 3. 核心业务流程

### 3.1 设备接入主流程（HTTP）

1. 先创建产品与物模型，明确 `productKey`、协议、节点类型、数据格式和厂商。
2. 创建设备并绑定产品。
3. 设备经 `/api/message/http/report` 上报。
4. 协议层标准化上报消息。
5. 写入消息日志与最新属性。
6. 更新设备在线/最近上报时间。

### 3.2 设备接入主流程（MQTT）

1. 设备按标准 Topic 或 `$dp` 上报。
2. 解析 Topic 中的 `productKey` 与 `deviceCode`，鉴别直连/子设备/兼容模式。
3. 协议适配器完成解码、安全校验、必要解密。
4. 写入日志与属性，刷新在线状态。
5. 异常链路记录为 `system_error` 审计，并可触发现有 webhook/企业消息通知。

### 3.3 告警与事件闭环流程

1. 规则触发告警（`iot_alarm_record`）。
2. 告警确认/抑制/关闭。
3. 必要时转事件（`iot_event_record`）并派发工单（`iot_event_work_order`）。
4. 工单执行并反馈，事件关闭形成闭环。
5. 报表聚合输出闭环指标。

补充口径：
- 2026-03-21 起，深部位移 `$dp` 子设备链路在属性落库后会发布风险判定事件，由 `alarm` 模块按 `risk_point_device` 绑定自动完成平台内闭环。
- 2026-04-02 起，自动闭环判级优先消费启用中的 `rule_definition`；`2026-04-06` 起，规则匹配会优先按 `riskMetricId` 命中风险指标目录，缺失目录时再回退到 `metricIdentifier`，仅当当前测点仍未命中可执行阈值策略时，才回退到 `iot.alarm.auto-closure.*` 默认阈值。
- 默认回退判级仍按毫米绝对值执行：蓝 `< 5`、黄 `[5, 10)`、橙 `[10, 20)`、红 `>= 20`。
- 黄色仅建告警；橙色、红色会继续自动转事件并派发工单；蓝色只更新风险点等级，不生成告警/事件。
- 自动闭环默认按 `deviceCode + metricIdentifier + level` 做 30 分钟冷却去重，风险点等级按全部绑定测点的最新值取最高等级后回写。
- 告警/事件留痕当前会写入 `policySource`、阈值文本、联动命中结果与预案命中结果，供风险运营与对象洞察统一回看。
- 自动闭环命中应急预案时，会优先筛选与“深部位移/位移/边坡/滑坡”及风险点上下文相关的启用预案；若只有无关通用预案，则留痕中的 `emergencyPlan` 为空，不再误挂无关预案。
- 已升级实例之间会额外使用 Redis 分布式锁抑制同一 `deviceCode + metricIdentifier + level` 的并发重复建单；若共享环境仍运行旧版 consumer，验收时可能出现新旧留痕并存，应先统一升级再判最终结果。

### 3.4 系统治理流程

1. `POST /api/auth/login` 登录，前端持久化 token。
2. 后端按用户角色返回动态菜单与按钮权限。
3. 前端请求自动附加 `Authorization: Bearer <token>`。
4. 业务页面按权限渲染按钮并调用对应接口。
5. 审计过滤器记录操作结果，失败和异常进入排障链路。

### 5.4 导航与搜索工作方式

- 前端共享 workspace schema 当前继续负责一级工作台分组顺序、概览入口与兼容路径扩展；同一路径的运行态菜单名称、简称和说明则优先复用 `/api/auth/me` 返回的动态菜单树，避免导航编排改名后左侧菜单仍停留在静态旧标题。
- 顶部搜索已升级为全局命令面板，支持 `Ctrl/Cmd + K` 快捷唤起，按当前账号可见菜单在五个工作台内检索功能入口。
- 右上角 `通知中心 / 帮助中心` 已升级为角色化壳层入口：通知中心按当前角色、工作台、最近操作和失败记录聚合 `系统事件 / 业务事件 / 错误事件`；帮助中心按当前菜单权限筛选 `业务类 / 技术类 / FAQ` 资料。
- 2026-04-02 起，右上角账号菜单统一只保留 `账号中心 / 修改密码 / 退出登录`；原 `实名认证`、`登录方式` 独立静态入口已并入账号中心抽屉，不再维持第二套壳层说明页。
- 2026-03-21 起，`spring-boot-iot-system` 已提供与壳层一致的后端支撑；2026-03-21 晚间起，前端壳层已优先消费站内消息与帮助文档接口，本地内容构造仅在接口不可用时作为兜底，分类口径继续保持 `系统事件 / 业务事件 / 错误事件` 与 `业务类 / 技术类 / FAQ`。
- 壳层摘要弹层当前只展示少量分组卡片，并在底部统一提供“查看更多”；通知摘要额外提供“全部已读”，帮助摘要继续只做消费入口，不新增独立菜单或私有页面模块。
- 账号中心当前收口为单一抽屉，固定展示 `基础资料 / 机构与角色 / 安全信息 / 实名资料` 四段；基础资料直接调用 `PUT /api/user/profile` 维护当前登录人的昵称、真实姓名、手机号、邮箱，机构与角色、安全信息统一复用 `/api/auth/me` 返回的租户、主机构、角色、登录方式、最近登录和数据范围摘要。
- 通知消费闭环当前已收口为“摘要 -> 列表抽屉 -> 详情抽屉 -> 显式已读”：打开摘要或详情都不会自动清空未读，只有点击“标记已读”或“全部已读”时才更新未读徽标。
- 2026-03-22 起，通知中心生产侧也已收口到统一内部发布服务：`system_error`、事件派工、工单接收/开始/完成都会自动生成站内消息，并继续由全局壳层消费，不新增独立通知页面。
- 通知中心当前固定来源类型为 `manual / system_error / event_dispatch / work_order / governance`；治理侧可筛选来源、查看统计，并对系统自动消息执行“查看或停用”，不再直接改正文或删除。
- 2026-03-22 深夜起，通知中心已新增“高优未读阈值后桥接通知渠道”闭环：
  - 只对 `high / critical` 站内消息生效。
  - 超过阈值仍未读时，后端按调度扫描并复用 `/api/system/channel` 既有渠道配置桥接到外部群通知，不新增新的多渠道通知产品页。
  - 桥接仍按渠道 `config.scenes` 过滤，场景名固定为 `in_app_unread_bridge`。
- 2026-03-22 晚间起，平台治理 `/in-app-message` 已在原消息治理列表下方补齐“桥接效果运营”专区：
  - 不新增独立菜单或路由，继续留在同页折叠区内完成桥接统计和日志治理。
  - 管理侧可查看桥接记录数、桥接成功数、待重试数、桥接成功率、渠道/来源分布，以及桥接日志分页。
  - 单条桥接记录可打开独立桥接详情抽屉，同时查看消息原文和按尝试序号倒序排列的逐次尝试明细。
- 帮助消费闭环当前已收口为“摘要 -> 列表抽屉 -> 详情抽屉”：列表抽屉支持关键字搜索、服务端分页、当前页相关资料优先排序，以及标题/摘要/关键词/正文四处安全高亮。
- 2026-03-21 夜间起，平台治理工作台已新增 `/in-app-message` 与 `/help-doc` 前端编排页，管理/超管可直接维护消息分类、投放范围、角色/用户定向、帮助分类、可见角色和关联页面路径。
- 2026-04-01 起，风险运营下的 `实时监测台 / GIS态势图 / 告警运营台 / 事件协同台`，以及平台治理下的 `组织架构 / 账号中心 / 角色权限 / 导航编排 / 区域版图 / 数据字典 / 通知编排 / 帮助文档管理` 已统一复用 `产品定义中心` 的列表交互合同：不再调整页面布局层级，只统一结果区骨架屏、已有结果后的局部刷新蒙层、空态分流、`10 / 20 / 50 / 100` 分页与请求乱序保护；其中 `GIS态势图` 继续保留“已定位 / 未定位”双分区，`组织架构 / 导航编排 / 区域版图` 继续保留“默认懒加载树 / 搜索后扁平分页”双模式。
- 原“最近访问标签栏”已移除，功能页顶部仅保留面包屑；页面定位继续由一级工作台、左侧二级菜单和命令面板承担。
