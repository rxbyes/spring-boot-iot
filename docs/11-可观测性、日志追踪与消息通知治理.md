# 11 可观测性、日志追踪与消息通知治理

> 文档定位：统一说明 TraceId、审计日志、系统异常、消息追踪、通知中心和渠道桥接的治理口径。
> 适用角色：运维、研发、测试、超级管理员、帮助中心技术稿维护者。
> 权威级别：一级权威。
> 上游来源：`AuditLogFilter`、`TraceContextFilter`、消息日志查询接口、`/api/system/in-app-message/**`、运行配置、真实环境验收文档。
> 下游消费：排障手册、帮助中心技术类文章、通知编排说明、运行验收。
> 变更触发条件：TraceId 机制、日志字段、通知场景、桥接策略、排障流程或相关 API 调整。
> 更新时间：2026-04-08

职责边界：
- 本文件只负责 TraceId、日志查询、通知中心、桥接策略和排障流程。
- 环境变量、升级脚本和共享环境安全基线，以 [07-部署运行与配置说明.md](./07-部署运行与配置说明.md) 为准。
- 帮助中心与系统内容治理规则，以 [12-帮助文档与系统内容治理.md](./12-帮助文档与系统内容治理.md) 为准。

## 1. 目标与边界

- 建设目标保持为“两条主线”：
  - 业务可审计：业务操作和结果可追溯。
  - 研发可排障：接入链路、后台异常和通知联动可定位。
- 当前不建设独立日志平台，也不把通知中心扩成新的多渠道消息产品。
- 帮助中心和通知中心都消费本治理规则，但不替代代码和数据库事实。

## 2. 当前能力清单

### 2.1 链路追踪

- `/api/*` 会自动生成或透传 `X-Trace-Id`。
- 同一条接入链路中的 HTTP、MQTT、系统异常审计和消息日志共享同一追踪号。
- 当前调度线程与应用线程池也会透传或补发 `traceId`，至少覆盖 `@Scheduled` 任务、平台线程池任务和 MQTT 接入主链路，避免 trace 只停留在 HTTP 主线程。
- 当前关键检索字段：
  - `trace_id`
  - `device_code`
  - `product_key`
  - `error_code`
  - `exception_class`

### 2.2 日志与查询面

- 业务审计：`/api/system/audit-log`，前端业务日志页默认排除 `system_error`。
- 系统日志：接入智维中的 `/system-log`，仍复用 `/api/system/audit-log`，固定查看 `operation_type=system_error`。
- `2026-04-08` 起，普通 HTTP 控制器异常即使已被 `GlobalExceptionHandler` 包装成统一失败响应，也会继续把原始异常透传给审计链路，并以 `system_error` 落入 `/system-log`；列表与详情应优先看到真实 `exceptionClass`、根因摘要和堆栈，而不是只有“系统繁忙，请稍后再试”。
- `接入失败归档` 与 `/system-log` 的边界继续固定：前者只承接 MQTT / `$dp` 接入失败样本，后者负责 Web/后台 `system_error`。像 `/api/risk-governance/coverage-overview` 这类 HTTP 查询接口抛出的 `BadSqlGrammarException`，应先在 `/system-log` 排查，而不是期待它出现在失败归档。
- 消息追踪：`GET /api/device/message-trace/page`，当前支持 `keyword` 精确命中 `TraceId / 设备编码 / 产品标识`，并继续兼容 `traceId / deviceCode / productKey / messageType / topic` 显式筛选；历史 `messageType=report` 会在服务层自动归一为 `property` 后查询。
- `message-flow` 时间线：`GET /api/device/message-flow/session/{sessionId}`、`GET /api/device/message-flow/trace/{traceId}`，用于查看固定 Pipeline 的阶段状态、耗时、处理类/方法和摘要。
- legacy `$dp` 的 `PROTOCOL_DECODE.summary` 当前会在家族观测开关打开时补充 `appId`、`familyCodes`、`normalizationStrategy`、`timestampSource`、`childSplitApplied`，便于按厂商 / 家族排查标准化差异。
- `TELEMETRY_PERSIST.summary` 当前会补齐 `legacyMappedMetricCount`、`legacyUnmappedMetricCount`、`fallbackMetricCount`、`fallbackReasons`；`NORMALIZED_FALLBACK_ONLY` 应视为治理信号，而不是自动等价于“家族已治理完成”。
- 排查 `iot_device_telemetry_point` 时，`reported_at` 表示设备采集时间，`ts` 表示通用回退表唯一行键时间；核对同一采集时刻是否发生覆盖，必须按 `device_code + reported_at + trace_id` 看多条 trace 是否共存，最新值读取口径应优先 `reported_at DESC`，再按 `ts DESC` 决定并列顺序。
- 当 `IOT_TELEMETRY_LEGACY_MAPPING_VALIDATE_ONLY=true` 时，协议层会输出 `event="legacy_dp_normalizer_validation_diff"` 的 `WARN` 摘要日志，用于比对 legacy `$dp` v1/v2 标准化与子消息拆分差异；该日志只做观测，不代表主链路已经切到 v2。
- `message-flow` 运维看板：`GET /api/device/message-flow/ops/overview`，用于查看运行时总量、相关性命中、lookup 健康度和阶段性能表。
- `message-flow` 最近会话：`GET /api/device/message-flow/recent`，用于在不预先记录 `sessionId` 的情况下恢复最近一批提交会话。
- 当前 `$dp` 兼容链路的 `PROTOCOL_DECODE.summary` 在 `iot.protocol.legacy-dp.family-observability-enabled=true` 时会补齐协议治理字段：
  - `appId`：仅加密外层包存在时返回
  - `familyCodes`：当前报文识别出的家族码列表
  - `normalizationStrategy`：默认 `LEGACY_DP`；若兼容开关关闭 V2，则返回 `LEGACY_DP_COMPAT`
  - `timestampSource`：当前区分 `PAYLOAD_TIMESTAMP` 与 `SERVER_TIME`
  - `childSplitApplied`：当前报文是否执行了基准站一包多测点拆分
- 当前 `legacy-compatible` 模式下的 `TELEMETRY_PERSIST.summary` 已补齐遥测治理字段：
  - `legacyMappedMetricCount`
  - `legacyUnmappedMetricCount`
  - `fallbackMetricCount`
  - `fallbackReason`
  其中 `fallbackReason` 用于标记 `MISSING_TDENGINE_LEGACY_MAPPING / MISSING_STABLE / MISSING_COLUMN / LEGACY_COLUMN_MISSING` 等治理缺口。
- `iot.telemetry.legacy-mapping-validate-only=true` 时，日志还会输出 `legacy_mapping_validate_only` 摘要，对比旧 metadata 快照与当前显式映射服务的命中差异；该模式只用于双轨观测，不改变本次持久化分支和结果。
- 前端 `message-trace` 页面当前已直接消费 `GET /api/device/message-trace/stats`，在同页 `notices + toolbar meta` 中展示最近 1h / 24h 追踪量、失败摘要量和高频消息 / 设备摘要，不再要求排障时先手工拼接口。
- 前端 `message-trace` 页面顶部当前已升级为完整 `message-flow` 运维看板：同页展示 `ops/overview` 指标卡、lookup 健康度和阶段性能表，并在旁侧补齐“最近 message-flow 会话”辅助区。
- 前端 `message-trace` 页面当前已补齐“失败归档”模式：同一路由下可切换查看 `GET /api/device/access-error/page` 与 `GET /api/device/access-error/stats`，直接联动 failureStage、契约快照和原始报文，不新增第二条受权限影响的静态路由。
- 前端 `message-trace` 页面当前已把 `链路追踪` 模式筛选交互收口为单行五项：`快速搜索（TraceId / 设备编码 / 产品标识） / 消息类型 / Topic / 查询 / 重置`，不再保留该模式下的“更多筛选”切换；正常消息行内动作只保留 `详情`，`观测` 只在 `失败归档` 等异常样本链路中保留。
- `GET /api/device/message-flow/detail/{id}` 当前在恢复 `Payload 对照` 时，会先尝试设备/产品当前协议，再按 topic/payload 继续补推 `mqtt-json` 候选；因此即使历史消息的当前协议信息已缺失，只要原始报文仍是 `$dp` 或 `/sys/...` 的 JSON 链路，详情仍应优先恢复真正的明文和解析结果，而不是把 `header.appId / bodies.body` 外层密文包回显成“解密后明文”。
- 前端 `/reporting` 页面当前只保留单次复盘语义：HTTP 立即展示 timeline，MQTT 先轮询 `sessionId` 等待绑定 trace，并在原页补齐“最近提交”面板用于恢复最近 session。
- `message-flow` 当前固定使用 Redis 短期留存：
  - `iot:message-flow:trace:{traceId}`
  - `iot:message-flow:session:{sessionId}`
  - `iot:message-flow:fingerprint:{fingerprint}`
  - `iot:message-flow:recent:sessions`
- 默认保留时长 `24` 小时；超过 TTL 后，消息日志与失败归档仍可查询，但时间线接口允许返回空结果。
- `message-flow` recent 索引当前只做 Redis 短期 ZSET，不新增数据库表，也不做多维二级索引；后端按最近窗口扫描 session JSON 并在查询时顺手清理已失效 sessionId。
- 开发态控制台噪音开关：当前已支持按类别控制 `mybatis.sql` 与 `org.mybatis.spring` 两类日志是否输出到控制台。
- 当前常规日志 pattern 已统一追加 `traceId`，并对最终输出统一做敏感字段脱敏。
- 当前已新增两条独立诊断文件链路：
  - `diagnostic.sql` -> `logs/diagnostic-sql.log`
  - `diagnostic.access` -> `logs/diagnostic-access.log`
- 当前慢日志能力：
  - `slow_sql`
  - `slow_http_request`
  - `slow_mqtt_dispatch`
- 慢日志只输出摘要字段，不回放整包 payload，也不污染 `spring-boot-iot-info.log`、`spring-boot-iot-error.log`。
- 设备接入链路当前额外输出 1 条 `message_flow_summary` 摘要日志，logger 名为 `message.flow`；其职责仅是索引 `traceId/sessionId` 和成功失败摘要，不再承担阶段展示责任。
- `message-flow` 当前还会记录 4 组低基数运行指标：
  - `iot.message.flow.session.total`，tags=`transportMode,status`
  - `iot.message.flow.stage.duration`，tags=`stage,transportMode,status`
  - `iot.message.flow.correlation.total`，tags=`result`
  - `iot.message.flow.lookup.total`，tags=`target,result`
- 当 MQTT 模拟发布超过 `session-match-window-seconds` 仍未命中真实消费回流时，系统会额外输出 `message_flow_correlation_miss` 摘要，并把 `correlation.total{result="missed"}` 累加到运维看板。
- 鉴权失败、MQTT 连接变化、协议分发失败、通知发送与规则化运维告警触发当前已统一为业务摘要日志，默认优先输出 `event / result / costMs / traceId / 主键字段`。
- 第二批业务里程碑日志当前已覆盖 `alarm_lifecycle`、`event_lifecycle`、`work_order_lifecycle` 三类摘要事件：
  - `alarm_lifecycle`：告警创建、确认、抑制、关闭
  - `event_lifecycle`：事件创建、派发、反馈、关闭
  - `work_order_lifecycle`：工单接收、开始处理、完成处理
- 上述里程碑日志默认只输出摘要字段，不重复打印 SQL、整包 JSON 或超长现场反馈。
- MQTT 无效上报治理当前已纳入首批运行口径：对 `DEVICE_NOT_FOUND` 与 `EMPTY_DECRYPTED_PAYLOAD` 先采样、后进入冷却抑制；原始空 payload 会在 consumer 入口直接丢弃。
- 失败归档：`GET /api/device/access-error/page`、`GET /api/device/access-error/{id}`，查看 MQTT / `$dp` 接入失败时的 `failureStage`、设备契约和原始 payload 快照；该表当前定位为“失败样本归档”，冷却期内重复坏包不再保证逐条留痕。
- 失败归档当前不接收普通 Web Controller / 管理端查询异常；若错误发生在 HTTP 控制器、Service 或 Mapper 查询链路，但并未进入 MQTT 接入 Pipeline，应统一落到 `sys_audit_log.system_error`，再由 `/system-log` 查看。
- MQTT 阻塞型 Pipeline 异常当前会优先携带真实阶段元数据写入失败归档与异常摘要，首批阶段口径固定为 `ingress / topic_route / protocol_decode / device_contract / message_log / payload_apply / telemetry_persist / device_state / risk_dispatch / message_dispatch`；只有在消费端完全拿不到阶段元数据时，才回退到旧的按上下文猜测阶段逻辑。
- 若失败归档在补建 `contractSnapshot` 时再次命中历史库 schema 漂移、产品查询异常或旧环境缺列，平台当前会记一条降级 `WARN` 并继续保留基础失败样本；`contractSnapshot` 可为空，但失败归档本身不再因为补快照失败被整条吞掉。
- `dispatch_failed` 失败轨迹当前也已跟随同一降级口径：即使按 `productKey` 补产品档案再次失败，仍会按缺省产品上下文继续写入消息日志，避免“异常已发生，但链路追踪和失败归档都没有样本”。
- 当前异常观测台不再保留页头右上角跳转菜单；需要联动 `message-trace` 的“失败归档”模式时，统一通过详情抽屉或正文动作承接，并继续自动带入 `traceId / deviceCode / productKey / topic / errorCode / exceptionClass` 等公共筛选条件。
- 聚合统计：
  - `GET /api/device/message-trace/stats`：查看最近 1h / 24h 追踪量、失败分发量，以及消息类型 / 产品 / 设备 / Topic 聚合。
  - `GET /api/device/access-error/stats`：查看最近 1h / 24h 失败量，以及 `failureStage / errorCode / exceptionClass / protocolCode / topic` 聚合；其中无筛选 `recentHourCount / recent24HourCount / topFailureStages` 当前读取 Redis `invalid-report` 分钟桶，其他 Top bucket 仍来自样本归档表。
- MQTT consumer 健康：`GET /actuator/health/mqttConsumer`，查看 `running`、`connected`、`clientId`、`brokerUrl`、`subscribeTopics`、`lastDisconnectAt` 以及最近一次连接/收包/成功分发/失败阶段摘要。
- MQTT 联调和 `$dp` 验证继续由 [14-MQTTX真实环境联调手册.md](./14-MQTTX真实环境联调手册.md) 提供操作步骤。
- 审计详情当前应继续保持敏感字段脱敏与超长内容截断，避免排障链路再次暴露 `password`、`token`、`secret` 一类原文。

开发态控制台类别说明：
- `mybatis.sql`：SQL 明细日志，包括 `Preparing`、`Parameters`、`Columns / Row / Total`。
- `org.mybatis.spring`：会话与连接噪音，包括 `JDBC Connection ... will not be managed by Spring`、`Closing non transactional SqlSession ...`。
- `diagnostic.sql`：慢 SQL 摘要，仅在 `IOT_OBSERVABILITY_DIAGNOSTIC_SQL_FILE_ENABLED=true` 时写入 `diagnostic-sql.log`。
- `diagnostic.access`：慢 HTTP / 慢 MQTT 摘要，仅在 `IOT_OBSERVABILITY_DIAGNOSTIC_ACCESS_FILE_ENABLED=true` 时写入 `diagnostic-access.log`。
- `message.flow`：固定 Pipeline 的摘要日志，默认在 `iot.observability.message-flow.enabled=true` 时输出 `INFO`。
- 这四类日志当前都不进入常规 info / error 文件日志。
- 若要精确关闭某个 Mapper 或某条语句的 SQL 明细，继续使用标准 `logging.level.mybatis.sql.<mapper-or-statement>=OFF|TRACE`；完整 logger 名必须带 `mybatis.sql.` 前缀。

### 2.3 通知中心

- 当前定位：全局壳层内的站内消息中心，不新增独立消费端菜单。
- 壳层后台自动刷新当前默认只拉 `GET /api/system/in-app-message/my/unread-count`；`GET /api/system/in-app-message/my/page` 只在用户显式打开通知面板或通知中心时才触发，避免焦点恢复、可见性恢复和跨标签同步时反复预拉摘要分页。
- 未读统计自动刷新当前必须同时满足两条减载约束：
  - 自动场景复用同一条进行中的未读统计请求，不得在前一条请求未完成时并发重发。
  - 自动场景需遵守最小刷新窗口：成功后默认 `60` 秒内不重复拉取，失败后按 `15` 秒退避重试；显式打开通知中心或跨标签已读同步允许强制刷新。
- `GET /api/system/in-app-message/my/unread-count` 服务端当前也已改为轻量查询：只读取未读统计所需字段，不再为角标统计额外选择正文列或执行整表排序，降低壳层自动刷新在共享环境中的连接占用时间。
- 当前固定消息来源类型：
  - `manual`
  - `system_error`
  - `event_dispatch`
  - `work_order`
  - `governance`
- 当前用户消费接口：
  - `GET /api/system/in-app-message/my/page`
  - `GET /api/system/in-app-message/my/unread-count`
  - `POST /api/system/in-app-message/my/read/{id}`
  - `POST /api/system/in-app-message/my/read-all`

### 2.4 外部通知渠道桥接

- 继续复用 `/api/system/channel` 的通知渠道，不另建第二套通知编排。
- 当前自动场景名固定为：
  - `system_error`
  - `observability_alert`
  - `in_app_unread_bridge`
- 支持渠道类型：
  - `webhook`
  - `wechat`
  - `feishu`
  - `dingtalk`
- `observability_alert` 与 `system_error` 明确分场景治理：前者只承接规则化运维阈值告警，后者继续承接后台异常审计自动通知，避免两类消息混在同一场景下失真。
- 规则化运维告警评估器当前固定每 `60` 秒执行一次，继续复用已有数据源：
  - `sys_audit_log` 中最近窗口 `system_error` 次数
  - `MqttConsumerRuntimeState` 中的连接/断连运行态
  - Redis `invalid-report` 分钟桶中的 `failureStage` 聚合
  - `iot_device_access_error_log` 的样本详情聚合
  - `sys_in_app_message_bridge_attempt_log` 的站内信桥接失败聚合
- 当前固定 5 类规则：
  - `system-error-burst`：最近 `10` 分钟 `system_error >= 5`
  - `mqtt-disconnect-timeout`：MQTT consumer 持续断连 `>= 5` 分钟
  - `failure-stage-spike`：最近 `10` 分钟同一 `failureStage >= 10`
  - `in-app-bridge-failure-burst`：最近 `10` 分钟同一渠道站内信桥接失败 `>= 3`
  - `risk-governance-missing-policy-burst`：当前同一 `riskMetricId / metricIdentifier` 下已绑定风险点但未启用阈值策略 `>= 3`
- 冷却策略：按 `ruleType + dimensionKey` 生成唯一 key，使用 Redis TTL 冷却 `30` 分钟；冷却期内重复命中不重复发通知。
- 触发结果：只在真正触发时发送外部通知并打印结构化 `WARN` 摘要；本阶段不新增告警表、不做恢复通知、不做独立告警大盘。
- `risk-governance-missing-policy-burst` 当前直接读取风险治理缺口聚合摘要：优先按 `riskMetricId` 聚合，缺失时回退到 `metricIdentifier`，只统计“已绑定风险点但没有启用阈值策略”的现存积压量，不把已启用规则的绑定重复计入。

## 3. 数据与接口口径

## 4. 排障主流程

### 4.1 标准流程

1. 发现问题：页面报错、链路异常、MQTT 上报失败、站内错误消息或外部群通知异常。
2. 取得 `TraceId`：优先从响应头、业务日志、系统日志或消息日志详情中获取。
3. 查询系统日志：先看 `/system-log` 是否存在对应 `system_error`，确认异常类型、模块、请求目标和结果。
   - 若报错来自管理端接口、控制面查询或浏览器页面接口，例如 `/api/risk-governance/coverage-overview`、`/api/risk-governance/dashboard-overview` 一类 HTTP 请求，默认先走这一步，而不是直接找失败归档。
   - MQTT `$dp` 场景下，`request_params` 当前会补齐 `deviceCode`、`productKey`、`protocolCode`、`clientId`、`topicRouteType`，优先用这些字段做接入契约比对。
4. 查询消息追踪：再到 `/message-trace` 按 `TraceId / deviceCode / topic` 复盘接入链路。
   - 页面顶部统计摘要会同步展示当前筛选条件下的最近 1h / 24h 数量、失败分发量和高频桶摘要，可先用来判断是否属于局部放大或全局性波动。
   - `message-flow` 运维看板会进一步展示运行时 session/correlation/lookup 指标与阶段性能表，便于快速判断是局部链路问题还是 Redis/相关性整体劣化。
  - 若当前链路来自 `$dp`，优先检查 `PROTOCOL_DECODE.summary.appId / familyCodes / normalizationStrategy / timestampSource / childSplitApplied`，先确认协议家族识别与时间戳来源是否符合预期，再继续判断后续 `PAYLOAD_APPLY / TELEMETRY_PERSIST` 是否异常。
  - 若 `TELEMETRY_PERSIST.branch` 为 `NORMALIZED_FALLBACK_ONLY` 或 `LEGACY_WITH_NORMALIZED_FALLBACK`，继续检查 `legacyMappedMetricCount / legacyUnmappedMetricCount / fallbackMetricCount / fallbackReason`，不要再把 fallback 命中视为“默认成功”。
   - 列表与详情中的显式 `UTC / offset` 上报时间当前会统一换算为中国时区 `Asia/Shanghai` 展示；无时区的历史时间串继续按原值显示，避免把已经是本地时间的日志再次偏移。若 `/message-trace` 内的 `reportTime` 相对 `createTime` 呈现典型 8 小时漂移，则前端会继续按链路追踪专用口径做中国时间纠偏，避免同一条消息在列表和详情里保留错误时差。
   - 详情抽屉会继续按 `traceId` 拉取 `message-flow` 时间线；若时间线已过期，页面会明确提示“时间线已过期，仅保留消息日志”，但 `Payload 对照` 仍应继续展示从消息日志恢复出的原始报文、解密明文与解析结果。
   - 详情抽屉标题行当前不再重复展示 `消息类型 / TraceId / Topic` 标签，也不再在左上角副标题区域回显 topic；识别信息统一以下方 `链路标识 / 接入上下文` 为准，其中 `链路标识` 固定为 `日志 ID / 创建时间 / TraceId`，`Topic` 固定保留在 `接入上下文`。
   - 列表中的 `Payload 摘要` 及其 hover 当前只展示截断后的短预览，不再默认暴露整段 payload；详情 `消息概览 / 接入上下文` 中的长 `产品标识` 也统一改为单行省略，并通过 hover 查看全量。
   - 若消息日志尚未落库或怀疑失败发生在设备校验前，可直接切换到同一路由下的“失败归档”模式，继续查看 failureStage、契约快照与原始 payload。
   - 若当前问题来自 `/reporting`，可直接先按 `sessionId` 查询 `/api/device/message-flow/session/{sessionId}`，HTTP 会立即返回 timeline，MQTT 需等待消费回流绑定 trace；若不记得 `sessionId`，可直接先查 `GET /api/device/message-flow/recent` 或使用页面中的“最近提交”面板恢复。
5. 若主链路在设备校验前失败：继续查询 `/api/device/access-error/page`，核对 `failureStage`、原始 payload、解码后的 `deviceCode / productKey / protocolCode` 与 `contractSnapshot.expectedProtocolCode / protocolSource`。
   - 只有当问题发生在 MQTT / `$dp` 接入主链路，或消息尚未进入 Web 控制器查询链路时，才应优先使用失败归档；不要把普通控制面 SQL 异常误判成接入归档缺失。
   - 对 MQTT `$dp` 阻塞型异常，`failureStage` 当前应优先看到真实 Pipeline 阶段，例如产品查档报错通常会落到 `device_contract`，而不应再被统一折叠成 `topic_route`。
   - 若当前失败样本没有 `contractSnapshot`，先不要直接判定为“未归档”；这通常表示归档时补建快照再次命中了旧 schema 或产品查询异常，基础失败样本仍应保留，可继续结合 `traceId / deviceCode / productKey / errorClass` 排查。
6. 对照运行配置：核对 [07-部署运行与配置说明.md](./07-部署运行与配置说明.md) 中的环境变量、依赖服务和脚本前置。
7. 核对通知闭环：若涉及 `system_error` 自动通知、高优未读桥接或规则化运维告警，再检查渠道 `config.scenes`、未读阈值、Redis 冷却键与桥接留痕。
8. 留存验收证据：将结论回写到 `logs/acceptance/*`、验收记录或当前缺陷单。

补充说明：
- 若当前排障需要查看 SQL 明细，可临时设置 `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SQL_ENABLED=true` 后重启应用。
- 若当前排障需要查看 MyBatis-Spring 会话行为，可临时设置 `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SESSION_ENABLED=true` 后重启应用。
- 若当前排障需要查看慢 SQL、慢 HTTP 或慢 MQTT 摘要，可临时设置 `IOT_OBSERVABILITY_DIAGNOSTIC_SQL_FILE_ENABLED=true` 或 `IOT_OBSERVABILITY_DIAGNOSTIC_ACCESS_FILE_ENABLED=true`，并按需调整对应慢日志阈值。
- 若只想屏蔽某一个 SQL logger，请直接在 `application-dev.yml` 或环境变量里设置 `logging.level.mybatis.sql.<mapper-or-statement>=OFF`，不要省略 `mybatis.sql.` 前缀。
- 排障结束后，开发环境建议恢复默认关闭，避免大批量 SQL 与 SqlSession 控制台噪音再次淹没有效日志。

### 4.2 最小检查单

- 是否拿到了同一条链路的 `TraceId`
- `/actuator/health/mqttConsumer` 是否显示 `connected=true`
- `sys_audit_log` 是否存在对应 `system_error`
- `iot_message_log` 是否存在对应上行或分发失败记录
- `iot_device_access_error_log` 是否存在对应失败归档
- `deviceCode / productKey / topic` 是否与产品建档一致
- `contractSnapshot.expectedProtocolCode / actualProtocolCode / protocolSource` 是否与设备/产品建档一致
- 相关环境变量和通知渠道 `scenes` 是否开启
- `observability_alert` 场景渠道是否已配置且 Redis 冷却键可写
- Redis 中 `message-flow` key 是否可写且未提前过期
- `GET /api/device/message-flow/ops/overview` 中 `lookupCounts` 是否出现连续 `error`
- MQTT 模拟会话是否在 `session-match-window-seconds` 窗口外仍停留 `correlationPending=true`
- 是否属于真实环境 schema 缺失或历史库未升级问题

## 5. 通知治理规则

### 5.1 system_error 自动通知

- 开关：`IOT_OBSERVABILITY_SYSTEM_ERROR_NOTIFY_ENABLED=true`
- 渠道前提：`config.url` 非空，且 `scenes` 中包含 `system_error`
- 当前目标：把后台异常同步到既有外部群渠道和站内消息中心

### 5.2 高优未读桥接

- 开关：`IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_ENABLED=true`
- 阈值配置：
  - `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_SCAN_INTERVAL_SECONDS`
  - `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_HIGH_THRESHOLD_MINUTES`
  - `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_CRITICAL_THRESHOLD_MINUTES`
- 当前策略：
  - 只扫描 `high / critical`
  - 只对仍未读对象生效
  - 每条消息在每个渠道上“成功一次即闭环”
  - 失败按渠道冷却后重试

### 5.3 规则化运维告警

- 总开关：`IOT_OBSERVABILITY_ALERTING_ENABLED=true`
- 渠道前提：`config.url` 非空，且 `scenes` 中包含 `observability_alert`
- 调度与冷却：
  - 评估周期固定 `IOT_OBSERVABILITY_ALERTING_EVALUATE_INTERVAL_SECONDS=60`
  - 冷却固定 `IOT_OBSERVABILITY_ALERTING_COOLDOWN_MINUTES=30`
  - Redis key 口径固定为 `规则类型 + 维度键`
- 当前固定规则：
  - `system-error-burst`
  - `mqtt-disconnect-timeout`
  - `failure-stage-spike`
  - `in-app-bridge-failure-burst`
- 当前边界：
  - 继续复用已有异常统计、MQTT 运行态、Redis `invalid-report` 分钟桶、接入失败样本归档和桥接失败统计，不新增独立告警表
  - 触发时只发送告警通知和打印 `WARN` 摘要，不生成恢复通知
  - 不在本阶段新增独立告警看板或手工处置流
