# 11 可观测性、日志追踪与消息通知治理

> 文档定位：统一说明 TraceId、审计日志、系统异常、消息追踪、通知中心和渠道桥接的治理口径。
> 适用角色：运维、研发、测试、超级管理员、帮助中心技术稿维护者。
> 权威级别：一级权威。
> 上游来源：`AuditLogFilter`、`TraceContextFilter`、消息日志查询接口、`/api/system/in-app-message/**`、运行配置、真实环境验收文档。
> 下游消费：排障手册、帮助中心技术类文章、通知编排说明、运行验收。
> 变更触发条件：TraceId 机制、日志字段、通知场景、桥接策略、排障流程或相关 API 调整。
> 更新时间：2026-03-23

职责边界：
- 本文件只负责 TraceId、日志查询、通知中心、桥接策略和排障流程。
- 环境变量、升级脚本和共享环境安全基线，以 [07-部署运行与配置说明.md](./07-部署运行与配置说明.md) 为准。
- 帮助中心与系统内容治理规则，以 [12-帮助文档与系统内容治理.md](./12-帮助文档与系统内容治理.md) 为准。

## 1. 目标与边界

- 建设目标保持为“两条主线”：
  - 业务可审计：业务操作和结果可追溯。
  - 研发可排障：接入链路、后台异常和通知联动可定位。
- 当前不建设独立日志平台，也不把通知中心扩成新的多渠道消息产品。
- 帮助中心和通知中心都消费本治理规则，但不替代代码和数据库事实。

## 2. 当前能力清单

### 2.1 链路追踪

- `/api/*` 会自动生成或透传 `X-Trace-Id`。
- 同一条接入链路中的 HTTP、MQTT、系统异常审计和消息日志共享同一追踪号。
- 当前调度线程与应用线程池也会透传或补发 `traceId`，至少覆盖 `@Scheduled` 任务、平台线程池任务和 MQTT 接入主链路，避免 trace 只停留在 HTTP 主线程。
- 当前关键检索字段：
  - `trace_id`
  - `device_code`
  - `product_key`
  - `error_code`
  - `exception_class`

### 2.2 日志与查询面

- 业务审计：`/api/system/audit-log`，前端业务日志页默认排除 `system_error`。
- 系统日志：接入智维中的 `/system-log`，仍复用 `/api/system/audit-log`，固定查看 `operation_type=system_error`。
- 消息追踪：`GET /api/device/message-trace/page`，支持按 `TraceId`、设备编码、产品标识、消息类型和 Topic 分页检索。
- `message-flow` 时间线：`GET /api/device/message-flow/session/{sessionId}`、`GET /api/device/message-flow/trace/{traceId}`，用于查看固定 Pipeline 的阶段状态、耗时、处理类/方法和摘要。
- 前端 `message-trace` 页面当前已直接消费 `GET /api/device/message-trace/stats`，在同页 `notices + toolbar meta` 中展示最近 1h / 24h 追踪量、失败摘要量和高频消息 / 设备摘要，不再要求排障时先手工拼接口。
- 前端 `message-trace` 页面当前已补齐“失败归档”模式：同一路由下可切换查看 `GET /api/device/access-error/page` 与 `GET /api/device/access-error/stats`，直接联动 failureStage、契约快照和原始报文，不新增第二条受权限影响的静态路由。
- `message-flow` 当前固定使用 Redis 短期留存：
  - `iot:message-flow:trace:{traceId}`
  - `iot:message-flow:session:{sessionId}`
  - `iot:message-flow:fingerprint:{fingerprint}`
- 默认保留时长 `24` 小时；超过 TTL 后，消息日志与失败归档仍可查询，但时间线接口允许返回空结果。
- 开发态控制台噪音开关：当前已支持按类别控制 `mybatis.sql` 与 `org.mybatis.spring` 两类日志是否输出到控制台。
- 当前常规日志 pattern 已统一追加 `traceId`，并对最终输出统一做敏感字段脱敏。
- 当前已新增两条独立诊断文件链路：
  - `diagnostic.sql` -> `logs/diagnostic-sql.log`
  - `diagnostic.access` -> `logs/diagnostic-access.log`
- 当前慢日志能力：
  - `slow_sql`
  - `slow_http_request`
  - `slow_mqtt_dispatch`
- 慢日志只输出摘要字段，不回放整包 payload，也不污染 `spring-boot-iot-info.log`、`spring-boot-iot-error.log`。
- 设备接入链路当前额外输出 1 条 `message_flow_summary` 摘要日志，logger 名为 `message.flow`；其职责仅是索引 `traceId/sessionId` 和成功失败摘要，不再承担阶段展示责任。
- 鉴权失败、MQTT 连接变化、协议分发失败、通知发送与规则化运维告警触发当前已统一为业务摘要日志，默认优先输出 `event / result / costMs / traceId / 主键字段`。
- 第二批业务里程碑日志当前已覆盖 `alarm_lifecycle`、`event_lifecycle`、`work_order_lifecycle` 三类摘要事件：
  - `alarm_lifecycle`：告警创建、确认、抑制、关闭
  - `event_lifecycle`：事件创建、派发、反馈、关闭
  - `work_order_lifecycle`：工单接收、开始处理、完成处理
- 上述里程碑日志默认只输出摘要字段，不重复打印 SQL、整包 JSON 或超长现场反馈。
- 失败归档：`GET /api/device/access-error/page`、`GET /api/device/access-error/{id}`，查看 MQTT / `$dp` 接入失败时的 `failureStage`、设备契约和原始 payload 快照。
- 当前异常观测台右上角已可直接跳到 `message-trace` 的“失败归档”模式，并自动带入 `traceId / deviceCode / productKey / topic / errorCode / exceptionClass` 等公共筛选条件。
- 聚合统计：
  - `GET /api/device/message-trace/stats`：查看最近 1h / 24h 追踪量、失败分发量，以及消息类型 / 产品 / 设备 / Topic 聚合。
  - `GET /api/device/access-error/stats`：查看最近 1h / 24h 失败量，以及 `failureStage / errorCode / exceptionClass / protocolCode / topic` 聚合。
- MQTT consumer 健康：`GET /actuator/health/mqttConsumer`，查看 `running`、`connected`、`clientId`、`brokerUrl`、`subscribeTopics`、`lastDisconnectAt` 以及最近一次连接/收包/成功分发/失败阶段摘要。
- MQTT 联调和 `$dp` 验证继续由 [14-MQTTX真实环境联调手册.md](./14-MQTTX真实环境联调手册.md) 提供操作步骤。
- 审计详情当前应继续保持敏感字段脱敏与超长内容截断，避免排障链路再次暴露 `password`、`token`、`secret` 一类原文。

开发态控制台类别说明：
- `mybatis.sql`：SQL 明细日志，包括 `Preparing`、`Parameters`、`Columns / Row / Total`。
- `org.mybatis.spring`：会话与连接噪音，包括 `JDBC Connection ... will not be managed by Spring`、`Closing non transactional SqlSession ...`。
- `diagnostic.sql`：慢 SQL 摘要，仅在 `IOT_OBSERVABILITY_DIAGNOSTIC_SQL_FILE_ENABLED=true` 时写入 `diagnostic-sql.log`。
- `diagnostic.access`：慢 HTTP / 慢 MQTT 摘要，仅在 `IOT_OBSERVABILITY_DIAGNOSTIC_ACCESS_FILE_ENABLED=true` 时写入 `diagnostic-access.log`。
- `message.flow`：固定 Pipeline 的摘要日志，默认在 `iot.observability.message-flow.enabled=true` 时输出 `INFO`。
- 这四类日志当前都不进入常规 info / error 文件日志。
- 若要精确关闭某个 Mapper 或某条语句的 SQL 明细，继续使用标准 `logging.level.mybatis.sql.<mapper-or-statement>=OFF|TRACE`；完整 logger 名必须带 `mybatis.sql.` 前缀。

### 2.3 通知中心

- 当前定位：全局壳层内的站内消息中心，不新增独立消费端菜单。
- 当前固定消息来源类型：
  - `manual`
  - `system_error`
  - `event_dispatch`
  - `work_order`
  - `governance`
- 当前用户消费接口：
  - `GET /api/system/in-app-message/my/page`
  - `GET /api/system/in-app-message/my/unread-count`
  - `POST /api/system/in-app-message/my/read/{id}`
  - `POST /api/system/in-app-message/my/read-all`

### 2.4 外部通知渠道桥接

- 继续复用 `/api/system/channel` 的通知渠道，不另建第二套通知编排。
- 当前自动场景名固定为：
  - `system_error`
  - `observability_alert`
  - `in_app_unread_bridge`
- 支持渠道类型：
  - `webhook`
  - `wechat`
  - `feishu`
  - `dingtalk`
- `observability_alert` 与 `system_error` 明确分场景治理：前者只承接规则化运维阈值告警，后者继续承接后台异常审计自动通知，避免两类消息混在同一场景下失真。
- 规则化运维告警评估器当前固定每 `60` 秒执行一次，继续复用已有数据源：
  - `sys_audit_log` 中最近窗口 `system_error` 次数
  - `MqttConsumerRuntimeState` 中的连接/断连运行态
  - `iot_device_access_error_log` 的 `failureStage` 聚合
  - `sys_in_app_message_bridge_attempt_log` 的站内信桥接失败聚合
- 当前固定 4 类规则：
  - `system-error-burst`：最近 `10` 分钟 `system_error >= 5`
  - `mqtt-disconnect-timeout`：MQTT consumer 持续断连 `>= 5` 分钟
  - `failure-stage-spike`：最近 `10` 分钟同一 `failureStage >= 10`
  - `in-app-bridge-failure-burst`：最近 `10` 分钟同一渠道站内信桥接失败 `>= 3`
- 冷却策略：按 `ruleType + dimensionKey` 生成唯一 key，使用 Redis TTL 冷却 `30` 分钟；冷却期内重复命中不重复发通知。
- 触发结果：只在真正触发时发送外部通知并打印结构化 `WARN` 摘要；本阶段不新增告警表、不做恢复通知、不做独立告警大盘。

## 4. 排障主流程

### 4.1 标准流程

1. 发现问题：页面报错、链路异常、MQTT 上报失败、站内错误消息或外部群通知异常。
2. 取得 `TraceId`：优先从响应头、业务日志、系统日志或消息日志详情中获取。
3. 查询系统日志：先看 `/system-log` 是否存在对应 `system_error`，确认异常类型、模块、请求目标和结果。
   - MQTT `$dp` 场景下，`request_params` 当前会补齐 `deviceCode`、`productKey`、`protocolCode`、`clientId`、`topicRouteType`，优先用这些字段做接入契约比对。
4. 查询消息追踪：再到 `/message-trace` 按 `TraceId / deviceCode / topic` 复盘接入链路。
   - 页面顶部统计摘要会同步展示当前筛选条件下的最近 1h / 24h 数量、失败分发量和高频桶摘要，可先用来判断是否属于局部放大或全局性波动。
   - 详情抽屉会继续按 `traceId` 拉取 `message-flow` 时间线；若时间线已过期，页面会明确提示“时间线已过期，仅保留消息日志”。
   - 若消息日志尚未落库或怀疑失败发生在设备校验前，可直接切换到同一路由下的“失败归档”模式，继续查看 failureStage、契约快照与原始 payload。
   - 若当前问题来自 `/reporting`，可直接先按 `sessionId` 查询 `/api/device/message-flow/session/{sessionId}`，HTTP 会立即返回 timeline，MQTT 需等待消费回流绑定 trace。
5. 若主链路在设备校验前失败：继续查询 `/api/device/access-error/page`，核对 `failureStage`、原始 payload、解码后的 `deviceCode / productKey / protocolCode` 与 `contractSnapshot.expectedProtocolCode / protocolSource`。
6. 对照运行配置：核对 [07-部署运行与配置说明.md](./07-部署运行与配置说明.md) 中的环境变量、依赖服务和脚本前置。
7. 核对通知闭环：若涉及 `system_error` 自动通知、高优未读桥接或规则化运维告警，再检查渠道 `config.scenes`、未读阈值、Redis 冷却键与桥接留痕。
8. 留存验收证据：将结论回写到 `logs/acceptance/*`、验收记录或当前缺陷单。

补充说明：
- 若当前排障需要查看 SQL 明细，可临时设置 `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SQL_ENABLED=true` 后重启应用。
- 若当前排障需要查看 MyBatis-Spring 会话行为，可临时设置 `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SESSION_ENABLED=true` 后重启应用。
- 若当前排障需要查看慢 SQL、慢 HTTP 或慢 MQTT 摘要，可临时设置 `IOT_OBSERVABILITY_DIAGNOSTIC_SQL_FILE_ENABLED=true` 或 `IOT_OBSERVABILITY_DIAGNOSTIC_ACCESS_FILE_ENABLED=true`，并按需调整对应慢日志阈值。
- 若只想屏蔽某一个 SQL logger，请直接在 `application-dev.yml` 或环境变量里设置 `logging.level.mybatis.sql.<mapper-or-statement>=OFF`，不要省略 `mybatis.sql.` 前缀。
- 排障结束后，开发环境建议恢复默认关闭，避免大批量 SQL 与 SqlSession 控制台噪音再次淹没有效日志。

### 4.2 最小检查单

- 是否拿到了同一条链路的 `TraceId`
- `/actuator/health/mqttConsumer` 是否显示 `connected=true`
- `sys_audit_log` 是否存在对应 `system_error`
- `iot_message_log` 是否存在对应上行或分发失败记录
- `iot_device_access_error_log` 是否存在对应失败归档
- `deviceCode / productKey / topic` 是否与产品建档一致
- `contractSnapshot.expectedProtocolCode / actualProtocolCode / protocolSource` 是否与设备/产品建档一致
- 相关环境变量和通知渠道 `scenes` 是否开启
- `observability_alert` 场景渠道是否已配置且 Redis 冷却键可写
- Redis 中 `message-flow` key 是否可写且未提前过期
- 是否属于真实环境 schema 缺失或历史库未升级问题

## 5. 通知治理规则

### 5.1 system_error 自动通知

- 开关：`IOT_OBSERVABILITY_SYSTEM_ERROR_NOTIFY_ENABLED=true`
- 渠道前提：`config.url` 非空，且 `scenes` 中包含 `system_error`
- 当前目标：把后台异常同步到既有外部群渠道和站内消息中心

### 5.2 高优未读桥接

- 开关：`IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_ENABLED=true`
- 阈值配置：
  - `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_SCAN_INTERVAL_SECONDS`
  - `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_HIGH_THRESHOLD_MINUTES`
  - `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_CRITICAL_THRESHOLD_MINUTES`
- 当前策略：
  - 只扫描 `high / critical`
  - 只对仍未读对象生效
  - 每条消息在每个渠道上“成功一次即闭环”
  - 失败按渠道冷却后重试

### 5.3 规则化运维告警

- 总开关：`IOT_OBSERVABILITY_ALERTING_ENABLED=true`
- 渠道前提：`config.url` 非空，且 `scenes` 中包含 `observability_alert`
- 调度与冷却：
  - 评估周期固定 `IOT_OBSERVABILITY_ALERTING_EVALUATE_INTERVAL_SECONDS=60`
  - 冷却固定 `IOT_OBSERVABILITY_ALERTING_COOLDOWN_MINUTES=30`
  - Redis key 口径固定为 `规则类型 + 维度键`
- 当前固定规则：
  - `system-error-burst`
  - `mqtt-disconnect-timeout`
  - `failure-stage-spike`
  - `in-app-bridge-failure-burst`
- 当前边界：
  - 继续复用已有异常统计、MQTT 运行态、接入失败归档和桥接失败统计，不新增数据库表
  - 触发时只发送告警通知和打印 `WARN` 摘要，不生成恢复通知
  - 不在本阶段新增独立告警看板或手工处置流
