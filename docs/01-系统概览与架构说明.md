# 01 系统概览与架构说明

> 文档定位：系统级权威概览，说明定位、架构风格、模块边界和运行基线。

## 1. 系统定位

`spring-boot-iot` 是一个基于 Spring Boot 4 + Java 17 的模块化单体 IoT 网关平台，面向“设备接入 + 风险预警 + 事件处置 + 系统治理”的一体化场景。

当前交付基线：
- Phase 1~3 主链路稳定可用（HTTP/MQTT 上报、协议解析、消息日志、属性更新、在线状态、最小下行）。
- Phase 4 核心能力可用（告警中心、事件处置、风险点、阈值规则、联动规则、应急预案、报表分析、系统治理）。
- 风险监测与 GIS（`/risk-monitoring`、`/risk-monitoring-gis`）已落地代码，但仍需真实环境验收后再计入“已交付范围”。

## 3. 逻辑分层与模块边界

- `common`：常量、异常、工具、统一响应模型。
- `framework`：安全、配置、MyBatis、Redis、全局处理器、观测能力。
- `auth`：登录与用户鉴权上下文。
- `system`：组织、用户、角色、菜单、区域、字典、通知渠道、审计日志。
- `device`：产品、设备、属性、消息日志查询、设备调试查询。
- `protocol`：协议适配、Topic 解析、消息解码、安全校验。
- `message`：HTTP/MQTT 入口与分发，下行发布入口。
- `telemetry`：TDengine telemetry v2 raw 主写、legacy 兼容镜像、latest 查询与历史遥测存储抽象。
- `alarm`：告警、事件、风险点、阈值规则、联动规则、应急预案、风险监测。
- `report`：统计报表聚合。
- `rule`：规则引擎占位与扩展基座。
- `admin`：应用装配与启动。

硬约束：
- Controller 仅处理请求/响应。
- Service 负责编排。
- Mapper 负责持久化。
- 不把持久化逻辑放进协议适配层。

## 4. 关键运行链路

### 4.1 HTTP 上报链路

`/api/message/http/report` -> `UpMessageProcessingPipeline` 固定阶段 -> `device` 各 stage handler -> 返回 `R<MessageFlowSubmitResultVO>`。

当前固定阶段顺序：
- `INGRESS`
- `TOPIC_ROUTE`（HTTP 直接标记 `SKIPPED/DIRECT_HTTP`）
- `PROTOCOL_DECODE`
- `DEVICE_CONTRACT`
- `MESSAGE_LOG`
- `PAYLOAD_APPLY`
- `TELEMETRY_PERSIST`
- `DEVICE_STATE`
- `RISK_DISPATCH`
- `COMPLETE`

每次 HTTP 模拟都会立即生成 `sessionId / traceId`，并把完整 `MessageFlowTimeline` 写入 Redis，供 `/reporting` 立即展示处理时间线。
`TELEMETRY_PERSIST` 当前只落标准化后的 `properties`；`reply`、文件载荷和空属性消息会直接跳过。dev / prod 基线下若启用 `iot.telemetry.primary-storage=tdengine-v2`，该阶段会先批量写入 telemetry v2 raw stable，再异步分发 latest 投影与 legacy mirror；TDengine 写失败时该步骤记为 `FAILED` 但不回滚主链路。

### 4.2 MQTT 上报链路

MQTT 模拟发布 -> 生成待匹配 `sessionId` 与 fingerprint -> Broker -> `MqttMessageConsumer` -> `UpMessageProcessingPipeline` 固定阶段 -> `device` 各 stage handler -> 绑定真实 `traceId`。

补充口径：
- 标准 `/sys/...` Topic 与历史 `$dp` 都统一进入同一条 Pipeline。
- MQTT 发布接口只返回 `sessionId`；消费链路在完成 `PROTOCOL_DECODE` 后按 `topic + resolvedDeviceCode + rawPayloadHash` 命中 fingerprint，再把 `sessionId` 绑定到真实 `traceId`。
- `/reporting` 只轮询 `sessionId`；绑定成功后才展示完整 `trace` 时间线。
- MQTT / `$dp` 子设备拆包后，父设备与各 `childMessages` 会分别进入相同固定阶段；标准化属性会随各自 `DeviceProcessingTarget` 进入 `TELEMETRY_PERSIST`。

### 4.3 告警到事件闭环

规则触发告警 -> 告警确认/抑制/关闭 -> 事件派发 -> 工单接收/开始/完成 -> 事件关闭 -> 报表统计。

### 4.4 系统治理链路

用户登录 -> JWT -> 动态菜单/按钮权限 -> 系统管理 CRUD -> 审计日志自动采集。

## 5. 安全与观测基线

### 5.1 认证与鉴权

- 默认策略：除白名单外全部接口要求 JWT Bearer。
- 公共白名单：
  - `/api/auth/login`
  - `/api/message/http/report`
  - `/api/cockpit/**`
  - `/actuator/**`
  - `/doc.html`
  - `/swagger-ui/**`
  - `/swagger-ui.html`
  - `/v3/api-docs/**`

### 5.2 Trace 与审计

- `TraceContextFilter` 作用于 `/api/*`，统一注入 `X-Trace-Id`。
- `AuditLogFilter` 自动采集 `/api/**`（排除 `/api/system/audit-log/**`、`/api/auth/login`）。
- 业务失败识别规则：HTTP 异常或响应 `code != 200` 计为失败。
- 后台异步异常（如 MQTT 连接/订阅/分发失败）以 `operation_type=system_error` 写入 `sys_audit_log`。
- 设备接入链路当前额外生成 `MessageFlowTimeline`，默认 Redis 保留 `24` 小时；控制台继续保留 1 条 `message_flow_summary` 摘要日志，但阶段排障以时间线接口和页面为主。

## 6. 部署拓扑（开发/验收）

【根据代码推断】默认真实环境依赖：
- MySQL（业务主库）
- Redis（缓存、部分聚合状态、`message-flow` 时间线短期留存、telemetry v2 投影任务队列）
- TDengine（telemetry v2 raw 主时序 + legacy stable 兼容镜像 + `iot_device_telemetry_point` 兼容回退表）
- MQTT Broker（设备消息通道）
- 前端 Vite 开发服务（联调）

统一基线配置文件：
- `spring-boot-iot-admin/src/main/resources/application-dev.yml`

## 7. 已知边界与待确认项

- `iot_message_log` 当前为视图，物理表仍为 `iot_device_message_log`。
- 风险监测 GIS 不包含第三方地图 SDK、底图瓦片与热力图能力。
- dev / prod 当前默认仍以 `iot.telemetry.storage-type=tdengine` 为基线，但主写已切到 `iot.telemetry.primary-storage=tdengine-v2`；`/api/telemetry/latest` 默认优先读 v2 latest MySQL 投影，历史与 aggregate 路由仍保留 legacy / placeholder 口径。
- TDengine `slave_1` 当前同时承接 telemetry v2 raw stable（`iot_raw_measure_point` / `iot_raw_status_point` / `iot_raw_event_point`）与 `iot_device_telemetry_point` 兼容回退表；legacy stable 继续由环境侧治理，代码只做 `DESCRIBE` 校验和镜像写入。
- MySQL 当前新增 `iot_device_metric_latest` 作为 telemetry v2 latest 持久投影表，用于 `/api/telemetry/latest` 默认读路径与 Redis 失效后的回源。
- 多租户、组织范围与数据权限的权威口径已统一收口到 [13-数据权限与多租户模型.md](./13-数据权限与多租户模型.md)；当前真实环境仍以默认租户基线为主。
