## 1. 测试策略总览

当前项目采用“单元测试 + 协议回归 + 真实环境验收脚本”的组合策略。

- 单元测试：覆盖核心服务逻辑、协议解析、安全校验、审计与通知。
- 协议回归：重点覆盖 MQTT Topic 解析、加密解密、签名防重放、二进制帧解析。
- 真实环境验收：以 `application-dev.yml` 所指向环境为基线，验证端到端业务可用性。

- 业务验收台真实环境最小检查：
  - 登录后进入 `/business-acceptance`，确认业务角色可见业务验收入口。
  - 只选择 `环境 / 账号模板 / 模块范围` 三类轻配置，并成功发起“一键执行验收”。
  - 结果页首屏必须直接显示 `是否通过`、`哪些模块没过`、`总耗时`。
  - 展开一个未通过模块后，必须能看到 `失败步骤 / 接口 / 页面动作`。
  - 点击“进入结果与基线中心”后，应落到 `/automation-results?runId=...`，并自动预选同一次运行。
- 系统内容最小回归接口：
  - `GET /api/system/help-doc/access/list`
  - `GET /api/system/help-doc/access/page`
  - `GET /api/system/help-doc/access/{id}`
  - `GET /api/system/in-app-message/my/page`

## 9. 角色化自动化测试正式验收意见（2026-04-05）

### 9.1 验收目标与证据范围

本轮正式验收只回答一个问题：当前自动化测试能力，是否已经能分别从研发视角和业务人员视角，做到“以少量辅助支撑功能性、业务性测试”。

本轮结论基于以下证据：

1. 当前质量工场、业务验收台、研发工场、执行中心、结果与基线中心的前后端实现。
2. `docs/21-业务功能清单与验收标准.md` 中已固化的角色矩阵、菜单基线和质量工场定位。
3. `sql/init-data.sql` 中当前角色、菜单和授权种子。
4. `config/automation/business-acceptance-packages.json` 中当前预置验收包、账号模板和模块映射。
5. `2026-04-05` 当天重新执行的定向验证：
   - `npm --prefix spring-boot-iot-ui test -- --run src/__tests__/views/AutomationWorkbenchViews.test.ts src/__tests__/composables/useBusinessAcceptanceWorkbench.test.ts src/__tests__/composables/useAutomationRegistryWorkbench.test.ts`
   - `mvn --% -pl spring-boot-iot-report -DskipTests=false -Dmaven.test.skip=false -Dsurefire.failIfNoSpecifiedTests=false -Dtest=BusinessAcceptanceServiceImplTest,BusinessAcceptanceControllerTest,AutomationResultQueryServiceImplTest test`
   - `node scripts/docs/check-topology.mjs`

### 9.2 正式验收结论

正式验收意见如下：

1. 研发视角下，当前自动化测试能力已达到“可编排、可执行、可追证据、可复盘”的可用状态，可以作为功能回归、链路联调、缺陷定位和交付前复验的正式辅助能力。
2. 业务人员视角下，当前已达到“轻配置、一键执行、首屏看结论、失败可下钻”的可用状态，可以在预置验收包范围内，以少量辅助完成大部分功能性、业务性验收。
3. 当前能力可以判定为“已具备投用条件”，但不应表述为“全角色零门槛、零维护、全场景自助自动化”。新增场景接入、验收包维护、失败颗粒度优化和底层证据治理，仍需研发或测试继续维护。

综合判定：

- 研发自动化能力：`通过`
- 业务轻量验收能力：`通过（限预置验收包范围）`
- 全角色完全自治自动化：`暂不通过`

### 9.3 分角色验收判定

| 角色 | 当前判定 | 已具备能力 | 当前边界 |
|---|---|---|---|
| 开发 / 研发 | 通过 | 可进入 `/rd-workbench` 组织资产、在 `/automation-execution` 执行统一注册表、在 `/automation-results` 查看历史运行、失败场景和证据 | 仍需负责场景编排、断言维护、验收包映射和结果粒度优化 |
| 测试 | 基本通过 | 可复用研发侧执行与结果能力，完成回归、复验和失败复盘 | 当前未单独固化独立 `TEST_STAFF` 运行时角色，更多通过研发或超管权限复用 |
| 验收人员 / 业务人员 | 通过（轻量消费） | 可在 `/business-acceptance` 只配置 `环境 / 账号模板 / 模块范围` 发起验收，并直接查看 `是否通过 / 哪些模块没过 / 总耗时` | 不能独立编排新自动化场景，新增验收范围仍依赖研发/测试预置 |
| 产品 | 通过（轻量消费） | 可复用业务验收台完成预置包验收、查看失败模块和失败动作 | 当前“产品”主要体现在预置包目标标签和账号模板，不是独立运行时角色模型 |
| 项目经理 / 管理人员 | 通过（轻量消费） | 可复用管理账号模板运行交付包、查看模块级结论和失败下钻 | 首页聚焦仍偏风险运营，未把业务验收台作为默认首页入口 |
| 运维 | 不作为本轮主入口验收通过项 | 可继续使用接入智维、链路追踪、异常观测等研发/运维路径完成排障 | 当前不是 `/business-acceptance` 的主要授权对象，不应把该入口当作运维主工作台 |
| 超级管理员 | 通过 | 具备全量查看、复核和补位执行能力 | 不替代角色化产品化治理本身 |

### 9.4 可投用范围

当前准予投用的范围为：

1. 共享 `dev` 环境下的预置验收包复验。
2. 研发主链路回归、浏览器配置驱动回归、业务烟测脚本复验和结果证据复盘。
3. 业务、产品、项目经理在交付清单已固化的模块范围内做“一键复验 + 结论查看 + 失败下钻”。

当前不应直接宣称覆盖的范围为：

1. 全部业务场景均已自动化。
2. 业务人员可以独立新增、维护和治理自动化资产。
3. 结果中心已经具备 CI 归档、跨环境汇聚、数据库级持久化和团队级共享检索能力。

### 9.5 当前不通过项与后续补强方向

当前仍存在以下边界，故不应把本能力描述为“完全自治”：

1. 业务验收包的 `targetRoles` 当前主要是配置标签，不是按当前登录角色做严格后端裁剪。
2. 业务/管理角色虽然已具备业务验收台权限，但默认首页与工作区焦点仍未完全切到质量工场。
3. 少量模块仍共享同一个底层场景引用，模块级失败结论在极端情况下会偏粗，不是所有模块都已做到独立断言、独立失败归因。
4. 结果与基线中心当前仍主要依赖 `logs/acceptance` 台账，不是集中式 CI 证据中心。

结论口径：

- 若问题是“当前是否已经能给研发和业务一起用起来”，答案是：`能`。
- 若问题是“是否已经做到所有角色都几乎不需要专业人员支持”，答案是：`还没有`。

## 10. 开发人员自动化测试使用指南

### 10.1 适用对象

本指南面向开发人员、测试人员、联调负责人和需要协助业务验收的交付负责人。

### 10.2 使用前提

1. 所有真实环境验收统一以 `spring-boot-iot-admin/src/main/resources/application-dev.yml` 为基线，或以覆盖该文件的环境变量为基线。
2. 禁止回退到旧 H2 验收链路、旧浏览器自动化废弃路径或任何只为“跑通”而构造的替代环境。
3. 若环境不可达，应明确记录为环境阻塞，不得把环境问题写成业务通过。

### 10.3 先判断自己该走哪条入口

| 目标 | 推荐入口 | 使用场景 |
|---|---|---|
| 只想让业务或产品快速复验交付包 | `/business-acceptance` | 预置验收包、一键执行、直接看是否通过 |
| 需要梳理自动化资产、页面覆盖、模板和计划 | `/rd-workbench` | 研发维护自动化资产的主入口 |
| 需要看统一注册表、执行依赖和命令预览 | `/automation-execution` | 研发执行前核对目标环境、场景依赖和编排 |
| 需要看历史运行、失败明细和证据原文 | `/automation-results` | 回归复盘、失败定位、交付证据核查 |

### 10.4 开发人员标准使用流程

1. 先确认本次目标是“业务复验”还是“研发回归”。
2. 若只是按交付清单复验，优先让业务或交付走 `/business-acceptance`，研发只负责预置包和异常协助。
3. 若是新增功能、自测回归或缺陷修复，先走研发链路：
   - 在 `/rd-workbench` 明确当前页面、模板、计划或交付整理入口。
   - 在 `/automation-execution` 核对本次运行将复用哪些注册表场景。
   - 执行定向脚本或统一注册表。
   - 在 `/automation-results` 看本次运行详情、失败场景和证据。
4. 若业务验收台报出失败模块，再让研发从同一次 `runId` 跳转到底层结果中心继续定位。

### 10.8 如何判断结果

1. `通过`：模块断言全部通过，可继续把本次结果作为交付或回归证据。
2. `未通过`：存在明确失败模块，应继续查看失败步骤、接口和页面动作。
3. `阻塞`：更偏环境问题、登录问题、依赖不可达或未读取到有效运行结果，不应直接按功能缺陷处理。

建议判断顺序：

1. 先看业务验收台首屏结论。
2. 再看失败模块列表。
3. 再展开单模块明细，看 `失败步骤 / 接口 / 页面动作`。
4. 仍不清楚时，再通过 `runId` 到 `/automation-results` 查看底层运行明细、证据清单和证据原文。

### 10.9 什么时候需要研发继续介入维护

出现以下情况时，不应要求业务自己处理：

1. 需要新增验收包、模块映射、账号模板或场景引用。
2. 需要把一个“粗模块结论”拆成多个更细的自动化断言。
3. 需要修正统一注册表、浏览器计划、业务烟测脚本或结果解析逻辑。
4. 需要补齐新的证据类型、图片/二进制产物或集中式归档能力。

### 10.10 交付留痕建议

每次开发自测或交付复验，建议至少保留以下产物：

1. 运行命令与执行时间。
2. `registry-run-*.json / .md` 或业务验收台对应 `runId`。
3. 若失败，补一条“失败模块 -> 失败步骤/接口/页面动作 -> 责任归属”的文字结论。
4. 若本轮改动影响行为、路由、配置或验收口径，原位更新 `docs/05`、`docs/21`、`README.md` 或 `AGENTS.md` 中对应文档。
