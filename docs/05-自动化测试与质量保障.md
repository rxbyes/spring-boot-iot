## 1. 测试策略总览

当前项目采用“单元测试 + 协议回归 + 真实环境验收脚本”的组合策略。

- 单元测试：覆盖核心服务逻辑、协议解析、安全校验、审计与通知。
- 协议回归：重点覆盖 MQTT Topic 解析、加密解密、签名防重放、二进制帧解析。
- 真实环境验收：以 `application-dev.yml` 所指向环境为基线，验证端到端业务可用性。

## 6. 测试数据准备与清理 SOP

### 6.1 准备规则

1. 真实环境验收数据统一使用 `accept-`、`ACCEPT_` 一类前缀，避免与共享演示数据混淆。
2. 默认继续在 `tenant_id=1 / tenant_code=default` 基线下准备验收数据；如任务要验证租户范围，必须额外标记并同步参考 [13-数据权限与多租户模型.md](./13-数据权限与多租户模型.md)。
3. 执行前先区分：
   - 新库初始化：`sql/init.sql`、`sql/init-data.sql`
   - 历史库对齐：以最新 `sql/init.sql`、`sql/init-data.sql` 为准重建或人工比对
4. 任何需要先做 schema / 数据对齐才能通过的场景，都必须在验收记录中写明“前置对齐已完成”，不能直接口头视为通过。

### 6.2 清理规则

1. 测试结束后，优先清理或禁用本轮新增的 `accept-*` 产品、设备、账号和辅助样例。
2. 若共享环境不允许直接删除历史记录，至少要在验收产物中标明：
   - 本轮使用的编码前缀
   - 影响到的表和接口
   - 是否保留为后续复验样例
3. 对消息日志、告警、事件这类链路留痕，默认允许保留测试记录，但必须能通过统一前缀快速检索和区分。

  2. 再补单元测试 / 控制器测试 / 冒烟脚本
  3. 最终仍以真实环境执行结果判定是否通过
- 若后续引入真正的契约测试，至少应覆盖：
  - 鉴权白名单与受保护接口边界
  - `R` 返回体结构稳定性
  - `PageResult` 分页结构稳定性
  - 关键消费端接口字段不回退

## 8. 浏览器自动化与视觉回归准入等级

- 当前浏览器自动巡检脚本的执行方式和产物见 [真实环境测试与验收手册.md](./真实环境测试与验收手册.md) 第 `13` 节。
- `config/automation/acceptance-registry.json` 当前已作为统一自动化目录，首轮场景通过 `node scripts/auto/run-acceptance-registry.mjs` 把 `browserPlan / apiSmoke / messageFlow / riskDrill` 编排为同一份 `registry-run-*.json / .md` 汇总。
- 质量工场当前按 `/quality-workbench` 总览 + `/rd-workbench` 研发总览 + `/rd-automation-inventory`、`/rd-automation-templates`、`/rd-automation-plans`、`/rd-automation-handoff` 四个研发专项页 + `/automation-execution`、`/automation-results` 两个共享中心组织；`/automation-assets` 与 `/automation-test` 仅保留兼容入口并默认落到研发工场总览。
- 研发工场链路当前收口为：`页面盘点台` 负责页面清单、覆盖缺口与人工补录；`场景模板台` 负责页面冒烟、表单提交与列表详情模板；`计划编排台` 负责正式计划导入导出、步骤与断言；`交付打包台` 负责执行建议、基线说明与交付备注；`执行中心` 负责目标环境、命令预览与统一验收注册表；`结果与基线中心` 负责优先读取最近运行结果（`GET /api/report/automation-results/recent`、`GET /api/report/automation-results/{runId}`）、查看证据清单与原文预览（`GET /api/report/automation-results/{runId}/evidence`、`GET /api/report/automation-results/{runId}/evidence/content`）、兼容导入 `registry-run-*.json`、查看失败场景与处理质量建议。
- 若共享环境仍返回旧 `automation-assets / automation-test` 菜单授权，前端当前会自动把权限扩展映射到 `研发工场 + 四个研发专项页 + 执行中心/结果与基线中心`，并在壳层导航优先展示新结构，避免质量工场继续只剩“总览 + 旧兼容入口”。
- `结果与基线中心` 当前只对 `logs/acceptance` 下、且被当前运行结果显式引用的文本证据开放预览；手工粘贴导入模式继续只保留兼容解析，不伪造后台证据清单。
- 当前准入等级统一收口为：
  - `delivery` 场景：阻断级。正式执行失败时应返回非零退出码，并作为前端高风险改动的必查项。
  - `baseline` 场景：告警级。用于扩大覆盖和维护基线，不单独替代交付结论。
  - 视觉基线刷新：仅在页面设计或统一样式治理已确认变更时执行 `--update-baseline`，不能把真实视觉回归直接覆盖掉。
- `config/automation/sample-web-smoke-plan.json` 当前已收口为 `/login`、`/products`、`/devices` 的最小无人值守基线；默认 `readySelector` 已切到 `#login-submit` / `#quick-search`，不再依赖历史 `console-page-title`、旧查询框 id 或缺失场景模块。
- 配置驱动视觉断言支持把单步声明为 `optional: true`；此时缺失基线或视觉不一致会继续输出 visual manifest / diff 建议，但只记为 `optional_failed`，不直接阻断场景通过和 CLI 退出码。
- 浏览器自动化是高价值证据，但不替代真实环境手工验收结论。
- `【待确认 DOC-Q-004】` 当前 `delivery / baseline` 已形成推荐执行口径，但是否在所有日常提测流程中固定为统一阻断门禁，仍待团队流程确认。
