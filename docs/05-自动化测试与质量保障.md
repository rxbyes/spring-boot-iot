## 1. 测试策略总览

当前项目采用“单元测试 + 协议回归 + 真实环境验收脚本”的组合策略。

- 单元测试：覆盖核心服务逻辑、协议解析、安全校验、审计与通知。
- 协议回归：重点覆盖 MQTT Topic 解析、加密解密、签名防重放、二进制帧解析。
- 真实环境验收：以 `application-dev.yml` 所指向环境为基线，验证端到端业务可用性。

- 后端：

```powershell

```


- 前端：

```powershell

```

- 验收场景基准：`docs/真实环境测试与验收手册.md`。
- 系统内容最小回归接口：
  - `GET /api/system/help-doc/access/list`
  - `GET /api/system/help-doc/access/page`
  - `GET /api/system/help-doc/access/{id}`
  - `GET /api/system/in-app-message/my/page`
  - `GET /api/system/in-app-message/my/unread-count`

- 浏览器自动化失败问题的历史台账已归档到 `docs/archive/22-automation-test-issues-20260316.md`；长期有效结论应回写本文件或 `docs/08-变更记录与技术债清单.md`。

## 4. 已知问题与环境差异

1. `DeviceMessageServiceImplTest` 在部分 JDK 17 环境可能因 Mockito inline agent 自附加失败而报错。
2. `scripts/start-backend-acceptance.ps1`、`scripts/start-frontend-acceptance.ps1`、`scripts/run-business-function-smoke.ps1`、`scripts/run-business-function-browser.ps1` 已改为根据脚本位置自动推导仓库根目录，不再依赖固定 Windows 盘符路径；其中 `start-backend-acceptance.ps1` 会在 `.mvn/settings.xml` 缺失时自动回退到 plain `mvn`。
3. `【待确认 DOC-Q-005】` 当前仓库未见统一 CI 配置（如 GitHub Actions/Jenkinsfile），自动化执行口径主要依赖本地或共享环境。
4. 本地沙箱命令环境与提权环境对 `localhost` 监听端口的可见性可能不同；若普通命令无法访问本机验收实例，应优先在同一提权环境内完成 HTTP 验收，并把差异记录为环境问题。

## 5. 质量门禁建议（当前执行口径）

每次需求合并前建议至少完成：

1. `mvn clean package -DskipTests`（若仓库提供 `.mvn/settings.xml`，可按团队约定追加 `-s .mvn/settings.xml`）。
2. 受影响模块单元测试。
3. 关键链路手工验收（HTTP 上报、MQTT 上报、告警/事件闭环、权限登录）。
4. 文档同步更新（核心文档 `01`~`08` + README）。

## 6. 测试数据准备与清理 SOP

### 6.1 准备规则

1. 真实环境验收数据统一使用 `accept-`、`ACCEPT_` 一类前缀，避免与共享演示数据混淆。
2. 默认继续在 `tenant_id=1 / tenant_code=default` 基线下准备验收数据；如任务要验证租户范围，必须额外标记并同步参考 [13-数据权限与多租户模型.md](./13-数据权限与多租户模型.md)。
3. 执行前先区分：
   - 新库初始化：`sql/init.sql`、`sql/init-data.sql`
   - 历史库升级：`sql/upgrade/*.sql`
4. 任何需要补脚本才能通过的场景，都必须在验收记录中写明“前置脚本已执行”，不能直接口头视为通过。

### 6.2 清理规则

1. 测试结束后，优先清理或禁用本轮新增的 `accept-*` 产品、设备、账号和辅助样例。
2. 若共享环境不允许直接删除历史记录，至少要在验收产物中标明：
   - 本轮使用的编码前缀
   - 影响到的表和接口
   - 是否保留为后续复验样例
3. 对消息日志、告警、事件这类链路留痕，默认允许保留测试记录，但必须能通过统一前缀快速检索和区分。

## 7. 契约测试策略

  - Controller / DTO / VO 源码

  1. 接口变更先更新 `03`
  2. 再补单元测试 / 控制器测试 / 冒烟脚本
  3. 最终仍以真实环境执行结果判定是否通过

  - 鉴权白名单与受保护接口边界
  - `R` 返回体结构稳定性
  - `PageResult` 分页结构稳定性
  - 关键消费端接口字段不回退

## 8. 浏览器自动化与视觉回归准入等级

- 当前浏览器自动巡检脚本的执行方式和产物见 [真实环境测试与验收手册.md](./真实环境测试与验收手册.md) 第 `13` 节。
- 当前准入等级统一收口为：
  - `delivery` 场景：阻断级。正式执行失败时应返回非零退出码，并作为前端高风险改动的必查项。
  - `baseline` 场景：告警级。用于扩大覆盖和维护基线，不单独替代交付结论。
  - 视觉基线刷新：仅在页面设计或统一样式治理已确认变更时执行 `--update-baseline`，不能把真实视觉回归直接覆盖掉。
- 浏览器自动化是高价值证据，但不替代真实环境手工验收结论。
- `【待确认 DOC-Q-004】` 当前 `delivery / baseline` 已形成推荐执行口径，但是否在所有日常提测流程中固定为统一阻断门禁，仍待团队流程确认。
