- 桥接仍依赖通知渠道 `config.scenes` 包含 `in_app_unread_bridge`；只开环境变量但渠道未配置该场景时，不会实际发出桥接通知。

### 3.7 基准站一包多子设备映射

当前深部位移基准站兼容链路新增配置：

```yaml
iot:
  device:
    sub-device-mappings:
      SK00FB0D1310195:
        L1_SW_1: 84330701
        L1_SW_2: 84330695
        L1_SW_3: 84330697
        L1_SW_4: 84330699
        L1_SW_5: 84330686
        L1_SW_6: 84330687
        L1_SW_7: 84330691
        L1_SW_8: 84330696
```

说明：
- 当前配置用于兼容历史 `$dp` 报文里“基准站一个包携带多个逻辑测点”的场景。
- 协议层会按 `基准站 deviceCode -> 逻辑测点编码 -> 子设备 deviceCode` 的映射关系拆分子消息，再按真实子设备编码入库。
- 当前放在 `application-dev.yml` 仅适合作为真实环境联调或临时兼容方案；长期建议迁移到设备绑定主数据或设备元数据表，不建议继续长期放在环境配置中。

### 3.8 深部位移自动闭环配置

当前真实环境基线已补齐：

```yaml
iot:
  alarm:
    auto-closure:
      enabled: true
      cooldown-minutes: 30
      yellow: 5
      orange: 10
      red: 20
```

说明：
- `enabled`：是否开启深部位移平台内自动闭环。
- `cooldown-minutes`：同一 `deviceCode + metricIdentifier + level` 的告警冷却窗口，默认 30 分钟。
- `yellow/orange/red`：毫米阈值，蓝色区间默认为 `< yellow`。
- 当前单位口径固定为毫米，服务端按绝对值判级，不做单位换算。
- 风险点负责人为空时，自动派工默认回退到用户 `1`。
- 自动闭环命中应急预案时，会优先匹配与深部位移场景、风险点或测点上下文相关的启用预案；若库里暂未配置场景化预案，事件留痕中的 `emergencyPlan` 会为空，不再误挂通用预案。
- 已升级实例之间会额外使用 Redis 锁 `iot:alarm:auto-closure:{deviceCode}:{metricIdentifier}:{alarmLevel}` 抑制并发重复建单；若共享环境仍有未升级旧实例在线，该锁无法约束旧逻辑实例，需要统一升级后再以最终留痕判定验收结果。

### 3.9 会话表缺失降级口径

- 若共享库暂未执行 `sql/upgrade/20260321_phase5_device_online_session.sql`，当前服务端会自动降级跳过 `iot_device_online_session` 的读写与聚合，不再因为缺表阻断 `$dp`、MQTT 或 HTTP 上报主链路。
- 降级期间仍会继续更新 `iot_device.online_status`、`last_online_time`、`last_report_time`，因此设备在线状态、消息日志、属性落库与深部位移自动闭环仍可继续验收。
- 降级期间产品详情中的 `avgOnlineDuration / maxOnlineDuration` 会返回 `null`；如需恢复在线时长统计，仍需补执行 `20260321_phase5_device_online_session.sql` 并重启服务。

### 3.10 审计结果消息长度口径

- `sys_audit_log.result_message` 当前按数据库 `VARCHAR(500)` 写入。
- 服务端会在入库前自动截断超长 `resultMessage`，并追加 `...(truncated)`，避免系统异常审计自身因长度超限再次写入失败。
- 若需要查看完整异常原文，请结合 `response_result`、`request_params`、应用日志或外部日志平台交叉排查。

## 4. 启动与构建命令

### 4.1 推荐预安装

```bash
mvn -s .mvn/settings.xml clean install -DskipTests
```

- 直接在 `spring-boot-iot-admin` 上执行 `spring-boot:run` 之前，建议先在仓库根目录执行上述命令，确保 `common`、`framework`、`device`、`alarm` 等依赖模块中的最新 Java 类已完成编译并安装到本地 Maven 仓库。
- 如果本次开发改动涉及多个模块，优先重复执行一次 `clean install -DskipTests`，不要只编译启动模块。

### 4.2 启动后端

macOS / Linux、Windows CMD：

```bash
mvn  -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev
```

Windows PowerShell：

```powershell
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run "-Dspring-boot.run.profiles=dev"
```

Windows PowerShell 也可使用 stop-parsing 写法：

```powershell
mvn --% -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev
```

### 4.3 启动前端

```bash
cd spring-boot-iot-ui
npm install
npm run acceptance:dev
```

- 当前 Vite 开发代理仅放行 `/api/**`；前端侧约定所有浏览器发起的后端接口访问都必须走 `/api(?:/|$)`。
- 开发态前端 history 路由如 `/devices`、`/device-access`、`/message-trace`、`/products` 不再允许通过 `/device/**`、`/message/**` 旧代理入口误命中后端；若后续新增页面名称接近后端接口前缀，仍需同步检查 `vite.config.ts`。

### 4.4 前端 History 路由刷新

- 当前后端已放行 `/`、`/login` 及多层前端 history 路由的 SPA 壳层访问，并允许同源静态资源 `/assets/**`、`/index.html`、`/favicon.ico` 直出。
- 同源部署时，浏览器直接刷新 `/products`、`/risk-disposal`、`/user` 以及更深层前端路由，应先返回 `index.html`；前端随后优先恢复本地 token 和 `authContext`，若后续 `/api/auth/me` 或业务接口返回 `401`，再统一跳转 `/login`。
- 浏览器直接访问 `/message/**`、`/device/**` 这类不符合 `/api(?:/|$)` 约定的 GET 路径时，也应先回到 SPA，再由前端统一跳转 `/login`，不再直接暴露后端错误页。
- 如果 token 失效、无效或目标路由未授权，前端会统一回到 `/login`，不再停留在后端返回的原始 `401` JSON 页面。

### 4.5 链路验证中心 HTTP / MQTT 模拟说明

- 路由：`/reporting`（链路验证中心）。
- 当前页面交互已收口为“设备编码查询 -> 接入契约反显 -> Payload 编辑与预演 -> 发送 -> 诊断复盘”：
  - `deviceCode` 是唯一主查询入口。
  - 查询成功后只读反显 `productKey`、`protocolCode`、`clientId=deviceCode`。
  - `tenantId` 继续展示且默认 `1`，但只有 HTTP 模拟会把该值提交到后端。
  - `topic` 默认 `$dp`，允许在 HTTP / MQTT 两种传输方式下手工修改。
- 当前支持两种传输方式：
  - HTTP：调用 `POST /api/message/http/report`，继续走现有 HTTP 上报主链路。
  - MQTT：调用 `POST /api/message/mqtt/report/publish`，把原始字节真实发布到 Broker，再由现有 MQTT consumer 回流进入主链路。
- 当前支持两种上报模式：
  - 明文：只有可识别 JSON 才自动识别 C.1 / C.2 / C.3，计算 Byte2~Byte3 大端长度（第 4 字节开始），并构造标准二进制帧。
  - 密文：直接透传加密封包 JSON（`header + bodies.body`），不计算类型和长度。
- payload 编辑区支持：
  - JSON：可一键格式化；合法 JSON 对象会进入帧识别与“实际发送内容”预演。
  - XML：可一键格式化，但只按原始文本发送，不参与 C.1 / C.2 / C.3 判定。
  - 普通文本：保持原样发送，不做协议帧识别。
- 明文帧发送时会附带 `payloadEncoding=ISO-8859-1`，后端按单字节映射恢复原始帧字节；历史调用不传该字段时继续按 UTF-8 处理。
- 当传输方式为 MQTT、`topic=$dp` 且 payload 为普通属性 JSON 且未包含设备身份字段时，前端会在“实际发送内容”中自动补入 `deviceCode`，以保证默认兼容主题可用。

## 5. 数据初始化与升级

### 5.1 新库初始化

1. 执行 `sql/init.sql`
2. 执行 `sql/init-data.sql`

### 5.2 历史库升级

按场景执行 `sql/upgrade/*.sql`。常见必备脚本：
- `20260316_phase4_real_env_schema_alignment.sql`
- `20260316_phase4_task3_risk_monitoring_schema_sync.sql`
- 若历史库尚未补齐右上角 `通知中心 / 帮助中心` 的后端数据模型，再执行：
  - `20260321_phase5_in_app_message_help_docs.sql`
  - 该脚本会补齐 `sys_in_app_message`、`sys_in_app_message_read`、`sys_help_document` 三张表，并写入系统事件 / 业务事件 / 错误事件消息样例，以及首批 `13` 篇帮助文档样例（`5` 篇业务类、`4` 篇技术类、`4` 篇 FAQ）
  - 帮助文档样例当前继续按 `visible_role_codes + related_paths + currentPath` 的既有消费模型组织，不新增富文本、附件或版本流能力
  - 若当前访问 `/api/system/help-doc/**` 或 `/api/system/in-app-message/**` 已收到“系统内容依赖表缺失，请先执行 `sql/upgrade/20260321_phase5_in_app_message_help_docs.sql`，若为旧库升级请继续执行 `sql/upgrade/20260322_phase5_notification_center_followup.sql`”业务错误，说明真实库仍缺少这批表或列，应优先执行对应脚本，不再把问题归因为通用后端异常
- 若历史库已建好系统内容表，但仍缺通知中心去重键、来源索引或旧来源口径回填，再执行：
  - `20260322_phase5_notification_center_followup.sql`
  - 该脚本会补齐 `sys_in_app_message.dedup_key`、检索与去重索引，并把历史样例来源值统一回填到 `manual / governance` 等现行口径
- 若历史库本轮需要开启“高优未读阈值后桥接通知渠道”，再继续执行：
  - `20260322_phase5_notification_channel_bridge.sql`
  - 该脚本会新增 `sys_in_app_message_bridge_log`，用于记录桥接成功/失败、最近尝试时间、响应状态和尝试次数
  - 该脚本不会自动改写既有渠道 `config.scenes`；是否启用桥接渠道仍由运维自行决定
- 若历史库还未补齐平台治理下的 `站内消息 / 帮助文档` 菜单和按钮权限，再执行：
  - `20260321_phase5_system_content_menu_governance.sql`
  - 该脚本会补齐 `/in-app-message`、`/help-doc` 页面菜单，以及新增/编辑/删除按钮权限，并为管理人员、超级管理员补齐真实菜单授权
- 产品详情在线时长统计如需生效，还需执行：
  - `20260321_phase5_device_online_session.sql`
- 深部位移基准站初始化子设备与风险点绑定时，可补执行：
  - `20260320_phase4_deep_displacement_sub_devices_bootstrap.sql`
  - 该脚本会基于现有基准站自动补齐 8 个子设备、`dispsX` / `dispsY` 产品物模型，以及 1 个初始风险点和绑定关系
  - 脚本已按“可重复执行”治理：会先清理同一风险点下 `device_code + metric_identifier` 的重复绑定，再仅插入缺失绑定，避免重复跑脚本后出现重复 `risk_point_device` 记录

## 6. 运维与排障入口

- Swagger：`/doc.html`、`/swagger-ui/index.html`
- 运行监控：`/actuator/**`
- 业务审计：`/api/system/audit-log/**`
- 接入追踪：`/api/device/message-trace/page`

建议排障顺序：
1. 先看 `X-Trace-Id`。
2. 再看 `sys_audit_log` 的 `system_error`。
3. 再看 `iot_device_message_log`（或视图 `iot_message_log`）。
4. 若是深部位移自动闭环，再看 `iot_alarm_record.remark`、`iot_event_record.review_notes` 是否写入 `traceId / metricIdentifier / linkage / emergencyPlan` 留痕。
5. 若产品详情在线时长为空，先确认共享库是否已存在 `iot_device_online_session`；若不存在，这是当前允许的兼容降级结果，不代表消息主链路失败。

## 7. 脚本现状与风险

- `scripts/start-backend-acceptance.ps1`、`scripts/start-frontend-acceptance.ps1`、`scripts/run-business-function-smoke.ps1`、`scripts/run-business-function-browser.ps1` 已改为基于脚本目录自动定位仓库根目录。
- Windows 10 共享环境常见工作区示例路径为 `E:\idea\ghatg\spring-boot-iot`；macOS/Linux 以当前仓库实际路径为准。
- 当前推荐做法是“保留 Windows 示例路径说明，但脚本执行仍按脚本位置自动推导仓库根目录”，避免把某一个绝对路径重新写死为唯一入口。
- 本地手工启动命令现已补充 macOS/Linux、Windows CMD 和 Windows PowerShell 三种写法；验收脚本仍以 PowerShell 入口为主。
- 前端相关脚本仍要求本机具备 `node`、`npm`；后端脚本仍要求本机具备 `java`、`mvn`。

## 8. 安全与合规提醒

- `application-dev.yml` 含默认连接账号与口令示例，建议在共享环境统一改为环境变量注入。
- `【待确认】` 仓库是否已纳入密钥扫描与配置脱敏流程。
