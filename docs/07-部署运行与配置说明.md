补充口径：
- 当前 `spring.datasource.dynamic.datasource.slave_1` 专用于 TDengine。
- 当前 `spring-boot-iot-framework` 的 `MasterDataSourceConfig` 已显式把 `spring.datasource.dynamic.datasource.master.hikari.*` 绑定到主库 `HikariDataSource`；共享环境若已更新 YAML 但运行时 `hikaricp.connections.max` 仍停留在 `10`，优先检查进程是否完成重启，而不是继续误判为单个 Mapper 故障。
- 当前 `application-dev.yml`、`application-prod.yml`、`application-test.yml` 已显式写入 MySQL 主库 Hikari 基线：默认 `maximum-pool-size=30`、`minimum-idle=5`、`connection-timeout=30000ms`、`validation-timeout=5000ms`、`idle-timeout=180000ms`、`max-lifetime=240000ms`、`keepalive-time=120000ms`、`leak-detection-threshold=20000ms`；dev 的 TDengine `slave_1` 默认 `maximum-pool-size=8`、`minimum-idle=1`。
- 2026-03-30 的真实环境回放显示主库连接在空闲约 `286994ms` 后触发 `Communications link failure`，因此当前基线故意把 `keepalive-time` 前移到 `120000ms`、把 `max-lifetime` 收口到 `240000ms`，优先避免共享环境里借出接近 5 分钟未活动的旧连接。
- 2026-03-31 起，`application-dev.yml`、`application-prod.yml`、`application-test.yml` 的默认 `IOT_MYSQL_URL` 已统一补齐 `allowPublicKeyRetrieval=true`。当目标 MySQL 8 账号使用 `caching_sha2_password` 且连接未启用 SSL 时，若缺少该参数，应用会在新建连接阶段直接报 `Public Key Retrieval is not allowed`，表现为 `DeviceMapper`、`UserMapper` 或 `$dp` topic 首跳查询失败。
- 若共享环境通过环境变量覆盖 `IOT_MYSQL_URL`，覆盖值也必须同步带上 `allowPublicKeyRetrieval=true`，否则即使 YAML 默认值已修复，运行态仍会继续出现同类握手失败；如现场具备正式 SSL 证书链，也可改走启用 SSL 的更严格方案，但不得回退到 H2 或其它废弃验收链路。
- 2026-04-02 起，`application-dev.yml`、`application-prod.yml`、`application-test.yml` 统一显式开启 `server.servlet.encoding.charset=UTF-8`、`enabled=true`、`force=true`。若共享环境通过外部配置覆盖 servlet 编码，必须保持 UTF-8，否则账号中心、用户治理等中文字段仍可能在请求或响应链路中表现为 `????` 或乱码。
- `application-dev.yml`、`application-prod.yml` 默认 `iot.telemetry.storage-type=tdengine`、`iot.telemetry.primary-storage=tdengine-v2`、`iot.telemetry.legacy-mirror.enabled=true`、`iot.telemetry.read-routing.latest-source=v2`、`iot.telemetry.read-routing.history-source=v2`；`application-test.yml` 默认 `mysql`，但仍保留同一组 telemetry v2 foundation 配置键。
- `2026-04-09` 起，对象洞察与其它历史趋势默认优先走 telemetry v2 raw/read 路由；若 legacy stable 主读链路、`iot_device_telemetry_point` 兼容表探测或查询失败，`POST /api/telemetry/history/batch` 会在 `legacy-read-fallback-enabled=true` 时直接保留仍可读取的 raw v2 / 主链路结果，并继续做缺失补齐，不再把整次查询打成通用 `500`。
- `2026-04-09` 起，telemetry v2 raw stable 已收口为普通 `metric_id` 列，不再依赖当前共享 TDengine 版本并不接受的 `COMPOSITE KEY` 语法；`TelemetryRawBatchWriter` 会按 `ingested_at` 基准为同一次 raw 批写分配唯一单调 `ts` 行键，并继续把设备真实上报时间保存在 `reported_at`，避免 `$dp` 同报文多测点在同一个 child table 中互相覆盖。对象洞察与其它历史趋势在 telemetry v2 raw 路径下也会优先按 `ingested_at/ts` 聚桶，缺失时再回退 `reported_at`。
- `2026-04-13` 起，published resolver snapshot 不新增独立配置开关：设备域默认按 `productId + 最新 releaseBatchId` 做进程内缓存，正式发布产生新批次后，对象洞察、history/latest 与 runtime 解析会自动切换到新快照，无需额外手工刷新或重建缓存。
- 本轮修复只保证后续新写入 raw 点位不再按同样方式互相覆盖；对 2026-04-09 之前已经被覆盖的历史点位，平台不会自动补回，需等待设备重新上报或按专项数据修复方案处理。
- `2026-04-01` 起，`iot.telemetry.aggregate.enabled`、`iot.telemetry.aggregate.hourly-enabled`、`iot.telemetry.aggregate.daily-enabled` 与 `iot.telemetry.cold-archive.enabled` 已作为长期容量治理配置固化到模型；当前只有在 `aggregate.enabled=true` 且 `hourly-enabled=true` 时，`TelemetryWriteFanoutService` 才会把 `MEASURE` 数值点位异步写入 `iot_agg_measure_hour`，默认值仍全部为 `false`。
- `spring-boot-iot-telemetry` 在应用启动且 `iot.telemetry.storage-type=tdengine` 时，会在 `slave_1` 上自动执行 telemetry v2 raw stable 与 `iot_device_telemetry_point` 的 `CREATE IF NOT EXISTS`；该阶段只负责 schema 初始化，不自动触发历史数据回放。legacy stable 继续由环境侧治理，代码不补 stable DDL。
- `sql/init-tdengine.sql` 当前除 `iot_device_telemetry_point` 与三张 raw stable 外，还负责初始化 `iot_agg_measure_hour`。该小时聚合 stable 必须通过脚本手动执行；应用运行时只会自动派生 `tb_ah_<tenantId>_<deviceId>` child table，不会自动建 stable。
- `iot_device_metric_latest` 已进入 `sql/init.sql`；新库初始化完成后，本轮 `/api/telemetry/latest` 可直接走 v2 读路径，历史库未对齐时才会退回 legacy fallback 或暴露缺失。
- 历史 TDengine 数据如需补迁，统一通过业务接口 `POST /api/telemetry/migrate-history` 手动触发；默认优先迁移 `iot_device_telemetry_point` 标准化兼容表，必要时再回放 legacy stable/subtable 映射，不把全量迁移塞进应用启动链路。
- 2026-03-24 起，`application-dev.yml` 额外引入 `$dp` 重构双轨开关：`IOT_PROTOCOL_LEGACY_DP_FAMILY_OBSERVABILITY_ENABLED`、`IOT_PROTOCOL_LEGACY_DP_NORMALIZER_V2_ENABLED`、`IOT_TELEMETRY_LEGACY_MAPPING_VALIDATE_ONLY`。
- 2026-04-09 起，`application-dev.yml` / `application-prod.yml` / `application-test.yml` 额外暴露 `iot.protocol.family-definitions` 与 `iot.protocol.decrypt-profiles`；其中 `decrypt-profiles` 只负责统一协议对象引用，不重复存储底层密钥。
- 2026-04-16 起，上述 YAML 只作为协议安全 fallback 真相：运行时会优先从 `iot_protocol_family_definition_snapshot / iot_protocol_decrypt_profile_snapshot` 中读取 `PUBLISHED` 快照重建 family 与 decrypt profile，只有正式快照缺失时才回退 YAML。
- 2026-04-16 起，页面层已有两处直接消费上述 DB 正式真相：`/protocol-governance` 会直接查看协议族定义 / 解密档案列表并提交发布或回滚审批，`/products -> 契约字段 -> 映射规则台账` 会直接查看映射规则草稿/正式版本并执行试命中/审批入口；因此共享环境若只改 YAML 而不补齐正式快照，页面仍会继续把数据库 `PUBLISHED` 状态视为正式真相，YAML 仅在运行时缺快照时兜底。
- 2026-04-18 起，`/device-onboarding` 无代码接入台不新增独立环境变量；共享环境只需对齐最新 `sql/init.sql` 中的 `iot_device_onboarding_case` 与 `iot_onboarding_template_pack`，以及 `sql/init-data.sql` 中的 `/device-onboarding` 菜单/角色授权即可。若手输路由能访问但分组首页或左侧导航缺入口，优先检查初始化数据是否已补齐；若页面能打开但模板包或批量能力报结构缺失，再回查最新结构是否已同步。
- 当前 `application-dev.yml` 的安全默认值为：`family-observability-enabled=true`、`normalizer-v2-enabled=false`、`legacy-mapping-validate-only=true`。
- 当 `IOT_TELEMETRY_LEGACY_MAPPING_VALIDATE_ONLY=true` 且未放量 `IOT_PROTOCOL_LEGACY_DP_NORMALIZER_V2_ENABLED` 时，协议层会并行执行 legacy `$dp` v1/v2 标准化比对并输出差异日志，但仍沿用 v1 结果，不改变固定 Pipeline 与外部接口。
- 共享环境若再次出现 `HikariPool-1 - Connection is not available, request timed out after 30000ms`，不要只盯住栈顶 `UserMapper` / `DeviceMapper`。这类错误通常表示主库连接池已被慢 SQL、定时任务和页面并发共同耗尽，应先检查 `/actuator/metrics/hikaricp.connections.active`、`/actuator/metrics/hikaricp.connections.pending`、`/actuator/metrics/hikaricp.connections.timeout`，再结合 `diagnostic-sql.log` 与泄漏检测日志定位占用方。

补充口径：
- Redis 当前除缓存与告警冷却外，还承接 `message-flow` 时间线短期留存、MQTT 无效上报治理分钟桶 / 冷却键，以及 telemetry v2 投影任务队列。
- 固定 key 口径：
  - `iot:message-flow:trace:{traceId}`
  - `iot:message-flow:session:{sessionId}`
  - `iot:message-flow:fingerprint:{fingerprint}`
  - `iot:message-flow:recent:sessions`
  - `iot:invalid-report:bucket:{yyyyMMddHHmm}:failure-stage:{failureStage}`
  - `iot:invalid-report:reason:{yyyyMMddHHmm}:{reasonCodeOrScopedKey}`
  - `iot:invalid-report:cooldown:{governanceKey}`
  - `iot:device:offline-timeout:leader`
  - `iot:telemetry:v2:projection:queue`
- 若 Redis 不可用，本轮 `message-flow` 时间线与 `/reporting`、`/message-trace` 处理阶段复盘视为环境阻塞，不回退到仅控制台日志模式。
- `IOT_DEVICE_OFFLINE_TIMEOUT_CLUSTER_SINGLETON_ENABLED=true` 时，设备离线超时调度会通过独立 Redis 锁竞争唯一 leader；只有拿到 `iot:device:offline-timeout:leader` 的节点会执行离线收口。
- 共享环境若仍混跑旧版本节点，或显式关闭 `IOT_DEVICE_OFFLINE_TIMEOUT_CLUSTER_SINGLETON_ENABLED`，设备列表的在线/离线状态仍可能被旧节点反复写回错误结果；修复该类问题时应先统一停旧实例，再观察一个完整上报周期。

- `IOT_MQTT_CLUSTER_LOCK_RENEW_INTERVAL_SECONDS`
- `IOT_MQTT_CLUSTER_LOCK_ACQUIRE_INTERVAL_SECONDS`

说明：
- 若未显式覆盖 `IOT_MQTT_CLIENT_ID`，默认值 `spring-boot-iot-mqtt-consumer` 会在运行时自动追加 `主机/PID` 后缀，降低共享 Broker 上多个 `dev` 实例互踢连接的风险。
- 若现场要求固定 `clientId`，请直接设置自定义且非默认的 `IOT_MQTT_CLIENT_ID`。
- 若 `application-dev.yml` 或其他 YAML 配置里显式声明 `default-subscribe-topics`，历史 `$dp` 主题应写成 `"$dp"`；未加引号时，运行时可能丢失该订阅，导致 legacy 回放超时。
- legacy `$dp` 双轨治理只在 `MqttJsonProtocolAdapter` 内部切换 v1/v2 标准化与子消息拆分实现，不改变 `MqttMessageConsumer -> UpMessageProcessingPipeline` 的固定 10 阶段顺序。
- `2026-04-09` 起新增 `iot.protocol.family-definitions.{familyCode}` 与 `iot.protocol.decrypt-profiles.{profileCode}` 两层配置。前者只声明 `protocol-code / decrypt-profile-code / normalization-strategy`，后者只声明 `algorithm / merchant-source / merchant-key / transformation / signature-secret`，不直接承载密钥明文。
- 当前 resolver 规则为：先按 DB `PUBLISHED` 快照重建运行时 family/decrypt profile，缺失时再回退 YAML；有 family hint 时优先按 `family-definitions -> decrypt-profile-code` 解析，没有 family hint 时按 `appId == merchant-key` 回退匹配 `decrypt-profiles`。legacy `$dp` 加密外壳在解密前天然没有 family hint，因此会先走 `appId` 回退。
- `IOT_MQTT_CLUSTER_SINGLETON_ENABLED=true` 时，MQTT consumer 会通过 Redis 租约竞争唯一 leader；只有 leader 节点真正连接 Broker 并订阅主题。
- standby 节点的 `/actuator/health` 仍返回 `UP`，但 `mqttConsumer.details.leadershipMode=STANDBY`、`consumerActive=false`、`connected=false`；这属于预期行为，不应再被误判为 MQTT 故障。
- standby 节点仍允许调用 `/api/message/mqtt/report/publish`；当本地没有活跃 consumer 连接时，会退化为一次性临时 publisher 客户端发包，再由 leader 节点统一消费回流。
- 共享环境若仍混跑旧版本节点或显式关闭 `IOT_MQTT_CLUSTER_SINGLETON_ENABLED`，同一 MQTT 上行仍可能被多个实例重复消费，导致同一 `sessionId` 绑定多个 `traceId`、MySQL 消息日志和 TDengine 时序重复写入。
- `2026-03-27` 起 MQTT consumer 额外启用首批无效上报治理：对 `DEVICE_NOT_FOUND` 与 `EMPTY_DECRYPTED_PAYLOAD` 先按治理键采样，再进入冷却抑制；其中原始空 payload 会在 consumer 入口直接丢弃，不再进入固定 Pipeline。

  - `iot.message.flow.session.total`，tags=`transportMode,status`
  - `iot.message.flow.stage.duration`，tags=`stage,transportMode,status`
  - `iot.message.flow.correlation.total`，tags=`result`
  - `iot.message.flow.lookup.total`，tags=`target,result`

- `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SQL_ENABLED=true` 时，会恢复控制台中的 SQL 明细日志，包括 `Preparing`、`Parameters`、`Columns / Row / Total`。
- `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SESSION_ENABLED=true` 时，会恢复控制台中的 MyBatis-Spring 会话噪音，包括 `JDBC Connection ... will not be managed by Spring`、`Closing non transactional SqlSession ...`。
- `IOT_OBSERVABILITY_DIAGNOSTIC_SQL_FILE_ENABLED=true` 时，会额外输出 `logs/diagnostic-sql.log`，用于承接慢 SQL 摘要，不污染常规 `info / error` 文件。
- `IOT_OBSERVABILITY_DIAGNOSTIC_ACCESS_FILE_ENABLED=true` 时，会额外输出 `logs/diagnostic-access.log`，用于承接慢 HTTP / 慢 MQTT 摘要。
- `IOT_OBSERVABILITY_PERFORMANCE_SLOW_SQL_THRESHOLD_MS`、`IOT_OBSERVABILITY_PERFORMANCE_SLOW_HTTP_THRESHOLD_MS`、`IOT_OBSERVABILITY_PERFORMANCE_SLOW_MQTT_THRESHOLD_MS` 默认为 `500 / 1000 / 1000`，超阈值时只打印摘要，不回放整包 payload。
- 上述两类控制台开关和两类诊断文件开关互不影响；`mybatis.sql`、`org.mybatis.spring` 仍只走控制台，`diagnostic.sql`、`diagnostic.access` 只走独立诊断文件。
- 若要精确屏蔽某个 Mapper 或某条语句的 SQL 明细，继续使用标准 logger level，例如：`logging.level.mybatis.sql.com.ghlzm.iot.system.mapper.InAppMessageMapper.selectList=OFF`。注意完整 logger 名必须带 `mybatis.sql.` 前缀。
- `IOT_OBSERVABILITY_IN_APP_UNREAD_BRIDGE_ENABLED=true` 后，后端会启用定时扫描任务，对 `high / critical` 且超过阈值仍未读的站内消息执行渠道桥接。
- 桥接仍依赖通知渠道 `config.scenes` 包含 `in_app_unread_bridge`；只开环境变量但渠道未配置该场景时，不会实际发出桥接通知。
- `IOT_OBSERVABILITY_ALERTING_ENABLED=true` 后，后端会每 `60` 秒评估一次规则化运维告警，并通过 `config.scenes` 匹配 `observability_alert` 场景的渠道发送通知。
- 若通知渠道 `config` 额外包含 `opsAlertTypes`，则只会接收声明过的治理运维告警类型；缺省或空数组代表继续接收 `observability_alert` 场景下全部告警。
- 规则化运维告警当前固定 8 类规则：`system-error-burst(10 分钟 >= 5)`、`mqtt-disconnect-timeout(持续断连 >= 5 分钟)`、`failure-stage-spike(10 分钟同一 failureStage >= 10)`、`in-app-bridge-failure-burst(10 分钟同一渠道桥接失败 >= 3)`、`risk-governance-missing-policy-burst(当前已绑定风险点但未启用阈值策略 >= 3)`、`risk-governance-field-drift-burst(字段漂移积压 >= 1)`、`risk-governance-contract-diff-burst(合同差异积压 >= 1)`、`risk-governance-missing-risk-metric-burst(风险指标缺失积压 >= 1)`。
- `risk-governance-missing-policy-burst` 当前直接复用风险治理缺口聚合：按 `riskMetricId` 优先、`metricIdentifier` 兼容回退统计当前“已绑定风险点但没有启用阈值策略”的积压量，不依赖新增表，也不依赖 `releaseBatchId` 回放链路。
- 治理运维三类规则阈值当前通过 `iot.observability.alerting.risk-governance-ops-alerts.*` 配置：`field-drift-threshold / contract-diff-threshold / missing-risk-metric-threshold`，对应环境变量为 `IOT_OBSERVABILITY_ALERTING_RISK_GOVERNANCE_OPS_ALERTS_*`。
- 渠道配置示例：

```json
{
  "url": "https://example.com/webhook/ops-field-drift",
  "scenes": ["observability_alert"],
  "opsAlertTypes": ["FIELD_DRIFT"],
  "timeoutMs": 3000,
  "minIntervalSeconds": 300
}
```
- `GET /api/device/access-error/stats` 的无筛选 `recentHourCount`、`recent24HourCount` 与 `topFailureStages`，以及 `failure-stage-spike` 告警规则，当前统一读取 Redis `invalid-report` 分钟桶聚合；`iot_device_access_error_log` 只继续承接样本详情、`topErrorCodes`、`topExceptionClasses`、`topProtocolCodes`、`topTopics` 等明细维度统计。
- 告警冷却当前固定依赖 Redis TTL，键口径为 `规则类型 + 维度键`，默认 `30` 分钟；若 Redis 不可用，本轮不会降级成新数据库表或恢复通知。
- 规则化运维告警触发时只补结构化 `WARN` 摘要日志，不额外放大 Spring Boot 启动日志、常规业务日志或 MyBatis SQL 控制台输出。
- `GET /actuator/health/mqttConsumer` 当前会额外返回 `lastDisconnectAt`，便于排查 `mqtt-disconnect-timeout` 是否具备触发条件。

### 3.8 深部位移自动闭环配置

当前真实环境基线已补齐：

```yaml
iot:
  alarm:
    auto-closure:
      enabled: true
      cooldown-minutes: 30
      yellow: 5
      orange: 10
      red: 20
```

说明：
- `enabled`：是否开启深部位移平台内自动闭环。
- `cooldown-minutes`：同一 `deviceCode + metricIdentifier + level` 的告警冷却窗口，默认 30 分钟。
- `yellow/orange/red`：毫米阈值，蓝色区间默认为 `< yellow`。
- 当前单位口径固定为毫米，服务端按绝对值判级，不做单位换算。
- 风险点负责人为空时，自动派工默认回退到用户 `1`。
- 自动闭环命中应急预案时，会优先匹配与深部位移场景、风险点或测点上下文相关的启用预案；若库里暂未配置场景化预案，事件留痕中的 `emergencyPlan` 会为空，不再误挂通用预案。
- 已升级实例之间会额外使用 Redis 锁 `iot:alarm:auto-closure:{deviceCode}:{metricIdentifier}:{alarmLevel}` 抑制并发重复建单；若共享环境仍有未升级旧实例在线，该锁无法约束旧逻辑实例，需要统一升级后再以最终留痕判定验收结果。

### 3.9 会话表缺失降级口径

- 若共享库尚未对齐当前 `sql/init.sql` 中的 `iot_device_online_session`，当前服务端会自动降级跳过 `iot_device_online_session` 的读写与聚合，不再因为缺表阻断 `$dp`、MQTT 或 HTTP 上报主链路。
- 降级期间仍会继续更新 `iot_device.online_status`、`last_online_time`、`last_report_time`，因此设备在线状态、消息日志、属性落库与深部位移自动闭环仍可继续验收。
- 降级期间产品详情中的 `avgOnlineDuration / maxOnlineDuration` 会返回 `null`；如需恢复在线时长统计，仍需按最新 `sql/init.sql` 对齐 `iot_device_online_session` 并重启服务。

### 3.10 审计结果消息长度口径

- `sys_audit_log.result_message` 当前按数据库 `VARCHAR(500)` 写入。
- 服务端会在入库前自动截断超长 `resultMessage`，并追加 `...(truncated)`，避免系统异常审计自身因长度超限再次写入失败。
- 若需要查看完整异常原文，请结合 `response_result`、`request_params`、应用日志或外部日志平台交叉排查。

### 3.11 设备接入契约巡检与修复脚本

当前仓库已补齐只读巡检脚本：

```bash
python scripts/audit-device-contracts.py --device-prefix SK11
```

常见用途：
- 扫描 `product_id=0`
- 扫描 `protocol_code` 为空
- 扫描设备协议与产品协议不一致
- 输出可直接用于排查的设备/产品对照快照

导出修复清单示例：

```bash
python scripts/audit-device-contracts.py \
  --device-prefix SK11 \
  --export-repair-csv logs/repair-checklists/sk11-device-contracts.csv
```

说明：
- 默认只做审计，不改库。
- 修复模式当前只接受显式清单：`python scripts/audit-device-contracts.py --repair-file logs/repair-checklists/sk11-device-contracts.csv --apply`。
- 修复清单固定列为 `device_code, product_key, protocol_code, reviewer, remark`；建议先人工确认产品归属，再分批执行。
- 脚本默认读取 `application-dev.yml` 中的 MySQL 连接基线，也允许通过 `IOT_MYSQL_*` 或命令行参数覆盖。

标准回放脚本示例：

```bash
python scripts/replay-mqtt-trace.py --trace-id 97daeef362184d349330e48b4401d3ea
```

若当前 dev 实例通过本地 SOCKS 代理访问 MQTT Broker，回放脚本也要显式走同一路径，例如：

```bash
python scripts/replay-mqtt-trace.py \
  --trace-id 97daeef362184d349330e48b4401d3ea \
  --app-base-url http://127.0.0.1:10099 \
  --mqtt-broker-url tcp://iot.ghatg.com:1883 \
  --socks-proxy-host 127.0.0.1 \
  --socks-proxy-port 7890
```

说明：
- 回放前会先检查 `/actuator/health/mqttConsumer` 是否 `connected=true`。
- 若 dev 进程带有 `socksProxyHost / socksProxyPort`，回放脚本也要同步加 `--socks-proxy-*`，否则容易出现“脚本 PUBACK 成功，但 consumer 实际收不到同一路径报文”的假阳性。
- 若已确认健康正常、代理路径也已对齐，但回放仍持续 `timeout` 且库中没有对应 payload 留痕，当前共享 Broker 很可能不会把平台侧手工 publish 的 `$dp` 报文重新投递给 consumer；此时应改用真实设备、专用模拟器或 Broker 侧白名单环境做最终回放验收。
- 回放后会轮询 `iot_message_log`（兼容视图；底层来源 `iot_device_message_log`）与 `iot_device_access_error_log`，输出“成功消息”或“失败归档 + contractSnapshot”。

## 4. 启动与构建命令

说明：

### 4.1 推荐预安装

```bash
mvn -s .mvn/settings.xml clean install -DskipTests
```

- 若本轮需要对跨模块改动做严格真实环境取证，优先改用 `mvn -pl spring-boot-iot-admin -am clean package -DskipTests` 后再执行 fat jar；这样更容易避免 `spring-boot:run` 仍引用旧依赖类。


Windows PowerShell：

```powershell
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run "-Dspring-boot.run.profiles=dev"
```

- `scripts/start-backend-acceptance.ps1` 当前会在构建成功后把 `spring-boot-iot-admin/target/*.jar` 复制到 `logs/backend-runtime/` 下再启动，避免运行中的 `target` 产物锁住后续 `clean package`。

### 4.4 前端 History 路由刷新

- 当前后端已放行 `/`、`/login` 及多层前端 history 路由的 SPA 壳层访问，并允许同源静态资源 `/assets/**`、`/index.html`、`/favicon.ico` 直出。
- 同源部署时，浏览器直接刷新 `/products`、`/risk-disposal`、`/user` 以及更深层前端路由，应先返回 `index.html`；前端随后优先恢复本地 token 和 `authContext`，若后续 `/api/auth/me` 或业务接口返回 `401`，再统一跳转 `/login`。
- 浏览器直接访问 `/message/**`、`/device/**` 这类不符合 `/api(?:/|$)` 约定的 GET 路径时，也应先回到 SPA，再由前端统一跳转 `/login`，不再直接暴露后端错误页。
- 如果 token 失效、无效或目标路由未授权，前端会统一回到 `/login`，不再停留在后端返回的原始 `401` JSON 页面。

### 4.5 链路验证中心 HTTP / MQTT 模拟说明

- 路由：`/reporting`（链路验证中心）。
- 当前页面交互已收口为“设备编码查询 -> 接入契约反显 -> Payload 编辑与预演 -> 发送 -> 诊断复盘”：
  - `deviceCode` 是唯一主查询入口。
  - 查询成功后只读反显 `productKey`、`protocolCode`、`clientId=deviceCode`。
  - `tenantId` 继续展示且默认 `1`，但只有 HTTP 模拟会把该值提交到后端。
  - `topic` 默认 `$dp`，允许在 HTTP / MQTT 两种传输方式下手工修改。
- 当前支持两种传输方式：
  - HTTP：调用 `POST /api/message/http/report`，继续走现有 HTTP 上报主链路。
  - MQTT：调用 `POST /api/message/mqtt/report/publish`，把原始字节真实发布到 Broker，再由现有 MQTT consumer 回流进入主链路。
- 当前发送成功后会在同页新增“处理时间线”面板：
  - HTTP：提交成功后立即按返回的 `sessionId` 拉取完整 timeline。
  - MQTT：先展示“等待消费回流”，再按 `sessionId` 轮询，直到 fingerprint 命中并绑定真实 `traceId`。
- 处理时间线当前必须包含 `TELEMETRY_PERSIST`；属性消息在 `tdengine` 模式下应看到 `persistedPointCount > 0`，跳过或失败时应看到对应 `branch / failedTargetCount`。
- `/message-trace` 详情抽屉当前会按 `traceId` 异步拉取同一条 timeline；若 Redis 中已过期，只保留消息日志详情与降级提示。
- 当前支持两种上报模式：
  - 明文：只有可识别 JSON 才自动识别 C.1 / C.2 / C.3，计算 Byte2~Byte3 大端长度（第 4 字节开始），并构造标准二进制帧。
  - 密文：直接透传加密封包 JSON（`header + bodies.body`），不计算类型和长度。
- payload 编辑区支持：
  - JSON：可一键格式化；合法 JSON 对象会进入帧识别与“实际发送内容”预演。
  - XML：可一键格式化，但只按原始文本发送，不参与 C.1 / C.2 / C.3 判定。
  - 普通文本：保持原样发送，不做协议帧识别。
- 明文帧发送时会附带 `payloadEncoding=ISO-8859-1`，后端按单字节映射恢复原始帧字节；历史调用不传该字段时继续按 UTF-8 处理。
- 当传输方式为 MQTT、`topic=$dp` 且 payload 为普通属性 JSON 且未包含设备身份字段时，前端会在“实际发送内容”中自动补入 `deviceCode`，以保证默认兼容主题可用。

## 7. 脚本现状与风险

- `scripts/start-backend-acceptance.ps1`、`scripts/start-frontend-acceptance.ps1`、`scripts/run-business-function-smoke.ps1`、`scripts/run-business-function-browser.ps1` 已改为基于脚本目录自动定位仓库根目录。
- Windows 10 共享环境常见工作区示例路径为 `E:\idea\ghatg\spring-boot-iot`；macOS/Linux 以当前仓库实际路径为准。
- 当前推荐做法是“保留 Windows 示例路径说明，但脚本执行仍按脚本位置自动推导仓库根目录”，避免把某一个绝对路径重新写死为唯一入口。
- 本地手工启动命令现已补充 macOS/Linux、Windows CMD 和 Windows PowerShell 三种写法；验收脚本仍以 PowerShell 入口为主。
- 前端相关脚本仍要求本机具备 `node`、`npm`；后端脚本仍要求本机具备 `java`、`mvn`。
- 若验收脚本以非默认端口起了新实例，而机上已有其他 `dev` 节点处于 MQTT leader，则新实例虽然 `/actuator/health` 仍为 `UP`，但 `mqttConsumer.details.leadershipMode` 可能是 `STANDBY`。此时 HTTP、登录和页面验收可继续打到新端口，MQTT 回流与 `message-flow` 复验则应切到 leader 节点。

## 8. 安全与合规提醒

- `application-dev.yml` 含默认连接账号与口令示例，真实共享环境应优先通过环境变量覆盖，不把示例值长期当成生产配置。
- 当前共享环境安全基线：
  - 连接账号、MQTT 口令、JWT 密钥、Webhook 地址优先走环境变量，不在文档、截图和验收产物中完整暴露。
  - 帮助中心正文、README 示例和验收截图中不写完整密钥、token、`Authorization` 头或真实群机器人地址。
  - 审计日志详情当前已对 `password / token / secret / authorization / accessToken / refreshToken / clientSecret` 等敏感字段自动脱敏；排障时应优先查看脱敏后的审计留痕，而不是复制原始密钥。
- 当前仓库未见强制性的 Secret Scan 流水线配置，因此现阶段最低要求是：
  - 提交前人工检查新增配置、脚本和文档中是否出现明文密钥
  - 共享环境示例仅保留“变量名 + 占位值”口径
  - 如后续接入密钥扫描工具，需同步回写本文件与 [08-变更记录与技术债清单.md](./08-变更记录与技术债清单.md)
