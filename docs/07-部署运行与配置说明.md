# 07 部署运行与配置说明

更新时间：2026-03-20

## 1. 运行基线

- 唯一启动模块：`spring-boot-iot-admin`。
- 唯一验收基线配置：`spring-boot-iot-admin/src/main/resources/application-dev.yml`。
- 不再使用：H2 验收 profile、H2 专用脚本、旧前端验收链路。

## 2. 依赖环境（dev 基线）

【根据代码推断】默认连接目标：

- MySQL：`jdbc:mysql://8.130.107.120:3306/rm_iot`
- TDengine：`jdbc:TAOS-RS://8.130.107.120:6041/iot`
- Redis：`8.130.107.120:6379`，DB `8`

### 3.7 基准站一包多子设备映射

当前深部位移基准站兼容链路新增配置：

```yaml
iot:
  device:
    sub-device-mappings:
      SK00FB0D1310195:
        L1_SW_1: 84330701
        L1_SW_2: 84330695
        L1_SW_3: 84330697
        L1_SW_4: 84330699
        L1_SW_5: 84330686
        L1_SW_6: 84330687
        L1_SW_7: 84330691
        L1_SW_8: 84330696
```

说明：
- 当前配置用于兼容历史 `$dp` 报文里“基准站一个包携带多个逻辑测点”的场景。
- 协议层会按 `基准站 deviceCode -> 逻辑测点编码 -> 子设备 deviceCode` 的映射关系拆分子消息，再按真实子设备编码入库。
- 当前放在 `application-dev.yml` 仅适合作为真实环境联调或临时兼容方案；长期建议迁移到设备绑定主数据或设备元数据表，不建议继续长期放在环境配置中。

## 4. 启动与构建命令

### 4.1 推荐预安装

```bash
mvn -s .mvn/settings.xml clean install -DskipTests
```

- 直接在 `spring-boot-iot-admin` 上执行 `spring-boot:run` 之前，建议先在仓库根目录执行上述命令，确保 `common`、`framework`、`device`、`alarm` 等依赖模块中的最新 Java 类已完成编译并安装到本地 Maven 仓库。
- 如果本次开发改动涉及多个模块，优先重复执行一次 `clean install -DskipTests`，不要只编译启动模块。

### 4.2 启动后端

macOS / Linux、Windows CMD：

```bash
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev
```

Windows PowerShell：

```powershell
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run "-Dspring-boot.run.profiles=dev"
```

Windows PowerShell 也可使用 stop-parsing 写法：

```powershell
mvn --% -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev
```

### 4.3 启动前端

```bash
cd spring-boot-iot-ui
npm install
npm run acceptance:dev
```

- 当前 Vite 开发代理仅放行 `/api/**`；前端侧约定所有浏览器发起的后端接口访问都必须走 `/api(?:/|$)`。
- 开发态前端 history 路由如 `/devices`、`/device-access`、`/message-trace`、`/products` 不再允许通过 `/device/**`、`/message/**` 旧代理入口误命中后端；若后续新增页面名称接近后端接口前缀，仍需同步检查 `vite.config.ts`。

### 4.4 前端 History 路由刷新

- 当前后端已放行 `/`、`/login` 及多层前端 history 路由的 SPA 壳层访问，并允许同源静态资源 `/assets/**`、`/index.html`、`/favicon.ico` 直出。
- 同源部署时，浏览器直接刷新 `/products`、`/risk-disposal`、`/user` 以及更深层前端路由，应先返回 `index.html`；前端随后优先恢复本地 token 和 `authContext`，若后续 `/api/auth/me` 或业务接口返回 `401`，再统一跳转 `/login`。
- 浏览器直接访问 `/message/**`、`/device/**` 这类不符合 `/api(?:/|$)` 约定的 GET 路径时，也应先回到 SPA，再由前端统一跳转 `/login`，不再直接暴露后端错误页。
- 如果 token 失效、无效或目标路由未授权，前端会统一回到 `/login`，不再停留在后端返回的原始 `401` JSON 页面。

## 5. 数据初始化与升级

### 5.1 新库初始化

1. 执行 `sql/init.sql`
2. 执行 `sql/init-data.sql`

### 5.2 历史库升级

按场景执行 `sql/upgrade/*.sql`。常见必备脚本：
- `20260316_phase4_real_env_schema_alignment.sql`
- `20260316_phase4_task3_risk_monitoring_schema_sync.sql`
- 深部位移基准站初始化子设备与风险点绑定时，可补执行：
  - `20260320_phase4_deep_displacement_sub_devices_bootstrap.sql`
  - 该脚本会基于现有基准站自动补齐 8 个子设备、`dispsX` / `dispsY` 产品物模型，以及 1 个初始风险点和绑定关系

## 6. 运维与排障入口

- Swagger：`/doc.html`、`/swagger-ui/index.html`
- 运行监控：`/actuator/**`
- 业务审计：`/api/system/audit-log/**`
- 接入追踪：`/api/device/message-trace/page`

建议排障顺序：
1. 先看 `X-Trace-Id`。
2. 再看 `sys_audit_log` 的 `system_error`。
3. 再看 `iot_device_message_log`（或视图 `iot_message_log`）。

## 7. 脚本现状与风险

- `scripts/start-backend-acceptance.ps1`、`scripts/start-frontend-acceptance.ps1`、`scripts/run-business-function-smoke.ps1`、`scripts/run-business-function-browser.ps1` 已改为基于脚本目录自动定位仓库根目录。
- Windows 10 共享环境常见工作区示例路径为 `E:\idea\ghatg\spring-boot-iot`；macOS/Linux 以当前仓库实际路径为准。
- 当前推荐做法是“保留 Windows 示例路径说明，但脚本执行仍按脚本位置自动推导仓库根目录”，避免把某一个绝对路径重新写死为唯一入口。
- 本地手工启动命令现已补充 macOS/Linux、Windows CMD 和 Windows PowerShell 三种写法；验收脚本仍以 PowerShell 入口为主。
- 前端相关脚本仍要求本机具备 `node`、`npm`；后端脚本仍要求本机具备 `java`、`mvn`。

## 8. 安全与合规提醒

- `application-dev.yml` 含默认连接账号与口令示例，建议在共享环境统一改为环境变量注入。
- `【待确认】` 仓库是否已纳入密钥扫描与配置脱敏流程。
