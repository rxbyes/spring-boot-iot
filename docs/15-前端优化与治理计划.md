# 15 前端优化与治理计划

> 文档定位：长期生效的前端治理规则、开放问题和下一步优化方向。
> 适用角色：前端研发、全栈研发、设计协作、代码审阅者、AI 前端助手。
> 权威级别：一级权威。
> 上游来源：`spring-boot-iot-ui` 当前实现、[06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md)、近期前端改造结论、归档快照。
> 下游消费：页面重构、风格收口、交互评审、技能文档、前端任务模板。
> 变更触发条件：共享组件规则变化、页面风格收口、列表/抽屉模式调整、前端稳定性问题暴露。
> 更新时间：2026-03-28

## 1. 当前长期有效规则

1. 新页面优先复用 `PanelCard`、`StandardPagination`、`useServerPagination`、`StandardTableToolbar`、`StandardTableTextColumn`、`StandardDetailDrawer`、`StandardFormDrawer`、`StandardDrawerFooter`、`confirmAction` 和共享列表样式。
2. 查询、新增、重置、删除、批量、刷新及其他高频动作，优先复用 `StandardButton` 表达统一按钮语义，不再在页面内散写第二套尺寸、配色和 hover 规则。
3. `StandardButton` 当前作为按钮权威入口，继续统一“主操作白字 / 次操作橙字 / 品牌链接橙字”的文字色和层级表达；共享组件与页面内高频动作不得再单独覆写按钮文案颜色。
4. 表格与卡片中的“详情 / 编辑 / 更多”类操作统一复用 `StandardActionLink`、`StandardActionMenu` 与 `StandardRowActions`；`StandardActionLink` 已移除 `tone`，`StandardActionMenu` item 已移除 `tone`，不得继续在页面里散写一组 `el-button link + el-dropdown` 模板或伪危险语义接口。
5. `StandardRowActions` 当前固定使用 `variant="table | card | editor"`：桌面表格操作列统一走 `table`，移动卡片操作区统一走 `card`，可换行编辑区统一走 `editor`；不得再依赖页面私有 `:deep(...)` 覆写去修按钮等宽、触控尺寸和下拉对齐。
6. 当列表操作列固定为 2~3 个轻操作时，间距必须通过 `StandardRowActions` 的共享 `gap` 档位表达；产品定义中心、设备资产中心、链路追踪台、异常观测台不得再在页面 CSS 中私自覆写操作列 gap。
7. 固定宽度下的 `消息类型`、`Topic`、编号、TraceId 等文本列，只要存在回落原始值或长度不可控的可能，就继续优先复用 `StandardTableTextColumn` 承接省略与 tooltip，不回退到裸 `el-table-column` 文本节点。
8. 按钮继续按品牌 / 辅助层级表达：主操作使用品牌橙渐变白字，次操作使用白底橙描边和橙字，`link / text` 中的主语义操作保持品牌橙；删除也跟随同一套橙色按钮语义，不再保留红色危险按钮分支。
9. 系统弹窗、抽屉底部和遗留全局按钮类必须跟随共享按钮层级收口；hover 以轻微加深或品牌浅底纹为主，不再额外叠加页面私有渐变和动效。
10. 系统正文与筛选标签统一优先使用蓝灰字体系，避免纯黑字在浅色控制台页面上显得发硬；操作区、操作列和主语义链接按钮继续复用品牌橙，`详情 / 编辑 / 更多 / 查看设备 / 删除 / 更换 / 洞察` 统一与查询按钮保持同一识别。
11. 传输方式、模式切换、分组选项等“点前 / 点后”状态按钮统一复用 `StandardChoiceGroup`；不得继续在页面里散写 `active` class 和局部 pill 按钮样式。
12. 页面配色继续统一到 `tokens.css` 与 `element-overrides.css` 的品牌 token；当前全局基线固定为“阿里云浅色控制台”方向，禁止为单页重新发明另一套蓝橙紫主色。
13. 工作台、抽屉、确认交互和概览卡必须保持“浅背景 + 白卡 + 轻描边 + 轻阴影”的控制台语法，不回退到高饱和营销式视觉。
14. 顶部壳层、分组导航、命令面板、通知中心、帮助中心继续收口到共享 shell 编排，不新增页面私有壳层能力。
15. 品牌层级继续固定为“品牌橙主操作 + 云蓝辅助信息 + 蓝灰正文与描边”；登录页、壳层、驾驶舱、loading 蒙层和工作台概览不得再各自维护另一套大面积渐变或纯黑字方案。
16. `/in-app-message` 的桥接效果运营继续保持“同页独立 workbench 分区 + 独立桥接详情抽屉”结构，复用共享分页、详情抽屉和表格列模式，不新增第二套桥接治理页面。
17. 新增页面和组件继续遵守 [06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md) 中的“路由页 / 壳层编排 / 共享标准件 / 业务包装层”分层与命名规则，不回退到无语义命名。
18. 当前前端最小工程门禁继续保持 `npm run build`、`npm run test`、`npm run component:guard`、`npm run list:guard`、`npm run style:guard`；其中 `component:guard` 独立负责共享动作/文本列契约守卫，`list:guard` 独立负责列表骨架守卫，不与当前 `style:guard` 的半径基线噪音绑定；若后续接入 ESLint / Stylelint，需要先更新 `06` 与本文件。
19. 视觉 token 必须优先复用 `var(--radius-*)`、`calc(var(--radius-*) + ...)` 与 `var(--shadow-*)`；不要继续在页面或组件里新增 `border-radius: 999px / 8px / 6px`、重复 `box-shadow: inset 0 1px 0 rgba(...)` 这类字面量，若现有 token 不够用，应先补 `tokens.css` 再落样式。
20. 产品定义中心、设备资产中心等高频台账的工具栏右侧动作较多时，必须优先通过共享工具栏的紧凑间距与自适应布局保持同组功能在桌面端单行展示；不要把同一组功能无规律挤到下一行。表格操作列如果包含 `更多` 这类下拉触发器，固定宽度要预留额外余量，避免右侧 fixed 列裁切触发器文案。
21. 当前共享动作/文本列纳管范围已从首批试点扩展到 `风险运营`、`风险策略`、`平台治理`、`质量工场` 代表页；后续新增批次页面时，必须先补 `component:guard` allowlist，再迁移页面接入，避免“先混改、后补规则”。
22. 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理等列表页，筛选区默认复用 `StandardListFilterHeader`；不要再回退到页面私有 `search-form + el-row/el-col + text-right` 模板。
23. 纳管列表页的已生效筛选状态统一复用 `useListAppliedFilters`；页面不再各自维护“标签拼装 + 单项移除 + 全部清空 + 高级筛选计数”的私有实现。组织架构、账号中心、角色权限、区域版图当前也已纳入同一口径。
24. `链路追踪台`、`异常观测台`、审计中心这类日志 / 诊断台账，若需要展示统计摘要，优先放入页面顶部紧凑状态条和 `StandardTableToolbar.meta-items`；只有确有必要时才补 `notices`，不要为单页再拆第二套概览卡或私有统计条。
25. 当新增诊断能力与现有已授权日志台账强耦合、且共享环境菜单路径可能未及时同步时，优先在同一路由内通过 `StandardChoiceGroup` 等共享切换组件承接双模式，不要先创建第二条静态路由再回头兜权限。
26. 产品定义中心内的产品物模型设计器继续复用同一抽屉，不新增一级路由；候选提炼必须保持“舞台头部层 + 真实证据概览 + 候选目录 + 候选主列表 + 确认写入 rail”结构，并把正式模型 CRUD 收口在同抽屉的“正式模型”模式。
27. 产品物模型候选提炼必须诚实表达证据边界：属性候选优先来自真实属性 / 消息证据，事件和服务在缺乏真实上报或命令证据时只展示空态提示，不得在 UI 中伪造候选占位数据。
28. 产品物模型设计器后续若继续优化视觉或交互，必须保持 `hero-stage / overview / candidate-workspace / candidate-rail / formal-overview / formal-stage / card-surface` 这一套结构语法，不得回退到页面私有 `summary / type-switcher / candidate-layout` 旧骨架。
29. 设备资产中心若继续承接“已登记资产 + 未登记上报名单”双名单能力，必须保持同一路由 `/devices`、同一套 `StandardWorkbenchPanel + StandardListFilterHeader + StandardTableToolbar + StandardDetailDrawer` 骨架；未登记行继续保持只读，不得为其再拆第二条列表路由或页面私有详情壳。
30. `接入智维总览` 只保留跨模块判断、最近使用、推荐处理顺序和全部能力；子页禁止再复制模块目录、能力墙或大段介绍文案。
31. `链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 的顶部只允许使用一句判断 + 一行状态/动作的紧凑条；如果状态已在顶部和工具条表达，就不要再叠加第二层概况卡。
32. `接入智维` 跨页诊断上下文统一复用 `src/utils/iotAccessDiagnostics.ts`；query 只保留 `deviceCode / traceId / productKey / topic` 这类短字段，来源页、传输方式、状态等辅助信息统一走带 TTL 的 `sessionStorage`，不要在页面里各自扩写新的 query 协议。
33. `产品定义中心`、`设备资产中心` 接收诊断跳转时，只允许通过 `StandardWorkbenchPanel` 的 `inline-state` 槽位展示一条紧凑来源提示；不要再加介绍卡、说明墙、重复统计卡或“从哪里来/接下来做什么”的长段文案。

## 2. 页面结构治理重点

### 2.1 列表页

- 默认结构保持为 `概览或标题区 -> 筛选区 -> 工具条 -> 列表 -> 分页`。
- 同一台账优先收口为一张主卡，避免“说明卡 + 列表卡 + 分页卡”层级过深。
- 已提交筛选条件必须有显式反馈，支持单项移除和一键清空。
- 平台治理和风险处置类列表继续沿用统一操作条、长文本省略和选择态计数，不回退到页面私有工具条。
- 组织架构、区域版图继续保持“默认懒加载树 / 搜索后扁平分页”双模式；模式提示统一放入 `StandardWorkbenchPanel` 的 `notices` 槽位，不再保留页面私有提示条位置。
- 账号中心、角色权限与其他普通列表页一致，统一采用 `StandardWorkbenchPanel + StandardListFilterHeader + StandardAppliedFiltersBar + StandardTableToolbar + StandardPagination` 节奏，不再保留旧列表壳。
- 站内消息管理、帮助文档管理、导航编排、数据字典、通知编排等治理类列表，如果筛选项超过 4 个，优先让 `StandardListFilterHeader` 自动承接首行收拢与展开，不再手写占位空列维持按钮对齐。
- `/message-trace`、`/audit-log` 与 `/system-log` 这类链路 / 日志台账页，筛选区也必须复用 `StandardListFilterHeader + StandardAppliedFiltersBar`；高频入口走快速搜索，其余条件收口到“更多筛选”，长文本列继续通过 `StandardTableTextColumn` 保持单行省略与悬浮补全。
- 链路 / 日志台账页若存在聚合统计，优先用顶部紧凑状态条表达页级判断、用 `toolbar meta` 表达高频数字，不再在主列表上方额外叠一层私有统计卡片。
- 若同一路由下承接“主链路台账 + 失败归档”双模式，模式切换入口统一放在 `StandardWorkbenchPanel` 的 `header-actions`，筛选骨架、详情抽屉和工具栏继续复用同一套共享标准件。
- `/devices` 当前在同一列表内承接“已登记资产 + 未登记上报名单”双口径时，区分入口应优先通过 `登记状态` 筛选和列表标签完成；未登记详情继续复用同一详情抽屉，只增加失败阶段、Topic、TraceId、原始载荷等排障字段，不再并行搭第二套失败列表页。
- `/products`、`/devices` 若从诊断页或失败归档承接治理动作，优先在台账顶部保留一条紧凑来源提示并让列表直接可用；即使当前列表尚无数据，也不要为了提示来源再额外堆一块说明卡或二级空态壳。
- 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理、审计中心 / 异常观测台、链路追踪台，当前统一纳入 `list:guard` 守卫范围；后续新增纳管页时，必须先补 allowlist，再迁移页面。

### 2.2 详情与抽屉

- 详情抽屉优先采用“摘要秒开 + 后台静默补数”的体验，不把正常刷新状态包装成红色错误提示。
- 详情抽屉内的加载、空态、错误态必须与 `StandardDetailDrawer` 统一，不为单页单独造一套提示组件。
- 表单抽屉继续复用 `StandardFormDrawer` 和 `StandardDrawerFooter`，避免重复底部按钮样式与关闭逻辑。
- 切换型按钮、pill 选项和链路验证中心中的“传输方式 / 上报模式”当前已统一抽到 `StandardChoiceGroup`；后续类似切换交互直接复用，不再在页面内书写 `--active` 私有样式。
- 桥接详情抽屉继续采用“摘要秒开 + 消息原文与尝试明细并行补数”的模式；尝试明细若为空或失败，只在抽屉内部表达，不影响上层治理列表继续使用。

### 2.3 链路验证中心

- `/reporting` 保持“左侧模拟上报、右侧诊断复盘”的双主卡结构。
- 发送结果下方的“处理时间线”必须复用 `StandardTraceTimeline`，按同一品牌和共享 token 展示阶段、状态、耗时、处理类/方法与摘要；不要再为单页散写私有 timeline 卡片或 rail。
- `deviceCode` 继续作为唯一主查询入口；`productKey`、`protocolCode`、`clientId` 保持只读派生。
- JSON / XML 格式化仅服务编辑体验；只有可识别 JSON 才参与协议帧判断。
- 页面级状态摘要与发送就绪度优先通过 `MetricCard` 收口，设备查询反馈优先通过 `StandardInlineState` 表达，不回退到页面私有提示条。
- MQTT 模拟的“等待消费回流 / 已绑定 trace / 时间线已过期”状态提示，继续放在共享工作台语义里表达，不新增蓝/橙/紫私有强调条。

### 2.3.1 链路 / 日志时间线

- `/message-trace` 详情抽屉中的“处理时间线”继续复用 `StandardTraceTimeline`，并与 `/reporting` 保持相同字段顺序、状态色和空态文案。
- 时间线过期后的降级提示必须明确说明“仅保留消息日志”，避免用户误判为页面加载失败或权限问题。

### 2.4 数据校验台

- `/file-debug` 优先收口到 `StandardWorkbenchPanel + StandardListFilterHeader + StandardInlineState` 的单入口工作台结构，不回退到页面私有散卡片表单。
- 顶部紧凑状态条、文件快照、固件聚合和原始响应继续复用共享语法，保持“状态条 -> 查询 -> 结果 -> 原始响应”的固定节奏；若顶部已经交代数量和刷新状态，就不要再补一块重复的概况卡。
- 壳层、登录页、驾驶舱和列表页结果区的高曝光渐变、弱纹理、hover 高亮与 loading 蒙层，必须优先来自全局 token 和共享样式入口；页面只允许做轻量语义化包装，不得继续散写另一套品牌 literal。

## 3. 稳定性与质量规则

1. 控制台 warning 需要区分“单次巡检采样噪音”和“可稳定复现问题”，只有复测成立的问题才进入活跃治理范围。
2. `v-permission` 不得直接挂在没有稳定原生根节点的组件上；组件型操作菜单优先在脚本层过滤 action 列表。
3. 移动端侧栏关闭前必须先移出焦点，并只在桌面折叠态使用 tooltip 包裹导航图标。
4. 浏览器自动化与视觉回归的失败证据继续保留在历史台账中，但长期有效规则要回写到本文件和 [06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md)。
5. 桥接响应摘要、目标摘要、帮助搜索高亮等消费态文本继续统一走安全文本切片渲染，不回退到 `v-html` 直出。
6. 共享动作列与文本列的防回退继续由 `npm run component:guard` 守卫；当前纳管页已覆盖产品/设备、链路追踪/异常观测、风险运营/风险策略、平台治理与质量工场代表页，后续新增批次页面仍需先定义 allowlist，再迁移，不要一次性全站混改。
7. 已经由共享请求层弹出友好提示的 `handled` 请求异常，不再额外输出页面级 `console.error`；浏览器控制台应优先保留真正未处理的前端错误，避免业务失败提示淹没真实缺陷。
8. `style:guard` 当前只允许在 baseline 内保留少量历史字面量余量；新增或回流的圆角 / 阴影字面量一律视为样式漂移回归，必须在提交前通过 token 化或共享样式收口恢复到基线以内。

## 4. 当前开放问题

### P1

- 持续扫描共享页面中组件节点上的权限指令、分页漂移和详情抽屉私有样式回流风险。
- 继续控制开发态 warning 和非业务型控制台噪音，避免掩盖真实缺陷。
- 持续补齐前端静态门禁，优先评估 ESLint / Stylelint 与现有 `style:guard` 的协同方案。

### P2
