# 15 前端优化与治理计划

> 文档定位：长期生效的前端治理规则、开放问题和下一步优化方向。
> 适用角色：前端研发、全栈研发、设计协作、代码审阅者、AI 前端助手。
> 权威级别：一级权威。
> 上游来源：`spring-boot-iot-ui` 当前实现、[06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md)、近期前端改造结论、归档快照。
> 下游消费：页面重构、风格收口、交互评审、技能文档、前端任务模板。
> 变更触发条件：共享组件规则变化、页面风格收口、列表/抽屉模式调整、前端稳定性问题暴露。
> 更新时间：2026-04-01

## 1. 当前长期有效规则

1. 新页面优先复用 `PanelCard`、`StandardPagination`、`useServerPagination`、`StandardTableToolbar`、`StandardTableTextColumn`、`StandardDetailDrawer`、`StandardFormDrawer`、`StandardDrawerFooter`、`confirmAction` 和共享列表样式。
2. 查询、新增、重置、删除、批量、刷新及其他高频动作，优先复用 `StandardButton` 表达统一按钮语义，不再在页面内散写第二套尺寸、配色和 hover 规则。
3. `StandardButton` 当前作为按钮权威入口，继续统一“主操作白字 / 次操作橙字 / 品牌链接橙字”的文字色和层级表达；共享组件与页面内高频动作不得再单独覆写按钮文案颜色。
4. 表格与卡片中的“详情 / 编辑 / 更多”类操作统一复用 `StandardActionLink`、`StandardActionMenu`、`StandardRowActions` 与 `StandardWorkbenchRowActions`；`StandardActionLink` 已移除 `tone`，`StandardActionMenu` item 已移除 `tone`，页级组合优先交给 `StandardWorkbenchRowActions`，不得继续在页面里散写一组 `el-button link + el-dropdown` 模板或伪危险语义接口。
5. `StandardRowActions` 当前固定使用 `variant="table | card | editor"`：桌面表格操作列统一走 `table`，移动卡片操作区统一走 `card`，可换行编辑区统一走 `editor`；`StandardWorkbenchRowActions` 继续在此基础上统一直出动作与“更多”菜单的组合，其中桌面 `table` 变体固定使用共享桌面间距，不再因 `2` 个动作还是 `3` 个动作切换成不同分布，移动端 `card` 变体继续保持自适应触控布局，不得再依赖页面私有 `:deep(...)` 覆写去修按钮等宽、触控尺寸和下拉对齐。
6. 当列表操作列固定为 1~2 个轻操作时，间距必须通过共享桌面基线表达；当操作列超过 2 个动作时，必须通过 `StandardWorkbenchRowActions` 收口为“前 2 项直出 + 更多”，但相邻动作间距仍沿用同一套共享标准。产品定义中心、设备资产中心、链路追踪台、异常观测台，以及风险运营下的 `实时监测台 / GIS态势图 / 告警运营台 / 事件协同台`，都不得再在页面 CSS 中私自覆写操作列 gap，也不得另起一套桌面 `distribution` 规则。
补充：纳管页的 `操作` 列必须声明 `class-name="standard-row-actions-column"`，由共享全局样式统一处理 fixed 列内的 `overflow / text-overflow / 字号 / 留白`。`StandardRowActions` 的桌面 `table` 变体必须占满整列，并统一采用左对齐分布，`standard-row-actions-column .cell` 也继续保持左贴齐起笔基线；不要再让默认 `.cell` ellipsis 把最后一个动作裁成尾部圆点，也不要让不同文案长度的页面重新长出私有对齐策略。
7. 固定宽度下的 `消息类型`、`Topic`、编号、TraceId 等文本列，只要存在回落原始值或长度不可控的可能，就继续优先复用 `StandardTableTextColumn` 承接省略与 tooltip，不回退到裸 `el-table-column` 文本节点。
8. 按钮继续按品牌 / 辅助层级表达：查询类主操作、对象洞察台“刷新洞察”、确认、新增、删除、批量等主操作统一使用品牌主橙渐变白字。次操作使用白底橙描边和橙字，`link / text` 中的主语义操作保持品牌橙；删除也跟随同一套橙色按钮语义，不再保留红色危险按钮分支。
9. 系统弹窗、抽屉底部和遗留全局按钮类必须跟随共享按钮层级收口；hover 以轻微加深或品牌浅底纹为主，不再额外叠加页面私有渐变和动效。
10. 系统正文与筛选标签统一优先使用蓝灰字体系，避免纯黑字在浅色控制台页面上显得发硬；操作区、操作列和主语义链接按钮继续复用品牌橙，`详情 / 编辑 / 更多 / 查看设备 / 删除 / 更换 / 洞察` 统一与查询按钮保持同一识别。
11. 传输方式、模式切换、分组选项等“点前 / 点后”状态按钮统一复用 `StandardChoiceGroup`；不得继续在页面里散写 `active` class 和局部 pill 按钮样式。
12. 页面配色继续统一到 `tokens.css` 与 `element-overrides.css` 的品牌 token；当前全局基线固定为“阿里云浅色控制台”方向，禁止为单页重新发明另一套蓝橙紫主色。
13. 工作台、抽屉、确认交互和概览卡必须保持“浅背景 + 白卡 + 轻描边 + 轻阴影”的控制台语法，不回退到高饱和营销式视觉。
14. 顶部壳层、分组导航、命令面板、通知中心、帮助中心继续收口到共享 shell 编排，不新增页面私有壳层能力。
15. 五个一级工作台下的功能页根容器，当前统一优先复用 `StandardPageShell`；页级节奏、breadcrumb 到首张主卡片的垂直留白、标题/副标题字距，不再由页面私有 `padding / background / border / shadow` 控制。
16. 品牌层级继续固定为“品牌橙主操作 + 云蓝辅助信息 + 蓝灰正文与描边”；登录页、壳层、驾驶舱、loading 蒙层和工作台概览不得再各自维护另一套大面积渐变或纯黑字方案。
17. `/in-app-message` 的桥接效果运营继续保持“同页独立 workbench 分区 + 独立桥接详情抽屉”结构，复用共享分页、详情抽屉和表格列模式，不新增第二套桥接治理页面。
18. 新增页面和组件继续遵守 [06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md) 中的“路由页 / 壳层编排 / 共享标准件 / 业务包装层”分层与命名规则，不回退到无语义命名。
19. 当前前端最小工程门禁继续保持 `npm run build`、`npm run test`、`npm run component:guard`、`npm run list:guard`、`npm run style:guard`；其中 `component:guard` 独立负责共享动作/文本列契约守卫，`list:guard` 独立负责列表骨架守卫，不与当前 `style:guard` 的半径基线噪音绑定；若后续接入 ESLint / Stylelint，需要先更新 `06` 与本文件。
20. 视觉 token 必须优先复用 `var(--radius-*)`、`calc(var(--radius-*) + ...)` 与 `var(--shadow-*)`；不要继续在页面或组件里新增 `border-radius: 999px / 8px / 6px`、重复 `box-shadow: inset 0 1px 0 rgba(...)` 这类字面量，若现有 token 不够用，应先补 `tokens.css` 再落样式。
21. 全站中文排版当前统一采用“宋感字体栈”基线：`--font-display` 与 `--font-body` 必须保持同一套宋感 serif 栈，标题、正文、表格、表单、按钮和抽屉跟随全局 token；只有 `--font-mono` 继续服务技术文本。不要在单页 scoped CSS 中把正文或标题重新切回无衬线字体。
22. 产品定义中心、设备资产中心等高频台账的工具栏右侧动作较多时，必须优先通过共享工具栏的紧凑间距与自适应布局保持同组功能在桌面端单行展示；不要把同一组功能无规律挤到下一行。表格操作列如果包含 `更多` 这类下拉触发器，固定宽度要预留额外余量，避免右侧 fixed 列裁切触发器文案。
23. 当前共享动作/文本列纳管范围已从首批试点扩展到 `风险运营`、`风险策略`、`平台治理`、`质量工场` 代表页；后续新增批次页面时，必须先补 `component:guard` allowlist，再迁移页面接入，避免“先混改、后补规则”。
24. 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理等列表页，筛选区默认复用 `StandardListFilterHeader`；不要再回退到页面私有 `search-form + el-row/el-col + text-right` 模板。
25. 纳管列表页的已生效筛选状态统一复用 `useListAppliedFilters`；页面不再各自维护“标签拼装 + 单项移除 + 全部清空 + 高级筛选计数”的私有实现。组织架构、账号中心、角色权限、区域版图当前也已纳入同一口径。
26. `链路追踪台`、`异常观测台`、审计中心这类日志 / 诊断台账，若需要展示统计摘要，优先放入 `StandardTableToolbar.meta-items`、分页邻近区域或详情区；只有确有必要时才补 `notices`，不要再回退到页面顶部紧凑状态条、辅战区或私有统计条。
27. 当新增诊断能力与现有已授权日志台账强耦合、且共享环境菜单路径可能未及时同步时，优先在同一路由内通过 `StandardChoiceGroup` 等共享切换组件承接双模式，不要先创建第二条静态路由再回头兜权限。
28. 产品定义中心内的产品物模型设计器继续复用同一抽屉，不新增一级路由；默认治理入口固定为“手动提炼”，并保持“舞台头部层 + 总览指标层 + 手动治理工作区 + 正式模型工作区”结构语法。
29. 产品物模型手动提炼只允许服务当前选中的产品，且单次只支持粘贴 `1` 个设备样本 JSON；UI 必须明确区分 `业务数据 / 状态数据 / 其他数据` 三种样本类型，不得在页面里偷偷放开多设备批量解析。
30. 产品物模型治理界面必须诚实表达证据边界：手动提炼出的数组、空对象和结构噪音只能作为 `ignoredFieldCount` 摘要展示，不得伪造成属性；`other` 样本提炼出的字段必须默认标记 `needsReview`，事件和服务继续在正式模型中人工维护。
31. 设备资产中心若继续承接“已登记资产 + 未登记上报名单”双名单能力，必须保持同一路由 `/devices`、同一套 `StandardWorkbenchPanel + StandardListFilterHeader + StandardTableToolbar + StandardDetailDrawer` 骨架；未登记行继续保持只读，不得为其再拆第二条列表路由或页面私有详情壳。
32. `接入智维总览` 只保留 `资产底座 / 诊断排障` 两组真业务入口、最近使用、推荐处理顺序和全部能力；子页禁止再复制模块目录、能力墙或大段介绍文案。
31. `链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 的顶部只允许使用轻量标题壳和必要动作；摘要优先下沉到共享工具条、`StandardInlineState` 或详情区，不要再叠加第二层概况卡。
32. `接入智维` 六个核心页的二轮视觉精修已统一采用“静稳控制台基线 + 少量现代 SaaS 留白”，继续保持当前主配色不变；后续页面现代化优先通过标题层级、筛选密度、工具条轻重、表头亮度和操作节奏实现，不得再以新增私有主题色换取所谓互联网感。
33. 单主列表页与真页签诊断页当前都必须复用同一套轻量视觉语法：更轻边框、更软阴影、更淡表头、短说明和稳定字距。页面若需要继续优化，应先扩共享组件或 token，不要在 scoped CSS 中各自再造第二套“高级感”样式。
34. `/products` 当前固定为“桌面表格主视图 + 移动卡片补充”语法：不再提供 `表格视图 / 卡片视图` 手动切换，也不允许回流桌面卡片网格。若浏览器验收认为页面不够精细，优先继续精修标题层级、筛选密度、工具条节奏和表头亮度，而不是再加第二主视图。
35. `接入智维` 跨页诊断上下文统一复用 `src/utils/iotAccessDiagnostics.ts`；query 只保留 `deviceCode / traceId / productKey / topic` 这类短字段，来源页、传输方式、状态等辅助信息统一走带 TTL 的 `sessionStorage`，不要在页面里各自扩写新的 query 协议。
36. `产品定义中心`、`设备资产中心` 接收诊断跳转时，只允许通过 `StandardWorkbenchPanel` 的 `inline-state` 槽位展示一条紧凑来源提示；不要再加介绍卡、说明墙、重复统计卡或“从哪里来/接下来做什么”的长段文案。
37. `接入智维` 单主列表页当前统一收口为“全局 `ShellBreadcrumb` + `StandardPageShell` + `StandardWorkbenchPanel`”两层半结构；`StandardPageShell` 只负责页级节奏，不得再输出第二条同路径页内导航。`产品定义中心 / 设备资产中心 / 异常台账 / 数据校验台` 这类页面不得再保留中间一级标题层、说明墙、第二条“接入智维 / 当前模块”导航条或三层导航壳；只有 `/device-access` 这类分组总览页才允许显式开启单行 breadcrumb。
38. 如果同一路由下需要两个真实业务视图，统一使用 `IotAccessTabWorkspace` 放在面包屑和主工作台之间；不得再把模式切换做成右上角命令菜单、header-actions 假页签或页面私有 pill。
39. `链路追踪台` 与 `数据校验台` 顶部工具条不得回流 `异常观测台 / 数据校验台 / 链路追踪台` 这类跨页功能按钮；跨页协同只保留行级动作、详情抽屉提示和来源上下文恢复，避免重新长出右上角功能菜单。
40. 单主列表页及其详情 / 表单 / 关联抽屉，不得再通过 `eyebrow` 重复表达当前模块名或英文工作区标签；如需补充上下文，统一改由 `title`、`subtitle`、`description` 或区块标题承担。`产品定义中心`、`设备资产中心`、`链路验证中心`、`链路追踪台`、`数据校验台`、`站内消息管理`，以及壳层 `通知中心 / 帮助中心 / 账号中心`、`风险监测详情`、`接入失败详情`、`批量导入设备`、`更换设备`、`产品物模型设计器`、`自动化计划导入 / 自定义页面`、`导出列设置`、`自动化工场`、`风险运营驾驶舱`、`未来能力实验室`、`对象洞察台` 内的属性趋势卡当前均按此规则收口。
41. 同一路由内通过 query 同步筛选、分页或 Tab 时，必须保持内容区滚动位置稳定；统一依赖 `src/router/scrollBehavior.ts`，不要再让 `router.replace` 把页面重新顶到首屏。
42. `实时监测台`、`GIS态势图`、`告警运营台`、`事件协同台` 当前也统一纳入“单一 `StandardWorkbenchPanel` 主工作台”语法；首屏摘要应收口到 `notices`、工具条统计或正文分区，不再回流独立 Hero 概况卡、第二张筛选卡和第三张列表卡的三段式堆叠。
43. `/device-access` 入口工作台与无权限空态不得再通过 `QUIET CONSOLE`、`ACCESS STATUS` 这类英文 eyebrow 营造层级；入口页只允许用真实业务页签、搜索和入口列表表达信息，不再补第二层状态语气卡。
44. `/message-trace` 下的 `接入失败归档台` 统计摘要应优先下沉到 `StandardInlineState` 与工具条元信息；不要为失败归档模式再叠加顶部双 `Alert`、说明墙或工具栏跨页跳转按钮。
45. `/insight` 的 `对象洞察台` 当前统一采用“单一 `StandardWorkbenchPanel` + `StandardListFilterHeader` + `StandardInlineState` + 正文内容分区”语法；相关页面联动动作应下沉到正文卡片或行内操作，不再保留英文 hero 眉标、首屏大按钮组或第二张概况 Hero 卡。风险等级、在线状态、监测对象档案、测点和趋势图必须优先来自 `/api/risk-monitoring/realtime/{bindingId}`，且页面固定只保留“4 个概览指标 + `基础档案` / `研判依据` / `属性趋势预览` / `关键监测指标` / `设备属性快照` / `风险分析草稿`”结构；建议动作卡组和独立消息日志卡视为回退。若设备未命中风险监测绑定，页面允许退化为设备上报分析视图，但不得把设备档案、属性快照和消息日志一并清空。
46. `风险对象中心`、`阈值策略`、`联动编排`、`应急预案库` 当前也统一纳入“单一 `StandardWorkbenchPanel` 主工作台”语法；新增主操作优先上收至 `header-actions`，摘要指标并入共享工具条元信息，不再保留独立 Hero 概况卡或双首屏结构。
47. 风险策略域抽屉不得再使用 `Risk Platform Form` 一类英文 eyebrow；抽屉层级优先通过中文标题、副标题和区块标题表达，不额外补一层英文装饰性眉标。
48. `接入智维`、`风险运营`、`风险策略`、`质量工场` 代表页当前已统一以 `StandardPageShell` 对齐平台治理的页级字距与导航到首卡片的高度；`/device-access`、`/reporting`、`/message-trace`、`/file-debug`、`/system-log`、`/insight`、`/report-analysis` 也已纳入同一页壳合同，页面根节点若再次回流私有 `padding / background / border / shadow`，视为样式治理回退，必须在同轮修复。
49. `自动化工场` 这类工程工作台，即使正文需要承载多块 `PanelCard`、编排器和结果面板，也必须把首屏标题与说明收口到同一 `StandardWorkbenchPanel`；不得再单独保留 Hero 页头或第二层概况壳。
50. 列表页 `操作` 列的动作间距当前统一跟随共享基线：表格态只允许使用 `StandardWorkbenchRowActions`；桌面 `table` 变体中的所有可见动作都必须按共享左对齐语法展开，行内默认最多 2 个直出动作，超过 2 个时必须进入 `更多`，且 `2` 个动作与 `详情 / 编辑 / 更多` 这类 `3` 个可见动作仍必须保持同一套固定共享间距。与此同时，`standard-row-actions-column` 的 fixed 列单元格必须共享左侧起笔基线，避免 `进入工作台 / 删除`、`详情 / 观测` 与单个 `详情` 这类长短文案页再次漂移。页面若重新出现 `gap="wide"`、`gap="compact"` 的表格动作组、额外显式桌面 `distribution`、`resolveWorkbenchActionColumnWidth({ gap: ... })`、`Math.max(160, ...)` 这类列宽 hack，或 3 个及以上直出动作，视为视觉回退。
51. `产品定义中心` 详情当前已固定为 `72rem` 统一 `产品经营工作台` 抽屉，并在 `2026-04-01` 切换到“2026 A 方案年报中枢式 / 指挥舱刊物式”内页骨架：头部不再承接设备规模，正文统一按“刊头 + 细线真页签 + 年报展开页 + 刊栏式治理工作区”收口，不得回流旧的横向数据带、趋势双表格、第二套概况卡、政企简报式中轴身份框或展签陈列式大头部。
- `产品定义中心` 当前固定把 `详情 / 物模型治理 / 关联设备 / 编辑治理` 统一收口到同一 `产品经营工作台` 抽屉；`/products` 页面不得再直接并行挂载独立 `DeviceListDrawer`、`ProductModelDesignerDrawer` 或编辑对象抽屉壳。

- `产品定义中心` 列表行内当前只允许保留 `进入工作台` 与 `删除` 两个直出动作，且不再显示 `更多`；这组双按钮必须保持紧凑间距，产品层的 `编辑 / 物模型治理 / 关联设备` 不得再作为同级直出按钮回流到主列表操作列。
- `产品经营工作台` 当前固定采用“2026 A 方案年报中枢式 / 指挥舱刊物式”：头部只保留刊头标题、短状态句、去重元信息行和细线真页签；不得再回流左右分栏头部、中轴身份框、头部设备指标带、第二组概况卡、展签眉标或厚 hero 背景。

- `物模型治理` 当前固定采用“刊头 + 摘要标尺 + 治理清单 / 正式模型工作区”；手动提炼区内部步骤说明必须改写成连续治理清单，不得再回流三张并列说明卡、产品元信息 chips、`当前产品` 重复文案、侧边治理轨道、厚 hero-stage 或第二套正式模型页面骨架。
- `关联设备` 当前固定采用“台账横标尺 + 设备清册主区”；不得再保留独立章节头、重复 `关联设备台账` 标题、台账导语或把台账主区继续挤到首屏之后。
- `编辑治理` 当前固定采用“修订刊头 + 修订标尺 + 内联治理提示 + 表单主区”；上下文条内不再承接解释性小字，表单前也不得再补导语说明，不得回流大标题 hero、双层说明卡或让表单继续退居次屏。
- `产品经营工作台` 四页当前都必须优先删除判断性文案、重复产品身份、重复导语和重复设备属性展示；后续若继续精修，应优先动排版秩序、区块对齐和文案压缩，而不是再补新卡片、新颜色或新提示层。
- 后续若产品工作台继续精修，优先动标题层级、节奏和区块对齐，不要第一反应增加新卡片、新色彩或新提示块；“更大气”的首选手段是排版秩序，不是视觉堆叠。
- `经营总览` 当前继续执行极简文案规则：优先保留标题、字段和值，删除重复解释、副说明、接入提示卡、状态胶囊、额外判断句、正文右侧指标栏、共享总外框和治理区；若板块里出现“字段后再补一整段解释”的写法，默认视为需要继续压缩，判断性内容应优先改写成契约线表或档案卡中的事实字段。
- `关联设备` 当前固定回到统一主色体系：指标横带不得再使用 `success / danger / accent` 一类整块卡片配色，页面级指标只允许使用主色层次；在线/离线、激活状态等语义色仅保留在表格行内 tag 或确有必要的局部状态位。
48. `告警详情`、`事件详情`、`异常详情` 与 `事件协同台` 内的派发 / 关闭抽屉，当前也统一纳入“无英文 eyebrow、只保留中文标题 / 副标题 / 区块标题”的规则；详情抽屉的层级表达不得再回流 `Alarm Detail`、`Event Detail`、`Audit Log Detail`、`Event Workflow` 一类装饰性眉题。
49. `产品定义中心`、`设备资产中心` 工具栏右侧若同时存在 `导出列设置 / 导出选中 / 导出当前结果 / 清空选中 / 批量删除` 等低频治理动作，统一通过 `StandardActionMenu label="更多操作"` 收口；只保留 `刷新列表` 和当前最核心入口直出，避免右侧工具条重新堆满同级文字按钮。
50. `/system-log` 的系统异常模式当前统一使用 `异常台账`、`异常详情`、`异常台账导出列设置` 三组标题语义，且顶部工具条不得再放 `链路追踪台 / 失败归档` 跨页按钮；跨页协同继续下沉到行级 `追踪` 和详情抽屉动作。
51. 复用 `standard-row-actions-column` 的所有桌面表格当前必须接入共享自适应操作列宽度，不得继续在单页写死 `150 / 180 / 210 / 224 / 288` 等固定值；列宽要按照“直出动作数量 + 是否存在 `更多` 菜单 + 标签长度”自动收口，并继续落到共享宽度分层：单动作桌面列固定 `96px`、双动作桌面列固定 `112px`、`前 2 项直出 + 更多` 的三段式桌面列固定 `160px`，且这些档位在桌面列表里都统一使用左对齐。`告警运营台`、`事件协同台` 这类存在 `详情 / 确认 / 抑制 / 关闭` 多个动作的运营台表格，以及 `实时监测台`、`GIS态势图` 这类单动作风险运营页，也必须直接复用 `StandardWorkbenchRowActions + standard-row-actions-column + resolveWorkbenchActionColumnWidth`，不得继续页面内手拼 `StandardRowActions`、显式传桌面 `distribution`、显式传 table `gap`，或额外追加页面私有列宽下限。
52. 当列表页的可见动作数量会随当前页数据状态变化时，桌面操作列宽必须按当前页真实可见动作的最大档位动态收口，而不是按页面理论最大动作数长期占位。`告警运营台`、`事件协同台` 当前已按此规则改用基于当前页行数据的宽度解析；如果当前页只剩双动作，不得继续保留三动作列宽空白。
51. 共享组件契约当前也必须同步收口：`eyebrow` 仅允许保留在最底层 `PanelCard`、`StandardWorkbenchPanel`；`StandardDetailDrawer`、`StandardFormDrawer`、`ResponsePanel`、`DeviceListDrawer`、`IotAccessWorkbenchHero`、`IotAccessSignalDeck` 这类页面中层包装组件不得再定义、转发或借由根节点透传 `eyebrow`，并通过静态守卫持续防回退。
52. `告警详情`、`事件详情`、`异常详情` 这类运维型详情抽屉，首屏摘要与后续上下文分区不得再重复堆叠同名字段标签；`当前状态 / 触发时间 / 责任人 / 操作时间 / 操作用户` 等高频字段应明确归属在单一分区内，避免抽屉阅读路径来回跳转。`告警运营台`、`事件协同台` 工具栏中的导出列设置、导出选中、导出当前结果、清空选中等低频动作也应统一收口到 `更多操作`，只保留刷新等高频动作直出。
53. `/audit-log` 与 `/system-log` 这类日志治理台账，工具栏也必须遵循同一节奏：桌面端只直出 `刷新列表`，`导出列设置 / 导出选中 / 导出当前结果 / 清空选中` 统一进入 `StandardActionMenu label="更多操作"`；不要因为审计与异常模式共用页面就回退到一排平铺文字按钮。
54. `/help-doc` 这类平台治理编排页，即使工具栏低频动作只剩 `清空选中` 一项，也继续通过 `StandardActionMenu label="更多操作"` 收口；不要为了少一次点击再把低频治理动作放回直出按钮区。
55. 第一批接入侧高频台账当前已按“来源页 + 目标页”方式推进：`/products` 作为模板来源页，`/devices`、`/message-trace`、`/system-log` 与失败归档作为首批目标页；后续批次必须继续沿该方式推进，不得跨域混改。
56. 第一批接入侧高频台账若已切到共享列表模板，页面 scoped CSS 只允许保留业务差异；`standard-list-surface`、`standard-mobile-record-*`、`StandardWorkbenchRowActions` 与自适应操作列宽度不得在后续页面回退为私有实现。

## 2. 页面结构治理重点

### 2.1 列表页

- 默认结构保持为 `概览或标题区 -> 筛选区 -> 工具条 -> 列表 -> 分页`。
- 同一台账优先收口为一张主卡，避免“说明卡 + 列表卡 + 分页卡”层级过深。
- 已提交筛选条件必须有显式反馈，支持单项移除和一键清空。
- 平台治理和风险处置类列表继续沿用统一操作条、长文本省略和选择态计数，不回退到页面私有工具条。
- 组织架构、区域版图继续保持“默认懒加载树 / 搜索后扁平分页”双模式；模式提示统一放入 `StandardWorkbenchPanel` 的 `notices` 槽位，不再保留页面私有提示条位置。
- 账号中心、角色权限与其他普通列表页一致，统一采用 `StandardWorkbenchPanel + StandardListFilterHeader + StandardAppliedFiltersBar + StandardTableToolbar + StandardPagination` 节奏，不再保留旧列表壳。
- 站内消息管理、帮助文档管理、导航编排、数据字典、通知编排等治理类列表，如果筛选项超过 4 个，优先让 `StandardListFilterHeader` 自动承接首行收拢与展开，不再手写占位空列维持按钮对齐。
- `/message-trace`、`/audit-log` 与 `/system-log` 这类链路 / 日志台账页，筛选区也必须复用 `StandardListFilterHeader + StandardAppliedFiltersBar`；高频入口走快速搜索，其余条件收口到“更多筛选”，长文本列继续通过 `StandardTableTextColumn` 保持单行省略与悬浮补全。
- 链路 / 日志台账页若存在聚合统计，优先放入顶部紧凑状态条和 `StandardTableToolbar.meta-items`；不要再为单页回流第二套私有统计条或概况卡。
- 若同一路由下承接“主链路台账 + 失败归档”双模式，模式切换入口统一放在面包屑下方的 `IotAccessTabWorkspace`；筛选骨架、详情抽屉和工具栏继续复用同一套共享标准件。
- `/devices` 当前在同一列表内承接“已登记资产 + 未登记上报名单”双口径时，区分入口应优先通过 `登记状态` 筛选和列表标签完成；未登记详情继续复用同一详情抽屉，只增加失败阶段、Topic、TraceId、原始载荷等排障字段，不再并行搭第二套失败列表页。
- `/products`、`/devices` 若从诊断页或失败归档承接治理动作，优先在台账顶部保留一条紧凑来源提示并让列表直接可用；即使当前列表尚无数据，也不要为了提示来源再额外堆一块说明卡或二级空态壳。
- `接入智维` 子页若存在多个并列内容域，应只在存在两个及以上真实业务视图时使用同页 Tab 工作区组织单主内容区；不要再新增页面私有 pill、假页签、说明墙或二级轻路由表达同页切换。
- 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理、审计中心 / 异常观测台、链路追踪台，当前统一纳入 `list:guard` 守卫范围；后续新增纳管页时，必须先补 allowlist，再迁移页面。

### 2.2 详情与抽屉

- 详情抽屉优先采用“摘要秒开 + 后台静默补数”的体验，不把正常刷新状态包装成红色错误提示。
- 详情抽屉内的加载、空态、错误态必须与 `StandardDetailDrawer` 统一，不为单页单独造一套提示组件。
- 表单抽屉继续复用 `StandardFormDrawer` 和 `StandardDrawerFooter`，避免重复底部按钮样式与关闭逻辑。
- 切换型按钮、pill 选项和链路验证中心中的“传输方式 / 上报模式”当前已统一抽到 `StandardChoiceGroup`；后续类似切换交互直接复用，不再在页面内书写 `--active` 私有样式。
- 桥接详情抽屉继续采用“摘要秒开 + 消息原文与尝试明细并行补数”的模式；尝试明细若为空或失败，只在抽屉内部表达，不影响上层治理列表继续使用。
- `/devices` 详情抽屉当前固定复用 `DeviceDetailWorkbench`，并与编辑页保持同一分区语法：已登记设备必须按 `基础档案 / 父子拓扑 / 状态与维护属性 / 认证字段 / 扩展信息` 组织，未登记设备必须按 `来源档案 / 失败摘要 / 最近载荷` 组织。标题右侧的 `已登记 / 在线 / 已激活 / 启用` 状态带必须与标题保持同一行，不得自动换行到下一层，且详情、编辑、更换抽屉默认不再展示自动补数提示带；详情字段值必须与编辑页保持同一常规字号和字重，不得再单独放大加粗。上述已登记板块若字段或属性全部为空，则直接隐藏整块，不再展示空模块占位；不得再回退到旧的双栏台账矩阵、小卡片墙或上下文跳跃的堆叠布局。

### 2.3 链路验证中心

- `/reporting` 当前已升级为“共享诊断状态头 + 结果复盘 / 模拟验证 / 最近记录”真页签主控台；后续只允许在现有真页签和共享状态头语法内迭代，不再回退到左右双栏工具页或说明墙。
- `链路验证中心` 顶部诊断状态头应保持“单一摘要面 + 侧栏卡点/动作 + 上下文 chips”的轻量层级，不再回流三张等权状态卡、装饰性 badge 或大写 eyebrow 式首屏语气。
- 发送结果下方的“处理时间线”必须复用 `StandardTraceTimeline`，按同一品牌和共享 token 展示阶段、状态、耗时、处理类/方法与摘要；不要再为单页散写私有 timeline 卡片或 rail。
- `deviceCode` 继续作为唯一主查询入口；`productKey`、`protocolCode`、`clientId` 保持只读派生。
- JSON / XML 格式化仅服务编辑体验；只有可识别 JSON 才参与协议帧判断。
- 页面级状态摘要与发送就绪度优先通过 `MetricCard` 收口，设备查询反馈优先通过 `StandardInlineState` 表达，不回退到页面私有提示条。
- MQTT 模拟的“等待消费回流 / 已绑定 trace / 时间线已过期”状态提示，继续放在共享工作台语义里表达，不新增蓝/橙/紫私有强调条。
- `最近记录` 内部的 `全部 / 失败 / 等待中 / 成功` 状态切换继续复用 `StandardChoiceGroup`，不要在局部列表里再散写第二套筛选 pill 和 active 样式。

### 2.3.1 链路 / 日志时间线

- `/message-trace` 详情抽屉中的“处理时间线”继续复用 `StandardTraceTimeline`，并与 `/reporting` 保持相同字段顺序、状态色和空态文案。
- 时间线过期后的降级提示必须明确说明“仅保留消息日志”，避免用户误判为页面加载失败或权限问题。

### 2.4 数据校验台

- `/file-debug` 优先收口到 `StandardWorkbenchPanel + StandardListFilterHeader + StandardInlineState` 的单入口工作台结构，不回退到页面私有散卡片表单。
- 查询区、文件快照、固件聚合和原始响应继续复用共享语法，保持“查询 -> 设备校验 -> 原始响应”的固定节奏；若查询区已经通过 `StandardInlineState` 交代数量和刷新状态，就不要再补一块重复的概况卡。
- 壳层、登录页、驾驶舱和列表页结果区的高曝光渐变、弱纹理、hover 高亮与 loading 蒙层，必须优先来自全局 token 和共享样式入口；页面只允许做轻量语义化包装，不得继续散写另一套品牌 literal。

## 3. 稳定性与质量规则

1. 控制台 warning 需要区分“单次巡检采样噪音”和“可稳定复现问题”，只有复测成立的问题才进入活跃治理范围。
2. `v-permission` 不得直接挂在没有稳定原生根节点的组件上；组件型操作菜单优先在脚本层过滤 action 列表。
3. 移动端侧栏关闭前必须先移出焦点，并只在桌面折叠态使用 tooltip 包裹导航图标。
4. 浏览器自动化与视觉回归的失败证据继续保留在历史台账中，但长期有效规则要回写到本文件和 [06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md)。
5. 桥接响应摘要、目标摘要、帮助搜索高亮等消费态文本继续统一走安全文本切片渲染，不回退到 `v-html` 直出。
6. 共享动作列与文本列的防回退继续由 `npm run component:guard` 守卫；当前纳管页已覆盖产品/设备、链路追踪/异常观测、风险运营/风险策略、平台治理与质量工场代表页，后续新增批次页面仍需先定义 allowlist，再迁移，不要一次性全站混改。
7. 共享 `StandardActionMenu`、`StandardRowActions` 一类动作容器必须把 `items` 视为可空输入：组件内部统一回落空数组并自动禁用触发器，不允许在 `computed` 或模板里直接读取未兜底的 `items.length`，避免权限切换、异步装配或路由首屏恢复时把整页渲染打断。
8. 已经由共享请求层弹出友好提示的 `handled` 请求异常，不再额外输出页面级 `console.error`；浏览器控制台应优先保留真正未处理的前端错误，避免业务失败提示淹没真实缺陷。
9. `style:guard` 当前只允许在 baseline 内保留少量历史字面量余量；新增或回流的圆角 / 阴影字面量一律视为样式漂移回归，必须在提交前通过 token 化或共享样式收口恢复到基线以内。
10. 壳层 `通知中心` 的后台自动刷新必须遵守“未读统计后台轻拉取、通知分页显式打开再加载、自动请求单飞、成功 `60` 秒冷却、失败 `15` 秒退避”的统一规则；`mount / visibilitychange / focus / BroadcastChannel / storage` 这类自动场景不得再直接并发预拉 `/api/system/in-app-message/my/page`，也不得在上一条 `/my/unread-count` 未返回时继续叠发下一条请求。
