# 15 前端优化与治理计划

> 文档定位：长期生效的前端治理规则、开放问题和下一步优化方向。
> 适用角色：前端研发、全栈研发、设计协作、代码审阅者、AI 前端助手。
> 权威级别：一级权威。
> 上游来源：`spring-boot-iot-ui` 当前实现、[06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md)、近期前端改造结论、归档快照。
> 下游消费：页面重构、风格收口、交互评审、技能文档、前端任务模板。
> 变更触发条件：共享组件规则变化、页面风格收口、列表/抽屉模式调整、前端稳定性问题暴露。
> 更新时间：2026-04-12

## 1. 当前长期有效规则

1. 新页面优先复用 `PanelCard`、`StandardPagination`、`useServerPagination`、`StandardTableToolbar`、`StandardTableTextColumn`、`StandardDetailDrawer`、`StandardFormDrawer`、`StandardDrawerFooter`、`confirmAction` 和共享列表样式。
2. 查询、新增、重置、删除、批量、刷新及其他高频动作，优先复用 `StandardButton` 表达统一按钮语义，不再在页面内散写第二套尺寸、配色和 hover 规则。
3. `StandardButton` 当前作为按钮权威入口，继续统一“主操作白字 / 次操作橙字 / 品牌链接橙字”的文字色和层级表达；共享组件与页面内高频动作不得再单独覆写按钮文案颜色。
4. 表格与卡片中的“详情 / 编辑 / 更多”类操作统一复用 `StandardActionLink`、`StandardActionMenu`、`StandardRowActions` 与 `StandardWorkbenchRowActions`；`StandardActionLink` 已移除 `tone`，`StandardActionMenu` item 已移除 `tone`，页级组合优先交给 `StandardWorkbenchRowActions`，不得继续在页面里散写一组 `el-button link + el-dropdown` 模板或伪危险语义接口。
5. `StandardRowActions` 当前固定使用 `variant="table | card | editor"`：桌面表格操作列统一走 `table`，移动卡片操作区统一走 `card`，可换行编辑区统一走 `editor`；`StandardWorkbenchRowActions` 继续在此基础上统一直出动作与“更多”菜单的组合，其中桌面 `table` 变体固定使用共享桌面间距，不再因 `2` 个动作还是 `3` 个动作切换成不同分布，移动端 `card` 变体继续保持自适应触控布局，不得再依赖页面私有 `:deep(...)` 覆写去修按钮等宽、触控尺寸和下拉对齐。
6. 当列表操作总数 `<= 3` 时，默认全部直出；只有总动作数 `> 3` 时，才通过 `StandardWorkbenchRowActions` 收口为“前 3 项直出 + 更多”。用户管理等只有 `编辑 / 重置密码 / 删除` 三项动作的治理页不得再保留多余 `更多`；相邻动作间距仍沿用同一套共享标准。产品定义中心、设备资产中心、链路追踪台、异常观测台，以及风险运营下的 `实时监测台 / GIS态势图 / 告警运营台 / 事件协同台`，都不得再在页面 CSS 中私自覆写操作列 gap，也不得另起一套桌面 `distribution` 规则。`风险对象中心` 当前统一按 `详情 / 编辑 / 维护绑定 / 待治理转正 / 删除` 的共享动作列语义收口，不再保留旧“绑定设备”文案或页面私有例外逻辑。
补充：纳管页的 `操作` 列必须声明 `class-name="standard-row-actions-column"`，由共享全局样式统一处理 fixed 列内的 `overflow / text-overflow / 字号 / 留白`。`StandardRowActions` 的桌面 `table` 变体必须占满整列，并统一采用左对齐分布，`standard-row-actions-column .cell` 也继续保持左贴齐起笔基线；不要再让默认 `.cell` ellipsis 把最后一个动作裁成尾部圆点，也不要让不同文案长度的页面重新长出私有对齐策略。
7. 固定宽度下的 `消息类型`、`Topic`、编号、TraceId 等文本列，只要存在回落原始值或长度不可控的可能，就继续优先复用 `StandardTableTextColumn` 承接省略与 tooltip，不回退到裸 `el-table-column` 文本节点。
8. 按钮继续按品牌 / 辅助层级表达：查询类主操作、对象洞察台“刷新洞察”、确认、新增、删除、批量等主操作统一使用品牌主橙渐变白字。次操作使用白底橙描边和橙字，`link / text` 中的主语义操作保持品牌橙；删除也跟随同一套橙色按钮语义，不再保留红色危险按钮分支。
9. 系统弹窗、抽屉底部和遗留全局按钮类必须跟随共享按钮层级收口；hover 以轻微加深或品牌浅底纹为主，不再额外叠加页面私有渐变和动效。
10. 系统正文与筛选标签统一优先使用蓝灰字体系，避免纯黑字在浅色控制台页面上显得发硬；操作区、操作列和主语义链接按钮继续复用品牌橙，`详情 / 编辑 / 更多 / 查看设备 / 删除 / 更换 / 洞察` 统一与查询按钮保持同一识别。
11. 传输方式、模式切换、分组选项等“点前 / 点后”状态按钮统一复用 `StandardChoiceGroup`；不得继续在页面里散写 `active` class 和局部 pill 按钮样式。
12. 页面配色继续统一到 `tokens.css` 与 `element-overrides.css` 的品牌 token；当前全局基线固定为“阿里云浅色控制台”方向，禁止为单页重新发明另一套蓝橙紫主色。
13. 工作台、抽屉、确认交互和概览卡必须保持“浅背景 + 白卡 + 轻描边 + 轻阴影”的控制台语法，不回退到高饱和营销式视觉。
14. 顶部壳层、分组导航、命令面板、通知中心、帮助中心继续收口到共享 shell 编排，不新增页面私有壳层能力。
15. 五个一级工作台下的功能页根容器，当前统一优先复用 `StandardPageShell`；页级节奏、breadcrumb 到首张主卡片的垂直留白、标题/副标题字距，不再由页面私有 `padding / background / border / shadow` 控制。
16. 品牌层级继续固定为“品牌橙主操作 + 云蓝辅助信息 + 蓝灰正文与描边”；登录页、壳层、驾驶舱、loading 蒙层和工作台概览不得再各自维护另一套大面积渐变或纯黑字方案。
17. `/in-app-message` 的桥接效果运营继续保持“同页独立 workbench 分区 + 独立桥接详情抽屉”结构，复用共享分页、详情抽屉和表格列模式，不新增第二套桥接治理页面。
18. 新增页面和组件继续遵守 [06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md) 中的“路由页 / 壳层编排 / 共享标准件 / 业务包装层”分层与命名规则，不回退到无语义命名。
19. 当前前端最小工程门禁继续保持 `npm run build`、`npm run test`、`npm run component:guard`、`npm run list:guard`、`npm run style:guard`；其中 `component:guard` 独立负责共享动作/文本列契约守卫，`list:guard` 独立负责列表骨架守卫，不与当前 `style:guard` 的半径基线噪音绑定；若后续接入 ESLint / Stylelint，需要先更新 `06` 与本文件。
20. 视觉 token 必须优先复用 `var(--radius-*)`、`calc(var(--radius-*) + ...)` 与 `var(--shadow-*)`；不要继续在页面或组件里新增 `border-radius: 999px / 8px / 6px`、重复 `box-shadow: inset 0 1px 0 rgba(...)` 这类字面量，若现有 token 不够用，应先补 `tokens.css` 再落样式。
21. 全站中文排版当前统一采用“宋感字体栈”基线：`--font-display` 与 `--font-body` 必须保持同一套宋感 serif 栈，标题、正文、表格、表单、按钮和抽屉跟随全局 token；只有 `--font-mono` 继续服务技术文本。不要在单页 scoped CSS 中把正文或标题重新切回无衬线字体。
22. 产品定义中心、设备资产中心等高频台账的工具栏右侧动作较多时，必须优先通过共享工具栏的紧凑间距与自适应布局保持同组功能在桌面端单行展示；不要把同一组功能无规律挤到下一行。表格操作列如果包含 `更多` 这类下拉触发器，固定宽度要预留额外余量，避免右侧 fixed 列裁切触发器文案。
23. 当前共享动作/文本列纳管范围已从首批试点扩展到 `风险运营`、`风险策略`、`平台治理`、`质量工场` 代表页；后续新增批次页面时，必须先补 `component:guard` allowlist，再迁移页面接入，避免“先混改、后补规则”。
24. 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理等列表页，筛选区默认复用 `StandardListFilterHeader`；不要再回退到页面私有 `search-form + el-row/el-col + text-right` 模板。
25. 纳管列表页的已生效筛选状态统一复用 `useListAppliedFilters`；页面不再各自维护“标签拼装 + 单项移除 + 全部清空 + 高级筛选计数”的私有实现。组织架构、账号中心、角色权限、区域版图当前也已纳入同一口径。
26. `风险策略` 下的 `风险对象中心 / 阈值策略 / 联动编排 / 应急预案库` 当前已统一接入 `产品定义中心` 的单主列表合同：结果区复用 `standard-list-surface`，首屏空数据展示骨架屏，已有结果刷新只叠加局部浅色蒙层，空态继续区分“无数据”与“筛选无结果”，分页固定为 `10 / 20 / 50 / 100` 四档并在 `total=0` 时隐藏。后续同批治理页要么一起沿用该合同，要么先在本文件记录例外原因，不能再各自长出第二套列表加载节奏。
补充：`风险运营` 下的 `实时监测台 / GIS态势图 / 告警运营台 / 事件协同台`，以及 `平台治理` 下的 `组织架构 / 账号中心 / 角色权限 / 导航编排 / 区域版图 / 数据字典 / 通知编排 / 帮助文档管理` 当前也已纳入同一合同。其中 `GIS态势图` 继续保留“已定位 / 未定位”双结果分区，但两个分区都必须独立复用 `standard-list-surface + StandardPagination`；`组织架构 / 导航编排 / 区域版图` 继续保留“默认懒加载树 / 搜索后扁平分页”双模式，但主列表请求仍必须遵守同一首屏骨架、局部刷新、空态分流与过期响应保护，不得再因为树结构而回退到旧式 `v-loading + 空白表格壳`。
27. `链路追踪台`、`异常观测台`、审计中心这类日志 / 诊断台账，若需要展示统计摘要，优先放入 `StandardTableToolbar.meta-items`、分页邻近区域或详情区；只有确有必要时才补 `notices`，不要再回退到页面顶部紧凑状态条、辅战区或私有统计条。
28. 当新增诊断能力与现有已授权日志台账强耦合、且共享环境菜单路径可能未及时同步时，优先在同一路由内通过 `StandardChoiceGroup` 等共享切换组件承接双模式，不要先创建第二条静态路由再回头兜权限。
29. 产品定义中心内的产品物模型设计器继续复用同一抽屉，不新增一级路由；`契约字段` 与 `契约字段描述` 必须留在同一页工作区，并保持“共享经营头部 + 治理摘要结果卡 + 样本输入/识别结果/本次生效/正式字段”结构语法；`版本台账 / 回滚试算` 只在存在真实批次数据时按需出现。页级标题、描述、来源提示与治理文案还必须显式回答“产品定义 + 契约治理 + 版本治理 + 风险目录入口”，不得回流“产品管理页”旧语义。不得再回流 `ProductModelDesignerDrawer`、第二层抽屉、静态阶段栏或页内重复入口。
30. 产品物模型提炼只允许服务当前选中的产品，且单次只支持粘贴 `1` 个设备样本 JSON；客户主路径固定展示 `业务数据 / 状态数据` 与 `单台设备 / 复合设备` 两组切换，不再回流 `自动提炼`、规范预设、运行期开关、`其他数据` 样本类型或人工补录大块表单。
31. 产品物模型治理界面必须诚实表达证据边界：手动提炼出的数组、空对象和结构噪音只能作为 `ignoredFieldCount` 摘要展示，不得伪造成属性；非结构化噪音默认停留在“继续观察”，不要为了看起来热闹而在结果区硬造字段。复合设备模式只允许暴露 `父设备编码 / 逻辑通道编码 / 子设备编码` 三类输入，不得再让用户接触 `sourceDeviceCode / extractMode` 这类后端专用字段名。
32. 设备资产中心若继续承接“已登记资产 + 未登记上报名单”双名单能力，必须保持同一路由 `/devices`、同一套 `StandardWorkbenchPanel + StandardListFilterHeader + StandardTableToolbar + StandardDetailDrawer` 骨架；未登记行继续保持只读，不得为其再拆第二条列表路由或页面私有详情壳。
33. `接入智维总览` 只保留 `资产底座 / 诊断排障` 两组真业务入口、最近使用、推荐处理顺序和全部能力；完整排障决策树只允许在 `/device-access` 总览页展示，子页禁止再复制模块目录、能力墙、大段介绍文案或整棵决策树。
34. `链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 的顶部只允许使用轻量标题壳和必要动作；摘要优先下沉到共享工具条、`StandardInlineState` 或详情区，不要再叠加第二层概况卡。四个诊断子页还必须统一只回答 `当前节点 + 下一步`：`/reporting=链路验证`、`/system-log=后台异常核对`、`/message-trace=主链路复盘`、`/file-debug=数据与原始响应校验`。
35. `接入智维` 六个核心页的二轮视觉精修已统一采用“静稳控制台基线 + 少量现代 SaaS 留白”，继续保持当前主配色不变；后续页面现代化优先通过标题层级、筛选密度、工具条轻重、表头亮度和操作节奏实现，不得再以新增私有主题色换取所谓互联网感。
36. 单主列表页与真页签诊断页当前都必须复用同一套轻量视觉语法：更轻边框、更软阴影、更淡表头、短说明和稳定字距。页面若需要继续优化，应先扩共享组件或 token，不要在 scoped CSS 中各自再造第二套“高级感”样式。
22. 宋感排版的优化方向当前固定为“强标题、弱正文、技术文本保留等宽”，而不是把所有文字都做成同强度宋体。页级标题、工作台标题、抽屉标题、区块标题、eyebrow 与 breadcrumb 必须优先复用共享字号/字距 token；筛选项、工具条元信息、表格正文、表单正文和帮助性文案必须保持克制，不得再为追求“高级感”把密集文本统一做大做重。
23. `tokens.css` 中的 `--type-title-*`、`--type-overline-size`、`--type-body-size`、`--type-caption-size`、`--type-label-size`、`--type-toolbar-meta-size` 与 `--font-letter-spacing-*` 当前已作为全站排版治理合同。后续若浏览器验收仍觉得页面不够稳，优先扩这些共享 token 或共享组件，不要在单页里重新散写第二套字号和字距。
24. `StandardPageShell`、`StandardWorkbenchPanel`、`StandardListFilterHeader`、`StandardTableToolbar` 当前已纳入同一宋感层级治理；任何页面若再次通过 scoped CSS 覆写页头标题、筛选提示、表头、工具条 meta 或 eyebrow 节奏，视为视觉回退，必须在同轮修复。
25. 产品定义中心、设备资产中心等高频台账的工具栏右侧动作较多时，必须优先通过共享工具栏的紧凑间距与自适应布局保持同组功能在桌面端单行展示；不要把同一组功能无规律挤到下一行。表格操作列如果包含 `更多` 这类下拉触发器，固定宽度要预留额外余量，避免右侧 fixed 列裁切触发器文案。
26. 当前共享动作/文本列纳管范围已从首批试点扩展到 `风险运营`、`风险策略`、`平台治理`、`质量工场` 代表页；后续新增批次页面时，必须先补 `component:guard` allowlist，再迁移页面接入，避免“先混改、后补规则”。
27. 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理等列表页，筛选区默认复用 `StandardListFilterHeader`；不要再回退到页面私有 `search-form + el-row/el-col + text-right` 模板。
28. 纳管列表页的已生效筛选状态统一复用 `useListAppliedFilters`；页面不再各自维护“标签拼装 + 单项移除 + 全部清空 + 高级筛选计数”的私有实现。组织架构、账号中心、角色权限、区域版图当前也已纳入同一口径。

- `契约字段` 当前固定采用“刊头 + 治理摘要结果卡 + 样本输入 / 识别结果 / 本次生效 / 正式字段工作区”；治理语义必须保持言简意赅，抽屉统一收口为同页工作区，不再拆出第二层页面，也不再保留 `待输入样本 / 待确认识别结果 / 待提交审批 / 审批中 / 已发布 / 可回滚` 这类静态阶段提示。样本输入区必须优先使用一条操作条承接 `样本类型 / 设备结构 / 格式化 JSON / 提取契约字段`，手动模式不再出现编号解释清单，改为面向客户的短文案与紧凑提示行。对比行先按 `可直接生效 / 待确认 / 继续观察 / 存在差异` 组织，再按 `property / event / service` 筛选；每张卡必须直接回答 `样例值`、`正式字段：已存在/暂无`、来源短标签和状态短句，底部确认区固定回答“已选 X 项，确认后将写入正式字段”。
- `契约字段` 在 `确认并生效` 成功后，当前 compare 会话的待生效选择必须立刻清空，按钮同步转为不可点击，防止同一批字段被重复提交；若生效接口返回错误，只允许通过局部错误提示或消息提示反馈，不得把整个工作区切成“加载失败”空页。
- `契约字段` 当前在复合设备模式下只允许显式提供 `父设备编码 / 逻辑通道编码 / 子设备编码` 三类输入；页面可以读取现有 `iot_device_relation` 关系回填表单，但 compare 请求必须直接提交当前映射行，不得再回流 `自动提炼`、`sourceDeviceCode`、`extractMode`、协议模板摘要或第二套关系维护弹窗。该模式的页面文案必须保持“已识别为复合设备样本，将自动归一到子产品字段”这类短提示，并明确父终端状态字段不会进入子产品 compare 结果。
- `关联设备` 当前固定只保留 `设备清单` 主区；不得再回流 `设备总数 / 在线设备 / 离线设备 / 在线比例` 重复统计尺、独立章节头或 `关联设备台账` 导语。
- `编辑档案` 当前固定作为头部动作进入“修订刊头 + 内联治理提示 + 表单主区”；不得再回流为并列页签，也不得回流 `产品 Key / 协议编码 / 节点与状态` 重复标尺、解释性小字或表单前导语。
- `产品经营工作台` 当前各工作区都必须优先删除判断性文案、重复产品身份、重复导语和重复设备属性展示；后续若继续精修，应优先动排版秩序、区块对齐和文案压缩，而不是再补新卡片、新颜色或新提示层。
- 后续若产品工作台继续精修，优先动标题层级、节奏和区块对齐，不要第一反应增加新卡片、新色彩或新提示块；“更大气”的首选手段是排版秩序，不是视觉堆叠。
- `经营总览` 当前继续执行极简文案规则：优先保留标题、字段和值，删除重复解释、副说明、接入提示卡、状态胶囊、额外判断句、正文右侧指标栏、共享总外框和治理区；若板块里出现“字段后再补一整段解释”的写法，默认视为需要继续压缩，判断性内容应优先改写成契约线表或档案卡中的事实字段。
- `关联设备` 当前固定回到统一主色体系：指标横带不得再使用 `success / danger / accent` 一类整块卡片配色，页面级指标只允许使用主色层次；在线/离线、激活状态等语义色仅保留在表格行内 tag 或确有必要的局部状态位。
57. `告警详情`、`事件详情`、`异常详情` 与 `事件协同台` 内的派发 / 关闭抽屉，当前也统一纳入“无英文 eyebrow、只保留中文标题 / 副标题 / 区块标题”的规则；详情抽屉的层级表达不得再回流 `Alarm Detail`、`Event Detail`、`Audit Log Detail`、`Event Workflow` 一类装饰性眉题。
58. `产品定义中心`、`设备资产中心` 工具栏右侧若同时存在 `导出列设置 / 导出选中 / 导出当前结果 / 清空选中 / 批量删除` 等低频治理动作，统一通过 `StandardActionMenu label="更多操作"` 收口；只保留 `刷新列表` 和当前最核心入口直出，避免右侧工具条重新堆满同级文字按钮。
59. `/system-log` 的系统异常模式当前统一使用 `异常观测台`、`异常详情`、`异常观测台导出列设置` 三组标题语义；页内来源提示还必须固定回答 `当前节点：后台异常核对` 与“下一步回链路追踪台或治理页继续排查”。顶部重复异常统计说明必须移除，统计摘要继续只放在共享工具条 `meta-items`，且顶部工具条不得再放 `链路追踪台 / 失败归档` 跨页按钮；跨页协同继续下沉到行级 `追踪` 和详情抽屉动作。
60. `/system-log` 的系统异常模式在总动作数不超过 `3` 时必须直出全部行动作；当前固定语义为 `详情 / 追踪 / 删除`，不得再保留行内 `更多`。
61. `异常详情` 当前必须复用设备资产中心的扁平详情工作台语言：首屏轻摘要卡 + 纵向表格化台账分区 + 请求/响应快照，不得回流说明式 hero、左右分栏模块、长段解释或重复概况句；同一主标识在 title、tags、摘要卡、台账中最多保留一个主展示位。
60. 复用 `standard-row-actions-column` 的所有桌面表格当前必须接入共享自适应操作列宽度，不得继续在单页写死 `150 / 180 / 210 / 224 / 288` 等固定值；列宽要按照“直出动作数量 + 是否存在 `更多` 菜单 + 标签长度”自动收口，并继续落到共享宽度分层：单动作桌面列固定 `96px`、双动作桌面列固定 `112px`、三个可见动作的基础档位固定 `160px`，总动作数超过 `3` 时再追加 `更多` 触发器并按标签长度自动扩展，且这些档位在桌面列表里都统一使用左对齐。`告警运营台`、`事件协同台` 这类存在 `详情 / 确认 / 抑制 / 关闭` 多个动作的运营台表格，以及 `实时监测台`、`GIS态势图` 这类单动作风险运营页，也必须直接复用 `StandardWorkbenchRowActions + standard-row-actions-column + resolveWorkbenchActionColumnWidth`，不得继续页面内手拼 `StandardRowActions`、显式传桌面 `distribution`、显式传 table `gap`，或额外追加页面私有列宽下限。
61. 当列表页的可见动作数量会随当前页数据状态变化时，桌面操作列宽必须按当前页真实可见动作的最大档位动态收口，而不是按页面理论最大动作数长期占位。`告警运营台`、`事件协同台` 当前已按此规则改用基于当前页行数据的宽度解析；如果当前页只剩双动作，不得继续保留三动作列宽空白。
62. 共享组件契约当前也必须同步收口：`eyebrow` 仅允许保留在最底层 `PanelCard`、`StandardWorkbenchPanel`；`StandardDetailDrawer`、`StandardFormDrawer`、`ResponsePanel`、`DeviceListDrawer`、`IotAccessWorkbenchHero`、`IotAccessSignalDeck` 这类页面中层包装组件不得再定义、转发或借由根节点透传 `eyebrow`，并通过静态守卫持续防回退。
63. `告警详情`、`事件详情`、`异常详情` 这类运维型详情抽屉，首屏摘要与后续上下文分区不得再重复堆叠同名字段标签；`当前状态 / 触发时间 / 责任人 / 操作时间 / 操作用户` 等高频字段应明确归属在单一分区内，避免抽屉阅读路径来回跳转。`告警运营台`、`事件协同台` 工具栏中的导出列设置、导出选中、导出当前结果、清空选中等低频动作也应统一收口到 `更多操作`，只保留刷新等高频动作直出。
64. `/audit-log` 与 `/system-log` 这类日志治理台账，工具栏也必须遵循同一节奏：桌面端只直出 `刷新列表`，`导出列设置 / 导出选中 / 导出当前结果 / 清空选中` 统一进入 `StandardActionMenu label="更多操作"`；不要因为审计与异常模式共用页面就回退到一排平铺文字按钮。
65. `/help-doc` 这类平台治理编排页，即使工具栏低频动作只剩 `清空选中` 一项，也继续通过 `StandardActionMenu label="更多操作"` 收口；不要为了少一次点击再把低频治理动作放回直出按钮区。
66. 第一批接入侧高频台账当前已按“来源页 + 目标页”方式推进：`/products` 作为模板来源页，`/devices`、`/message-trace`、`/system-log` 与失败归档作为首批目标页；后续批次必须继续沿该方式推进，不得跨域混改。
67. 第一批接入侧高频台账若已切到共享列表模板，页面 scoped CSS 只允许保留业务差异；`standard-list-surface`、`standard-mobile-record-*`、`StandardWorkbenchRowActions` 与自适应操作列宽度不得在后续页面回退为私有实现。
- 68. `/products` 的 `对象洞察配置`、`编辑档案` 与 `契约字段 -> 当前已生效字段`，以及 `/insight` 的趋势区当前必须共享同一套客户命名：`监测数据 / 状态事件 / 运行参数`。产品侧不再出现 `设为状态趋势`、`状态数据` 这类旧名称；旧版 `group=status` 仅允许作为兼容输入存在，读取后必须自动归并到三分组新口径。正式 `property` 字段一旦改名，`iot_product.metadata_json.objectInsight.customMetrics[]` 中对应 `displayName` 也必须同步刷新，确保 `/insight` 的趋势图例、tooltip、属性快照与核心快照指标继续显示同一客户中文名。
- 68.1 正式 `property` 字段的单位当前也必须沿用同一路径维护，并写回 `specsJson.unit`；对象洞察趋势图例、tooltip 与综合分析在存在单位时必须自动拼接 `（unit）`，但若正式名称本身已显式带了同一单位，则不得重复追加。
- 69. `状态事件` 图表当前固定遵守 `0=正常` 的客户语义，缺报补点必须落到独立“未上报”哨兵值；tooltip、坐标轴与图例不得再把缺报点伪装成 `0`，也不得向客户暴露“补零补齐”一类实现说明。若未来再调整补点策略，必须先在本文件记录不会与真实状态码冲突的替代方案；与此同时，非状态类补位桶若值为 `0`，tooltip 必须直接显示数值 `0`，不得再翻译成 `否`。
- 70. `契约字段 -> 当前已生效字段` 当前允许直接删除正式字段，但删除动作必须同步清理 `iot_product.metadata_json.objectInsight.customMetrics[]` 中同标识符残留项，避免对象洞察继续引用已删除字段；若后续改成二次确认或审批链，仍必须保持“正式字段删除后对象洞察配置不留脏引用”这条底线。
- 71. `/insight` 的趋势图区继续保持低压阅读节奏：底部时间标签与折线末端之间必须保留稳定下边距，避免标签贴图；这类版式留白应通过共享图表配置收口，不允许后续在单一设备模板里写死私有偏移量。
- 72. 状态事件如果当前只落在 `正常 / 异常` 等双状态区间，图表必须继续使用更紧凑的共享高度与纵向状态带，避免仅有两条状态线时仍占满整张高图。
- `风险运营驾驶舱` 管理视角当前已接入治理经营聚合口径，KPI/队列/待办必须继续由 `GET /api/risk-governance/dashboard-overview` 驱动；后续若补充经营指标，优先扩该接口字段，不得在页面内重新回流静态“预设数字”。
- `产品定义中心` 当前已把任务化治理提示下沉到同页 `notices`，并且只允许基于 `coverage-overview + contract-release-batches` 计算“待发布合同 / 待发布风险指标目录 / 待绑定风险点 / 待补阈值策略”；后续迭代继续复用该链路，不新增平行待办页或私有计数源。

## 2. 页面结构治理重点

### 2.1 列表页

- 默认结构保持为 `概览或标题区 -> 筛选区 -> 工具条 -> 列表 -> 分页`。
- 同一台账优先收口为一张主卡，避免“说明卡 + 列表卡 + 分页卡”层级过深。
- 已提交筛选条件必须有显式反馈，支持单项移除和一键清空。
- 平台治理和风险处置类列表继续沿用统一操作条、长文本省略和选择态计数，不回退到页面私有工具条。
- 组织架构、区域版图继续保持“默认懒加载树 / 搜索后扁平分页”双模式；模式提示统一放入 `StandardWorkbenchPanel` 的 `notices` 槽位，不再保留页面私有提示条位置。
- 任何消费区域树的写入侧抽屉或表单，也不得在页面挂载时预加载 `/api/region/tree` 整棵树。`风险对象中心` 当前已收口为“打开抽屉后先拉根区域、展开节点再走 `GET /api/region/list?parentId=...`”的树选择合同；后续若组织、区域相关写入表单继续扩展，也必须优先沿用同一按需加载语法。
- `/risk-point` 的主表当前不得再展示任何绑定状态、设备数、测点数或待治理摘要；列表只负责承载风险点基础档案字段与详情入口，正式绑定设备卡片、测点列表和治理提示都必须下沉到详情抽屉。
- `/risk-point` 的风险点名称与行内“详情”必须统一打开只读详情抽屉；详情抽屉负责收纳绑定概况、风险点档案、负责人、正式绑定设备和测点明细，并从底部继续分流到“编辑风险点 / 维护绑定 / 待治理转正”。
- `/risk-point` 的“维护绑定”和“待治理转正”必须保持两个独立抽屉：前者只维护正式 `risk_point_device` 关系，后者只处理 pending 治理记录与转正留痕，不得再回流成同一混合面板。
- `/risk-point` 的新增/编辑抽屉中，`orgId / regionId / responsibleUser` 这类树选择或账号选择值必须继续按原始 `IdType` 保存与提交，不得在页面内再做 `Number()` 强转；若真实环境返回 19 位 Long，树选择器仍必须稳定回显组织/区域名称，而不是裸 ID。
- `/risk-point` 的负责人默认链路当前固定为“机构管理员优先，当前登录用户兜底”：选择所属组织后，若该机构存在 `leaderUserId`，默认带出该管理员与联系电话；若机构未配置管理员，则改为默认带出当前具备修改能力的登录用户资料。页面不得再把空负责人直接回退成机构座机或空白占位。
- 账号中心、角色权限与其他普通列表页一致，统一采用 `StandardWorkbenchPanel + StandardListFilterHeader + StandardAppliedFiltersBar + StandardTableToolbar + StandardPagination` 节奏，不再保留旧列表壳。
- `/role` 的授权抽屉固定为“三段式授权工作区”：页面树只承载目录/页面勾选，已选页面列表只承载当前页切换与移除，按钮权限只在当前页面面板内细化。桌面端优先保持“左树 + 右侧上下两段”的共享高度紧凑布局，移动端再顺序回落；树节点不再回流路由文本。目录节点不得参与授权提交，新勾选页面默认不自动附带按钮；禁止再把按钮节点塞回树里，或通过加宽抽屉、堆叠第二棵树来解决拥挤问题。
- 壳层账号中心保持现有控制台风格的单一轻量抽屉，不再拆出“实名认证 / 登录方式”独立静态抽屉；内容固定为 `基础资料 / 机构与角色 / 安全信息 / 实名资料` 四段，资料编辑与密码修改分别走当前用户资料保存和独立密码抽屉。
- 站内消息管理、帮助文档管理、导航编排、数据字典、通知编排等治理类列表，如果筛选项超过 4 个，优先让 `StandardListFilterHeader` 自动承接首行收拢与展开，不再手写占位空列维持按钮对齐。
- `/governance-task` 当前必须继续复用 `StandardWorkbenchPanel + StandardListFilterHeader + StandardButton + StandardTableToolbar` 的共享工作台语法；首屏只保留一个统一快速搜索框，固定支持 `审批单号 / 发布批次号 / 产品标识 / 设备编码 / TraceId`，不得再拆成多个孤立输入或页头私有搜索条。
- `/governance-task` 的 `全部 / 推荐优先处理 / 待审批 / 待发布合同 / 待绑定风险点 / 待补阈值 / 待补联动/预案 / 待运营复盘` 分类入口必须继续由 URL query 驱动；其中 `待审批` 固定映射 `executionStatus=PENDING_APPROVAL`，`推荐优先处理` 只允许在同一路由内重排现有列表并把推荐数量写入 `meta-items / 摘要卡`，不再回流第二套子路由、私有 pill 或说明墙。
- `/message-trace`、`/audit-log` 与 `/system-log` 这类链路 / 日志台账页，筛选区也必须复用 `StandardListFilterHeader + StandardAppliedFiltersBar`；高频入口优先走快速搜索，其余条件是否下沉到“更多筛选”由页面真实筛选密度决定，长文本列继续通过 `StandardTableTextColumn` 保持单行省略与悬浮补全。
- `/message-trace` 当前必须继续锁定“快速搜索优先、异常动作不回流正常链路”的治理规则：`链路追踪` 模式首屏固定同一行展示 `快速搜索（TraceId / 设备编码 / 产品标识） / 消息类型 / Topic / 查询 / 重置`，不再保留“更多筛选”切换；正常消息行内只保留 `详情`，`观测` 只允许留在 `失败归档` 等异常样本场景。
- `/message-trace` 下的 `接入失败归档台` 行级动作当前固定直出 `详情 / 追踪 / 观测` 三项，总动作数未超过 `3` 时不得再保留行内 `更多`；详情抽屉固定采用 `失败态势与处理概况 / 链路与主体台账 / 异常诊断与回跳 / 契约与报文快照` 四段纵向结构，与异常详情保持同一扁平台账语法。
- `/message-trace` 的 `链路追踪详情` 也必须继续沿用 `设备资产中心` 的详情阅读路径：首屏只保留轻摘要卡，`链路与接入台账` 内部固定拆为 `链路概览 / 接入概览` 两段纵向台账块，长字段使用宽行台账卡，`模板执行` 一类多段证据必须分行展示，不得再回退成被挤压的单条长文本。
- `/message-trace` 的 `Payload 对照` 必须继续沿用同一套详情卡节奏：每张 payload 卡片首屏直接展示 `当前状态` 与用途说明，完整内容再按需展开；不要再回退成只剩标题和 `复制 / 展开` 动作的薄行按钮区。
- `/message-trace` 的 `处理时间线` 首屏必须继续保持 `当前状态 / 处理节点 / Trace 归属 / 存储提示` 四张摘要卡，不要再把过期、存储异常和 Trace 归属压回单块提示文案；完整 `StandardTraceTimeline` 仍只在用户主动展开后渲染。
- 链路 / 日志台账页若存在聚合统计，优先放入顶部紧凑状态条和 `StandardTableToolbar.meta-items`；不要再为单页回流第二套私有统计条或概况卡。
- 若同一路由下承接“主链路台账 + 失败归档”双模式，模式切换入口统一放在面包屑下方的 `IotAccessTabWorkspace`；筛选骨架、详情抽屉和工具栏继续复用同一套共享标准件。
- `/devices` 当前在同一列表内承接“已登记资产 + 未登记上报名单”双口径时，区分入口应优先通过 `登记状态` 筛选和列表标签完成；顶部快速搜索必须保持单输入框，并统一支持 `设备编码 / 设备名称 / 产品 Key / 产品名称` 四字段命中。列表里的 `最近上报时间` 若携带显式 `UTC / offset` 信息，必须统一按 `Asia/Shanghai` 展示；未登记详情继续复用同一详情抽屉，只增加失败阶段、Topic、TraceId、原始载荷等排障字段，不再并行搭第二套失败列表页。
- `/products`、`/devices` 若从诊断页或失败归档承接治理动作，优先在台账顶部保留一条紧凑来源提示并让列表直接可用；即使当前列表尚无数据，也不要为了提示来源再额外堆一块说明卡或二级空态壳。
- `接入智维` 子页若存在多个并列内容域，应只在存在两个及以上真实业务视图时使用同页 Tab 工作区组织单主内容区；不要再新增页面私有 pill、假页签、说明墙或二级轻路由表达同页切换。
- 告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理、审计中心 / 异常观测台、链路追踪台，当前统一纳入 `list:guard` 守卫范围；后续新增纳管页时，必须先补 allowlist，再迁移页面。

### 2.2 详情与抽屉

- 详情抽屉优先采用“摘要秒开 + 后台静默补数”的体验，不把正常刷新状态包装成红色错误提示。
- 详情抽屉内的加载、空态、错误态必须与 `StandardDetailDrawer` 统一，不为单页单独造一套提示组件。
- 表单抽屉继续复用 `StandardFormDrawer` 和 `StandardDrawerFooter`，避免重复底部按钮样式与关闭逻辑。
- 任何支持“切换主对象后重新加载远程数据”的共享抽屉，都必须显式做 request-id 或等价的失效保护，避免旧响应回写当前上下文；抽屉 `close` 事件也不得因底层组件同时触发 `update:modelValue` 与 `close` 而重复向上传播。
- 切换型按钮、pill 选项和链路验证中心中的“传输方式 / 上报模式”当前已统一抽到 `StandardChoiceGroup`；后续类似切换交互直接复用，不再在页面内书写 `--active` 私有样式。
- 桥接详情抽屉继续采用“摘要秒开 + 消息原文与尝试明细并行补数”的模式；尝试明细若为空或失败，只在抽屉内部表达，不影响上层治理列表继续使用。
- `/devices` 详情抽屉当前固定复用 `DeviceDetailWorkbench`，并按“外层抽屉保留标题说明、正文使用扁平台账”的方式组织：设备场景下 `StandardDetailDrawer` 继续保留标题和说明，但详情正文不再保留内部 hero 区、右上 `已登记 / 在线 / 已激活 / 启用` 状态带或附件区。已登记设备固定按 `资产概览 / 运行概览 / 身份与部署台账 / 运行与认证台账 / 关系与建档补充 / 扩展元数据快照` 分段，未登记设备固定按 `来源档案与失败摘要 / 最近载荷` 分段；若对应字段全部为空则直接隐藏整块。详情字段值继续与编辑页保持同一常规字号和字重，不得回流说明墙、重复字段描述、重复状态摘要或第二套设备详情壳。

### 2.3 链路验证中心

- `/reporting` 当前已升级为“共享诊断状态头 + 结果复盘 / 模拟验证 / 最近记录”真页签主控台；后续只允许在现有真页签和共享状态头语法内迭代，不再回退到左右双栏工具页或说明墙。
- `/reporting` 当前固定作为接入排障起点；共享诊断状态头必须直接回答 `当前节点：链路验证` 和下一步进入哪条诊断分支，不得在子页重复 `/device-access` 的整棵决策树。
- `链路验证中心` 顶部诊断状态头应保持“单一摘要面 + 侧栏卡点/动作 + 上下文 chips”的轻量层级，不再回流三张等权状态卡、装饰性 badge 或大写 eyebrow 式首屏语气。
- 发送结果下方的“处理时间线”必须复用 `StandardTraceTimeline`，按同一品牌和共享 token 展示阶段、状态、耗时、处理类/方法与摘要；不要再为单页散写私有 timeline 卡片或 rail。
- `deviceCode` 继续作为唯一主查询入口；`productKey`、`protocolCode`、`clientId` 保持只读派生。
- JSON / XML 格式化仅服务编辑体验；只有可识别 JSON 才参与协议帧判断。
- 页面级状态摘要与发送就绪度优先通过 `MetricCard` 收口，设备查询反馈优先通过 `StandardInlineState` 表达，不回退到页面私有提示条。
- MQTT 模拟的“等待消费回流 / 已绑定 trace / 时间线已过期”状态提示，继续放在共享工作台语义里表达，不新增蓝/橙/紫私有强调条。
- `最近记录` 内部的 `全部 / 失败 / 等待中 / 成功` 状态切换继续复用 `StandardChoiceGroup`，不要在局部列表里再散写第二套筛选 pill 和 active 样式。

### 2.3.1 链路 / 日志时间线

- `/message-trace` 当前固定使用 `主链路复盘` 语义：描述、来源提示和 inline-state 只回答 `当前节点：主链路复盘` 与下一步进入异常观测、数据校验或治理页；`失败归档` 继续只作为同一路由下的异常样本模式，不得在信息架构上与链路追踪并列成第二条一级诊断支路。
- `/message-trace` 详情抽屉中的“处理时间线”继续复用 `StandardTraceTimeline`，并与 `/reporting` 保持相同字段顺序、状态色和空态文案。
- 时间线过期后的降级提示必须明确说明“时间线已过期，但 payload 对照仍可继续查看”，避免用户误判为页面加载失败、权限问题，或误以为明文/解析结果同样不可读。
- `/message-trace` 详情抽屉当前固定采用扁平工作台语法：抽屉标题行承担轻刊头，但不得再重复展示 `消息类型 / TraceId / Topic` 标签，也不得在左上角副标题位回显 topic；这些字段统一由正文中的 `消息态势与处理概况 + 链路与接入台账` 承接。正文固定保留 `消息态势与处理概况 + 链路与接入台账 + Payload 对照 + 处理时间线` 四个基础章节；当详情恢复接口返回 `protocolMetadata.templateEvidence` 时，允许在 `链路与接入台账` 与 `Payload 对照` 之间插入轻量 `协议模板证据`，但仍必须复用同一套浅色控制台台账语法，不得长出第二套诊断页头、说明墙或模板管理子页面。`链路与接入台账` 当前固定回答 `TraceId / 日志 ID / 创建时间 / 设备编码 / 路由类型 / 产品标识 / Topic`，统一改用纵向单列表格行，不再保留 `消息概览 lead-sheet`、`链路标识 / 接入上下文` 左右分组或其他列式布局。显式带 `UTC / offset` 的上报时间当前统一按 `Asia/Shanghai` 展示，无时区的历史时间串继续按原值显示；若 `reportTime` 相对 `createTime` 呈现典型 8 小时漂移，则 `/message-trace` 内必须继续按链路追踪专用纠偏口径对齐到中国时间，避免列表与详情继续显示错误时差。消息类型若命中 `status`，前端统一展示为 `状态上报`。列表中的 `Payload 摘要` 必须保持单行截断，hover 也只允许展示截断后的短预览，不得再把整段 payload 作为默认 tooltip 暴露。`Payload 对照` 必须保留 `原始 Payload / 解密后明文 / 解析结果` 三个阅读单元，但形态固定为三条纵向折叠板块：默认收起、按需展开、每条都提供复制动作，并优先消费消息日志详情恢复接口返回的数据，不得再只依赖 timeline summary，也不要回流“有内容 / 暂无内容”辅助状态 chip 干扰排查。时间线过期时，payload 对照仍需保留恢复后的明文与解析结果，只在时间线区块表达过期提示，不得把处理时间线脚注化或把其他章节挤没；详情抽屉整体配色必须继续与系统浅色控制台风格保持同系列，不再为单页保留过重的私有渐变和深色代码底板。
- `/message-trace` 的 `Payload 对照` 复制动作必须同时兼容 `navigator.clipboard.writeText` 与 `document.execCommand('copy')` 降级路径；只要当前浏览器仍具备基础文本选择能力，就不应再给出“当前环境暂不支持复制”的直接拒绝提示。
- `/message-trace` 详情抽屉中的详情恢复接口或时间线查询若发生异常，当前统一收口为抽屉内降级表达：Payload 对照继续按原始 Payload 兜底，处理时间线章节展示存储异常提示，不再弹出“系统繁忙，请稍后重试”一类红色全局错误提示打断排查流程。
- `/message-trace` 详情抽屉中的“处理时间线”当前默认折叠，首屏固定展示 `当前状态 / 处理节点 / Trace 归属 / 存储提示` 四张摘要卡，并在过期/存储异常时补充说明；用户主动展开后才渲染完整 `StandardTraceTimeline`。这条规则与 `Payload 对照默认折叠` 一起构成链路追踪详情的收纳基线，后续不得再回流为“所有 payload 与时间线首屏常显”的旧布局。

### 2.4 数据校验台

- `/file-debug` 优先收口到 `StandardWorkbenchPanel + StandardListFilterHeader + StandardInlineState` 的单入口工作台结构，不回退到页面私有散卡片表单。
- `/file-debug` 的说明与来源提示必须固定回答 `当前节点：数据与原始响应校验` 与“下一步确认是否回链路追踪台或治理页”；不得在页头复刻完整排障树或跨页功能菜单。
- 查询区、文件快照、固件聚合和原始响应继续复用共享语法，保持“查询 -> 设备校验 -> 原始响应”的固定节奏；若查询区已经通过 `StandardInlineState` 交代数量和刷新状态，就不要再补一块重复的概况卡。
- 壳层、登录页、驾驶舱和列表页结果区的高曝光渐变、弱纹理、hover 高亮与 loading 蒙层，必须优先来自全局 token 和共享样式入口；页面只允许做轻量语义化包装，不得继续散写另一套品牌 literal。

## 3. 稳定性与质量规则

1. 控制台 warning 需要区分“单次巡检采样噪音”和“可稳定复现问题”，只有复测成立的问题才进入活跃治理范围。
2. `v-permission` 不得直接挂在没有稳定原生根节点的组件上；组件型操作菜单优先在脚本层过滤 action 列表。
3. 移动端侧栏关闭前必须先移出焦点，并只在桌面折叠态使用 tooltip 包裹导航图标。
4. 浏览器自动化与视觉回归的失败证据继续保留在历史台账中，但长期有效规则要回写到本文件和 [06-前端开发与CSS规范.md](./06-前端开发与CSS规范.md)。
5. 桥接响应摘要、目标摘要、帮助搜索高亮等消费态文本继续统一走安全文本切片渲染，不回退到 `v-html` 直出。
6. 共享动作列与文本列的防回退继续由 `npm run component:guard` 守卫；当前纳管页已覆盖产品/设备、链路追踪/异常观测、风险运营/风险策略、平台治理与质量工场代表页，后续新增批次页面仍需先定义 allowlist，再迁移，不要一次性全站混改。
7. 共享 `StandardActionMenu`、`StandardRowActions` 一类动作容器必须把 `items` 视为可空输入：组件内部统一回落空数组并自动禁用触发器，不允许在 `computed` 或模板里直接读取未兜底的 `items.length`，避免权限切换、异步装配或路由首屏恢复时把整页渲染打断。
8. 已经由共享请求层弹出友好提示的 `handled` 请求异常，不再额外输出页面级 `console.error`；浏览器控制台应优先保留真正未处理的前端错误，避免业务失败提示淹没真实缺陷。
9. 页面或组合式函数在 `catch` 中处理请求异常时，必须优先复用 `@/api/request` 暴露的 `resolveRequestErrorMessage(error, fallbackMessage)`；HTTP `500` 不得再被页面私有文案覆写成“获取列表失败 / 加载失败 / 查询失败”等局部提示，统一显示“系统繁忙，请稍后重试！”。
10. 若错误已经由共享请求层标记为 `handled` 并弹出过全局提示，页面本地不得再追加第二次 `ElMessage.error`；同一条 `500` 或业务失败提示在同一次操作链路内只能出现一次。
11. `style:guard` 当前只允许在 baseline 内保留少量历史字面量余量；新增或回流的圆角 / 阴影字面量一律视为样式漂移回归，必须在提交前通过 token 化或共享样式收口恢复到基线以内。
12. 壳层 `通知中心` 的后台自动刷新必须遵守“未读统计后台轻拉取、通知分页显式打开再加载、自动请求单飞、成功 `60` 秒冷却、失败 `15` 秒退避”的统一规则；`mount / visibilitychange / focus / BroadcastChannel / storage` 这类自动场景不得再直接并发预拉 `/api/system/in-app-message/my/page`，也不得在上一条 `/my/unread-count` 未返回时继续叠发下一条请求。

## 5. 近期优先方向
