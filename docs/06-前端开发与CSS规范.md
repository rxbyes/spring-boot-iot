# 06 前端开发与 CSS 规范

更新时间：2026-03-20

## 1. 技术栈与目录

- 框架：Vue 3 + TypeScript + Vite。
- UI：Element Plus。
- 图表：ECharts。
- 关键目录：
  - `spring-boot-iot-ui/src/views`
  - `spring-boot-iot-ui/src/components`
  - `spring-boot-iot-ui/src/api`
  - `spring-boot-iot-ui/src/router`
  - `spring-boot-iot-ui/src/styles`

## 2. 前端架构基线

### 2.1 鉴权与请求

- 请求入口：`src/api/request.ts`。
- 拦截器：`src/api/interceptors.ts`。
- 行为基线：
  - 自动注入 `Authorization: Bearer <token>`。
  - 当后端返回业务 `code=401` 或 HTTP `401`，统一清理登录态并跳转 `/login`。
  - 统一错误提示与空数据兜底。

### 2.2 路由与权限

- 路由定义：`src/router/index.ts`。
- 工作台导航 schema：`src/utils/sectionWorkspaces.ts`。
- 登录页：`/login`。
- 除登录页外，业务页面默认要求鉴权。
- 浏览器刷新当前业务路由时，应先回到 SPA 壳层，再由前端优先恢复本地 token + `authContext`；后续若 `/api/auth/me` 或业务接口返回 `401`，再统一跳回 `/login`。
- 访问未授权路由或无资源链接时，前端统一清理登录态并跳转 `/login`，不保留原始 401 页面。
- 菜单与按钮权限来源：后端动态菜单与权限编码。
- 一级导航、二级导航、分组首页、路由 `title/description` 与角色默认首页，优先从共享 workspace schema 派生；不要再在壳层、分组首页和路由表里重复维护第二份菜单元数据。
- 顶部全局搜索统一使用 `Ctrl/Cmd + K` 命令面板口径；新增入口需保证能被命令面板与角色化推荐检索到。
- `AppShell` 只通过 `src/composables/useShellOrchestrator.ts` 承接壳层编排；后续若新增壳层交互、导航规则或路由副作用，应优先扩展对应 composable，并最终由 orchestrator 统一装配，不再回到壳层主文件直接拼接多组状态。
- 壳层共享类型统一沉淀到 `src/types/shell.ts`；`AppShell`、壳层子组件与 shell composables 不再互相反向引用类型，避免组件层直接依赖实现层。

## 3. 页面开发统一模式

根据现有共享能力，新增页面优先复用以下标准件：

1. `PanelCard`：卡片容器与页内分区。
2. `MetricCard`：统一概览页、报表页、工作台概况区的 KPI 卡片；当页面首屏需要压缩高度时，优先复用紧凑规格而不是另写一套私有统计卡。
3. `StandardPagination`：统一分页组件。
4. `useServerPagination`：统一分页状态与翻页行为。
5. `StandardTableToolbar`：统一列表操作条与统计信息展示。
6. `StandardTableTextColumn`：统一长文本列省略与悬浮展示完整内容。
7. `StandardInfoGrid`：统一工作台查询结果、协议预演等只读摘要卡片布局。
8. `StandardFlowRail`：统一流程导引、检查步骤、路线衔接等纵向步骤条布局。
9. `StandardInlineSectionHeader`：统一编辑器、编排页、复杂表单中的区块标题与右侧轻操作入口。
10. `StandardActionGroup`：统一工作台、表单区、编排页和卡片操作按钮组的排列、换行与常用间距。
11. 全局列表样式（`standard-list-view` 体系）。
12. 统一详情抽屉：`StandardDetailDrawer`。
13. 统一表单抽屉：`StandardFormDrawer`。
14. `StandardDrawerFooter` + `confirmAction`：统一新增/修改/删除/确认类底部按钮和二次确认交互。

禁止：
- 在同类列表页重复造分页样式和状态管理。
- 在列表页重复实现“表格工具条 + 文本溢出 tooltip + 删除确认弹窗 + 抽屉底部按钮”。
- 同时存在“本地散装分页”与“标准分页契约”的双轨实现。

## 4. CSS 与视觉 Token 规范

### 4.1 Token 文件

- `src/styles/tokens.css`：颜色、字体、圆角、阴影、间距、过渡。
- `src/styles/global.css`：全局布局、公共组件样式、列表基线。
- `src/styles/element-overrides.css`：Element Plus 主题映射。

### 4.2 主色与主题

- 当前主品牌色通过 `--brand` 及其衍生变量统一控制。
- Element Plus 主色通过 `--el-color-primary` 等变量映射到 `--brand`。

### 4.3 列表页约定

- 结构：筛选区 -> 操作栏 -> 表格区 -> 分页区。
- 当页面定位说明、全局主操作、筛选条、统计栏和列表都服务同一台账时，优先收口到同一张主卡片，避免再拆成“说明卡 + 列表卡”的双卡结构造成首屏割裂。
- 当页面只保留一个全局主操作且台账卡不再展示独立标题时，可把主按钮锚定到页面主标题右侧；若该主操作直接服务当前筛选查询（如新增、批量导入），可并入查询行并放在“查询/重置”之后，同一操作不得在主标题区和查询行重复出现。
- 当筛选只服务当前台账且固定在 3 个以内高频条件时，可直接内嵌到台账卡片头部与统计栏之间；移动后应删除独立“筛选条件”卡和字段 label，仅保留 placeholder 与查询/重置按钮，避免首屏形成额外纵向层级。
- 手机端优先保证筛选项可自适应展开，不要用固定宽度把输入框锁死；应通过响应式 grid/flex 和统一 gap 保留字段间距，在窄屏时自然降为两列或一列。
- 已提交的筛选条件必须提供可见反馈，优先通过标签行展示当前生效条件，并支持单项移除和一键清空；标签状态应反映“最近一次已执行查询”，而不是用户尚未提交的临时输入。
- 产品定义页的工具栏统计应坚持产品口径；`在线设备` 等资产运行指标不应再作为产品页总览主指标。
- 当桌面端表格在窄屏下可读性明显下降时，应补充卡片列表模式，并保留基础操作和选中能力，不要只依赖横向滚动。
- 同一批列表数据若同时提供“桌面表格 + 移动卡片”两种视图，两侧必须复用同一份选中态、批量操作状态和空状态口径；禁止桌面与移动各自维护独立选择结果，导致批量删除、导出选中等行为不一致。
- 空状态至少区分“当前还没有任何数据”与“筛选后没有匹配结果”两类，前者优先引导新增，后者优先引导调整或清空筛选，不得继续共用一条笼统的无结果文案。
- 列表页的查询/刷新 `loading` 若只作用于卡片内结果区，应优先使用浅色局部蒙层并保留底板与圆角，避免直接沿用全局深色遮罩造成“列表瞬间黑屏”的跳变感；深色遮罩只保留给全屏或强打断场景。
- 高频台账页在冷启动、切换筛选后未命中缓存或跨页返回未恢复结果时，结果区应优先提供与最终列表结构接近的骨架屏，而不是只留一个空白容器加 spinner；骨架需同时兼顾桌面表格和移动卡片两种版式节奏。
- 共享壳层统一采用“顶部一级导航固定 + 左侧二级菜单固定 + 右侧内容区独立滚动”布局；当正文超过一屏时，只允许 `AppShell` 的 `cloud-content` 承接纵向滚动，不再让 `body/window` 与左侧菜单一起滚动。
- 页面根容器应继续复用 `.page-stack`、`.standard-list-view`、`.ops-workbench`、`.section-landing`、`.report-analysis-view` 等共享基线，并保持 `min-width: 0`、不声明页面级 `100vh`/整页 `overflow-y: auto`，避免共享壳层切换为内容区滚动后再次出现局部外溢或双滚动。
- 五个一级工作台下的功能页顶部导航统一为仅保留阿里云式面包屑的极简结构；不得再重复渲染最近访问标签栏、主标题、同组横向切换，以及 `当前分区`、`页面类型`、`可见菜单`、`返回概览` 这类说明性文案。
- 最近访问标签栏已下线，禁止继续新增 `trackTab`、`visitedTabs`、`TabsView` 一类旧机制；功能页定位统一依赖面包屑、左侧菜单和命令面板。
- 页面说明文案只在分组首页、空态页或确有必要的引导场景保留；普通功能页默认不再展示“列表/筛选/详情抽屉”一类低信息密度说明。
- 当筛选项超过 3 个时，优先采用“常用条件 + 更多条件”的可展开筛选结构；收起后仍需通过已启用条件数、状态提示或标签，让用户感知隐藏筛选仍在生效。
- 产品定义中心、设备资产中心等高频台账页的查询行默认顺序为“常用筛选项 -> 查询 -> 重置 -> 主操作按钮（如新增、批量导入）”；后续同类页面重构优先复用该顺序。
- 查询行中的主操作按钮若属于当前台账主入口（如新增产品、新增设备），默认与“查询”保持同一主色口径，避免使用低对比度的 `primary plain` 样式导致文案可读性下降。
- 桌面表格“操作”列若动作超过 2 项，应按“高频动作直出 + 低频动作进入更多下拉”的口径收口，避免横向平铺过多按钮挤压列表可读性。
- 当同类查询头结构在 3 个及以上列表页重复出现时，下一轮应统一抽离为公共组件（建议命名 `ListFilterHeader`），禁止继续页面内复制模板和样式。
- 列表页若要支持刷新、回退或跨页面跳转后恢复筛选，应把高频条件与分页状态同步到路由 query；至少保持 `productName / nodeType / status / pageNum / pageSize` 这类高频台账状态可恢复，不要只在组件内留瞬时状态。
- 路由 query 恢复不得只覆盖文本输入；枚举型筛选与分页参数也必须同步恢复。以设备台账页为例，应继续覆盖 `deviceId / productKey / deviceCode / deviceName / onlineStatus / activateStatus / deviceStatus / pageNum / pageSize`，避免回退后只恢复一半状态。
- 列表页在快速切筛选、分页、刷新或路由恢复时，应主动取消已过期请求，避免旧响应回写覆盖新结果；不要只依赖“后发请求大概率更晚回来”的偶然顺序。
- 默认 10 条分页的高频台账页，若存在前后页往返、刷新后恢复或详情返回列表的高频操作，应优先按“筛选条件 + pageNum + pageSize”缓存当前分页结果；命中缓存时先秒开当前页，再静默校验并按需预取下一页，显式刷新或增删改成功后再失效并重建当前页缓存。
- 若页面存在“跳去详情关联页再返回原列表”的高频链路，分页结果缓存不应只停留在组件内存；应按短 TTL 落到 `sessionStorage` 一类会话级存储，并在页面重挂载时先恢复最近缓存，再静默校验是否过期。
- 若列表页的首屏只需要台账数据，新增/编辑/替换抽屉才依赖额外选项数据，则不要在页面挂载时预加载这些抽屉专用下拉数据；应改为抽屉首次打开时再懒加载，避免列表首屏被无关请求拖慢。
- 列表页的新增、编辑、删除提交成功后，应优先本地更新当前页可见结果，再后台静默刷新校正排序和分页，不要每次都阻塞等待整页重刷后才反馈结果。
- 若列表页已接入分页缓存，增删改成功后的“后台静默刷新”不得被同一时刻刚写入的新鲜缓存短路；需要继续复用当前页本地结果秒开，但必须显式发起一次静默校验请求。
- 当列表页已展示可用结果，只是在后台静默校验最新数据时，应优先在结果区上方展示轻量页内提示；静默校验失败时默认继续保留当前结果并改为页内错误提示，不要直接用打断式 toast 覆盖用户操作节奏。
- 若 Hero 区指标与台账统计栏表达的是同一组启用/停用汇总信息，应优先删除 Hero 重复指标，不要在普通列表页重复堆叠同义统计卡。
- 长文本默认通过 `StandardTableTextColumn` 单行省略，悬浮显示完整内容。
- 表格操作和导出行为保持统一入口。
- 概览页、报表页、工作台概况区的 KPI 卡优先通过 `MetricCard` 统一数值层级、徽标和卡片密度；普通列表页首屏如需降低视觉高度，优先切换到紧凑规格，不再新增页面私有 mini-card。
- 查询结果、协议预演、只读摘要卡优先通过 `StandardInfoGrid` 统一信息密度、对齐和长值展示策略。
- 流程导引、发送后检查、路线桥接等步骤型内容优先通过 `StandardFlowRail` 统一索引、标题和说明布局；不再在页面内重复手写 `flow-rail` 模板。
- 工作台、表单和编排页中的按钮组优先通过 `StandardActionGroup` 统一，不再继续保留 `button-row + 内联 style`、`action-row + 局部样式` 的双轨实现。
- 场景/步骤编排编辑器中的区块头、场景卡头、步骤卡头等“标题 + 轻操作”结构优先通过 `StandardInlineSectionHeader` 统一；不再新增页内独立的 `inline-block__header`、`scenario-card__header`、`step-editor__header` 一类样式实现。
- 产品、设备等台账详情抽屉应采用“顶部概览 + 分组字段卡 + 治理提示”结构；禁止回退为只靠大字号文本纵向堆叠的无层次详情页。
- 台账详情抽屉首屏应至少有一个主信息焦点区，用于突出当前状态、治理阶段或核心身份；不要把所有摘要卡做成同权重平铺，导致阅读焦点缺失。
- 同一属性在详情抽屉中应尽量只保留一个权威展示位置；不要在标题 tags、摘要卡、字段卡中重复出现 `状态 / 节点类型 / 协议编码 / 数据格式` 等同一信息。
- 产品类详情至少拆分为“产品汇总 / 接入契约 / 产品档案 / 维护治理”四个层次；汇总数据、接入语义和建档档案不得混写在同一组字段卡中。
- 面向业务侧的详情抽屉标题、区块标题、标签和说明文案优先保持中文一致性；除确有必要的协议名、技术缩写外，避免无意义的中英混排或偏研发黑话。
- 接入契约类区块优先采用单列纵向卡片，方便逐项核对协议、节点角色和数据格式；长编号、长 Key 这类字段应提供完整 hover 文本，并尽量放入更宽的档案区承接。
- 维护与治理区块优先拆成“当前建议 / 固定规则 / 变更前确认”等分层卡片，不要把多类说明挤在两块笼统大段文本里。
- 详情抽屉内同层级卡片的字号、行距、内边距和区块间距应保持统一节奏；正文说明优先使用通俗业务语言，避免因文案过长或术语过重进一步放大页面松散感。
- 详情抽屉若接口补数慢于列表摘要，应优先使用列表行已有字段秒开抽屉，再异步补齐完整详情，并以轻量状态提示说明“摘要已展示、完整详情补充中”；不要一律先清空抽屉再让用户等整屏 loading。
- 同一详情抽屉若频繁重复打开相同对象，应优先按 `id + updateTime` 复用本地缓存的完整详情；只有列表行版本时间变化或缓存信息不足时才重新请求，避免重复点开同一条记录仍重复打接口。
- 若详情/编辑抽屉存在“跨路由返回后继续重复打开同一对象”的高频链路，完整详情缓存不应只停留在组件内存；应按短 TTL 和有限条数落到 `sessionStorage` 一类会话级存储，并继续以列表行 `updateTime` 作为补数校验依据。
- 编辑抽屉若需要秒开，应优先使用列表行摘要或本地缓存预填表单，再后台补齐最新详情；若用户已经开始编辑，后台返回的新数据不得自动覆盖当前表单输入。
- 若详情抽屉同层级信息卡仍出现“有的很高、有的很短”的失衡感，应优先统一 label / value / helper 的字号梯度，并把长说明拆到独立宽卡承接，不要继续把长文塞在档案卡里把整行节奏撑散。
- 当详情抽屉进入商业化细抛光阶段时，应优先把区块说明收口为单行短句，并压缩抽屉页眉到首个区块、区块到区块、卡片到卡片之间的纵向距离；若共享 `info-chip / info-grid` 基线引入额外高度，应在页面内显式覆盖，避免“卡片结构正确但卡与卡之间显得发飘”。
- 当页面已完成结构治理但视觉仍被评价为“同质、发灰、像堆卡片”时，可在不脱离品牌 token 的前提下引入“分区大板块 + 版面主次 + 区块差异底纹”的重设计方案；禁止继续只通过机械缩小 `gap` 反复微调。
- 共享组件接管后，必须在同一变更中同步删除页面内对应的旧结构 class 与伴生 scoped CSS；禁止保留失效的 `info-grid / info-chip / button-row / flow-rail` 等遗留样式造成“模板已统一、CSS 仍分叉”。
- 台账页所需的关联统计（如关联设备数、在线数、最近上报）应直接消费分页接口返回的聚合字段；若后端仍通过加载整页关联明细后在应用层汇总导致列表变慢，必须同步改成数据库聚合，前端不要再追加串行补查。

### 4.4 概览页与抽屉配色约定

- 分组概览、风险处置工作台、详情抽屉、表单抽屉与确认弹窗统一复用 `--brand`、`--accent`、`--panel-border` 等全局 token。
- 不再在概览页或工作台局部写死另一套蓝色主题；局部强调色只能作为 `--accent` 的补充，而不是替换整页品牌色。
- 如需局部例外主题，必须在 `docs/15-frontend-optimization-plan.md` 记录原因、影响范围和回退策略。

- 涉及文案修改时必须自检中文可读性。

## 6. 回归检查清单（前端改动后）

1. 登录态恢复、401 回退流程是否正常。
2. 菜单与按钮权限是否按角色生效。
3. 列表分页是否统一使用 `StandardPagination` + `useServerPagination`。
4. 列表操作条和长文本列是否使用 `StandardTableToolbar` + `StandardTableTextColumn`。
5. 摘要卡、流程导引和按钮组是否分别使用 `StandardInfoGrid`、`StandardFlowRail`、`StandardActionGroup`。
6. 详情/表单交互是否使用 `StandardDetailDrawer`、`StandardFormDrawer`、`StandardDrawerFooter` 和 `confirmAction`。
7. 视觉 Token 是否复用，无页面级“临时色值漂移”。
8. 共享组件替换后，页面私有结构类和失效 scoped CSS 是否已同步清理。
9. 功能页顶部是否仍残留最近访问标签栏、主标题、同组横向导航或“当前分区 / 页面类型 / 可见菜单 / 返回概览”等重复说明文案；若有，必须在交付前删除。
10. 页面超长时是否只有右侧内容区滚动，左侧二级菜单与顶部一级导航是否保持固定；若仍出现整页滚动，必须优先修正共享壳层或页面内异常高度样式。
11. 当筛选区存在“更多条件”折叠层时，默认态、展开态和已命中隐藏条件时的提示是否一致可感知；不得出现“高级条件已生效但页面无提示”的状态。
