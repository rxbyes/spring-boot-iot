## 1. 当前基线

### 1.1 已确认事实

- 当前仓库与初始化脚本中存在 `sys_tenant`，大量核心业务表保留 `tenant_id` 字段。
- `sql/init-data.sql` 当前只提供默认租户样例：`tenant_code=default`。
- 当前真实环境验收基线仍以单租户样例为主，默认按租户 `1 / default` 组织演示数据、账号、角色和业务记录。
- 菜单、按钮和角色授权已形成可用闭环，但这只解决“能不能看到 / 能不能操作”，不等于“能看到哪些数据”已经完整建模。
- 2026-04-02 起，身份模型已补齐第一轮治理基线：`sys_user.org_id` 作为登录用户主机构，`sys_role.data_scope_type` 作为角色声明的数据范围，`/api/auth/me` 统一返回 tenant/org/login/security/data-scope 摘要供壳层账号中心、用户管理和角色管理共用。

### 1.2 当前不应误读的点

- 当前项目不能因为表里有 `tenant_id` 就被视为“完整多租户产品已交付”。
- 当前项目也不能因为只有默认租户样例，就被视为“完全没有租户隔离设计”。
- 更准确的口径是：
  - 数据表层已经为租户隔离预留了主键外的逻辑边界。
  - 真实环境验收当前只验证了默认租户基线。
  - 完整租户生命周期、跨租户共享、租户级运维流程仍未形成单独交付闭环。

## 2. 权限模型分层

### 2.1 第一层：租户隔离

- `tenant_id` 是当前最外层的数据隔离边界。
- 同一类主数据、业务数据、日志数据原则上都应先落在租户范围内，再谈角色和组织范围。
- 当前文档口径只覆盖“共享库 + 逻辑隔离”的基线，不承诺“独库独实例”一类物理隔离方案。

### 2.2 第二层：角色、菜单与按钮权限

- 用户与角色：`sys_user`、`sys_role`、`sys_user_role`
- 角色与菜单：`sys_role_menu`
- 登录后权限上下文：`/api/auth/login`、`/api/auth/me`

当前作用：
- 决定用户可进入哪些工作台、页面和按钮动作。
- 决定管理页是否展示新增、编辑、删除、导出、测试通知等操作。
- 通过 `sys_role.data_scope_type` 为当前用户声明 `ALL / TENANT / ORG_AND_CHILDREN / ORG / SELF` 五档数据范围，并在 `/api/auth/me` 中汇总最高优先级摘要；2026-04-03 起，这套声明已先落到 `用户管理 / 组织管理 / 角色权限 / 菜单管理 / 风险点管理 / 区域管理 / 字典管理 / 通知渠道管理 / 帮助文档管理 / 审计日志管理 / 设备资产中心` 第一批查询侧执行，其中角色、菜单、区域、字典、通知渠道、帮助文档、审计日志当前先按租户侧收口，设备资产中心则已进入 `tenant + org` 治理。
- `2026-04-10` 起，治理审批固定复核人策略已独立建模为 `sys_governance_approval_policy`：表结构预留 `GLOBAL / TENANT` 两级 scope，当前初始化基线只启用 `tenant_id=0` 的全局策略，为产品合同发布 / 回滚自动解析固定复核人。该表属于“治理配置真相”，不直接替代审批主单或一般业务数据的 `tenant_id` 隔离语义。

当前边界：
- 菜单 / 按钮权限不直接替代数据范围控制。
- 即使用户拥有页面访问权限，数据层仍应继续受租户、组织、区域或业务对象范围约束。
- 当前这套数据范围声明不再只停留在身份上下文展示，但也不等于所有业务查询都已经自动按该范围过滤；当前仅完成 `用户管理 / 组织管理 / 角色权限 / 菜单管理 / 风险点管理 / 区域管理 / 字典管理 / 通知渠道管理 / 帮助文档管理 / 审计日志管理 / 设备资产中心` 第一批服务层收口，其余模块仍待继续推广。

### 2.3 第三层：组织、区域与业务对象范围

当前系统已存在以下业务范围维度：

- 组织：`sys_organization`
- 区域：`sys_region`
- 风险对象：`risk_point`
- 设备资产：`iot_device`
- 产品主数据：`iot_product`

当前可确认口径：
- 工作台、菜单和帮助中心会基于角色权限过滤入口。
- 组织、区域、风险点和设备等业务对象已形成独立主数据，可以作为后续数据范围控制的承载体。
- `sys_user.org_id` 当前表示登录用户主机构，用户管理页以此维护“用户 -> 主机构”关系；当前尚未引入用户-多机构关系表。
- 用户管理、角色管理与壳层账号中心当前统一使用同一身份上下文：用户侧维护主机构/角色绑定，角色侧维护 `dataScopeType`，账号中心只消费当前登录用户资料，不再另起第二套静态模型。
- `risk_point` 当前已显式落库 `org_id / org_name`，风险点新增与编辑都以组织树所选组织作为责任归属；2026-04-02 起该字段已作为第一批组织范围权限入口，风险点列表、分页、详情、删除、绑定/解绑与绑定设备查询会按当前登录人的租户、主机构和数据范围执行过滤或越权校验。
- `iot_device` 当前也已显式落库 `org_id / org_name`，设备新增、批量导入、编辑与设备更换生成新设备时统一按当前登录人的主机构自动回填；设备详情、分页、设备选项、属性快照、消息日志、链路追踪与失败归档当前都已改为按 `tenant + org + dataScopeType` 收口。
- `sys_menu`、`sys_region`、`sys_dict`、`sys_notification_channel`、`sys_help_document`、`sys_audit_log` 当前已作为同一批系统治理租户侧主数据收口：列表/分页/详情与新增/编辑/删除统一先按当前登录人的 `tenantId` 过滤或越权校验；`/api/auth/me`、`/api/menu/tree` 与角色授权树当前也只返回当前租户的启用菜单；帮助文档消费端 `access/*` 会在租户过滤后继续叠加角色与页面路径权限。

当前仍未完全证实的口径：
- `【待确认 DOC-Q-003】` 共享环境是否已存在跨租户真实数据混跑，仍需基于真实环境数据库与查询行为进一步核查。
- `【待确认 DOC-Q-008】` 是否所有核心查询都已经在服务层或 Mapper 层严格按“租户 + 角色数据范围 + 主机构”统一过滤，并形成可复用的数据权限表达，而不仅是页面筛选条件。

### 2.4 2026-04-03 第一批查询侧落地口径

- 当前后端使用统一的 `DataPermissionContext` 收口第一批数据权限上下文，核心字段为 `userId / tenantId / orgId / dataScopeType / superAdmin`，来源仍是 `sys_user`、`sys_user_role`、`sys_role` 与组织树。
- 当前第一批覆盖范围明确限定为 `用户管理 / 组织管理 / 角色权限 / 菜单管理 / 风险点管理 / 区域管理 / 字典管理 / 通知渠道管理 / 帮助文档管理 / 审计日志管理 / 设备资产中心（tenant + org）`，采用服务层显式传入当前登录用户主键的方式执行过滤或越权校验，不额外引入全局 SQL 拦截器。
- 当前作用语义如下：
  - `ALL / TENANT`：当前租户内可见全部组织、用户和风险点。
  - `ORG_AND_CHILDREN`：按当前主机构及其组织子树收口组织、用户和风险点。
  - `ORG`：按当前主机构节点收口组织和用户，风险点只保留该机构下记录。
  - `SELF`：用户治理只允许查看或操作当前登录用户本人；组织治理只暴露当前主机构节点；风险点治理只暴露 `responsibleUser=当前用户` 或 `createBy=当前用户` 的记录。
- 设备治理当前复用同一套声明，但业务语义单独固定为：
  - `ALL / TENANT`：当前租户内可见全部设备。
  - `ORG_AND_CHILDREN`：当前主机构及子树下设备。
  - `ORG`：当前主机构下设备。
  - `SELF`：设备域当前显式等价收敛为 `ORG`，不伪造“责任人用户”维度过滤。
- 角色治理当前作为同一批身份链路的租户侧补充：
  - 角色列表、分页、详情、新增、编辑、删除固定先按当前登录用户 `tenantId` 收口。
  - `GET /api/role/user/{userId}` 除租户一致外，还会先校验目标用户是否位于当前登录人的可访问范围内。
  - 当前未把 `dataScopeType` 本身反向作用到“角色是否可见”的组织范围过滤；角色档案仍视为租户级治理主数据。
- 菜单治理当前作为同一批身份链路的租户侧补充：
  - `GET /api/menu/tree`、`/list`、`/page`、`/{id}` 与 `POST /api/menu/add`、`PUT /api/menu/update`、`DELETE /api/menu/{id}` 固定先按当前登录用户 `tenantId` 收口。
  - `sys_menu.menu_code` 唯一性、父子菜单校验、详情读取、更新删除以及兼容写库逻辑当前都改为租户侧生效，不再把 `tenant_id` 写死为默认租户。
  - `/api/auth/me` 与角色授权树当前只消费当前租户的启用菜单，避免跨租户菜单混入导航树或授权树。
- 区域、字典、通知渠道、帮助文档、审计日志当前作为同一批系统治理租户侧扩展：
  - 列表、分页、详情统一先按当前登录用户 `tenantId` 收口。
  - 新增、编辑写入不再信任请求体中的 `tenantId`，统一以当前登录人上下文回填。
  - 删除、测试通知与帮助文档访问会先校验目标记录是否属于当前租户；帮助文档消费端 `access/list|page|{id}` 会在租户过滤之后，再叠加角色与页面路径权限。
- 设备资产中心当前作为同一批 IoT 业务 `tenant + org` 扩展：
  - `GET /api/device/{id}`、`GET /api/device/code/{deviceCode}`、`GET /api/device/page`、`GET /api/device/list`、`GET /api/device/{deviceId}/metrics` 与 `GET /api/device/{deviceCode}/properties` 当前都会先按当前登录用户 `tenantId` 过滤，再按 `org_id + dataScopeType` 继续收口或做越权校验。
  - `GET /api/device/{deviceCode}/message-logs`、`GET /api/device/message-trace/page`、`GET /api/device/message-trace/stats`、`GET /api/device/access-error/page`、`GET /api/device/access-error/stats` 与 `GET /api/device/access-error/{id}` 当前也统一透传当前登录用户主键；服务层会先通过设备主档校验当前登录人的机构可见范围，再读取消息日志、追踪统计和失败归档查询，组织受限场景下 `access-error/stats` 不再回读全局 `invalid-report` 聚合桶。
  - `POST /api/device/add`、`POST /api/device/batch-add`、`PUT /api/device/{id}`、`POST /api/device/{id}/replace`、`DELETE /api/device/{id}`、`POST /api/device/batch-delete` 当前也统一透传当前登录用户主键，设备建档写入与替换会按当前租户校验产品、父设备和目标设备归属，并自动把当前登录人的主机构回填到 `iot_device.org_id / org_name`，不再允许前端手工指定设备归属机构。
  - `GET /api/device/page` 合并未登记上报名单时，`iot_device_invalid_report_state`、`iot_device_access_error_log` 与 `iot_device_message_log(messageType=dispatch_failed)` 的只读补档视图当前在 `ALL / TENANT` 下继续按 `tenant_id` 收口；组织受限账号不再暴露这些无法稳定归属机构的未登记记录。
  - 风险点设备候选列表当前已从泛化 `/api/device/list` 收口为 `GET /api/risk-point/bindable-devices/{riskPointId}`；返回结果只保留当前用户可见范围内、与风险点同机构、且未被其他风险点占用的设备，当前风险点已绑定设备允许继续回显。
- 当前写入侧同步补齐了首批越权校验：
  - 用户新增、编辑会校验目标 `orgId` 是否位于当前登录人的可写组织范围内。
  - 风险点新增、编辑会校验目标 `orgId` 是否位于当前登录人的可写组织范围内。
  - 风险点删除、绑定、解绑、绑定设备查询会先校验目标风险点是否位于当前登录人的数据范围内。
- 当前设备侧口径已升级为：
  - `iot_device.org_id / org_name` 已成为设备域正式组织事实来源，设备查询和诊断链路不再停留在租户级隔离。
  - 风险点绑定会额外校验设备组织与风险点组织一致，并拒绝“一设备多风险点”占用；这条规则当前只推广到风险对象中心，尚未同步扩到告警、事件、报表等更多业务域。
- 当前组织树已拆分为“查询口径”和“写入口径”两条接口：`GET /api/organization/tree` 继续服务查询展示，`GET /api/organization/writable-tree` 专门服务用户/风险点表单中的组织选择。其中 `ORG` 在写入口径下允许选择当前主机构及其子树，`SELF` 仍只允许当前主机构节点；前端必须把 `orgId / roleIds` 当作字符串形态的 Long ID 处理，不得再转成 `Number`。
- 当前非目标保持不变：虽然设备资产中心及首批设备诊断接口已完成 `tenant + org` 治理，但同一套表达尚未继续推广到告警、事件、报表等更多业务查询，也尚未引入用户-多机构关系表或跨租户共享模型。

| 系统内容 | `sys_in_app_message`、`sys_help_document` | `tenant_id`、目标范围字段 | 系统内容消费范围仍应先服从租户边界 |

## 4. 当前设计约束

1. 所有新增表、索引、接口和验收脚本都应继续显式考虑 `tenant_id`，不能把默认租户样例误写成“无租户模型”。
2. 租户隔离、角色授权、组织范围是三层不同问题，文档和代码都不应混写成同一条规则。
3. 任何跨组织、跨区域、跨风险对象的数据可见性讨论，都必须先说明它属于：
   - 租户内共享
   - 角色授权
   - 组织树范围
   - 临时查询筛选
4. 若未来要把多租户作为正式交付能力，需要同时更新 `02 / 04 / 13 / 21`，而不是只改某一处表结构或菜单说明。
5. 当前基线只落“主机构 + 角色数据范围声明”，不额外引入用户-多机构关系表；若未来扩成多机构成员模型，必须同步更新 `04 / 13 / 21` 与用户治理页面合同。

## 5. 当前验收基线
