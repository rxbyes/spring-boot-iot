## 1. 当前基线

### 1.1 已确认事实

- 当前仓库与初始化脚本中存在 `sys_tenant`，大量核心业务表保留 `tenant_id` 字段。
- `sql/init-data.sql` 当前只提供默认租户样例：`tenant_code=default`。
- 当前真实环境验收基线仍以单租户样例为主，默认按租户 `1 / default` 组织演示数据、账号、角色和业务记录。
- 菜单、按钮和角色授权已形成可用闭环，但这只解决“能不能看到 / 能不能操作”，不等于“能看到哪些数据”已经完整建模。

### 1.2 当前不应误读的点

- 当前项目不能因为表里有 `tenant_id` 就被视为“完整多租户产品已交付”。
- 当前项目也不能因为只有默认租户样例，就被视为“完全没有租户隔离设计”。
- 更准确的口径是：
  - 数据表层已经为租户隔离预留了主键外的逻辑边界。
  - 真实环境验收当前只验证了默认租户基线。
  - 完整租户生命周期、跨租户共享、租户级运维流程仍未形成单独交付闭环。

## 2. 权限模型分层

### 2.1 第一层：租户隔离

- `tenant_id` 是当前最外层的数据隔离边界。
- 同一类主数据、业务数据、日志数据原则上都应先落在租户范围内，再谈角色和组织范围。
- 当前文档口径只覆盖“共享库 + 逻辑隔离”的基线，不承诺“独库独实例”一类物理隔离方案。

### 2.2 第二层：角色、菜单与按钮权限

- 用户与角色：`sys_user`、`sys_role`、`sys_user_role`
- 角色与菜单：`sys_role_menu`
- 登录后权限上下文：`/api/auth/login`、`/api/auth/me`

当前作用：
- 决定用户可进入哪些工作台、页面和按钮动作。
- 决定管理页是否展示新增、编辑、删除、导出、测试通知等操作。

当前边界：
- 菜单 / 按钮权限不直接替代数据范围控制。
- 即使用户拥有页面访问权限，数据层仍应继续受租户、组织、区域或业务对象范围约束。

### 2.3 第三层：组织、区域与业务对象范围

当前系统已存在以下业务范围维度：

- 组织：`sys_organization`
- 区域：`sys_region`
- 风险对象：`risk_point`
- 设备资产：`iot_device`
- 产品主数据：`iot_product`

当前可确认口径：
- 工作台、菜单和帮助中心会基于角色权限过滤入口。
- 组织、区域、风险点和设备等业务对象已形成独立主数据，可以作为后续数据范围控制的承载体。
- `risk_point` 当前已显式落库 `org_id / org_name`，风险点新增与编辑都以组织树所选组织作为责任归属；该字段用于后续组织范围权限收口，本轮尚未在查询侧按登录组织自动过滤。

当前仍未完全证实的口径：
- `【待确认 DOC-Q-003】` 共享环境是否已存在跨租户真实数据混跑，仍需基于真实环境数据库与查询行为进一步核查。
- `【待确认 DOC-Q-008】` 是否所有核心查询都已经在服务层或 Mapper 层严格按“租户 + 组织范围”双维度统一过滤，并形成可复用的数据权限表达，而不仅是页面筛选条件。

## 3. 关键表与范围字段索引

| 主题 | 关键表 | 关键范围字段 | 当前说明 |
|---|---|---|---|
| 租户主数据 | `sys_tenant` | `id`、`tenant_code` | 当前仅确认默认租户样例进入初始化数据 |
| 用户与角色 | `sys_user`、`sys_role`、`sys_user_role` | `tenant_id`、`role_id`、`user_id` | 承载账号归属与角色绑定 |
| 菜单与按钮权限 | `sys_menu`、`sys_role_menu` | `tenant_id`、`menu_id`、`role_id` | 承载页面入口与按钮授权 |
| 组织与区域 | `sys_organization`、`sys_region` | `tenant_id`、层级字段 | 承载组织树和地理范围 |
| 产品与设备 | `iot_product`、`iot_device` | `tenant_id`、`product_id` | 设备与产品身份均在租户内唯一管理 |
| 消息与属性 | `iot_message_log`（兼容视图；物理表 `iot_device_message_log`）、`iot_device_property` | `tenant_id`、`device_id` | 运行数据需跟随设备和租户归属 |
| 风险处置 | `risk_point`、`risk_point_device`、`iot_alarm_record`、`iot_event_record` | `tenant_id`、`org_id`、业务对象主键 | 风险对象与事件留痕不应跨租户混用，`risk_point.org_id` 作为后续组织范围过滤入口 |
| 系统内容 | `sys_in_app_message`、`sys_help_document` | `tenant_id`、目标范围字段 | 系统内容消费范围仍应先服从租户边界 |

## 4. 当前设计约束

1. 所有新增表、索引、接口和验收脚本都应继续显式考虑 `tenant_id`，不能把默认租户样例误写成“无租户模型”。
2. 租户隔离、角色授权、组织范围是三层不同问题，文档和代码都不应混写成同一条规则。
3. 任何跨组织、跨区域、跨风险对象的数据可见性讨论，都必须先说明它属于：
   - 租户内共享
   - 角色授权
   - 组织树范围
   - 临时查询筛选
4. 若未来要把多租户作为正式交付能力，需要同时更新 `02 / 04 / 13 / 21`，而不是只改某一处表结构或菜单说明。

## 5. 当前验收基线

- 当前真实环境验收仍按默认租户样例执行，不单独验跨租户切换。
- 测试数据建议继续使用 `accept-`、`ACCEPT_` 前缀，并保持在默认租户中完成一轮完整验证，避免与历史演示数据混淆。
- 若任务明确涉及“租户切换、跨租户隔离、组织范围收敛、跨组织共享”，必须把结果区分为：
  - 已由代码和环境事实证实
  - 仅根据表结构推断
  - 仍待产品 / 环境 / 文档确认
