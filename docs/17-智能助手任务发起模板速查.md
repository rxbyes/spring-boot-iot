# 17 智能助手任务发起模板速查

> 文档定位：`spring-boot-iot` 项目内可直接复制的智能助手任务发起模板速查页。
> 适用角色：研发、测试、文档维护者、智能助手协作人员。
> 权威级别：协作速查入口，负责“直接复制什么话给助手”。
> 上游来源：`AGENTS.md`、[09-GPT接管提示模板.md](./09-GPT接管提示模板.md)、[10-智能助手技能与任务选型指南.md](./10-智能助手技能与任务选型指南.md)、[skills/ai-task-intake/SKILL.md](./skills/ai-task-intake/SKILL.md)。
> 下游消费：日常发任务、跨模块协作、复杂需求接手、排障、回归、文档治理与技能建设。
> 变更触发条件：高频任务类型变化、协作规则变化、最小阅读集变化、相关技能入口变化。
> 更新时间：2026-03-24

职责边界：
- 本文只提供“直接复制给助手”的提示词模板，不重复解释技能定义。
- 要先判断“该用哪个技能”，看 [10-智能助手技能与任务选型指南.md](./10-智能助手技能与任务选型指南.md)。
- 要看完整任务卡、证据附片和长模板母版，回到 [09-GPT接管提示模板.md](./09-GPT接管提示模板.md)。
- 要做更短的任务收口，优先看 [skills/ai-task-intake/SKILL.md](./skills/ai-task-intake/SKILL.md)。

## 1. 使用规则

1. 先选最接近的任务场景，再复制对应模板，不要先写一大段自然语言。
2. 如果你还没想清楚该走设计、计划、调试还是执行，先看 [10-智能助手技能与任务选型指南.md](./10-智能助手技能与任务选型指南.md)。
3. 如果你只想先让助手分析，不要在模板里写“直接开始修改”。
4. 如果已有日志、截图、接口响应、SQL 或验收证据，把它们追加到第 `4` 节附片模板后，不要塞进正文。
5. 任何涉及真实环境验收的任务，都必须明确基线是 `spring-boot-iot-admin/src/main/resources/application-dev.yml`。
6. 模板是起点，不是上限；如果任务超出模板承载能力，再升级到 [09-GPT接管提示模板.md](./09-GPT接管提示模板.md) 第 `6` 节或第 `9` 节。

## 2. 十条日常短提示词

### 2.1 先分析，不改代码

```md
请接手 `spring-boot-iot` 的分析任务。

我要分析的是：【问题 / 页面 / 接口 / 模块】。
当前现象是：【一句话】。
影响范围大致是：【页面 / 接口 / 模块 / 表 / 环境】。

现在先不要改代码，请先输出：
1. 当前理解
2. 最可能根因
3. 影响范围
4. 建议补读文档
5. 建议实施方案

请先按最小阅读集建立认知，再只补读和本任务直接相关的文档，不要全量扫描 `docs/`。
```

### 2.2 新增后端业务能力

```md
请接手 `spring-boot-iot` 的新增后端功能任务。

目标能力是：【新增什么】。
业务背景是：【为什么要做】。
涉及范围是：【模块 / 接口 / 表 / 文档 / 验收】。
验收标准是：【怎么才算完成】。
非目标是：【这次不要顺手改什么】。

请先总结任务、列出受影响模块、给出实现计划和假设，然后开始实施。
如果行为、接口、表结构或验收口径变化，必须同步更新对应文档。
```

### 2.3 新增前端页面或页面优化

```md
请接手 `spring-boot-iot-ui` 的前端页面任务。

页面 / 路由是：【填写】。
任务目标是：【新增页面 / 优化交互 / 调整信息结构 / 收口样式】。
当前问题或背景是：【填写】。
禁止改动的是：【API / 业务流程 / 信息结构 / 其他边界】。
验收方式是：【页面状态 / 交互动作 / 截图点 / 数据回显】。

请优先复用共享模式：`PanelCard`、`StandardPagination`、`StandardTableToolbar`、`StandardTableTextColumn`、`StandardFormDrawer`、`StandardDetailDrawer`、`confirmAction`。
请保持 UTF-8，可读可检；若暴露了样式治理问题，结束前同步更新 `docs/15-前端优化与治理计划.md`。
```

### 2.4 错误修复 / 回归问题

```md
请接手 `spring-boot-iot` 的错误修复任务。

问题现象是：【一句话异常】。
复现步骤是：【怎么触发】。
预期结果是：【本来应该怎样】。
实际结果是：【现在怎样】。
影响范围是：【页面 / 接口 / 模块 / 数据 / 环境】。
回归关注点是：【还有哪些地方可能受影响】。

请先定位根因，再开始修改。
不要先大范围重构，也不要先猜修复方案。
```

### 2.5 MQTT / 协议 / message-flow / telemetry 排障

```md
请接手 `spring-boot-iot` 的接入链路排障任务。

当前现象是：【MQTT 不回流 / traceId 绑定失败 / timeline 不完整 / TDengine 没写入 / latest 查询异常】。
已知信息是：【topic / deviceCode / sessionId / traceId / 阶段名 / 日志】。
复现方式是：【如何触发】。
预期结果是：【链路应该在哪一步完成】。

请先判断问题位于发布、consumer、pipeline、持久化还是查询层。
请按 `INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> TELEMETRY_PERSIST -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE` 的链路顺序排查。
不要直接猜代码修改点。
```

### 2.6 真实环境复验 / 提测 / 交付核对

```md
请接手 `spring-boot-iot` 的真实环境复验任务。

本轮目标是：【复验什么能力】。
验收范围是：【接口 / 页面 / 脚本 / 链路】。
验收基线固定为：`spring-boot-iot-admin/src/main/resources/application-dev.yml`。
已有证据是：【日志 / 截图 / 接口响应 / SQL / traceId】。

请先给出本轮验收步骤、核对点和阻塞风险，再执行验证。
如果环境访问受阻，请明确标记为“环境阻塞”，不要引入 H2 回退链路。
```

### 2.7 处理 code review 意见

```md
请接手 `spring-boot-iot` 的 code review 处理任务。

review 意见如下：
【粘贴意见】

请先逐条判断：
1. 这条意见是否准确
2. 是否符合当前模块边界和业务背景
3. 是否会影响现有行为

不要不验证就盲改，也不要只做表态式回复。
```

### 2.10 文档治理 / 协作规则整理


## 5. 何时先分析，何时直接修改

| 情况 | 建议方式 | 推荐模板 |
|---|---|---|
| 你不知道问题在哪一层 | 先分析，不改代码 | `2.1` 或 `3.3` |
| 任务跨多个模块或多个技术面 | 先分析，再设计，再计划 | `3.1`、`3.2`、`3.5` |
| 目标已明确，只差实施 | 总结理解后直接修改 | `2.2`、`2.3`、`2.8`、`2.9` |
| 明确是 bug，但根因未知 | 先定位根因，再改 | `2.4` 或 `2.5` |
| 明确是 review 落地 | 先验证意见，再改 | `2.7` |
| 明确是验收复验 | 直接列步骤并验证 | `2.6` |
