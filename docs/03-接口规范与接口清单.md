# 03 接口规范与接口清单

更新时间：2026-03-22

本清单以 Controller 源码为准，目标是作为二次开发与联调的主索引。

## 1. 通用规范

### 1.1 基础协议

- 接口风格：REST + JSON。
- 统一返回：`R`，典型格式：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

- 业务错误：`code != 200`，前端统一按失败处理。

### 1.2 鉴权规范

- 默认要求：`Authorization: Bearer <token>`。
- 公共白名单：
  - `POST /api/auth/login`
  - `POST /api/message/http/report`
  - `/api/cockpit/**`
  - `/actuator/**`
  - `/doc.html`
  - `/swagger-ui/**`
  - `/swagger-ui.html`
  - `/v3/api-docs/**`

### 1.3 链路追踪

- 请求/响应可携带 `X-Trace-Id`。
- `/api/*` 会注入或透传 traceId。

### 1.4 分页约定

【根据代码推断】多数分页接口采用 `pageNum`、`pageSize`，返回 `PageResult`（`records`、`total`、`pageNum`、`pageSize`）。

## 2. 认证接口（auth）

- `POST /api/auth/login`：账号登录。
- `GET /api/auth/me`：当前登录用户信息。

## 3. 设备接入与调试接口（device/message）

### 3.1 产品与设备

- `POST /api/device/product/add`
- `GET /api/device/product/page`
- `GET /api/device/product/list`
- `GET /api/device/product/{id}`
- `PUT /api/device/product/{id}`
- `DELETE /api/device/product/{id}`
- `POST /api/device/add`
- `POST /api/device/batch-add`
- `GET /api/device/{id}`
- `GET /api/device/code/{deviceCode}`
- `GET /api/device/page`
- `PUT /api/device/{id}`
- `POST /api/device/{id}/replace`
- `DELETE /api/device/{id}`
- `POST /api/device/batch-delete`
- `GET /api/device/list`
- `GET /api/device/{deviceId}/metrics`

补充口径：
- `POST /api/device/product/add` 与 `PUT /api/device/product/{id}` 当前共用 `ProductAddDTO`，必填字段为 `productKey`、`productName`、`protocolCode`、`nodeType`，可选字段为 `dataFormat`、`manufacturer`、`description`、`status`。
- `productKey` 是产品的机器侧唯一标识；【根据代码推断】当前后端按 `(tenant_id, product_key)` 做唯一约束，且更新产品时若试图修改 `productKey` 会直接返回业务异常。
- `POST /api/device/add`、`PUT /api/device/{id}` 与 `POST /api/device/batch-add` 当前都要求传入 `productKey`；后端会先按 `productKey` 查产品，再把产品的 `protocolCode`、`nodeType` 继承到设备。
- `POST /api/device/add`、`PUT /api/device/{id}` 与 `POST /api/device/{id}/replace` 当前支持 `parentDeviceId`（父设备主键）与 `parentDeviceCode`（父设备编码，二者不可同时填写）；批量导入建议优先使用 `parentDeviceCode`。
- 当目标设备产品的 `nodeType=3`（网关子设备）且已选择父设备时，后端会按父设备链路自动回填 `gatewayId`；详情、分页结果和设备选项接口都会同步返回父设备与关联网关的展示字段。
- `DELETE /api/device/product/{id}` 当前会在删除前校验关联设备数量，若仍存在库存设备会拒绝删除。
- `GET /api/device/product/list` 当前只返回“未删除且启用中”的产品，供设备建档和前端下拉框使用。
- `GET /api/device/product/{id}` 当前除基础产品档案外，还会返回 `todayActiveCount`、`sevenDaysActiveCount`、`thirtyDaysActiveCount`、`avgOnlineDuration`、`maxOnlineDuration` 五个详情增强字段。
- 活跃设备数口径基于 `iot_device.last_report_time` 的自然日窗口聚合。
- `avgOnlineDuration`、`maxOnlineDuration` 当前基于 `iot_device_online_session` 聚合，统计近 30 天内会话的平均/最大在线分钟数。
- 在线会话结束时间当前按 `最后上报时间 + iot.device.online-timeout-seconds` 推断；默认超时阈值为 `120` 秒。
- 在线时长不回填历史老会话，若产品在会话表中暂无记录，则这两个字段返回 `null`。
- `PUT /api/device/product/{id}` 当前在把产品状态改为停用时，会先校验该产品下是否仍存在启用中的关联设备。
- 停用产品当前会被后端拒绝用于 `POST /api/device/add`、`POST /api/device/batch-add`、设备更换建档、设备上报接入，以及 `POST /api/message/mqtt/down/publish` 下行指令下发。
- `GET /api/device/list` 默认只返回启用中的设备选项；当传入 `includeDisabled=true` 时，也会返回禁用设备，供设备资产中心在编辑/更换场景回显现有父设备关系。

### 3.2 上报与下发

- `POST /api/message/http/report`
- `POST /api/message/mqtt/report/publish`
- `POST /api/message/mqtt/down/publish`
  - 若目标设备所属产品已停用，后端会直接返回业务错误，拒绝继续下发。

补充口径：
- 链路验证中心当前已按“设备编码 -> 查询设备 -> 发送模拟”交互收口：前端先通过 `GET /api/device/code/{deviceCode}` 反查设备接入契约，再只读回显 `productKey`、`protocolCode`、`clientId=deviceCode`；`tenantId` 默认 `1` 且仅 HTTP 生效，`topic` 默认 `$dp` 且允许手工修改。
- `POST /api/message/http/report` 当前支持链路验证中心双模式：
  - 明文模式：只有可识别 JSON 才按 C.1/C.2/C.3 自动构造“类型字节 + 大端长度 + 正文/文件流”二进制帧。
  - 明文 XML / 普通文本：不做帧识别，直接按原始文本透传。
  - 密文模式：前端直接透传 `{"header":{"appId":"..."},"bodies":{"body":"..."}}` 封包。
- 当深部位移子设备已绑定 `risk_point_device` 且 `iot.alarm.auto-closure.enabled=true` 时，`POST /api/message/http/report` / MQTT 上报在完成 `$dp -> 解密/解析 -> 子设备映射 -> 属性落库` 后，会继续自动执行红橙黄蓝判级、告警/事件/工单闭环；该能力不新增对外 API 路径。
- 自动闭环命中应急预案时，会优先从启用预案中筛选名称/描述/响应步骤/联系人文本里能命中深部位移场景关键词或风险点上下文的记录；若无场景化预案，留痕允许为空。
- 自动闭环内部会额外使用 Redis 分布式锁保护同一 `deviceCode + metricIdentifier + level` 的并发去重；该治理不改变现有接口入参、出参和响应结构。
- 新增可选字段 `payloadEncoding`：
  - 默认或空值：按 `UTF-8` 取字节（兼容历史调用）。
  - `ISO-8859-1` / `latin1`：按单字节映射取字节，适用于明文二进制帧模拟。
- `POST /api/message/mqtt/report/publish` 为“原始 MQTT 上行模拟发布”接口，请求体固定字段如下：
  - 必填：`deviceCode`、`productKey`、`protocolCode`、`topic`、`payload`
  - 可选：`payloadEncoding`、`qos`、`retained`
  - 发送语义：直接把原始字节发布到 Broker，再由现有 MQTT consumer 回流进入主链路；不复用 `DownMessageService` 下行编码逻辑。
  - 发送前校验：MQTT 客户端已连接、设备存在、设备所属产品与传入 `productKey` 一致，且设备协议编码若存在则需与 `protocolCode` 一致。
  - 前端兼容规则：当传输方式为 MQTT、`topic=$dp`、明文内容为普通属性 JSON 且未携带设备身份字段时，前端会先补入 `deviceCode` 再调用该接口。

### 5.8 站内消息 `/api/system/in-app-message`

- 管理端：
  - `GET /api/system/in-app-message/page`
  - `GET /api/system/in-app-message/stats`
  - `GET /api/system/in-app-message/{id}`
  - `POST /api/system/in-app-message/add`
  - `PUT /api/system/in-app-message/update`
  - `DELETE /api/system/in-app-message/delete/{id}`
- 当前用户消费端：
  - `GET /api/system/in-app-message/my/page`
  - `GET /api/system/in-app-message/my/unread-count`
  - `GET /api/system/in-app-message/my/{id}`
  - `POST /api/system/in-app-message/my/read/{id}`
  - `POST /api/system/in-app-message/my/read-all`

补充口径：
- 消息类型当前固定为 `system / business / error`；优先级固定为 `critical / high / medium / low`。
- 来源类型当前固定收口为 `manual / system_error / event_dispatch / work_order / governance`；旧样例中的 `system_maintenance / daily_report / governance_task` 在升级后会分别回填到 `manual / manual / governance`。
- 推送范围当前固定为 `all / role / user`，分别通过 `target_type`、`target_role_codes`、`target_user_ids` 表达；当前先不引入单独的发布明细关系表。
- 管理端 `GET /api/system/in-app-message/page` 当前已支持 `sourceType` 筛选；`GET /api/system/in-app-message/stats` 当前返回发布时间范围、消息类型、来源类型、总发送量、已读量、未读量、已读率、趋势桶、来源占比和高未读消息榜单。
- `sys_in_app_message` 当前已新增 `dedup_key`；后端去重规则固定为 `sourceType + sourceId + target fingerprint + messageType`，自动消息默认通过服务层查重复用，不额外加数据库唯一约束。
- `GET /api/system/in-app-message/my/unread-count` 当前返回总未读数和按 `system / business / error` 分类的未读数，便于壳层通知中心直接显示摘要指标。
- 当前用户消息列表只返回“已发布、未删除、未过期且当前用户有权访问”的消息；已读状态由 `sys_in_app_message_read` 单独记录。
- 当前用户消息列表排序已固定为“未读优先 -> 优先级 -> 发布时间倒序”；壳层会在登录后、标签页重新激活、通知摘要/列表打开以及显式已读后立即刷新，并通过 `BroadcastChannel` / `storage` 事件同步多标签页未读徽标。
- 系统自动消息当前只允许查看或停用，不允许在管理端随意改正文或删除；手工广播与治理类消息仍可通过治理页新增、编辑、删除。
- 当前高优未读桥接规则：
  - 不新增对外 HTTP 接口，完全由后端调度任务在壳层消息闭环外部执行。
  - 仅扫描 `high / critical` 且超过阈值仍未读的消息。
  - 当前按“每条站内消息 + 每个渠道一次成功闭环、失败按 `minIntervalSeconds` 冷却重试”的策略执行，并把结果写入 `sys_in_app_message_bridge_log`。
- 当前已接入的自动消息源：
  - `system_error`：来源于后台异常审计链路，目标角色固定为 `OPS_STAFF / DEVELOPER_STAFF / SUPER_ADMIN`，关联页面 `/system-log`，默认 24 小时过期。
  - `event_dispatch`：来源于事件派工，目标用户为 `receiveUser`，关联页面 `/event-disposal`，默认 7 天过期。
  - `work_order`：来源于工单接收 / 开始 / 完成，接收与开始发给 `receiveUser`，完成发给 `assignUser`，关联页面 `/event-disposal`，默认 7 天过期。
- 当前壳层消费约定：
  - 摘要弹层继续使用 `GET /api/system/in-app-message/my/page?pageSize=6` + `GET /api/system/in-app-message/my/unread-count`。
  - “查看更多”列表抽屉继续使用 `GET /api/system/in-app-message/my/page`。
  - 详情抽屉继续使用 `GET /api/system/in-app-message/my/{id}`。
  - 打开摘要弹层或详情抽屉不会自动改写已读状态；只有 `POST /api/system/in-app-message/my/read/{id}` 或 `POST /api/system/in-app-message/my/read-all` 才会更新未读统计。

  - `related_paths` 为空时，不额外限制页面权限。
  - 当 `related_paths` 有值时，后端会按当前用户已授权菜单路径过滤无权页面资料。
- `GET /api/system/help-doc/access/list` 支持 `docCategory`、`keyword`、`currentPath`、`limit` 参数；其中 `currentPath` 主要用于提升“当前页面最相关资料”的排序优先级。
- `GET /api/system/help-doc/access/page` 支持 `docCategory`、`keyword`、`currentPath`、`pageNum`、`pageSize` 参数，返回 `PageResult<HelpDocumentAccessVO>`。
- 当前壳层消费约定：
  - 摘要弹层继续使用 `GET /api/system/help-doc/access/list?limit=6`。
  - “查看更多”列表抽屉统一使用 `GET /api/system/help-doc/access/page` 做服务端分页检索。
  - 详情抽屉继续使用 `GET /api/system/help-doc/access/{id}`，并保留 `currentPath` 排序与当前页相关标识。
- 若真实环境历史库尚未执行 `sql/upgrade/20260321_phase5_in_app_message_help_docs.sql`，`/api/system/help-doc/**` 与 `/api/system/in-app-message/**` 现在会返回明确业务错误；若旧库还未补齐通知中心后续治理字段，还需继续执行 `sql/upgrade/20260322_phase5_notification_center_followup.sql`；若本轮需要开启高优未读桥接，还需继续执行 `sql/upgrade/20260322_phase5_notification_channel_bridge.sql`。
