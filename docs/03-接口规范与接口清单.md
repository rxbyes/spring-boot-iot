## 1. 通用规范

### 1.1 基础协议

- 接口风格：REST + JSON。
- 统一返回：`R`，典型格式：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

- 业务错误：`code != 200`，前端统一按失败处理。

### 1.2 鉴权规范

- 默认要求：`Authorization: Bearer <token>`。
- 公共白名单：
  - `POST /api/auth/login`
  - `POST /api/message/http/report`
  - `/api/cockpit/**`
  - `/actuator/**`
  - `/doc.html`
  - `/swagger-ui/**`
  - `/swagger-ui.html`
  - `/v3/api-docs/**`

### 1.3 链路追踪

- 请求/响应可携带 `X-Trace-Id`。
- `/api/*` 会注入或透传 traceId。

### 1.4 分页约定

【根据代码推断】多数分页接口采用 `pageNum`、`pageSize`，返回 `PageResult`（`records`、`total`、`pageNum`、`pageSize`）。

### 1.5 OpenAPI 与 SDK 生成发布口径

- 当前运行时 OpenAPI 入口继续使用：
  - `/v3/api-docs/**`
  - `/swagger-ui/**`
  - `/swagger-ui.html`
  - `/doc.html`
- 当前权威顺序为：
  1. Controller 与 DTO / VO 源码
  2. 本文件中的接口口径与兼容说明
  3. 运行时 OpenAPI 文档
- 当前仓库未见“自动生成 SDK 并发布到制品库”的独立流水线配置，因此：
  - OpenAPI 可作为联调和生成 SDK 的输入源；
  - 但生成产物不是新的权威源，不能反向覆盖本文件和代码事实。
- 若后续要对外发布 SDK，至少需要同步记录：
  - 生成日期与对应分支 / 提交
  - 使用的 `basePath`
  - 覆盖的模块范围
  - 是否包含未纳入当前交付范围的灰度接口
- 风险监测、非 Reactor 模块或灰度接口如需出现在 SDK 中，必须先与 [19-第四阶段交付边界与复验进展.md](./19-第四阶段交付边界与复验进展.md) 和 [21-业务功能清单与验收标准.md](./21-业务功能清单与验收标准.md) 的交付口径对齐。

## 2. 认证接口（auth）

- `POST /api/auth/login`：账号登录。
- `GET /api/auth/me`：当前登录用户信息。
  - 当前除菜单、按钮与角色信息外，还统一返回租户、主机构、昵称/展示名、头像、登录方式、最近登录信息和角色数据范围摘要，供壳层账号中心与治理页共用。
  - `authStatus` 当前只返回真实资料态文案：`已填写实名信息`、`未填写实名信息`。

补充口径：
- `POST /api/device/product/add` 与 `PUT /api/device/product/{id}` 当前共用 `ProductAddDTO`，必填字段为 `productKey`、`productName`、`protocolCode`、`nodeType`，可选字段为 `dataFormat`、`manufacturer`、`description`、`metadataJson`、`status`。
- `GET /api/device/product/page` 当前仍使用 `productName` 作为 `/products` 页面快速搜索入参，但服务端已按“产品名称 / 产品 Key / 厂商”三字段执行统一模糊匹配；页面不会再把产品 Key 搜索误判为无结果。
- `productKey` 是产品的机器侧唯一标识；【根据代码推断】当前后端按 `(tenant_id, product_key)` 做唯一约束，且更新产品时若试图修改 `productKey` 会直接返回业务异常。
- `POST /api/device/add`、`PUT /api/device/{id}` 与 `POST /api/device/batch-add` 当前都要求传入 `productKey`；后端会先按 `productKey` 查产品，再把产品的 `protocolCode`、`nodeType` 继承到设备。
- `POST /api/device/add`、`PUT /api/device/{id}` 与 `POST /api/device/{id}/replace` 当前支持 `parentDeviceId`（父设备主键）与 `parentDeviceCode`（父设备编码，二者不可同时填写）；批量导入建议优先使用 `parentDeviceCode`。
- `GET /api/device/{id}`、`GET /api/device/code/{deviceCode}` 返回的 `metadataJson` 当前继续保持设备扩展元数据透传；`/insight` 仍会兼容读取其中的 `objectInsight.customMetrics[]` 做单设备扩展覆盖，但这不改变设备主档新增/编辑接口的字段结构，也不作为对象洞察重要指标的正式治理真相源。
- `GET /api/device/{deviceCode}/topology-role` 当前为对象洞察设备拓扑角色真相接口，返回 `COLLECTOR_PARENT / COLLECTOR_CHILD / STANDALONE` 三态之一，`/insight` 统一按该返回值决定父设备视图、子设备视图与趋势/快照过滤，不再前端分散硬编码 `nodeType` 兼容。
- `GET /api/device/{deviceCode}/insight/properties` 当前为对象洞察属性快照专用接口，响应结构固定为 `topologyRole + properties[]`。其中 `properties[]` 已在后端按角色过滤：父采集器不返回子设备业务字段（如 `dispsX / dispsY / value / sensor_state`），子设备不回流父采集器运行参数（如 `signal_4g / battery / temp / humidity / ext_power_volt` 等）。
- `GET /api/device/{deviceCode}/collector-children/overview` 当前为采集器对象洞察读侧聚合接口，仅在父设备语义下汇总子设备最新监测值、`sensor_state` 与链路可达性，不会回写采集器契约字段。响应主结构固定包含 `parentDeviceCode / parentOnlineStatus / childCount / reachableChildCount / sensorStateReportedCount / missingChildCount / staleChildCount / recommendedMetricCount / children[]`；`children[]` 单项当前包含 `logicalChannelCode / childDeviceCode / childDeviceName / childProductKey / collectorLinkState / sensorStateValue / sensorStateHealth / lastReportTime / recommendedMetricIdentifiers / metrics[]`，其中 `sensorStateHealth` 固定为 `REPORTED_NORMAL / REPORTED_ABNORMAL / MISSING / STALE`，`metrics[]` 单项又会补齐 `recommended`，用于对象洞察页直接标记“建议优先纳入趋势”的子设备指标。该接口不新增汇总表，继续基于既有 `iot_device_relation`、设备主档与 latest 属性读侧拼装。`2026-04-19` 起，`nf-collect-rtu-v1` 虽仍按 `nodeType=1` 建档，`/insight` 读侧也必须继续把它视为采集器父设备并调用该接口。
- `GET /api/device/product/{productId}/collector-children/recommended-metrics` 当前按产品返回采集器子设备对象洞察建议指标，结果来源固定为当前产品下“已启用且 `insightEnabled=1`”的正式风险指标目录 `contractIdentifier` 去重集合。该接口只提供读侧建议，不会自动改写 `metadataJson.objectInsight.customMetrics[]`，也不会替代 `/products` 的正式配置写侧。
- `GET /api/device/product/{id}` 当前除设备活跃度统计增强字段外，也会继续透传 `metadataJson`；其中 `metadataJson.objectInsight.customMetrics[]` 已作为对象洞察重要指标的产品级单一真相源，由 `/products` 的新增/编辑表单、产品详情页 `编辑档案`，以及 `/products/:productId/contracts -> 当前已生效字段` 的趋势快捷动作共同维护，`/insight` 再把它作为正式配置读取。若当前产品已存在正式发布批次，这里的 `customMetrics[].identifier` 也必须命中已发布 canonical identifier。
- 产品 `metadataJson.objectInsight.customMetrics[]` 当前正式字段固定为 `identifier / displayName / group / unit / includeInTrend / includeInExtension / analysisTitle / analysisTag / analysisTemplate / enabled / sortNo`；其中 `group` 只允许 `measure / status / runtime`，分别对应对象洞察里的 `监测数据 / 状态事件 / 运行参数`。后端会拒绝非法 JSON、重复 `identifier`、非法 `group`、超过 `20` 项和 `analysisTemplate` 超过 `300` 字的脏配置。若当前产品已存在已发布合同快照，`identifier` 原则上必须命中已发布 canonical identifier：仅大小写不同的 canonical 写法会在保存时自动归一，`L1_LF_1.value` 这类 raw alias 不再允许写入正式对象洞察配置。`2026-04-23` 起，为兼容历史残留短标识，若入参是无点号短标识且在已发布标识中能唯一命中同名后缀（如 `gX -> L1_JS_1.gX`），写侧会自动升级为该已发布标识；若命中不唯一或无命中，仍按“对象洞察指标必须使用已发布合同标识符”拒绝。`/insight` 运行时会按这四个治理字段消费：`enabled=false` 或 `includeInTrend=false` 的指标不会进入趋势/历史查询，`includeInExtension=false` 的重点趋势指标会按 `sortNo` 前置展示；平台也不再默认按运行时属性自动拼出 `监测数据 / 状态事件 / 运行参数` 三组趋势项，未手工配置时趋势请求会保持空集合。`2026-04-21` 起，前端保存对象洞察配置时也会透传 `customMetrics[].unit`，且 `/insight` 在运行态缺单位时会按 `property.unit -> customMetrics[].unit -> property.specsJson.unit` 回退拼接趋势图例与属性快照单位。`POST /api/telemetry/history/batch` 在进入 TDengine 查询前，还会优先按已发布快照解析 raw alias / 大小写漂移，再回退产品元数据与当前属性，把请求统一改写为 canonical / 运行时 identifier。`2026-04-19` 起，前端对象洞察读侧还会对历史遗留短标识做一层有限兼容：只有当短标识能在当前 latest 属性里唯一命中同名后缀全路径标识时，才升级为该运行态全路径标识后再发起历史查询；若存在多个候选后缀，则保持原标识不动。`2026-04-20` 起，若对象洞察正式配置本身保留全路径正式标识，而 `POST /api/telemetry/history/batch` 的 `points[].identifier` 返回 bare/canonical 标识，前端也会在 exact path 未命中时按单一 bare/canonical 候选回绑到该正式字段；若 bare/canonical 候选不唯一，则继续保持未命中，避免把多通道 `value` 一类趋势误绑到错误测点。
- `/products` 的产品新增/编辑与 `编辑档案` 中的对象洞察配置区，当前会额外读取 `GET /api/device/product/{productId}/models` 返回的正式 `property` 模型作为快速加入候选；前端不会为此新建对象洞察专属写接口，最终仍复用 `PUT /api/device/product/{id}` 回写产品 `metadataJson`。
- `POST /api/device/{id}/secret-rotate` 当前用于设备密钥轮换，要求请求头 `X-Governance-Approver-Id`，并执行双人复核（执行人与复核人不得为同一账号）。返回字段固定包括 `deviceId / deviceCode / productKey / rotationBatchId / rotatedBy / approvedBy / rotatedAt / reason`。
- `POST /api/device/{id}/secret-rotate` 当前要求执行人具备 `iot:secret-custody:rotate`、复核人具备 `iot:secret-custody:approve`；轮换日志会落库 `iot_device_secret_rotation_log`，并写入审计日志。
- `GET /api/device/secret-rotation-logs` 当前按 `pageNum / pageSize` 分页返回密钥轮换台账，支持 `deviceCode / productKey / rotationBatchId / rotatedBy / approvedBy / beginTime / endTime` 过滤；响应记录固定包含 `id / deviceId / deviceCode / productKey / rotationBatchId / reason / previousSecretDigest / currentSecretDigest / rotatedBy / approvedBy / rotateTime`，只暴露摘要不暴露明文。
- `GET /api/device/secret-rotation-logs` 当前要求 `iot:secret-custody:view`；非超管会按当前登录用户 `tenantId` 收口查询范围。
- 当目标设备产品的 `nodeType=3`（网关子设备）且已选择父设备时，后端会按父设备链路自动回填 `gatewayId`；详情、分页结果和设备选项接口都会同步返回父设备与关联网关的展示字段。
- `POST /api/device/add`、`POST /api/device/batch-add`、`PUT /api/device/{id}`、`POST /api/device/{id}/replace`、`DELETE /api/device/{id}`、`POST /api/device/batch-delete`、`GET /api/device/{id}`、`GET /api/device/code/{deviceCode}`、`GET /api/device/page`、`GET /api/device/list`、`GET /api/device/{deviceId}/metrics` 与 `GET /api/device/{deviceCode}/properties` 当前都统一透传当前登录用户主键；服务层会先按当前登录人的 `tenantId + dataScopeType + orgId` 收口产品、设备、父设备与未登记补档视图。设备写入链路会忽略前端自带的机构字段，并统一按当前登录人的主机构回填 `iot_device.org_id / org_name`；当前账号缺少主机构或主机构失效时会直接返回业务错误。
- `GET /api/device/{deviceId}/metrics` 当前返回 `identifier / name / dataType`；若该设备所属产品的正式字段已经发布到启用中的 `risk_metric_catalog`，返回项还会额外带上 `riskMetricId`，供风险点绑定、待治理转正和阈值策略直接围绕目录指标收口。
- `GET /api/device/onboarding/suggestion` 当前按 `traceId` 返回未登记线索的接入建议，响应固定包含 `traceId / deviceCode / deviceName / assetSourceType / productKey / protocolCode / lastFailureStage / lastErrorMessage / lastReportTopic / lastPayload / recommendedProductId / recommendedProductKey / recommendedProductName / recommendedFamilyCode / recommendedFamilyName / recommendedDecryptProfileCode / recommendedTemplateCode / recommendedTemplateName / suggestionStatus / ruleGaps[]`。当前 `suggestionStatus` 只区分 `READY / PARTIAL`：只有推荐产品已命中正式产品且已存在正式合同发布、协议族与协议模板已命中 published 快照、并且 trace 携带样本 payload 时才会返回 `READY`；否则统一按 `PARTIAL + ruleGaps[]` 回答卡点。
- `POST /api/device/onboarding/batch-activate` 当前接收 `traceIds[]` 与 `confirmed`，其中 `confirmed=true` 必填。服务端会先裁剪并去重 `traceIds`，再逐条复用接入建议口径校验；只有 `suggestionStatus=READY` 且 `ruleGaps[]` 为空的线索才会继续复用既有 `POST /api/device/add` 真相链创建正式设备，并默认写入 `activateStatus=1 / deviceStatus=1`。单条失败不会回滚整批，响应固定返回 `requestedCount / activatedCount / rejectedCount / activatedTraceIds / activatedDeviceCodes / errors[]`，其中 `errors[]` 单项包含 `traceId / deviceCode / message`。
- `GET /api/device/onboarding/template-packs` 当前按 `tenantId / keyword / status / scenarioCode / deviceFamily / pageNum / pageSize` 分页返回模板包；`keyword` 统一按 `packCode / packName / scenarioCode / deviceFamily` 做模糊匹配，读侧只返回未逻辑删除记录。
- `POST /api/device/onboarding/template-packs` 与 `PUT /api/device/onboarding/template-packs/{packId}` 当前共用最小 upsert 契约：`tenantId?`、`packCode`、`packName`、`scenarioCode?`、`deviceFamily?`、`status?`、`protocolFamilyCode?`、`decryptProfileCode?`、`protocolTemplateCode?`、`defaultGovernanceConfigJson?`、`defaultInsightConfigJson?`、`defaultAcceptanceProfileJson?`、`description?`、`remark?`；若 `tenantId` 为空，服务端默认按租户 `1` 落库，同租户下 `packCode` 必须唯一。模板包创建时固定 `versionNo=1`，更新时会递增 `versionNo`。

- `specsJson`、`serviceInputJson`、`serviceOutputJson` 当前都在服务层执行 JSON 合法性校验；更新与删除时都会校验 `modelId` 属于当前 `productId`。`specsJson` 在更新链路中已支持显式写 `null`，不再把“清空单位”误判为“未修改字段”。
- 受真实库 `iot_product_model.data_type NOT NULL` 约束影响，`event` / `service` 当前在落库时会内部写入兼容值 `json`；但 `GET/POST/PUT /api/device/product/{productId}/models*` 对外仍保持 `dataType=null`，前端无需补填该字段。
- 同一 `productId` 下 `identifier` 当前必须唯一；控制器路径固定挂在产品资源下，不新增顶层菜单接口。
- 手动样本治理当前不再暴露 `GET/POST /model-candidates*` 兼容入口，也不再暴露 `manualDraftItems[]`、`governanceMode`、`normativePresetCode`、`selectedNormativeIdentifiers[]`、`includeRuntimeCandidates` 等旧 compare 参数。
- 活跃设备数口径基于 `iot_device.last_report_time` 的自然日窗口聚合。
- `avgOnlineDuration`、`maxOnlineDuration` 当前基于 `iot_device_online_session` 聚合，统计近 30 天内会话的平均/最大在线分钟数。
- 在线会话结束时间当前优先按 `iot_device.last_online_time + iot.device.online-timeout-seconds` 推断；若历史数据尚未补齐 `last_online_time`，则回退到 `last_report_time + iot.device.online-timeout-seconds`。当前默认超时阈值为 `7200` 秒。
- 在线时长不回填历史老会话，若产品在会话表中暂无记录，则这两个字段返回 `null`。
- `PUT /api/device/product/{id}` 当前在把产品状态改为停用时，会先校验该产品下是否仍存在启用中的关联设备。
- 停用产品当前会被后端拒绝用于 `POST /api/device/add`、`POST /api/device/batch-add`、设备更换建档、设备上报接入，以及 `POST /api/message/mqtt/down/publish` 下行指令下发。
- `POST /api/device/add`、`POST /api/device/batch-add` 与 `POST /api/device/{id}/replace` 成功后，当前会自动把同 `productKey + deviceCode` 的 `DEVICE_NOT_FOUND` 无效上报最新态标记为 `resolved=1`，避免未登记名单继续保留已补录设备的历史噪音。
- `/devices` 当前允许对未登记行直接点击 `编辑` 完成建档；前端不会调用新的专用接口，而是复用现有 `POST /api/device/add` 提交完整设备表单。建档成功后，该记录会转入 `iot_device` 主档，并继续复用现有 invalid report resolve 逻辑收口未登记最新态。
- `/devices` 当前对带 `lastTraceId` 的未登记行还支持“建议优先、人工确认”的半自动转正：前端可先调用 `GET /api/device/onboarding/suggestion` 查看推荐产品、协议族、模板与 `ruleGaps[]`，再在人工确认后通过 `POST /api/device/onboarding/batch-activate` 批量转正式设备；缺少 `TraceId`、建议仍为 `PARTIAL` 或存在 `ruleGaps[]` 的线索则继续只能走人工补录。
- `GET /api/device/page` 当前新增可选筛选参数 `registrationStatus`：`1` 只返回已登记设备，`0` 只返回未登记上报名单，空值则在同一分页结果内先返回已登记设备，再按剩余页容量补入未登记设备。
- `GET /api/device/page` 当前新增可选筛选参数 `keyword` 与 `productName`：`keyword` 在已登记设备与未登记名单两条来源上都统一按 `deviceCode / deviceName / productKey / productName` 执行 OR 模糊匹配，`productName` 则作为显式产品名称筛选；未登记名单会通过 `iot_product.product_name` 反查对应 `productKey`，避免只知道产品名称时无法命中历史上报。
- `GET /api/device/page` 返回项当前新增 `registrationStatus`、`assetSourceType`、`sourceRecordId`、`lastFailureStage`、`lastErrorMessage`、`lastReportTopic`、`lastTraceId`、`lastPayload`；其中未登记行 `id=null`、`registrationStatus=0`，仅用于列表展示与详情排障。
- 未登记名单当前优先来源于 `iot_device_invalid_report_state` 中 `reasonCode=DEVICE_NOT_FOUND` 且 `resolved=0` 的最新态记录；若共享环境尚未补齐该表或列结构，则回退到 `iot_device_access_error_log` 最近一条失败样本，再回退到仅存在 `iot_device_message_log(messageType=dispatch_failed)` 的最近失败轨迹。
- `GET /api/device/page` 在未登记名单场景下，当前已支持 `keyword`、`productName`、`productKey`、`deviceCode` 与 `registrationStatus` 过滤；若同时传入 `deviceId`、`onlineStatus`、`activateStatus`、`deviceStatus`，未登记名单仍会按“无命中”处理，不伪造运行态字段筛选结果。
- `GET /api/device/list` 默认只返回启用中的设备选项；当传入 `includeDisabled=true` 时，也会返回禁用设备，供设备资产中心在编辑/更换场景回显现有父设备关系。
- `/devices` 打开未登记详情时，只有 `assetSourceType=access_error` 的行才保证可通过 `GET /api/device/access-error/{id}` 补全失败归档详情；若当前行来自 `invalid_report_state` 或 `dispatch_failed` 来源，则页面继续以列表摘要为准，不额外伪造不存在的失败归档明细。

### 3.2 上报与下发

- `POST /api/message/http/report`
- `POST /api/message/mqtt/report/publish`
- `POST /api/message/mqtt/down/publish`
  - 若目标设备所属产品已停用，后端会直接返回业务错误，拒绝继续下发。

补充口径：
- 链路验证中心当前已按“设备编码 -> 查询设备 -> 发送模拟”交互收口：前端先通过 `GET /api/device/code/{deviceCode}` 反查设备接入契约，再只读回显 `productKey`、`protocolCode`、`clientId=deviceCode`；`tenantId` 默认 `1` 且仅 HTTP 生效，`topic` 默认 `$dp` 且允许手工修改。
- `POST /api/message/http/report` 当前支持链路验证中心双模式：
  - 明文模式：只有可识别 JSON 才按 C.1/C.2/C.3 自动构造“类型字节 + 大端长度 + 正文/文件流”二进制帧。
  - 明文 XML / 普通文本：不做帧识别，直接按原始文本透传。
  - 密文模式：前端直接透传 `{"header":{"appId":"..."},"bodies":{"body":"..."}}` 封包。
- `POST /api/message/http/report` 当前返回 `R<MessageFlowSubmitResultVO>`，`data` 固定字段：
  - `sessionId`
  - `traceId`
  - `status`
  - `timelineAvailable`
  - `correlationPending`
- 当深部位移子设备已绑定 `risk_point_device` 且 `iot.alarm.auto-closure.enabled=true` 时，`POST /api/message/http/report` / MQTT 上报在完成 `$dp -> 解密/解析 -> 子设备映射 -> 属性落库` 后，会继续自动执行红橙黄蓝判级、告警/事件/工单闭环；该能力不新增对外 API 路径。
- 自动闭环命中应急预案时，会优先从启用预案中筛选名称/描述/响应步骤/联系人文本里能命中深部位移场景关键词或风险点上下文的记录；若无场景化预案，留痕允许为空。
- 自动闭环内部会额外使用 Redis 分布式锁保护同一 `deviceCode + metricIdentifier + level` 的并发去重；该治理不改变现有接口入参、出参和响应结构。
- 新增可选字段 `payloadEncoding`：
  - 默认或空值：按 `UTF-8` 取字节（兼容历史调用）。
  - `ISO-8859-1` / `latin1`：按单字节映射取字节，适用于明文二进制帧模拟。
- `POST /api/message/mqtt/report/publish` 为“原始 MQTT 上行模拟发布”接口，请求体固定字段如下：
  - 必填：`deviceCode`、`productKey`、`protocolCode`、`topic`、`payload`
  - 可选：`payloadEncoding`、`qos`、`retained`
  - 发送语义：直接把原始字节发布到 Broker，再由现有 MQTT consumer 回流进入主链路；不复用 `DownMessageService` 下行编码逻辑。
  - 发送前校验：MQTT 客户端已连接、设备存在、设备所属产品与传入 `productKey` 一致；协议校验优先使用设备协议，设备协议为空时回退到产品协议；若仍为空则返回“目标设备未配置协议”，若不一致则返回 `expected/actual` 明细。
  - 前端兼容规则：当传输方式为 MQTT、`topic=$dp`、明文内容为普通属性 JSON 且未携带设备身份字段时，前端会先补入 `deviceCode` 再调用该接口。
  - 返回 `R<MessageFlowSubmitResultVO>`，`data` 固定字段：
    - `sessionId`
    - `traceId`（发布成功时允许为空，待消费回流绑定）
    - `status=PUBLISHED`
    - `timelineAvailable=false`
    - `correlationPending=true`
- MQTT consumer 当前已启用首批无效上报治理，但不改变接口出入参契约：对 `DEVICE_NOT_FOUND` 与 `EMPTY_DECRYPTED_PAYLOAD` 会先采样、后进入冷却抑制；其中原始空 payload 会在 consumer 入口直接丢弃，冷却期内重复坏包不再保证继续写入 `dispatch_failed`、`iot_device_access_error_log` 或 `sys_audit_log`。

- `GET /api/device/{deviceCode}/properties` 当前会按 `identifier` 叠加所属产品最新正式字段名；即使 `iot_device_property.property_name` 仍是历史英文名，只要 `iot_product_model.model_name` 已更新，接口也会优先返回最新中文名称，供对象洞察快照与趋势预览直接展示。
- `2026-04-21` 起，`GET /api/device/{deviceCode}/properties` 在正式字段未命中时，还会继续复用运行态字段显示规则补齐 `propertyName / unit`；当前显示优先级固定为 `正式字段名称/单位 -> 运行态显示规则 -> latest 属性自带名称/单位 -> raw identifier`，其中运行态显示规则只影响读侧返回，不会反写 latest 真相表。
- `GET /actuator/health/mqttConsumer` 当前返回 `running`、`connected`、`clientId`、`brokerUrl`、`subscribeTopics`、`lastDisconnectAt` 以及最近一次连接/收包/成功分发/失败阶段摘要，供真实环境回放前先确认 consumer 是否在线。
- 规则化运维告警评估当前继续复用 `sys_audit_log`、Redis `invalid-report` 分钟桶、`iot_device_access_error_log` 样本归档、`/actuator/health/mqttConsumer` 与通知渠道 `scenes` 做调度评估和通知分发；当前调度规则已覆盖 `system-error-burst`、`mqtt-disconnect-timeout`、`failure-stage-spike`、`in-app-bridge-failure-burst`、`risk-governance-missing-policy-burst`，以及治理运维三类规则 `risk-governance-field-drift-burst`、`risk-governance-contract-diff-burst`、`risk-governance-missing-risk-metric-burst`。读侧 `GET /api/risk-governance/ops-alerts` 聚合输出 `FIELD_DRIFT / CONTRACT_DIFF / MISSING_RISK_METRIC` 三类告警，不新增独立写接口；若通知渠道 `config` 额外声明 `opsAlertTypes`，调度器会在同一 `observability_alert` 场景内继续按告警类型做二次订阅过滤。

补充口径：
- `POST /api/risk-point/add` / `POST /api/risk-point/update` 当前至少需要 `riskPointName`、`orgId`、`regionId`、`riskPointLevel` 四个核心字段；`responsibleUser` 为可选档案字段，`orgName`、`regionName` 由后端根据平台治理组织/区域档案回填。若旧 payload 仍只传 `riskLevel` 而未补 `riskPointLevel`，后端会明确返回“风险点档案等级已改为 riskPointLevel”。
- `riskPointCode` 不再要求前端人工录入；新增时由服务端按“所属组织 + 风险点名称 + 风险点档案等级”生成唯一编号，且需避开同租户内已软删除历史记录占用的编号；若并发写入导致撞码，服务端应自动重试下一个后缀，更新时保留历史编号；新增态不展示编号输入，编辑态只读回显历史编号。
- 风险点档案等级当前统一使用 `GET /api/dict/code/risk_point_level` 返回的启用字典项，标准值为 `level_1 / level_2 / level_3`。列表/分页主查询参数为 `riskPointLevel`，并兼容旧查询参数名 `riskLevel` 作为别名。
- `GET /api/risk-point/list` 与 `GET /api/risk-point/page` 自 `2026-04-08` 起额外支持 `keyword`，统一面向 `riskPointCode / riskPointName / regionName` 三字段检索；当 `keyword` 完整等于任一字段值时，服务端会优先走精确匹配并直接返回精确命中结果，只有精确结果为空时才回退到三字段 OR 模糊查询。旧查询参数 `riskPointCode` 继续保留并作为 `keyword` 的兼容别名，便于历史页面和脚本平滑过渡。
- 风险点运行态风险当前独立为 `currentRiskLevel`，标准值为 `red / orange / yellow / blue`。读侧继续保留 `riskLevel` 兼容字段，但其语义仅作为 `currentRiskLevel` 的镜像，不再承担档案等级含义。
- 风险点表单的“所属组织”当前应复用平台治理 `GET /api/organization/tree` 的树结构结果，不再使用手工文本输入；`createBy / updateBy` 由当前登录用户自动填充，并仅在编辑态只读回显。
- 风险点接口当前已落第一批查询侧数据权限：`GET /api/risk-point/list`、`GET /api/risk-point/page`、`GET /api/risk-point/get/{id}`、`POST /api/risk-point/delete/{id}`、`POST /api/risk-point/bind-device`、`POST /api/risk-point/bind-device-capability`、`POST /api/risk-point/unbind-device`、`GET /api/risk-point/bound-devices/{riskPointId}` 都会按当前登录用户的 `tenantId + dataScopeType + orgId` 收口范围。`ALL / TENANT` 表示当前租户内可见，`ORG_AND_CHILDREN` 表示当前主机构及其子树，`ORG` 表示当前主机构，`SELF` 只暴露 `responsibleUser = 当前用户` 或 `createBy = 当前用户` 的记录。
- `POST /api/risk-point/bind-device`、`POST /api/risk-point/bind-device-capability`、`POST /api/risk-point/unbind-device` 与 `POST /api/risk-point/pending-bindings/{pendingId}/promote` 当前统一返回 `GovernanceSubmissionResult`，字段固定为 `workItemId`、`approvalOrderId`、`approvalStatus`、`executionStatus`。`executionStatus=DIRECT_APPLIED` 表示动作已在风险域直接落真相并同步收口治理工作项；`executionStatus=PENDING_APPROVAL` 表示当前只创建治理工作项与审批单，尚未改动 `risk_point_device / risk_point_device_capability_binding / risk_point_device_pending_promotion` 真相。
- `POST /api/risk-point/add`、`POST /api/risk-point/update` 当前会校验提交的 `orgId` 是否位于当前登录人的可写组织范围内；越权时返回业务错误，不再允许把风险点写入范围外组织。
- `POST /api/risk-point/bind-device` 当前按 `riskPointId + deviceId + metricIdentifier` 组合判重；缺少任一关键字段时直接返回业务错误。请求体支持可选 `riskMetricId`：若携带，则后端会先校验该目录指标属于当前设备产品，再把 `metricIdentifier / metricName` 统一收口到目录合同字段；若同时显式传入的 `metricIdentifier` 与目录 `contractIdentifier` 不一致，则直接返回业务错误“目录指标与测点标识符不一致”，不再静默覆盖错误字段。目录生命周期当前只允许 `ACTIVE`：若命中目录 `lifecycleStatus != ACTIVE`，后端会直接返回“风险指标目录当前不可绑定: <id>”。`2026-04-13` 起，通过“风险绑定工作台 -> 维护绑定”发起的正式测点绑定已继续收口为“目录已发布字段必选”口径：若未携带 `riskMetricId`，也只允许按 `GET /api/risk-point/devices/{deviceId}/formal-metrics` 返回的 `identifier` 命中当前设备所属产品已发布到启用中目录的字段，再自动补齐 `riskMetricId / metricName`；未进入目录的正式字段、设备运行期属性或纯治理语义字段会直接返回业务错误“当前测点未发布到风险指标目录，不能用于正式绑定”。若同设备当前已存在设备级正式绑定，也会直接返回“当前设备已存在设备级正式绑定，请先整机解绑后再切换为正式测点绑定”。除同风险点同测点重复外，服务层还会拒绝“同一设备已被其他风险点有效绑定”“设备 `org_id` 为空”，并继续执行组织/租户门禁：非超级管理员必须满足“风险点与设备属于同一租户，且组织关系为同节点或父子链路”；超级管理员允许跨租户、跨组织绑定。`2026-04-11` 起，该接口会先按 `sys_governance_approval_policy` 判断是否需要审批：直写模式下会先打开治理工作项、成功后立即写入 `risk_point_device` 并把工作项收口为 `DIRECT_APPLIED`；审批模式下只创建工作项和审批单，真正绑定写入延后到告警域 `RiskPointGovernanceApprovalExecutor` 审批通过后执行。
- `POST /api/risk-point/bind-device-capability` 当前要求登录态，只服务预警型和视频类设备的设备级正式绑定；请求体固定包含 `riskPointId / deviceId`，并支持可选 `deviceCapabilityType`。服务端会按设备能力类型与产品能力推断做归一；若当前设备 `productId` 为空、`<=0` 或产品主档缺失，则会回退使用设备编号/设备名称做能力兜底识别，不再因为缺少产品主数据直接返回通用 `500`。若最终识别结果仍具备正式测点能力，则直接返回“监测型设备必须选择正式测点，不能按设备级绑定”。若同设备当前已存在正式测点绑定，则会要求先整机解绑再切换为设备级正式绑定。视频类绑定成功后会把 `extensionStatus` 固定写为 `AI_EVENT_RESERVED`，作为后续 AI 事件分析扩展位。
- `POST /api/risk-point/unbind-device` 当前按 `riskPointId + deviceId` 逻辑删除该风险点下该设备的全部有效绑定；前端当前只把它用于“整机解绑”，不再复用为单测点删除。该接口会同时删除 `risk_point_device` 与 `risk_point_device_capability_binding` 两类正式绑定，解绑成功后，`GET /api/risk-point/bound-devices/{riskPointId}` 与 `GET /api/risk-point/binding-groups/{riskPointId}` 都不应再返回该设备。`2026-04-11` 起，该接口同样切换为审批感知提交：命中审批策略时只返回 `PENDING_APPROVAL` 与审批单号，不会提前删除正式绑定；审批通过后的整机解绑仍由告警域执行器在领域内完成。
- `POST /api/risk-point/delete/{id}` 当前走逻辑删除；删除成功后，再次调用 `GET /api/risk-point/get/{id}` 应返回“风险点不存在”。
- `GET /api/risk-point/bindable-devices/{riskPointId}` 当前专门服务绑定设备弹窗：只返回当前登录人可见范围内、且未被其他风险点占用的设备；对非超级管理员，服务层会继续在候选阶段执行“同租户 + 父子组织”门禁，对超级管理员则允许跨租户、跨组织候选。若当前风险点已经绑定过某设备，则该设备仍允许在候选清单中回显，便于编辑同一风险点下的多测点绑定。`2026-04-13` 起，该接口返回项还会显式补齐 `deviceCapabilityType / supportsMetricBinding / aiEventExpandable`，供前端按设备能力在同一工作台内分流“正式测点绑定”与“设备级正式绑定”；当设备缺少有效产品主档时，接口会继续回退读取设备编号/名称做能力识别，命中“视频 / 摄像 / 监控”等线索的设备仍返回 `VIDEO`，避免视频设备在候选列表中被误收口为 `UNKNOWN`。
- `GET /api/risk-point/binding-summaries` 当前要求登录态，并至少提供 `riskPointIds`。接口按当前页风险点批量返回正式绑定摘要，字段固定为 `riskPointId`、`boundDeviceCount`、`boundMetricCount`、`pendingBindingCount`；当前不再直接渲染列表列，而是作为风险点详情抽屉的秒开摘要和维护/转正入口上下文，避免主表继续承载绑定状态信息。
- `GET /api/risk-point/binding-groups/{riskPointId}` 当前要求登录态，返回目标风险点下按设备分组的正式绑定结果。`2026-04-13` 起，每组固定包含 `deviceId / deviceCode / deviceName / bindingMode / deviceCapabilityType / aiEventExpandable / extensionStatus / metricCount / metrics[]`：监测型设备继续通过 `metrics[]` 返回 `bindingId`、`riskMetricId`、`metricIdentifier`、`metricName`、`bindingSource`、`createTime`；预警型与视频类设备则通过 `bindingMode=DEVICE_ONLY`、`deviceCapabilityType` 与 `extensionStatus` 返回设备级正式绑定语义，供风险对象详情抽屉与“维护绑定”工作台统一聚合展示。
- `GET /api/risk-point/get/{id}` 与 `GET /api/risk-point/binding-groups/{riskPointId}` 当前共同构成 `/risk-point` 详情抽屉的标准读路径：前者返回风险点档案、负责人、描述与审计时间，后者返回正式绑定设备与测点/设备级分组；列表行内已有的 `riskPointName + bindingSummary` 只作为秒开首屏缓存，详情抽屉异步刷新必须以后返回的最新请求为准，避免快速切换风险点时旧响应覆盖新上下文。
- `GET /api/risk-point/devices/{deviceId}/formal-metrics` 当前要求登录态，只服务“风险绑定工作台 -> 维护绑定”里监测型设备的正式绑定读侧，返回当前设备所属产品已发布到启用中 `risk_metric_catalog` 的目录字段；响应项固定包含 `identifier / name / dataType / riskMetricId`，不再回流设备运行期属性、未入目录正式字段或纯治理语义字段。预警型和视频类设备不需要通过该接口补测点。
- `POST /api/risk-point/bindings/{bindingId}/remove` 当前要求登录态，只删除一条正式测点绑定记录，不影响同设备下其他测点。删除成功后，若该设备在当前风险点下仍有其他正式绑定，则设备卡片应继续保留；设备级正式绑定不通过该接口删除。
- `POST /api/risk-point/bindings/{bindingId}/replace` 当前要求登录态，请求体固定包含 `metricIdentifier / metricName`，并支持可选 `riskMetricId`。服务端会以事务方式完成“新增新测点绑定 + 删除旧绑定”，若目标测点与当前测点相同或已在同设备下存在正式绑定，则直接返回业务错误，避免重复；当 `riskMetricId` 存在时，也会同步校验目录指标与当前设备产品的一致性，并拒绝非 `ACTIVE` 生命周期目录。`2026-04-13` 起，该接口也只允许把测点替换为 `GET /api/risk-point/devices/{deviceId}/formal-metrics` 返回的已发布目录字段；若请求命中未入目录字段，则直接返回业务错误“当前测点未发布到风险指标目录，不能用于正式绑定”。
- `GET /api/risk-point/pending-bindings` 当前要求登录态，并至少提供 `riskPointId`；支持附带 `deviceCode`、`resolutionStatus`、`batchNo`、`pageNum`、`pageSize` 做分页过滤，返回 `risk_point_device_pending_binding` 的待治理分页结果。
- `GET /api/risk-point/pending-bindings/{pendingId}/candidates` 当前要求登录态，只允许 `PENDING_METRIC_GOVERNANCE`、`PARTIALLY_PROMOTED` 两类待治理记录查看候选；返回值会同时包含 pending 主记录摘要、推荐候选列表以及 `risk_point_device_pending_promotion` 既往转正历史。若目标记录已经是 `PROMOTED / IGNORED` 等非可治理状态，后端当前返回业务 `400` 和明确提示“当前待治理状态不支持查看候选测点”，不再落成通用 `500`。若候选已经命中风险指标目录，候选项会一并返回 `riskMetricId`，供转正与后续策略配置直接复用。
- `GET /api/risk-point/pending-bindings/{pendingId}/candidates` 自 `2026-04-18` 起，若候选 `metricIdentifier` 命中当前产品已启用风险目录，还会额外返回 `catalogRecommended=true`，并把该候选的 `recommendationLevel` 抬升为 `HIGH`、`recommendationScore` 至少提升到 `95`、`reasonSummary` 前缀补齐“已命中正式风险目录，建议优先绑定”；该接口仍只返回建议结果，不会自动执行正式绑定。
- pending 推荐候选当前固定合并三路证据：产品物模型、设备最新属性、近期 property/status 类消息日志叶子字段。日志侧会忽略 `traceId / deviceCode / timestamp` 等根元数据，只把真实属性叶子作为候选，不把事件报文或根元信息误判为测点。
- `GET /api/risk-point/pending-bindings/{pendingId}/candidates` 自 `2026-04-03` 起新增“业务重点测点治理”口径：候选会先按监测业务规范白名单、设备类型识别和字段别名归一做二次收口。默认不再把 `battery_dump_energy / signal_4g / lat / lon / sw_version` 等设备状态参数暴露为正式候选；若上报里同时存在规范字段与厂商字段，例如 `L1_SW_1.dispsX`，接口返回的 `metricIdentifier` 会统一收口为 `dispsX`，并在 `reasonSummary` 中保留原始证据字段。
- `GET /api/risk-point/pending-bindings/{pendingId}/candidates` 的目录发布副作用当前无接口字段变化，但内部实现已改为“规则组件决策可发布指标”而非服务内硬编码。当前默认规则会按场景只发布裂缝/激光 `value`、GNSS `gpsTotalX / gpsTotalY / gpsTotalZ`、深部位移 `dispsX / dispsY` 与翻斗式雨量计 `value` 到 `risk_metric_catalog`，用于保持四类治理场景的一致闭环口径。
- `POST /api/risk-point/pending-bindings/{pendingId}/promote` 当前要求登录态，只允许 `PENDING_METRIC_GOVERNANCE`、`PARTIALLY_PROMOTED` 两类待治理记录提交；请求体包含 `metrics[]`、`completePending`、`promotionNote`，其中 `metrics[]` 支持可选 `riskMetricId`。一次提交可把同一 pending 下的多个推荐测点转正为正式 `risk_point_device` 绑定，并把每个测点的处理结果写入 `risk_point_device_pending_promotion` 留痕。`2026-04-11` 起，该接口同样先走审批策略判断：直写模式会立即完成转正并收口治理工作项；审批模式只创建 `workItemId + approvalOrderId`，待审批通过后再由告警域执行器调用待治理转正服务完成正式绑定与留痕写入。
- `POST /api/risk-point/pending-bindings/{pendingId}/promote` 的单测点返回状态当前固定为 `SUCCESS / DUPLICATE_SKIPPED / INVALID_METRIC`。`SUCCESS` 表示已成功写入正式绑定；`DUPLICATE_SKIPPED` 表示正式表内已存在相同 `riskPointId + deviceId + metricIdentifier` 绑定，本次只补转正历史；`INVALID_METRIC` 表示提交的测点未命中候选列表，或虽为手工构造请求但不属于当前设备上下文允许的规范测点，不会写入正式绑定。
- `POST /api/risk-point/pending-bindings/{pendingId}/promote` 当前会先复用 pending 治理规则做字段归一。请求里的厂商别名如 `L1_SW_1.dispsX` 会按规范字段 `dispsX` 落正式表；若提交的是状态参数或不在规范白名单内的字段，则直接返回 `INVALID_METRIC`，不会绕过候选列表写脏数据。
- 待治理主记录的 `resolutionStatus` 当前按以下规则收口：`completePending=true` 且本次至少出现一条 `SUCCESS / DUPLICATE_SKIPPED` 时记为 `PROMOTED`；否则如出现成功类结果则记为 `PARTIALLY_PROMOTED`；没有成功类结果时保持原治理状态。
- `POST /api/risk-point/pending-bindings/{pendingId}/ignore` 当前要求登录态，只允许 `PENDING_METRIC_GOVERNANCE`、`PARTIALLY_PROMOTED` 两类待治理记录提交；请求体包含 `ignoreNote`，提交后会把该 pending 主记录记为 `IGNORED`，不向 `risk_point_device` 写入任何正式绑定。

  - `totalBridgeCount` = 汇总表筛选结果条数。
  - `successCount` = `bridge_status = 1`。
  - `pendingRetryCount` = `bridge_status = 0`。
  - `totalAttemptCount` = `sum(attempt_count)`。
  - `successRate` = `successCount / totalBridgeCount`。
  - `trend` 按 `DATE(last_attempt_time)` 聚合，`channelBuckets` 按 `channel_code` 聚合，`sourceTypeBuckets` 按消息 `source_type` 聚合。
- 桥接尝试明细当前由 `sys_in_app_message_bridge_attempt_log` 承载：
  - 每次桥接扫描落一条尝试记录，`attempt_no` 固定等于汇总表更新后的 `attempt_count`。
  - `bridge_status` 口径固定为 `1=本次桥接成功`、`0=本次桥接失败`。
  - `response_body` 继续沿用服务端 `1000` 字符截断口径，不额外持久化请求报文。
- 当前已接入的自动消息源：
  - `system_error`：来源于后台异常审计链路，目标角色固定为 `OPS_STAFF / DEVELOPER_STAFF / SUPER_ADMIN`，关联页面 `/system-log`，默认 24 小时过期。
  - `event_dispatch`：来源于事件派工，目标用户为 `receiveUser`，关联页面 `/event-disposal`，默认 7 天过期。
  - `work_order`：来源于工单接收 / 开始 / 完成，接收与开始发给 `receiveUser`，完成发给 `assignUser`，关联页面 `/event-disposal`，默认 7 天过期。

- `GET /api/governance/work-items` 当前支持 `workItemCode / workStatus / executionStatus / subjectType / subjectId / productId / riskMetricId / assigneeUserId / keyword / pageNum / pageSize` 查询参数，返回 first-class 控制面工作项分页。分页请求始终优先读取 `iot_governance_work_item` 已落库记录；若 `GovernanceWorkItemContributor` 超过 `5` 分钟未刷新，后端会在后台异步触发一次节流刷新，不再在请求线程内同步全量固化六类治理积压。
- `keyword` 当前统一匹配 `approvalOrderId / releaseBatchId / productKey / deviceCode / traceId` 五类主定位字段：当入参是纯数字时，后端会优先对 `approvalOrderId / releaseBatchId` 做精确匹配；`productKey / deviceCode / traceId` 则统一先做等值命中，再以模糊匹配兜底，便于同一个搜索框同时承接审批单号、发布批次号、产品标识、设备编码和 TraceId 查询。

- `2026-04-11` 起，工作项读侧会继续回传生命周期枢纽字段 `taskCategory / domainCode / actionCode / executionStatus / recommendationSnapshotJson / evidenceSnapshotJson / impactSnapshotJson / rollbackSnapshotJson`。其中审批桥接场景下，`approvalOrderId` 与 `executionStatus` 会直接反映审批主单当前进度，控制面无需再靠页面本地推断“这条任务是否仍在待审批、是否已执行、是否已驳回/撤销”。

- `GET /api/governance/work-items/{id}/decision-context` 当前用于返回单条任务的排序解释对象，固定字段包括 `workItemId`、`priorityLevel`、`priorityScore`、`problemSummary`、`reasonCodes[]`、`affectedModules[]`、`affectedCount`、`recommendedAction`、`rollbackable`、`rollbackPlanSummary`。接口只读，不承担任何 ack/block/close 或领域写动作。

- `2026-04-10` 起，`/governance-task` 中的 `PENDING_REPLAY` 工作项已支持直接打开治理链路复盘抽屉；后端读侧会优先把 `productKey / releaseBatchId` 回填到正式工作项字段，且仅在该 replay 信号唯一对应单台设备时才回填 `deviceCode`。页面会优先复用记录自身的 `traceId / deviceCode / productKey / releaseBatchId`，若任务本身未显式携带 `productKey` 且存在 `productId`，则会先调用 `GET /api/device/product/{id}` 反查产品标识，再调用 `GET /api/risk-governance/replay`。
- `POST /api/governance/work-items/replay-feedback` 当前用于显式提交复盘结论，请求体支持 `workItemId / approvalOrderId / releaseBatchId / traceId / deviceCode / productKey / recommendedDecision / adoptedDecision / executionOutcome / rootCauseCode / operatorSummary`；其中 `adoptedDecision / executionOutcome / rootCauseCode` 必填，`operatorSummary` 用于沉淀人工结论说明。
- 若请求体未显式携带 `workItemId`，后端会按 `releaseBatchId / traceId / deviceCode / productKey` 解析最近一条 `PENDING_REPLAY` 工作项，因此治理运维台复盘抽屉也可复用同一接口，不要求先把告警强行映射为本地任务卡。
- 该接口当前只回写控制面表 `sys_governance_replay_feedback` 并把命中的 replay 工作项收口为 `workStatus=CLOSED + executionStatus=CLOSED`；`feedbackJson` 至少保留 `recommendedDecision / operatorSummary / traceId / deviceCode / productKey / overrideRecommendation`，供后续 recommendation 质量复盘使用，不直接改产品、风险点或规则真相源。
- `snapshotJson` 当前继续作为上下文补充载荷，至少会按场景携带 `productKey / deviceCode / releaseBatchId / dimensionKey / dimensionLabel / metricIdentifier / coverageType` 等字段，供 `/governance-task` 在正式字段仍不足以拼出完整 replay 查询上下文时做展示和补参回退。
- `2026-04-09` 起，后台同步刷新不会覆盖人工处理痕迹：`ACKED / BLOCKED / CLOSED` 状态及人工备注会被保留，只有 `RESOLVED` 会在同一自然键再次出现时重新打开。
- `ack / block / close` 都要求登录态；请求体当前只接收可选 `comment`，用于沉淀人工处理说明。
- `/governance-task` 当前按 query 驱动读取该接口；`/products` 中已具备正式工作项语义的提示、首页管理视角治理经营卡片，以及平台治理菜单都会深链到本页。前端当前已显式提供 `确认 / 阻塞 / 关闭` 三类人工处理按钮，并在操作成功后刷新列表；`2026-04-11` 起，卡片首屏还会直接展示 `P1 / P2 / P3`，并通过“决策说明”抽屉调用 `GET /api/governance/work-items/{id}/decision-context` 展示 `reasonCodes / affectedModules / recommendedAction / rollbackPlanSummary`。

## 6. 时序接口

- `GET /api/telemetry/latest?deviceId={deviceId}`（`spring-boot-iot-telemetry`）
- `POST /api/telemetry/history/batch`（`spring-boot-iot-telemetry`）
- `POST /api/telemetry/migrate-history`（`spring-boot-iot-telemetry`）

说明：
- 当前接口已纳入活跃 Reactor 构建链路。
- 入参固定为 `deviceId`。
- 返回结构固定包含 `deviceId`、`deviceCode`、`productId`、`productKey`、`storageType`、`reportTime`、`traceId`、`properties`。
- `POST /api/telemetry/history/batch` 请求体固定为 JSON：
  - `deviceId`：必填，单设备主键。
  - `identifiers[]`：必填，需查询的指标标识集合。
  - `rangeCode`：固定支持 `1d / 7d / 30d / 365d`。
  - `fillPolicy`：当前固定为 `ZERO`，用于无数据桶补零。
- `POST /api/telemetry/history/batch` 返回结构固定包含 `deviceId`、`rangeCode`、`bucket`、`points[]`；其中 `points[]` 的单项固定包含 `identifier`、`displayName`、`seriesType`、`buckets[]`，而 `buckets[]` 固定包含 `time`、`value`、`filled`。
- `POST /api/telemetry/history/batch` 的 `points[].identifier` 当前允许返回已发布 resolver 归一后的 canonical/bare 标识，不保证一定与请求时的全路径 `identifiers[]` 字面量完全一致。例如对象洞察请求 `S1_ZT_1.sound_state`、`S1_ZT_1.battery_dump_energy`、`S1_ZT_1.sensor_state.L4_LB_1` 时，响应可分别返回 `sound_state`、`battery_dump_energy`、`L4_LB_1`；消费方必须按“先 exact path，再单一 canonical/bare 候选回绑”的口径处理，不能把这种返回值直接误判成“历史点不存在”。
- `2026-04-21` 起，`POST /api/telemetry/history/batch` 在正式字段元数据未命中时，还会继续复用运行态字段显示规则补齐 `points[].displayName`；该规则只影响历史读侧显示，不会把 raw identifier 升格为正式合同字段。
- `POST /api/telemetry/history/batch` 当前只接受 TDengine 读路径；bucket 粒度按 `rangeCode` 固定映射：`1d -> hour`、`7d/30d -> day`、`365d -> month`，并在服务端统一完成时间桶对齐与补零。接口会先按 `rangeCode` 计算查询窗口，再把窗口下传给具体历史读取链路，不再采用“全量历史 + LIMIT 命中旧点后再由服务端丢弃”的旧口径。
- 对象洞察台当前使用该接口承接产品级正式配置筛选后的单设备趋势查询，不再回退到消息日志构造折线图；`enabled=false` 或 `includeInTrend=false` 的指标不会进入 `identifiers[]`，`includeInExtension=false` 的重点趋势指标会按 `sortNo` 前置组织监测/状态分组。趋势图例组装时会先保留 `points[].displayName` / latest `propertyName` 中已经存在的人类可读名称；只有当这些名称仍是 raw alias / 空白时，前端才会回退到正式字段 `modelName / unit`。
- `POST /api/telemetry/history/batch` 在 telemetry v2 场景下当前会优先读取 `iot_raw_measure_point / iot_raw_status_point` 对应 child table 的历史点位，并按请求 `identifiers[]` 与统一窗口过滤；若兼容表 `iot_device_telemetry_point` 或 legacy stable 中仍有缺失历史，继续按现有 fallback 规则合并补齐，但 fallback 读链也必须命中同一请求窗口。
- `POST /api/telemetry/history/batch` 当前在 telemetry v2 raw 路径下会优先按 `ingested_at` 聚桶；若某条历史点缺少 `ingested_at`，再回退到 `reported_at`。`reported_at` 继续只表示设备真实采集/上报时间，不再直接作为对象洞察趋势的主时间轴。
- 当 `iot.telemetry.storage-type=tdengine` 且 `iot.telemetry.read-routing.latest-source=v2` 时，接口当前优先从 MySQL 投影表 `iot_device_metric_latest` 读取 v2 latest；若 `legacy-read-fallback-enabled=true`，缺失指标会继续从 legacy stable + `iot_device_telemetry_point` 兼容结果补齐。
- 当 `iot.telemetry.storage-type=tdengine` 且读路由仍停留在 `legacy` 时，接口继续按产品物模型中的 `specsJson.tdengineLegacy` 映射从 legacy stable 查询最新一行，再按 `metric_code` 回填为标准属性；若 `legacy-read-fallback-enabled=true`，对象洞察等历史查询会继续从 telemetry v2 raw child table 与 `iot_device_telemetry_point` 兼容表补齐缺失趋势，避免主写已切到 v2 raw 后出现“最新有值但历史折线为空”。
- v2 latest 路径下返回的 `reportTime / traceId` 优先取 `iot_device_metric_latest` 中最新 `reported_at / trace_id`；legacy 兼容结果仍按 `iot_message_log.trace_id` 与 `device_id + report_time` 的邻近时间窗口做最佳努力回补，当前兼容视图底层仍来源于物理表 `iot_device_message_log`。
- 当 `iot.telemetry.storage-type=mysql` 时，接口兼容回退到 `iot_device_property` 最新属性表。
- `POST /api/telemetry/migrate-history` 请求体固定为 JSON：
  - `deviceId`：单设备迁移目标；与 `productId` 至少传一个。
  - `productId`：按产品批量迁移历史设备；与 `deviceId` 至少传一个。
  - `batchSize`：单次读取批量大小，默认 `500`。
  - `preferLegacy`：是否强制优先 legacy stable；默认 `false`。
- `POST /api/telemetry/migrate-history` 返回结构固定包含 `source`、`migratedDeviceCount`、`scannedPointCount`、`writtenPointCount`、`latestProjectedPointCount`。
- 迁移源优先级固定为：`iot_device_telemetry_point` 标准化兼容表优先，若无历史数据或显式 `preferLegacy=true`，则退回 legacy stable/subtable 映射读取。
- 迁移结果会复用 telemetry v2 raw writer 与 latest projector，把历史点回放到 `iot_raw_measure_point / iot_raw_status_point / iot_raw_event_point` 及 `iot_device_metric_latest`；启动阶段只补 schema，不会自动执行历史全量迁移。

## 7. 兼容与注意事项

- 消息日志统一对外口径为 `iot_message_log`；当前兼容视图底层仍写入物理表 `iot_device_message_log`。
- 部分“更新/删除”接口仍采用 `POST`（如风险点、规则、预案），属于当前历史风格，重构时需评估兼容性。
- OpenAPI 文档当前可作为联调与 SDK 生成输入，但对外发布口径仍需继续遵守 `代码 -> 本文 -> 运行时 OpenAPI` 的权威顺序。
> 2026-04-07 补充：治理关键写接口权限已收口。产品契约链路中，`models` 维护仅允许 `iot:normative-library:write`；`compare` 仅允许 `iot:product-contract:govern`；`apply` 需执行人具备 `iot:product-contract:release + risk:metric-catalog:tag` 且固定复核人具备 `iot:product-contract:approve`；`rollback` 需执行人具备 `iot:product-contract:rollback` 且固定复核人具备 `iot:product-contract:approve`。`2026-04-10` 起，`/products` 页面不再要求操作者手工填写 `approverUserId`，产品合同发布 / 回滚 / 原单重提会按 `sys_governance_approval_policy` 自动解析固定复核人。`2026-04-20` 起，映射规则台账的 `submit-publish / submit-rollback` 也沿用同一口径：执行人继续分别要求 `iot:product-contract:govern / iot:product-contract:rollback`，固定复核人则按 `VENDOR_MAPPING_RULE_PUBLISH / VENDOR_MAPPING_RULE_ROLLBACK` 自动解析。`2026-04-15` 起，协议治理链路 `GET|POST /api/governance/protocol/**` 当前统一要求 `iot:protocol-governance:edit`，而 `PROTOCOL_FAMILY_* / PROTOCOL_DECRYPT_PROFILE_*` 审批动作的 `approve/reject` 要求 `iot:protocol-governance:approve`，固定复核人同样由 `sys_governance_approval_policy` 自动解析。风险策略链路中，阈值/联动/预案写接口统一要求 `X-Governance-Approver-Id`，分别执行 `risk:rule-definition:edit|approve`、`risk:linkage-rule:edit|approve`、`risk:emergency-plan:edit|approve`。设备密钥轮换接口 `POST /api/device/{id}/secret-rotate` 要求 `iot:secret-custody:rotate|approve` 双人复核。
