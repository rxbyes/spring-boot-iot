# 前端工作区与监测预警平台首页

## 目标
为当前 `spring-boot-iot` Phase 1 / Phase 2 后端能力提供一套独立的 Vue 3 前端，覆盖：
- 产品管理联调
- 设备管理联调
- HTTP 模拟设备上报
- 设备最新属性查询
- 设备消息日志查询

同时把现有“研发调试台”逐步改造成更接近商业产品的驾驶舱，服务：
- 一线工作人员
- 运维维护人员
- 开发与实施人员

并为后续这些方向预留前端扩展面：
- 点位数据图表
- 数字孪生
- 网关与子设备拓扑
- 规则 / 告警 / OTA 操作面板

## 目录
前端工作区位于：

`spring-boot-iot-ui`

说明：
- 该目录不加入 Maven reactor
- 采用 Vue 3 + Vite 风格工程结构
- 公共壳层采用“顶部双层导航 + 左侧业务菜单 + 右侧内容区”的产品化结构，围绕监测预警业务分区组织页面
- 公共壳层视觉已切换为浅色产品控制台风格，品牌统一为 `监测预警平台`
- 当前已经接入 `Element Plus` 组件体系和 `ECharts` 图表能力

## 页面结构

### 公共壳层（2026-03）
- 顶部第一行：品牌入口、页面搜索、平台状态摘要
- 顶部第二行：一级业务分区导航（平台首页 / 设备接入 / 预警处置 / 系统治理）
- 左侧：当前一级分区下的二级菜单，显示页面名称与用途说明
- 右侧：页面上下文、登录态操作和业务内容；平台首页不再展示最近访问标签，联调地址配置收纳到折叠式“接入设置”面板
- 移动端：侧边菜单切换为抽屉式展开

### 视觉一致性优化（2026-03）
- 全局 Token、Element Plus 组件覆盖和通用卡片样式已切换为浅色控制台体系
- 已优先完成驾驶舱、产品工作台、设备工作台、上报工作台、监测对象工作台等核心页面的深色面板替换
- 已完成风险监测增强页（`/risk-monitoring`、`/risk-monitoring-gis`）及监测详情抽屉的浅色统一
- 已统一驾驶舱与风险监测相关 ECharts 的 tooltip、坐标轴和主题色，避免深色图表残留

### 1. 监测预警平台首页
- 以商业产品首页方式展示平台定位、业务能力构成、交付进展和风险处置闭环
- 首页围绕告警中心、事件处置、风险点管理、分析报表和系统治理组织业务入口
- 不再展示费用与成本、代理模式、常用入口等与当前产品定位不符的内容
- 首页不再保留最近访问标签与“最近操作动态”作为主体，右侧改为待办中心
- 当前作为公开首页，未登录状态也可直接访问

**商业化改造（2026-03）**：
- Hero 区：统一使用“监测预警平台”品牌与业务化定位描述
- KPI 卡片：聚焦核心业务域、已交付模块、处置闭环和运行节奏
- 能力构成：用图表表达设备接入、预警处置、分析决策和系统治理四类能力
- 产品能力进展：明确已交付模块与实时监测 / GIS 的阶段进度
- 风险处置闭环：把风险点建档、规则编排、告警触发、事件处置、分析复盘、组织治理串成执行路径
- 角色视角：加入值班人员、运维主管、管理层三种首页 preset，首页内容随视角切换
- 核心业务入口：收敛为告警中心、事件处置、风险点管理、规则预案、分析报表五条经营主线
- 实施支撑能力：设备接入与验证不再占据主入口位，改为单独的配套支撑卡片
- 配套治理能力：突出组织、用户、角色、区域、渠道和审计日志
- 待办中心：按角色视角给出经营待办，本地 activity 仅作为最近处理痕迹插入到底部
- 首页主视图当前采用静态能力编排与本地 activity store 兜底，不把旧 `/api/cockpit/*` 演示接口作为首屏主数据源，也不伪装为真实聚合统计首页
- 首页外层不再显示标签工作区，顶部上下文条改为轻量状态条，避免产品首页继续保留后台工作区感

### 2. 产品模板中心
- 对接 `POST /device/product/add`
- 对接 `GET /device/product/{id}`
- 用于维护产品建模模板，支撑设备归属和协议引用

### 3. 设备运维中心
- 对接 `POST /device/add`
- 对接 `GET /device/{id}`
- 对接 `GET /device/code/{deviceCode}`
- 用于验证设备建档、状态字段和最近上报时间
- 后续承接远程控制、阈值管理与运维操作
- 页面已按商业产品方式重组为：
  - 设备健康状态
  - 认证基线核查
  - 运维动作建议
  - 远程运维预留位
  - 研发排障提示

### 4. 接入验证中心
- 对接 `POST /message/http/report`
- 支持快速切换属性、状态、事件三类示例 payload
- 自动推导推荐 topic
- 提供 curl 预览，方便与文档或终端联调命令对照
- 后续扩展 MQTT / TCP 接入联调

### 5. 监测对象工作台
- 对接 `GET /device/code/{deviceCode}`
- 对接 `GET /device/{deviceCode}/properties`
- 对接 `GET /device/{deviceCode}/message-logs`
- 聚合设备详情、属性快照和消息日志，作为一线风险研判核心工作页
- 前端会基于现有数据做轻量风险分级、动作建议和报告草稿生成
- 页面结构按“一线人员 / 运维人员 / 研发人员”三类角色重组信息
- 当前已增加基于最近消息日志的 `ECharts` 趋势图，后续可直接衔接时序库查询

### 6. 数据完整性校验
- 对接 `GET /device/{deviceCode}/file-snapshots`
- 对接 `GET /device/{deviceCode}/firmware-aggregates`
- 用于验证表 C.3 文件快照和表 C.4 固件分包聚合是否已经被后端消费
- 展示 Redis 快照、分包进度、重组长度和 MD5 校验结果

### 7. 未来演进蓝图
- 展示点位图表、数字孪生、拓扑与规则的预留设计位
- 说明当前页面结构与未来模块的衔接方式
- 作为商业产品路线图的前端展示位

## 运行方式
1. 启动后端：`mvn -pl spring-boot-iot-admin spring-boot:run`
2. 进入前端目录：`cd spring-boot-iot-ui`
3. 安装依赖：`npm install`
4. 启动开发环境：`npm run dev`

默认行为：
- 若 `VITE_API_BASE_URL` 为空，则前端走 Vite 本地代理
- 默认代理目标为 `http://localhost:9999`
- 开发环境推荐保持 `VITE_API_BASE_URL` 为空，这样浏览器请求会走相对路径，避免 `http://localhost:5173` 直连 `http://127.0.0.1:9999` 时触发 CORS
- 页面中的“接入设置”折叠面板支持运行时切换接入地址；当输入框清空并保存时，会恢复为当前站点同源接入
- 后端当前已提供最小 CORS 支持，默认允许 `http://localhost:*` 与 `http://127.0.0.1:*` 作为浏览器开发来源
- `/` 可作为公开首页直接访问；其他受保护页面在未登录时会跳转到 `/login`

可通过 `spring-boot-iot-ui/.env.example` 参考配置：
- `VITE_API_BASE_URL`
- `VITE_PROXY_TARGET`

当前 Node 基线：
- 前端开发与构建统一以 Node 24 为基线
- 推荐直接使用 `spring-boot-iot-ui/.nvmrc`
- 若终端仍命中旧版 Node，需要先切换到 Node 24 再执行安装、开发和构建命令

## 当前设计原则
- 不改动后端主链路
- 前后端保持独立目录和清晰职责
- 当前首页优先向商业产品形态靠拢，但不牺牲联调效率
- 所有未来功能入口都基于已有数据面板扩展，不推翻现有结构
- 文件 / 固件调试台优先服务 MQTT C.3 / C.4 联调，不提前扩展成完整 OTA 运维面板
- 关键查询状态逐步同步到 URL，方便分享调试链接和恢复现场
