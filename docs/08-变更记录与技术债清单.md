## 1. 当前有效变更摘要

### 1.1 系统内容与帮助中心

- 壳层 `通知中心 / 帮助中心` 已完成“权威资料层 + 消费层”联动治理。
- 平台治理已补齐 `/in-app-message` 与 `/help-doc` 编排页，系统内容不再只停留在壳层消费端。
- 系统内容 schema 缺失时，后端返回明确升级脚本提示，不再直接落成通用 `500`。

### 1.2 可观测与通知

- `TraceId` 已打通 HTTP、MQTT、消息日志和系统异常审计。
- 当前控制台、常规文件日志与诊断文件已统一带 `traceId`，并在最终输出前统一脱敏 `password / token / secret / authorization / clientSecret` 等敏感字段。
- 开发态控制台日志当前已补齐分类开关：MyBatis SQL 明细与 MyBatis-Spring 会话噪音默认关闭，可通过 `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SQL_ENABLED`、`IOT_OBSERVABILITY_CONSOLE_MYBATIS_SESSION_ENABLED` 按需恢复到控制台，且不再写入常规 info / error 文件日志。
- 当前已新增独立诊断文件链路：`diagnostic-sql.log`、`diagnostic-access.log`，默认关闭，专门承接慢 SQL、慢 HTTP、慢 MQTT 摘要，不再把排障摘要塞进主 `info / error` 文件。
- 调度线程与应用线程池当前已补齐 MDC / `traceId` 透传，避免只有 HTTP 主线程稳定带追踪号。
- 鉴权失败、MQTT 连接变化、协议分发失败、系统异常通知发送、高优未读桥接发送当前已统一改为摘要日志格式，默认至少带 `event / result / costMs / traceId / 主键字段`，不再重复输出长 SQL 和超长响应正文。
- 第二批业务里程碑日志当前已补齐告警、事件、工单三条主链路摘要：
  - `alarm_lifecycle`：创建 / 确认 / 抑制 / 关闭
  - `event_lifecycle`：创建 / 派发 / 反馈 / 关闭
  - `work_order_lifecycle`：接收 / 开始处理 / 完成处理
- 上述日志继续只补可检索摘要字段，不改告警 / 事件 / 工单业务逻辑，也不回流到 MyBatis SQL 明细链路。
- 第三批运维聚合当前先落最小只读视图：`/api/device/message-trace/stats` 与 `/api/device/access-error/stats`，用于补齐最近 1h / 24h 统计和 Top bucket 聚合，不引入新表、不改现有消息与失败归档写入链路。
- `链路追踪台` 当前已接入 `/api/device/message-trace/stats` 同页摘要，默认在 `notices + toolbar meta` 展示最近 1h / 24h 追踪量、失败摘要量和高频桶，不新增第二个观测页面。
- `链路追踪台` 当前已进一步扩展为同路由双模式：保留消息追踪分页与统计，同时补齐“失败归档”模式消费 `/api/device/access-error/page`、`/stats`、`/{id}`，并通过顶部模式切换和异常观测台跳转串联排障，不额外引入受菜单路径约束的新路由。
- 设备接入主链路当前已重构为显式 `UpMessageProcessingPipeline`，固定阶段为 `INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE`；控制台不再作为阶段顺序判定依据。
- `framework` 当前已新增 `message-flow` 观测模型与 Redis 存储：统一生成 `MessageFlowSession / MessageFlowTimeline / MessageFlowStep`，默认保留 `24` 小时，并通过 fingerprint 关联 MQTT 模拟发布与真实消费回流。
- 可观测接口当前已补齐 `GET /api/device/message-flow/session/{sessionId}` 与 `GET /api/device/message-flow/trace/{traceId}`，供 `/reporting` 与 `/message-trace` 共享同一条处理时间线。
- MQTT `$dp` 分发失败审计当前会补齐解码后的 `deviceCode / productKey / protocolCode / clientId` 到 `request_params`，减少“只有 topic 和 traceId”的定位盲区。
- 设备接入失败归档表 `iot_device_access_error_log` 已补齐初始化/升级脚本与真实环境 schema sync 支持，`/api/device/access-error/**` 不再只停留在代码层预留；失败详情会额外写入 `contractSnapshot`，直接落库预期协议、协议来源与设备/产品契约快照。
- 设备建档链路当前已统一补上资产契约防线：新增、编辑、批量新增、设备更换都会在服务层拦截“产品未绑定有效协议 / 节点类型”的脏建档。
- actuator 已新增 `mqttConsumer` 健康项，回放 `$dp` 报文前可直接先看 consumer 是否已连接、已订阅以及最近失败阶段。
- `system_error` 已支持自动站内消息发布，并可选桥接到外部通知渠道。
- 规则化运维告警闭环当前已落地 `observability_alert` 场景：固定评估 `system-error-burst`、`mqtt-disconnect-timeout`、`failure-stage-spike`、`in-app-bridge-failure-burst` 四类规则，按 `规则类型 + 维度键` 走 Redis `30` 分钟冷却，并在真正触发时输出结构化 `WARN` 摘要。
- 高优未读桥接当前固定为“站内消息未读后桥接到既有群渠道”，不扩展成新的多渠道通知产品。
- `/in-app-message` 已补齐“桥接效果运营”专区：治理侧可在同一页面的独立 workbench 分区内查看桥接成功率、待重试数、渠道/来源分布、桥接日志分页，以及单条桥接记录的逐次尝试明细。

### 1.3 接入与风险处置

- Phase 1~3 主链路继续作为长期稳定基线。
- 深部位移 `$dp` 自动闭环已进入可用基线，但风险监测与 GIS 仍以 [19-第四阶段交付边界与复验进展.md](./19-第四阶段交付边界与复验进展.md) 的真实环境结论为准。
- 设备接入主链路当前已调整为“设备存在 -> 产品绑定/产品存在 -> 产品归属 -> 协议”顺序校验；协议比对会在设备协议为空时回退到产品协议，并在异常里输出 `expected/actual` 明细。
- 仓库已新增升级版 `python scripts/audit-device-contracts.py`，支持“巡检 + 导出修复清单 + 按清单修复”三步治理，不再允许按前缀直接批量改库。
- 仓库已新增 `python scripts/replay-mqtt-trace.py`，用于按历史 `traceId` 回放真实 MQTT 报文，并自动输出成功消息或失败归档；当 dev 实例经由本地 SOCKS 代理访问 Broker 时，脚本也支持用 `--socks-proxy-*` 显式复用同一路径完成验收。
- `/reporting` 已收口为设备编码驱动的链路验证中心，继续沿用统一工作台语法和共享 token。
- `/reporting` 当前已在原页新增“处理时间线”面板：HTTP 发送后立即展示 timeline，MQTT 发送后按 `sessionId` 轮询直至绑定 trace，不新增第二条独立观测路由。
- `/message-trace`、`/audit-log` 与 `/system-log` 当前已统一切换到 `StandardListFilterHeader + StandardAppliedFiltersBar + StandardTableToolbar + StandardTableTextColumn + StandardPagination` 的共享台账模式；长文本继续保持单行省略与悬浮补全。
- `/message-trace` 详情抽屉当前已补齐 `StandardTraceTimeline` 共享时间线区块；Redis 时间线过期时明确显示“时间线已过期，仅保留消息日志”，不再依赖页面私有空态语法。
- `产品定义中心`、`设备资产中心`、`链路追踪台`、`异常观测台` 当前已把“详情 / 编辑 / 更多 / 追踪”类操作列间距收口到 `StandardRowActions` 共享档位；链路追踪台的“消息类型”也已恢复到共享单行省略列口径。
- 前端共享动作/文本列当前已进入“组件契约治理”阶段：`StandardActionLink` 移除 `tone` 伪接口，`StandardActionMenu` item 移除 `tone`，`StandardRowActions` 收口为 `table / card / editor` 三种变体，试点页固定宽度纯文本列继续统一回到 `StandardTableTextColumn`。
- `spring-boot-iot-ui` 当前已新增 `npm run component:guard`，独立守卫共享动作/文本列契约，避免 `tone` 回流、页面 `*actions` 区块深度覆写回流，以及试点页纯文本列退回裸 `el-table-column`。
- `spring-boot-iot-ui` 当前已新增 `useListAppliedFilters` 与 `npm run list:guard`：前者统一承接已生效筛选标签、单项移除、全部清空与高级筛选命中数，后者独立守卫纳管页的 `StandardListFilterHeader + StandardTableToolbar + StandardPagination` 列表骨架，不再允许 `search-form` 与 `text-right` 旧模板回流；本轮已把平台治理遗留的组织架构、账号中心、角色权限、区域版图补入纳管。
- 共享动作/文本列治理当前已从首批试点推广到 `风险运营`、`风险策略`、`平台治理`、`质量工场` 代表页：告警运营台、事件协同台、实时监测台、GIS 态势图、风险对象中心、阈值策略、联动编排、应急预案库，以及组织/用户/角色/菜单/区域/字典/通知/站内消息/帮助文档与自动化工场等列表页，已统一接入 `StandardButton link + StandardRowActions variant="table"`，并把新增纳管页纳入 `component:guard`。
- 上述代表页当前已进一步完成列表骨架收口：告警运营台、事件协同台、风险对象中心、阈值策略、联动编排、应急预案库，以及组织架构、账号中心、角色权限、区域版图、导航编排、数据字典、通知编排、站内消息管理、帮助文档管理等页已统一切回 `StandardWorkbenchPanel + StandardListFilterHeader + StandardAppliedFiltersBar + StandardTableToolbar + StandardPagination`；其中组织架构、区域版图继续保留“默认懒加载树 / 搜索后扁平分页”模式并把提示移入 `notices` 槽位，审计中心 / 异常观测台与链路追踪台也已同步改用共享筛选组合式函数和统一“更多筛选”逻辑。
- 日志 / 链路类台账的统计摘要当前也统一向共享工作台槽位收口：优先放入 `StandardWorkbenchPanel.notices` 和 `StandardTableToolbar.meta-items`，不再为单页额外拼第二层概览卡。

## 3. 文档治理规则

1. `docs/` 是权威资料层；帮助中心是消费层，不得反向主导研发口径。
2. 所有影响行为、接口、表结构、配置、启动方式、页面结构、验收步骤、角色可见性的改动，都必须同步更新对应文档。

4. 历史文档默认进入 `docs/archive/`；旧编号兼容入口统一收口到 [历史兼容入口索引.md](./历史兼容入口索引.md)，不再保留多份根目录短跳转页。

7. 活跃层文档命名优先使用中文文件名；archive 保留历史文件名，不因活跃层收口而做二次迁移。
8. 日常智能助手协作优先使用技能文档中的“你本人专用版六条超短清单”或 `09` 中的短模板；跨模块、跨验收、跨数据库任务再升级为长模板。
