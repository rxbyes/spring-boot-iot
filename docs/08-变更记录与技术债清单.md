## 1. 当前有效变更摘要

- 2026-03-29：仓库级协作规则已补充 Git 分支治理。`master` 明确只作为生产分支，本地禁止直接开发、提交、合并或推送；智能助手默认只允许在 `codex/dev` 开工，且每次编码前必须先检查当前分支。若任务需要临时分支，产出也必须先回到 `codex/dev`，不得直接把本地分支内容合入 `master`。

### 1.3 可观测与通知

- `TraceId` 已打通 HTTP、MQTT、消息日志和系统异常审计。
- 2026-03-28：修复设备离线超时判定误用 `last_report_time` 的问题。此前 `DeviceSessionTimeoutScheduler` 会直接按报文时间推断离线，导致设备详情里的“最近在线时间”晚于“最近上报时间”时，资产中心仍可能提前在 `last_report_time + 120s` 被收口为离线；当前已改为优先按 `last_online_time` 判定超时，仅对缺失该字段的历史记录回退到 `last_report_time`，并补充调度单测防回归。
- 2026-03-28：为设备离线超时调度补齐独立 Redis 单实例锁。此前共享环境即使已把 `online-timeout-seconds` 调整为 `7200` 秒，只要仍有旧实例或多节点同时执行 `DeviceSessionTimeoutScheduler`，设备状态仍可能被旧节点按 `120` 秒规则反复写回离线；当前已新增 `DeviceOfflineTimeoutLeadershipService`，默认使用 `iot:device:offline-timeout:leader` 竞争唯一 leader，并补充锁服务与调度跳过路径单测。
- 2026-03-28：修复设备会话服务把处理时间误写入 `last_online_time` / Redis `lastSeenTime` 的问题。此前 `DeviceStateStageHandler` 虽然先按真实 `report_time` 更新设备状态，但随后 `DeviceSessionServiceImpl.online / refreshLastSeen` 又会用 `LocalDateTime.now()` 覆盖在线时间，导致数据库出现“最近在线时间晚于最近上报时间”、并干扰资产中心排障；当前已新增带 `seenTime` 的会话接口重载，并统一透传真实上报时间，补充状态阶段与会话服务单测防回归。
- 2026-03-28：修复 `RedisInvalidReportCounterStore` 在 Spring Boot 4 / Spring 7 下的多构造器装配问题。此前该组件为测试额外暴露 `Clock` 构造器后，运行时不再满足“唯一构造器自动注入”前提，导致容器退回默认实例化并在 `mqttInvalidReportGovernance -> redisInvalidReportCounterStore` 链路上以 `No default constructor found` 启动失败；当前已显式标注生产构造器注入入口，并补充最小 Spring 上下文测试防回归。
- 当前控制台、常规文件日志与诊断文件已统一带 `traceId`，并在最终输出前统一脱敏 `password / token / secret / authorization / clientSecret` 等敏感字段。
- 2026-03-31：补齐 TDengine telemetry v2 raw schema 与历史迁移治理。此前启动初始化只补 `iot_device_telemetry_point`，`iot_raw_measure_point / iot_raw_status_point / iot_raw_event_point` 仍依赖首写惰性建表，且 raw stable 的 `metric_id` 未声明 `COMPOSITE KEY`，同一设备子表在同一 `ts` 下写入多个标准化测点时存在覆盖风险。当前已改为启动阶段统一补齐三张 raw stable，并把 `metric_id` 固化为复合主键列；同时新增手动业务接口 `POST /api/telemetry/migrate-history`，默认优先从 `iot_device_telemetry_point` 迁移历史，缺失时再按 `tdengineLegacy` 映射回放 legacy stable，并同步回灌 telemetry v2 raw 与 `iot_device_metric_latest`。
- 2026-03-31：修复 MySQL 8 握手兼容参数缺失导致的新建连接失败。真实环境 `$dp` 回放与页面访问日志显示，异常链路已从此前的“连接池耗尽”收敛为 `SQLNonTransientConnectionException: Public Key Retrieval is not allowed`，且深层失败点位于 `HikariDataSource.getConnection()` 建连阶段，而非 `DeviceMapper` / `UserMapper` 业务 SQL。本轮已为 `application-dev.yml` / `application-prod.yml` / `application-test.yml` 的默认 `IOT_MYSQL_URL` 统一补齐 `allowPublicKeyRetrieval=true`，并新增回归测试锁定该参数，避免共享环境在 MySQL 8 `caching_sha2_password` + 非 SSL 组合下再次因握手失败把 `$dp`、权限查询和定时任务统一打成 `Failed to obtain JDBC Connection`。
- 2026-03-30：为真实环境 profile 显式补齐 Hikari 连接池基线。此前 `application-dev.yml` / `application-prod.yml` / `application-test.yml` 未声明主库连接池容量，运行时继续沿用默认 `10` 连接；当站内消息慢查询、设备离线巡检与登录并发叠加时，容易在共享环境触发 `HikariPool-1 - Connection is not available`。当前主库默认已收口为 `maximum-pool-size=30`、`minimum-idle=5`、`keepalive-time=300000`、`max-lifetime=1800000`，并通过 `IOT_MYSQL_HIKARI_*` 暴露覆盖入口；dev 的 TDengine `slave_1` 也补齐了独立 Hikari 基线。
- 2026-03-30：继续收紧主库 Hikari 保活窗口，响应真实环境回放中“连接空闲约 `286994ms` 后即报 `Communications link failure`”的证据。当前 `application-dev.yml` / `application-prod.yml` / `application-test.yml` 已把 MySQL 主库默认值进一步调整为 `idle-timeout=180000ms`、`max-lifetime=240000ms`、`keepalive-time=120000ms`，目标是在共享环境中优先淘汰和探活 4 分钟内的旧连接，降低 `$dp` 回放时在 `DeviceMapper` 首跳查询借到陈旧连接的概率。
