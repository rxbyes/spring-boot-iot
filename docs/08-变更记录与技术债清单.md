## 1. 当前有效变更摘要

### 1.1 系统内容与帮助中心

- 壳层 `通知中心 / 帮助中心` 已完成“权威资料层 + 消费层”联动治理。
- 平台治理已补齐 `/in-app-message` 与 `/help-doc` 编排页，系统内容不再只停留在壳层消费端。
- 系统内容 schema 缺失时，后端返回明确升级脚本提示，不再直接落成通用 `500`。

### 1.2 可观测与通知

- `TraceId` 已打通 HTTP、MQTT、消息日志和系统异常审计。
- 2026-03-28：修复设备离线超时判定误用 `last_report_time` 的问题。此前 `DeviceSessionTimeoutScheduler` 会直接按报文时间推断离线，导致设备详情里的“最近在线时间”晚于“最近上报时间”时，资产中心仍可能提前在 `last_report_time + 120s` 被收口为离线；当前已改为优先按 `last_online_time` 判定超时，仅对缺失该字段的历史记录回退到 `last_report_time`，并补充调度单测防回归。
- 2026-03-28：为设备离线超时调度补齐独立 Redis 单实例锁。此前共享环境即使已把 `online-timeout-seconds` 调整为 `7200` 秒，只要仍有旧实例或多节点同时执行 `DeviceSessionTimeoutScheduler`，设备状态仍可能被旧节点按 `120` 秒规则反复写回离线；当前已新增 `DeviceOfflineTimeoutLeadershipService`，默认使用 `iot:device:offline-timeout:leader` 竞争唯一 leader，并补充锁服务与调度跳过路径单测。
- 2026-03-28：修复设备会话服务把处理时间误写入 `last_online_time` / Redis `lastSeenTime` 的问题。此前 `DeviceStateStageHandler` 虽然先按真实 `report_time` 更新设备状态，但随后 `DeviceSessionServiceImpl.online / refreshLastSeen` 又会用 `LocalDateTime.now()` 覆盖在线时间，导致数据库出现“最近在线时间晚于最近上报时间”、并干扰资产中心排障；当前已新增带 `seenTime` 的会话接口重载，并统一透传真实上报时间，补充状态阶段与会话服务单测防回归。
- 2026-03-28：修复 `RedisInvalidReportCounterStore` 在 Spring Boot 4 / Spring 7 下的多构造器装配问题。此前该组件为测试额外暴露 `Clock` 构造器后，运行时不再满足“唯一构造器自动注入”前提，导致容器退回默认实例化并在 `mqttInvalidReportGovernance -> redisInvalidReportCounterStore` 链路上以 `No default constructor found` 启动失败；当前已显式标注生产构造器注入入口，并补充最小 Spring 上下文测试防回归。
- 当前控制台、常规文件日志与诊断文件已统一带 `traceId`，并在最终输出前统一脱敏 `password / token / secret / authorization / clientSecret` 等敏感字段。
- 开发态控制台日志当前已补齐分类开关：MyBatis SQL 明细与 MyBatis-Spring 会话噪音可通过 `IOT_OBSERVABILITY_CONSOLE_MYBATIS_SQL_ENABLED`、`IOT_OBSERVABILITY_CONSOLE_MYBATIS_SESSION_ENABLED` 独立控制，且不再写入常规 info / error 文件日志。
- 当前已新增独立诊断文件链路：`diagnostic-sql.log`、`diagnostic-access.log`，用于承接慢 SQL、慢 HTTP、慢 MQTT 摘要，不再把排障摘要塞进主 `info / error` 文件。
- 调度线程与应用线程池当前已补齐 MDC / `traceId` 透传，避免只有 HTTP 主线程稳定带追踪号。
- 鉴权失败、MQTT 连接变化、协议分发失败、系统异常通知发送、高优未读桥接发送当前已统一改为摘要日志格式，默认至少带 `event / result / costMs / traceId / 主键字段`，不再重复输出长 SQL 和超长响应正文。
- 第二批业务里程碑日志当前已补齐告警、事件、工单三条主链路摘要：
  - `alarm_lifecycle`：创建 / 确认 / 抑制 / 关闭
  - `event_lifecycle`：创建 / 派发 / 反馈 / 关闭
  - `work_order_lifecycle`：接收 / 开始处理 / 完成处理
- 上述日志继续只补可检索摘要字段，不改告警 / 事件 / 工单业务逻辑，也不回流到 MyBatis SQL 明细链路。
- 第三批运维聚合当前先落最小只读视图：`/api/device/message-trace/stats` 与 `/api/device/access-error/stats`，用于补齐最近 1h / 24h 统计和 Top bucket 聚合，不引入新表、不改现有消息与失败归档写入链路。
- `链路追踪台` 当前已接入 `/api/device/message-trace/stats` 同页摘要，默认通过 `StandardTableToolbar.meta-items` 展示最近 1h / 24h 追踪量和失败摘要，并与同路由 `失败归档` 页签共用筛选、详情抽屉和回跳排障，不新增第二个观测页面。
- `链路追踪台` 当前已进一步扩展为同路由双模式：保留消息追踪分页与统计，同时补齐“失败归档”模式消费 `/api/device/access-error/page`、`/stats`、`/{id}`，并通过顶部模式切换和异常观测台跳转串联排障，不额外引入受菜单路径约束的新路由。
- 设备接入主链路当前已重构为显式 `UpMessageProcessingPipeline`，固定阶段为 `INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> TELEMETRY_PERSIST -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE`；控制台不再作为阶段顺序判定依据。
- `spring-boot-iot-telemetry` 已恢复为活跃 Reactor 模块；标准化属性当前会在 `PAYLOAD_APPLY` 之后优先进入 TDengine legacy stable，未映射属性再兼容回退到通用表 `iot_device_telemetry_point`，`reply` / 文件载荷 / 空属性消息直接跳过时序落库。
- 2026-03-23：真实环境复验补齐 TDengine 兼容修复，新增 `reported_at` 保留真实上报时间，`ts` 改为唯一行键时间，避免同一条报文的多指标在单表中互相覆盖；`/api/telemetry/latest` 同步改为顺序查询后按 `metric_code` 选取最新点，规避 TDengine 对 `MAX(ts)+JOIN` 的兼容限制。
- 2026-03-23：`/api/telemetry/latest` 已补齐 `int` 指标的数值回填，真实环境下如 `humidity` 不再因全局 `Long -> String` 序列化策略返回 `"59"`，而是按 JSON 数值 `59` 输出。
- 2026-03-23：修复共享 dev 环境 `/api/message/mqtt/report/publish` 重复消费事故。根因不是 Broker 单实例回环，而是多个应用实例在共享 MySQL / Redis / TDengine 环境里同时订阅同一批 MQTT 主题，导致同一次 publish 被多节点重复消费，并把不同 `traceId` 绑定回同一个 `sessionId`。当前已改为 Redis 租约式 MQTT consumer `cluster-singleton`，只允许 leader 节点订阅；standby 节点保留健康态并通过临时 publisher 客户端继续支持 publish。
- 2026-03-23：TDengine 遥测当前已切到 `legacy-compatible` 默认模式；`specsJson.tdengineLegacy` + `metadataJson.tdengineLegacy` 负责 legacy stable/列/tag 映射，`iot_device_telemetry_point` 退化为兼容回退表。
- 2026-03-24：已为 `south_rtu / south_multi_displacement / south_gnss_monitor / south_deep_displacement` 四类 live 产品新增映射治理台账与 dry-run SQL 草案 `sql/upgrade/20260324_phase5_tdengine_mapping_gap_draft.sql`。当前只固化 MySQL 侧 `iot_product_model` property 基线和 `tdengineLegacy` 占位结构，不把未经实库核验的 `stable/column` 直接写入真实环境。
- `/api/telemetry/latest` 已从硬编码示例改为真实查询：`tdengine` 模式优先读取 legacy stable，再补 `iot_device_telemetry_point` 缺失指标，并按 `iot_message_log`（兼容视图；底层物理表 `iot_device_message_log`）最佳努力回补 legacy 结果的 `traceId`；`mysql` 模式兼容回退到 `iot_device_property`。
- 2026-03-24：legacy `$dp` 标准化治理进入“双轨切换”阶段。`IotProperties` 新增 `iot.protocol.legacy-dp.family-observability-enabled`、`iot.protocol.legacy-dp.normalizer-v2-enabled`、`iot.telemetry.legacy-mapping-validate-only`；`MqttJsonProtocolAdapter` 默认继续走 v1 兼容路径，`validate-only` 模式下并行输出 `legacy_dp_normalizer_validation_diff` 差异日志，只有显式打开 `normalizer-v2-enabled` 后才切到 v2 内部实现。
- 2026-03-27：telemetry v2 foundation 已落地。dev / prod 默认新增 `iot.telemetry.primary-storage=tdengine-v2`、`iot.telemetry.legacy-mirror.enabled=true`、`iot.telemetry.read-routing.latest-source=v2`；`TELEMETRY_PERSIST` 现会先批量写入 `iot_raw_measure_point / iot_raw_status_point / iot_raw_event_point` raw stable，再异步发布 Redis 投影任务、回写 MySQL latest 投影表 `iot_device_metric_latest`，并按配置镜像 legacy stable。
- 2026-03-27：`/api/telemetry/latest` 当前已补齐 v2 latest 读路由。默认先读 `iot_device_metric_latest`，若 `legacy-read-fallback-enabled=true` 则继续从 legacy stable + `iot_device_telemetry_point` 补齐缺失指标；`mysql` 存储模式仍兼容回退到 `iot_device_property`。
- 2026-03-25：修复单设备深部位移 `$dp` 的逻辑测点前缀残留问题。当前在“无 `iot.device.sub-device-mappings`，且 payload family 仅包含 `S1_ZT_1` 与单个 `L1_SW_*` 逻辑测点”场景下，协议层会把 `L1_SW_1.dispsX / L1_SW_1.dispsY` 折叠为当前设备的 `dispsX / dispsY`；基准站拆子设备路径保持不变，适配器层回归也已补齐。
- 2026-03-24：修复 `iot_device_telemetry_point` 在“不同报文共享同一 `reported_at`”场景下的覆盖写问题。根因是通用回退表此前把 `ts` 固定为 `reported_at + pointIndex`，导致后到报文复用同一批主时间键；当前已改为全局单调唯一的写入行键时间，并把 fallback latest 查询顺序调整为 `reported_at DESC, ts DESC`。共享 dev 复验证据：`SK00E60D1306931` 在同一 `reported_at=2026-03-24T07:18:03.000Z` 下同时保留 17 条状态点与 2 条雨量点；`SK00EA0D1307992` 在同一 `reported_at=2026-03-24T07:17:38.000Z` 下同时保留 19 条状态点与 3 条裂缝点。
- `TELEMETRY_PERSIST` 当前采用非阻塞失败语义：TDengine 写失败时，该步骤会在 `message-flow` 时间线中标记 `FAILED/NON_BLOCKING_FAILURE`，但不回滚 MySQL 消息日志、最新属性和在线状态。
- `framework` 当前已新增 `message-flow` 观测模型与 Redis 存储：统一生成 `MessageFlowSession / MessageFlowTimeline / MessageFlowStep`，默认保留 `24` 小时，并通过 fingerprint 关联 MQTT 模拟发布与真实消费回流。
- 2026-03-23：修复 MQTT publish 关联成功后时间线 `sessionId` 展示不一致问题。当前同一条 timeline 的顶层 `sessionId` 与 `INGRESS.summary.sessionId` 都统一回填为 publish 返回的会话编号，不再暴露消费侧临时 session。
- `UpMessageDispatcher` 当前已降级为 legacy/compatibility 类；活文档和主链路测试矩阵不再把它作为默认消息主链路说明对象。
- `message-flow` 当前已补齐低基数指标记录：session 总量、stage 时延、correlation 结果与 lookup 命中/缺失/错误统一通过 Micrometer 归集，并由 `/api/device/message-flow/ops/overview` 暴露给接入诊断页的共享摘要能力；当前前台主要由 `/message-trace` 的链路追踪模式消费，并收口到共享工具条与详情排查语义。
- Redis 当前已新增 `iot:message-flow:recent:sessions` 最近会话索引，并通过 `GET /api/device/message-flow/recent` 向 `/reporting` 暴露最近记录恢复能力；同一批 session 数据继续供接入诊断链路复用，不新增数据库表。
- 可观测接口当前已补齐 `GET /api/device/message-flow/session/{sessionId}`、`GET /api/device/message-flow/trace/{traceId}`、`GET /api/device/message-flow/ops/overview` 与 `GET /api/device/message-flow/recent`，供接入诊断流共享同一条处理时间线、运行期摘要与最近记录恢复能力，避免新增独立 observability 路由。
- `/reporting` 当前已补齐最近提交面板与统一降级态文案：HTTP session 缺失、MQTT pending 超时、trace 过期和 Redis 存储异常都按页面上下文稳定展示，不再依赖后端额外包装缺失原因。
- 仓库当前已新增 `python scripts/run-message-flow-acceptance.py`，用于在真实环境基线上重复执行 HTTP 提交、MQTT publish 轮询和可选 expired trace 降级校验；`scripts/start-backend-acceptance.ps1` 也已在 `.mvn/settings.xml` 缺失时自动回退到 plain `mvn`。
- MQTT `$dp` 分发失败审计当前会补齐解码后的 `deviceCode / productKey / protocolCode / clientId` 到 `request_params`，减少“只有 topic 和 traceId”的定位盲区。
- 设备接入失败归档表 `iot_device_access_error_log` 已补齐初始化/升级脚本与真实环境 schema sync 支持，`/api/device/access-error/**` 不再只停留在代码层预留；失败详情会额外写入 `contractSnapshot`，直接落库预期协议、协议来源与设备/产品契约快照。
- 2026-03-27：MQTT 无效上报治理已落地首批运行时闭环。`DEVICE_NOT_FOUND` 与 `EMPTY_DECRYPTED_PAYLOAD` 当前按治理键执行 Redis 分钟桶计数与冷却抑制，`iot_device_access_error_log` 收口为失败样本归档，`iot_device_invalid_report_state` 承载最新态与命中计数，`/api/device/access-error/stats` 与 `failure-stage-spike` 改读 Redis 聚合，不再把“详情展示”和“频次统计”耦合在同一张表上。
- 设备建档链路当前已统一补上资产契约防线：新增、编辑、批量新增、设备更换都会在服务层拦截“产品未绑定有效协议 / 节点类型”的脏建档。
- actuator 已新增 `mqttConsumer` 健康项，回放 `$dp` 报文前可直接先看 consumer 是否已连接、已订阅以及最近失败阶段。
- `system_error` 已支持自动站内消息发布，并可选桥接到外部通知渠道。
- 规则化运维告警闭环当前已落地 `observability_alert` 场景：固定评估 `system-error-burst`、`mqtt-disconnect-timeout`、`failure-stage-spike`、`in-app-bridge-failure-burst` 四类规则，按 `规则类型 + 维度键` 走 Redis `30` 分钟冷却，并在真正触发时输出结构化 `WARN` 摘要。
- 高优未读桥接当前固定为“站内消息未读后桥接到既有群渠道”，不扩展成新的多渠道通知产品。
- `/in-app-message` 已补齐“桥接效果运营”专区：治理侧可在同一页面的独立 workbench 分区内查看桥接成功率、待重试数、渠道/来源分布、桥接日志分页，以及单条桥接记录的逐次尝试明细。

### 1.3 接入与风险处置

- Phase 1~3 主链路继续作为长期稳定基线。
- 深部位移 `$dp` 自动闭环已进入可用基线，但风险监测与 GIS 仍以 [19-第四阶段交付边界与复验进展.md](./19-第四阶段交付边界与复验进展.md) 的真实环境结论为准。
- 设备接入主链路当前已调整为“设备存在 -> 产品绑定/产品存在 -> 产品归属 -> 协议”顺序校验；协议比对会在设备协议为空时回退到产品协议，并在异常里输出 `expected/actual` 明细。
- 仓库已新增升级版 `python scripts/audit-device-contracts.py`，支持“巡检 + 导出修复清单 + 按清单修复”三步治理，不再允许按前缀直接批量改库。
- 仓库已新增 `python scripts/replay-mqtt-trace.py`，用于按历史 `traceId` 回放真实 MQTT 报文，并自动输出成功消息或失败归档；当 dev 实例经由本地 SOCKS 代理访问 Broker 时，脚本也支持用 `--socks-proxy-*` 显式复用同一路径完成验收。
- `/reporting` 已收口为设备编码驱动的链路验证中心，继续沿用统一工作台语法和共享 token。
- `/reporting` 当前已在原页新增“处理时间线”面板：HTTP 发送后立即展示 timeline，MQTT 发送后按 `sessionId` 轮询直至绑定 trace，不新增第二条独立观测路由。
- 2026-03-28：`接入智维` 页面结构已按“面包屑/标题 + 真业务页签 + 筛选区 + 单主内容区”统一收口。`/device-access` 改为 `资产底座 / 诊断排障` 双页签入口，删除重复介绍墙、假工作区和状态条，继续复用系统既有品牌 token 与共享列表骨架。
- 2026-03-28：`接入智维` 六个核心页已在既有两层结构基础上完成第二轮视觉精修，统一采用“静稳控制台”基线：保持当前品牌橙、蓝灰正文与浅灰背景不变，通过共享工作台、筛选头、工具条、真页签和页面字距微调提升 2026 年简约互联网控制台感，不再依赖页面私有主题色或大块说明卡。

3. 前端需继续控制页面私有样式、分页漂移和详情抽屉异步刷新提示回退风险，并继续按批次推进共享动作/文本列治理；`2026-03-24` 已将跨平台统一入口 `scripts/run-quality-gates.mjs` 卡住的主要样式漂移收口，后续仍需继续优先沿 `tokens.css` 与共享 token 口径演进；详细规则见 [15-前端优化与治理计划.md](./15-前端优化与治理计划.md)。

## 3. 文档治理规则

1. `docs/` 是权威资料层；帮助中心是消费层，不得反向主导研发口径。
2. 所有影响行为、接口、表结构、配置、启动方式、页面结构、验收步骤、角色可见性的改动，都必须同步更新对应文档。
3. 需要瘦身的长文档先完整归档，再重写当前权威版本；禁止直接删除历史上下文。
4. 历史文档默认进入 `docs/archive/`；旧编号兼容入口统一收口到 [历史兼容入口索引.md](./历史兼容入口索引.md)，不再保留多份根目录短跳转页。
5. 新增智能助手模板、技能或协作规则时，必须同步更新 [09-GPT接管提示模板.md](./09-GPT接管提示模板.md)、`docs/template/README.md` 或 `docs/README.md`。
6. 当前仍未关单的文档不确定项统一使用 `DOC-Q-*` 编号；活跃文档中禁止再出现未编号的 `【待确认】` 标记。
7. 活跃层文档命名优先使用中文文件名；archive 保留历史文件名，不因活跃层收口而做二次迁移。
8. 日常智能助手协作优先使用技能文档中的“你本人专用版六条超短清单”或 `09` 中的短模板；跨模块、跨验收、跨数据库任务再升级为长模板。
