## 1. 系统范围与模块清单

系统定位：模块化单体 IoT 网关平台，统一打通设备接入、协议适配、消息处理、设备资产管理，并在此基础上扩展风险监测预警与处置能力。当前前端一级导航已按 `接入智维 / 风险运营 / 风险策略 / 平台治理 / 质量工场` 五大工作台完成重构。

硬约束（验收前置条件）：
1. 项目名：`spring-boot-iot`
2. Base package：`com.ghlzm.iot`
3. 唯一启动模块：`spring-boot-iot-admin`
4. 不破坏模块边界，不把业务逻辑写进 Controller，不把持久化逻辑写进协议适配层

模块（按 `docs/01-系统概览与架构说明.md`）：
- spring-boot-iot-common：通用模型、异常、返回体、工具
- spring-boot-iot-framework：全局配置、安全、缓存、MyBatis 等
- spring-boot-iot-auth：认证
- spring-boot-iot-system：系统管理（用户、角色、菜单、组织、区域、字典、审计、通知渠道等）
- spring-boot-iot-device：产品、设备、属性、消息日志、驾驶舱数据
- spring-boot-iot-gateway：网关与子设备拓扑（能力闭环相关）
- spring-boot-iot-protocol：协议适配器、编解码、安全校验
- spring-boot-iot-message：接入入口与分发（HTTP/MQTT 上行，MQTT 下行发布）
- spring-boot-iot-telemetry：TDengine 时序落库、latest 查询与历史遥测存储抽象
- spring-boot-iot-rule：规则引擎（按规划逐步落地）
- spring-boot-iot-alarm：告警中心、事件处置、风险点、阈值规则、联动与预案
- spring-boot-iot-report：分析报表
- spring-boot-iot-ota：OTA（按规划逐步落地）
- spring-boot-iot-admin：应用启动与整合

35. pending 候选当前必须满足两条验收标准：一是设备状态参数如 `battery_dump_energy / signal_4g / lat / lon / sw_version` 默认不能作为正式候选；二是当上报同时出现规范字段与厂商字段时，候选主标识必须统一收口到规范字段名，例如把 `L1_SW_1.dispsX` 展示为 `dispsX`，同时在候选说明里保留原始证据字段。
36. `POST /api/risk-point/pending-bindings/{pendingId}/promote` 的结果必须显式返回单测点处理状态 `SUCCESS / DUPLICATE_SKIPPED / INVALID_METRIC`；成功类结果应写入 `risk_point_device_pending_promotion` 留痕，并在必要时更新 `risk_point_device` 正式绑定。转正写侧必须复用同一套治理规则，把厂商别名按规范字段写入正式绑定，并对状态参数或非规范字段返回 `INVALID_METRIC`。pending 主记录的状态流转要求与后端一致：`completePending=true` 且本次至少有一条 `SUCCESS / DUPLICATE_SKIPPED` 时记为 `PROMOTED`；否则如存在成功类结果则记为 `PARTIALLY_PROMOTED`；人工忽略时记为 `IGNORED`。
37. 新增/编辑抽屉中的所属区域选择当前不再在页面挂载时预加载 `/api/region/tree`；页面只在打开抽屉后请求根区域，并在展开节点时通过 `GET /api/region/list?parentId=...` 按需加载子区域，避免真实环境大区域树拖慢风险对象中心首屏。
38. `POST /api/risk-point/bind-device`、`POST /api/risk-point/bind-device-capability`、`POST /api/risk-point/unbind-device` 与 `POST /api/risk-point/pending-bindings/{pendingId}/promote` 当前必须统一返回 `GovernanceSubmissionResult`。若未命中审批策略，接口返回 `executionStatus=DIRECT_APPLIED`，领域真相需立即落库并同步收口治理工作项；若命中审批策略，则必须返回 `approvalOrderId + approvalStatus=PENDING + executionStatus=PENDING_APPROVAL`，且在审批通过前不得提前写入 `risk_point_device / risk_point_device_capability_binding / risk_point_device_pending_promotion`。
39. 审批模式下，`/risk-point` 不得跳转第二套页面，也不得在提交后立即关闭当前抽屉或工作台；页面必须在原上下文内联展示 `待审批 / 审批单 xxxx` 提示。控制面只负责调度、审批和留痕，真正的风险绑定/待治理转正写入仍必须由告警域执行器在领域内完成。

### 3.2 Phase 1 验收（HTTP 主链路）

真实环境验收（参考 `docs/真实环境测试与验收手册.md`）：
1. 使用 `application-dev.yml` 启动后端主服务
2. 同一套场景覆盖：产品新增、设备新增、HTTP 上报、属性查询、消息日志查询、非法协议、不存在设备
3. 当前交付/验收边界补充：`接入智维` 已新增研发专项自动化基线，专项首轮（Phase 1）仅要求 `6` 个现有入口各具备 `1` 条稳定主用例，并补齐 `产品 -> 设备`、`上报 -> 链路追踪`、`异常观测 -> 链路追踪` 三条跨入口联动链路；不把全部按钮、全部异常和全部视觉回归并入专项首轮（Phase 1）完成定义。

手工验收（参考 `docs/03-接口规范与接口清单.md`、`docs/真实环境测试与验收手册.md`）：
1. 初始化数据库：执行 `sql/init.sql`、`sql/init-data.sql`
2. 启动应用（`dev` 真实环境）
3. 新增产品：调用 `POST /api/device/product/add`，返回 `code=200` 且返回产品主键
4. 新增设备：调用 `POST /api/device/add`，返回 `code=200`
5. 设备治理核对：`iot_device.org_id / org_name` 应自动落为当前登录人主机构，且 `GET /api/device/{id}`、`GET /api/device/page` 返回结果需能直接看到设备所属机构字段
6. 组织范围核对：使用不同机构账号访问 `GET /api/device/page`、`GET /api/device/{deviceCode}/message-logs` 与 `GET /api/device/access-error/page` 时，设备主档与诊断结果需按各自机构范围收口；组织受限账号不再继续看到未登记名单
7. 发送 HTTP 上报：调用 `POST /api/message/http/report`，返回 `code=200`
8. 查询属性：`GET /api/device/{deviceCode}/properties`，能返回上报的属性键值
9. 查询日志：`GET /api/device/{deviceCode}/message-logs`，能返回至少 1 条日志，topic 与 payload 正确
10. 数据库核对：`iot_message_log` 新增记录，`iot_device_property` 更新，`iot_device.online_status=1`，`last_online_time/last_report_time` 刷新
11. 产品详情核对：`GET /api/device/product/{id}` 返回 `todayActiveCount / sevenDaysActiveCount / thirtyDaysActiveCount`；若产品已有在线会话明细，还会返回 `avgOnlineDuration / maxOnlineDuration`
12. 若真实环境尚未按当前 `sql/init.sql` 对齐 `iot_device_online_session`，当前允许产品详情在线时长字段返回 `null`；但设备在线状态、属性落库、消息日志与深部位移自动闭环仍应继续通过，不得因缺表把主链路判失败

错误路径验收（参考 `docs/03-接口规范与接口清单.md`、`docs/真实环境测试与验收手册.md`）：
1. protocolCode 非法时返回业务错误（示例：`bad-protocol`）
2. deviceCode 不存在时返回业务错误（示例：`missing-device`）
3. 当前账号未绑定主机构时，`POST /api/device/add`、`POST /api/device/batch-add`、`PUT /api/device/{id}` 与 `POST /api/device/{id}/replace` 应返回明确业务错误，拒绝继续维护设备归属
4. 产品停用后，`POST /api/device/add` 应返回业务错误，拒绝继续建档
5. 产品停用后，设备继续上报 `POST /api/message/http/report` 或 MQTT 标准 Topic，应返回业务错误，拒绝继续接入

### 3.5.1 Phase 4 真实环境验收前置条件

1. 后端统一以 `application-dev.yml` 启动：`mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev`；若仓库不存在 `.mvn/settings.xml`，直接省略 `-s .mvn/settings.xml`
2. 真实环境数据库应以最新 `sql/init.sql` 与 `sql/init-data.sql` 为基线；无法重建时，先执行 `python scripts/run-real-env-schema-sync.py` 或按 `docs/04-数据库设计与初始化数据.md` 人工比对对齐
3. 建议验收数据统一使用 `accept-`、`ACCEPT_` 前缀，避免污染历史演示数据
4. 页面验收优先按路由访问真实页面，再同步核对接口返回与数据库结果
5. 以 `docs/19-第四阶段交付边界与复验进展.md` 为“是否纳入交付范围”的权威口径；本节同时反映当前代码基线的实际可验能力

#### 审计日志

```sql
SELECT id, operation_module, operation_type, request_url, request_method, user_name, operation_result, result_message, operation_time
FROM sys_audit_log
WHERE operation_module IN ('organization', 'user', 'role', 'region', 'dict', 'channel', 'message.mqtt')
ORDER BY id DESC
LIMIT 20;
```

说明：
- HTTP 业务失败即使返回 `HTTP 200`，只要统一响应体中的 `code != 200`，也应在审计日志中体现为失败。
- MQTT 启动失败、订阅失败、连接断开、消息分发失败等异步异常应以 `operation_type=system_error`、`request_method=MQTT`、`user_name=SYSTEM` 写入 `sys_audit_log`。
- `sys_audit_log.result_message` 当前按 `VARCHAR(500)` 口径治理，超长异常摘要允许被自动截断并追加 `...(truncated)`；这不视为缺陷，审计能稳定落库优先于保留完整长文本。
- 历史共享库若尚未补齐 `sys_audit_log.trace_id/device_code/product_key/error_code/exception_class`，执行角色权限、导航编排、账号中心等平台治理新增/更新时，也不应再因审计表缺列导致主业务失败；审计中心与异常观测台至少要能完成列表、分页、详情与删除。
- 若目标库仍使用旧字段 `log_type` / `operation_uri`，后端查询应自动兼容；如需完整的 Trace 检索与异常定位能力，仍需按当前 `sql/init.sql` / `sql/init-data.sql` 或 `python scripts/run-real-env-schema-sync.py` 对齐审计表结构。
- 当 `iot.observability.system-error-notify-enabled=true` 且存在匹配 `system_error` 场景的通知渠道时，上述后台异常还应触发通知发送；通知失败只记录应用日志，不反向生成新的审计日志。

### 5.1.1 平台治理分页基线补充（2026-03-17）

适用页面：
- `/organization`
- `/user`
- `/role`
- `/menu`
- `/region`
- `/dict`
- `/channel`
- `/audit-log`
- `/system-log`
- `/message-trace`

基线要求：
1. 分页组件切换到第 1 页、每页 10 条时，表格实际展示记录数不得超过 10 条。
2. 账号中心、角色权限、数据字典、通知编排、审计中心、异常观测台统一通过后端 `/page` 接口获取当前页数据，禁止前端基于全量数据二次 `slice` 假分页。
3. 组织架构、区域版图、导航编排在无筛选条件时仅分页查询根节点，展开节点时再通过 `/list?parentId=...` 懒加载子节点。
4. 组织架构、区域版图、导航编排在筛选模式下返回扁平分页结果，避免为搜索再次回退到整树全量查询。
5. `/api/menu/tree` 仅继续用于角色授权树等完整树场景；导航编排页主列表不再依赖该接口做全量展示。
6. 审计日志分页必须基于数据库分页查询，不允许先查全量再在服务层 `subList`。

建议验收动作：
1. 分别在账号中心、角色权限、数据字典、通知编排页将 `pageSize` 设为 `10`，确认表格实际展示 10 条以内且翻页后请求新的 `/page` 数据。
2. 在组织架构、区域版图、导航编排页清空筛选条件后进入列表，确认首屏仅加载根节点；展开任一节点时才触发子节点请求。
3. 在组织架构、区域版图、导航编排页输入筛选条件后查询，确认返回扁平分页结果，不再触发整树全量加载。
4. 在审计中心 / 异常观测台页切换分页，确认接口请求参数包含 `pageNum`、`pageSize`，且结果总数与当前页记录数一致。
