## 1. 系统范围与模块清单

系统定位：模块化单体 IoT 网关平台，统一打通设备接入、协议适配、消息处理、设备资产管理，并在此基础上扩展风险监测预警与处置能力。当前前端一级导航已按 `接入智维 / 风险运营 / 风险策略 / 平台治理 / 质量工场` 五大工作台完成重构。

硬约束（验收前置条件）：
1. 项目名：`spring-boot-iot`
2. Base package：`com.ghlzm.iot`
3. 唯一启动模块：`spring-boot-iot-admin`
4. 不破坏模块边界，不把业务逻辑写进 Controller，不把持久化逻辑写进协议适配层

模块（按 `docs/01-系统概览与架构说明.md`）：
- spring-boot-iot-common：通用模型、异常、返回体、工具
- spring-boot-iot-framework：全局配置、安全、缓存、MyBatis 等
- spring-boot-iot-auth：认证
- spring-boot-iot-system：系统管理（用户、角色、菜单、组织、区域、字典、审计、通知渠道等）
- spring-boot-iot-device：产品、设备、属性、消息日志、驾驶舱数据
- spring-boot-iot-gateway：网关与子设备拓扑（能力闭环相关）
- spring-boot-iot-protocol：协议适配器、编解码、安全校验
- spring-boot-iot-message：接入入口与分发（HTTP/MQTT 上行，MQTT 下行发布）
- spring-boot-iot-telemetry：TDengine 时序落库、latest 查询与历史遥测存储抽象
- spring-boot-iot-rule：规则引擎（按规划逐步落地）
- spring-boot-iot-alarm：告警中心、事件处置、风险点、阈值规则、联动与预案
- spring-boot-iot-report：分析报表
- spring-boot-iot-ota：OTA（按规划逐步落地）
- spring-boot-iot-admin：应用启动与整合

### 1.1 前端业务结构与角色视角

旧版把 `设备接入 / 预警处置 / 系统治理` 直接作为一级菜单时，存在“接入调试与资产建档混杂、风险处置与策略配置耦合、工程工具稀释治理入口”的问题。当前验收基线已调整为按角色关注面组织工作台：

补充说明：
1. 当前 HTTP / MQTT 上行固定阶段为 `INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> TELEMETRY_PERSIST -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE`。
2. `TELEMETRY_PERSIST` 当前只落标准化后的 `properties`；默认 `legacy-compatible` 会优先写 legacy stable，未映射属性再回退到 `iot_device_telemetry_point`；`reply`、文件载荷和空属性消息会跳过该步骤。
3. `iot_product_model` 当前既承接设备属性 latest、测点选项和遥测映射链路消费，也承接产品维度物模型设计器的 CRUD；首轮设计器继续挂在 `/products` 内，不新增顶层路由。
4. 产品物模型设计器已于 2026-03-25 完成真实环境接口、数据库与页面验收，但仍作为下一阶段增强推进，尚未计入 Phase 4 已交付范围；详见 [2026-03-24-product-model-designer-implementation-plan.md](./superpowers/plans/2026-03-24-product-model-designer-implementation-plan.md) 与 [19-第四阶段交付边界与复验进展.md](./19-第四阶段交付边界与复验进展.md)。
5. `/devices` 当前在同一路由内承接“已登记设备 + 未登记上报名单”双名单：`GET /api/device/page` 传 `registrationStatus=1` 只看主档，传 `0` 只看未登记名单，空值则同页混排展示。
6. 未登记名单当前优先来自 `iot_device_access_error_log` 最近一条失败归档，并按设备补齐仅存在 `iot_device_message_log(messageType=dispatch_failed)` 的最近失败轨迹；共享环境若缺表或缺列，会整体回退到 `dispatch_failed` 来源。这部分记录保持只读，只用于补档与排障，不等同于正式资产主档。
7. 业务治理层面的“产品 -> 设备 -> 物模型 -> 属性/遥测 -> 消息日志”关系说明，以及变形监测字段治理标准，属于业务口径整理资产，用于指导建档、联调、风险绑定和字段治理，不自动扩大正式交付范围；当前正式交付、当前可用和规划中的边界，仍以本文和 [19-第四阶段交付边界与复验进展.md](./19-第四阶段交付边界与复验进展.md) 为准。
8. 业务人员如需查看字段治理标准表和建档联调步骤，参考 [appendix/iot-field-governance-and-sop.md](./appendix/iot-field-governance-and-sop.md)。

### 2.2 设备接入域（MQTT 上下行与 Topic 规范）

业务功能点：
1. MQTT 上行接入（直连设备标准 Topic）
2. MQTT 上行接入（历史兼容 `$dp`）
3. 网关代子设备 Topic 解析结构预留
4. 历史 `$dp` 基准站一包多测点拆分入链
5. MQTT 下行最小发布能力

Topic 规范（参考 `docs/05-protocol.md`、`docs/03-接口规范与接口清单.md`）：
1. 上行标准 Topic：`/sys/{productKey}/{deviceCode}/thing/{property|event|status}/post`
2. 上行兼容 Topic：`$dp`
3. 子设备上行预留：`/sys/{gatewayProductKey}/{gatewayDeviceCode}/sub/{subDeviceCode}/thing/{property|event|status}/post`
4. 下行标准 Topic：`/sys/{productKey}/{deviceCode}/thing/property/set`、`/sys/{productKey}/{deviceCode}/thing/service/{serviceIdentifier}/invoke`
5. 子设备下行预留：`/sys/{gatewayProductKey}/{gatewayDeviceCode}/sub/{subDeviceCode}/thing/property/set`、`/sys/{gatewayProductKey}/{gatewayDeviceCode}/sub/{subDeviceCode}/thing/service/{serviceIdentifier}/invoke`

下行 API（参考 `docs/03-接口规范与接口清单.md`）：
1. `POST /api/message/mqtt/down/publish`

补充说明：
1. `/reporting` 当前已同时支持 HTTP / MQTT 两种模拟方式；MQTT 模拟上报通过 `POST /api/message/mqtt/report/publish` 把原始字节发布到 Broker，再回流进入既有 MQTT 主链路。
2. `/reporting` 当前同页补齐“处理时间线”面板与“最近提交”辅助区：HTTP 提交后应立即显示 timeline，MQTT 提交后先显示等待回流，再按 `sessionId` 绑定 trace，并允许从最近 session 一键恢复复盘。
3. MQTT publish 回流绑定成功后，处理时间线顶层 `sessionId` 与 `INGRESS` 阶段摘要中的 `sessionId` 必须保持一致，统一展示 publish 返回的会话编号。

### 2.3 协议适配与安全校验（mqtt-json 与 `$dp` 兼容）

业务功能点（参考 `docs/05-protocol.md`）：
1. `ProtocolAdapter` 插件化适配（`getProtocolCode`、`decode`、`encode`）
2. mqtt-json 上行解码为统一 `DeviceUpMessage`
3. mqtt-json 下行编码统一 `DeviceDownMessage`
4. `$dp` 三类历史报文兼容
5. `$dp` 深部位移基准站类型 2 报文可按配置拆分为多个真实子设备上报
6. AES 解密与多商户 key 选择（`header.appId` -> `spring.cloud.aes.merchants`）
7. DES/3DES 扩展解密
8. 安全校验：签名、时间戳、nonce 防重放（缺失时兼容旧设备，不阻断主链路）
9. 二进制帧兼容：C.1/C.2/C.3/C.4（按文档定义）

调试辅助 API（参考 `docs/03-接口规范与接口清单.md`）：
1. `GET /api/device/{deviceCode}/file-snapshots`，查看 C.3 文件快照 Redis 最小持久化结果
2. `GET /api/device/{deviceCode}/firmware-aggregates`，查看 C.4 固件分包 Redis 聚合结果与 MD5 校验

## 3. 验收标准

### 3.1 全局验收标准（所有阶段通用）

1. 构建通过：`mvn -pl spring-boot-iot-admin -am clean package -DskipTests`
2. 启动通过：`mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev`；若仓库不存在 `.mvn/settings.xml`，直接省略 `-s .mvn/settings.xml`
3. API 返回体统一：`{"code":200,"msg":"success","data":...}`（参考 `docs/03-接口规范与接口清单.md`）
4. 关键原则不破坏：模块边界、Controller 无业务逻辑、协议适配层不做持久化（参考 `docs/01-系统概览与架构说明.md`）
5. 原始报文保留，最新属性独立维护（参考 `docs/02-业务功能与流程说明.md`）
6. 若任务涉及多租户、组织范围或数据权限，不得仅凭默认租户样例宣布“已完成验收”；需继续参考 `docs/13-数据权限与多租户模型.md` 区分已证实事实与待确认范围。

### 3.2 Phase 1 验收（HTTP 主链路）

真实环境验收（参考 `docs/真实环境测试与验收手册.md`）：
1. 使用 `application-dev.yml` 启动后端主服务
2. 同一套场景覆盖：产品新增、设备新增、HTTP 上报、属性查询、消息日志查询、非法协议、不存在设备

手工验收（参考 `docs/03-接口规范与接口清单.md`、`docs/真实环境测试与验收手册.md`）：
1. 初始化数据库：执行 `sql/init.sql`、`sql/init-data.sql`
2. 启动应用（`dev` 真实环境）
3. 新增产品：调用 `POST /api/device/product/add`，返回 `code=200` 且返回产品主键
4. 新增设备：调用 `POST /api/device/add`，返回 `code=200`
5. 发送 HTTP 上报：调用 `POST /api/message/http/report`，返回 `code=200`
6. 查询属性：`GET /api/device/{deviceCode}/properties`，能返回上报的属性键值
7. 查询日志：`GET /api/device/{deviceCode}/message-logs`，能返回至少 1 条日志，topic 与 payload 正确
8. 数据库核对：`iot_message_log` 新增记录，`iot_device_property` 更新，`iot_device.online_status=1`，`last_online_time/last_report_time` 刷新
9. 产品详情核对：`GET /api/device/product/{id}` 返回 `todayActiveCount / sevenDaysActiveCount / thirtyDaysActiveCount`；若产品已有在线会话明细，还会返回 `avgOnlineDuration / maxOnlineDuration`
10. 若真实环境尚未按当前 `sql/init.sql` 对齐 `iot_device_online_session`，当前允许产品详情在线时长字段返回 `null`；但设备在线状态、属性落库、消息日志与深部位移自动闭环仍应继续通过，不得因缺表把主链路判失败

错误路径验收（参考 `docs/03-接口规范与接口清单.md`、`docs/真实环境测试与验收手册.md`）：
1. protocolCode 非法时返回业务错误（示例：`bad-protocol`）
2. deviceCode 不存在时返回业务错误（示例：`missing-device`）
3. 产品停用后，`POST /api/device/add` 应返回业务错误，拒绝继续建档
4. 产品停用后，设备继续上报 `POST /api/message/http/report` 或 MQTT 标准 Topic，应返回业务错误，拒绝继续接入

### 3.3 Phase 2 验收（MQTT 上下行最小闭环）

真实环境验收（参考 `docs/真实环境测试与验收手册.md`）：
1. 基于 `application-dev.yml` 完成 MQTT 标准 Topic、`$dp`、下行发布联调
2. 协议侧安全与兼容回归通过：`MqttDeviceAesDataTests`、`MqttPayloadSecurityValidatorTest`、`MqttBinaryFormatParserTest`、`MqttJsonProtocolAdapterTest`、`MqttPayloadDecryptorRegistryTest`

手工验收（参考 `docs/真实环境测试与验收手册.md`、`docs/05-protocol.md`）：
1. 启动并开启 MQTT（共享 dev 环境口径详见 `docs/真实环境测试与验收手册.md`）
2. MQTT 上行：使用 MQTTX 向标准 Topic 发布属性上报，预期属性与日志可用，在线状态刷新
3. MQTT 下行：调用 `POST /api/message/mqtt/down/publish`，并订阅推荐下行 Topic，预期订阅端收到编码后的 JSON 下行消息
4. `$dp` 兼容：按文档给出的 3 类示例分别上报，预期都能进入统一主链路并产生属性更新与日志
5. 深部位移基准站兼容：对 `SK00FB0D1310195` 发送类型 2 `$dp` 报文，预期基准站保留原始日志，`84330701` ~ `84330696` 8 个子设备各自写入 `dispsX` / `dispsY`
6. 停用产品后，再次调用 `POST /api/message/mqtt/down/publish`，预期后端返回业务错误，拒绝继续下发
7. 连续发送同一不存在设备的 MQTT 标准 Topic 或 `$dp` 上报，预期未登记名单只保留最新记录；`iot_device_access_error_log` 允许只保留首批样本，不再按每条重复坏包都插入详情行
8. 连续发送空 payload / 解密后空 payload，预期原始空包会在 consumer 入口直接丢弃；冷却期内重复坏包不再持续写入 `dispatch_failed`、失败归档和 `system_error`，但 `GET /api/device/access-error/stats` 与 `failure-stage-spike` 仍能继续反映累计频次

2. 日期区间参数可选，格式固定为 `YYYY-MM-DD`
3. 风险趋势接口返回 `date`、`alarmCount`、`eventCount`
4. 告警统计接口返回 `total`、`critical`、`high`、`medium`、`low`
5. 事件闭环接口返回 `total`、`closed`、`unclosed`
6. 设备健康接口返回 `onlineRate`、`healthy`、`warning`、`critical`，并可解释为基于 `online_status` 与 `last_report_time` 的当前健康分层

深部位移自动闭环验收：
1. 保证深部位移子设备已存在 `risk_point_device` 绑定，且 `application-dev.yml` 开启 `iot.alarm.auto-closure.enabled=true`。
2. 发送正常小位移样本（例如 `0.0x mm`），验证消息日志、最新属性和在线状态仍正常，且不产生橙/红事件。
3. 发送超阈值样本（例如 `dispsX=25.6`），验证自动生成红色告警、事件与工单。
4. 用同一 `deviceCode + metricIdentifier + level` 在 30 分钟内重复发送，验证命中冷却窗口不重复建告警。
5. 查询 `GET /api/alarm/list`、`GET /api/event/list`、`GET /api/event/work-orders`，验证闭环结果可见。
6. 查询 `GET /api/report/alarm-statistics`，验证 `critical/high/medium/low` 统计口径与闭环结果一致。
7. 核对 `iot_alarm_record.remark`、`iot_event_record.review_notes`，确认写入 `traceId`、`metricIdentifier` 与联动规则留痕；若库中已配置深部位移场景化应急预案，则应写入对应 `planId / emergencyPlan`，若只有无关通用预案，则应保持为空而不是误挂。
8. 核对 `risk_point.risk_level`，确认按全部绑定测点的最新值取最高等级回写，而不是按最后一条上报盲目覆盖。
9. 若共享 Broker/数据库环境仍有未升级旧版 consumer 在线，可能出现同一条 `$dp` 上报同时生成新旧两套留痕；此时应先统一升级环境，再对最终闭环结果作通过/失败判定。

平台治理验收（对应表：`sys_organization`、`sys_region`、`sys_dict`、`sys_audit_log` 及用户角色相关表）：
1. 组织机构支持树结构查询与维护
2. 账号中心支持新增、编辑、禁用、重置密码、用户角色分配
3. 角色权限支持角色 CRUD、菜单授权、按钮权限授权
4. 导航编排支持菜单树查看、菜单 CRUD、父子关系维护
5. 区域版图、数据字典可配置且可被业务模块引用
6. 通知编排可查询；系统内容编排页支持站内消息与帮助文档的分页、详情、新增、更新、删除；消费端接口支持通知中心与帮助中心调用
7. 系统内容消费端最小验收接口固定包含 `GET /api/system/help-doc/access/list`、`GET /api/system/help-doc/access/page`、`GET /api/system/help-doc/access/{id}`、`GET /api/system/in-app-message/my/page`、`GET /api/system/in-app-message/my/unread-count`
8. 历史库缺少 `sys_help_document`、`sys_in_app_message`、`sys_in_app_message_read` 时，上述接口应返回“请先按最新 `sql/init.sql` / `sql/init-data.sql` 对齐初始化基线”的明确业务提示，而不是通用 `500`
9. 平台治理菜单验收必须核对 `/api/auth/me` 或 `/api/menu/tree` 中是否存在 `/help-doc`、`/in-app-message`；若缺失，应先按最新初始化基线对齐菜单与授权，再重新登录复验
10. 日志详情可查看请求参数与响应摘要，敏感字段需自动脱敏，超长内容应有截断标记；列表与详情链路中的雪花 `id` 应保持字符串语义，前端请求层需在 `JSON.parse` 前兜底保留 `id/*Id` 超大整数，避免因精度丢失打开空详情或误报“记录不存在”
11. 异常观测台支持按 `TraceId`、设备编码、产品标识、异常编码、异常类型排查，并可直接跳转链路追踪台
12. 链路追踪台支持按 `TraceId` / Topic / 设备编码检索 `iot_message_log`，并可回跳异常观测台完成双向联查
13. 链路追踪台详情抽屉应按 `traceId` 拉取处理时间线；若 Redis 已过期，必须明确提示“时间线已过期，仅保留消息日志”
13. 审计中心与异常观测台均支持按 `operationResult`（成功/失败）筛选，并在页面展示统计概览：业务侧聚合总量/今日/成功失败/活跃用户，系统侧聚合异常总量/今日/MQTT/链路数/高频模块
14. 当 `iot.observability.alerting.enabled=true` 且存在 `scenes` 包含 `observability_alert` 的有效渠道时，应能按固定 4 类规则触发运维告警通知；关闭总开关后不再触发

#### 审计日志

```sql
SELECT id, operation_module, operation_type, request_url, request_method, user_name, operation_result, result_message, operation_time
FROM sys_audit_log
WHERE operation_module IN ('organization', 'user', 'role', 'region', 'dict', 'channel', 'message.mqtt')
ORDER BY id DESC
LIMIT 20;
```

说明：
- HTTP 业务失败即使返回 `HTTP 200`，只要统一响应体中的 `code != 200`，也应在审计日志中体现为失败。
- MQTT 启动失败、订阅失败、连接断开、消息分发失败等异步异常应以 `operation_type=system_error`、`request_method=MQTT`、`user_name=SYSTEM` 写入 `sys_audit_log`。
- `sys_audit_log.result_message` 当前按 `VARCHAR(500)` 口径治理，超长异常摘要允许被自动截断并追加 `...(truncated)`；这不视为缺陷，审计能稳定落库优先于保留完整长文本。
- 历史共享库若尚未补齐 `sys_audit_log.trace_id/device_code/product_key/error_code/exception_class`，执行角色权限、导航编排、账号中心等平台治理新增/更新时，也不应再因审计表缺列导致主业务失败；审计中心与异常观测台至少要能完成列表、分页、详情与删除。
- 若目标库仍使用旧字段 `log_type` / `operation_uri`，后端查询应自动兼容；如需完整的 Trace 检索与异常定位能力，仍需按当前 `sql/init.sql` / `sql/init-data.sql` 或 `python scripts/run-real-env-schema-sync.py` 对齐审计表结构。
- 当 `iot.observability.system-error-notify-enabled=true` 且存在匹配 `system_error` 场景的通知渠道时，上述后台异常还应触发通知发送；通知失败只记录应用日志，不反向生成新的审计日志。

### 5.1.1 平台治理分页基线补充（2026-03-17）

适用页面：
- `/organization`
- `/user`
- `/role`
- `/menu`
- `/region`
- `/dict`
- `/channel`
- `/audit-log`
- `/system-log`
- `/message-trace`

基线要求：
1. 分页组件切换到第 1 页、每页 10 条时，表格实际展示记录数不得超过 10 条。
2. 账号中心、角色权限、数据字典、通知编排、审计中心、异常观测台统一通过后端 `/page` 接口获取当前页数据，禁止前端基于全量数据二次 `slice` 假分页。
3. 组织架构、区域版图、导航编排在无筛选条件时仅分页查询根节点，展开节点时再通过 `/list?parentId=...` 懒加载子节点。
4. 组织架构、区域版图、导航编排在筛选模式下返回扁平分页结果，避免为搜索再次回退到整树全量查询。
5. `/api/menu/tree` 仅继续用于角色授权树等完整树场景；导航编排页主列表不再依赖该接口做全量展示。
6. 审计日志分页必须基于数据库分页查询，不允许先查全量再在服务层 `subList`。

建议验收动作：
1. 分别在账号中心、角色权限、数据字典、通知编排页将 `pageSize` 设为 `10`，确认表格实际展示 10 条以内且翻页后请求新的 `/page` 数据。
2. 在组织架构、区域版图、导航编排页清空筛选条件后进入列表，确认首屏仅加载根节点；展开任一节点时才触发子节点请求。
3. 在组织架构、区域版图、导航编排页输入筛选条件后查询，确认返回扁平分页结果，不再触发整树全量加载。
4. 在审计中心 / 异常观测台页切换分页，确认接口请求参数包含 `pageNum`、`pageSize`，且结果总数与当前页记录数一致。

### 5.1.2 顶部账号与导航收口补充（2026-03-17）

适用页面：
- 全部采用 `AppShell` 壳层的登录后页面

基线要求：
1. 页面标题区不再提供显式 `退出登录` 按钮，个人操作统一收口到右上角头像菜单。
2. 顶部头像菜单需至少提供：账号中心、实名认证说明、登录方式、修改密码、退出登录。
3. `平台治理` 分组页面默认不展示最近访问标签，避免与左侧导航重复表达当前定位。
4. 页面头部不再展示 `接入设置` 展开区，避免与业务内容竞争视觉焦点。
5. 左侧导航统一保留轻量分组摘要（分组标题、菜单数量、页面类型）与标准二级菜单列表；二级导航以菜单名称为主，说明通过悬浮提示提供；内容区标题栏统一采用“面包屑 + 标题说明 + 状态卡”的标准页头，不再使用厚卡片式页头。
6. `GET /api/auth/me` 返回的 `authContext` 需包含当前用户手机号、邮箱、账号类型、实名状态和可用登录方式，供头像菜单与账号抽屉直接渲染。
7. 一级导航点击后应优先进入分组首页（`/device-access`、`/risk-disposal`、`/risk-config`、`/system-management`、`/quality-workbench`），再从概览页进入具体功能页面。
8. 告警运营台、事件协同台、实时监测台、GIS态势图的列表操作条、提示条、空态卡与资源卡需统一采用浅色轻卡片风格，不再使用厚重工具条或强营销式装饰。
