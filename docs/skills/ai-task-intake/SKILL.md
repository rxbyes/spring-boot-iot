# 智能助手任务提问收口

当用户还不准备直接实现业务代码，而是需要先把任务说清楚、减少智能助手的理解成本和文档扫描范围时，使用这个技能。

典型请求：
- “帮我整理怎么和助手说需求”
- “这个错误该怎么提给助手”
- “给我一个最短模板”
- “我该先让助手分析还是直接修改”
- “怎么减少文档阅读量”

本技能只负责任务收口和提示词整理，不负责直接实现业务功能。

## 核心目标

把用户的模糊想法收口成 5 个输出：

1. 任务类型
2. 执行方式：先分析还是直接修改
3. 一张短任务卡
4. 合适的证据附片
5. 最少补读文档

## 固定项目前提

始终按 `spring-boot-iot` 当前规则处理：
- 从 `AGENTS.md` 里的最小阅读集起步
- 不让助手全量扫描整个 `docs/` 目录
- 不把 `docs/archive/*` 当主输入，除非任务明确是历史追溯
- 真实环境验收仍以 `spring-boot-iot-admin/src/main/resources/application-dev.yml` 为准
- 不建议任何 H2 回退验收路径

## 收口流程

按下面的顺序处理任务：

1. 先判断任务类型
2. 再判断要先分析还是直接修改
3. 再写最短可用任务卡
4. 只有在能降低歧义时才补证据
5. 最后告诉助手只需要补读哪些文档

## 第一步：判断任务类型

优先只选一个主类型：
- `分析任务`
- `新增功能`
- `错误修复`
- `功能优化`
- `前端样式优化`
- `接口改造`
- `数据库任务`
- `验收 / 回归`

如果任务跨多个方面，也只保留一个主类型，把其他内容写进“影响范围”，不要并行开两套模板。

## 第二步：判断执行方式

以下情况使用 `先分析，不改代码`：
- 用户自己也不确定问题在哪一层
- 任务可能同时涉及前端、后端、配置、SQL 或环境
- 风险较高，范围还不清楚
- 用户明确想先看原因、方案或范围

以下情况使用 `先总结理解，然后直接开始修改`：
- 目标已经明确
- 用户已经能判断这是新增、修复、优化还是样式问题
- 验收标准已经足够清楚

## 第三步：使用五行任务卡

默认任务卡：

```md
任务类型：
目标结果：
当前现象 / 业务背景：
影响范围：
验收标准：
```

使用规则：
- `任务类型` 只保留一个主类型
- `目标结果` 只写最终想看到什么，不写实现过程
- `当前现象 / 业务背景` 只写事实，不先给结论
- `影响范围` 尽量写到页面、接口、模块、表、脚本、文档
- `验收标准` 尽量能观察、能验证、能打勾

如果证据很多，不要把日志和样例塞进任务卡正文，直接追加后面的证据附片。

## 第四步：只在必要时补证据

### 什么时候补日志

以下情况强烈建议补日志：
- 后端 `500`
- 启动失败
- 登录或鉴权失败
- 协议 / MQTT 解析问题
- 同样操作在不同环境结果不同
- 任何不是纯视觉问题的异常

日志附片：

```md
日志证据：
- 时间：
- 环境：
- TraceId：
- 接口 / 任务：
- 异常栈 / 日志片段：
```

### 什么时候补截图

以下情况强烈建议补截图说明：
- 布局问题
- 间距、层级、排版漂移
- 抽屉、弹窗、表格、分页不一致
- “不好看”“不统一”这类前端反馈

页面附片：

```md
页面证据：
- 页面 / 路由：
- 浏览器 / 终端：
- 操作步骤：
- 当前表现：
- 期望界面 / 交互：
- 截图说明：
```

### 什么时候补接口证据

以下情况强烈建议补接口证据：
- 接口异常
- 返回结构变化
- 新增接口
- 鉴权、分页、筛选、详情加载问题

接口附片：

```md
接口证据：
- 路径：
- 方法：
- 请求参数：
- 响应结果：
- 登录令牌 / 鉴权方式：
- 报错信息：
```

### 什么时候补 SQL 证据

以下情况强烈建议补 SQL 证据：
- 结构漂移
- 缺字段
- 初始化或升级脚本失败
- 明显与数据或表结构相关的运行异常

SQL 附片：

```md
SQL 证据：
- 涉及表 / 视图：
- 涉及字段：
- 数据样例：
- 升级脚本 / 初始化脚本：
- 是否真实环境问题：
```

## 第五步：选择最少补读文档

先从 `AGENTS.md` 中的最小阅读集起步，再按任务补读：

- `新增功能`：优先补 `docs/02` `docs/03` `docs/04`
- `错误修复`：按层级补读
  - 接口 / 鉴权 / 返回问题：`docs/03`
  - 启动 / 配置 / 环境问题：`docs/07`
  - 验收 / 回归问题：`docs/05` `docs/真实环境测试与验收手册.md` `docs/21-业务功能清单与验收标准.md`
  - 协议问题：`docs/05-protocol.md` `docs/14-MQTTX真实环境联调手册.md`
- `功能优化`：优先补 `docs/02` `docs/03`，只在交付边界确实受影响时再补 `docs/21` 或 `docs/19`
- `前端样式优化`：优先补 `docs/06` `docs/15`，不要默认去读 `docs/04`
- `接口改造`：优先补 `docs/03` `docs/04`
- `数据库任务`：优先补 `docs/04` `docs/07`，如果验收受影响再补 `docs/21`
- `验收 / 回归`：优先补 `docs/05` `docs/真实环境测试与验收手册.md` `docs/21-业务功能清单与验收标准.md`

2. 只补一句目标和一句现象
3. 如有证据，再追加日志 / 截图 / 接口 / SQL

## 判断表

- 用户自己也不知道问题在哪一层：先用“先分析，不改代码”
- 目标已经明确：直接用对应任务口令
- 纯视觉问题：先补截图说明，再决定是否补日志
- 后端或链路问题：先补日志，再做推断
- 接口问题：至少补接口路径和响应结果
- 结构问题：至少补表、字段或脚本信息
- 文档重复、入口混乱、阅读成本高：直接用“文档治理 / 资料收口”
- 启动失败、共享环境异常、MQTT 不通或登录报错：直接用“真实环境排障”
