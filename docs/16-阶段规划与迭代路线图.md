# 16 阶段规划与迭代路线图

> 权威级别：一级权威。
> 上游来源：`19-第四阶段交付边界与复验进展.md`、`08-变更记录与技术债清单.md`、当前代码与验收基线。

## 1. 当前阶段定位

- Phase 1~3 主链路已经形成长期稳定基线。
- Phase 4 当前重点不再是“继续扩散功能面”，而是“收口已实现能力、完成真实环境复验、补齐治理闭环”。
- Phase 4 的交付边界、阻塞项和真实环境复验状态，继续以 [19-第四阶段交付边界与复验进展.md](./19-第四阶段交付边界与复验进展.md) 为唯一阶段权威。

## 1.1 当前工作全盘总结（2026-04-24）

### 稳定基线

- Phase 1~3 设备接入主链路已长期稳定，HTTP / MQTT 上报统一进入固定 Pipeline。
- `spring-boot-iot-admin` 仍是唯一启动模块，父 `pom.xml` 当前活跃 `12` 个模块。
- 真实环境验收继续固定使用 `spring-boot-iot-admin/src/main/resources/application-dev.yml` 或其环境变量覆盖。
- telemetry v2 已进入默认基线，dev / prod 默认写 TDengine v2 raw，并保留 MySQL latest 投影与 legacy fallback。
- 数据库结构真相源固定为 `schema/`，对象退场与 seed 治理真相源固定为 `schema-governance/`。

### 已完成并进入当前基线

- 产品定义中心已拆为 `/products` 主列表和 `/products/:productId/overview`、`devices`、`contracts`、`mapping-rules`、`releases` 五段详情子路由。
- 契约字段、映射规则、发布批次、风险指标目录、对象洞察重点指标和运行态显示规则已形成最小治理闭环。
- 风险监测、GIS 态势图、告警、事件、风险策略、报表分析、系统治理和系统内容治理已纳入 Phase 4 真实环境基线。
- 质量工场已形成业务验收台、自动化治理台的可用链路。
- `/device-onboarding` 已具备统一接入案例、模板包、验收触发和批量编排的第一阶段能力。
- 治理控制面已具备治理任务台、治理运维台、治理审批台、权限与密钥治理页的最小闭环。

### 部分完成或仍需推进

- 零代码接入仍是阶段性底座能力，不代表任意设备已完成全自动接入。
- 对象洞察按设备拓扑角色分场景展示仍处于推进中，后续需继续收口父采集器、子设备、单台设备的快照和趋势差异。
- 设备资产中心的能力操作台账和命令执行链路已完成设计，仍需按产品能力类型继续实施和复验。
- 统一 CI / 契约测试流水线仍未成为正式基线，当前仍以本地质量门禁和共享真实环境验收为主。
- 风险指标桥层、协议治理和映射规则治理仍需扩大设备族覆盖，并继续沉淀运行时自动生成与恢复编排能力。

### 明确未完成或不纳入当前承诺

- GIS SDK、第三方地图底图、热力图和驾驶舱内嵌风险监测子模块不属于当前已交付范围。
- 任意设备厂商的无人值守零代码接入不属于当前已交付范围。
- 首页完整驾驶舱不等同于所有角色视角真实环境验收完成。
- 多机构成员模型、角色授权快照、统一数据权限全域推广仍属于后续演进。

### 下一轮建议主线

1. 设备资产能力操作：按产品类型展示可执行能力、参数模板、常用命令和命令台账。
2. 对象洞察拓扑角色化：优先完成读侧止血，再推进多 profile 配置治理和存量巡检。
3. 零代码接入扩面：沿协议治理、映射规则、模板包和正式合同发布继续推进，不新增平行真相源。
4. 质量门禁工程化：把当前本地脚本和真实环境验收逐步固化为可复用的团队级流水线。
5. 文档减负：入口文档保持摘要化，过程文档进入 [superpowers/README.md](./superpowers/README.md) 索引，不再进入最小阅读集。

## 2. 未来两个迭代的主线

### 2.1 第一主线：Phase 4 收口

1. 完成风险监测与 GIS 的真实环境复验。
2. 继续维持 `application-dev.yml` 作为唯一验收基线，不回退旧 H2 路线。
3. 清理环境阻塞、schema 漂移和命名兼容问题，形成可稳定复用的交付基线。

### 2.2 第二主线：Phase 5 入口

1. 以文档体系、日志治理、帮助中心治理和质量工场为支点，建设更稳定的研发与运维协同底座。
2. 把产品定义中心、设备资产中心、系统内容治理、质量工场等“已进入可用闭环”的模块，持续从演示态收口到工程化长期维护态。
3. 在不破坏模块化单体边界的前提下，推进后续能力增强。

## 4. 重点工作流

### 4.1 研发与交付

- 产品定义中心和设备资产中心继续围绕“稳定体验 + 真实环境基线 + 文档闭环”推进，而不是重新扩张概念。
- 风险处置与系统治理继续优先做“已交付能力收口”，避免阶段边界再次模糊。

### 4.2 运维与排障

- 统一采用 [11-可观测性、日志追踪与消息通知治理.md](./11-可观测性、日志追踪与消息通知治理.md) 的 TraceId 和通知治理路径。
- 真实环境阻塞统一在阶段文档和验收记录中显式标记，不用降级方案替代结论。

### 4.3 质量与自动化

- 单元测试、协议回归、真实环境验收和浏览器巡检继续并行，但系统通过结论仍以真实环境为准。
- 视觉回归、配置驱动计划和失败问题历史台账继续作为质量工场的工程化入口，而不是单次活动。

### 4.4 平台治理与账号中心后续演进建议（待整体链路打通后实施）

触发前提：

- 已完成系统治理第一批查询侧收口，并把同一套 `tenant + org + dataScopeType` 继续稳定推广到设备、告警、事件、报表等核心主链路。
- 设备归属、风险点绑定、消息诊断、账号中心与角色治理在真实环境下形成连续闭环，且不再依赖临时隔离账号或一次性补录操作才能复验。
- 历史数据补齐、逻辑删除、唯一键复用、跨模块引用关系等基础治理问题已有稳定操作手册或自动化脚本，不再阻塞日常运维。

建议演进方向：

1. 账号中心从“当前登录人主机构 + 角色摘要”升级为“主机构、参与机构、有效角色、数据范围预览、关键授权变更历史”的统一治理入口。
2. 用户模型从单一 `sys_user.org_id` 演进为“主机构必填 + 多机构成员关系可选”的双层模型，但前提是不破坏当前已落地的主机构归属规则和既有验收口径。
3. 角色模型从“菜单/按钮权限 + dataScopeType 声明”演进为“角色模板 + 数据范围声明 + 关键对象授权快照”，用于支撑更细粒度的跨机构协同，但不得先于主链路稳定前落地。
4. 数据权限执行从首批治理模块继续扩展到告警中心、事件处置、报表分析、风险监测等模块，形成统一的服务层过滤表达，而不是每个页面单独拼装查询条件。
5. 账号中心补齐“关联机构切换预览、角色变更影响预览、设备/风险点/工单影响面提示”，避免机构调整后只改身份表、不校验业务对象归属。
6. 增加治理级批处理工具，统一处理历史设备归属补齐、风险点冲突绑定排查、机构迁移后的权限重算和异常审计补录，而不是继续依赖人工 SQL 清洗。
7. 对主机构变更、角色升降权、跨机构授权、关键绑定解绑等动作补齐审批留痕和审计回放，确保后续扩面时仍能追溯“谁在什么时间改了什么”。

实施约束：

- 在上述前提达成前，本轮已落地规则继续保持不变：设备归属仍以当前登录人主机构为准，风险点绑定仍坚持“一设备一风险点”，前端不开放机构手选回填。
- 后续若正式启动该批演进，必须同步更新 `02 / 03 / 04 / 13 / 21`，必要时再补 `08` 的技术债状态，不允许只改页面原型或局部表结构。

## 5. 当前不纳入本轮规划的事项

- 不引入新的架构形态，继续保持模块化单体。
- 不把通知中心扩展成新的多渠道产品。
- 不在 Phase 4 未完全收口前，把风险监测、GIS 或驾驶舱演示态能力直接宣告为“全面已交付”。
