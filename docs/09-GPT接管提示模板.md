# 09 GPT 接管提示模板

> 文档定位：智能助手接手项目时的标准模板、任务卡和协作提示。
> 适用角色：研发、测试、文档维护者、智能助手协作人员。
> 权威级别：一级权威。
> 上游来源：`AGENTS.md`、核心权威文档、当前交付边界和文档体系。
> 下游消费：`docs/template/README.md`、技能文档、日常任务提问模板。
> 变更触发条件：最小阅读集、任务分流、帮助中心治理、可观测路径、阶段规划或智能助手协作方式变化。
> 更新时间：2026-03-24

本文件是日常向智能助手发任务的统一入口。默认目标不是写一大段自然语言，而是用更短的任务卡让模型先读懂你的目标，再按最小阅读集和任务补读矩阵进入编码。

如果你已经明确知道任务场景，只想直接复制现成提示词，优先查阅 [17-智能助手任务发起模板速查.md](./17-智能助手任务发起模板速查.md)；本文件继续作为任务卡、证据附片和长模板母版。

目标：
- 让模型优先读取最小阅读集，而不是全量扫描 `docs/`
- 让你先给出清晰任务边界，而不是把问题丢给模型自己猜
- 让高频任务优先走“短任务卡”，复杂任务再走“长模板”
- 明确项目硬约束、交付边界、真实环境基线与文档更新规则
- 让 `docs/template/README.md` 只做快捷索引，不再维护第二份模板正文
- 如果需要把“提问顺序清单”作为可复用技能使用，可参考 `docs/skills/ai-task-intake/SKILL.md`

## 1. 高效交互原则

日常任务统一按以下顺序与智能助手交互：

1. 先选任务类型。
2. 再写一张 `5` 行任务卡。
3. 如有证据，再追加日志 / 截图 / SQL / 接口样例。
4. 明确“只按任务补读，不要全量扫描 `docs/`”。

高价值输入：
- 报错日志、异常栈、TraceId
- 接口路径、请求方式、请求参数、响应样例
- 页面路由、截图说明、复现步骤
- SQL、表名、字段名、升级脚本
- 预期结果、验收标准、不能改动的边界

低价值输入：
- “帮我优化一下”
- “你先看看哪里有问题”
- “把整个仓库和所有文档都分析一下”
- 只有目标，没有现象、边界或验收标准

固定提醒：
- 默认最小阅读集不变，仍以 `AGENTS.md` 为准。
- 真正要减的是“补读范围”和“理解成本”，不是绕过项目硬约束。
- 如果任务已经很明确，不要再额外投喂历史归档文档。

## 2. 统一五行任务卡

所有日常任务都可以先用下面这张卡片起手：

```md
任务类型：
目标结果：
当前现象 / 业务背景：
影响范围：
验收标准：
```

使用规则：
- `任务类型` 只写一种主类型，例如：`新增功能 / 错误修复 / 功能优化 / 前端样式优化`
- `目标结果` 写你最终想看到什么，不写实现过程
- `当前现象 / 业务背景` 只写事实，不先下结论
- `影响范围` 尽量写到模块、页面、接口、表、脚本、文档
- `验收标准` 尽量可观察、可验证、可打勾

如果证据较多，不要把它们塞进任务卡正文，直接在任务卡后追加第 `4` 节中的证据附片即可。

## 3. 最小补读矩阵

所有任务默认先读：
`README.md`、`docs/README.md`、`docs/01-系统概览与架构说明.md`、`docs/02-业务功能与流程说明.md`、`docs/03-接口规范与接口清单.md`、`docs/04-数据库设计与初始化数据.md`、`docs/07-部署运行与配置说明.md`、`docs/08-变更记录与技术债清单.md`。

在此基础上，再按任务类型补读：

| 任务类型 | 额外优先补读 | 何时继续补读 |
|---|---|---|
| 新增功能 | `02 / 03 / 04` | 涉及交付边界或验收再补 `21`；涉及前端再补 `06 / 15`；涉及 Phase 4 边界再补 `19` |
| 错误修复 | 根据层级补 `03`、`07`、`05 / 真实环境测试与验收手册 / 21`、`05-protocol / 14` | 接口 / 鉴权 / 返回问题优先补 `03`；运行 / 环境 / 配置问题优先补 `07`；验收 / 回归问题补 `05 / 真实环境测试与验收手册 / 21`；协议问题补 `05-protocol / 14` |
| 功能优化 | `02 / 03` | 涉及交付能力或范围变化再补 `21`；涉及阶段边界再补 `19`；涉及前端体验再补 `06 / 15` |
| 前端样式优化 | `06 / 15` | 页面属于 Phase 4 或验收页面再补 `19 / 21`；只有在样式问题背后涉及接口字段、结构或数据约束时，才继续深挖 `03 / 04` |

补读边界：
- 不要因为“可能有关”就把 `docs/` 全扫一遍。
- 不要默认把 `docs/archive/*` 当主输入。
- 只要当前任务不涉及协议、验收、阶段边界或数据权限，就不要顺手加读对应文档。

## 6. 标准模板（复杂任务通用）

- 项目名必须保持：`spring-boot-iot`
- 基础包必须保持：`com.ghlzm.iot`

- 唯一真实环境验收基线：`spring-boot-iot-admin/src/main/resources/application-dev.yml`
- 可以通过环境变量覆盖该文件中的连接配置
- 不允许回退到旧 H2 验收路径、独立 H2 脚本或 H2-only 验收方案
- Windows 10 共享环境常见工作区示例路径可能是：`E:\idea\ghatg\spring-boot-iot`
- 其他系统路径可能不同，不要把某一个绝对路径当成唯一合法路径；优先按脚本位置、仓库根目录或当前环境推导路径
- 涉及后端联调、重启、发布验证时，必须优先复用现有已启动服务（默认 `9999`），不要改用容器、沙盒或其他替代运行链路
- 若后端本次没有新增更新内容，不要重复拉起其他运行方式；直接复用现有服务完成验证
- 若后端服务未启动或需要重启，先停止 `9999` 服务，再按下列命令执行打包和启动
- Windows 环境下执行：
  - `mvn -s .mvn/settings.xml clean install -DskipTests`
  - `mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run "-Dspring-boot.run.profiles=dev"`
- Mac/Linux 环境下执行：
  - `mvn clean install -DskipTests`
  - `mvn -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev`

## 8. 推荐投喂策略

推荐顺序：

1. 如果你已经非常明确要做什么，优先直接使用第 `5.5` 节中的“你本人专用版六条超短清单”。
2. 如果六条超短清单装不下，再用第 `5.5` 节里的十条速用口令。
3. 如果你只知道大方向，还没法一句话说清，再投喂第 `2` 节的统一任务卡，或直接使用第 `5` 节对应的高频任务卡模板。
4. 如有证据，再追加第 `4` 节的证据附片。
5. 如果任务跨模块、跨验收、跨数据库或背景复杂，再升级为第 `6` 节标准模板或第 `9` 节执行型长模板。

不推荐：

- 一开始就让模型“扫描整个仓库所有文档”
- 把历史归档文档和核心文档混在一起投喂
- 不加任务边界就让模型自由发挥
- 没有证据时先写一大段分析结论

## 10. 如何选择模板

可按以下规则直接选用：

- 日常新增功能：优先用 `5.1`
- 日常错误修复：优先用 `5.2`
- 日常功能优化：优先用 `5.3`
- 日常前端样式优化：优先用 `5.4`
- 要重构资料、整理知识库、归档文档：用 `9.1`
- 任务背景复杂、跨模块实现新功能：用 `6` 或 `9.2`
- 已经有明确报错、失败日志、回归缺陷：用 `9.3`
- 主要工作在 `spring-boot-iot-ui` 页面、交互、样式：用 `9.4`
- 主要工作是新增 / 修改 / 修复后端接口：用 `9.5`
- 主要工作是 `sql/init.sql`、`sql/init-data.sql` 或 schema 对齐：用 `9.6`

如果任务同时跨两类场景，推荐顺序是：
1. 先贴第 `2` 节任务卡
2. 再加第 `4` 节证据附片
3. 最后按需升级为第 `6` 节或第 `9` 节长模板
