# 项目交付培训与产品使用说明书设计

## 1. 背景

当前 `spring-boot-iot` 已经形成面向真实项目交付的五大工作台基线：

- `接入智维`
- `风险运营`
- `风险策略`
- `平台治理`
- `质量工场`

系统能力已经覆盖设备接入、风险运营、治理控制面、审批闭环与业务验收，但面向客户交付培训时，现有权威文档仍主要服务研发、运维、测试与验收，不足以直接回答以下问题：

- 管理者登录后先看什么
- 一线使用者每天主要在哪些页面工作
- 运维者遇到接入异常时先查哪里
- 研发者在交付现场如何支撑联调、排障和回归
- 多角色如何围绕同一条“建档 -> 联调 -> 风险纳管 -> 审批治理 -> 验收复盘”主线协同

因此需要补一份面向内部培训与项目交付场景的《产品使用说明书》，把现有能力从“模块说明”转化为“角色可用、培训可讲、现场可查”的交付手册。

## 2. 目标

- 形成一份面向内部培训和客户交付培训的正式说明书
- 说明书以“角色使用 + 项目协同 + 培训讲义 + 排障速查”四位一体方式组织
- 明确管理者、一线使用者、运维者、研发者四类核心角色的：
  - 默认首页
  - 常用页面路径
  - 典型任务
  - 协同边界
  - 最小排障入口
- 明确系统当前交付边界，避免对客户形成“任意设备零代码接入”或“所有角色完全自治”的误解
- 从架构师视角与客户视角总结当前流程问题，并给出可执行的改进建议

## 3. 非目标

- 不替代 [docs/03-接口规范与接口清单.md](../../03-接口规范与接口清单.md) 的完整接口字典
- 不替代 [docs/07-部署运行与配置说明.md](../../07-部署运行与配置说明.md) 的完整部署与环境手册
- 不替代 [docs/11-可观测性、日志追踪与消息通知治理.md](../../11-可观测性、日志追踪与消息通知治理.md) 的完整排障治理规则
- 不把说明书写成研发设计文档、数据库字典或 API 白皮书
- 不承诺当前系统已经具备“全角色零门槛、零维护、全场景自助”

## 4. 约束与写作原则

- 文档必须基于真实环境基线与当前权威文档，不得引入 H2、废弃验收路径或历史失效流程
- 文档必须显式写明五大工作台和四类核心角色的映射关系
- 文档应使用客户可理解的业务语言组织正文，但允许在内部培训版中保留页面路径、演示账号、关键接口与排障入口
- 文档必须主动管理预期，明确：
  - 当前支持的治理切片
  - 当前控制面与领域页的边界
  - 当前排障链路的推荐顺序
  - 当前仍需研发/运维支持的场景
- 文档成稿后应作为 `docs/` 活跃权威层的新文档维护，而不是进入 `archive/`、`template/` 或帮助中心消费层

## 5. 目标读者与角色边界

### 5.1 核心读者

- 管理者
- 一线使用者
- 运维者
- 研发者

### 5.2 补充读者

- 产品经理
- 测试人员
- 项目经理
- 验收人员

### 5.3 四类核心角色的培训边界

- 管理者：关注经营指标、治理任务、审批决策与复盘闭环，不要求掌握协议细节
- 一线使用者：关注实时监测、告警处置、事件协同与业务验收，不要求掌握底层接口与库表
- 运维者：关注建档、联调、链路追踪、失败归档与运行维护，需要掌握最小排障闭环
- 研发者：关注协议联调、规则验证、治理扩展与自动化回归，需要承担复杂问题兜底

## 10. 培训顺序设计

建议培训讲解顺序为：

1. 管理者
2. 一线使用者
3. 运维者
4. 研发者
5. 联合演练

原因：

- 先建立项目全局协同认知
- 再建立业务使用认知
- 再进入运维落地和研发兜底
- 最后用一条完整主线把多角色串起来

## 11. 场景演练设计

建议把联合演练固定为一条“从上游到下游”的主线：

1. 创建/确认产品与设备主档
2. 发起 HTTP 或 MQTT 联调
3. 查看链路验证结果与 message-flow 时间线
4. 进入风险运营确认监测、告警、事件结果
5. 进入治理控制面查看待办、审批或复盘
6. 进入业务验收台完成结果核验

联合演练需要明确四件事：

- 谁触发
- 谁观察结果
- 谁处理异常
- 谁对最终通过负责

## 12. 流程问题与建议框架

第 9 章建议采用统一问题模板：

1. 问题现象
2. 当前影响角色
3. 客户视角的感受/风险
4. 架构师视角的根因
5. 建议方案
6. 建议优先级

本轮建议至少收口以下 6 类问题：

1. 工作台和入口较多，客户不易判断“先去哪”
2. 治理控制面与领域页分离，职责边界不够直观
3. 接入排障入口较多，排障顺序不够简单
4. 产品契约治理能力较强，但客户容易高估为“零代码万能接入”
5. 真实环境依赖强，培训与演示前置检查不足
6. 菜单权限、数据范围、组织范围已经成体系，但客户仍难理解“为什么我看不到”

## 14. 成稿验收标准

说明书成稿后，应至少满足以下标准：

- 能让四类角色各自回答“我先去哪个页面、做什么、怎么看结果”
- 能让培训讲师按同一目录完成完整授课，不需要临时拼接多份文档
- 能让项目团队按“建档 -> 联调 -> 风险纳管 -> 治理审批 -> 验收复盘”完成联合演练
- 能明确说明控制面与领域页的边界，避免客户误解
- 能提供最小排障地图，而不是把全部问题都交给研发口头解释
- 能主动说明当前交付边界与非目标，避免过度承诺

## 15. 后续实施建议

正式撰写说明书时，建议按以下顺序执行：

1. 先建立成稿目录和章节骨架
2. 先写第 1~3 章，固定培训语言和协同主线
3. 再按统一模板写四类角色章节
4. 再写排障地图与流程问题建议
5. 最后补附录、账号速查和培训演示脚本
6. 成稿后同步检查 `README.md`、`docs/README.md`、`docs/12`、`AGENTS.md`

## 16. 风险与防守

### 16.1 主要风险

- 把说明书写成“功能列表拼盘”，导致客户仍不知道怎么用
- 把说明书写成“研发文档”，导致一线角色读不懂
- 把当前能力写成“全自动、全自治、零代码”，导致交付预期失控
- 忽略治理控制面与领域真相源的边界，导致培训时讲错责任分工
- 忽略真实环境依赖，导致培训材料脱离现场可用性

### 16.2 防守措施

- 严格按角色与项目协同主线组织内容，不按模块堆砌
- 正文优先使用业务语言，技术细节收敛到“关键接口与排障入口”
- 对治理能力、自动化能力和零代码接入边界进行主动声明
- 把“先去哪里、再看哪里、最后找谁”写成固定路径
- 成稿前按四类角色逐章自检，确认都能回答“日常怎么用”
