# `/system-log` V3 表格扫读优化设计

## 1. 背景

`/system-log` 当前已经完成一轮结构收口：

1. `异常排查` 改成了明细优先、异常分组按需进入
2. `观测热点` 改成了热点主表 + 单一下钻区
3. `归档治理` 改成了精简指标 + 批次主表 + 详情抽屉

这说明页面的主次关系已经比早期版本清楚很多，但从实际使用体验看，还存在一个更细但很重要的问题：**主表可扫读性还不够稳定**。

主要表现为：

1. 同属 `/system-log` 的不同页签，表格的阅读节奏还不一致
2. 主字段、状态字段、次级说明和操作列之间仍有相互抢注意力的情况
3. 用户需要横向扫较多信息，才能判断“这一行值不值得继续点”
4. `观测热点` 和 `归档治理` 虽然都已经减负，但视觉上仍不够像一套统一工作台

因此，这一轮不再新增功能，也不继续大改结构，而是聚焦一个更克制、更有收益的目标：**统一 `观测热点` 与 `归档治理` 的表格语言，让用户更快扫懂每一行。**

## 2. 目标与非目标

### 2.1 目标

1. 提升 `观测热点` 与 `归档治理` 的主表扫读效率
2. 统一两页的列宽节奏、状态层级、次级文本样式和操作列密度
3. 保持“主表负责判断，深层区域负责展开”的现有结构优势
4. 在不新增后端接口的前提下，明显降低表格阅读负担
5. 让两页更像同一套控制台，而不是两个各自为政的模块

### 2.2 非目标

1. 本轮不新增任何 observability 后端接口
2. 本轮不重做 `异常排查` 结构
3. 本轮不改详情抽屉协议
4. 本轮不把 `观测热点` 或 `归档治理` 改成完全相同的模板页
5. 本轮不触碰仓库里与 `riskPoint` 相关的在研改动

## 3. 方案对比

### 3.1 方案 A：只调间距与颜色

做法：

1. 只微调样式，不改变表格列组织
2. 通过字号、间距、颜色做轻美化

优点：

1. 风险最低
2. 改动最小

缺点：

1. 主次结构问题无法真正解决
2. “难扫读”的根因仍然保留

### 3.2 方案 B：统一扫读语言，保留业务差异（推荐）

做法：

1. `观测热点` 与 `归档治理` 共用一套表格阅读规则
2. 每页仍保留自己的业务主角和字段语义
3. 统一主列节奏、状态位置、次级文本样式和操作列重量

优点：

1. 一致性强
2. 收益高于纯样式微调
3. 不会把业务语义压扁成同一个壳

缺点：

1. 需要同时修改两页组件与测试
2. 需要更仔细地收敛列结构

### 3.3 方案 C：强制统一为同一套表格模板

做法：

1. 两页尽量使用同样的列分布和头部形态
2. 通过模板复用来实现统一

优点：

1. 形式上最整齐
2. 更容易继续抽象共用组件

缺点：

1. 热点与归档的业务语义会被压扁
2. 页面容易显得机械，不像针对场景设计

### 3.4 推荐

采用 **方案 B：统一扫读语言，保留业务差异**。

原因：

1. 这能同时解决“统一性”和“可理解性”
2. 用户切换页签时能复用阅读习惯
3. 不需要为了统一而引入过度抽象

## 4. 统一表格语言

这一轮统一的不是“长得一模一样”，而是统一**读表规则**。

### 4.1 每行只保留一个主判断点

1. `观测热点`：这条热点现在值不值得继续追
2. `归档治理`：这个批次现在值不值得立即处理

页面不再要求用户在一行里同时完成三四种不同层级的判断。

### 4.2 第一列永远是对象身份

第一列作为最宽列，用来回答“我在看谁”：

1. `观测热点`：热点名称 + 目标对象 / 最新 trace
2. `归档治理`：批次号 + 来源表 / 保留策略

这样用户每次切页签时，目光落点保持一致。

### 4.3 状态永远前置

状态相关的判断必须靠前，不再埋在中后部：

1. `观测热点`：风险状态
2. `归档治理`：执行状态 + 对比结论

这样用户扫过几行时，不需要左右来回找“哪里才是风险”。

### 4.4 次级说明统一降噪

次级说明包括：

1. 来源信息
2. 辅助时间
3. 保留策略
4. 辅助统计说明

这些信息统一用次级文字承接：

1. 字重更轻
2. 字号更小
3. 不使用高饱和颜色
4. 不和主字段并排争主视觉

### 4.5 操作列轻量且稳定

操作列原则：

1. 永远固定在最右
2. 宽度稳定，不随内容波动
3. 只保留主入口，不把更多诊断信息再堆进表面

## 5. `观测热点` 收口方案

## 5.1 主表字段重排

`观测热点` 主表收成 5 列：

1. `热点对象`
2. `风险状态`
3. `性能信号`
4. `最近情况`
5. `操作`

### 5.1.1 热点对象

主文本：

1. 热点名称

次文本：

1. 目标对象
2. 最新 traceId

用途：让用户先知道“我当前看的是哪个热点对象”。

### 5.1.2 风险状态

风险状态使用轻标签或短状态组表达，不再优先堆数字。

建议信号类型：

1. `高延迟`
2. `高错误`
3. `波动`
4. `稳定`

如果一行同时有多个信号，主状态保留一个，次状态退到次级文本。

### 5.1.3 性能信号

只保留最关键的紧凑数值组：

1. `P95`
2. `P99`
3. `错误率`

表现方式以短组为主，而不是把数字分散到多个结构块中。

### 5.1.4 最近情况

保留：

1. 最近发生时间
2. 最近窗口内总次数

用途：帮助用户快速判断这是刚发生的热点，还是只是被累计出来的热点。

### 5.1.5 操作

操作列保留轻入口，例如：

1. `查看`

用户点击整行负责选中热点对象；更多诊断由下钻区承接。

## 5.2 下钻区保持现有承接分工

不把样本、趋势、相关任务搬回主表：

1. 主表负责判断
2. 下钻区负责展开

下钻区仍保持：

1. `最近样本`
2. `趋势`
3. `相关任务`

这符合“扫一眼先判断，再深入”的流程。

## 5.3 视觉规则

1. 第一列允许双层文本
2. `性能信号` 列保持一行短组，避免折成多层
3. 状态色只用于风险标签，不染色整列
4. 最新 traceId 用次级文本而不是主标题拼接

## 6. `归档治理` 收口方案

### 6.1 主表字段重排

`归档治理` 主表收成 6 列：

1. `归档批次`
2. `执行状态`
3. `对比结论`
4. `风险信号`
5. `最近时间`
6. `操作`

### 6.1.1 归档批次

主文本：

1. 批次号

次文本：

1. 来源表
2. 保留策略

用途：先确认“这一行是哪一批”。

### 6.1.2 执行状态

只表达任务运行是否正常：

1. `SUCCESS`
2. `FAILED`
3. `PARTIAL`
4. `RUNNING`

保持独立列，不和对比结论混写。

### 6.1.3 对比结论

只表达 compare 结果：

1. `MATCHED`
2. `DRIFTED`
3. `PARTIAL`
4. `UNAVAILABLE`

这列单独存在，是因为它和执行状态不是一回事。

### 6.1.4 风险信号

只保留最有判断价值的短信号：

1. 偏差量
2. 剩余过期量
3. 报告完整性

表现形式以短句或轻标签为主，不再摊成长段。

### 6.1.5 最近时间

主时间只保留一个，例如：

1. 创建时间

另一个时间信息，如截止时间，降为次级说明。

### 6.1.6 操作

主入口保留：

1. `详情`

compare、report、异常原因等继续由详情抽屉承接。

## 6.2 顶部区域继续做轻

`归档治理` 的顶部只做轻支撑，不和主表争主角：

1. 三个摘要指标继续保留
2. “最近异常批次”继续作为轻量焦点入口
3. 筛选区作为定位工具，而不是视觉主角

## 6.3 视觉规则

1. 第一列双层文本，保持与热点页一致
2. 状态列与结论列各自独立，不再塞进一列
3. 风险信号列用短组，不再做高密度说明堆叠
4. 截止时间等辅助信息退到次级文本

## 7. 统一样式落点

本轮不新增总壳组件，先在现有组件内统一以下样式语法：

1. 主表行高
2. 主字段与次字段字重层级
3. 状态标签颜色与边框
4. 次级说明颜色
5. 操作列按钮间距与宽度

建议统一方向：

1. 行高控制在接近的紧凑范围内
2. 第一列采用“标题 + 次说明”的双层节奏
3. 状态色只保留红 / 橙 / 绿 / 灰蓝四类语义
4. 数值信号采用短组，不再分裂成多个块

## 8. 文件边界

本轮主要修改边界：

1. `spring-boot-iot-ui/src/components/auditLog/AuditLogHotspotTabPanel.vue`
2. `spring-boot-iot-ui/src/components/auditLog/AuditLogArchiveTabPanel.vue`
3. `spring-boot-iot-ui/src/views/AuditLogView.vue`
4. 对应组件级与视图级测试文件

明确不动：

1. 后端 observability 接口
2. `AuditLogErrorTabPanel.vue` 的异常排查主流程
3. 抽屉协议与深链行为
4. 仓库内不属于 `system-log` 的在研改动

## 9. 测试策略

至少补齐或更新以下验证：

1. `观测热点` 主表列和下钻区的视图测试
2. `归档治理` 主表列、焦点入口和详情入口的视图测试
3. `AuditLogView` 中两页签切换后仍保持状态与筛选行为
4. 前端构建验证

## 10. 成功标准

这轮优化完成后的目标不是“更花”，而是“更快读懂”。

用户进入两页后的自然动作应变成：

### 10.1 观测热点

1. 先看热点对象
2. 立刻判断风险状态
3. 用简短性能信号决定是否继续下钻

### 10.2 归档治理

1. 先看批次身份
2. 立刻判断执行状态和对比结论
3. 用简短风险信号决定是否打开详情

如果用户在不放大页面、不逐列细读的情况下，也能快速选出下一步要点哪一行，这轮优化就算达标。
