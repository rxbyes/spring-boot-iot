# 质量工场 flaky 标记与重跑建议设计

- 日期：`2026-04-26`
- 范围：`spring-boot-iot-report`、`spring-boot-iot-ui`、`docs/*`
- 状态：`proposed`

## 1. 背景

质量工场当前已经形成四层基础能力：

1. `/quality-workbench` 已收口为 `业务验收台 + 自动化治理台` 两条主路径。
2. `generate-acceptance-coverage.mjs`、`diff-acceptance-coverage.mjs`、`run-acceptance-readiness.mjs` 已补齐覆盖治理、趋势对比和 readiness 证据。
3. `generate-automation-result-archive-index.mjs` 已把 `logs/acceptance/registry-run-*.json` 收口成稳定的 run 级归档索引，并支持按 `runId / status / runnerType / packageCode / environmentCode / 日期范围` 检索历史运行。
4. `2026-04-26` 起，结果归档索引和读侧详情已补齐失败归因，业务验收结果页与自动化治理台能够回答“更像什么问题”。

但当前结果治理仍有一个明显缺口：

1. 用户已经能知道“更像环境、接口、UI 还是断言问题”，但还不能快速判断“这次失败值不值得先重跑”。
2. 自动化治理台结果证据工作区仍主要依赖人工阅读历史台账，再自己判断某个失败是稳定缺陷还是偶发波动。
3. 对研发、测试和管理员来说，“疑似偶发失败”和“稳定失败”是完全不同的处理路径，但当前页面没有给出明确建议。
4. 后续若要推进一键重跑、flaky 治理或 CI 特殊口径，仍缺少首版稳定性判断底座。

因此，本轮应在不改执行链路、不改 readiness gate、不新增数据库真相表的前提下，把质量工场从“更会判断失败”继续推进到“更会处理失败”。

## 2. 目标

本轮目标是为自动化治理台补齐首版 flaky 标记与重跑建议能力：

1. 对当前失败场景基于同一 `scenarioId` 的最近历史结果做稳定性判断。
2. 首版稳定性口径固定为 `稳定失败 / 疑似 flaky / 样本不足` 三态。
3. 自动化治理台结果证据工作区对失败场景展示 `稳定性判断 + 重跑建议`。
4. 自动化治理台结果证据工作区补充运行级稳定性分布摘要。
5. 所有 flaky 判断只作为读侧辅助信息，不改变 `passed / failed / blocked`。

## 3. 非目标

本轮明确不做以下事项：

1. 不改 `run-acceptance-registry.mjs`、业务验收台运行入口或任何真实执行脚本。
2. 不新增一键重跑按钮、场景级重跑编排或命令复制入口。
3. 不把 flaky 结果写回归档索引生成链，不改 `automation-result-index.latest.json` 的结构。
4. 不把 flaky 判断接入 readiness gate、CI 阻断或特殊 warning 降级逻辑。
5. 不把 flaky 标记下放到业务验收结果页。
6. 不按环境、验收包、执行器做更细历史分桶。
7. 不把 `blocked` 纳入 flaky 主判断。

## 4. 方案选型

### 4.1 可选方案

#### 方案 A：查询时动态判断 flaky

当用户打开 `/automation-governance?tab=evidence&runId=...` 时，后端基于当前失败场景的 `scenarioId`，从 `automation-result-index.latest.json` 回看同一场景最近 `10` 次结果，现场计算三态稳定性结论和建议文本。

优点：

1. 不需要重写归档索引结构。
2. 判断规则可以快速调整，不必重刷历史索引。
3. 只在治理台详情场景触发，范围最克制。

代价：

1. flaky 结果主要存在于读侧，而不是预先落好的静态归档产物。
2. 详情查询需要额外做一次历史聚合。

#### 方案 B：在归档索引生成时预计算 flaky

每次生成 `automation-result-index` 时，顺手对每个场景做历史波动分析，并把 flaky 结论写回索引。

优点：

1. 前端读取更轻。
2. 多个读侧页面可以统一复用。

代价：

1. 索引结构和生成链会显著变重。
2. 规则变化后通常需要整体重刷索引。
3. 首版只为了治理台详情展示，投入过大。

#### 方案 C：独立 flaky 离线报告

新增一个单独的 CLI，扫描历史运行并输出 `flaky-analysis-*.json/.md`，治理台暂不展示。

优点：

1. 最保守，不会干扰现有读链。
2. 适合纯离线复盘。

代价：

1. 用户仍需在治理台与离线报告之间切换。
2. 无法直接帮助当前失败场景判断“要不要先重跑”。

### 4.2 推荐方案

本轮采用 **方案 A：查询时动态判断 flaky**。

原因：

1. 当前质量工场最缺的是“当前失败如何处理”的即时反馈，而不是再新增一个离线报告。
2. 归档索引已经承担 run 级稳定读入口，但首版 flaky 还不值得把结构再做重。
3. 详情读侧动态补充最适合快速迭代规则，并为下一阶段的一键重跑保留空间。

## 5. 数据口径

### 5.1 输入来源

flaky 判断只依赖现有真相源：

1. 当前 `runId` 对应详情里的失败场景。
2. `automation-result-index.latest.json` 中同一 `scenarioId` 的最近历史结果。

### 5.2 判断对象

首版只对 **当前失败场景** 做 flaky 判断。

原因：

1. 当前目标是帮助研发/测试判断“这次失败值不值得先重跑”。
2. 通过场景展示 flaky 标记价值较低，反而会增加页面负担。
3. 这样可以保持结果证据工作区聚焦在失败处理链路上。

### 5.3 历史窗口

首版固定回看同一 `scenarioId` 的最近 `10` 次结果。

口径说明：

1. 历史结果按 `updatedAt` 倒序选取。
2. 只统计状态明确为 `passed` 或 `failed` 的样本。
3. `blocked`、空状态、未知状态不纳入 flaky 主判断。
4. 若有效样本不足，则返回 `样本不足`。

## 6. 三态规则

首版稳定性判断固定为：

1. `稳定失败`
2. `疑似 flaky`
3. `样本不足`

### 6.1 `样本不足`

满足以下任一条件即判定为 `样本不足`：

1. 最近 `10` 次内有效样本数 `< 2`
2. 当前场景虽然失败，但历史没有足够记录支撑波动判断

### 6.2 `疑似 flaky`

满足以下条件时判定为 `疑似 flaky`：

1. 最近 `10` 次有效样本中，同时存在 `passed` 和 `failed`

这是首版唯一的 flaky 判定规则。

原因：

1. 规则直观，用户容易理解。
2. 不需要引入失败率阈值等更复杂统计口径。
3. 适合作为后续更细粒度 flaky 规则的第一层基础。

### 6.3 `稳定失败`

满足以下条件时判定为 `稳定失败`：

1. 最近 `10` 次有效样本数 `>= 2`
2. 且有效样本全部为 `failed`

该结论表达的是“更像稳定缺陷”，而不是“永远不需要重跑”。页面只给出处理建议，不强制阻断用户操作。

## 7. 建议文案

首版不输出动作按钮或命令，只输出固定建议文本。

### 7.1 `疑似 flaky`

建议文案固定为：

`最近 10 次同场景结果有通过也有失败，疑似偶发失败，建议先重跑一次后再判断是否需要修复。`

### 7.2 `稳定失败`

建议文案固定为：

`最近 10 次同场景持续失败，更像稳定缺陷，建议先排查接口、页面或断言，不建议直接依赖重跑。`

### 7.3 `样本不足`

建议文案固定为：

`当前历史样本不足，暂时无法判断是否偶发，建议结合失败归因和原始证据人工判断。`

## 8. 后端设计

### 8.1 读侧位置

flaky 判断只挂在 `GET /api/report/automation-results/{runId}` 详情读侧。

原因：

1. 自动化治理台结果证据当前就是围绕详情 run 展开。
2. `page` 和 `recent` 台账保持轻量，不需要在首版分页列表里做历史回看。
3. 后续若要扩展到列表或独立面板，可在此基础上复用同一判断逻辑。

### 8.2 建议新增 VO

建议新增一个场景稳定性 VO，例如：

```json
{
  "stabilityStatus": "suspected_flaky",
  "rerunAdvice": "最近 10 次同场景结果有通过也有失败，疑似偶发失败，建议先重跑一次后再判断是否需要修复。",
  "sampleSize": 6,
  "passedCount": 2,
  "failedCount": 4,
  "latestStatuses": ["failed", "passed", "failed", "failed", "passed"]
}
```

建议字段：

1. `stabilityStatus`
2. `rerunAdvice`
3. `sampleSize`
4. `passedCount`
5. `failedCount`
6. `latestStatuses`

其中 `latestStatuses` 只保留最近 `5` 个用于解释，不要求前端首版全部展示。

### 8.3 挂载位置

该 VO 挂到失败场景详情上，与上一阶段的失败归因并列：

1. `diagnosis` 负责回答“更像什么问题”
2. `stability` 负责回答“值不值得先重跑”

### 8.4 服务逻辑

后端判断顺序固定为：

1. 获取当前 `runId` 的详情。
2. 找出当前运行中的失败场景。
3. 基于 `automation-result-index.latest.json` 找同一 `scenarioId` 的历史 run。
4. 取最近 `10` 次，并过滤出 `passed / failed` 的有效样本。
5. 按三态规则得出 `stabilityStatus`。
6. 生成固定 `rerunAdvice` 文案。
7. 把稳定性结果挂回当前失败场景详情。

## 9. 前端设计

### 9.1 展示范围

首版只在自动化治理台结果证据工作区展示。

不放到业务验收结果页，原因是：

1. 业务页当前目标是先回答“是否通过、哪些模块没过”。
2. flaky 更偏研发/测试处置语义。
3. 业务用户不一定需要承担“是否先重跑”的判断责任。

### 9.2 失败场景明细

失败场景表格新增两块信息：

1. `稳定性判断`
2. `重跑建议`

推荐展示顺序：

1. 场景
2. 执行器
3. 阻断级别
4. 主分类
5. 稳定性判断
6. 判断理由
7. 重跑建议
8. 证据摘要

其中：

1. `主分类 / 判断理由 / 证据摘要` 复用上一阶段失败归因结果
2. `稳定性判断 / 重跑建议` 使用本轮新增读侧结果

### 9.3 运行级摘要

在结果证据工作区的运行级摘要区，补一个轻量分布：

1. `稳定失败 x 条`
2. `疑似 flaky x 条`
3. `样本不足 x 条`

这样用户先看整体，再下钻到单场景处理建议。

## 10. 测试方案

### 10.1 Java 测试

必须覆盖以下场景：

1. 同一 `scenarioId` 最近 `10` 次里同时存在 `passed` 和 `failed` 时，返回 `suspected_flaky`
2. 同一 `scenarioId` 最近有效样本全部为 `failed` 时，返回 `stable_failed`
3. 有效样本 `< 2` 时，返回 `insufficient_history`
4. `blocked` 不计入 flaky 主判断
5. 详情 VO 能带出建议文本和样本摘要

建议扩展的测试文件：

1. `AutomationResultQueryServiceImplTest`
2. 与结果证据详情相关的 controller / service 测试

### 10.2 前端测试

必须覆盖以下场景：

1. 自动化治理台结果证据工作区显示 `稳定性判断`
2. 自动化治理台结果证据工作区显示 `重跑建议`
3. 运行级稳定性分布能正确汇总
4. 失败归因和 flaky 判断能并列展示，不互相覆盖

建议扩展的测试文件：

1. `useAutomationRegistryWorkbench.test.ts`
2. `AutomationWorkbenchViews.test.ts`
3. 如有需要，再补 `useAutomationResultsWorkbench` 对稳定性汇总的断言

## 11. 风险与控制

### 11.1 误报风险

首版只用同一 `scenarioId` 最近 `10` 次判断，可能会出现：

1. 不同环境下的偶发差异被混在一起
2. 同一场景近期刚修复但历史仍混杂旧失败

控制方式：

1. 页面明确标记为 `疑似 flaky`
2. 只输出建议文本，不直接触发重跑
3. 不把结论接入 gate

### 11.2 历史不足风险

新场景或低频场景常见的问题是样本少。

控制方式：

1. 明确返回 `样本不足`
2. 建议结合失败归因和证据人工判断
3. 不猜测、不硬判 flaky

### 11.3 规则过早做重

若首版就引入环境维度、失败率阈值或命令生成，会明显扩大范围。

控制方式：

1. 本轮固定只看最近 `10` 次
2. 本轮固定只给建议文本
3. 本轮固定只上自动化治理台

## 12. 验收标准

本轮完成后，必须满足：

1. 自动化治理台结果证据工作区的失败场景能展示：
   - `主分类`
   - `稳定性判断`
   - `重跑建议`
   - `证据摘要`
2. 同一失败场景最近 `10` 次里同时出现 `passed` 和 `failed` 时，显示 `疑似 flaky`
3. 同一失败场景最近有效样本持续失败时，显示 `稳定失败`
4. 样本不足时，显示 `样本不足`
5. 运行级摘要能显示三态分布
6. 不改变现有 `passed / failed / blocked`
7. 不改真实执行入口，不新增一键重跑
