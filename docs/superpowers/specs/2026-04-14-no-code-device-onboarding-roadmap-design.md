# 零代码设备接入四期实施路线设计

> 日期：2026-04-14
> 状态：已在会话内确认蓝图，待用户审阅书面 spec
> 适用范围：`spring-boot-iot-protocol`、`spring-boot-iot-message`、`spring-boot-iot-device`、`spring-boot-iot-ui`、文档体系与真实环境验收口径
> 目标：在保持模块化单体、固定 `Pipeline`、真实环境验收基线和既有产品/设备工作台语义不破坏的前提下，把当前“部分设备族的配置化治理能力”升级为“面向 `mqtt-json / legacy $dp` 族系设备的无代码接入实施路线”，分四期推进规则治理平台化、协议模板配置化、设备资产半自动接入和下游业务联动化。

## 1. 背景

截至 `2026-04-14`，仓库已经具备以下稳定事实：

1. `iot_vendor_metric_mapping_rule` 已升级为一级治理对象，并已被 compare、apply、审批执行与运行时 `PAYLOAD_APPLY` 消费。
2. `/products -> 契约字段` 已补齐“映射规则建议 -> 采纳为 `PRODUCT + DRAFT` 草稿规则”的同页最小闭环。
3. `legacy $dp` 已形成 `Template Registry -> Template Matcher -> Template Executor -> Splitter Orchestrator` 四层结构，模板执行证据也已进入 `message-trace` 与产品治理 compare/apply 消费面。
4. 协议安全层已补齐 `iot.protocol.family-definitions` 与 `iot.protocol.decrypt-profiles` 配置结构，运行时支持 `family:<familyCode>` 协议族继承。
5. `/devices` 已支持从未登记上报线索进入正式设备建档，但仍以人工补档为主。

同时也有明确边界：

1. 项目入口文档已明确声明：当前能力只覆盖裂缝、GNSS、激光测距、深部位移与雨量计等治理最小切片，**不代表任意设备厂商已经实现零代码接入**。
2. 协议模板当前仍以内置代码注册为主，新增模板仍需新增 Java 类。
3. 协议族与解密档案当前主要停留在 `application-*.yml` 配置层，并没有完整的治理工作台、审批、发布与回滚谱系。
4. 未登记设备虽然可以人工转建设备主档，但还没有形成“系统推荐 -> 人工确认 -> 批量转正”的半自动接入闭环。
5. 风险目录、对象洞察、风险点绑定与阈值策略仍主要依赖人工继续推进，尚未形成接入发布后的自动建议链。

因此，当前最合理的路线不是继续用零散补丁去堆叠“看起来更像无代码”的局部能力，而是明确分期，把“无代码接入”拆成可独立验收的四个阶段。

## 2. 问题定义

本设计要解决的问题不是“让所有协议都可配置”，而是：

1. 在现有 `mqtt-json / legacy $dp` 主链路内，让新增同类设备或同类厂商变种时，不再要求研发反复改动 `ProductModelServiceImpl`、`LegacyDp*` 或运行时归一逻辑。
2. 让接入配置从“开发可配”升级为“业务可治理、可审批、可发布、可回滚、可追溯”。
3. 让“字段归一成功”继续向下游传导为“设备主档可转正、对象洞察可建议、风险指标可生成、治理积压可追踪”，而不是停留在协议解码或契约 compare 这一层。

本设计中的“真正达标”定义为：

1. 新增同协议族设备时，不需要新增 Java 类或修改既有协议适配器。
2. 配置变更必须通过正式治理对象、审批和发布生效，不依赖手改 `application-*.yml` 或手工 SQL。
3. 设备接入后的主档、合同发布与下游业务建议都能通过统一工作台收口，而不是分散在失败归档、开发脚本和临时文档里。

## 3. 非目标

本路线明确不承诺以下事情：

1. 不承诺在本轮实现“任意新传输协议都完全无代码”。新增真正的全新传输协议适配器，仍可能需要协议层开发。
2. 不承诺一步到位实现“无人值守自动建档、自动审批、自动发布、自动绑定风险点”。
3. 不新增一套平行的“零代码接入平台”去替代 `/device-access`、`/products`、`/devices`。
4. 不破坏当前固定 `Pipeline`、模块化单体边界和 `spring-boot-iot-admin` 唯一启动模块约束。
5. 不把配置治理直接做成“跳过审批的运行态热生效开关”。

## 4. 方案比较

### 4.1 方案 A：一次性建设完整零代码平台

做法：

1. 一次性补齐规则治理、模板配置化、资产自动建档、风险自动联动。
2. 同时新增统一平台工作台、审批链、运行时装载与回滚体系。

优点：

1. 目标最完整。
2. 业务上最接近“理想状态”的零代码接入。

缺点：

1. 范围过大，难以与当前稳定基线隔离。
2. 很容易在协议、资产、风险三条线同时引入新不确定性。
3. 不利于真实环境逐段复验。

结论：

- 拒绝。

### 4.2 方案 B：按能力阶梯分四期推进，推荐

做法：

1. 第一阶段先治理“配置对象如何成为正式真相”。
2. 第二阶段再治理“协议模板为什么还要写代码”。
3. 第三阶段补“设备主档为什么还靠人工补录”。
4. 第四阶段才把接入成果向对象洞察与风险闭环扩散。

优点：

1. 与当前仓库事实最一致。
2. 每一期都能独立验收、独立回滚。
3. 允许继续复用现有 `/products`、`/devices`、`/device-access` 工作台语义，不必推倒重来。

缺点：

1. 业务侧要接受“不是一轮做完”的现实。
2. 早期阶段只能做到“强配置化/低代码”，还不能对外宣称完全无代码。

结论：

- 采用本方案。

### 4.3 方案 C：继续沿当前路径做点状增强

做法：

1. 继续针对单一设备族补规则、补 compare、补页面提示。
2. 不做统一对象模型和路线治理。

优点：

1. 短期投入最小。

缺点：

1. 长期会把厂商特例继续塞回协议层和业务服务层。
2. 文档、工作台、运行时与审批口径会越来越分裂。
3. 无法形成真正可复制的接入模式。

结论：

- 拒绝。

## 5. 设计结论

本轮固定采用“**四期实施路线**”：

1. 第一期：规则治理平台化
2. 第二期：协议模板配置化
3. 第三期：设备资产半自动接入
4. 第四期：下游业务自动联动

同时固定以下设计原则：

1. 优先复用现有真相源，不新增无必要的平行主表。
2. 自动化必须建立在治理对象、审批谱系和发布回滚之上。
3. “建议优先于自动执行”，先解决“如何发现和确认”，再解决“如何自动生效”。
4. 产品级治理继续优先留在 `/products`，资产级治理继续留在 `/devices`，接入全局视角继续留在 `/device-access`。
5. 所有阶段都必须可在真实 `dev` 环境单独验收。

## 6. 第一期：规则治理平台化

### 6.1 目标

把当前“能配、能消费”的配置对象升级为“可治理、可审批、可发布、可回滚”的正式平台对象。

这一期聚焦三类对象：

1. 厂商字段映射规则
2. 协议族定义
3. 解密档案

### 6.2 当前基础

当前已有：

1. `iot_vendor_metric_mapping_rule`
2. `/api/device/product/{productId}/vendor-mapping-rules`
3. `/api/device/product/{productId}/vendor-mapping-rule-suggestions`
4. 运行时对 `family:<familyCode>` 的消费能力

当前缺口：

1. 协议族与解密档案没有正式治理工作台和持久化治理对象。
2. 规则没有完整的发布谱系和独立回滚视角。
3. 当前产品工作台只能创建 `DRAFT` 规则，缺批量操作、独立审批、共享 scope 治理和命中观测。

### 6.3 能力清单

第 1 期必须补齐：

1. 规则台账、详情、批量导入、批量状态切换。
2. 规则审批提交、发布、生效、回滚和命中观测。
3. `DEVICE_FAMILY / SCENARIO / PROTOCOL` 共享 scope 的正式治理入口。
4. 协议族定义与解密档案的后端 CRUD、审批和发布。
5. 规则/协议族/解密档案的测试命中接口与回放校验。

第 1 期明确不做：

1. 模板 DSL
2. 模板解释执行器替换
3. 自动设备建档
4. 风险自动绑定

### 6.4 模块落点

后端：

1. `spring-boot-iot-device`
   - 厂商字段映射规则治理对象、审批、发布与命中观测
2. `spring-boot-iot-framework`
   - 协议族与解密档案配置对象装载与运行时装配
3. `spring-boot-iot-protocol`
   - 继续只消费已发布配置，不承担治理流程

前端：

1. `/products`
   - 继续承接产品级规则治理、suggestions 与产品内规则台账
2. `接入智维`
   - 在现有 `接入智维` 路由族内新增共享协议治理能力页，承接协议族与解密档案对象
   - `/device-access` 继续只承担总览与排障决策树语义，不改造成协议配置编辑页，也不新增第二个总览中心
3. `/governance-approval`
   - 复用现有审批控制面，不再造新审批体系

### 6.5 数据对象建议

优先沿用：

1. `iot_vendor_metric_mapping_rule`

建议新增正式治理对象：

1. `iot_protocol_family_definition`
2. `iot_protocol_decrypt_profile`

如需发布留痕，可补：

1. `iot_vendor_metric_mapping_rule_snapshot`
2. `iot_protocol_family_snapshot`
3. `iot_protocol_decrypt_profile_snapshot`

但原则上不新增平行“草稿主表”，仍以对象主表 + 审批单 + 快照组合解决发布问题。

### 6.6 页面与 API 建议

页面建议：

1. `/products -> 契约字段 -> 映射规则建议 / 映射规则台账`
2. `接入智维 -> 协议治理工作台`
   - 作为现有接入能力树下的独立治理页承接，不替代 `/device-access` 总览

API 建议：

1. 规则对象：`list/detail/create/update/submit-approval/publish/rollback/test-hit`
2. 协议族对象：`list/detail/create/update/submit-approval/publish/rollback`
3. 解密档案对象：`list/detail/create/update/submit-approval/publish/rollback/test-decrypt`

### 6.7 验收标准

第 1 期完成后，必须做到：

1. 新增同协议族厂商别名差异时，不改 compare/apply/runtime 代码，只改规则配置。
2. 协议族和解密档案不再只存在于 `application-*.yml`。
3. 所有配置对象都能看到审批、生效版本、回滚记录与命中证据。

## 7. 第二期：协议模板配置化

### 7.1 目标

把当前“内置模板类 + 注册表”的实现，升级为“模板定义 + 模板解释执行”的配置驱动模式。

### 7.2 当前基础

当前已有：

1. 模板匹配/执行框架
2. 模板执行证据
3. 模板证据进入 `message-trace` 与产品治理 compare/apply

当前缺口：

1. 模板本体仍是代码注册。
2. 新增模板仍要新增 Java 类。
3. 模板没有正式治理对象、版本和灰度验证。

### 7.3 能力清单

第 2 期必须补齐：

1. 模板定义模型
2. 模板匹配条件配置
3. 模板字段抽取/映射/状态镜像/父字段剔除规则配置
4. 模板样本验证与运行时回放验证
5. 模板发布、灰度和回滚

第 2 期建议坚持：

1. 解释执行优先于脚本执行
2. 样本验证优先于真实生效
3. 模板证据仍继续进入 `message-trace` 与 `/products`

### 7.4 模块落点

后端：

1. `spring-boot-iot-protocol`
   - 模板定义模型、模板解释执行器、模板装载器
2. `spring-boot-iot-device`
   - 模板治理消费、compare/apply 证据联动
3. `spring-boot-iot-message`
   - 保持编排，不承载模板治理逻辑

前端：

1. `接入智维 -> 协议治理工作台`
   - 新增模板定义、样本调试、发布回执与命中回放
   - 继续沿用接入侧既有页面骨架与导航层级，不单独新造平台外壳
2. `/message-trace`
   - 继续显示模板命中证据
3. `/products -> 契约字段`
   - 继续消费模板证据，不改页面主语义

### 7.5 数据对象建议

建议新增：

1. `iot_protocol_template_definition`
2. `iot_protocol_template_revision`
3. `iot_protocol_template_validation_run`

原则：

1. 模板治理对象与运行时证据分离
2. 正式生效模板有明确 revision 和生效快照

### 7.6 页面与 API 建议

页面建议：

1. 模板列表
2. 模板编辑器
3. 样本调试台
4. 发布/回滚历史

API 建议：

1. `list/detail/create/update`
2. `validate-by-sample`
3. `replay-by-message-log`
4. `submit-approval/publish/rollback`

### 7.7 验收标准

第 2 期完成后，必须做到：

1. 新增一种同类子拆分或 canonicalization 规则时，不新增模板 Java 类。
2. 模板上线前可以用真实样本做验证与回放。
3. 模板发布后，运行时命中结果和治理证据仍保持一致。

## 8. 第三期：设备资产半自动接入

### 8.1 目标

把当前“未登记线索 -> 人工建档”升级为“系统推荐 -> 人工确认 -> 批量转正”的半自动接入闭环。

### 8.2 当前基础

当前已有：

1. 未登记上报名单
2. 未登记设备详情与排障上下文
3. 从未登记线索进入正式建档

当前缺口：

1. 仍需从空白或半空白表单人工录入关键字段。
2. 没有批量转正能力。
3. 没有父子关系、候选产品、认证方式和组织归属的自动推荐。

### 8.3 能力清单

第 3 期必须补齐：

1. 基于未登记上报的产品推荐
2. 基于协议族/模板/字段证据的关系推荐
3. 基于已有设备/产品的认证参数推荐
4. 批量确认建档
5. 建档后自动解封未登记记录和补齐关系

第 3 期明确不做：

1. 无人值守自动建档
2. 绕过人工确认直接创建正式资产

### 8.4 模块落点

后端：

1. `spring-boot-iot-device`
   - 未登记线索聚合
   - 接入建议生成
   - 批量建档编排

前端：

1. `/devices`
   - 继续承接“已登记 + 未登记”双名单
   - 增强为“接入建议 + 批量转正”工作台

### 8.5 数据对象建议

保持正式主档不变：

1. `iot_device` 仍是唯一正式资产真相

建议新增编排留痕：

1. `iot_device_onboarding_batch`
2. `iot_device_onboarding_decision_log`

原则：

1. 不新增平行正式资产表
2. 未登记线索仍主要来自 `iot_device_invalid_report_state` 及其回退来源

### 8.6 页面与 API 建议

页面建议：

1. `/devices` 内新增“建档建议条”
2. 批量转正抽屉
3. 推荐关系预览
4. 推荐认证信息预览

API 建议：

1. `preview-onboarding-suggestions`
2. `preview-relation-mappings`
3. `preview-auth-profile`
4. `batch-register-devices`
5. `resolve-unregistered-leads`

### 8.7 验收标准

第 3 期完成后，必须做到：

1. 一批同产品未登记设备能在 `/devices` 一次性确认转正。
2. 系统能自动推荐大部分产品、拓扑和认证信息。
3. 建档成功后，未登记线索自动解封，设备进入正式治理链。

## 9. 第四期：下游业务自动联动

### 9.1 目标

让接入成果从“字段能进来”继续扩散到“对象洞察、风险指标、风险点绑定与策略配置能得到正式建议”。

### 9.2 当前基础

当前已有：

1. 发布批次
2. resolver snapshot
3. 风险指标目录最小闭环
4. 对象洞察重点指标正式真相源

当前缺口：

1. 发布后缺少统一的下游治理建议链。
2. 风险指标、风险点绑定、阈值模板仍主要依赖人工继续推进。
3. 对象洞察和风险策略之间没有显式的接入后续经营闭环。

### 9.3 能力清单

第 4 期建议补齐：

1. 对象洞察候选指标建议
2. 风险指标目录建议
3. 风险点待绑定建议
4. 阈值模板建议
5. 发布后的影响分析与治理待办固化

第 4 期原则：

1. 仍以“自动建议”优先
2. 不直接跳过人工确认

### 9.4 模块落点

后端：

1. `spring-boot-iot-device`
   - 发布后对象洞察候选生成
2. `spring-boot-iot-alarm`
   - 风险指标、风险点和阈值模板建议
3. `spring-boot-iot-report`
   - 经营分析与治理进度视图

前端：

1. `/products`
   - 显示合同发布后的下游建议摘要
2. `/risk-point`
   - 承接正式绑定建议
3. `/rule-definition`
   - 承接阈值模板建议
4. `/device-access`
   - 承接接入后续治理总览

### 9.5 数据对象建议

优先复用：

1. `risk_metric_catalog`
2. `releaseBatchId`
3. `iot_product.metadata_json.objectInsight.customMetrics[]`

必要时新增建议对象：

1. `risk_metric_binding_suggestion`
2. `object_insight_metric_suggestion`

但这些对象应是治理建议，不得替代正式真相表。

### 9.6 页面与 API 建议

API 建议：

1. `list-object-insight-suggestions`
2. `list-risk-metric-suggestions`
3. `list-risk-binding-suggestions`
4. `list-threshold-template-suggestions`

页面建议：

1. 发布结果页增加“下一步治理建议”
2. 风险绑定与阈值规则工作台支持从建议直接进入正式动作

### 9.7 验收标准

第 4 期完成后，必须做到：

1. 新设备族完成合同发布后，平台能自动回答“哪些字段可进对象洞察、哪些字段可进风险闭环、还缺哪些动作”。
2. 运维和产品不再需要靠经验跨多个页面手工补齐全部后续治理动作。

## 10. 跨阶段依赖关系

四期必须按顺序推进：

1. 没有第 1 期，配置对象就没有正式真相，不适合继续扩大自动化范围。
2. 没有第 2 期，新增协议模板仍要写代码，“无代码接入”只能停留在字段别名层。
3. 没有第 3 期，报文即使接进来了，设备主档仍然是人工补录。
4. 没有第 4 期，接入成果无法扩散为经营和风险价值。

同时固定实施拆分原则：

1. 本文是路线级 spec，不直接展开四期总实现单。
2. 下一步 `writing-plans` 只展开第 1 期“规则治理平台化”的实现计划。
3. 第 2 至第 4 期在前一期完成并通过真实环境验收后，再分别细化为独立实施计划。

## 11. 里程碑与冻结边界

### 11.1 M1：规则治理平台化完成

冻结边界：

1. 不碰模板解释执行器
2. 不做自动建档
3. 不做风险自动联动

### 11.2 M2：协议模板配置化完成

冻结边界：

1. 只覆盖现有 `legacy $dp` 模板主链
2. 不承诺所有传输协议都配置化

### 11.3 M3：设备资产半自动接入完成

冻结边界：

1. 只做到“系统推荐 + 人工确认”
2. 不承诺无人值守自动建档

### 11.4 M4：下游业务自动联动完成

冻结边界：

1. 只做到正式建议链
2. 不跳过人工审批或正式治理动作

## 12. 风险与控制策略

### 12.1 主要风险

1. 配置对象快速增多后，治理复杂度上升。
2. 模板配置化若缺少验证和回放，容易把运行态错误前移到生产。
3. 资产半自动建档若推荐错误，会污染正式主档。
4. 下游业务自动联动若直接做自动执行，容易扩大错误影响面。

### 12.2 控制策略

1. 所有对象先做“建议、审批、发布、回滚”，再考虑自动执行。
2. 样本验证和真实日志回放必须作为模板/规则发布前门禁。
3. 正式主档与正式合同继续保持唯一真相，不引入平行正式表。
4. 真实环境复验必须逐期进行，不允许只靠本地测试宣称完成。

## 14. 结论

当前项目离“真正的无代码接入”并不是零到一的差距，而是“底座已具备，但治理对象、模板引擎、资产编排和业务联动还没有连续收口”的差距。

因此最合理的推进方式不是再继续做单点补丁，而是按本 spec 固定为四期路线：

1. 先让配置成为正式治理真相。
2. 再让模板脱离代码注册。
3. 再让设备主档从人工补录升级为半自动接入。
4. 最后让接入结果自动扩散到对象洞察和风险闭环。

只有四期都完成后，项目才适合对外正式宣称：

`在既有 mqtt-json / legacy $dp 接入族系内，平台已经具备面向同类新设备的无代码接入能力。`
