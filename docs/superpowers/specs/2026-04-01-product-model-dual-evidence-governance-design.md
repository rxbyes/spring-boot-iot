# 产品物模型双证据并列治理设计

**Date:** 2026-04-01  
**Status:** Approved in-session for spec drafting  
**Audience:** 设备中心 / 产品治理 / 前后端实现  
**Scope:** 在 `/products` 的 `产品经营工作台 -> 物模型治理` 内，引入“手动提炼 + 自动提炼”并列对比治理能力，覆盖 `property / event / service` 三类模型，并继续以 `iot_product_model` 作为唯一正式契约存储。

## 1. 背景

当前仓库已经形成以下稳定现状：

1. `产品经营工作台` 内的物模型治理默认链路已经收口为“手动提炼 -> 正式模型 -> 确认写库”，默认 UI 不再自动暴露运行期候选入口。
2. 后端仍保留 `GET /api/device/product/{productId}/model-candidates` 运行期候选提炼能力：
   - `property` 候选优先来自 `iot_device_property`，并辅以最近真实消息日志。
   - `event` 候选来自 `iot_device_message_log` 中稳定事件证据。
   - `service` 候选来自 `iot_command_record` 中稳定命令证据。
3. 手动提炼当前只支持围绕当前产品粘贴 `1` 个设备样本 JSON，并只自动生成 `property` 候选；`event / service` 仍依赖正式模型人工维护。
4. 当前确认写库接口本质上是“候选批量新增”：
   - 对不存在的 `identifier` 执行新增。
   - 对已存在 `identifier` 只返回 `skipped/conflict`，不支持“纳入修订”。
5. `对象洞察台` 中展示的 `设备属性快照` 属于运行期最新值，用于补充核查监测对象关联设备的当前状态，不是产品级正式契约。

结合当前代码和文档，系统中三类对象的关系应固定理解为：

1. `产品物模型`
   - 产品级正式字段标准。
   - 用于 latest 属性解释、测点选项、命令定义、规则与报表口径收口。
2. `设备属性快照`
   - 设备实例级运行期最新值。
   - 用于对象洞察台和设备补充核查，不替代正式契约。
3. `消息日志 / 命令记录`
   - 原始或半结构化运行证据。
   - 用于排障、候选提炼和证据补强。

本轮设计的核心诉求不是简单“把自动提炼按钮放回来”，而是把手动证据、运行证据和正式契约放进同一套治理视图中，用于发现漏项、命名漂移和证据差异，并继续保持人工确认是唯一入正式模型的门槛。

## 2. 目标

本设计服务以下目标：

1. 在同一个 `产品经营工作台` 中，把 `手动提炼` 与 `自动提炼` 并列展示并做统一对比。
2. 让 `property / event / service` 三类模型都能进入同一套“候选对比 -> 决策 -> 正式模型”治理链路。
3. 把“发现漏项、命名漂移和证据差异”作为第一优先级，同时兼顾建模效率与误入库风险控制。
4. 保持 `iot_product_model` 仍是唯一正式契约表，不新增 `draft`、`schema` 或平行草稿表。
5. 让对象洞察台、设备 latest、消息日志、命令记录继续作为证据来源，但不反向变成正式模型写入入口。

## 3. 非目标

本轮明确不做以下事情：

1. 不新增新的一级路由或把物模型治理拆出 `/products`。
2. 不把 `/insight` 或风险运营页改造成正式模型写入入口。
3. 不允许“自动提炼结果直接落正式模型”。
4. 不按设备维度维护独立正式模型分支，正式模型仍只按产品维度治理。
5. 不引入模糊自动合并，把疑似同义字段直接折算成一个正式 `identifier`。
6. 不新增 `iot_product_model_draft`、`iot_product_schema`、治理会话表或中间态持久化表。

## 4. 方案对比

### 4.1 方案 A1：双来源并列，单一确认出口

- `手动提炼` 与 `自动提炼` 在同一治理页并列存在。
- 两路候选会被统一折叠成“对比结果行”。
- 最终只通过一个“正式模型确认区”执行新增或修订。

优点：

1. 能同时解决漏项发现、命名漂移和证据差异问题。
2. 保留人工确认门槛，不会让运行期脏字段直接入正式模型。
3. 符合当前系统“正式模型是唯一契约源”的治理原则。

缺点：

1. 对比规则和前后端数据结构会更复杂。
2. 当前 `confirm` 仅支持新增，需要新增“修订应用”能力。

### 4.2 方案 A2：自动优先生成，手动只做修订

- 运行期候选作为主入口。
- 手动提炼只做补充或修正。

优点：

1. 初始录入速度最快。

缺点：

1. 运行期噪音、历史兼容字段和命名漂移更容易进入正式模型。
2. 不符合本轮“以完整性和风险控制为主”的治理目标。

### 4.3 方案 A3：双来源可看，不做真正合并

- 手动提炼和自动提炼都展示。
- 但不生成统一对比结果，仅供人工肉眼对照。

优点：

1. 前端改造最轻。

缺点：

1. 不能统一收口冲突判定、建议动作和确认出口。
2. 价值不足，仍会把大量治理判断留给人工记忆。

## 5. 选型

本轮采用 `方案 A1：双来源并列，单一确认出口`。

原因：

1. 用户已明确确认，需要把“手动提炼 + 自动提炼”作为并列入口，并用于对比后提取为正式模型。
2. 目标优先级已确认是：同时覆盖效率、质量和风险控制，但以“发现漏项和命名漂移”优先。
3. 当前后端已经具备手动提炼和运行期候选两条基础能力，只缺统一对比域模型和统一应用出口。
4. 当前库与文档已明确“不新增平行草稿表”，因此最合适的升级方式是“增强治理会话能力”，而不是扩表做流程系统。

## 6. 核心概念与职责

### 6.1 四层角色

本轮统一把物模型治理拆成四个角色层次：

1. `手动证据层`
   - 来源：样本 JSON 提炼 + 人工补录。
   - 价值：承接业务理解和厂家协议理解。
2. `自动证据层`
   - 来源：`iot_device_property`、`iot_device_message_log`、`iot_command_record`。
   - 价值：承接真实运行情况和稳定证据。
3. `正式契约层`
   - 来源：`iot_product_model`。
   - 价值：系统唯一正式标准。
4. `运行核查层`
   - 来源：对象洞察台、设备 latest、消息日志、风险监测详情。
   - 价值：辅助判断，不直接写正式契约。

### 6.2 对象洞察台与物模型治理的关系

对象洞察台中的 `设备属性快照` 与产品经营工作台中的 `正式物模型` 不是同一类数据：

1. `设备属性快照`
   - 是设备实例当前值。
   - 只回答“当前这个设备现在上报了什么、值是多少、何时更新”。
2. `正式物模型`
   - 是产品级字段标准。
   - 回答“平台承认哪些字段、字段叫什么、类型是什么、事件和服务语义如何定义”。

因此，关系应固定为：

1. 对象洞察台提供“这个字段在真实设备上是否存在、是否稳定、最近值是什么”的证据。
2. 产品物模型治理负责决定“这个字段是否应该进入产品正式契约，以及以什么名称和结构进入”。
3. 对象洞察台不提供直接写库动作；物模型治理也不直接消费单条 latest 值替代正式定义。

### 6.3 正式模型在整个系统中的作用

正式物模型不是单纯的页面配置，而是 IoT 主链路中的正式解释层，至少承担以下职责：

1. 为 `PAYLOAD_APPLY` 阶段提供 `propertyName / dataType` 等属性解释元数据。
2. 为 latest 属性展示、设备指标选项和对象洞察核查提供字段名称标准。
3. 为事件定义、服务定义和命令输入输出约束提供契约基线。
4. 为规则、告警、报表和字段治理提供统一标识符口径。
5. 为跨设备、跨项目复用提供产品级标准，而不是让实例数据各自漂移。

## 7. 工作台设计

### 7.1 页面位置与总结构

本轮继续复用现有：

- `/products`
- `产品经营工作台`
- `物模型治理` 视图

不新增一级路由，不新增并行抽屉壳。工作台内页升级为四段式：

1. `证据入口区`
2. `治理摘要区`
3. `对比工作区`
4. `正式模型确认区`

### 7.2 证据入口区

证据入口区分成两栏，但位于同一页：

1. `手动提炼`
   - 继续支持当前 `sampleType + samplePayload` 的单设备样本 JSON 提炼。
   - 用于生成 `property` 手动候选。
   - 同区新增 `人工补录候选` 能力：
     - `event`
     - `service`
     - 手工补录的 `property`
   - 这样才能满足三类模型都进入统一治理链路，而不把 `event / service` 排除在手动侧之外。
2. `自动提炼`
   - 触发运行期候选提炼。
   - 用于获取 `property / event / service` 三类自动候选。
   - 继续保留“无证据时返回空结果 + 诚实提示”的现有语义。

### 7.3 治理摘要区

治理摘要区统一显示：

1. 手动命中数
2. 自动命中数
3. 双证据一致数
4. 正式已存在数
5. 冲突待裁决数
6. 证据不足数

摘要区不再只表达“本次提炼了多少条”，而是表达“本次治理的风险与收获”。

### 7.4 对比工作区

对比工作区按 `property / event / service` 三类切换，但共享同一套表格语法。

每一行围绕一个候选标识展示：

1. `模型类型`
2. `候选标识`
3. `手动证据`
4. `自动证据`
5. `正式模型现状`
6. `风险标记`
7. `建议动作`

必要时可在侧边或下钻区展示：

1. 最近消息日志证据
2. 最近命令证据
3. latest 属性证据
4. 疑似同义标识提示
5. 正式模型当前版本内容

### 7.5 正式模型确认区

最终确认区只承接用户已经明确选择的决策项，且只允许以下决策：

1. `纳入新增`
2. `纳入修订`
3. `继续观察`
4. `人工裁决`
5. `忽略`

其中：

1. `纳入新增`
   - 写入新正式模型。
2. `纳入修订`
   - 更新现有正式模型。
3. `继续观察`
   - 不写库，只留在本次治理会话中。
4. `人工裁决`
   - 先补齐人工调整，再决定新增或修订。
5. `忽略`
   - 不写库。

正式确认区只展示 `新增/修订` 待应用项；`继续观察/忽略` 不做持久化。

## 8. 对比规则

### 8.1 自动合并规则

只有当以下条件同时满足时，系统才允许自动判定“这是同一候选”：

1. `modelType` 完全一致
2. `identifier` 完全一致

例如：

1. `property: temperature` 只会和 `property: temperature` 自动对齐
2. `event: alarm_raise` 只会和 `event: alarm_raise` 自动对齐
3. `service: reboot` 只会和 `service: reboot` 自动对齐

### 8.2 疑似同义规则

以下情况只做“疑似冲突/命名漂移”提示，不自动合并：

1. 大小写差异
2. 下划线与简写差异
3. 常见拼写漂移，如 `signal_nb` 与 `singal_nb`
4. 业务近义词，如 `temp` 与 `temperature`
5. 服务别名，如 `reboot` 与 `restart`

这类候选必须进入人工裁决，不允许系统直接折算成一个正式 `identifier`。

### 8.3 行状态

对比结果统一输出以下 6 种状态：

1. `double_aligned`
   - 手动候选和自动候选都命中同一标识，正式模型中尚不存在。
2. `manual_only`
   - 只有手动候选命中。
3. `runtime_only`
   - 只有自动候选命中。
4. `formal_exists`
   - 正式模型已存在该标识，候选仅作为旁证。
5. `suspected_conflict`
   - 存在同义异名、字段类型不一致、事件级别不一致、服务输入输出不一致等冲突。
6. `evidence_insufficient`
   - 当前只有提示性线索、结构不完整证据，或人工补录缺少关键字段，暂不适合入正式模型。

### 8.4 状态优先级

同一行状态按以下优先级判定，避免一个候选同时命中多个状态：

1. 若存在“正式模型同标识，但核心定义不一致”，优先判为 `suspected_conflict`
2. 否则若正式模型同标识已存在，判为 `formal_exists`
3. 否则若手动与自动都命中同标识，判为 `double_aligned`
4. 否则若只命中手动侧，判为 `manual_only`
5. 否则若只命中自动侧且已经形成可展示候选项，判为 `runtime_only`
6. 其他情况统一判为 `evidence_insufficient`

### 8.5 三类模型的判定重点

#### `property`

重点对比：

1. `identifier`
2. `dataType`
3. `specsJson`
4. `valueType`
5. 最近证据时间
6. 来源表

#### `event`

重点对比：

1. `identifier`
2. `eventType`
3. 最近消息日志证据
4. Topic / 报文稳定性

#### `service`

重点对比：

1. `identifier`
2. `serviceInputJson`
3. `serviceOutputJson`
4. 最近命令记录证据
5. 服务标识字段是否齐备

### 8.6 正式模型的角色

正式模型在对比结果中不是“第三个候选来源”，而是“当前裁决基线”。

也就是说：

1. 手动与自动回答“应该新增什么、修订什么”。
2. 正式模型回答“现在系统承认的是什么”。
3. 自动证据再强，也不能越过正式模型直接生效。

### 8.7 建议动作映射

为了避免前后端各自推断“建议动作”，本轮建议固定以下默认映射：

1. `double_aligned`
   - 默认建议：`纳入新增`
2. `manual_only`
   - 默认建议：`继续观察`
   - 若人工已补齐关键定义且业务确认充分，可转 `纳入新增`
3. `runtime_only`
   - 默认建议：`继续观察`
   - 若证据稳定、命名清晰且业务已确认，可转 `纳入新增`
4. `formal_exists`
   - 默认建议：`忽略`
   - 若候选与正式模型存在明确内容差异，可转 `纳入修订`
5. `suspected_conflict`
   - 默认建议：`人工裁决`
6. `evidence_insufficient`
   - 默认建议：`继续观察`
   - 若人工确认该线索为噪音，可转 `忽略`

## 9. 接口设计

### 9.1 兼容保留的现有接口

以下接口继续保留，作为底层能力或兼容入口：

1. `GET /api/device/product/{productId}/models`
2. `POST /api/device/product/{productId}/models`
3. `PUT /api/device/product/{productId}/models/{modelId}`
4. `DELETE /api/device/product/{productId}/models/{modelId}`
5. `GET /api/device/product/{productId}/model-candidates`
6. `POST /api/device/product/{productId}/model-candidates/manual-extract`
7. `POST /api/device/product/{productId}/model-candidates/confirm`

其中旧 `confirm` 接口继续保留为“仅新增兼容入口”，但不再作为新治理工作台的主应用接口。

### 9.2 新增对比接口

建议新增：

`POST /api/device/product/{productId}/model-governance/compare`

请求体建议包含：

1. `manualExtract`
   - 是否启用手动样本提炼
   - `sampleType`
   - `samplePayload`
2. `manualDraftItems`
   - 人工补录候选数组
   - 允许录入 `property / event / service`
3. `includeRuntimeCandidates`
   - 是否同时拉取运行期候选，默认 `true`
4. `runtimeWindowDays`
   - 可选，未传时继续沿用现有默认窗口

响应体建议包含：

1. `summary`
2. `manualSummary`
3. `runtimeSummary`
4. `formalSummary`
5. `compareRows`

`compareRows` 每行至少包含：

1. `modelType`
2. `identifier`
3. `manualCandidate`
4. `runtimeCandidate`
5. `formalModel`
6. `compareStatus`
7. `riskFlags`
8. `suggestedAction`
9. `suspectedMatches`

### 9.3 新增应用接口

建议新增：

`POST /api/device/product/{productId}/model-governance/apply`

请求体 `items[]` 每项建议包含：

1. `decision`
   - `create`
   - `update`
   - `skip`
2. `targetModelId`
   - `update` 时必填
3. `modelType`
4. `identifier`
5. `modelName`
6. `dataType`
7. `specsJson`
8. `eventType`
9. `serviceInputJson`
10. `serviceOutputJson`
11. `sortNo`
12. `requiredFlag`
13. `description`
14. `compareStatus`
15. `evidenceSources`

响应体建议包含：

1. `createdCount`
2. `updatedCount`
3. `skippedCount`
4. `conflictCount`
5. `lastAppliedAt`

### 9.4 为什么当前 `confirm` 已不够

当前 `confirmModelCandidates` 只支持：

1. 新建不存在的 `identifier`
2. 已存在时返回 `skipped/conflict`

这不支持以下新治理动作：

1. `纳入修订`
2. `人工裁决后更新正式模型`
3. 基于正式模型现状做显式决策

因此本轮必须新增“治理应用接口”，而不是继续把治理决策挤进现有“批量新增接口”。

## 10. 后端设计

### 10.1 模块边界

本轮改造继续落在：

1. `spring-boot-iot-device`
   - 控制器负责请求收发
   - Service 负责候选提炼、对比编排、治理决策应用
2. `spring-boot-iot-ui`
   - 负责工作台渲染、交互和视图状态
3. `docs`
   - 负责更新行为、接口、治理口径和前端约束

不把业务规则写进 Controller，不把治理逻辑下沉到前端随意拼装。

### 10.2 服务编排原则

建议在 `ProductModelService` 侧新增“对比域模型”编排能力：

1. 手动样本提炼
2. 手工补录候选接入
3. 运行期候选提炼
4. 正式模型加载
5. 对比结果构建
6. 应用决策执行

前端不负责重新发明对比规则，只消费后端给出的统一结果。

### 10.3 数据库边界

本轮不做任何新表设计，不新增迁移脚本。数据库边界固定为：

1. 正式模型仍只写 `iot_product_model`
2. 候选仅存在于接口返回和前端治理会话中
3. `继续观察 / 忽略` 不做持久化

## 11. 前端设计

### 11.1 入口不变

继续使用：

1. `/products`
2. `产品经营工作台`
3. `物模型治理`

不新增新的治理路由，不新增页面私有壳层。

### 11.2 交互原则

1. 进入 `物模型治理` 后，首屏先看到治理摘要和证据入口，而不是直接进入一堆候选卡片。
2. 对比工作区统一使用共享列表/卡片语法，不回退为两套完全独立的“候选目录墙”。
3. `property / event / service` 三类切换共享一套结构，不为单页再造三套交互风格。
4. 正式模型确认区只承接最终决策项，不把所有候选再次平铺一遍。

### 11.3 手动侧补强要求

为满足三类模型都进入统一治理链路，前端必须补齐：

1. `property`
   - 样本 JSON 提炼
   - 可选人工补录
2. `event`
   - 人工补录入口
3. `service`
   - 人工补录入口

不要求手动样本提炼自动生成 `event / service`，但要求手动侧能够把这两类能力纳入同一治理会话。

## 12. 错误处理与治理边界

### 12.1 治理边界

以下边界必须固定：

1. 对象洞察台只提供证据，不直接参与正式模型写入。
2. 自动提炼永远是建议源，不是自动落库源。
3. 正式物模型是产品级契约，不按设备做正式分叉。
4. 本轮继续坚持“不新增草稿表”。

### 12.2 错误处理原则

以下场景必须采用“部分失败、诚实提示”的策略，而不是整体不可用：

1. `service` 证据字段未对齐
2. `event` 当前没有稳定证据
3. 手动样本 JSON 非法
4. 自动提炼某一类模型返回空结果
5. compare 成功但 apply 部分项冲突

页面要明确告诉用户：

1. 哪一部分证据已获取
2. 哪一部分证据不足
3. 哪一部分只能人工补录
4. 哪些项未写库以及原因

### 12.3 显式决策原则

`apply` 必须要求显式声明：

1. 新增
2. 修订
3. 跳过

不允许因为用户勾选了候选，就隐式覆盖已有正式模型。
