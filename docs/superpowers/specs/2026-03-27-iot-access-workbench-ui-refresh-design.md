# IoT Access Workbench UI Refresh Design

**Date:** 2026-03-27
**Status:** Draft for review
**Scope:** `接入智维总览`、`产品定义中心`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 的统一视觉升级、结构收口与职责边界优化

## 1. 背景

当前 `接入智维` 一级工作台已经完成菜单分组与页面落位，但视觉和结构仍存在三类问题：

1. `接入智维总览` 与 6 个子模块页面的职责边界不够鲜明，容易出现“总览像首页、子页也像首页”的重复感。
2. `产品定义中心 / 设备资产中心` 与 `链路验证 / 异常观测 / 链路追踪 / 数据校验` 两类页面的工作属性不同，但当前缺少统一又有区分度的视觉语法。
3. 多个页面虽然已接入共享组件体系，但首屏焦点、主工作区层级、提醒信息与详情抽屉的呈现方式仍偏松散，未形成“接入智维”一级模块应有的统一门面感。

用户已明确要求：

- 配色继续沿用系统现有品牌体系，不引入新的私有主色
- 由 Codex 帮助设计其余视觉与结构细节
- 本轮不仅优化 6 个子页，也要把 `接入智维总览` 一起纳入同一套设计体系

## 2. 目标

本轮设计目标如下：

1. 把 `接入智维总览` 升级为真正的“中枢台”，负责跨模块态势、入口分发和待办引导。
2. 把 6 个子模块升级为“单域专台”，避免重复承担首页分发职责。
3. 在统一品牌语法下，建立两条页面变体：
   - 资产治理组：`产品定义中心`、`设备资产中心`
   - 诊断实验组：`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台`
4. 统一 Hero、指标卡、提醒带、主工作区和详情抽屉语法，提升一级模块下的整体专业感和完成度。
5. 保持现有共享组件、浅色控制台风格和品牌橙/云蓝/蓝灰体系，不回退到营销化大屏或页面私有色板。

## 3. 非目标

1. 本轮不新增一级路由，也不调整现有菜单归属。
2. 本轮不改变后端接口语义、权限模型和业务闭环。
3. 本轮不为单页引入独立主题系统、独立设计 token 或第二套共享组件前缀。
4. 本轮不把 `接入智维总览` 做成重型驾驶舱大屏，不承接子页完整明细。

## 4. 设计结论

### 4.1 总体方案

采用“总览中枢、页面专台”的双轨母版方案：

- `接入智维总览`：中枢台，只负责“整体什么情况、先去哪处理”
- `7 个作业页`：专台，只负责“在这个域里现在怎么处理”

### 4.2 母版方向

统一母版优先采用 **A. 指挥甲板型**，但按页面属性做双轨收口：

- `资产治理组` 向 **B. 资产治理型** 倾斜
- `诊断实验组` 向 **C. 诊断实验室型** 倾斜

### 4.3 视觉原则

1. **颜色继续使用现有品牌体系**
   - 品牌橙：主操作、重点 CTA、轻强调边界
   - 云蓝：信息提示、结果辅助态、诊断次强调
   - 蓝灰：正文、说明、表格、描边
2. **统一浅色控制台语法**
   - 浅背景
   - 白卡
   - 轻描边
   - 轻阴影
3. **避免重复首页化**
   - 总览页表达跨模块态势
   - 子页只表达本域状态与本域动作

## 5. 信息架构与职责边界

### 5.1 `接入智维总览`

负责：

- 跨模块态势摘要
- 角色优先入口
- 最近动作恢复
- 推荐处理顺序
- 进入 6 个子模块的主跳转

不负责：

- 完整产品/设备台账
- 完整异常日志列表
- 完整 Trace 列表
- 完整数据校验明细
- 重筛选区与批量操作区

### 5.2 资产治理组

#### `产品定义中心`

只回答：

- 产品契约是否完整
- 产品库存和启停状态如何
- 物模型治理是否存在阻塞
- 当前批量治理应该怎么做

#### `设备资产中心`

只回答：

- 设备运行态是否稳定
- 拓扑关系是否清晰
- 认证、激活、状态是否齐全
- 当前设备治理该从哪类设备切入

### 5.3 诊断实验组

#### `链路验证中心`

只回答：

- 当前模拟配置是否正确
- 本次报文是否发送成功
- 时间线、响应和下一步排查建议是什么

#### `异常观测台`

只回答：

- 当前异常主要集中在哪个模块/设备/通道
- 哪些异常最值得优先筛查
- 下一步是否应联动到追踪或归档

#### `链路追踪台`

只回答：

- 当前 Trace / session / Topic 链路是否清晰
- message-flow 是否命中
- 最近会话和失败归档怎样联动复盘

#### `数据校验台`

只回答：

- 文件快照与固件聚合是否完整
- MD5 / 分包 / 最近抓取是否正常
- 当前设备的数据校验结论是否成立

## 6. 统一组件语法

所有页面统一采用四层语法，但每层内容按页面类型分化：

### 6.1 Hero 中枢头部

统一作用：

- 表达页面身份
- 给出当前页主判断
- 承接主动作和强相关联动入口

统一结构：

1. 左区：页面身份、主标题、主判断、辅助标签
2. 右区：主动作组、相关联动入口、当前状态摘要

约束：

- Hero 只表达本页上下文，不再复刻导航首页
- 资产页 Hero 更稳重
- 诊断页 Hero 更强调结果和步骤

### 6.2 指标与提醒层

统一采用：

- `1` 张主提醒卡
- `3~4` 张指标卡

语义分工：

- 主提醒卡：回答“先做什么、为什么做”
- 指标卡：回答“当前域内状态如何”

约束：

- 禁止重新回到“4 张几乎同权重的平铺小卡”
- 总览页的指标使用跨模块口径
- 子页只使用本域口径

### 6.3 主工作区

统一包含：

1. 条件区
2. 主战区
3. 辅战区

其中：

- 资产治理组：主战区是台账，辅战区是治理提示或联动摘要
- 诊断实验组：主战区偏结果舞台，辅战区偏时间线/原始响应/建议动作

### 6.4 详情与抽屉语法

统一采用：

- 摘要秒开
- 顶部 Hero 摘要
- 分区字段卡
- 建议动作 / 联动区

约束：

- 产品、设备、Trace、异常详情都属于同一家族
- 不再出现“有的抽屉很运营，有的抽屉像纯接口回显页”的割裂感

## 7. 逐页设计

### 7.1 `接入智维总览`

定位：中枢台

页面节奏：

1. 中枢 Hero
2. 跨模块态势卡
3. 高优入口区
4. 最近动作恢复区
5. 推荐处理顺序区
6. 全部能力入口区

Hero 内容建议：

- 当前角色关注域
- 当前推荐优先处理域
- 最近 24h 接入稳定性摘要
- 最近动作恢复按钮

指标建议：

- 设备在线稳定性
- 最近 trace/验证活跃度
- system_error 趋势
- 最近需要继续处理的工作数

关键约束：

- 不放完整明细表格
- 不放重筛选
- 不做二级模块重复统计

### 7.2 `产品定义中心`

定位：资产治理专台

页面节奏：

1. 产品契约治理 Hero
2. 指标与治理提醒层
3. 台账主战区
4. 产品详情抽屉 / 物模型设计器抽屉

Hero 重点：

- 产品总量、启停状态、物模型治理提醒
- 新增产品、导出、批量操作、跳转关联设备

提醒层重点：

- 物模型候选提炼提醒
- 停用产品存在库存阻塞
- 最近活跃产品变化

主战区重点：

- 列表仍是主角
- 查询、批量治理、导出行为不绕路

### 7.3 `设备资产中心`

定位：资产治理专台

页面节奏：

1. 资产运行 Hero
2. 指标与运行态摘要层
3. 设备台账主战区
4. 设备详情 / 编辑 / 导入 / 更换抽屉

Hero 重点：

- 在线率、激活率、停用设备、拓扑异常
- 新增设备、批量导入、刷新和联动入口

提醒层重点：

- 父子拓扑异常
- 长时间未上报设备
- 激活或禁用治理建议

### 7.4 `链路验证中心`

定位：诊断实验专台

页面节奏：

1. 诊断 Hero
2. 当前状态指标层
3. 左操作右结果的主工作区
4. 时间线、响应和建议动作区

左区：

- 设备查询
- 传输方式
- 上报模式
- Topic / tenant
- Payload 编辑
- 模板和发送动作

右区：

- 当前发送状态
- 处理时间线
- 原始响应 / 请求回显
- 后续排查建议

### 7.5 `异常观测台`

定位：诊断实验专台

页面节奏：

1. 异常 Hero
2. 异常摘要指标层
3. 筛选与聚焦条
4. system_error 主台账
5. 详情抽屉

重点：

- 不做第二个首页
- 做“可快速聚焦的异常台账”
- 强化与链路追踪、失败归档的联动

### 7.6 `链路追踪台`

定位：诊断实验专台

页面节奏：

1. 追踪 Hero
2. 追踪摘要和 message-flow 指标层
3. 模式切换梁
4. 追踪列表主战区
5. 运维看板 / 最近会话辅战区
6. 详情抽屉

重点：

- 强化“追踪模式 / 失败归档模式”切换感
- 让最近会话和运维看板更像辅战区，而不是零散补丁卡

### 7.7 `数据校验台`

定位：诊断实验专台

页面节奏：

1. 校验 Hero
2. 校验摘要层
3. 设备查询入口
4. 概况 / 文件快照 / 固件聚合 / 原始响应四段式结果区

重点：

- 维持单设备工作台特性
- 提升结果区层级，不再像普通结果页

## 8. 总览与子页的防重叠规则

为避免与 `接入智维总览` 功能重叠，本轮统一遵守以下规则：

1. `接入智维总览` 保留“常用入口 / 最近使用 / 推荐操作 / 全部能力”。
2. 子页不再复刻“全部能力入口墙”。
3. 总览只展示跨模块摘要；子页只展示本域摘要。
4. 总览负责“先去哪”；子页负责“在这里怎么做”。
5. 子页跨页入口只保留 `1~2` 个强相关联动动作，不再承担导航中心职能。

## 9. 组件与实现约束

本轮视觉重构继续遵守项目现有前端治理规则：

1. 优先复用现有共享组件：`StandardWorkbenchPanel`、`PanelCard`、`MetricCard`、`StandardListFilterHeader`、`StandardAppliedFiltersBar`、`StandardTableToolbar`、`StandardDetailDrawer`、`StandardFormDrawer`、`StandardDrawerFooter`、`StandardButton`、`StandardChoiceGroup`。
2. 不为 7 个页面各自引入新的品牌色或另一套按钮系统。
3. 资产页与诊断页允许布局重心不同，但必须共享同一层级语法。
4. 详情与抽屉统一使用“摘要秒开 + 分区字段卡 + 建议动作”的同一家族设计。

建议按以下顺序落地：

2. `产品定义中心`
3. `设备资产中心`
4. `链路追踪台`
5. `异常观测台`
6. `链路验证中心`

- 先稳定总览与资产治理骨架
- 再收口最关键的诊断台
- 最后统一剩余实验型页面
