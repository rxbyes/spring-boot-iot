# 风险对象中心客户视角详情抽屉与去摘要列表设计

> 日期：2026-04-04
> 范围：`spring-boot-iot-ui`、`docs/02-业务功能与流程说明.md`、`docs/03-接口规范与接口清单.md`、`docs/15-前端优化与治理计划.md`、`docs/21-业务功能清单与验收标准.md`
> 页面：`/risk-point`
> 主题：把风险对象中心从“列表摘要 + 运维维护抽屉”优化为“客户可直接浏览详情、绑定信息下沉到详情抽屉、列表不再展示绑定摘要”的统一体验
> 状态：设计已确认，待书面 spec 审阅

## 1. 背景

风险对象中心最近已经补齐了正式绑定维护能力：

- 列表页可以看到风险点主档和绑定概览摘要。
- 行级可以进入“维护绑定”抽屉，按设备和测点维护正式 `risk_point_device` 关系。
- “待治理转正”仍保持独立抽屉，负责 pending 记录转正。

但从客户视角看，当前仍有三个体验问题：

- 列表里的“绑定概览”承担了过多详情信息，用户需要在主表中理解设备数、测点数、待治理数，视觉负担偏重。
- 页面没有只读“详情抽屉”，客户若只是想看清一个风险点当前绑定了哪些设备、哪些正式测点，也必须进入偏操作型的“维护绑定”抽屉。
- 当前浏览路径偏“后台维护心智”，不够像“客户先看档案和现状，再决定是否继续操作”的对象中心。

结合现有前端治理规则，也已有明确方向：

- `/risk-point` 的主列表应承载摘要，不应把完整绑定设备卡片和测点明细重新塞回主表。
- 正式绑定详情应继续下沉到抽屉。
- 详情、编辑、维护绑定、待治理转正应保持职责分离，不混成同一个大面板。

因此，这一轮优化的重点不是增加新业务动作，而是重构客户看到的阅读路径和信息层级。

## 2. 目标

- 让客户在风险对象中心里更自然地“先看详情，再做维护决策”。
- 把当前过重的列表绑定概览从主表彻底移除，避免列表承担任何绑定说明责任。
- 新增只读详情抽屉，统一承接绑定概况、风险点档案、设备与测点详情、待治理提醒。
- 同时支持“点风险点名称进入详情”和“行操作进入详情”，减少客户学习成本。
- 保持现有编辑、维护绑定、待治理转正三个动作边界不变，只把入口组织得更顺。

## 3. 非目标

- 本轮不新增风险对象中心的独立详情页面或新路由。
- 本轮不把“详情抽屉”和“编辑抽屉”合并。
- 本轮不把“维护绑定”和“待治理转正”合并。
- 本轮不新增后端接口，只优先复用现有 `getRiskPointById`、`listBindingSummaries`、`listBindingGroups`。
- 本轮不修改 `risk_point_device`、pending 治理、设备主档或产品物模型的业务语义。

## 4. 方案对比

### 4.1 方案 A：列表保留详细绑定概览，不新增详情抽屉

- 列表继续显示设备数、测点数、待治理数，用户通过“维护绑定”抽屉看细节。

优点：

- 实现改动最小。

缺点：

- 客户浏览体验仍旧偏“运维后台”。
- 主表继续承担解释性内容，视觉负担不降反升。
- “想看看现状”和“准备修改关系”仍然共用同一个入口，心智不清晰。

### 4.2 方案 B：新增详情抽屉，列表取消绑定状态列

- 列表不再保留任何绑定状态或摘要展示。
- 风险点名称可点击进入详情抽屉。
- 行操作新增“详情”入口。
- 详情抽屉内展示绑定概况、风险点档案、设备与测点明细，并承接后续动作入口。

优点：

- 最符合客户“先看对象详情，再决定后续动作”的阅读路径。
- 与项目现有“列表看摘要，抽屉看详情”的治理规则一致。
- 不改变现有维护动作语义，只优化入口组织和视觉层次。

缺点：

- 需要新增一个详情抽屉组件和对应测试。

### 4.3 方案 C：把绑定明细并入编辑抽屉

- 不新增详情抽屉，直接在编辑抽屉中加入绑定概览和设备测点信息。

优点：

- 组件数量更少。

缺点：

- 浏览和编辑混在一起，客户打开即进入“可改状态”，阅读压迫感强。
- 容易让详情字段和表单字段混排，破坏现有抽屉职责边界。
- 与“编辑抽屉是写入心智，详情抽屉是阅读心智”的共性模式不一致。

## 5. 选型

采用方案 B：`去摘要列表 + 独立只读详情抽屉 + 原有维护抽屉继续保留`。

原因：

- 它同时满足“更美观”和“更符合设计”的诉求，但不引入新路由或新业务语义。
- 它把客户最在意的绑定信息完全从主表搬到详情抽屉，降低列表密度。
- 它保留现有维护绑定、待治理转正的实现资产，属于稳定增量，而不是推倒重来。

## 6. 页面与交互设计

### 6.1 列表层

主表调整为以下口径：

- 保留风险点编号、名称、组织、区域、等级、风险态势、负责人、状态等基础列。
- 主表不再展示任何绑定状态、设备数、测点数或待治理摘要。
- 列表只保留风险点基础档案字段与详情入口，不再承担绑定信息解释责任。

### 6.2 详情入口

详情支持双入口：

- 风险点名称点击进入详情抽屉。
- 行操作新增 `详情`，和名称点击保持同一结果。

这样兼顾两类用户：

- 熟悉系统的用户，可以直接点名称快速浏览。
- 依赖明确动作按钮的用户，也能通过“详情”入口完成同样操作。

### 6.3 详情抽屉总体结构

新增 `RiskPointDetailDrawer`，统一复用 `StandardDetailDrawer`。

抽屉首屏结构建议为：

1. `绑定概况`
2. `正式绑定设备`
3. `风险点档案`

如存在待治理记录，在概览区和设备区之间插入轻量治理提示。

### 6.4 绑定概况

这一段要让客户一眼回答三个问题：

- 它现在绑定到了什么程度。
- 是否还有待治理残留。
- 下一步应该走维护绑定还是待治理转正。

建议展示内容：

- 绑定状态
- 正式绑定设备数
- 正式绑定测点数
- 待治理记录数

这部分优先使用概览卡表达，不直接出现维护动作按钮墙。

### 6.5 正式绑定设备

这一段承接客户最关心的绑定可见性。

展示规则：

- 按设备分组展示，每台设备一张卡片。
- 卡片头展示设备名称、设备编码、正式测点数量。
- 卡片内展示该设备下已生效的正式测点清单。
- 每个测点展示：
  - 测点名称
  - `metricIdentifier`
  - 绑定来源标签（如 `人工维护`、`待治理转正`）

如果当前风险点尚无正式绑定：

- 显示统一空态，文案聚焦“当前暂无正式绑定”。
- 若同时存在待治理记录，则在空态附近保留轻量提示，引导继续进入“待治理转正”。

### 6.6 风险点档案

这一段承接风险点自身的基础资料与审计信息，保持“只读详情”语义。

建议展示：

- 所属组织 / 所属区域
- 负责人 / 联系方式
- 风险点编号
- 风险点等级
- 当前风险态势
- 启停状态
- 风险点描述
- 备注
- 创建时间
- 更新时间

如果后续认为 `createBy / updateBy` 对客户可读性不高，本轮可不额外展示。

### 6.7 详情内动作入口

详情抽屉底部或正文尾部提供三个明确动作：

- `编辑风险点`
- `维护绑定`
- `待治理转正`

动作语义保持不变：

- `编辑风险点` 继续打开现有编辑抽屉。
- `维护绑定` 继续打开现有正式绑定维护抽屉。
- `待治理转正` 继续打开现有待治理转正抽屉。

详情抽屉本身不直接承接写入表单，避免把阅读和写入混在一起。

## 7. 数据与状态设计

### 7.1 复用现有接口

本轮优先复用已有接口：

- `GET /api/risk-point/get/{id}`
- `GET /api/risk-point/binding-summaries`
- `GET /api/risk-point/binding-groups/{riskPointId}`

职责分配如下：

- 列表页继续用 `listBindingSummaries` 预取当前页风险点的绑定摘要，用于详情抽屉秒开与动作分流，但不再直接渲染列表列。
- 详情抽屉打开时，用当前行数据先秒开头部，再静默请求 `getRiskPointById` 拉主档最新值。
- 详情抽屉同时请求 `listBindingGroups` 拉正式绑定设备与测点分组。

### 7.2 详情抽屉的秒开策略

详情抽屉采用“先秒开、后补数”的共享体验：

- 打开瞬间先使用列表行里的 `riskPointName / riskPointCode / orgName / regionName / responsibleUser / responsiblePhone` 和当前页 `bindingSummaryMap` 中的摘要数值渲染第一屏。
- 后台补拉 `getRiskPointById` 和 `listBindingGroups`。
- 若后台请求返回更完整或更新的值，再刷新详情内容。

这样可以避免详情抽屉打开时出现整屏等待。

### 7.3 过期响应保护

和现有维护绑定抽屉一致，详情抽屉也必须对以下场景做 request-id 或等价失效保护：

- 用户快速切换不同风险点详情。
- 用户关闭详情后，旧请求才返回。
- 用户从列表连续点击多个风险点名称。

要求：

- 旧请求不得回写新抽屉上下文。
- 关闭抽屉时应重置当前详情状态。

## 8. 组件与实现边界

### 8.1 新增组件

新增 `spring-boot-iot-ui/src/components/riskPoint/RiskPointDetailDrawer.vue`。

组件职责：

- 承接风险点只读详情。
- 负责组合风险点基础档案、绑定摘要和绑定分组数据。
- 向上抛出动作事件，例如：
  - `edit`
  - `maintain-binding`
  - `pending-promotion`
  - `close`

### 8.2 列表页调整

`RiskPointView.vue` 负责：

- 新增详情抽屉状态管理。
- 把风险点名称改造成可点击详情入口。
- 给行级动作新增 `详情`。
- 把“绑定概览”列替换为“绑定状态”列。
- 在详情抽屉动作触发时，复用现有编辑、维护绑定、待治理转正流程。

### 8.3 不新增后端接口

因为当前详情诉求主要是信息重组而不是数据新能力，本轮不建议扩后端新接口。

只有在实现中确认以下问题时，才再考虑补接口：

- 详情抽屉必须额外字段而 `getRiskPointById` 无法提供。
- `listBindingGroups` 缺少展示所需的绑定来源或审计字段。

若现有返回已满足页面展示，则继续保持最小后端改动。

## 9. 测试设计

### 9.1 前端组件与视图测试

至少补齐以下回归：

- 点击风险点名称会打开详情抽屉。
- 点击行操作 `详情` 会打开相同详情抽屉。
- 列表“绑定状态”在无绑定、有绑定、待治理残留等情况下展示正确。
- 详情抽屉首屏能使用列表已有数据秒开。
- 详情抽屉能在异步补数后展示正式绑定设备和测点。
- 用户快速切换不同风险点详情时，旧请求不会污染当前抽屉。
- 在详情抽屉内点击 `编辑风险点 / 维护绑定 / 待治理转正`，会正确转到现有流程。

### 9.2 回归边界

必须确保以下既有行为不回退：

- 维护绑定抽屉继续只负责正式绑定维护。
- 待治理转正抽屉继续只负责 pending 记录治理。
- 风险点编辑抽屉继续只负责主档写入。
- 列表分页、摘要刷新和现有绑定摘要加载逻辑不被打断。

## 11. 验收口径

本轮验收通过标准：

- 客户可通过风险点名称或行操作 `详情` 打开详情抽屉。
- 列表主表不再承载多行绑定概览堆叠，而是只显示轻状态。
- 详情抽屉内可清晰看到：
  - 风险点基础档案
  - 已绑定设备
  - 每台设备下的正式测点
  - 待治理提醒
- 客户在详情抽屉内可继续前往：
  - 编辑风险点
  - 维护绑定
  - 待治理转正
- 原有维护绑定与待治理转正行为不回退。

## 12. 风险与约束

- 如果列表直接渲染过多详情信息，容易再次回到“主表太重”的旧问题，因此必须坚持轻状态列。
- 如果把详情和编辑混在一个抽屉里，后续维护会重新出现阅读与写入冲突，因此必须保持只读详情抽屉独立。
- 如果详情抽屉不做失效保护，快速切换风险点时会再次出现旧响应回写问题。
- 当前仓库已有风险点维护相关未提交改动，实施时必须只在本需求涉及文件内工作，不回退其他脏改动。
