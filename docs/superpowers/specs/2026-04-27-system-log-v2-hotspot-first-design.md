# `/system-log` V2 体验升级设计（热点优先切片）

## 1. 背景

`/system-log` 当前已经形成了 `异常排查 / 观测热点 / 归档治理` 三张平级选项卡，也已经具备真实可用的数据基线和证据链能力：

1. `异常排查` 已经收敛为异常分组主表、异常明细、详情/证据/追踪动作。
2. `观测热点` 已经能查看慢点汇总、慢点明细、趋势 bucket 和调度任务台账。
3. `归档治理` 已经能查看异常摘要、批次台账、批次对比和确认报告预览。

但从用户体验上看，这三张页签还没有真正收成一套统一的“工作台语法”：

1. `异常排查` 已经更像工作台，`观测热点` 仍偏卡片堆叠，`归档治理` 仍偏治理台账直铺。
2. 三张页签的“主对象”不够统一，用户进入页面后仍然要先重新理解“这一屏到底该先看哪里”。
3. 页签之间都能下钻，但下钻方式和视觉层级不一致，导致整体美观性和熟悉感还不够稳定。

上一轮分析里已经形成了明确推荐：

1. 不再先给 `异常排查` 加新块，而是继续做减法和上下文固定。
2. 下一轮优先升级 `观测热点`，因为它最容易从“信息堆叠”进化成“真正的诊断工作区”。
3. 第二优先升级 `归档治理`，把它从“台账感”进一步拉向“治理工作区”。

因此本轮不做“三张页签一次性全改”，而是按推荐路径执行：

1. 先定义 `/system-log` V2 的统一体验语言。
2. 首轮真正落代码的切片只覆盖 `观测热点`，并顺手补齐少量页级统一样式。
3. `归档治理` 和 `异常排查` 在本 spec 中给出后续落位，但不在首轮实现里一起硬上。

## 2. 目标与非目标

### 2.1 目标

1. 为 `/system-log` 定义一套统一的 V2 工作台语法：`焦点条 -> 主对象列表 -> 联动下钻区 -> 行动入口`。
2. 首轮把 `观测热点` 改造成真正的主从联动工作台，而不是慢点卡片和调度台账简单堆叠。
3. 保持 `/system-log` 仍是一条路由、三张平级选项卡，不引入新的子路由或新的全局导航层。
4. 保留现有慢点、趋势、调度、证据链和批次详情接口，不通过新后端接口换体验。
5. 给后续 `归档治理` 和 `异常排查` 的统一收口提前留出一致的视觉和交互边界。

### 2.2 非目标

1. 本轮不重写 `/audit-log` 审计中心模式。
2. 本轮不同时重做三张页签的所有布局。
3. 本轮不新增 `system-log` 子路由、URL query 持久化协议或新的后端聚合接口。
4. 本轮不把证据抽屉、归档批次详情抽屉改成独立详情页。
5. 本轮不新增新的观测指标、治理动作或告警联动规则。

## 3. 方案对比与推荐

### 3.1 方案 A：只做轻量美化

做法：

1. 保持三张页签的结构不变。
2. 只微调间距、颜色、标题、卡片边框和空态。

优点：

1. 改动最小。
2. 风险最低。

缺点：

1. 结构问题不解决。
2. `观测热点` 和 `归档治理` 仍然不是工作台，只是更好看的信息堆叠。

### 3.2 方案 B：统一 V2 语法，首轮优先改 `观测热点`

做法：

1. 定义三张页签统一的工作台语法。
2. 首轮把 `观测热点` 改造成“热点清单 + 联动详情 + 调度伴随台账”的主从工作区。
3. 页级保留现有总览带和选项卡，只补最少量统一状态样式。

优点：

1. 收益最大，范围仍然可控。
2. 完全符合当前推荐执行顺序。
3. 能先把最有改造价值的一块做成样板，再复制到 `归档治理`。

缺点：

1. 首轮完成后，三张页签的统一程度会明显提升，但不会一步到位完全一致。

### 3.3 方案 C：一次性把三张页签全部改成新工作台

做法：

1. 同一轮同时重构 `异常排查 / 观测热点 / 归档治理`。
2. 把所有页签全部切到新视觉和新交互语法。

优点：

1. 最完整。
2. 最像一次正式“V2 大版本”。

缺点：

1. 范围过大，测试成本和回归风险都明显上升。
2. 容易把“体验升级”做成“大范围换皮 + 多点联动一起出问题”。

### 3.4 推荐

采用 **方案 B：统一 V2 语法，首轮优先改 `观测热点`**。

原因：

1. 这是收益最高、风险最稳的切片。
2. 它既回应了用户对整体美观性和体验一致性的诉求，也避免把首轮改动做得太散。
3. `观测热点` 天然适合做成样板：对象清晰、数据关系明确、联动价值高。

## 4. V2 总体体验语言

### 4.1 页级不变的骨架

`/system-log` 继续保留：

1. 轻量页头。
2. 顶部总览带。
3. `异常排查 / 观测热点 / 归档治理` 三张平级选项卡。
4. 页级共享抽屉：异常详情、证据抽屉、归档批次详情。

本轮不改成新的路由层，也不叠第二层导航。

### 4.2 三张页签统一的工作台语法

V2 统一采用：

1. `焦点条`
   - 用一句话回答“当前用户正在看哪一个主对象”
   - 顺手给出一到两个最重要的上下文字段
2. `主对象列表`
   - 这一屏真正的主角
   - 点击某一行后，下面或旁边的联动区跟着切换
3. `联动下钻区`
   - 不再是孤立的第二块内容
   - 而是“当前选中主对象”的细部视图
4. `行动入口`
   - 行级详情、证据、追踪、复制等动作保留在离对象最近的位置

这套语法对应到三张页签的主对象分别是：

1. `异常排查`：异常簇
2. `观测热点`：慢点热点
3. `归档治理`：归档批次

### 4.3 状态与视觉统一

三张页签统一采用：

1. 同一套 section 标题语法。
2. 同一套 `loading / empty / error / selected` 状态层级。
3. 同一套“主对象选中后”的高亮和焦点条样式。
4. 同一套紧凑表格/台账边框和背景语言。

目标不是把所有内容都做成表格，而是让“主对象列表”和“联动区”一眼能分清层级。

## 5. 三张页签的 V2 目标形态

### 5.1 `异常排查`

这一块已经最接近 V2 目标形态，本轮只定义后续方向，不做大改：

1. 保留当前异常分组主表 + 展开明细表。
2. 在主表上方补一条更明确的 `当前排查对象焦点条`：
   - 当前异常模块
   - 异常类型
   - 异常编码
   - 发生次数 / 影响范围 / 最近时间
3. 把“当前没有选中异常簇”“正在查看全部异常明细”这类状态收敛到焦点条里，不再分散提示。

它在 V2 中的角色是“默认首屏、最直接的排查工作台”，不再继续加杂项信息块。

### 5.2 `观测热点`

这是本轮首要落代码的对象。

目标形态：

1. 顶部是一条 `热点焦点条`
   - 当前选中的慢点对象名
   - 当前趋势窗口
   - 最近 trace
   - 最近发生时间
2. 中间是 `慢点热点主表`
   - 一行一个热点对象
   - 行内字段只保留最能判断优先级的列
3. 下方是 `联动下钻区`
   - 左侧或上半区：最近样本明细
   - 右侧或下半区：趋势 bucket
4. 调度任务台账不再孤立放成另一大块，而是降为同一工作区里的伴随台账：
   - 作为“与当前热点同屏的系统背景信号”
   - 保持证据动作，但不抢主角

它在 V2 中要回答的是：

1. 现在最慢的是哪类热点？
2. 这类热点最近的真实样本是什么？
3. 它是偶发尖刺还是持续趋势？
4. 同一时间段调度任务是否也出现异常信号？

### 5.3 `归档治理`

本轮不完整实现，但先定目标形态，避免后续再换语言：

1. 顶部是一条 `治理焦点条`
   - 当前筛选口径
   - 当前选中的异常摘要卡
   - 最近异常批次定位状态
2. 主对象列表是 `归档批次表`
   - 一行一个批次
   - 重点列围绕状态、对比结论、确认值、删除值、剩余过期量
3. 联动区继续由详情抽屉承接：
   - 批次对比
   - 报告预览

它在 V2 中的定位是“治理工作区”，而不是仅仅展示批次台账。

## 6. `观测热点` 首轮实现设计

### 6.1 主对象定义

`观测热点` 的主对象固定为 `slowSummaryRows` 中的一条慢点汇总记录。

也就是说：

1. 用户先选“哪类慢点最值得追”
2. 然后再看该热点的最近样本和趋势
3. 调度任务台账作为旁证，不再和热点本身争主角

### 6.2 布局重构

把当前“卡片式慢点 Top + 条目式慢点明细 + 条目式趋势 + 独立调度任务台账”改成三段：

1. `热点焦点条`
2. `热点主表`
3. `热点联动区`

#### 6.2.1 热点焦点条

固定放在 `观测热点` 页签头部，显示：

1. 当前热点名称
2. 当前目标对象或目标 URL
3. 最近 traceId
4. 最近发生时间
5. 当前趋势窗口（`24 小时 / 7 天`）

行为：

1. 若尚未选中热点，则默认选中首个热点并显示其信息。
2. 若慢点汇总为空，则焦点条显示“暂无热点”空态副本，不再显示无意义占位。

#### 6.2.2 热点主表

把当前 `audit-log-slow-summary__grid` 卡片列表改成紧凑表格或等价的表格化列表。

建议列：

1. 热点名称
2. 目标对象
3. 最新时间
4. 最大耗时
5. 平均耗时
6. 总次数
7. 操作

操作只保留高频动作：

1. `证据`
2. `最近样本`
3. `趋势`

交互规则：

1. 点击整行或 `最近样本`，切换当前热点并联动最近样本区。
2. 点击 `趋势`，切换当前热点并联动趋势区。
3. `证据` 继续直达 trace 证据抽屉。

#### 6.2.3 热点联动区

联动区保留两块，但要更明确地作为“当前热点的细部视图”：

1. `最近样本`
2. `趋势观察`

推荐做成同一块 surface 内的上下双区，而不是两个完全平级的分散 section。

最近样本区展示：

1. span 名称
2. traceId
3. 状态
4. 开始时间
5. 耗时
6. 证据动作

趋势观察区展示：

1. bucket 时间
2. 总次数
3. P95
4. P99
5. 平均值
6. 最大值
7. 错误率

趋势窗口切换继续保留在联动区头部，但文案、布局和焦点条保持同一语气。

### 6.3 调度任务台账的降噪处理

调度任务台账继续保留在 `观测热点` 页签内，但角色降级为“伴随台账”：

1. 放在联动区下方或侧下方，视觉层级低于热点主表和热点联动区。
2. 标题更明确为“最近调度信号”或“伴随调度台账”，避免看起来像第二个主列表。
3. 仍然保留 trace 证据按钮。

这样页面会更像：

1. 先判断慢点热点
2. 再看该热点的细部证据
3. 最后顺手核对调度背景

而不是三块信息并列抢注意力。

### 6.4 状态记忆

首轮在现有本地状态基础上补充并明确：

1. 当前选中的慢点热点 key
2. 当前联动视角默认保持 `最近样本`
3. 当前趋势窗口继续记忆
4. 切出页签后返回，优先恢复上一次选中的热点

### 6.5 降级与错误处理

1. `slowSummary` 失败：
   - 热点焦点条显示失败态
   - 主表显示错误态
   - 不阻断调度任务台账
2. `slowSpan` 失败：
   - 只影响最近样本区
3. `slowTrend` 失败：
   - 只影响趋势区
4. `scheduledTask` 失败：
   - 只影响伴随台账

V2 的原则是：错误留在当前子区，不把整张页签一起拖成“全屏不可用”。

## 8. 测试与验收

### 8.1 前端测试

至少覆盖：

1. `观测热点` 默认命中首个热点并渲染焦点条。
2. 点击热点主表行后，最近样本和趋势联动到该热点。
3. 切换趋势窗口时，只刷新趋势区，不影响热点主表和调度台账。
4. `证据` 动作仍然按 traceId 打开共享证据抽屉。
5. `scheduledTask` 错误不会阻断热点主表显示。
6. 页签切换后返回，热点选择状态仍然恢复。

### 8.2 浏览器复核

本地浏览器至少核对：

1. `观测热点` 首屏阅读顺序是否变成“先主表、再联动、再调度”。
2. 主对象选中态是否足够明显但不过度跳。
3. 趋势窗口切换是否自然。
4. 小屏下表格是否会回退成合理的紧凑列表，而不是重新变回大卡片堆叠。

1. `node node_modules\\vitest\\vitest.mjs --run src/__tests__/views/AuditLogView.test.ts src/__tests__/components/auditLog/AuditLogHotspotTabPanel.test.ts`

## 10. 结论

本轮 `/system-log` V2 不做一次性大翻修，而是采用“**统一语言，热点优先**”的推荐路径：

1. 页级继续保留现有总览带和三张平级选项卡。
2. 三张页签统一朝 `焦点条 -> 主对象列表 -> 联动下钻区 -> 行动入口` 的工作台语法演进。
3. 首轮代码实现只聚焦 `观测热点`，把它从信息堆叠升级为真正的热点诊断工作台。

这样既能让页面更美观、更统一，也能把风险控制在一个真正值得优先投入的切片里。
