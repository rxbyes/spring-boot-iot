# `/system-log` 异常排查减负改版设计

## 1. 背景

当前 `/system-log` 的 `异常排查` 已经具备：

1. 筛选区  
2. 异常分组主表  
3. 分组展开后的异常明细  
4. 详情 / 证据 / 追踪等排查动作

这套结构在“信息关系”上是成立的，但在真实使用体验上暴露出两个明显问题：

1. **首屏信息负荷过高**  
   用户一进页就同时面对筛选、分组主表、分组展开明细三层内容，注意力很容易被过早切散。

2. **默认加载链路偏重**  
   当前页面会默认加载异常分组主表，再围绕选中的分组去带明细。对真实排查来说，这不是“按需分析”，而是“页面替用户先做了聚合分析”，导致首屏更慢、更重。

用户已经明确反馈：

1. 首屏一次性加载太多信息，认知负荷高  
2. 当前结构虽然有逻辑，但对用户查看和体验不够优化  
3. 异常分组主表更适合通过查询条件按需查看，而不是默认首屏直出

因此，这一轮不再继续强化“同屏聚合 + 同屏明细”的结构，而是要把 `异常排查` 收回到更轻、更符合排查节奏的模式。

## 2. 目标与非目标

### 2.1 目标

1. 把 `异常排查` 改成 **明细优先** 的默认首屏  
2. 把 `异常分组主表` 改成 **按需查询** 的分析视图，而不是默认首屏内容  
3. 降低异常排查首屏的数据加载量与用户认知压力  
4. 保留“异常分组帮助缩小排查范围”的价值，但把它变成用户主动进入的工具  
5. 保留现有的异常详情、证据、追踪、复制等排查能力

### 2.2 非目标

1. 本轮不新增后端接口  
2. 本轮不重做 `观测热点` 和 `归档治理`  
3. 本轮不把 `异常排查` 拆成独立子路由  
4. 本轮不重新引入“同屏分组主表 + 同屏展开明细”的双层堆叠结构  
5. 本轮不改变异常详情抽屉、证据抽屉、追踪入口的 contract

## 3. 方案对比与推荐

### 3.1 方案 A：明细优先 + 分组整页切换 + 选中后回明细

做法：

1. 默认首屏只显示筛选区和异常明细  
2. 在筛选区右侧提供明显的 `按异常分组查看` 按钮  
3. 点击后，列表区域整块切换成异常分组主表  
4. 用户点击某个异常分组后，自动回到异常明细，并用该分组条件缩小结果范围

优点：

1. 首屏最轻  
2. 分组真正变成按需分析工具  
3. 用户操作链路自然：先看真实异常，再决定是否需要聚合判断  
4. 性能收益最直接：首屏不再自动调用分组接口

缺点：

1. 分组和明细不再同屏，需要用户在两个视图间切换  

### 3.2 方案 B：明细优先 + 分组页内折叠区

做法：

1. 默认首屏只看明细  
2. 点击 `按异常分组查看` 后，在明细页内插入分组表

优点：

1. 改动相对保守  
2. 视图跳转成本低

缺点：

1. 最终仍会回到“同屏两块甚至三块内容”的状态  
2. 很容易重新出现用户不喜欢的页面压力

### 3.3 方案 C：查询优先 + 空态首屏

做法：

1. 首屏只显示筛选区和少量统计  
2. 用户点查询后再看明细或分组

优点：

1. 默认加载最轻  

缺点：

1. 对日常排查来说多了一步  
2. 牺牲了异常明细首屏的直接性

### 3.4 推荐

采用 **方案 A：明细优先 + 分组整页切换 + 选中后回明细**。

原因：

1. 它最符合用户已经明确表达的体验偏好  
2. 它兼顾了信息减负、性能优化和排查效率  
3. 它把“异常分组”从默认负担改成真正的分析工具

## 4. 总体交互语言

`异常排查` 改版后统一只有三种页面状态：

1. **异常明细默认态**  
   默认首屏，只显示筛选区、分组入口、异常明细表

2. **异常分组查看态**  
   用户主动进入后，列表区域整块切换为分组主表

3. **分组定位后的异常明细态**  
   用户从分组视图选中某个分组后回到明细页，明细表按该分组条件过滤，并显示轻量上下文提示

不再保留：

1. 同屏分组主表 + 同屏展开明细  
2. 默认首屏就加载聚合结果  
3. 大段解释性回退文案堆在列表上方

## 5. 异常明细默认态

### 5.1 页面结构

默认首屏只保留：

1. 筛选区  
2. `按异常分组查看` 按钮  
3. 异常明细表

不显示：

1. 异常分组主表  
2. 分组展开态  
3. 与分组相关的额外说明块

### 5.2 首屏加载策略

默认首屏只做两类请求：

1. 异常明细列表请求  
2. 异常排查必要的统计或页内辅助请求

明确取消：

1. 首屏自动加载异常分组接口

### 5.3 分组入口

`按异常分组查看` 放在筛选区右侧，作为明显的次工作流入口。

它的语义不是“只切页面”，而是：

1. 以**当前筛选表单值**作为查询条件  
2. 发起异常分组查询  
3. 进入异常分组查看态

这里强调使用的是“当前表单值”，而不是“上一次已生效查询条件”。  
这样用户修改了筛选项但还没点查询时，直接点 `按异常分组查看` 也能得到预期结果。

## 6. 异常分组查看态

### 6.1 页面结构

进入异常分组查看态后，列表区域整块切换成：

1. `返回异常明细`  
2. 异常分组主表

筛选区仍然保留在上方，不额外新增二级导航。

### 6.2 分组主表字段

分组主表只负责回答“哪类异常值得看”，建议保留这些字段：

1. 异常模块  
2. 异常类型  
3. 异常编码  
4. 发生次数  
5. 影响设备数 / 影响 Trace 数  
6. 最近发生时间  
7. 最近入口或最近摘要

不再在该页里展开明细表。

### 6.3 分页

分组主表保留分页，并比明细分页更轻：

1. 默认 `10条/页`

原因：

1. 分组页是辅助分析页，不是主排查页  
2. 更小的分页有助于降低感知负担

## 7. 点击分组后的回流方式

### 7.1 回流原则

用户点击某个异常分组后：

1. 自动返回 `异常明细视图`  
2. 自动按该分组条件缩小异常明细结果

用户不需要停留在分组页再多点一步。

### 7.2 回填字段

回到明细页时，自动把该分组的核心条件写回筛选：

1. `异常模块`
2. `异常编码`
3. `异常类型`

同时保留用户原本已设置的其他筛选条件，例如：

1. `TraceId`
2. `请求通道`
3. `目标 / URL`
4. `设备编码`
5. `产品标识`
6. `操作结果`

### 7.3 回流后的明细上下文条

当明细页处于“按分组定位”状态时，在明细表上方显示一条轻量上下文条，例如：

1. `当前按分组定位：message.mqtt / BizException / 500`

同时提供两个动作：

1. `清除分组定位`
2. `返回异常分组结果`

这条上下文条只承载定位语义，不做大段说明。

## 8. 分组结果缓存与失效

### 8.1 返回分组结果

当用户从分组页点中某个分组并回到明细页后，如果还想换另一组，不应该重新从零开始。

因此保留 `返回异常分组结果` 轻入口，并优先复用上一次分组查询结果。

### 8.2 失效规则

如果用户在明细页手动修改了筛选条件，则：

1. 原分组结果上下文失效  
2. `返回异常分组结果` 入口消失  
3. 分组定位上下文也同步清理

避免用户回到一份已经不对应当前筛选条件的旧分组结果。

## 9. 加载、空态与失败态

### 9.1 明细默认态

首屏只展示明细加载状态，不再额外出现分组相关 loading。

### 9.2 分组查看态空态

如果当前条件下没有异常分组结果，只显示轻提示：

1. `当前条件下没有异常分组结果`

并保留：

1. `返回异常明细`
2. 当前筛选区

### 9.3 分组查看态失败

如果分组接口失败：

1. 仍停留在分组查看态  
2. 显示简洁失败提示  
3. 提供：
   - `重试`
   - `返回异常明细`

不再把失败解释混入明细页，也不再自动回退成复杂混合态。

### 9.4 明细定位态清理

当用户点击 `清除分组定位` 时：

1. 清掉分组写回的三个核心字段  
2. 保留用户原本非分组来源的其他筛选值  
3. 重新刷新明细列表

## 10. 状态模型建议

前端建议新增并明确区分三类状态：

1. `errorViewMode`
   - `detail`
   - `clusters`

2. `clusterQueryContext`
   - 记录这次分组查询使用的筛选条件签名

3. `selectedClusterRefiner`
   - 记录当前明细页是由哪个分组定位回来的

4. `clusterRowsSnapshot`
   - 缓存最近一次可回访的分组结果

状态变化原则：

1. 进入分组页时写入 `clusterQueryContext`
2. 点分组回明细时写入 `selectedClusterRefiner`
3. 手工改筛选时清掉 `selectedClusterRefiner` 和 `clusterRowsSnapshot`

## 12. 测试与验收

### 12.1 前端测试至少覆盖

1. 默认首屏只加载异常明细，不自动加载异常分组  
2. 点击 `按异常分组查看` 后，使用当前表单值查询分组  
3. 分组视图整页切换成功，显示 `返回异常明细`  
4. 点击某个分组后，自动回到明细，并带入分组条件  
5. 明细页显示轻量分组定位上下文  
6. 修改筛选后，分组结果回访入口与定位上下文失效  
7. 分组空态与失败态行为正确

### 12.2 浏览器验收重点

1. 首屏是否明显变轻  
2. 进入分组页是否足够直接、不绕  
3. 点中分组后回到明细是否顺手  
4. 页面是否不再出现同屏多层内容的压迫感  
5. 首屏加载感知是否明显改善

## 13. 结论

这次 `异常排查` 改版的核心不是“去掉异常分组”，而是把它从**默认负担**改成**按需工具**。

最终目标是：

1. 首屏先让用户看到真实异常明细  
2. 需要聚合判断时，再主动进入异常分组  
3. 选中分组后自然回到明细继续排查  
4. 把性能压力和认知压力都从首屏移走

这比继续强化“同屏分组主表 + 同屏展开明细”的结构，更符合用户当前对体验和加载效率的真实诉求。
