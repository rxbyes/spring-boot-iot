# 风险绑定正式绑定设备搜索与目录接口收口设计

**Date:** 2026-04-25  
**Status:** Approved in-session for spec drafting  
**Audience:** 风险对象中心 / 前端实现 / 风险域后端实现 / 测试验收 / 文档维护  
**Scope:** 在不新增路由、不扩展视频绑定分支的前提下，收口 `/risk-point -> 风险绑定 -> 新增正式绑定` 的监测设备正式绑定主链，支持按设备编号搜索，并强制测点只消费正式风险目录接口。

## 1. 背景

当前风险绑定正式绑定链路已经有明确领域规则：

1. 正式绑定只允许绑定已发布到风险指标目录的正式字段。
2. 后端已有专用接口 `GET /api/risk-point/devices/{deviceId}/formal-metrics`。
3. 写侧也已有“非目录字段不得写入正式绑定”的一致性校验。

但当前页面实现仍有两个问题：

1. “新增正式绑定”设备下拉不支持按设备编号快速搜索，一线用户难以从大量候选里快速定位设备。
2. 页面测点加载实际仍走通用设备测点接口，而不是风险绑定专用正式目录接口，导致页面语义与既有设计、后端校验和用户预期不完全一致。

用户当前明确要求只修正“监测设备正式绑定”主链，不在本轮把摄像机等视频设备纳入该入口。

## 2. 目标

本轮只完成以下目标：

1. “新增正式绑定”中的设备选择支持按 `deviceCode` 快速搜索。
2. 设备选中后，测点只加载正式风险目录接口结果。
3. 当设备所属产品没有可绑定目录字段时，页面稳定给出统一空态提示。
4. 保持正式绑定领域规则、页面表现、接口调用三者一致。

## 3. 非目标

本轮明确不做：

1. 不新增一级路由、独立绑定页或第二套正式绑定页面。
2. 不把摄像机、IPC、视频设备纳入本次正式测点绑定入口。
3. 不新增视频能力绑定或设备级绑定交互。
4. 不改写风险域写侧规则，不放宽非目录字段写入。
5. 不改造待治理转正主链。

## 4. 方案比较

### 4.1 方案 A：仅修前端搜索

做法：

1. 设备下拉增加输入搜索。
2. 测点仍沿用通用设备测点接口。

优点：

1. 改动最小。

缺点：

1. 页面语义仍与正式绑定目录规则不一致。
2. 不能解决“页面看的是通用字段，提交时又被后端目录规则拒绝”的认知落差。

### 4.2 方案 B：新增正式绑定入口专用收口，推荐

做法：

1. 设备下拉增加过滤能力，重点支持按 `deviceCode` 搜索。
2. 测点加载统一切换到 `/api/risk-point/devices/{deviceId}/formal-metrics`。
3. 空态文案继续固定为“当前设备所属产品暂无可用于风险绑定的正式目录字段”。
4. 页面仍只承接监测设备测点级正式绑定。

优点：

1. 与既有设计文档和后端规则完全一致。
2. 不扩散范围，不引入视频绑定分支。
3. 回归面可控，能直接修正当前实现偏差。

缺点：

1. 需要同步回归正式绑定相邻交互，避免同页不同口径。

### 4.3 方案 C：在方案 B 上增加更细空态分类

做法：

1. 除方案 B 外，再细分“非监测设备”和“无目录字段”等空态原因。

优点：

1. 一线诊断更直接。

缺点：

1. 需要把更多设备能力语义显式暴露到页面，超出本轮最小修复范围。

### 4.4 选型

本轮采用 **方案 B：新增正式绑定入口专用收口**。

原因：

1. 它正好匹配当前用户确认的边界。
2. 它可以修正“页面走错接口”的实现偏差。
3. 它不引入摄像机支持等额外范围。

## 5. 目标态行为

### 5.1 新增正式绑定

`/risk-point -> 风险绑定 -> 新增正式绑定` 调整为：

1. 设备下拉仍保持单选下拉形态。
2. 设备下拉支持输入过滤，优先按 `deviceCode` 匹配，同时保留现有 `deviceCode - deviceName` 展示。
3. 选中设备后，前端只请求正式目录接口。
4. 测点下拉只展示正式目录接口返回的字段。
5. 当测点为空时，页面显示统一提示：`当前设备所属产品暂无可用于风险绑定的正式目录字段。`

### 5.2 摄像机与非本轮对象

本轮不在该入口中支持摄像机、IPC 或其它视频设备：

1. 不新增视频绑定动作。
2. 不在正式测点绑定抽屉中增加第二套交互。
3. 若某类设备不满足监测测点绑定前提，不在本轮通过页面扩展去兼容。

### 5.3 写侧一致性

写侧继续保持现有规则：

1. 前端提交的测点若不在正式目录接口结果内，后端仍会拒绝写入。
2. 不因为本轮前端修复而放宽领域规则。

## 6. 组件与模块边界

### 6.1 前端

前端改动范围限定在现有风险点页面与已复用的 API 封装：

1. `spring-boot-iot-ui/src/views/RiskPointView.vue`
2. `spring-boot-iot-ui/src/api/riskPoint.ts`（如需仅做调用对齐或类型复用）

页面不新增私有路由、不拆新页面。

### 6.2 后端

后端不新增新接口，复用现有正式目录接口：

1. `GET /api/risk-point/devices/{deviceId}/formal-metrics`

后端写侧校验和目录过滤逻辑保持现状，不做语义扩展。

### 6.3 模块职责

1. `spring-boot-iot-ui`：负责设备编号搜索和正式目录测点加载。
2. `spring-boot-iot-alarm`：继续承载正式目录接口和写侧约束。
3. 不把风险绑定目录语义重新回退到设备域通用测点接口。

## 7. 数据流

正式绑定数据流收口为：

1. 页面打开时加载风险点可绑定设备列表。
2. 用户在设备框输入设备编号，前端在现有候选列表上做本地过滤。
3. 用户选中设备后，前端调用 `/api/risk-point/devices/{deviceId}/formal-metrics`。
4. 前端使用该接口结果渲染测点下拉。
5. 用户提交绑定。
6. 后端继续按现有目录规则校验并写入正式绑定。

## 8. 异常与兼容策略

### 8.1 设备可选但没有目录字段

若设备可选中，但正式目录接口返回空列表：

1. 测点下拉为空。
2. 页面显示统一空态提示。
3. 不允许用户误绑到通用字段。

### 8.2 用户绕过前端直接提交

若用户绕过前端直接构造非目录字段请求：

1. 后端继续返回现有业务错误。
2. 本轮不新增额外旁路兼容。

### 8.3 历史绑定兼容

历史数据兼容策略保持现状：

1. 本轮只修新增正式绑定入口。
2. 不自动迁移、不回滚已有绑定数据。

## 9. 验收与测试

本轮最小验收集包括：

1. 设备下拉支持按设备编号搜索，展示仍为 `deviceCode - deviceName`。
2. 选中设备后，请求的是正式目录接口，而不是通用设备测点接口。
3. 有目录字段的监测设备能正常选择测点并提交正式绑定。
4. 无目录字段的设备会显示统一空态提示，且不能误绑。
5. 后端对绕过前端的非目录字段提交仍保持拒绝。

## 11. 风险与回归关注点

1. 同页若仍存在其它正式绑定相关动作，需确认是否也应继续沿用同一正式目录语义。
2. 前端若仅改设备搜索但未切接口，会继续造成行为偏差。
3. 若真实环境某设备产品确实没有风险目录字段，页面空态属于正确结果，不应误判为前端故障。
