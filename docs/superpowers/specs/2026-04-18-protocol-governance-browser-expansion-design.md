# 协议治理浏览器专项扩面设计

**日期**: 2026-04-18

**目标**: 在不扩大共享 `dev` 环境审批台账污染的前提下，把 `/protocol-governance` 浏览器专项从“草稿保存 + 试算 + 模板回放”扩展到“详情加载 + 模板发布快照 + 前端异常约束”，继续提升真实环境证据覆盖率。

## 1. 设计结论

本轮采用“低污染优先”方案，先补只读与低写入风险场景，不把协议族/解密档案的审批提交链路纳入浏览器专项。

本轮新增覆盖范围固定为：

1. 协议族详情加载并回填草稿编辑区
2. 解密档案详情加载并回填草稿编辑区
3. 协议模板详情加载并回填草稿编辑区
4. 协议模板发布快照
5. 批量操作的前端约束与异常提示

本轮明确不做：

1. 协议族单条发布审批浏览器验收
2. 协议族单条回滚审批浏览器验收
3. 解密档案单条发布/回滚审批浏览器验收
4. 协议族/解密档案批量审批真实写链路浏览器验收
5. 新增后端聚合接口或临时验收专用接口

## 2. 背景与约束

当前 `/protocol-governance` 已在真实 `dev` 环境完成首轮浏览器闭环，最新基线为：

- `logs/acceptance/protocol-governance-browser-summary-20260417163727.json`
- `logs/acceptance/protocol-governance-browser-results-20260417163727.json`
- `logs/acceptance/protocol-governance-browser-report-20260417163727.md`

首轮已覆盖：

- 登录
- 页面可达
- 协议族/解密档案/协议模板三块内容域可见
- 解密档案草稿保存
- 协议族草稿保存
- 解密命中试算
- 协议模板草稿保存
- 模板回放命中 `提取子通道 1`

当前页面实现与单测已经具备下一阶段所需的基础能力：

- `ProtocolGovernanceWorkbenchView.vue` 已有详情加载、模板发布快照、批量提交和错误提示逻辑
- `ProtocolGovernanceWorkbenchView.test.ts` 已覆盖详情加载、模板发布、批量发布/回滚的部分前端约束

共享环境约束保持不变：

- 分支固定为 `codex/dev`
- 必须使用真实 `application-dev.yml`
- 不回退 H2
- 浏览器专项应优先减少对共享审批台账和治理工作项的污染

## 3. 方案比较

### 3.1 方案 A：低污染扩面

在现有专项计划上新增详情、模板发布和异常约束场景，尽量只复用现有 `GET detail`、`POST template publish` 与前端本地校验。

优点：

- 改动最小
- 对共享 `dev` 污染低
- 能继续扩大真实环境证据
- 后续若审批链失败，问题边界更清晰

缺点：

- 仍未覆盖协议族/解密档案审批写链路

### 3.2 方案 B：一次打满审批链

直接把协议族与解密档案单条/批量发布回滚都纳入浏览器专项。

优点：

- 覆盖最全

缺点：

- 会持续向共享 `dev` 写审批单
- 更依赖固定复核人、审批状态与脏数据清理
- 一旦失败，浏览器计划、页面逻辑和审批链路会混在一起排障

### 3.3 方案 C：只补单测

只扩前端 `vitest`，不再增加真实浏览器专项。

优点：

- 实现最快

缺点：

- 不增加真实环境验收覆盖
- 不满足当前阶段“继续执行下一阶段”的目标

### 3.4 选型

本轮固定选择方案 A。

## 4. 实现设计

### 4.1 浏览器计划扩面

更新 `config/automation/protocol-governance-web-smoke-plan.json`，在现有两个场景基础上新增或细化以下步骤：

1. 详情加载步骤
2. 模板发布步骤
3. 批量异常约束步骤

计划应继续复用已有 `data-testid`，避免为自动化再造页面结构。

优先使用的选择器固定为：

- `protocol-family-detail-*`
- `protocol-profile-detail-*`
- `protocol-template-detail-*`
- `protocol-template-publish-*`
- `protocol-family-batch-submit-*`
- `protocol-profile-batch-submit-*`

### 4.2 详情加载验收口径

详情加载不是打开独立抽屉，而是命中后端详情接口后把结果回填到当前草稿编辑区。

浏览器专项应断言：

1. 点击详情按钮后命中对应详情接口
2. 草稿编辑区关键字段发生变化
3. 页面仍停留在 `/protocol-governance`

### 4.3 模板发布快照验收口径

模板发布是当前唯一可接受的低风险写场景，因为它不走独立审批链，只写模板发布快照。

浏览器专项应断言：

1. 点击模板发布按钮后命中 `POST /api/governance/protocol/templates/{templateId}/publish`
2. 发布成功后页面刷新列表
3. 页面保持在当前工作台

### 4.4 批量异常约束验收口径

本轮不把批量审批接口真实写入当作主目标，而是优先验证前端约束确实工作：

1. 未勾选任何记录时，批量操作应在前端直接失败
2. 批量回滚只允许已发布记录
3. 错误提示应可被浏览器断言到

若当前配置驱动执行器不足以直接消费 Element Plus message，则本轮允许用以下任一方式收口：

1. 为错误提示增加稳定文本断言点
2. 为批量无效操作增加不会触发 API 的页面断言

原则是尽量不改业务逻辑，只补稳定可验收的前端证据点。

### 4.5 前端测试补强

同步补强 `spring-boot-iot-ui/src/__tests__/views/ProtocolGovernanceWorkbenchView.test.ts`，确保以下语义有单测锁定：

1. 详情加载成功后草稿字段被覆盖
2. 模板发布调用正确接口并展示成功提示
3. 批量空选择时不会误发请求
4. 批量回滚只保留已发布记录
5. 部分失败或纯失败路径能展示 warning/error

## 6. 验证方案

本轮实现后的验证顺序固定为：

1. `vitest` 定向回归协议治理页面测试
2. 启动真实 `dev` 后端与前端
3. 执行 `node scripts/auto/run-browser-acceptance.mjs --plan=config/automation/protocol-governance-web-smoke-plan.json --no-append-issues`
4. 检查 `logs/acceptance/protocol-governance-browser-*.json/.md`
5. `git diff --check`

通过标准：

1. 计划新增步骤全部通过
2. 真实环境结果文件明确反映新增覆盖点
3. 无额外未说明的审批台账污染方案引入

## 7. 风险与后续阶段

本轮主要风险：

1. 配置驱动执行器对页面瞬时消息的可见性不足
2. 模板发布后列表刷新时序可能导致断言不稳定
3. 共享 `dev` 现有治理数据波动可能影响详情场景数据命中

对应策略：

1. 优先使用接口命中 + 草稿字段回填的双证据
2. 避免把 toast 本身作为唯一成功依据
3. 尽量复用当前已存在、已发布的稳定样例数据

下一阶段预留内容：

1. 协议族单条发布/回滚审批浏览器专项
2. 解密档案单条发布/回滚审批浏览器专项
3. 批量审批真实写链路浏览器专项
4. 协议治理与治理审批台联动专项
