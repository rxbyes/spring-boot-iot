# 产品定义中心语义减负与接入智维排障决策树设计

## 背景

当前 `/products` 实际已经承载了四层能力：

- 产品定义与台账维护
- 契约治理入口
- 版本治理入口
- 风险目录入口

但页面标题、描述、跨页跳转文案和接入智维总览口径仍容易把它误读成“只是产品管理页”。与此同时，`接入智维` 当前虽已存在：

- `/reporting` 链路验证中心
- `/message-trace` 链路追踪台
- `/system-log` 异常观测台
- `/file-debug` 数据校验台
- `/products` 产品定义中心
- `/devices` 设备资产中心

这些页面之间也已有一定联动，但“链路验证 -> 诊断分流 -> 产品/设备治理修正”的排障顺序仍主要散落在文案、按钮和经验里，没有被固化成统一决策树。

## 目标

本轮目标是同时完成两件事：

1. 让 `/products` 继续保持“产品定义中心”的名称与默认列表主视图，但显式表达它已经是产品定义主入口，并承接契约治理、版本治理与风险目录入口。
2. 把 `接入智维` 排障路径固定为统一决策树：`链路验证 -> 链路追踪 / 异常观测 / 数据校验 -> 产品定义中心 / 设备资产中心`，并明确总览页与各诊断页的职责边界。

## 设计结论

### 方案选择

采用“平衡收口”方案：

1. `/products` 保持原路由、原标题和列表主视图，不改造成治理驾驶舱。
2. 通过页级描述、治理提示、单产品工作区语义和跨页跳转文案，显式补齐“契约治理 + 版本治理 + 风险目录入口”的正式语义。
3. 由 `/device-access` 总览承担完整排障决策树。
4. 各诊断页只表达“当前节点 + 当前判断 + 下一步”，不重复展开完整树。
5. 产品定义中心与设备资产中心作为决策树终点页，只承接修正动作，不重复诊断说明墙。

## `/products` 设计

### 1. 页级定位

- 路由保持为 `/products`
- 页标题保持为 `产品定义中心`
- 页描述升级为正式语义，例如：
  - `统一维护产品定义，并承接契约治理、版本治理与风险目录入口。`

该描述必须同时回答两件事：

1. 这里首先仍是产品定义主入口。
2. 这里不是单纯产品台账页，而是治理链路的正式承接入口。

### 2. 首屏结构

首屏继续保持以下主结构：

1. 筛选区
2. 已生效筛选
3. notices / inline-state
4. 工具条
5. 产品列表
6. 分页

核心原则：

- 列表仍是默认主视图
- 不把治理待办墙放到列表之前取代主内容
- 不新增治理驾驶舱式首屏

### 3. 治理提示收口

`notices` 固定收口为两层语义：

1. **固定主语义提示**
   - 回答：这里不仅维护产品台账，也承接契约治理、版本追溯和风险目录入口。
2. **条件性治理提示**
   - 有治理待办时：回答当前聚焦产品仍有哪些治理动作未完成。
   - 无治理待办时：回答治理链路已收口，可继续抽检正式发布与策略有效性。

这样既保留台账主视图，又把治理语义显式化。

### 4. 单产品工作区语义

进入单产品后，继续保留现有工作区，不新增一级路由：

- `产品总览`
- `关联设备`
- `契约字段`

其中 `契约字段` 明确收口为单产品治理工作区，除 compare / apply 外，还继续承接：

- 契约治理
- 版本台账
- 回滚试算
- 风险目录联动入口

本轮不新增独立的“版本治理”或“风险目录”一级页签；仍保持这些能力挂在 `契约字段` 工作区内，以避免把 `/products` 再拆成新的多路由体系。

### 5. 跨页跳转语义

从链路追踪、异常观测、失败归档或链路验证跳转到 `/products` 时，目标语义统一改为：

- 去 `产品定义中心` 核对产品定义与契约基线
- 必要时继续进入 `契约字段` 完成治理修正

不再把 `/products` 的跳转含义弱化成“仅查看产品信息”。

## `接入智维` 排障决策树设计

### 1. 标准决策树

本轮固定的排障顺序为：

1. `链路验证中心`
2. 分流到：
   - `链路追踪台`
   - `异常观测台`
   - `数据校验台`
3. 最终收口到：
   - `产品定义中心`
   - `设备资产中心`

对应语义为：

- 诊断页负责给出证据和判断
- 治理页负责完成修正

### 2. `/device-access` 总览职责

`/device-access` 总览负责完整展示排障决策树，是唯一显式展开整棵树的页面。

它需要直接回答：

1. 先去哪处理
2. 再去哪看证据
3. 最后去哪修正

总览上的决策树应明确表达：

- **起点**：先到链路验证中心拿到本次模拟验证的 verdict、session、trace 或失败信号
- **诊断分流**：
  - 看固定 Pipeline 阶段与失败归档 -> 链路追踪台
  - 看后台 `system_error` 或异步异常 -> 异常观测台
  - 看原始响应、文件快照、聚合调试结果 -> 数据校验台
- **修正终点**：
  - 产品 / 协议 / 契约问题 -> 产品定义中心
  - 设备 / 认证 / 拓扑 / 在线状态问题 -> 设备资产中心

### 3. 诊断页职责边界

#### `/reporting` 链路验证中心

固定为排障起点页，只表达：

- 当前节点：排障起点
- 当前判断：尚未验证 / 验证成功 / 等待回流 / 验证失败 / 复盘受限
- 下一步：继续链路追踪 / 查看异常观测 / 打开数据校验

不再让该页重复承担整棵树的说明。

#### `/message-trace` 链路追踪台

固定为主链路复盘页，只表达：

- 当前节点：主链路复盘
- 当前用途：查看 Pipeline 阶段、Payload 对照、失败归档
- 下一步：
  - 后台异常证据不足 -> 异常观测台
  - 产品或契约问题 -> 产品定义中心
  - 设备身份或资产问题 -> 设备资产中心
  - 原始校验结果问题 -> 数据校验台

#### `/system-log` 异常观测台

固定为后台异常核对页，只表达：

- 当前节点：后台异常核对
- 当前用途：查看 `system_error` 与异步异常
- 下一步：
  - 有 trace -> 回链路追踪继续复盘
  - 明确是产品/契约问题 -> 产品定义中心
  - 明确是设备/认证问题 -> 设备资产中心

#### `/file-debug` 数据校验台

固定为数据与原始响应校验页，只表达：

- 当前节点：数据与原始响应校验
- 当前用途：查看文件快照、聚合调试和原始响应
- 下一步：
  - 数据结构或契约不匹配 -> 产品定义中心
  - 设备身份或资产不匹配 -> 设备资产中心
  - 需要回主链路核对 -> 链路追踪台

## 统一来源上下文设计

继续复用现有 `src/utils/iotAccessDiagnostics.ts` 上下文协议，不新增新的 query 协议。

来源提示统一收口为紧凑来源语义，只回答三件事：

1. 从哪来
2. 为什么来
3. 现在该做什么

例如：

- `来自链路追踪台，建议核对产品定义与契约基线。`
- `来自异常观测台，建议核对设备资产与认证信息。`

不再在目标页顶部新增说明墙、介绍卡或整棵决策树回放。

## 非目标

本轮不做以下改动：

- 不改 `/products` 路由
- 不新增 `/product-governance` 一类一级路由
- 不把 `/products` 改造成治理首页
- 不把“版本治理”或“风险目录”拆成独立一级 Tab
- 不新增“排障中心”一级页面
- 不让每个诊断页都展示完整决策树
- 不重写诊断上下文协议
- 不修改后端 API、数据库或真实环境验收链路

## 验收口径

### `/products`

- 页面标题仍为 `产品定义中心`
- 页面默认主视图仍为产品列表，不变成治理驾驶舱
- 页描述必须显式表达“产品定义 + 契约治理 + 版本治理 + 风险目录入口”
- 页级 notices 必须显式回答治理入口语义，而不只剩产品台账说明
- 从诊断页跳转到 `/products` 时，来源提示必须明确表达“核对产品定义与契约基线”

### `/device-access`

- 总览页必须显式展示完整排障决策树
- 决策树必须表达 `链路验证 -> 诊断分流 -> 治理修正终点`
- 总览不再只是松散入口集合或推荐顺序文案

### 诊断页

- `链路验证中心` 只表达起点语义和下一步
- `链路追踪台`、`异常观测台`、`数据校验台` 只表达当前节点和下一步
- 各诊断页不得重复展示完整决策树
- 产品定义中心和设备资产中心只承接修正动作，不回流诊断说明墙

### 来源上下文

- 继续复用现有诊断上下文能力
- 目标页只展示紧凑来源提示
- 不新增长说明、二级壳层或跨页大菜单
