# `/system-log` 异常排查双层联动表格设计

## 1. 背景

当前 `/system-log` 已经拆成 `异常排查 / 观测热点 / 归档治理` 三个工作台页签，系统模式入口在结构上比旧版更清楚，但 `异常排查` 首屏仍然存在两个体验问题：

1. 顶部横向总览条同时承担“摘要”和“导航”两种职责，用户能看到数量，但很难直接判断这些摘要与下面异常明细表之间的关系。
2. 下方异常明细表是一条日志一行，适合进入排查，但不适合先回答“当前最值得看的异常是哪一类”。

从现有实现看，这两层数据确实相关，但不是同一层：

- 顶部摘要来自 `getSystemErrorStats(...)`，是聚合统计。
- 下方表格来自 `pageLogs(...)`，是 `system_error` 明细分页。

因此，这次改动不应只是把两个视觉块拼得更近，而是要把“先判断异常簇，再下钻明细”的排查路径做成同一个工作区。

## 2. 目标与非目标

### 2.1 目标

1. 让 `/system-log` 的 `异常排查` 首屏先回答“哪类异常最值得看”，再无缝进入该类异常的真实记录。
2. 取消“顶部一条摘要横栏 + 底部一张明细表”之间的割裂感，改成以表格为中心的工作台。
3. 保留现有异常明细排查能力，包括筛选、分页、行内动作、详情抽屉、证据链和消息追踪跳转。
4. 保持 `观测热点 / 归档治理` 两个页签的职责和现有交互边界不变。

### 2.2 非目标

1. 不重做 `/audit-log` 业务审计模式。
2. 不改动异常详情抽屉、Trace 证据抽屉、归档治理抽屉的核心信息结构。
3. 不把 `观测热点` 与 `归档治理` 混入同一张超大表。
4. 不引入新的异常治理审批链或新的持久化真相表。

## 3. 结论

采用“**双层联动表格**”方案，并将现有顶部总览条降级为更轻的页签计数信息，不再保留它作为异常排查首屏的主要内容。

首屏从“摘要横栏 + 明细表”调整为：

1. `异常概览表`
   - 一行表示一个异常簇，而不是一条日志。
   - 用于回答“当前哪类异常最集中、影响面最大、最近还在发生”。
2. `关联异常明细表`
   - 一行仍然表示一条真实异常日志。
   - 自动跟随上方选中的异常簇，只展示该簇对应的明细。

这会把摘要层真正并入排查路径里，而不是把摘要放在明细上方当作装饰。

## 4. 信息架构

### 4.1 页面级结构

`/system-log` 系统模式仍保留三张一级工作台：

1. `异常排查`
2. `观测热点`
3. `归档治理`

但系统模式首屏不再保留独立的 `AuditLogSystemOverviewStrip` 作为主工作区第一层。改为：

1. 页签本身继续作为一级导航。
2. 页签标题展示轻量计数信息：
   - `异常排查` 展示系统异常总数。
   - `观测热点` 展示慢点摘要数量。
   - `归档治理` 展示异常批次数量。
3. `任务` 不再作为单独总览入口；它继续归属于 `观测热点` 工作台，由该页签内的调度任务台账承接。

这一步的目的，是消除“已经有页签导航，顶部又有一条伪页签横栏”的重复导航。

### 4.2 `异常排查` 工作台结构

`异常排查` 工作台改成单页双区：

1. 顶部保留现有筛选区和工具条。
2. 中部新增 `异常概览表`。
3. 下部保留 `异常明细表`，但数据源改为跟随概览表选中项联动。

布局顺序固定为：

1. 筛选区
2. 已应用筛选条
3. 工具条
4. 异常概览表
5. 关联异常明细表
6. 明细分页

## 5. 异常概览表设计

### 5.1 行粒度

异常概览表一行代表一个“异常簇”。本次设计将异常簇定义为以下维度的组合：

1. `operationModule`
2. `exceptionClass`
3. `errorCode`

说明：

- `operationModule` 回答“哪一段链路或哪个模块出了问题”。
- `exceptionClass` 回答“抛出的异常类型是什么”。
- `errorCode` 回答“业务侧是否已归类出稳定异常编码”。

`requestMethod`、`requestUrl`、`deviceCode`、`productKey` 不作为主聚合键，只作为概览辅助信息或明细过滤输入，避免异常概览表过度碎片化。

### 5.2 列定义

异常概览表默认列如下：

1. `异常簇`
   - 主标题：`operationModule`
   - 副信息：`exceptionClass`
   - 补充标签：`errorCode`
2. `发生次数`
3. `影响 Trace 数`
4. `影响设备数`
5. `最近发生时间`
6. `最近请求入口`
   - 优先展示 `requestUrl`，为空则展示 `requestMethod`
7. `最近异常摘要`
   - 展示最近一条 `resultMessage` 的压缩摘要

### 5.3 排序与默认行为

1. 默认按 `发生次数 desc` 排序，同次数按 `最近发生时间 desc` 排序。
2. 首次加载成功后，自动选中第一行异常簇。
3. 用户点击概览行后，下方明细表立刻刷新为该异常簇对应的日志集合。
4. 当筛选条件变化时：
   - 重新查询概览表。
   - 若原选中簇仍存在，则尽量保留选中。
   - 若原选中簇不存在，则自动选中新的第一行。

## 6. 关联异常明细表设计

### 6.1 角色定位

明细表继续承接真实排查动作，因此保留当前 `AuditLogErrorTabPanel` 的核心能力：

1. 行内查看详情
2. 跳转消息追踪
3. 打开 Trace 证据链
4. 设备、产品、请求、异常信息展示
5. 分页

### 6.2 明细表与概览表的联动规则

当上方选中某个异常簇时，明细表自动附加三项隐藏过滤条件：

1. `operationModule = selected.operationModule`
2. `exceptionClass = selected.exceptionClass`
3. `errorCode = selected.errorCode`

表现层面需要让用户感知当前正在看“联动后的局部结果”，因此：

1. 明细表标题改为 `关联异常明细`。
2. 标题旁展示当前选中异常簇摘要。
3. 提供一个清晰的 `查看全部异常明细` 或 `退出联动` 动作。
4. 退出联动后，明细表回到仅受顶部筛选控制的全量异常列表。

### 6.3 明细表字段

保留现有字段为主，不在本次改动中做大规模删改。默认展示优先级建议为：

1. `TraceId`
2. `deviceCode`
3. `productKey`
4. `requestUrl`
5. `operationModule`
6. `errorCode`
7. `exceptionClass`
8. `resultMessage`
9. `operationTime`
10. `operationResult`

## 7. 数据与接口设计

### 7.1 现状评估

现有后端已经提供：

1. `GET /api/system/audit-log/page`
   - 返回 `system_error` 明细分页。
2. `GET /api/system/audit-log/system-error/stats`
   - 返回总数、今日数量、去重 Trace 数，以及 `topModules / topExceptionClasses / topErrorCodes`。

这些接口足够支撑“顶部摘要 + 明细表”，但不足以支撑真正的异常簇联动表格，因为：

1. `system-error/stats` 只有单维 Top 桶，没有组合维度分页。
2. 用前端拿到的单页明细临时聚合，会把“异常概览表”做成当前页数据假象，不是真实全局结果。

### 7.2 推荐接口方案

新增系统异常簇分页接口，建议命名为：

`GET /api/system/audit-log/system-error/clusters/page`

查询参数沿用现有异常筛选口径：

1. `traceId`
2. `operationModule`
3. `requestMethod`
4. `requestUrl`
5. `deviceCode`
6. `productKey`
7. `errorCode`
8. `exceptionClass`
9. `operationResult`
10. `pageNum`
11. `pageSize`

返回行建议包含：

1. `clusterKey`
2. `operationModule`
3. `exceptionClass`
4. `errorCode`
5. `count`
6. `distinctTraceCount`
7. `distinctDeviceCount`
8. `latestOperationTime`
9. `latestRequestUrl`
10. `latestRequestMethod`
11. `latestResultMessage`

### 7.3 明细接口复用

下方明细表继续复用现有 `GET /api/system/audit-log/page`，不新增第二套明细真相链。

原因：

1. 现有明细表动作、详情抽屉、证据链已围绕这条接口构建。
2. 新增簇接口只负责回答“先看哪类异常”，不重复承担日志明细读取职责。

## 8. 前端组件边界

### 8.1 `AuditLogView.vue`

继续作为 `/system-log` 工作台总编排层，负责：

1. 页签切换
2. 筛选状态
3. 概览表状态
4. 明细表状态
5. 详情抽屉与证据抽屉状态

### 8.2 `AuditLogErrorTabPanel.vue`

从“单一异常表面板”升级为“异常双层工作台面板”，内部新增两个区块：

1. `异常概览表`
2. `关联异常明细表`

但它仍然只负责 `异常排查` 页签，不接管 `观测热点 / 归档治理` 的状态。

### 8.3 `AuditLogSystemOverviewStrip.vue`

本组件不再作为系统模式首屏必要结构。推荐处理方式：

1. 若无其他页面复用价值，则从 `/system-log` 中移除。
2. 若短期不删除文件，则停用组件引用，避免影响本次结构改造。

## 9. 交互与异常处理

### 9.1 正常流程

1. 用户进入 `/system-log?systemLogTab=errors`。
2. 页面加载异常概览表。
3. 页面自动选中第一条异常簇。
4. 明细表自动加载该异常簇对应日志。
5. 用户点击别的异常簇时，明细区切换。
6. 用户退出联动时，明细区回到全部异常列表。

### 9.2 空态

1. 概览表无结果：
   - 直接展示“当前筛选条件下没有异常簇”。
   - 明细区不展示旧数据。
2. 概览表有结果但联动明细为空：
   - 允许出现。
   - 提示“该异常簇暂无匹配明细，请调整时间范围或查看全部异常”。

### 9.3 错误态与回退

若新增异常簇接口失败，不应让整个 `异常排查` 工作台不可用。推荐回退策略：

1. 概览表区域显示 inline error。
2. 明细表自动回退为当前顶部筛选条件下的全量异常列表。
3. 工具条提示“异常概览加载失败，已回退为全部异常明细”。

这样即使新能力出现问题，用户仍能继续排查，不会因为首层聚合失败而失去明细工作台。

### 10.1 前端

新增或更新 `AuditLogView.test.ts` / `AuditLogErrorTabPanel` 相关测试，覆盖：

1. 概览表加载成功后自动选中首行。
2. 点击概览行会刷新明细表查询条件。
3. 退出联动后明细表恢复全量查询。
4. 概览接口失败时明细表回退到旧行为。
5. 筛选变化时概览与明细同时刷新。

### 10.2 后端

新增系统异常簇接口的 service/controller 测试，覆盖：

1. 组合维度聚合正确。
2. 空字段的归一处理稳定。
3. 筛选条件能同时作用于簇分页和明细分页。
4. 分页与排序结果稳定。

### 10.3 验证

至少执行：

1. `spring-boot-iot-ui` 相关单测

3. 前端本地工作台冒烟，重点检查：
   - 异常排查页签

## 11. 预期结果

完成后，`/system-log` 的 `异常排查` 首屏会从“两个相关但割裂的信息块”变成“一个先聚合判断、再联动明细”的排查工作台：

1. 用户第一眼能看出哪类异常最集中。
2. 用户第二眼就能进入该类异常的真实记录。
3. 现有证据链和排查动作继续可用。
4. 系统模式的导航层级更干净，不再出现重复的页签式摘要条。
