## 2. 三大业务域总览

### 2.1 三域定位

当前系统可以稳定归纳为三大后端业务域：

1. `IoT 主链路`
   - 核心对象：产品、设备、物模型、属性/遥测、消息日志
   - 核心职责：承接设备接入、协议解码、数据标准化、运行数据落库
   - 业务定位：上游数据源头
2. `风险平台`
   - 核心对象：风险点、绑定、规则、联动、预案、告警、事件、工单
   - 核心职责：把设备数据变成风险研判与处置闭环
   - 业务定位：消费 IoT 数据的业务中枢
3. `系统管理`
   - 核心对象：组织、区域、字典、通知渠道、审计日志
   - 核心职责：提供权限、归属、字典、通知和留痕能力
   - 业务定位：横向治理和支撑底座

### 2.2 三域主从关系

三域之间不是并列孤岛，而是稳定的主从关系：

1. `IoT 主链路 -> 风险平台`
   - IoT 主链路输出设备属性、遥测、状态和消息日志。
   - 风险平台消费这些结果，完成风险点识别、阈值命中、告警、事件和工单闭环。
2. `系统管理 -> IoT 主链路`
   - 组织、区域、字典、权限、审计为产品和设备治理提供范围和规则。
   - 通知渠道和审计日志为接入异常、系统异常和治理动作提供外部触达与留痕。
3. `系统管理 -> 风险平台`
   - 组织和区域决定风险点归属和处置范围。
   - 通知渠道承接告警、事件、工单外部触达。
   - 审计日志记录治理和处置行为。
4. `风险平台 -> IoT 主链路`
   - 风险平台通常不反向定义接入契约。
   - 但会反向约束“哪些设备、哪些测点、哪些字段”必须纳入重点治理和持续上报。

### 2.3 交付边界

为避免业务说明和交付口径混淆，本文固定使用三层边界：

1. `正式交付`
   - IoT 主链路主闭环
   - 风险平台主闭环
   - 系统管理基础治理能力
2. `当前可用`
   - 风险监测 / GIS 已进入真实环境基线
   - 产品物模型设计器真实可用，但仍按下一阶段增强管理
3. `规划中`
   - 设备中心更深扩展
   - GIS SDK / 底图 / 热力图
   - 远程维护、维修工单联动等

## 3. IoT 主链路五层治理模型

### 3.1 五层定义

`产品 -> 设备 -> 物模型 -> 属性/遥测 -> 消息日志` 不是简单的对象列表，而是五层递进关系：

1. `产品`
   - 定义一类设备的接入模板和业务身份
2. `设备`
   - 定义某台实际资产实例
3. `物模型`
   - 定义平台认可的正式字段标准
4. `属性/遥测`
   - 定义运行期最新值和历史值
5. `消息日志`
   - 定义原始报文证据和处理留痕

前 3 层属于“主数据治理”，后 2 层属于“运行数据治理”。

### 3.2 正确的业务顺序

业务治理顺序必须固定为：

1. 先定义产品标准
2. 再建设备实例
3. 再确认物模型字段
4. 再接入真实上报
5. 最后消费属性/遥测和消息日志结果

若顺序颠倒，会导致：

1. 字段解释不统一
2. 风险规则不可复用
3. 设备补档困难
4. 报表口径不一致
5. 排障无法回溯

## 4. 产品层：定义与录入规则

### 4.1 产品的本质

产品不是一台设备，而是“平台对一类设备接入契约的定义”。

当前产品层至少要定义：

1. `productKey`
2. `productName`
3. `manufacturer`
4. `protocolCode`
5. `nodeType`
6. `dataFormat`

### 4.2 产品层业务规则

1. `先建产品，后建设备`
   - 没有产品，设备不能正式建档。
2. `productKey` 是接入身份，不允许投运后随意修改
   - 它会影响 Topic、日志检索、设备建档和协议匹配。
3. `是否新建产品`，看接入契约是否变化，不看项目是否变化
   - 协议、Topic 结构、物模型、认证方式、下行能力、节点类型任一变化，都应评估拆成新产品。
4. `停用和删除产品` 必须先清库存
   - 删除前清理关联设备，停用前核查是否仍有启用设备。
5. `productName` 面向人读，`productKey` 面向机器识别
   - 二者职责不能混淆。

### 4.3 面向业务人员的录入口径

业务人员在建产品时，不应直接从字段入手，而应先回答：

1. 这类设备属于什么监测业务
2. 这类设备属于哪家厂商
3. 这类设备按什么协议和 Topic 接入
4. 这类设备需要承接哪些监测类型

### 4.4 产品层最小录入字段清单

建议产品录入时，至少固化以下字段：

1. `productKey`
   - 机器侧唯一身份
2. `productName`
   - 面向业务和运营的可读名称
3. `manufacturer`
   - 厂商标准名
4. `protocolCode`
   - 协议编码
5. `nodeType`
   - 直连 / 网关 / 子设备等节点类型
6. `dataFormat`
   - 上报数据格式
7. `description`
   - 产品业务说明
8. `status`
   - 启用 / 停用

### 4.5 产品命名与拆分规则

建议采用以下口径：

1. `productKey`
   - 推荐模式：`{vendorCode}-{scene}-{category}[-{protocolOrVersion}]`
   - 禁止包含中文、空格、项目名、站点名、客户名等易变信息
2. `productName`
   - 推荐模式：`厂商 + 场景/用途 + 品类 [+ 协议/版本]`
3. `manufacturer`
   - 只填厂商品牌标准名，不夹带项目和品类
4. 如果只是项目名、站点名、安装位置变化，不应新建产品
5. 如果协议、物模型、命令、Topic 结构变化，应评估新建产品

## 5. 设备层：定义与录入规则

### 5.1 设备的本质

设备是某个产品下的具体资产实例。

设备层至少要定义：

1. `deviceCode`
2. 所属产品
3. 安装位置
4. 区域 / 组织归属
5. 在线状态与最近上报时间
6. 必要的接入凭据或扩展信息

### 5.2 设备层业务规则

1. `一个设备必须且只能属于一个产品`
2. `deviceCode` 是实例唯一身份，不允许重复
3. 设备只补充实例信息，不重定义产品语义
4. 未建档上报不能直接算正式资产
   - 只能进入失败归档或未登记名单，待补档后转正
5. 基准站一包多测点场景下，父设备和子设备职责必须拆分
   - 父设备：负责原始上报、接入证据、在线状态
   - 子设备：负责测点属性、风险绑定、风险闭环

### 5.3 与当前系统实现的对应关系

当前系统已支持：

1. 已登记设备与未登记上报名单双名单治理
2. 通过 `iot.device.sub-device-mappings` 实现基准站一包多测点映射
3. 父设备保留原始 `$dp` 日志与在线状态
4. 子设备承接 `dispsX`、`dispsY` 等属性和后续风险闭环

### 5.4 设备层最小录入字段清单

建议设备录入时，至少固化以下字段：

1. `deviceCode`
2. `productKey / productId`
3. `deviceName`
4. `region`
5. `organization`
6. 安装位置 / 点位说明
7. `status`
8. 必要的认证或扩展信息

若存在父子设备关系，还应补充：

1. `parentDeviceId` 或 `parentDeviceCode`
2. 父子关系说明
3. 是否作为风险判定主体

### 5.5 父设备 / 子设备治理规则

面向业务人员，建议固定如下判断：

1. 若一台设备自己就是一个监测主体，则直接建单设备
2. 若一包上报中承载多个逻辑测点，则应拆分“父设备 + 子设备 / 子测点责任”
3. 父设备负责接入，不等于父设备负责风险判定
4. 能直接绑定风险点和规则的，应该是承担监测责任的真实设备或子设备

## 6. 物模型层：定义与录入规则

### 6.1 物模型的本质

物模型不是厂家原始字段表，而是平台认可的正式字段标准。

当前平台物模型继续沿用三类：

1. `property`
2. `event`
3. `service`

业务说明的重点是 `property`，因为它直接参与 latest、遥测、规则和报表。

### 6.2 物模型层业务规则

1. 不是所有原始字段都进入正式物模型
2. 只有长期展示、预警、统计、分析需要的字段才进入正式模型
3. 每个正式字段都必须明确：
   - 字段标识
   - 中文名称
   - 数据类型
   - 单位
   - 业务解释
4. 若字段需要公式计算，必须明确它是“原始字段”还是“派生字段”
5. 候选字段可以先从真实上报中提炼，只有人工确认后才能进入正式物模型

### 6.3 当前系统中的设计器口径

当前系统已具备：

1. 产品维度物模型设计器
2. 候选提炼 + 正式模型双模式
3. `iot_product_model` 同时服务运行链路与设计器治理

该能力当前真实可用，但仍按下一阶段增强管理，不并入 Phase 4 正式交付。

### 6.4 物模型层最小录入字段清单

建议每一个正式字段至少固化以下内容：

1. `modelType`
   - `property / event / service`
2. `identifier`
   - 平台标准字段标识
3. `modelName`
   - 中文名称
4. `dataType`
   - 字段类型
5. `unit`
   - 单位
6. `requiredFlag`
   - 是否必填 / 必传
7. `description`
   - 业务解释
8. `specsJson`
   - 取值范围、精度、映射等补充规则

### 6.5 物模型治理规则

1. 同一产品下 `identifier` 必须唯一
2. 字段名一旦进入正式模型，不应随意改名
3. 同一业务含义的字段，在不同产品间应尽量统一命名
4. “备注”中出现的业务解释、公式来源、上传规则，不应只停留在备注，应进入正式字段定义
5. 需要时序落库的字段，应同时确认 latest 和 telemetry 消费口径

## 7. 标准上报规范提炼：从图片到平台标准

### 7.1 原始规范的业务意义

用户提供的标准表头包括：

1. 监测内容
2. 监测内容编码
3. 监测类型
4. 监测类型编码
5. 数据字段
6. 单位
7. 备注

这张表本质上是“字段来源标准”，但若要指导业务人员工作，还需要提升成“平台治理标准”。

### 7.2 建议提升为三层标准

#### 第一层：监测业务 / 产品族标准

字段建议：

1. 监测内容
2. 监测内容编码
3. 适用产品类别
4. 业务说明
5. 是否已纳入正式交付

示例：

1. 监测内容：变形监测
2. 监测内容编码：`L1`

#### 第二层：监测类型标准

字段建议：

1. 监测类型
2. 监测类型编码
3. 所属监测内容编码
4. 说明
5. 对应风险对象类型
6. 是否已有现网接入

基于图片当前可提炼为：

1. 裂缝 `LF`
2. 地表位移 `GP`
3. 深部位移 `SW`
4. 加速度 `JS`

#### 第三层：字段治理标准

字段建议：

1. 所属监测内容编码
2. 所属监测类型编码
3. 字段标识
4. 中文名称
5. 单位
6. 数据类型
7. 是否 latest
8. 是否遥测
9. 是否核心判定字段
10. 是否平台计算字段
11. 上报要求
12. 业务解释
13. 特殊计算规则

### 7.3 基于图片提炼的字段治理初稿

#### `LF.value`

1. 中文语义：裂缝张开度 / 累计变化量
2. 单位：`mm`
3. 业务归类：核心判定字段
4. latest：是
5. 遥测：是
6. 风险判定：是
7. 说明：表示裂缝位移随时间累计变化量

#### `GP.gpsInitial`

1. 中文语义：GNSS 原始观测基础数据
2. 单位：原始对象或无量纲
3. 业务归类：原始资料字段
4. latest：可选
5. 遥测：可选或按资料保留
6. 风险判定：否
7. 说明：作为 `gpsTotalX/Y/Z` 的计算基础，不宜直接作为判警主指标

#### `GP.gpsTotalX / gpsTotalY / gpsTotalZ`

1. 中文语义：相对初始位置的 X/Y/Z 方向累计位移量
2. 单位：`mm`
3. 业务归类：核心判定字段
4. latest：是
5. 遥测：是
6. 风险判定：是
7. 说明：虽然需要公式计算，但属于正式业务指标

#### `SW.dispsX / dispsY`

1. 中文语义：深部位移累计变形量
2. 单位：`mm`
3. 业务归类：核心判定字段
4. latest：是
5. 遥测：是
6. 风险判定：是
7. 说明：当前应作为深部位移风险判定主指标

#### `JS.gX / gY / gZ`

1. 中文语义：三个坐标轴的加速度周期代表值
2. 单位：图示主单位为 `m/s²`，展示时可视业务换算为 `mg`
3. 业务归类：周期聚合字段
4. latest：是
5. 遥测：是
6. 风险判定：待规则定义
7. 说明：图中强调“一个上传周期内取绝对值最大时刻”，因此它不是原始连续波形流

### 7.4 业务上必须明确的字段分类

不是所有字段都按同等权重管理。建议固定分为四类：

1. `核心判定字段`
   - 直接用于阈值、风险、告警和报表
2. `辅助分析字段`
   - 用于展示和解释，但不直接判警
3. `原始资料字段`
   - 用于保留底层依据，不直接作为主业务指标
4. `周期聚合字段`
   - 表示一个上传周期提炼后的代表值，而非原始连续采样值

### 7.5 建议补充到字段治理标准表的列

若要形成可执行的业务标准表，建议在原始图片字段之外，再增加这些列：

1. 中文名称
2. 数据类型
3. 是否必传
4. 是否 latest
5. 是否遥测
6. 是否核心判定字段
7. 是否平台计算
8. 是否允许空值
9. 风险规则是否直接消费
10. 报表是否直接消费
11. 责任归属
    - 设备侧计算 / 平台侧计算 / 人工维护

### 7.6 字段治理表的使用方式

业务人员应按以下顺序使用字段治理表：

1. 先按 `监测内容编码` 找到业务大类
2. 再按 `监测类型编码` 找到该类设备的字段集合
3. 再判断字段属于：
   - 必传字段
   - 可选字段
   - 核心判定字段
   - 原始资料字段
4. 最后再决定这些字段是否进入：
   - 产品正式物模型
   - latest 展示
   - 遥测落库
   - 风险规则
   - 报表分析

### 7.7 南方测绘多维检测仪的单设备多能力建模示例

针对 `SK11E80D1307426AZ` 这类“一台设备同时监测倾角、加速度、裂缝”的场景，当前推荐固定为：

1. `1 个产品`
2. `N 台设备`
3. `24 个 property 定义（产品级）`

也就是：

1. 产品层只维护一套正式物模型
2. 同类实物设备都挂在这个产品下
3. 每台设备实例按同一套 `identifier` 上报对应的 latest / telemetry

这样既保持“单设备多能力”建模，又避免把“某一台样例设备”和“产品层建模口径”混写。

#### 建模前提

1. `deviceCode` 唯一，三个监测能力共用一个资产主体、一个安装点和一个总看板。
2. `L1_QJ_1`、`L1_JS_1`、`L1_LF_1` 是同一设备下的逻辑测点组，不是三个独立设备身份。
3. `S1_ZT_1` 是设备运行状态组，应与业务测点字段分组治理，但仍归属于同一设备。
4. 当前业务重点是统一建档、统一详情、统一趋势和统一看板，而不是按测点拆成独立责任对象。

#### 产品与设备口径

1. 推荐产品名称：`南方测绘多维检测仪`
2. 推荐 `productKey`：`south-survey-multi-detector-v1`
3. 产品说明建议固定为：`单台设备同时承载倾角、加速度、裂缝三类监测能力，状态组统一通过 S1_ZT_1 上报。`
4. 样例设备 `deviceCode=SK11E80D1307426AZ` 只是该产品下的一台实例，不新增子设备。
5. 后续同类设备应继续复用这个产品，按设备编码独立建档。
6. 若同类设备出现 `L1_QJ_2`、`L1_LF_2` 等新增逻辑测点，仍应优先沿用同一产品，在物模型中扩展新的 `identifier`。

#### 标识规则

当前推荐固定采用“`测点编码.字段名`”命名，而不是把厂家原始结构压平成新的私有缩写：

1. 监测属性：`L1_QJ_1.X`、`L1_JS_1.gX`、`L1_LF_1.value`
2. 状态属性：`S1_ZT_1.temp`、`S1_ZT_1.signal_4g`
3. 传感器状态：`S1_ZT_1.sensor_state.L1_QJ_1`

这样做有三个好处：

1. 保留厂家测点语义，便于联调和排障。
2. 同一产品下 `identifier` 仍然唯一，符合当前 `iot_product_model` 约束。
3. 后续风险规则、报表和测点扩展都可以直接围绕 `metricIdentifier` 工作。

3. `S1_ZT_1.sensor_state.*` 当前主要用于表达测点健康态，不应与业务测量值混写。

#### 当前业务下的治理结论

1. 当前明确保持“单设备多能力”，不拆父设备 / 子设备。
2. 产品层口径应表达为“`1 个产品 + N 台设备 + 24 个 property 定义`”，而不是“`1 个产品 + 1 台设备`”。
3. 风险规则、报表和看板应围绕同一设备下的不同 `metricIdentifier` 消费，不额外创造第二套资产身份。
4. 若后续业务改为“倾角 / 加速度 / 裂缝分别独立绑定风险点、独立承接工单和独立定位”，再升级为“父设备接入 + 子责任对象”模式。

## 8. 属性 / 遥测层：定义与治理规则

### 8.1 属性和遥测的区别

1. `属性`
   - 表达“当前最新值”
2. `遥测`
   - 表达“随时间变化的历史值”

当前系统中对应的主要落点为：

1. `iot_device_property`
2. `iot_device_metric_latest`
3. TDengine raw / latest / legacy 兼容结构
4. `iot_device_telemetry_point` 兼容回退表

### 8.2 latest 与 telemetry 的使用边界

建议业务上固定这样理解：

1. `latest`
   - 用于看“现在值是多少”
   - 用于列表、详情、当前状态、风险点当前值
2. `telemetry`
   - 用于看“这段时间怎么变化”
   - 用于趋势图、统计、回放、阈值分析和报表

若一个字段只保留 latest，不保留 telemetry，则业务上只能做“当前值查看”，难以做趋势分析。
若一个字段只保留 telemetry，不保留 latest，则列表和详情页难以稳定展示“当前值”。

### 8.3 属性 / 遥测层业务规则

1. 凡是需要实时查看当前值的字段，应具备 latest 口径
2. 凡是需要趋势、报表、规则和风险分析的字段，应进入遥测口径
3. 同一字段必须明确其业务类别：
   - 原始值
   - 累计值
   - 计算值
   - 极值
   - 状态值
4. latest 和遥测不是二选一，而是职责不同
5. 核心判定字段必须具备稳定 latest 和时序留存能力

### 8.4 属性 / 遥测层面向业务的检查项

业务联调时，建议每个核心字段至少检查：

1. latest 是否生成
2. latest 字段名是否与物模型一致
3. latest 单位和业务解释是否一致
4. 遥测是否成功落库
5. 遥测时间是否与实际上报时间一致
6. 风险规则读到的是哪个字段
7. 报表统计使用的是哪个字段

## 9. 消息日志层：定义与留痕规则

### 9.1 消息日志的本质

消息日志不是业务指标本身，而是“这次上报的原始证据和处理轨迹”。

它回答的是：

1. 哪台设备上报
2. 何时上报
3. 原始报文是什么
4. 平台按什么链路处理
5. 最终是否成功

### 9.2 消息日志与属性 / 遥测的关系

同一次上报，平台会产生两类结果：

1. `消息日志`
   - 保留原始证据和处理过程
2. `属性 / 遥测`
   - 保留业务值结果

### 9.3 消息日志的最小业务信息

建议业务上把消息日志理解为至少要能回答以下问题：

1. 谁上报的
2. 什么时候上报的
3. 用什么 Topic / 方式上报的
4. 原始报文是什么
5. 报文被解成了什么字段
6. 是否成功写入 latest / telemetry
7. 若失败，失败在哪个阶段

### 9.4 消息日志层业务规则

1. 消息日志必须保留原始上下文，不能只留解析结果
2. 失败上报也要留痕
   - 包括未建档、空报文、解密失败、解析失败等
3. 不能只看 latest，不看日志
4. 基准站 / 子设备场景下，父消息和子消息的职责要明确拆分
   - 父消息：接入证据
   - 子消息：业务结果与风险责任

## 10. IoT 主链路与运行流程说明

### 10.1 当前固定处理阶段

当前 HTTP / MQTT 上报统一进入固定 Pipeline：

1. `INGRESS`
2. `TOPIC_ROUTE`
3. `PROTOCOL_DECODE`
4. `DEVICE_CONTRACT`
5. `MESSAGE_LOG`
6. `PAYLOAD_APPLY`
7. `TELEMETRY_PERSIST`
8. `DEVICE_STATE`
9. `RISK_DISPATCH`
10. `COMPLETE`

### 10.2 面向业务人员的解释

这 10 个阶段可翻译为业务视角：

1. 接入平台
2. 识别 Topic / 接入方式
3. 解码协议和标准化字段
4. 校验这台设备是否存在、归属哪个产品
5. 记录原始消息日志
6. 刷新 latest 属性
7. 写入时序遥测
8. 刷新在线状态
9. 交给风险平台判定
10. 形成完整处理结果

## 11. 风险平台：对象、交互与业务规则

### 11.1 核心对象

风险平台当前核心对象为：

1. 风险点
2. 风险点与设备 / 指标绑定
3. 阈值规则
4. 联动规则
5. 应急预案
6. 告警
7. 事件
8. 工单

### 11.2 与 IoT 主链路的关系

风险平台不是自己产生监测数据，而是消费 IoT 主链路结果：

1. 设备 / 子设备为风险点提供数据来源
2. 物模型字段为规则提供判定对象
3. latest / 遥测为阈值和趋势提供输入
4. 消息日志为处置后的追溯提供证据

### 11.3 当前已确认的深部位移闭环规则

当前系统已形成的深部位移自动闭环口径包括：

1. 按 `risk_point_device` 绑定自动识别风险归属
2. 按毫米绝对值执行等级判定
3. 黄色生成告警
4. 橙色 / 红色继续自动生成事件并派工
5. 蓝色只更新风险点等级，不生成告警 / 事件
6. 默认按 `deviceCode + metricIdentifier + level` 做冷却去重

## 13. 面向业务人员的实际操作顺序

建议业务人员按以下 6 步执行：

1. `先判定监测业务类别`
   - 先确认监测内容，例如 `L1`
   - 再确认监测类型，例如 `LF / GP / SW / JS`
2. `再创建产品`
   - 明确产品名称、产品编码、厂商、协议、节点类型、数据格式
3. `再配置物模型`
   - 先定义核心判定字段，再补辅助字段和原始资料字段
4. `再建设备`
   - 明确设备编码、位置、区域、组织和父子关系
5. `再做上报联调`
   - 同时检查消息日志、latest 和遥测是否正常
6. `最后进入风险平台`
   - 绑定风险点、配置规则、联动和预案，再观察告警 / 事件 / 工单闭环

### 13.1 每一步的业务交付物

为便于业务执行，建议每一步都形成明确产物：

1. 监测业务类别
   - 输出：监测内容编码、监测类型编码确认结果
2. 创建产品
   - 输出：产品台账记录
3. 配置物模型
   - 输出：产品正式字段清单
4. 建设设备
   - 输出：设备台账和父子关系
5. 上报联调
   - 输出：消息日志、latest、遥测验证结果
6. 风险绑定
   - 输出：风险点绑定、规则、联动、预案和闭环验证记录

## 14. 面向业务人员的校验规则

建议固定以下最小校验规则：

1. 没有产品，不允许正式建档设备
2. 没有物模型，不允许宣称字段已纳入平台标准

7. 父设备和子设备责任边界必须明确
