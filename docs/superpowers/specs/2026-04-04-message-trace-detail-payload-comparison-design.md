# 链路追踪台详情三层 Payload 对照账本设计

> 日期：2026-04-04
> 范围：`spring-boot-iot-ui`、`spring-boot-iot-message`、必要时少量 `spring-boot-iot-protocol`
> 页面：`/message-trace`
> 主题：在链路追踪台详情抽屉中同时展示原始 Payload、解密后明文和解析结果，并保持现有处理时间线风格

## 1. 背景

`/message-trace` 当前已经完成列表与详情抽屉的第一轮治理：

- 列表页保持 `StandardWorkbenchPanel + StandardListFilterHeader + StandardTableToolbar + StandardDetailDrawer` 的共享语法。
- 详情抽屉已经包含 `消息概览`、`链路信息`、`处理时间线` 和 `消息内容` 四段。
- `处理时间线` 已固定复用 `StandardTraceTimeline`，并承担固定 Pipeline 的阶段复盘。

但当前详情抽屉仍有一个直接影响排障效率的问题：

- 列表里虽然能看到 `Payload 摘要`，详情里也能看到完整 `Payload`，但用户只能看到“日志里保存的原始内容”。
- 对于 `$dp` 加密报文、历史兼容报文或协议归一化场景，页面无法直观看到“系统实际解密出了什么”和“协议最终解析成了什么”。
- 用户排查上报异常时，需要在脑中来回推断“收到的密文、解出来的明文、解析后的结构”三层关系，成本偏高。

用户已经明确希望：

1. 详情里同时保留三层视角：`原始 Payload`、`解密后明文`、`解析结果`。
2. 三层内容是同权信息，不做聪明隐藏。
3. 即使其中某层当前没有数据，也要明确占位，而不是整块消失。
4. 处理时间线的设计风格保持现状。
5. 详情布局可以参考设备资产中心详情的 stage 组织方式，但不要把链路追踪台改造成另一套新页面。

## 2. 目标

- 让链路追踪台详情抽屉能够连续回答三个问题：
  - 设备实际上报了什么？
  - 系统解密后拿到了什么？
  - 协议最终解析成了什么？
- 保持现有 `消息概览 -> 链路信息 -> 处理时间线` 的阅读顺序不变，只增强时间线下方的内容区。
- 使用与设备资产中心详情一致的“stage 分区 + 统一代码块”语法，避免再长出页面私有视觉系统。
- 在 Redis 时间线过期时，仍能稳定展示原始 Payload，并对缺失的解密/解析结果给出诚实占位。
- 为后续实现计划提供明确的数据来源边界，避免前端先写布局、后端再临时补字段。

## 3. 非目标

- 不新增链路追踪台独立详情页或页签。
- 不重做 `StandardTraceTimeline` 的视觉样式、状态色或阶段顺序。
- 不改造链路追踪列表列结构、页头工具条或跨页导航方式。
- 不在本轮把解密结果或解析结果持久化到 `iot_device_message_log` 新字段。
- 不把文件类二进制内容直接原样灌入详情代码块。

## 4. 设计决策

采用 `A 方案：三分区对照账本`。

核心判断：

- 三层内容必须同权，最适合使用“同一区块内三张并列账本卡”的语法，而不是主次分明的主副布局。
- 用户已经明确要求“即使没数据也不要隐藏”，因此三张卡必须始终存在。
- 详情抽屉的整体节奏仍应沿用当前 stage 组织方式，避免把内容区改造成强交互式控制台。

## 5. 详情结构设计

### 5.1 总体顺序

详情抽屉保留现有上半段结构：

1. `消息概览`
2. `链路信息`
3. `处理时间线`

将现有单一的 `消息内容` 区块升级为：

4. `Payload 对照`

### 5.2 Payload 对照区

`Payload 对照` 是一个大的详情 stage，内部固定包含三张同权卡片：

1. `原始 Payload`
2. `解密后明文`
3. `解析结果`

三张卡片的语义定义如下：

- `原始 Payload`
  - 展示当前消息日志中保存的原始内容。
  - 对应用户“系统到底收到了什么”的第一视角。
- `解密后明文`
  - 展示协议解码阶段拿到的明文快照。
  - 对应用户“密文或兼容外层包被解开之后是什么”的第二视角。
- `解析结果`
  - 展示协议层归一化后的结构化上行结果快照。
  - 对应用户“后续业务处理真正消费的结构是什么”的第三视角。

### 5.3 视觉与排版

- 三张卡都使用相同的标题等级、边框、背景和代码块风格，不人为制造主次。
- 每张卡标题下允许保留一行短说明，说明数据来源。
- 代码块统一使用现有详情区的深色报文块语法，不引入新配色。
- 桌面端优先采用 `3` 列并排布局。
- 中等宽度降为 `2 + 1` 或单列堆叠，但阅读顺序必须始终固定为：
  - `原始 Payload -> 解密后明文 -> 解析结果`

### 5.4 缺失态

即使当前某层没有数据，也保留整张卡，并在代码块区域显示明确占位：

- `当前无解密结果`
- `当前无解析结果`

缺失态的设计原则：

- 不隐藏卡片。
- 不显示红色错误态。
- 不把“暂无结果”误导成接口异常。
- 必要时补一行轻量说明，例如“当前报文未经过解密”或“当前时间线已过期，无法恢复解析快照”。

## 6. 数据来源与后端边界

### 6.1 原始 Payload

继续沿用当前详情抽屉已经拿到的 `detailData.payload`。

结论：

- 这一层不需要新增接口。
- 即使 Redis 时间线过期，原始 Payload 仍然稳定可看。

### 6.2 解密后明文

优先从 `message-flow` 时间线中 `PROTOCOL_DECODE` 这一步的 `summary` 获取。

原因：

- 解密后的明文只在协议解码阶段最准确。
- 现有 `MessageFlowStep.summary` 已经具备承载结构化快照的能力。
- 这条链路不要求先改消息日志表结构。

设计要求：

- 加密或兼容外层包场景，将“解密后的可诊断明文快照”写入 `PROTOCOL_DECODE.summary`。
- 非加密普通报文允许为空。
- 内容应限制为可诊断文本，不为展示目的无限扩容 Redis 时间线。

### 6.3 解析结果

同样优先从 `PROTOCOL_DECODE.summary` 获取协议层归一化快照。

建议最小快照字段：

- `messageType`
- `deviceCode`
- `productKey`
- `dataFormatType`
- `properties`
- `events`
- `filePayload` 摘要
- `childMessageCount`

展示要求：

- 前端按 JSON 代码块原样展示。
- 不在本轮强制增加“字段摘要 + JSON 双展示”。
- 文件或二进制类报文只展示描述型摘要，不直接展开完整二进制。

### 6.4 时间线过期时的降级策略

时间线过期时：

- `原始 Payload` 仍然展示。
- `解密后明文` 和 `解析结果` 继续保留卡片，但显示缺失占位。
- 详情上方现有“时间线已过期，仅保留消息日志”提示保持不变。

这样可以把页面语义稳定收口为：

- 日志层信息仍可查。
- 解密/解析快照依赖时间线短期留存。
- 当前不是页面异常，而是观测窗口已过期。

## 7. 协议与 pipeline 改动边界

### 7.1 `spring-boot-iot-message`

`UpMessageProcessingPipeline.protocolDecode` 需要补充 `PROTOCOL_DECODE.summary`：

- 写入解密后明文快照（有则写，无则不写）。
- 写入解析结果快照。
- 保持当前已有的 `routeType`、`messageType`、`dataFormatType`、`childMessageCount` 等摘要字段不回退。

### 7.2 `spring-boot-iot-protocol`

仅在需要时补“可诊断快照提炼”逻辑：

- 协议层负责提供可靠的明文/解析结果原始素材。
- 不把页面样式、占位文案或前端布局判断塞进协议层。

### 7.3 `spring-boot-iot-ui`

前端只负责：

- 从详情时间线中提炼三层展示数据。
- 渲染 `Payload 对照` stage。
- 处理空态、过期态和响应式布局。

前端不负责：

- 自行猜测密文如何解密。
- 自行二次拼凑协议解析结果。

## 8. 组件与前端实现边界

建议保持 `MessageTraceView.vue` 仍为详情抽屉承载文件，但把 `Payload 对照` 提炼成局部可复用的展示单元，避免详情 template 继续膨胀。

推荐职责：

- `MessageTraceView.vue`
  - 管理详情抽屉状态、时间线加载、缺失态判断。
- 新增或内聚的局部展示组件
  - 负责三张内容卡和代码块的统一渲染。
- `types/api.ts`
  - 明确 `PROTOCOL_DECODE.summary` 中前端需要消费的快照结构。

设计原则：

- 不把解析 `summary` 的逻辑散落到多个 computed。
- 不让模板里直接堆过多 `if/else` 和 JSON 提炼逻辑。

## 9. 验证策略

至少覆盖以下场景：

1. 普通非加密 JSON 上报
   - 原始 Payload 有值
   - 解密后明文为空占位
   - 解析结果有值
2. `$dp` 加密报文
   - 原始 Payload、解密后明文、解析结果三层都可见
3. 时间线过期
   - 原始 Payload 仍显示
   - 解密后明文、解析结果显示明确占位
   - 现有过期提示不回退
4. 文件类或二进制类报文
   - 解析结果只展示可诊断摘要，不展开二进制正文
5. 响应式布局
   - 桌面端三列
   - 中小屏顺序不变地自动堆叠

前端测试建议补充：

- `MessageTraceView.test.ts`
  - 新增三层展示断言
  - 新增占位场景断言
  - 新增时间线过期场景断言

## 11. 风险与取舍

### 11.1 不扩消息日志表的取舍

优点：

- 改动小，风险低。
- 不引入新的持久化字段治理成本。
- 先复用现有 message-flow 时间线能力。

代价：

- 解密后明文和解析结果依赖 Redis 时间线短期留存。
- 一旦时间线过期，就只能展示原始 Payload 和占位提示。

这是本轮有意接受的取舍。

### 11.2 快照大小控制

如果把明文和解析结果完整无裁剪地写进时间线：

- 会放大 Redis 占用。
- 会让详情代码块失控。

因此本轮只接受“可诊断快照”，不接受“无限原文转储”。

## 12. 结论

链路追踪台详情本轮固定采用 `A 方案：三分区对照账本`：

- 上半段保留 `消息概览`、`链路信息`、`处理时间线`
- 下半段新增 `Payload 对照`
- 对照区固定展示：
  - `原始 Payload`
  - `解密后明文`
  - `解析结果`

数据来源优先使用 `消息日志 + PROTOCOL_DECODE 时间线摘要` 的组合模式，不在本轮扩消息日志持久化字段。

当时间线过期时，页面继续保留三张卡，但只稳定展示原始 Payload，其余两张卡使用明确占位，保持诊断语义诚实、稳定、可预期。
