# 设备资产中心未登记设备编辑转建档设计

> 日期：2026-04-12
> 范围：`spring-boot-iot-ui/src/views/DeviceWorkbenchView.vue`、`spring-boot-iot-ui/src/api/device.ts`、`spring-boot-iot-ui/src/__tests__/views/DeviceWorkbenchView.test.ts`、`spring-boot-iot-device`、`docs/02`、`docs/03`、`docs/08`
> 主题：让 `/devices` 设备资产中心中的未登记设备也可通过“编辑”入口直接完成建档，并在提交后切换为已登记状态
> 状态：设计已确认，已完成自检，待用户审阅

## 1. 背景

当前仓库中的资产治理台账以 `/devices` `设备资产中心` 为准。虽然任务描述中提到“产品资产中心列表”，但现有实现、文档和菜单口径都表明：已登记/未登记双名单联查发生在 `设备资产中心`，不是 `/products` `产品定义中心`。

当前设备资产中心已经形成如下基线：

1. `已登记` 行来自 `iot_device` 主档。
2. `未登记` 行来自 `iot_device_invalid_report_state`、`iot_device_access_error_log` 或 `dispatch_failed` 回退轨迹。
3. 页面已经有统一的设备抽屉表单，可维护产品、设备编码、父设备、状态、认证字段、部署位置和 `metadataJson`。
4. 前端当前只允许已登记设备进入编辑链路；未登记行会直接提示“未登记设备暂不支持编辑，请先完成建档”。
5. 后端现有 `POST /api/device/add` 在建档成功后，已经会自动把同 `productKey + deviceCode` 的 `DEVICE_NOT_FOUND` 无效上报最新态标记为 `resolved=1`，因此未登记噪音会自动收口。

当前缺口不是“没有建档能力”，而是“未登记行不能直接复用已有表单完成建档”。这导致用户已经在未登记名单里发现设备后，仍然必须额外走“新增设备”入口重新录入一次，资产治理路径割裂。

## 2. 目标

1. 让 `/devices` 中的未登记行也出现 `编辑` 入口。
2. 让未登记行打开后复用当前完整设备表单，而不是另做一套轻量补录表单。
3. 让未登记行提交后直接完成正式建档，并切换为 `已登记` 设备。
4. 继续复用现有 `POST /api/device/add` 建档能力和未登记状态自动消解逻辑，不新增重复接口。
5. 保持已登记设备的编辑、更换、删除、洞察等现有行为不变。

## 3. 非目标

1. 本轮不改 `/products` 产品定义中心。
2. 本轮不新增“未登记临时草稿”或任何平行资产表。
3. 本轮不开放未登记行的设备更换、删除或对象洞察。
4. 本轮不改未登记名单的数据来源与查询口径。
5. 本轮不引入新的审批、复核或审计流程。

## 4. 已确认决策

对话中已经明确确认以下边界：

1. 未登记行点击 `编辑` 后，应当直接完成建档并转成 `已登记`。
2. 未登记行打开的不是缩水版表单，而是与已登记设备一致的完整同款表单。
3. 用户优先接受“复用现有编辑抽屉，但未登记提交时实际走建档接口”的方案。

## 5. 方案对比与决策

### 5.1 方案 A：复用现有编辑抽屉，未登记行走预填建档

结构：

1. 未登记行也显示 `编辑`。
2. 打开当前统一设备表单抽屉。
3. 前端内部区分 `create / edit / register` 三种模式。
4. `register` 模式提交时调用现有 `POST /api/device/add`。

优势：

1. 与用户预期“像已登记设备一样修改参数”最一致。
2. 复用现有表单、校验、产品选项、父设备选项和大部分列表刷新逻辑，前端改动最小。
3. 后端直接复用现有建档闭环，不新增第二套接口语义。
4. 建档成功后可自动吃到已有的 invalid report resolve 能力。

代价：

1. 前端需要把“编辑按钮”背后的提交语义拆成“更新已有设备”和“基于未登记线索建档”两种。
2. 抽屉标题、按钮文案和成功提示必须更明确，避免用户误解为是在编辑一条真实已落库记录。

### 5.2 方案 B：未登记行单独增加“登记/补录”按钮

优势：

1. 语义最直观。

问题：

1. 与用户这轮明确提出的“未登记记录行中增加编辑功能”不一致。
2. 会在同一列表里形成两套入口文案，增加操作心智成本。

### 5.3 方案 C：新增后端“未登记转建档”专用接口

优势：

1. 服务端语义最纯粹，后续可扩展更强的来源审计。

问题：

1. 当前 `addDevice` 已能满足建档与自动消噪需求，短期新增接口收益不足。
2. 会扩大本轮范围，增加前后端联调与文档成本。

### 5.4 最终决策

采用 **方案 A：复用现有编辑抽屉，未登记行走预填建档**。

## 6. 详细设计

### 6.1 页面交互

`/devices` 列表保持现有 `详情 / 编辑 / 更多` 结构，但调整未登记行的动作口径：

1. 未登记行新增 `编辑` 直达动作。
2. `详情` 继续保持可用，用于查看来源档案与失败摘要。
3. `更多` 菜单仍不开放 `更换 / 洞察 / 删除`。

这样做的含义是：

1. `详情` 负责排障取证。
2. `编辑` 负责正式建档。
3. 未登记行不再只是只读发现名单，而是“发现后可直接转建档”的治理入口。

### 6.2 抽屉模式模型

当前抽屉主要通过 `editingDeviceId` 区分“新增/编辑”。本轮需要显式引入表单模式，建议前端内部增加：

1. `create`：普通新增设备。
2. `edit`：已登记设备编辑。
3. `register`：未登记行转正式建档。

对应语义：

| 模式 | 来源 | 提交接口 | 标题 | 确认按钮 | 权限 |
|---|---|---|---|---|---|
| `create` | 顶部“新增设备” | `POST /api/device/add` | `新增设备` | `提交设备建档` | `iot:devices:add` |
| `edit` | 已登记行“编辑” | `PUT /api/device/{id}` | `编辑设备` | `保存设备变更` | `iot:devices:update` |
| `register` | 未登记行“编辑” | `POST /api/device/add` | `登记设备` | `提交设备建档` | `iot:devices:add` |

这样可以避免继续把“是否有 `id`”当成唯一分支条件，也能让按钮、标题和权限控制更清晰。

### 6.3 表单字段与预填策略

`register` 模式复用现有完整设备表单，不裁字段：

1. 产品
2. 设备名称 / 设备编码
3. 父设备
4. 激活状态 / 设备状态
5. 设备密钥 / Client ID / 用户名 / 密码
6. 固件版本 / IP 地址 / 部署位置
7. `metadataJson`

预填规则：

1. 以当前未登记行摘要为基础预填：`productKey / deviceCode / deviceName / metadataJson / 最近载荷相关摘要` 中已有的可映射字段优先进入表单。
2. 若当前未登记行来自 `access_error` 且可补全失败归档详情，则继续沿用现有详情补全逻辑刷新摘要，但不要求为此新增写侧接口。
3. 对于未登记行天然不存在的正式设备字段（如真实 `id`、在线状态台账、正式父设备关系），不伪造运行态数据，只让用户在表单中补齐。

### 6.4 提交流程

#### 6.4.1 已登记编辑

保持现状：

1. 用户打开已登记行 `编辑`
2. 前端加载正式设备详情
3. 提交 `PUT /api/device/{id}`
4. 列表就地合并更新

#### 6.4.2 未登记转建档

新流程：

1. 用户打开未登记行 `编辑`
2. 前端进入 `register` 模式，并用未登记摘要预填完整表单
3. 用户补齐/修正字段
4. 前端提交 `POST /api/device/add`
5. 后端正式创建设备主档
6. 后端自动调用现有 invalid report resolve 逻辑，将同 `productKey + deviceCode` 的 `DEVICE_NOT_FOUND` 最新态标记为 `resolved=1`
7. 前端本地列表把该未登记行移除，并按筛选条件决定是否展示新建后的已登记设备
8. 最终统一再触发一次后台刷新，确保分页结果与服务端一致

### 6.5 列表刷新与本地状态

`register` 模式成功后需要区别于当前普通新增：

1. 当前筛选为 `registrationStatus=0` 时：
   - 直接从本地表格中移除该未登记行
   - 总数减一
   - 不强行本地插入已登记结果
2. 当前筛选为 `全部` 或 `已登记` 且新结果命中筛选时：
   - 可以先本地插入/合并新建后的正式设备
   - 同时移除旧未登记摘要行
3. 无论哪种情况，都继续执行一次静默 `loadDevicePage(force=true)`，用真实服务端分页结果校准本地缓存

缓存相关要求：

1. 同步清理该未登记摘要对应的详情缓存
2. 写入新建后正式设备详情缓存
3. 重新构建当前页缓存，避免同页同时残留“旧未登记 + 新已登记”双影子

### 6.6 权限口径

本轮权限保持现有菜单语义：

1. 未登记行显示 `编辑` 的前提是拥有 `iot:devices:add`，因为其本质是建档。
2. 已登记行显示 `编辑` 的前提仍是 `iot:devices:update`。
3. 若用户只有更新权限但没有新增权限，则未登记行不显示 `编辑`。
4. 若用户只有新增权限但没有更新权限，则已登记行不显示 `编辑`，但未登记行可以显示 `编辑`。

这样可以让“同一个按钮文案，不同记录类型走不同权限语义”保持可解释：

1. 已登记编辑是在改存量主档。
2. 未登记编辑是在创建正式主档。

### 6.7 文案与提示

需要明确区分两类语义，避免误导：

1. 未登记打开表单时，抽屉标题使用 `登记设备`。
2. 抽屉副标题和维护提示应明确说明：当前基于未登记上报线索创建正式设备档案。
3. 成功提示使用 `登记成功`。
4. 失败提示沿用后端真实错误，不包装成“更新失败”。

已登记设备维持：

1. 标题：`编辑设备`
2. 按钮：`保存设备变更`
3. 成功提示：`更新成功`

## 7. 错误处理

`register` 模式不新增专用异常模型，沿用现有后端真实业务错误：

1. 产品不存在或已停用
2. 设备编码已存在
3. 父设备不存在、不可访问或不满足拓扑要求
4. 当前登录人缺少主机构或数据权限不足
5. 表单字段校验不通过

前端要求：

1. 失败提示必须按“登记失败”语义表达，不混用“更新失败”。
2. 失败后保留用户已填写内容，不自动清空表单。
3. 若后台静默刷新失败，不回滚已经成功的建档结果，只提示“列表刷新失败，请手动刷新确认最新结果”。

## 8. 测试与验收

### 8.1 前端测试

至少补齐以下测试点：

1. 未登记行在满足 `iot:devices:add` 权限时也显示 `编辑`。
2. 未登记行点击 `编辑` 后，抽屉标题与按钮文案为登记语义。
3. `register` 模式提交调用 `addDevice`，而不是 `updateDevice`。
4. 建档成功后，本地列表会移除旧未登记行，并按筛选条件处理新已登记设备。
5. 已登记设备现有编辑行为不回归。

### 8.2 后端测试

若需要补齐后端回归，重点是保证现有建档闭环仍成立：

1. `addDevice` 成功后仍会执行 invalid report resolve。
2. 不因本轮前端改造而要求新增接口。

### 8.3 真实环境验收

按仓库真实环境规则，验收应使用 `application-dev.yml`：

1. 在 `/devices` 切到 `未登记`
2. 选中一条未登记上报记录，点击 `编辑`
3. 补齐完整字段并提交
4. 校验新设备出现在已登记名单
5. 校验原未登记记录不再继续回流
