# 产品物模型设计器二期治理工作流设计

**Date:** 2026-04-11  
**Status:** Approved in-session for spec drafting  
**Audience:** 产品定义中心 / 设备中心 / 治理控制面 / 前后端实现 / 文档维护  
**Scope:** 在不新增顶层路由、不扩散到资产扩展或远维工单的前提下，把现有 `/products -> models` 收束为一条完整、可理解、可审批、可追溯、可回滚、可验收的产品契约治理主流程。

## 1. 背景

当前 `/products` 内的 `ProductModelDesignerWorkspace` 已经不是单纯的“物模型编辑器”，而是同时承载了以下能力：

1. 手动样本输入与 compare 提取。
2. 契约字段发布审批与回执。
3. 发布批次台账、批次差异和风险目录联动。
4. 回滚试算、回滚审批和回滚结果。
5. 正式字段台账维护。
6. 采集器 / 子设备复合设备关系映射和边界治理。

这意味着一阶段能力并不缺少，而是已经逐步扩展成一个治理工作区。但当前仍存在几个结构性问题：

1. 页面信息密度高，多个治理块并列堆叠，用户难以快速判断当前所处阶段。
2. compare、审批、发布、回滚各自能用，但缺少统一的语义编排，用户容易混淆“已提交审批”和“已正式发布”。
3. 发布批次、风险目录联动和正式字段现状之间缺少清晰的阅读顺序，页面先后关系不稳定。
4. 采集器 / 子设备的治理边界虽已在后端逐步固化，但前端工作区仍需要更明确的阶段提示和职责表达。
5. 设备中心下一阶段已经冻结为“产品物模型设计器二期”这一条增强主线，继续散点扩展会削弱交付主题。

因此，本轮目标不是继续叠加孤立功能，而是把现有能力收束成一个稳定、可交付、可验收的治理主流程。

## 2. 设计目标

本轮只服务以下目标：

1. 把 `/products -> models` 明确定义为“产品契约治理工作台”，而不是松散的功能拼盘。
2. 让用户在一个工作区中顺序完成：样本输入、compare 决策、提交审批、发布追溯、回滚处理。
3. 把 compare、apply、release batch、rollback 的语义固定成明确的数据合同，便于前端、审批链路和后续 CI 门禁共用。
4. 明确并固定采集器 / 子设备的治理边界，避免父子产品职责再次摇摆。
5. 把当前能力沉淀为可复验的冻结增强项，并同步更新正式文档口径。

## 3. 非目标

本轮明确不做：

1. 不新增顶层页面、一级路由或独立治理子系统。
2. 不优先交付批量编辑器、模板化字段录入或复制已有字段能力。
3. 不把契约测试门禁总台、运行时健康驾驶舱或治理运营驾驶舱并入本轮主体。
4. 不扩散到设备资产扩展、远程维护、维修工单联动或上下架审批。
5. 不把当前治理能力定义为“任意厂商零代码接入平台”。

## 4. 方案对比与选型

### 4.1 方案 A：单工作区治理主流程收口，推荐

做法：

1. 保持 `/products -> models` 不变。
2. 把现有样本输入、compare、审批、发布、回滚和正式字段整理成一个主流程加两个辅助区。
3. 以“这一次治理如何推进”为主线组织页面，而不是继续堆叠功能块。

优点：

1. 完全延续现有 compare/apply/审批/回滚/批次台账能力。
2. 不新增路由和大范围领域模型。
3. 最适合本轮“主体先完成、范围不再发散”的交付要求。

缺点：

1. 需要重整已有工作区的信息层次。
2. 需要同步收口前端状态表达和部分返回语义。

### 4.2 方案 B：同页双视图收口

做法：

1. 在 `models` 工作区内拆成“治理流程视图”和“正式台账视图”两个主视图。
2. 让用户在两个视图间切换处理。

优点：

1. 治理中和正式现状的概念区分更强。

缺点：

1. 用户来回跳转成本更高。
2. 容易把本轮扩张成新的页面层级。

### 4.3 方案 C：治理流程独立化

做法：

1. 把 compare、审批、批次、回滚抽成新的治理工作区。
2. 正式字段台账仍然保留在现有产品页。

优点：

1. 架构边界最清楚。

缺点：

1. 改动面最大。
2. 容易从“二期收口”演变成“新系统建设”，不符合当前冻结增强项定位。

### 4.4 选型

本轮采用 `方案 A：单工作区治理主流程收口`。

原因：

1. 当前能力已经具备治理闭环雏形，缺的是工作区主流程的表达，而不是新系统。
2. 该方案与已完成的契约治理、审批、mapping runtime、风险目录发布是一条连续主线。
3. 该方案最符合“先完成主体，再列出下一步功能点”的执行原则。

## 5. 二期定位与边界

### 5.1 二期定位

“产品物模型设计器二期”不再定义成继续增加零散功能，而定义成：

**把现有产品物模型工作区收束为一条完整、可理解、可追溯、可验收的产品契约治理主流程。**

也就是说，二期的主语不是“编辑器”，而是：

**产品契约治理工作台。**

### 5.2 本期主目标

本期只完成 4 件事：

1. 工作流收口  
   让用户按固定顺序完成：样本输入 -> compare -> 生效决策 -> 审批 -> 发布追溯 / 回滚。
2. 状态反馈收口  
   明确用户当前所处阶段：未识别、已识别、待提交、审批中、已发布 / 可回滚。
3. 批次与风险联动收口  
   让发布批次、风险目录和影响范围可读、可追踪。
4. 采集器 / 子设备边界收口  
   固定哪些字段能在当前产品治理，哪些字段只能到子产品治理。

### 5.3 本轮完成后的用户结果

用户应能在一个工作区内回答以下问题：

1. 这次样本识别出了什么字段。
2. 哪些字段准备生效，为什么。
3. 当前审批走到哪一步。
4. 这次发布生成了哪个批次，并影响了哪些风险指标。
5. 如果需要撤回，能回滚什么，会影响什么。

## 6. 页面结构与主流程编排

### 6.1 页面结构不变，但语义重组

仍然保留：

`/products -> 产品经营工作台 -> models`

不新增路由，不拆新系统，但把当前工作区重组为：

1. 主流程区
2. 发布批次与风险联动辅助区
3. 当前正式字段台账辅助区

### 6.2 主流程区

主流程区固定为五段：

1. 流程总览
2. 样本输入
3. compare 决策
4. 提交审批
5. 审批结果 / 发布结果

顶部不再只是统计卡片，而要形成稳定的治理阶段条，建议固定五个阶段：

1. `待输入样本`
2. `待确认识别结果`
3. `待提交审批`
4. `审批中`
5. `已发布 / 可回滚`

### 6.3 样本输入区

样本输入区仍保留当前已有能力，但只保留必要动作：

1. `样本类型`：业务数据 / 状态数据
2. `设备结构`：单台设备 / 复合设备
3. 复合设备映射：父设备编码 + 逻辑通道编码 + 子设备编码
4. 操作按钮：格式化 JSON、提取契约字段

在复合设备模式下继续沿用既有边界：

1. 采集器产品页只治理采集器自身状态字段。
2. 子设备监测值和子设备 `sensor_state` 去子产品治理。
3. 当前页只提供边界提示和映射输入，不允许误把子设备正式字段发布到采集器产品。

### 6.4 compare 决策区

compare 区不再只是“识别结果展示区”，而是本轮治理的决策主战场，应突出四类信息：

1. 厂商原始标识与证据来源
2. 归一后的正式标识和字段语义
3. 规范字段匹配与风险就绪状态
4. 本次治理决策：create / update / skip / review

用户在这一段内要能同时看清：

1. 这个字段为什么出现。
2. 它映射到了哪个正式字段。
3. 当前建议动作是什么。
4. 为什么建议这样处理。

### 6.5 审批与发布反馈区

apply 回执、审批跟踪和执行结果应被组织成连续反馈区，而不是分散卡片：

1. 提交审批后立刻显示审批单状态。
2. 审批通过后同一区域直接切换为发布结果。
3. 同时呈现审批单号、发布批次号、新增 / 修订 / 跳过数量、审批意见和执行时间。

这样可以明确区分以下阶段：

1. 已提交审批
2. 审批中
3. 审批通过但前端尚未刷新执行结果
4. 审批通过且已生成正式发布批次

### 6.6 辅助区

主流程区下方固定两个辅助区：

1. 发布批次与风险联动  
   用于查看批次台账、批次差异、风险目录联动结果、回滚影响试算和回滚审批状态。
2. 当前已生效字段  
   用于查看和维护当前正式字段，保留 `property / event / service` 类型切换及必要的基础维护动作。

整个页面应形成稳定心智：

1. 上半区处理“这一次要不要发”。
2. 下半区处理“以前发过什么、现在生效什么”。

## 7. 后端契约与数据语义收口

### 7.1 总体原则

本轮不新增平行草稿表、平行治理表或第二套领域模型，继续复用现有主链路：

1. `compareProductModelGovernance`
2. `applyProductModelGovernance`
3. `pageProductContractReleaseBatches`
4. `rollbackProductContractReleaseBatch`
5. 审批单详情 / 重提
6. 风险目录与批次差异查询

本轮重点是固定语义和返回口径，而不是重建一套新系统。

### 7.2 compare 的固定语义

compare 在二期中应被定义为：

**一次基于样本输入形成的治理候选快照。**

返回结果至少要稳定表达以下六类信息：

1. 样本来源  
   业务数据 / 状态数据；单台设备 / 复合设备。
2. 原始字段证据  
   原始 identifier、逻辑通道编码、样本值、值类型、证据次数。
3. 归一结果  
   formal identifier、字段中文名、单位、数据类型。
4. 规范匹配  
   `normativeIdentifier`、`normativeName`、`riskReady`。
5. 与正式字段的对比结果  
   `double_aligned`、`formal_exists`、`suspected_conflict`、`manual_only`、`runtime_only`、`evidence_insufficient`。
6. 默认治理建议  
   默认建议 create / update / observe / review 的依据。

### 7.3 apply 的固定语义

`applyProductModelGovernance` 的默认返回值在二期中应固定为：

**审批提交回执，而不是“已正式生效”结果。**

返回口径明确分成两层：

1. 提交层  
   `approvalOrderId`、`approvalStatus`、`submittedItemCount`
2. 执行层  
   仅在审批通过并实际执行后才出现：`releaseBatchId`、`createdCount`、`updatedCount`、`skippedCount`、`executedAt`

### 7.4 release batch 的固定语义

`releaseBatch` 是物模型治理正式落库后的唯一追溯单位。以下语义统一围绕批次展开：

1. 这次哪些字段正式生效。
2. 是否同步发布了风险指标目录。
3. 对应的审批单是谁。
4. 发布理由和发布状态是什么。
5. 后续回滚时回滚的是哪一个正式版本。

### 7.5 rollback 的固定语义

回滚不是简单删字段，而是：

**基于发布批次的一次逆向发布操作。**

其流程固定为两层：

1. 回滚前  
   返回 impact preview，说明删除、恢复、回退的字段，以及受影响的风险指标、风险点、规则、联动和预案。
2. 回滚后  
   返回正式执行结果：`rolledBackBatchId`、`targetBatchId`、`restoredFieldCount`、`rollbackTime`、`approvalOrderId`、`approvalStatus`

### 7.6 采集器 / 子设备边界

二期继续固定并显式表达以下规则：

1. 采集器产品页只治理采集器自身状态或父设备字段。
2. 子设备产品页治理子设备监测值和子设备状态。
3. 采集器页不得发布子设备正式字段。
4. 复合设备模式通过父设备编码、逻辑通道编码和子设备编码完成归一映射。
5. compare 阶段负责归一化，apply 阶段负责越界拦截，evidence / mapping 保存 raw identifier 到规范字段的证据链。

## 8. 实施切分

本轮主体任务只做四项，按顺序推进，但都服务于同一个整体目标。

### 8.1 任务 1：工作区主流程收口

目标：

1. 重组 `ProductModelDesignerWorkspace` 信息架构。
2. 增加治理阶段总览。
3. 把样本输入、compare 决策、审批反馈、历史辅助区组织成固定顺序。

### 8.2 任务 2：compare / apply / 批次 / 回滚语义收口

目标：

1. 固定 compare = 治理候选快照。
2. 固定 apply = 审批提交回执。
3. 固定 release batch = 正式发布单位。
4. 固定 rollback = 逆向发布结果。

### 8.3 任务 3：采集器 / 子设备治理边界收口

目标：

1. 前端明确提示采集器边界。
2. 后端严格阻止采集器产品发布子设备正式字段。
3. 让激光测距、深部位移等复合设备场景的治理口径稳定下来。

### 8.4 任务 4：验收与文档口径收口

目标：

1. 把二期写成正式可验收的冻结增强项。
2. 同步业务、接口、交付边界和验收标准文档。
3. 保证后续实施和复验都以同一口径为准。

## 9. 验收口径

### 9.1 二期完成后应能直接回答的业务问题

验收时系统应能稳定回答以下六个问题：

1. 当前这次治理处于哪个阶段。
2. compare 里每个字段为什么会出现。
3. 这次准备发布哪些字段、哪些跳过、为什么。
4. 审批当前是什么状态，审批通过后是否生成正式批次。
5. 当前产品的正式字段是什么，哪些是本轮新生效的。
6. 如果回滚，会影响哪些字段和哪些风险治理对象。

### 9.2 最小测试集

前端至少覆盖：

1. 工作区主流程阶段展示和空态。
2. 采集器复合设备模式边界提示。
3. compare 结果到本次生效列表的联动。
4. 审批回执、审批刷新、原单重提展示。
5. 发布批次、回滚试算、正式字段台账展示。

后端至少覆盖：

1. compare 返回规范匹配和风险就绪字段。
2. apply 回执与正式执行结果区分。
3. 采集器产品禁止发布子设备字段。
4. 回滚 impact 与回滚结果语义稳定。
5. 批次与风险目录联动查询结果稳定。

构建与文档至少覆盖：

1. `mvn -pl spring-boot-iot-admin -am clean package -DskipTests`
2. 前端定向测试和 build
3. 文档拓扑检查
4. 必要时共享 `dev` 环境最小浏览器冒烟

## 11. 下阶段功能点

以下内容不属于本轮主体，但必须在规格中显式列出，避免后续失焦：

1. 契约与运行时对齐看板  
   正式字段与 runtime 映射命中率、未治理 raw identifier 观测、mapping rule 覆盖率。
2. 契约测试 / CI 门禁  
   产品样本回归、compare 结果基线校验、发布前契约差异门禁。
3. 编辑效率增强  
   批量改名、批量单位修订、模板化字段录入。
4. 治理运营视图  
   产品治理进度、待审批积压、批次发布频次、风险目录覆盖率。

原则固定为：

**先完成二期主体工作流，再进入下一步功能点。**

## 12. 最终结论

“产品物模型设计器二期”的完整定义是：

**在现有 `/products -> models` 工作区内，把产品契约治理流程收束成一条可操作、可审批、可追溯、可回滚、可验收的主流程，并固定采集器 / 子设备治理边界。**

这一定义将作为后续实施计划、编码、测试、文档同步和共享环境复验的统一基线。
