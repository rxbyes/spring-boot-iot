# 风险对象中心四色等级与档案补齐设计

> 日期：2026-04-02
> 主题：将风险对象中心升级为“四色风险等级 + 数据字典驱动 + 区域/负责人/审计字段补齐”的统一治理基线，并兼容历史三档风险等级数据迁移

## 1. 背景

当前风险对象中心 `/risk-point` 已经具备风险点主档、设备绑定和组织归属的基础能力，但仍存在一组直接影响后续治理和真实环境验收的问题：

- 风险等级仍使用 `critical / warning / info` 三档口径，页面文案对应“严重 / 警告 / 提醒”，与用户要求的行业通行“红 / 橙 / 黄 / 蓝”四色风险分级不一致。
- 风险等级来源散落在后端枚举、服务比较逻辑和前端页面文案中，当前是硬编码实现，没有通过数据字典统一治理。
- 风险点主档虽然数据库已经保留 `region_id / region_name / responsible_user / create_by / update_by`，但风险对象中心页面没有把区域、负责人、创建人编号、更新人编号完整纳入录入和回填链路。
- 风险对象中心新增页面仍展示只读“风险点编号”，而编号本身已改为服务端自动生成，这会造成页面信息噪音和错误心理预期。
- 风险对象中心编辑功能没有完整回填缺失字段，导致区域、负责人、审计字段在“新增可写、编辑不可见”或“数据库有、页面无”的状态之间脱节。
- 自动闭环、风险监测详情、对象洞察、事件/告警详情等风险等级消费链路，当前仍按三档口径解释 `riskLevel`，如果只改风险对象中心而不改消费链路，会造成前后台口径不一致。

结合代码和初始化脚本，可以确认以下事实：

- `risk_point` 已有 `org_id / org_name / region_id / region_name / responsible_user / create_by / update_by` 列，其中 `create_by / update_by` 也已落在数据库表结构中。
- `RiskPoint` 实体当前把 `createBy / updateBy` 标成了 `@TableField(exist = false)`，导致页面和接口无法正常透出真实审计字段。
- `sql/init-data.sql` 当前已经内置 `risk_level` 字典，但字典项仍是旧三档：`critical / warning / info`。
- 自动闭环枚举 `AutoClosureSeverity` 已有“红橙黄蓝”颜色语义，但最终回写到风险点等级时仍把橙/黄折叠成 `warning`、蓝折叠成 `info`。
- 前端 `RiskPointView.vue`、`RiskMonitoringDetailDrawer.vue`、`DeviceInsightView.vue` 等页面对风险等级的展示仍是三档硬编码，不具备动态字典能力。

本轮目标不是单纯把下拉文案改成红橙黄蓝，而是把“风险等级标准、风险点主档字段、数据迁移、风险等级消费链路”一起收口成一套真实可维护的基线。

## 2. 目标

- 把风险等级统一升级为四色内码：`red / orange / yellow / blue`。
- 风险等级的权威来源改为数据字典 `risk_level`，前后端都不再写死正式选项。
- 在保留现有 `risk_level` 字段名的前提下，完成从历史三档数据到四色内码的兼容迁移。
- 风险对象中心新增/编辑链路补齐以下字段：
  - `regionId / regionName`
  - `responsibleUser`
  - `createBy / updateBy`
- 风险对象中心新增页面取消展示风险点编号，编辑页面保留只读展示历史编号。
- 风险对象中心列表、抽屉和编辑回填链路同时支持“所属组织 + 所属区域 + 负责人”三类档案字段。
- `create_by / update_by` 按当前登录用户自动填充，只读透出，不允许在风险对象中心人工修改。
- 自动闭环、风险监测详情、对象洞察、告警/事件详情等风险等级消费链路统一切到四色标准，并兼容共享环境残留旧值。

## 3. 非目标

- 本轮不重构 `alarm_level` 的独立语义，不把告警等级体系一并升级成四色治理闭环。
- 本轮不实施新的组织范围或区域范围权限过滤规则，只补齐主档数据和展示字段。
- 本轮不批量回填历史风险点的区域、负责人等缺失字段，只保证新增、编辑和迁移后链路一致。
- 本轮不新增第二套风险等级字段，例如 `risk_level_color`、`risk_severity` 等平行列。
- 本轮不重做风险对象中心整体布局，继续复用现有共享列表/抽屉骨架。

## 4. 已确认决策

本轮设计基于以下已确认用户决策：

1. 采用方案 A：原位升级现有 `risk_level` 数据字典与业务字段，不新增平行等级体系。
2. 风险等级必须使用四色内码，并兼容历史数据迁移。
3. 风险等级必须从数据字典获取，而不是通过前后端硬编码实现。
4. `create_by / update_by` 按当前登录用户自动回填，只读透出，不允许在风险对象中心人工编辑。
5. 新增风险点页面不再展示风险点编号；编号仍由服务端自动生成。
6. 风险对象中心编辑功能需要补齐区域、负责人和缺失档案字段回填。

## 5. 方案对比

### 5.1 方案 A：原位升级 `risk_level` 并做兼容迁移

- 继续使用现有 `sys_dict.dict_code = 'risk_level'`。
- 把 `sys_dict_item` 从旧三档值升级为 `red / orange / yellow / blue`。
- 业务表 `risk_point.risk_level`、`iot_event_record.risk_level`、`emergency_plan.risk_level` 等仍使用原字段名，但存储四色内码。
- 旧值通过 SQL 迁移脚本和运行期兼容映射兜底。

优点：

- 权威口径唯一，不会形成新旧双轨字典。
- 数据库、接口、前端、文档都只有一套风险等级定义。
- 后续维护成本最低。

缺点：

- 会触达风险点、事件、预案、自动闭环和多个前端详情展示链路。
- 必须一次性把兼容映射做扎实。

### 5.2 方案 B：新增新的四色字典编码，旧 `risk_level` 保留

- 新增例如 `risk_level_color` 字典和业务映射。
- 新功能读新字典，旧链路继续沿用旧值。

优点：

- 表面上对现有链路冲击更小。

缺点：

- 会形成双口径。
- 每个消费链路都要判断“旧等级还是新等级”，维护复杂度高。
- 与“统一治理”和“后续管理标准化”目标相悖。

### 5.3 方案 C：页面显示四色，数据库继续存三档

- 页面风险等级选项来自字典，但提交时仍翻译回 `critical / warning / info`。

优点：

- 改动最少。

缺点：

- 实质上没有完成四色内码升级。
- 无法满足“采用四色内码等级并兼容迁移”的要求。

## 6. 选型

本轮采用方案 A。

原因：

- 用户已经明确要求使用四色内码并以数据字典为源。
- 风险对象中心只是入口，真正影响治理一致性的是事件、监测和自动闭环链路；方案 A 能一次性消除长期分叉。
- 现有 `risk_level` 字典和业务字段已经存在，原位升级比新增平行体系更合理。

## 7. 核心业务语义

### 7.1 四色风险等级标准

统一内码：

- `red`
- `orange`
- `yellow`
- `blue`

统一展示文案：

- `red -> 红色`
- `orange -> 橙色`
- `yellow -> 黄色`
- `blue -> 蓝色`

统一比较顺序：

- `red > orange > yellow > blue`

### 7.2 历史三档兼容口径

共享环境可能仍残留以下旧值：

- `critical`
- `warning`
- `info`

本轮统一兼容映射：

- `critical -> red`
- `info -> blue`
- `warning -> orange`

其中 `warning -> orange` 采用保守迁移默认值。原因是历史三档数据无法无损区分“橙 / 黄”；而从风险治理视角看，把旧 `warning` 默认迁到更高风险的 `orange`，比默认降到 `yellow` 更安全。后续真实监测重算或人工编辑可以把个别记录修正为 `yellow`。

### 7.3 风险点档案语义

风险点主档本轮保持以下字段语义：

- `org_id / org_name`：责任归属组织
- `region_id / region_name`：地理区域/GIS/风险处置定位
- `responsible_user`：责任人用户编号
- `responsible_phone`：责任联系电话
- `create_by / update_by`：审计字段，只读透出

组织和区域同时保留，不能互相替代。

## 8. 数据模型设计

### 8.1 `sys_dict` / `sys_dict_item`

继续使用：

- `sys_dict.dict_code = 'risk_level'`

字典项调整为：

- `red / 红色`
- `orange / 橙色`
- `yellow / 黄色`
- `blue / 蓝色`

排序建议：

1. `red`
2. `orange`
3. `yellow`
4. `blue`

要求：

- 只有 `status = 1` 的字典项才视为有效风险等级。
- 所有风险等级选项必须由字典项驱动生成。
- 不允许在前端组件或后端服务中把四色正式选项写成长期硬编码数组。

### 8.2 `risk_point`

不新增新列，沿用并修正已有列映射：

- `risk_level`
- `region_id`
- `region_name`
- `responsible_user`
- `create_by`
- `update_by`

实体修正要求：

- `createBy / updateBy` 去掉 `@TableField(exist = false)`，恢复真实映射。
- 保持 `riskPointCode` 自动生成且编辑不改。

### 8.3 关联业务表

本轮需要统一考虑的等级字段：

- `risk_point.risk_level`
- `iot_event_record.risk_level`
- `emergency_plan.risk_level`

说明：

- `iot_alarm_record.alarm_level` 仍保持原告警等级语义，不在本轮直接四色化。
- 风险监测详情、对象洞察等若直接读取 `riskPoint.riskLevel` 或 `event.riskLevel`，都要支持四色新值和旧值兼容。

## 9. 数据迁移设计

### 9.1 初始化数据升级

需要同步更新：

- `sql/init-data.sql`

调整内容：

- 把 `risk_level` 字典项从 `critical / warning / info` 改为四色。
- 若 `alarm_level` 仍保留旧三档，不在本轮一并改动。

### 9.2 共享环境 Schema/数据同步

需要同步更新：

- `scripts/run-real-env-schema-sync.py`

职责：

- 补齐真实环境中 `risk_level` 字典项为四色标准。
- 将旧 `risk_level` 数据迁移到新内码。

### 9.3 数据迁移规则

迁移语句统一遵循：

- `critical -> red`
- `warning -> orange`
- `info -> blue`

作用范围至少覆盖：

- `risk_point`
- `iot_event_record`
- `emergency_plan`

如果真实环境还存在其他直接存储风险等级且由风险对象链路消费的表，也应纳入同步迁移清单，并在实施文档里逐表列出。

### 9.4 运行期兼容

即使迁移脚本执行后，运行期仍需兼容旧值，原因包括：

- 共享环境历史脏数据未完全清理
- 外部旧链路或测试数据仍可能写入旧值
- 真实环境局部回归可能发生旧包未同步

因此本轮需要保留一层“风险等级标准化”逻辑，对读取到的旧值先标准化，再参与比较和展示。

## 10. 后端设计

### 10.1 数据字典读取能力

为避免前端“先查字典主表，再查字典项”的双请求和后端自行硬编码，本轮采用以下设计：

- 扩展现有 `GET /api/dict/code/{dictCode}` 能力，使其返回的 `Dict` 同时带出已启用、按排序排列的 `items`
- `risk_level` 的前端页面直接通过该接口获取动态等级选项

要求：

- `DictService.getByCode` 或新增等价能力，在读取字典主表后装载 `status = 1` 且 `deleted = 0` 的字典项
- 字典项顺序按 `sort_no asc, id asc`
- 若字典不存在或有效项为空，风险等级相关页面和服务要返回明确错误，而不是静默退回硬编码

### 10.2 风险等级标准化支持层

在 `spring-boot-iot-alarm` 增加统一风险等级支持层，职责包括：

- 标准化旧值和新值
- 输出权重顺序
- 校验输入值是否属于当前启用字典
- 提供统一展示文本/标签辅助方法给 controller/service/VO 层使用

该支持层不应依赖前端常量，必须以后端字典读取为真源。

### 10.3 风险点新增/编辑

`RiskPointController` 和 `RiskPointServiceImpl` 需要调整为：

- Controller 从当前 `Authentication` 中提取登录用户编号
- 新增时：
  - 校验 `riskPointName`、`orgId`、`regionId`、`riskLevel`
  - 校验组织存在且启用
  - 校验区域存在且启用
  - `riskLevel` 必须属于当前启用风险等级字典
  - 回填 `orgName / regionName`
  - 自动生成 `riskPointCode`
  - 写入 `createBy / updateBy`
- 更新时：
  - 先查原记录
  - 保留历史 `riskPointCode`
  - 重新校验并回填 `orgName / regionName`
  - 刷新 `updateBy`

`responsibleUser` 处理口径：

- 允许为空，以兼容当前“负责人为空时自动派工回退到用户 1”的既有基线
- 若传值不为空，则必须校验用户存在且启用

### 10.4 风险点删除

删除逻辑本轮不改变语义：

- 继续逻辑删除
- 继续返回明确业务错误

但若实现中发现“已绑定设备时删除会导致明显业务破坏”，允许在服务层补一条显式保护，并在实施文档中说明。

### 10.5 自动闭环

`AutoClosureSeverity` 与 `DeepDisplacementAutoClosureService` 需要收口为四色真实落库：

- `RED -> red`
- `ORANGE -> orange`
- `YELLOW -> yellow`
- `BLUE -> blue`

不再允许：

- `ORANGE / YELLOW -> warning`
- `BLUE -> info`

同时：

- 风险点等级刷新时按四色权重比较
- 事件和应急预案匹配若读取 `risk_level`，也要先过标准化层

### 10.6 审计字段写入方式

本轮不扩展全局 `MetaObjectHandler` 去写 `createBy / updateBy`，原因是那会扩大到全仓库实体风格统一。

采用窄实现：

- `RiskPointController` 获取当前用户 `userId`
- 传给 `RiskPointService` 新增/更新方法
- 服务层在新增/更新时显式写入 `createBy / updateBy`

这样既满足“自动回填当前登录用户”，又把范围控制在风险对象中心本轮需求内。

## 11. 前端设计

### 11.1 风险对象中心列表

列表字段调整为至少包含：

- 风险点编号
- 风险点名称
- 所属组织
- 所属区域
- 风险等级
- 负责人
- 负责人电话
- 状态
- 创建时间

显示规则：

- `responsibleUser` 优先映射为用户姓名，无法映射时显示编号
- `createBy / updateBy` 不强制放入主列表，避免列表过宽；作为抽屉只读档案字段展示
- 若 `orgName / regionName / responsibleUser` 为空，显示清晰占位文案

### 11.2 风险对象中心新增抽屉

新增态字段：

- 不展示“风险点编号”字段
- 风险点名称
- 所属组织
- 所属区域
- 风险等级
- 负责人
- 负责人电话
- 状态
- 描述

说明文案需要改为：

- 编号由系统保存后自动生成，不再在新增态展示
- 风险等级按数据字典加载的四色标准选择

### 11.3 风险对象中心编辑抽屉

编辑态字段：

- 风险点编号：只读显示
- 风险点名称
- 所属组织
- 所属区域
- 风险等级
- 负责人
- 负责人电话
- 状态
- 描述
- 创建人编号：只读
- 更新人编号：只读

要求：

- 编辑打开时完整回填上述字段
- 负责人、区域、风险等级都必须能回显当前值

### 11.4 选项数据来源

页面需要新增三类动态选项加载：

其中风险等级为正式治理口径，禁止再写死：

- `critical / warning / info`
- “严重 / 警告 / 提醒”

### 11.5 风险等级展示统一

风险对象中心、风险监测详情、对象洞察、告警/事件详情等页面统一改为：

- 标签颜色仍可继续复用 Element Plus 的 `danger / warning / success / info` 风格
- 但业务值和业务文案必须是四色口径

建议展示映射：

- `red -> 红色`
- `orange -> 橙色`
- `yellow -> 黄色`
- `blue -> 蓝色`

兼容旧值时前端显示：

- `critical -> 红色`
- `warning -> 橙色`
- `info -> 蓝色`

### 11.6 风险对象中心编辑功能补齐

本轮要修复的编辑问题包括：

- 编辑打开时区域没有正确回填
- 编辑打开时负责人没有正确回填
- 编辑页面没有透出创建人编号和更新人编号
- 风险等级仍显示旧三档文案

修复后要求：

- 编辑抽屉具备完整档案回填
- 提交后接口成功、列表刷新、抽屉关闭、状态重置完整闭环

## 12. 错误处理设计

### 12.1 后端

统一使用 `BizException` 返回明确错误，例如：

- 风险等级字典不存在
- 风险等级字典没有启用项
- 风险等级不在允许值范围内
- 所属组织不存在/已停用
- 所属区域不存在/已停用
- 负责人不存在/已停用
- 风险点不存在

### 12.2 前端

以下操作失败时必须给出显式用户提示：

- 加载风险等级字典失败
- 加载组织树失败

## 13. 测试设计

### 13.1 后端单测

至少新增/调整以下测试：

- 风险点新增时校验四色风险等级来自字典
- 风险点新增时回填 `regionName`
- 风险点新增时写入 `createBy/updateBy`
- 风险点更新时保留原编号并刷新 `updateBy`
- 负责人存在性校验
- 风险等级旧值标准化比较
- 自动闭环回写四色等级而不是旧三档

### 13.2 前端单测

至少新增/调整以下测试：

- 新增态不展示风险点编号
- 编辑态展示只读风险点编号
- 风险等级选项来自字典接口而不是硬编码
- 编辑态完整回填区域、负责人、创建人编号、更新人编号
- 列表展示所属区域和负责人

### 13.3 真实环境验收

最小真实环境验收链路：

1. 使用 `application-dev.yml` 启动真实环境
2. 确认 `risk_level` 字典已升级为四色
3. 新增风险点，完成组织、区域、风险等级、负责人录入
4. 确认新增页不展示编号，保存后列表出现自动生成编号
5. 编辑该风险点，确认区域、负责人、创建人编号、更新人编号可见且正确回填
6. 完成设备绑定，确认风险监测/对象洞察等消费链路的风险等级展示仍正确

若真实环境字典或历史数据未同步，应明确标记为环境阻塞，不能回退到已废弃验收链路。

## 15. 风险与缓解

### 15.1 旧 `warning` 无法无损拆分黄/橙

缓解：

- SQL 迁移默认映射到 `orange`
- 运行期兼容旧值
- 自动闭环实时刷新后可逐步把风险点等级矫正到四色真实值

### 15.2 数据字典接口失效导致页面无选项

缓解：

- 后端提供稳定的“按编码取字典及启用项”能力
- 页面失败时给出明确错误
- 不保留正式硬编码兜底，避免表面可用、实际脱离治理

### 15.3 审计字段自动写入扩大为全局改造

缓解：

- 本轮只在风险点 controller/service 内窄实现
- 不顺手推广到全仓库实体

### 15.4 风险对象中心列表过宽

缓解：

- `createBy / updateBy` 只在编辑抽屉只读展示
- 主列表保留组织、区域、负责人等高价值治理字段

## 16. 实施边界总结

本轮实施必须完成四件事：

1. 把 `risk_level` 升级为数据字典驱动的四色内码体系。
2. 完成历史三档风险等级向四色内码的迁移与运行期兼容。
3. 把风险对象中心新增/编辑链路补齐区域、负责人、审计字段，并移除新增态编号展示。
4. 把依赖风险等级的核心风险消费链路切到统一四色标准。

本轮不应扩大到：

- 告警等级全链路重构
- 新增平行风险等级字段
- 全局自动审计字段注入框架
- 组织/区域权限模型重构
