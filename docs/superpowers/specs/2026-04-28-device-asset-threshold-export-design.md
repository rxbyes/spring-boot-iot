# 2026-04-28 设备资产中心阈值查看与搜索结果导出设计

## 背景

当前设备资产中心已经完成分页筛选、详情查看、设备操作、批量治理和本地 CSV 导出，但仍有 4 个明显缺口：

1. 桌面列表把 `设备名称 + 设备编号` 混在同一列，台账扫描效率一般。
2. `接入协议` 继续占据列表列宽，但当前资产盘点场景下不是高频阅读信息。
3. 行内没有“查看设备阈值”入口，用户无法在设备资产中心直接确认设备当前真正生效的阈值来源。
4. `导出当前` 仍然只导出当前页 `tableData`，没有复用后端分页筛选真相，跨页后结果会失真。

这次改造的目标不是重做设备资产中心，而是在保留现有分页、详情、设备操作和列设置机制的前提下，把“列表可读性”“阈值可见性”“导出真相一致性”一起收口。

## 目标

1. 桌面列表把 `设备名称` 与 `设备编号` 拆成两列，提升扫描效率。
2. 列表不再展示 `接入协议`，移动端卡片也同步去掉该字段。
3. 行内新增 `查看设备阈值`，可直接展示该设备全部可用阈值，并明确最终生效值与来源。
4. 把 `导出当前` 升级为 `导出搜索结果`，按当前已生效筛选条件跨分页导出全部命中设备。
5. 导出与分页查询复用同一套后端筛选真相，不允许前端仅凭当前页状态拼全量结果。

## 非目标

1. 不重做设备资产中心整体交互框架。
2. 不新增异步导出任务中心或文件归档能力。
3. 不修改阈值策略治理主工作台 `/rule-definition` 的职责边界。
4. 不改写现有阈值优先级语义，只在设备资产中心做只读聚合展示。
5. 不把“查看设备阈值”并入设备详情抽屉，避免再次混淆职责。

## 现状问题

### 1. 列表信息密度与阅读路径不理想

当前桌面表格使用复合列 `设备`，主文本为 `deviceName`，副文本为 `deviceCode`。这种写法适合空间紧张场景，但设备资产中心本身就是台账工作台，用户经常需要横向核对名称、编号、产品、机构和状态。把名称与编号挤在同一列，会降低批量比对效率。

### 2. 接入协议占位但不是当前主场景高频信息

`protocolCode` 目前同时出现在桌面表格与移动端卡片中。对接入治理阶段它有价值，但对资产盘点、状态核查、批量转正、导出清单这类主场景来说，它不是一线用户的高频阅读字段，反而挤占了更重要的名称/编号/状态空间。

### 3. 设备阈值只能绕行其他治理页面确认

当前系统里阈值真相实际分散在：

1. 设备正式可绑定指标列表。
2. 风险点正式绑定。
3. 阈值策略规则表，且存在 `METRIC / PRODUCT_TYPE / PRODUCT / DEVICE / BINDING` 多层作用域。

设备资产中心没有一个直接回答“这台设备现在哪些指标有阈值、最终命中哪条、来自哪里”的入口。用户只能跳去风险点或阈值策略页反查，排障路径过长。

### 4. 导出结果与分页查询真相脱节

当前前端 `导出当前` 直接把当前页 `tableData` 走本地 `downloadRowsAsCsv(...)`。这意味着：

1. 只导出当前页，不是当前筛选条件下的全部结果。
2. 已登记、未登记、混合视图的分页真相由后端统一组装，但导出只截取当前页缓存。
3. 用户翻页后导出会丢数据，且无法从页面文案判断这一点。

## 方案比较

### 方案 A：前端拼装阈值与分页导出

做法：

1. 前端拆列表列。
2. 前端新增阈值抽屉，串多个现有接口拼装结果。
3. 前端循环请求 `/api/device/page` 全分页结果，再本地导 CSV。

优点：

1. 上手快。
2. 不需要新增很多后端接口。

缺点：

1. 阈值真相分散在前端，容易与 `/rule-definition` 口径漂移。
2. 前端要处理多层规则优先级、正式指标集合、风险绑定集合，复杂度偏高。
3. 导出仍然是“前端自己猜全量”，对大结果集、分页口径和后端后续演进都不稳。

### 方案 B：后端收口只读聚合与导出真相

做法：

1. 前端只负责列表展示和抽屉呈现。
2. 后端新增设备阈值只读接口，统一返回设备全部可用指标、各来源阈值和最终生效结论。
3. 后端新增设备导出查询接口，复用设备分页筛选真相返回全量命中结果。

优点：

1. 设备页、阈值策略页、导出口径基于同一套后端真相。
2. 前端改动集中在展示层，后续维护更稳。
3. 可以明确控制结果上限、空结果提示和异常文案。

缺点：

1. 需要补充新的 VO / Service / Controller 和测试。
2. 一次性改动会覆盖前后端两侧，工作量中等。

### 方案 C：直接跳转阈值策略页并保留前端本地导出

做法：

1. 列表拆列、去掉协议列。
2. 行内增加跳转 `/rule-definition` 的快捷入口。
3. 导出文案调整，但仍导当前页。

优点：

1. 实现最省。

缺点：

1. 没有真正解决“设备页直接看最终生效阈值”的诉求。
2. 导出误导仍然存在。
3. 用户需要跨页理解设备、绑定、规则三层上下文。

## 结论

采用方案 B。

这次需求本质上要求“列表查询真相”和“阈值命中真相”在设备资产中心直接可见，因此应由后端收口只读聚合，再由前端消费，不宜继续把核心语义留给页面临时拼装。

## 设计

### 1. 列表结构调整

#### 桌面表格

1. 现有复合列 `设备` 拆成两个独立列：
   - `设备名称`
   - `设备编号`
2. 删除桌面表格中的 `接入协议` 列。
3. 其余产品、机构、状态、固件、最近上报、创建时间、操作列保持现状。

#### 移动端卡片

1. 卡片头部仍保持 `设备名称 + 设备编号`。
2. 信息区删除 `接入协议` 字段，保证移动端与桌面端口径一致。

### 2. 行内“查看设备阈值”入口

1. 在设备资产中心行内菜单新增 `查看设备阈值`。
2. 入口仅对已登记设备显示；未登记上报线索不显示该动作。
3. 入口放在菜单项中，不新增为直出按钮，避免进一步拉宽操作列。

### 3. 设备阈值抽屉

#### 抽屉定位

设备阈值抽屉是一个独立只读工作区，不并入 `详情` 抽屉，也不复用 `设备操作` 抽屉。

#### 头部信息

顶部固定展示：

1. 设备名称
2. 设备编号
3. 产品名称
4. 已命中阈值指标数
5. 未配置阈值指标数

#### 内容分区

抽屉按来源与结论展示以下区域：

1. `当前生效`
   - 每个指标只展示当前最终生效的一条阈值结论。
2. `绑定个性`
   - 展示命中 `BINDING` 作用域的规则。
3. `设备个性`
   - 展示命中 `DEVICE` 作用域的规则。
4. `产品默认`
   - 展示命中 `PRODUCT` 作用域的规则。
5. `通用兜底`
   - 展示命中 `PRODUCT_TYPE` 或 `METRIC` 作用域的规则。

若某区块为空，则默认不渲染，避免产生大片空白。

#### 指标展示字段

每条指标至少展示：

1. 指标名称
2. `identifier`
3. 当前生效告警等级
4. 阈值表达式
5. 来源标签
6. 作用对象说明

对于绑定个性项，还应额外展示：

1. `riskPointDeviceId`
2. 风险点名称或绑定说明

#### 空状态

如果设备当前没有任何正式可用监测指标，则显示：

1. “当前设备暂无可展示阈值”
2. 原因说明：未找到正式可绑定监测指标或当前产品暂无风险闭环指标

### 4. 设备阈值后端只读接口

建议新增接口：

`GET /api/device/{deviceId}/thresholds`

返回结构应覆盖三类信息：

1. 设备摘要
2. 设备全部可用阈值项
3. 每个指标的最终生效结论与来源明细

建议 VO 草图：

```ts
interface DeviceThresholdOverviewVO {
  deviceId: string
  deviceCode: string
  deviceName: string
  productId?: string | null
  productName?: string | null
  matchedMetricCount: number
  missingMetricCount: number
  items: DeviceThresholdMetricItemVO[]
}

interface DeviceThresholdMetricItemVO {
  metricIdentifier: string
  metricName: string
  riskMetricId?: string | null
  effectiveRule?: DeviceThresholdRuleVO | null
  bindingRules: DeviceThresholdRuleVO[]
  deviceRules: DeviceThresholdRuleVO[]
  productRules: DeviceThresholdRuleVO[]
  fallbackRules: DeviceThresholdRuleVO[]
}

interface DeviceThresholdRuleVO {
  ruleId?: string | null
  ruleName?: string | null
  ruleScope: string
  ruleScopeText: string
  expression?: string | null
  alarmLevel?: string | null
  sourceType: 'BINDING' | 'DEVICE' | 'PRODUCT' | 'PRODUCT_TYPE' | 'METRIC'
  sourceLabel: string
  targetLabel?: string | null
  riskPointDeviceId?: string | null
  riskPointId?: string | null
  riskPointName?: string | null
}
```

### 5. 阈值聚合口径

#### 可用指标集合

设备“全部可用阈值”的候选集合由两部分并集组成：

1. 当前设备正式可绑定监测指标。
2. 当前设备已存在风险点正式绑定的指标。

按 `metricIdentifier + riskMetricId` 去重后形成最终指标集合。

这样可以同时满足：

1. 有产品默认阈值但暂未绑定风险点的指标仍然可见。
2. 已有绑定个性规则的指标不会被遗漏。

#### 生效优先级

最终生效规则严格沿用现有语义：

1. `BINDING`
2. `DEVICE`
3. `PRODUCT`
4. `PRODUCT_TYPE`
5. `METRIC`

设备资产中心只展示该结论，不改写任何后端规则判定逻辑。

#### 命中范围

规则筛选必须同时满足：

1. 指标匹配当前 `riskMetricId` 或 `metricIdentifier`
2. 作用域命中当前设备 / 当前产品 / 当前绑定对象
3. 规则状态为启用
4. 逻辑删除标记为有效

### 6. 导出搜索结果

#### 交互调整

1. 保留 `导出列设置`
2. 保留 `导出选中`
3. 将 `导出当前` 更名为 `导出搜索结果`

`导出搜索结果` 的语义固定为：

“按当前已生效筛选条件，导出全部命中设备，而不是当前页结果”

#### 后端导出接口

建议新增接口：

`GET /api/device/export`

查询参数与 `/api/device/page` 保持一致：

1. `deviceId`
2. `keyword`
3. `productKey`
4. `productName`
5. `deviceCode`
6. `deviceName`
7. `onlineStatus`
8. `activateStatus`
9. `deviceStatus`
10. `registrationStatus`

返回值可直接复用 `List<DevicePageVO>`，不额外创建第二套导出模型。设备资产中心已经基于 `DevicePageVO` 维护状态文案和 CSV 列映射，继续复用可降低口径分叉。

#### 查询真相复用

导出查询必须复用与 `/api/device/page` 相同的筛选语义，包括：

1. 已登记视图
2. 未登记视图
3. 混合视图
4. 关键字匹配产品
5. 显式产品筛选
6. 数据权限与组织范围

不允许前端通过分页循环自行拼接全量结果，也不允许导出查询只查已登记设备。

#### 结果保护

1. 当前筛选结果为空时，返回空列表并由前端提示“当前筛选没有可导出的设备”。
2. 如果需要保护上限，后端应显式返回业务提示，不做静默截断。
3. 列设置与格式化仍由前端复用现有 `CsvColumnSettingDialog`、`resolveCsvColumns(...)` 和状态文本格式化。

### 7. 前端数据流调整

#### 列表

`DeviceWorkbenchView.vue` 继续通过 `/api/device/page` 加载分页数据，不改变原有缓存、路由同步和筛选交互。

#### 阈值抽屉

新增设备阈值加载状态：

1. 打开抽屉时按 `deviceId` 请求 `/api/device/{deviceId}/thresholds`
2. 请求中显示局部 loading
3. 失败时提示“设备阈值加载失败”
4. 不影响原有详情抽屉与设备操作抽屉状态

#### 导出搜索结果

点击 `导出搜索结果` 时：

1. 读取当前已生效筛选条件
2. 请求 `/api/device/export`
3. 使用当前选择的列设置导出 CSV
4. 导出成功后提示导出条数

### 8. 错误处理

1. 设备阈值接口查不到设备时，返回明确业务错误，不展示空抽屉假象。
2. 设备阈值接口存在部分指标无规则时，仍返回该指标条目，但 `effectiveRule=null`，前端归入“未配置”统计。
3. 导出接口失败时，提示后端返回的业务错误，不降级为本地当前页导出。
4. 导出接口若因筛选条件命中 0 条，前端只提示无数据，不生成空文件。

## 影响范围

### 前端

1. `spring-boot-iot-ui/src/views/DeviceWorkbenchView.vue`
2. `spring-boot-iot-ui/src/api/device.ts`
3. 新增设备阈值抽屉组件或在现有设备工作台组件旁新增轻量只读抽屉
4. `spring-boot-iot-ui/src/__tests__/views/DeviceWorkbenchView.test.ts`
5. 新增设备阈值抽屉组件测试

### 后端

1. `spring-boot-iot-device` 新增设备导出查询接口与 service 能力
2. `spring-boot-iot-device` 新增设备阈值只读接口与 VO
3. 复用 `spring-boot-iot-alarm` 现有规则查询与风险绑定数据
4. 需要补充相应 service 单测 / controller 回归测试

## 验收标准

1. 设备资产中心桌面列表明确拆出 `设备名称` 与 `设备编号` 两列。
2. 列表和移动端卡片都不再显示 `接入协议`。
3. 已登记设备行内菜单可打开 `查看设备阈值` 抽屉。
4. 阈值抽屉能展示该设备全部可用指标，并明确 `当前生效 + 来源区块`。
5. 当设备没有设备个性或绑定个性规则时，能够正确回落展示产品默认阈值。
6. `导出搜索结果` 会按当前筛选条件导出全部命中设备，不受当前页限制。
7. 导出结果与分页列表筛选语义一致，尤其是 `已登记 / 未登记 / 混合` 三种视图保持一致。

## 测试设计

### 前端

1. 列表列结构测试
   - 存在 `设备名称`
   - 存在 `设备编号`
   - 不再存在 `接入协议`
2. 行菜单测试
   - 已登记设备出现 `查看设备阈值`
   - 未登记设备不出现该动作
3. 阈值抽屉测试
   - 成功渲染 `当前生效 / 绑定个性 / 设备个性 / 产品默认 / 通用兜底`
   - 无规则项显示未配置状态
4. 导出测试
   - `导出搜索结果` 调用新的后端全量导出接口
   - `导出选中` 保持原逻辑不变

### 后端

1. 设备阈值聚合测试
   - 命中 `BINDING` 时优先于 `DEVICE / PRODUCT`
   - 缺 `BINDING / DEVICE` 时回落 `PRODUCT`
   - 仅存在 `PRODUCT_TYPE / METRIC` 时能正确归入兜底
2. 设备导出测试
   - 与分页查询在相同筛选条件下返回相同命中集合
   - 已登记 / 未登记 / 混合场景一致
   - 显式产品筛选和关键字产品匹配一致

## 风险与缓解

1. 风险：设备阈值聚合会重复实现一套规则判定。
   - 缓解：尽量复用现有规则优先级和有效规则筛选逻辑，不在前端重写。
2. 风险：导出接口与分页接口后续演进时口径再次分叉。
   - 缓解：在 service 层抽出共享查询方法，导出与分页共用。
3. 风险：结果集过大时同步导出压力升高。
   - 缓解：先保留同步导出模式，但在后端增加明确上限和业务提示，为后续异步化留边界。

## 实施顺序

1. 后端抽出设备分页共用查询真相。
2. 新增设备导出接口并补测试。
3. 新增设备阈值只读聚合接口并补测试。
4. 前端调整设备列表列结构与菜单动作。
5. 前端接入设备阈值抽屉。
6. 前端把 `导出当前` 切换为 `导出搜索结果` 并接入新接口。
7. 运行定向前后端回归并完成手工验收。
