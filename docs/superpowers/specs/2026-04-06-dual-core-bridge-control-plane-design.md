# 双核心域 + 桥层 + 控制面 + 运行底座架构升级设计

**日期**: 2026-04-06

**目标**: 在不破坏 `spring-boot-iot` 现有模块化单体边界与真实环境基线的前提下，把当前“语义契约治理 + 风险运营闭环”的可用最小切片，升级为可持续扩展、可治理、可回滚、可复盘的长期架构。

## 1. 设计结论

本轮架构升级不再把平台表达为“双核心域”即可，而是统一收口为五层结构：

1. 核心域 A：语义契约治理域
2. 核心域 B：风险运营闭环域
3. 中间桥梁：风险指标语义层
4. 上方控制面：治理与运营控制面
5. 下方运行底座：接入链路、证据沉淀、可观测与审计

其中：

- `iot_product_model` 继续作为正式合同真相表。
- 风险闭环不再默认直接消费 `iot_product_model`，而是优先消费 `risk_metric_catalog`。
- 控制面负责“谁改了什么、为什么能发、出了问题怎么回滚、当前覆盖率如何、治理断点在哪”。
- 运行底座负责 trace、证据、漂移、告警、回放和审计，不与业务工作台混写。

## 2. 七条提升方向的落地边界

### 2.1 治理与运营控制面

新增统一控制面对象与流程，最小能力至少包括：

- 正式合同发布批次的版本化视图
- 合同发布审批留痕与发布理由
- 风险指标目录发布留痕
- 影响分析：某合同字段、风险指标、风险点绑定、阈值策略、联动、预案之间的引用关系
- 回滚入口：从发布批次维度回滚合同与风险指标启用状态
- 覆盖率指标：正式合同覆盖率、风险指标覆盖率、绑定覆盖率、阈值覆盖率、联动覆盖率、预案覆盖率
- 真实环境复盘入口：按 `traceId / deviceCode / productKey / releaseBatchId` 复盘

### 2.2 风险指标语义层

当前已有 `risk_metric_catalog` 最小切片，但需要正式升级为桥层真相：

- 所有允许进入风险闭环的正式字段，必须先进入风险指标目录
- 风险对象、阈值策略、联动编排、预案、GIS、对象洞察、运营分析，统一优先依赖 `risk_metric_id`
- 继续保留 `metric_identifier` 兼容读写，但只作为过渡回退口径
- 风险指标目录需要补齐语义字段：单位、量纲、方向性、趋势能力、GIS 能力、洞察能力、分析能力、启用状态、来源发布批次

### 2.3 经营与治理驾驶舱

管理者视图不再只展示风险运营结果，还要展示治理经营结果：

- 规范治理完成率
- 厂商/产品原始字段滞留率
- 风险点正式指标绑定覆盖率
- 阈值/联动/预案覆盖率
- 新厂商接入平均耗时与卡点分布
- 漂移告警与待处理数量

该驾驶舱应建立在真实 API 聚合之上，不再依赖静态预置数据。

### 2.4 从表单系统升级为任务系统

一线治理与配置页面统一补齐“下一步该做什么”的任务表达：

- 待治理产品
- 待发布合同
- 待发布风险指标
- 待绑定风险点
- 待补阈值策略
- 待补联动/预案
- 待复盘事件

对象详情页、产品工作台、风险对象中心与策略页都只展示与当前对象直接相关的待办，不回流大而全说明墙。

### 2.5 可运维性设计

运行底座必须覆盖以下新增能力：

- 字段漂移告警
- 厂商报文与正式合同差异告警
- 风险指标缺失告警
- 发布回滚后的影响面复盘
- 按 `traceId / deviceCode / productKey / releaseBatchId` 的治理与风险链路回放
- 配置变更审计

### 2.6 扩展模型

未来“少改代码、主要配配置”的前提，是现在先把扩展模型抽出来：

- 协议族定义
- 解密配置解析与密钥解析接口
- 字段映射规则
- 父子设备归一规则
- 规范字段匹配规则
- 风险指标生成规则

本轮不追求一次性抽象成完整 DSL，但要先把硬编码从主业务服务里拆成可替换规则组件或 SPI。

### 2.7 权限与密钥治理

控制面与桥层新增后，权限必须同步升级：

- 规范库维护权限
- 合同 compare/apply/发布/回滚 权限
- 风险指标发布/停用 权限
- 风险点绑定与策略配置权限
- 联动/预案治理权限
- 控制面查看与导出权限

密钥治理先完成“配置域隔离 + 审计留痕 + 使用者收口”，不在本轮直接引入外部 KMS。

## 3. 工作树拆分原则

这七条提升方向不能简单拆成七个独立窗口，因为共享写集过多。按最小冲突原则，统一拆成四条工作流：

### 工作流 A：语义桥层与控制面基础

承接原始方向：

- 第 1 条 治理与运营控制面
- 第 2 条 风险指标语义层

原因：

- 二者共享合同发布、风险指标目录、影响分析、回滚、审计、覆盖率等核心对象
- 若拆开并行，最容易在表结构、服务接口和文档口径上冲突

### 工作流 B：治理经营驾驶舱与任务化工作台

承接原始方向：

- 第 3 条 经营与治理驾驶舱
- 第 4 条 表单系统升级为任务系统

原因：

- 二者共享同一组聚合指标、缺口统计与待办视图
- 主要冲突集中在前端工作台、聚合接口与页面文案

### 工作流 C：可运维性与链路复盘

承接原始方向：

- 第 5 条 可运维性设计

原因：

- 该部分主要集中在 observability、trace、回放、告警与审计，不宜和工作台 UI 混改
- 需要消费工作流 A 的发布批次与风险指标对象，但可以保持独立交付

### 工作流 D：扩展模型与安全治理

承接原始方向：

- 第 6 条 扩展模型
- 第 7 条 权限和密钥治理

原因：

- 二者都属于“未来持续演进”的底座性工作
- 共享协议层、权限码、菜单种子、配置解析与审计口径

## 4. 推荐工作树与分支命名

统一以 `codex/dev` 为集成基线，从干净 HEAD 派生四个工作树：

- `.worktrees/semantic-bridge-control-plane` -> `codex/semantic-bridge-control-plane`
- `.worktrees/governance-task-workbench` -> `codex/governance-task-workbench`
- `.worktrees/operability-drift-replay` -> `codex/operability-drift-replay`
- `.worktrees/extensibility-security-governance` -> `codex/extensibility-security-governance`

## 5. 合并顺序

推荐统一回收顺序如下：

1. `codex/semantic-bridge-control-plane`
2. `codex/operability-drift-replay`
3. `codex/extensibility-security-governance`
4. `codex/governance-task-workbench`

原因：

- 工作流 A 先提供桥层对象与控制面基础真相
- 工作流 C 与 D 在 A 的对象稳定后可较低风险合入
- 工作流 B 最终对接真实聚合指标、待办与覆盖率，适合最后整体验收

## 7. 本轮设计的非目标

本轮先不做以下事项：

- 直接把平台改成微服务
- 引入外部工作流引擎或 BPM 平台
- 引入外部 KMS 或统一配置中心
- 一次性消灭所有 `metric_identifier` 兼容字段
- 以 H2 或单机 mock 数据替代真实环境验收

## 8. 验收口径

本轮架构升级完成后，至少应达到以下口径：

- 控制面可以回答“发布了什么、谁发布的、为什么发布、影响了什么、如何回滚”
- 风险闭环默认消费风险指标目录，而不是直接耦合正式合同字段
- 管理者能看到治理成效与覆盖率
- 一线人员能看到任务导向的待办与断点
- 运维人员能看到漂移、差异、缺失与回放证据
- 开发人员能在协议、归一、语义映射、风险生成处落可替换扩展点
- 权限与密钥口径不会让“免代码接入”演变成“无治理改线上”
