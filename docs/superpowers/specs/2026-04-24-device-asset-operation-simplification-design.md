# 2026-04-24 设备资产操作页轻量化设计

## 背景

当前设备资产中心里，“查看详情”和“设备操作”实际打开的是同一个详情抽屉。这个抽屉里同时承载：

1. 资产状态与接入概况
2. 资产概览
3. 身份与部署台账
4. 运行与认证台账
5. 关系与建档补充
6. 设备能力与命令
7. 扩展元数据快照

结果是“操作页”看起来几乎等于“详情页”，但真正可执行的能力反而藏得很深。对于 `6260370286` 这类声光报警器，用户会先经历一大段详情，再去找操作按钮；一旦设备离线、未激活或产品元数据未命中可执行能力，还会进一步感知为“页面复杂且无法操作”。

## 目标

1. 把“设备操作”从详情职责中拆出去，变成独立的轻量能力入口。
2. 让用户一眼区分“看详情”和“做操作”。
3. 对可执行设备，能力与命令直接可见；对不可执行设备，原因前置且明确。
4. 不新增新的重型页面框架，继续复用现有抽屉与能力组件。

## 非目标

1. 不重做设备资产中心列表页。
2. 不新增新的设备能力业务模型。
3. 不改变后端能力解析规则和命令下发主链路。
4. 不把详情页改成两个并列重页面。

## 现状问题

### 1. 入口职责混淆

当前 `设备操作` 行为最终还是打开详情抽屉，内部再渲染完整设备详情与能力区。用户点击“操作”，却先看到一整套资产台账，这会造成认知偏差。

### 2. 能力区被详情稀释

设备能力与命令虽然已实现，但位于详情页底部，用户需要先扫过大量信息才能到达真正的动作区。

### 3. 无法操作时缺少前置解释

以 `6260370286` 声光报警器为代表，设备操作是否可执行，取决于：

1. 产品元数据是否命中 `WARNING` 及具体子类
2. 设备是否已激活
3. 设备是否停用
4. 能力是否要求在线且设备是否在线

这些条件目前只在能力按钮禁用态里体现，缺少一个“为什么不能操作”的前置摘要。

## 方案比较

### 方案 A：独立“设备操作”抽屉，详情仍保留原抽屉

优点：

1. 职责清晰。
2. 详情与操作彻底解耦。
3. 最符合用户“我现在要操作，不是先读档案”的直觉。

缺点：

1. 需要新增一个轻量能力入口组件或抽屉容器。

### 方案 B：同一抽屉内默认切到能力模式，详情折叠

优点：

1. 改动较少。
2. 复用现有抽屉壳层。

缺点：

1. 仍然是一个“什么都装”的大抽屉。
2. 详情与操作边界不够清晰。

### 方案 C：仅保留详情抽屉，在顶部加能力锚点

优点：

1. 代码改动最小。

缺点：

1. 只是“更快滚到底部”，没有真正变轻。

## 结论

采用方案 A。

## 设计

### 1. 入口分离

设备资产中心列表保留两个明确动作：

1. `查看详情`：打开完整详情抽屉。
2. `设备操作`：打开独立能力抽屉。

其中“设备操作”只负责执行，不再承载资产台账。

### 2. 能力抽屉内容

能力抽屉只保留三块内容：

1. 设备摘要：设备编码、设备名称、产品名称、在线/激活/状态摘要。
2. 可执行能力：按产品元数据分组展示按钮。
3. 命令记录：最近下发记录与刷新动作。

如果设备当前不可执行，则在摘要下方直接展示统一原因，例如：

1. 产品已停用
2. 设备已停用
3. 设备未激活
4. 设备离线，当前能力需要在线执行

### 3. 声光报警器行为

`6260370286` 这类声光报警器的能力展示按产品治理元数据决定：

1. 命中 `WARNING + BROADCAST` 时，展示广播预警能力组。
2. 命中 `WARNING + LED` 时，展示情报板能力组。
3. 命中 `WARNING + FLASH` 时，展示爆闪灯能力组。
4. 若产品元数据缺失或无法解析，直接显示“当前产品暂未开放可执行能力”。

页面不再暗示“点了操作就一定能操作”，而是明确把不可执行原因放到能力抽屉顶部。

### 4. 详情抽屉保持原职责

详情抽屉继续展示：

1. 资产状态与接入概况
2. 资产概览
3. 身份与部署台账
4. 运行与认证台账
5. 关系与建档补充
6. 扩展元数据快照

不再在详情抽屉内渲染“设备能力与命令”区块。

## 验收标准

1. 点击 `设备操作` 时，只打开能力抽屉，不再先展示完整详情。
2. 点击 `查看详情` 时，只打开详情抽屉，不再混入能力执行区。
3. `6260370286` 声光报警器在命中正确元数据时，能看到对应能力分组。
4. 当设备离线、未激活或停用时，能力抽屉顶部能直接说明原因，按钮保持禁用。
5. 详情页不再重复渲染能力与命令模块。

## 测试设计

1. 列表操作测试
   - `设备操作` 入口应打开能力抽屉。
   - `查看详情` 入口应打开详情抽屉。
2. 能力面板测试
   - 可执行设备展示能力按钮与命令表。
   - 不可执行设备展示统一 disabled reason。
3. 详情面板测试
   - 详情页只保留资产/台账/元数据内容，不再包含能力区。

## 风险与缓解

1. 风险：用户仍把“操作”和“详情”混用。
   - 缓解：按钮命名保持明显区分，抽屉标题也分别强调“详情”与“能力”。
2. 风险：能力抽屉过于空。
   - 缓解：无能力时明确展示空态和原因，而不是留白。
3. 风险：`6260370286` 这类设备因治理元数据不完整仍无法操作。
   - 缓解：保持 disabled reason 前置，不用页面结构掩盖数据缺口。

## 实施顺序

1. 拆分列表入口行为。
2. 新增/调整轻量能力抽屉。
3. 从详情抽屉移除能力区。
4. 补齐或调整相关单测。
5. 运行前端定向回归并同步文档。
