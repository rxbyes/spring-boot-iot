# IoT Access Aliyun Tab Workspace Design

**Date:** 2026-03-28
**Status:** Draft for review
**Scope:** 在现有 `接入智维` 页面收口和诊断联动能力基础上，进一步重构为“阿里云式轻量页面骨架 + 同页 Tab 工作区 + 公共组件母版”

## 1. 背景

`接入智维` 在 2026-03-27 已完成第一轮 UI 收口，在 2026-03-28 已补上第一阶段诊断联动，当前已经具备这些基础：

1. `接入智维总览`、`产品定义中心`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 已统一回到系统现有品牌 token，不再各自维护私有配色。
2. 总览与子页的职责边界已经明确：
   - 总览负责“先去哪”
   - 子页负责“在这里怎么做”
3. `链路验证 -> 链路追踪 -> 异常观测 -> 数据校验 / 失败归档 -> 产品 / 设备治理` 已形成第一阶段跨页排查流。

但当前仍有新的结构问题：

1. **页面母版仍偏散。** 各页虽然共用标准件，但页面头部、筛选区、状态条、结果区的组合方式还不够统一。
2. **同一页面内多内容仍分散在按钮、抽屉或模式切换里。** 用户切换上下文时缺少稳定的同页工作区心智。
3. **页面里仍残留说明性内容。** 一些页面还会出现“功能介绍 / 用途说明 / 重复导流文案”，与简约控制台风格冲突。
4. **公共复用层级不够。** 现在更像“大家都在用同一批标准件”，还不是“大家都挂在同一套接入智维母版上”。
5. **局部交互缺陷暴露了结构松散问题。** 例如产品定义中心操作列的动作间距、下拉触发器视觉噪音，以及同页 query 更新触发的滚动跳顶，说明接入页还缺一层统一约束。

用户当前明确要求：

1. 页面布局参考阿里云控制台的简约、分层清晰风格。
2. 配色保持当前系统既有配色，不更换主色体系。
3. 一个页面内若有多个内容域，优先改为选项卡模式。
4. 把 `接入智维` 下的所有模块按同一母版重构，沉淀公共组件，方便后续复用。
5. 去掉页面中的无效内容描述，尽可能保持简约。

## 2. 目标

本轮设计目标如下：

1. 为 `接入智维` 建立一套统一的阿里云式轻量页面母版，覆盖总览、治理类页面和诊断类页面。
2. 把“同一页面中多个内容域”统一收口为同页 Tab 工作区，而不是继续散落在独立区域、描述卡或临时模式切换中。
3. 抽出一套 `接入智维` 专用公共组件层，统一承接页面头部、Tab、筛选区、结果区和轻状态条。
4. 保持现有共享组件体系和品牌 token，不新建第二套主题系统。
5. 通过结构约束直接修复一批现有页面缺陷：
   - 操作列动作间距不一致
   - “更多”触发器出现多余视觉噪音
   - 同页筛选/分页引发内容区滚动跳顶

## 3. 非目标

本轮明确不做：

1. 不新增 `接入智维` 一级路由，不变更菜单归属。
2. 不改变后端接口、权限模型或业务闭环。
3. 不把所有页面都抽成强配置驱动平台，避免首轮就过度框架化。
4. 不改变现有品牌橙、云蓝、蓝灰的视觉基线。
5. 不把普通结果页重新做成营销式大 Hero、大概况卡或多层装饰页面。

## 4. 设计结论

### 4.1 总体方案

采用“**阿里云式轻量母版 + 同页 Tab 工作区 + 共享结果区**”方案，作为 `接入智维` 第二轮结构升级。

相比 2026-03-27 的“页面视觉收口”，本轮进一步把重心放在：

1. **页面骨架统一**
2. **同页工作区统一**
3. **模块内容组织统一**

### 4.2 抽象深度结论

采用三种候选方式中的 **B. 工作区母版方案**：

1. 抽出统一 `接入智维页面壳`
2. 抽出统一 `Tab 工作区`
3. 抽出统一 `筛选 / 工具条 / 结果区` 外壳
4. 保留各页面自己的业务结果内容，不进一步做重配置化渲染

原因：

1. 比纯视觉包装更稳定，能真实提升后续复用能力。
2. 比全量配置驱动更易落地，不会让现有页面难以维护。
3. 能兼容当前已经存在的 `StandardWorkbenchPanel`、`StandardListFilterHeader`、`StandardTableToolbar`、`StandardInlineState`、`StandardPagination` 等标准件。

### 4.3 视觉方向结论

页面布局参考阿里云控制台，但只借鉴其结构秩序，不复制其品牌色。

保留现有系统配色，统一视觉规则为：

1. 轻量面包屑
2. 单标题主视图
3. 线性 Tab 工作区
4. 简洁筛选区
5. 主结果表格 / 卡片优先
6. 弱化说明，强化可执行动作

## 5. 统一设计原则

### 5.1 标题必须轻

页面顶部只保留：

1. 面包屑
2. 页面标题
3. 右侧 1~2 个主动作

普通页面默认不再展示长副标题、介绍型说明段落或“这里可以做什么”的导语。

### 5.2 同页多内容优先 Tab

只要一个页面存在两个及以上并列内容域，优先采用 Tab 工作区承接，而不是：

1. 再堆一层说明卡
2. 再做一排自定义 pill
3. 再拆第二条轻路由
4. 把次要内容塞进不稳定的局部按钮状态

### 5.3 主战区优先

结果台账、验证结果、追踪结果、校验结果必须优先占据首屏主体，不允许被大段说明、能力目录或重复统计墙压到第二屏。

### 5.4 描述最小化

页面说明只允许保留三类：

1. 面包屑层级信息
2. 轻状态条中的来源 / 判断 / 下一步建议
3. 空态或错误态中的必要说明

其余介绍性内容一律删除或下沉。

### 5.5 结构修复优先于样式打补丁

像“操作列间距不等”“更多后有多余视觉噪音”“分页和查询触发滚动跳顶”这类问题，不通过页面级零散补丁修，而是通过统一组件和统一路由行为收口。

## 6. 公共组件设计

本轮新增或重构出的公共层统一服务 `接入智维`，不扩展成全站第二套壳层系统。

### 6.1 `IotAccessPageShell`

职责：

1. 承接面包屑
2. 承接页面标题
3. 承接右侧主动作
4. 可选承接一条极轻量状态 / 来源条

规则：

1. 页面不再直接堆叠 `eyebrow + title + description + notices` 的自由组合。
2. 默认不渲染长描述文本。
3. 允许治理页和诊断页共享同一壳，但右侧动作数量保持克制。

### 6.2 `IotAccessTabWorkspace`

职责：

1. 承接同页多内容 Tab
2. 管理当前激活内容区
3. 支持页面级默认 Tab
4. 支持与路由 query 轻量同步当前 Tab

规则：

1. Tab 名称优先是“对象 / 动作”，不使用长句。
2. 默认 Tab 必须是当前页面最高频任务。
3. 同页切 Tab 不新增二级路由。
4. 若当前页面已有模式切换语义，优先统一迁移到这里。

### 6.3 `IotAccessFilterBar`

职责：

1. 承接常用筛选项
2. 承接“展开更多筛选”
3. 统一搜索 / 重置 / 主操作布局

规则：

1. 常用筛选与动作区必须在同一个控制台条带里完成。
2. 展开更多筛选后仍保持同一页面壳，不另起说明分区。
3. 默认不渲染额外字段说明文字，依赖 placeholder 和 label 即可。

### 6.4 `IotAccessResultSection`

职责：

1. 承接工具条
2. 承接 meta 信息
3. 承接 `StandardInlineState`
4. 承接表格 / 卡片 / 分页外壳

规则：

1. 同类页面统一这一层的纵向节奏和留白。
2. 结果区上方不再额外堆第二层概况卡。
3. 即便列表为空，只要有来源上下文或当前查询态，也优先留在同一结果区表达。

### 6.5 与现有标准件的关系

本轮不是替换 `StandardWorkbenchPanel` 家族，而是形成一层更薄的 `接入智维编排层`：

1. `IotAccessPageShell` 负责页面头部和壳层秩序
2. `IotAccessTabWorkspace` 负责同页内容切换
3. 现有 `StandardWorkbenchPanel`、`StandardListFilterHeader`、`StandardTableToolbar`、`StandardInlineState`、`StandardPagination` 继续承接业务区块

## 7. 模块信息架构

### 7.1 `接入智维总览`

Tab：

1. `推荐处理`
2. `最近使用`
3. `全部能力`

规则：

1. 总览继续只做入口，不承接完整业务明细。
2. 通过 Tab 组织入口信息，而不是通过多块说明墙堆叠。

### 7.2 `产品定义中心`

Tab：

1. `产品台账`
2. `物模型治理`
3. `关联设备`

规则：

1. `产品台账` 为默认页签。
2. 物模型不再只是“从列表动作里进入的额外抽屉语义”，而是页面内明确可见的工作区。
3. `关联设备` 承接当前产品与库存设备的关系查看，减少跨层跳转感。

### 7.3 `设备资产中心`

Tab：

1. `资产台账`
2. `未登记上报`
3. `拓扑关系`

规则：

1. `资产台账` 为默认页签。
2. `未登记上报` 统一承接当前只读失败补档视角。
3. `拓扑关系` 承接父子资产结构和相关筛查，不再把拓扑语义散落在列表细节里。

### 7.4 `链路验证中心`

Tab：

1. `模拟验证`
2. `结果复盘`
3. `最近记录`

规则：

1. `模拟验证` 保持默认页签。
2. `结果复盘` 承接当前发送结果、时间线和诊断结论。
3. `最近记录` 承接最近一次上下文恢复，不再通过额外说明文案提示。

### 7.5 `链路追踪台`

Tab：

1. `消息追踪`
2. `失败归档`
3. `时间线复盘`

规则：

1. 当前已有的双模式能力正式收口为统一 Tab。
2. 不再用页面私有模式切换器承接主内容域。

### 7.6 `异常观测台`

Tab：

1. `异常台账`
2. `聚合视图`
3. `回跳治理`

规则：

1. `异常台账` 为默认页签。
2. `聚合视图` 承接高频统计与聚合口径，不再额外堆卡。
3. `回跳治理` 只承接去产品 / 设备治理和来源上下文，不做说明页。

### 7.7 `数据校验台`

Tab：

1. `设备校验`
2. `原始响应`
3. `历史快照`

规则：

1. `设备校验` 为默认页签。
2. `原始响应` 与 `历史快照` 不再与主校验结果混排在同一长页面里。

## 8. 页面行为约束

### 8.1 操作列统一约束

`产品定义中心`、`设备资产中心` 以及后续接入页列表操作列统一遵守：

1. `详情 / 编辑 / 物模型 / 更多` 这类动作统一放入共享行操作骨架。
2. 列宽必须为包含下拉触发器预留稳定余量，避免文字被挤压或错位。
3. `更多` 触发器不得额外暴露默认 caret、伪标点或视觉噪点。
4. 间距只能来自共享 `gap` 档位和共享 action 容器，不允许页面私自补丁。

### 8.2 同页筛选和分页不重置内容滚动

对于 `产品定义中心`、`设备资产中心`、`链路追踪台`、`异常观测台` 等同一路由内依赖 query 切换列表状态的页面：

1. 同页 query 变化默认不触发内容区滚动到顶部。
2. 只有真正的跨页面导航，才执行滚动归顶。
3. 避免用户在翻页、搜索、切筛选时出现“下拉刷新 / 页面抽动”感。

### 8.3 说明文案剔除规则

以下内容默认从页面中移除：

1. 模块能力介绍墙
2. “这里可以做什么”的长说明
3. 重复总览已表达过的信息
4. 第二层概况卡中的解释性文字

保留内容只有：

1. 页面标题
2. Tab 名称
3. 轻状态条
4. 空态 / 错误态必要说明

## 9. 响应式与兼容

1. 桌面端保持标准控制台表格主视图。
2. 中窄屏下 Tab 仍保持同页切换，但筛选区自动折行。
3. 移动端继续保留现有卡片视图与抽屉能力，不强行展示过多 Tab 内容在同屏。
4. 对于现有共享环境已经稳定的业务路径，优先保持路由与接口不变，通过前端结构升级完成重构。

## 10. 实施批次建议

推荐分两批推进：

### 第一批

1. `产品定义中心`
2. `设备资产中心`
3. `链路验证中心`
4. `链路追踪台`

原因：

1. 这四个页面最能验证 Tab 工作区是否成立。
2. 其中既包含治理页，也包含诊断页，足以验证公共组件边界。

### 第二批

1. `异常观测台`
2. `数据校验台`
3. `接入智维总览`

原因：

1. 第一批稳定后再收口剩余页面，可降低整体返工风险。
2. 总览适合最后接入，以便根据子页最终结构回调入口组织方式。

## 11. 验收标准

完成后应满足：

1. `接入智维` 下所有页面都能识别出同一套母版家族。
2. 同页多内容模块统一通过 Tab 工作区切换。
3. 页面中无无效说明墙、重复描述或第二层概况卡。
4. 产品定义中心操作列的动作间距、触发器样式和列宽表现稳定。
5. 同页查询、翻页、切筛选不再触发内容区跳顶。
6. 新增页面或后续增强可以直接复用公共壳，而不是再各写一套页面头和切换条。

## 12. 风险与取舍

1. 首轮若把公共层抽得过深，会增加实现复杂度，因此本轮保持“薄编排层 + 既有标准件承载业务”。
2. `物模型治理` 和 `关联设备` 当前已有抽屉与跳转语义，迁移到 Tab 后需要谨慎处理现有入口，避免用户短期失去熟悉路径。
3. `异常观测台` 与 `/audit-log` 共用视图文件，必须保证本轮只影响接入智维语义，不误改审计中心模式。

## 13. 下一步

在本 spec 获批后，下一份 implementation plan 应按以下顺序展开：

1. 先落 `IotAccessPageShell / IotAccessTabWorkspace / IotAccessFilterBar / IotAccessResultSection`
2. 再实施第一批页面迁移
3. 再实施第二批页面迁移
4. 最后补文档、测试与质量门禁
