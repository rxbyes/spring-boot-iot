# IoT Access Aliyun Page Consolidation Design

**Date:** 2026-03-28
**Status:** Approved design baseline
**Scope:** `接入智维` 全部页面统一收口为“面包屑/标题 + 真正有业务意义的页签 + 筛选区 + 单主内容区”

## 1. 背景

`接入智维` 在 2026-03-27 至 2026-03-28 的第一轮页面收口中，已经补齐了一批统一壳层与诊断联动能力，但新的页面结构也暴露出明显问题：

1. 页面顶部重复展示状态摘要，例如“当前结果 30 条，启用 10 个，停用 0 个”，而这些信息在工具栏和分页中已经存在。
2. `物模型治理`、`关联设备`、`未登记上报`、`拓扑关系`、`聚合视图`、`回跳治理` 等区域里，存在明显的说明性卡片、占位入口或重复入口。
3. 同一能力同时出现在顶部页签、正文卡片、工具栏或表格表头中，造成重复建设。
4. 页面从“主业务区”退化成“说明区 + 辅助区 + 主表格区”的双主区结构，不符合用户想要的阿里云控制台式简洁秩序。

用户已经明确确认新的收口方向：

1. 参考第三张图的结构，不再沿用“工作区卡片 + 说明区”的设计。
2. 所有 `接入智维` 页面统一收成“面包屑/标题 + 真正有业务意义的页签 + 筛选区 + 单主内容区”。
3. 只保留能切换真实业务内容的页签；不能切换真实内容的页签、卡片和说明区全部删除。
4. `产品定义中心`、`设备资产中心`、`异常观测台` 这类本质上只有一个主业务面的页面，不保留无意义页签。
5. 只保留真实可用的业务入口，不为未实现能力保留占位按钮。

## 2. 目标

本轮设计目标如下：

1. 为 `接入智维` 全部页面建立统一的页面骨架。
2. 删除重复状态条、说明卡、假工作区、假入口、假页签。
3. 让页面主体重新回到“筛选后处理业务结果”这条主线。
4. 保留现有真实业务能力，但把入口收回到行操作、工具栏、详情抽屉或强相关跳转区。
5. 后续实现时优先复用现有共享组件，不再新增页面私有布局模式。

## 3. 非目标

本轮明确不做：

1. 不新增后端接口，不补做当前并不存在的业务能力。
2. 不增加新的一级路由或菜单分组。
3. 不改变既有品牌配色、设计令牌和标准表单/表格体系。
4. 不为了追求形式统一而强行让所有页面都保留页签。
5. 不把占位入口包装成“即将支持”或“辅助说明”卡片继续留在页面中。

## 4. 设计结论

### 4.1 收口方案

本轮采用用户确认的 **A 方案：表格/结果优先**。

核心结论只有一句话：

`接入智维` 页面统一收成“面包屑/标题 + 真正有业务意义的页签 + 筛选区 + 单主内容区”。

### 4.2 为什么不是旧方案

不再采用上一轮的“同页工作区卡片 + 结果区”思路，原因如下：

1. 顶部状态摘要与工具栏、分页信息重复。
2. 工作区卡片中大量内容属于说明、建议和占位入口，不是真正的主业务内容。
3. 用户进入页面后仍然需要再判断“到底主战区在哪”，破坏操作效率。
4. 多入口并存容易导致后续继续重复建设。

### 4.3 真页签的定义

只有满足以下条件的页签才允许保留：

1. 切换后会更换一整块真实业务内容，而不是只换提示文案。
2. 每个页签下都有独立、可操作、可验证的主内容区。
3. 页签内承载的内容不是现有列表行操作或详情抽屉的重复镜像。

不满足以上条件的页签必须删除。

## 5. 统一结构规则

### 5.1 页面头部

页面顶部只保留：

1. 面包屑
2. 页面标题
3. 少量强相关跳转按钮

页面顶部不再保留：

1. 长副标题
2. 状态摘要条
3. “当前结果 xx 条、启用 xx 个”这类重复信息
4. 介绍型说明文案

### 5.2 页签规则

页签只在页面存在两个及以上真实业务视图时保留。

以下情况必须取消页签：

1. 页面本质上只有一个主业务面
2. 页签切换的只是说明卡、推荐卡、摘要卡
3. 页签承载的是已有表格行操作、详情抽屉或右上角跳转的重复入口

### 5.3 筛选区规则

筛选区始终位于页签下方、主内容区上方：

1. 列表页使用搜索与筛选表单
2. 工具页使用查询条件或前置参数表单
3. 筛选动作只保留查询、重置和必要主操作

不再在筛选区周围增加说明卡或第二层导流区。

### 5.4 单主内容区规则

每个页面同一时刻只允许出现一个主内容区：

1. 列表页是主表格或主列表
2. 工具页是主工作面板
3. 详情、表单、关联查看通过抽屉承接

禁止再次出现“上层工作区 + 下层表格区”的双主区布局。

### 5.5 入口收口规则

二级能力入口统一回收到以下位置：

1. 与当前记录绑定的动作回到表格行操作
2. 与当前结果集绑定的动作回到工具栏
3. 与当前页面强相关的跨页跳转保留在右上角
4. 与当前记录深入查看相关的内容回到详情抽屉

不再在正文里单独造“治理区”“辅助区”“建议区”。

### 5.6 统计信息规则

同一组统计信息只能保留一个主要落点：

1. 优先放在工具栏 meta
2. 或者保留在分页附近
3. 或者放在详情区内部

同一统计信息不得同时出现在页面头、状态条、工具栏和分页中。

## 6. 页面逐页设计

### 6.1 `接入智维总览`

结构：

1. 面包屑 + 标题
2. 业务分组页签
3. 统一入口筛选区
4. 单列表主内容区

保留页签：

1. `资产底座`
2. `诊断排障`

收口规则：

1. 主内容区只展示页面名称、职责说明和进入按钮。
2. 通过页签区分“产品/设备”与“验证/追踪/观测/校验”两大业务域。
3. 删除 `推荐处理`、`最近使用`、`全部能力` 这类说明型页签和卡片。

### 6.2 `产品定义中心`

结构：

1. 面包屑 + 标题
2. 筛选区
3. 产品列表主内容区

页签结论：

1. 不保留页签

收口规则：

1. 删除顶部 `产品台账`、`物模型治理`、`关联设备` 页签。
2. 删除正文中的治理说明卡、关联说明卡和状态摘要。
3. `物模型治理` 回收到表格行操作和详情抽屉。
4. `关联设备` 回收到表格行操作或关联设备抽屉。
5. 产品列表继续作为页面唯一主内容区。

### 6.3 `设备资产中心`

结构：

1. 面包屑 + 标题
2. 筛选区
3. 设备列表主内容区

页签结论：

1. 不保留页签

收口规则：

1. 删除顶部 `资产台账`、`未登记上报`、`拓扑关系` 页签。
2. 删除正文中的未登记/拓扑说明卡和重复摘要。
3. `未登记上报` 通过现有 `登记状态` 筛选直接承接。
4. `拓扑关系` 回收到表格字段、行操作和详情抽屉。
5. 设备列表继续作为页面唯一主内容区。

### 6.4 `链路验证中心`

结构：

1. 面包屑 + 标题 + 强相关跳转
2. 真实业务页签
3. 当前页签的查询/配置区
4. 当前页签的单主工作面板

保留页签：

1. `模拟验证`
2. `结果复盘`
3. `最近记录`

收口规则：

1. 这三个页签都属于真实业务阶段，继续保留。
2. 每个页签内部只有一个主工作面板，不再额外叠加说明卡。
3. 右上角只保留强相关跳转：`链路追踪台`、`异常观测台`、`数据校验台`。

### 6.5 `链路追踪台`

结构：

1. 面包屑 + 标题 + 强相关跳转
2. 真实业务页签
3. 筛选区
4. 当前页签的单主内容区

保留页签：

1. `链路追踪`
2. `失败归档`

收口规则：

1. 顶部只保留真正的双业务模式页签。
2. 删除 `消息追踪`、`失败归档`、`时间线复盘` 这套工作区页签。
3. `时间线复盘` 下沉到 Trace 详情抽屉中。
4. `最近会话`、`运维看板` 等支持性区块不再占正文主区，可转为详情区或后续独立入口。

### 6.6 `异常观测台`

结构：

1. 面包屑 + 标题 + 强相关跳转
2. 筛选区
3. 异常列表主内容区

页签结论：

1. 不保留页签

收口规则：

1. 删除 `异常台账`、`聚合视图`、`回跳治理` 页签。
2. 页面只保留异常列表一个主内容区。
3. 聚合统计只保留在工具栏 meta 或分页邻近区域一次。
4. 回跳入口只保留在右上角和表格行操作里。
5. `/audit-log` 的业务审计模式不在本次页面结构收口范围内。

### 6.7 `数据校验台`

结构：

1. 面包屑 + 标题 + 强相关跳转
2. 真实业务页签
3. 设备编码查询区
4. 当前页签的单主内容区

保留页签：

1. `设备校验`
2. `原始响应`

删除页签：

1. `历史快照`

收口规则：

1. `设备校验` 承接文件快照与固件聚合结果。
2. `原始响应` 承接文件快照和固件聚合原始返回。
3. `历史快照` 与现有结果重复，且不构成独立业务视图，因此删除。

## 7. 共享组件与实现边界

### 7.1 保留的共享组件

继续保留并复用：

1. `IotAccessPageShell`
2. `StandardWorkbenchPanel`
3. `StandardListFilterHeader`
4. `StandardTableToolbar`
5. `StandardPagination`
6. `StandardDetailDrawer`
7. `StandardFormDrawer`
8. 现有共享列表样式与设计令牌

### 7.2 组件使用规则

`IotAccessPageShell` 只承载：

1. 面包屑
2. 页面标题
3. 右上角强相关操作

`IotAccessPageShell` 不再承载：

1. 状态摘要条
2. 长说明
3. 顶部重复统计

`IotAccessTabWorkspace` 只在页面存在两个及以上真实业务视图时继续使用。

`IotAccessResultSection` 这类说明型或辅助型正文区块，不再用于 `接入智维` 页面主结构。

### 7.3 禁止回潮的设计

后续实现中不得重新加入：

1. 顶部状态摘要条
2. 推荐卡、说明卡、辅助治理卡、能力墙
3. 同一能力的多套重复入口
4. 上层工作区加下层表格区的双主区布局
5. 为未实现能力保留的占位按钮

## 8. 交互约束

1. 强相关跨页跳转只允许出现在页面右上角。
2. 与当前记录绑定的能力必须放在表格行操作或详情抽屉。
3. 与当前筛选结果绑定的能力放在工具栏。
4. 已有真实流程的入口必须保留可达，但需要收回到统一位置。
5. 不存在真实流程的按钮直接删除，不做视觉保留。

## 9. 实现范围与验证口径

### 9.1 实现范围

本轮是页面结构收口，不是业务扩展，因此：

1. 不新增业务接口
2. 不新增占位能力
3. 不补做当前后端未提供的流程
4. 只整理现有真实入口、真实动作和真实内容区

### 9.2 验证重点

实现完成后重点验证：

1. `接入智维` 页面结构是否统一为同一家族骨架
2. 页面顶部是否已经去掉重复状态和说明信息
3. 页签是否只保留真实业务视图
4. 单主内容区是否成立
5. 既有真实业务入口是否仍然可达
6. 已删除的占位区和重复区是否没有以其他形式回流

## 10. 风险与取舍

1. `产品定义中心` 与 `设备资产中心` 之前已经加入了工作区页签和辅助区，实现阶段需要谨慎拆回，避免误伤现有抽屉和详情流程。
2. `链路验证中心` 与 `数据校验台` 仍然保留页签，但必须保证每个页签切换的是完整业务内容，而不是只换局部说明。
3. `异常观测台` 与 `/audit-log` 共用视图文件，实现时必须只影响接入智维语义，不误改审计中心模式。

## 12. 下一步

本设计稿确认后，后续 implementation plan 需要按以下顺序展开：

1. 先清理 `接入智维` 页面中的状态条、说明卡和假工作区
2. 再按本设计逐页收口真实页签和单主内容区
3. 再补齐测试、文档和前端治理记录
