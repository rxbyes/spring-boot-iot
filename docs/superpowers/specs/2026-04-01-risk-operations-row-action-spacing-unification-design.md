# 风险运营模块列表操作列统一设计

> 日期：2026-04-01
> 范围：`spring-boot-iot-ui`
> 主题：将风险运营模块的数据列表操作列完全收口到全局共享间距与宽度规则

## 1. 背景

前一轮“列表操作列间距统一”已经把桌面端表格动作间距和列宽基线收口到共享组件：

- `StandardWorkbenchRowActions`
- `resolveWorkbenchActionColumnWidth`

但风险运营模块下的 `实时监测台`、`GIS态势图`、`告警运营台`、`事件协同台` 虽然也已接入共享组件，页面内仍保留了历史参数表达差异，例如显式传入桌面 `gap="compact"`、对不同动作组合保留页面自己的宽度理解等。这会带来两个问题：

- 用户从页面对比上仍会感知到“同样是共享组件，但有的操作列更宽、有的更窄”。
- 后续开发者容易误判“既然页面还能传 gap，那本页是不是允许再单独调一套”。

此前已经确认过一次根因：问题不完全是“有没有使用共享组件”，还包括共享宽度分档曾经对双动作场景给了过宽下限，导致 `详情 / 观测` 这类短文案组合相比 `进入工作台 / 删除` 这类长文案组合更容易出现右侧留白偏大的错觉。因此本次风险运营收口必须同时锁定“共享组件入口”和“共享宽度档位解释权”。

这次收口的目标不是再做一套“风险运营子规范”，而是明确风险运营模块不再拥有任何操作列间距的私有设计权，必须彻底服从全局共享规则。

## 2. 目标

- 将风险运营模块四个页面的列表操作列统一到和 `产品定义中心` 同一套桌面端视觉基线。
- 明确页面不得再通过私有参数表达“本页更紧凑”或“本页更宽松”的桌面操作列语义。
- 保持移动端继续使用共享组件的自适应能力，不引入风险运营模块自己的移动端例外规则。
- 将这次收口补充到测试和文档，避免后续回流成“同组件多风格”。

## 3. 非目标

- 不新增风险运营专用的行操作组件。
- 不修改 `StandardActionLink`、`StandardActionMenu` 的颜色、字号和交互语义。
- 不为单个页面保留新的 scoped CSS 间距补丁。
- 不把移动端强行做成桌面端同像素宽度；移动端仍以触达和自适应为优先。

## 4. 方案比较

### 4.1 方案 A：继续使用全局共享组件，清理页面私有参数与心智

做法：

- 风险运营页面继续使用 `StandardWorkbenchRowActions`。
- 桌面端 `table` 场景统一服从共享 gap 基线与共享列宽分档。
- 页面不再保留“我这里只有 1 个动作所以要更窄”“我这里 3 个功能所以要更宽”的私有表达。

优点：

- 标准只有一份，符合“全系统同一套风格”的治理目标。
- 后续新增风险运营列表时无需再判断“是否沿用 risk 版规则”。
- 与当前已完成的产品定义中心、链路追踪台等共享规则完全一致。

缺点：

- 个别页面的列宽会跟随共享规则微调，不能再维持历史视觉差异。

### 4.2 方案 B：为风险运营模块增加一层专用封装

做法：

- 在全局共享组件外再包一层 `RiskWorkbenchRowActions`，由风险运营四页统一引用。

优点：

- 风险运营模块内收口快，看上去改动集中。

缺点：

- 会形成“全局标准 + 模块子标准”两级体系。
- 以后其他模块也可能复制同样做法，导致再次出现多套风格。

### 4.3 方案 C：逐页做 CSS 或列宽微调

做法：

- 针对每个页面当前按钮数量和文案长度，逐页调 `width`、padding 或 scoped 样式。

优点：

- 短期见效快。

缺点：

- 本质仍是页面补丁，未来动作数量和文案变化后还会漂移。
- 与“规范定义后不能随便改”的目标相冲突。

## 5. 决策

采用 `方案 A`。

风险运营模块不新增任何专用操作列样式抽象，不保留页面级桌面间距例外，只消费现有共享组件与共享宽度工具。

## 6. 页面级设计

### 6.1 实时监测台

- 当前动作只有 `详情`。
- 继续使用 `StandardWorkbenchRowActions` 和 `resolveWorkbenchActionColumnWidth`。
- 页面不再传入会暗示“桌面紧凑版”的私有 `gap` 语义。
- 最终视觉应落在共享单动作列宽档位，而不是页面单独压窄或拉宽。

### 6.2 GIS 态势图

- 表格区动作只有 `详情`，资源卡区仍使用共享 `card` 变体。
- 表格操作列和 `实时监测台` 一样，统一落在共享单动作规则。
- 卡片区不做桌面表格宽度对齐要求，继续保持共享卡片动作布局自适应。

### 6.3 告警运营台

- 桌面端继续采用“最多 2 个直出动作 + `更多`”的共享拆分策略。
- 无论具体状态下呈现为 `详情 + 关闭`、`详情 + 确认 + 更多`，都只允许由共享组件拆分，不允许页面再派生第二套操作列宽度逻辑。
- 页面列宽继续调用 `resolveWorkbenchActionColumnWidth`，并完全接受共享可见动作数分档。

### 6.4 事件协同台

- 与告警运营台一致，桌面端继续走共享拆分规则。
- `详情 / 派发 / 关闭` 等组合产生的直出与收拢方式，不允许页面私有定义。
- 页面显式传入的桌面紧凑参数需要清理，避免阅读代码时误以为本页可以单独定制间距。

## 7. 统一规则定义

### 7.1 桌面端表格

- 统一入口：`StandardWorkbenchRowActions` 的 `table` 变体。
- 统一间距：桌面端相邻功能按钮间距只允许使用共享 fixed gap 基线，当前共享档位为 `wide`，对应动作间距 `12px`。
- 统一分布：页面不得显式覆写桌面 `distribution`。
- 统一列宽：通过 `resolveWorkbenchActionColumnWidth` 按共享可见动作数分档，当前固定为：
  - `1` 个可见动作：`96px`
  - `2` 个可见动作：`112px`
  - `2` 个直出动作 + `更多`：`160px`

### 7.2 移动端与卡片场景

- 移动端仍由共享组件负责折行、收拢和点击面积控制。
- `card` 变体继续保持触控优先布局，不强制使用桌面等宽列宽。
- 锁死的是“共享规则”，不是要求所有终端呈现同像素宽度。

## 8. 测试与验证

### 8.1 组件与工具测试

- 在 `StandardWorkbenchRowActions` 测试中补充桌面 `table` 变体忽略页面私有 gap 语义的断言。
- 在 `adaptiveActionColumn` 测试中锁定共享单动作、双动作、`2 + 更多` 的列宽分档。
- 补充短文案组合与长文案组合的回归断言，确保右侧留白遵从同一套共享规则。

### 8.2 页面回归测试

- `实时监测台`、`GIS态势图`：验证单动作操作列宽度与共享规则一致。
- `告警运营台`、`事件协同台`：验证 2 个直出动作与 `更多` 的组合仍由共享组件拆分，状态变化不再导致两套视觉宽度。
- 优先复用已有视图测试文件；没有现成测试时，只补最小必要回归，不引入页面私有测试框架。

### 8.3 守卫与脚本

- 保持 `list:guard`、`component:guard`、`style:guard` 通过。
- 如现有门禁尚未覆盖“页面不得显式传桌面 gap / distribution”这一点，则在实现阶段补足脚本守卫。

## 10. 风险与控制

- 风险：风险运营四页的部分操作列视觉会与历史截图略有不同。
  - 控制：本次目标就是和产品定义中心彻底对齐，历史局部差异不再保留。

- 风险：页面开发者继续沿用 `gap="compact"` 一类写法，误导后续维护者。
  - 控制：在实现阶段清理多余参数，并用测试和守卫防止回流。

- 风险：移动端因桌面规则调整而受影响。
  - 控制：桌面规则只收口 `table` 变体，`card` 与移动端布局继续走共享自适应逻辑。

## 11. 实施入口

本设计确认后，下一步仅编写对应 implementation plan，不直接跳过计划进入编码。实现阶段必须以共享组件和共享工具为唯一切入点，风险运营页面只做参数清理和回归验证。
