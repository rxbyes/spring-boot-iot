# 产品治理与对象洞察统一标识符设计

**日期：** 2026-04-14  
**范围：** `spring-boot-iot-device`、`spring-boot-iot-alarm`、`spring-boot-iot-ui`、数据库清理脚本、治理与对象洞察相关文档  
**主题：** 把产品经营工作台、对象洞察台、设备属性快照、测点下拉、风险目录与策略链路统一到同一套正式字段标识符口径

---

## 1. 背景

当前平台已经不是单一的“产品管理页 + 设备详情页”结构，而是同一份产品正式字段同时驱动以下链路：

1. `/products` 产品经营工作台中的契约字段 compare/apply 与当前已生效字段
2. `iot_product.metadata_json.objectInsight.customMetrics[]` 中的对象洞察重要指标配置
3. `iot_device_property` 中的设备最新属性快照
4. 对象洞察台趋势查询与设备属性快照展示
5. 风险指标目录、风险点绑定、阈值策略、后续测点属性选择

一旦这些链路使用不同的 `identifier`，平台就会持续产生以下问题：

1. 同一物理含义出现两套字段，如 `signal_4g` 与 `S1_ZT_1.signal_4g`
2. 产品经营工作台、对象洞察台、风险绑定页面看到的可选字段不一致
3. 历史快照、当前已生效字段、风险目录与策略配置无法稳定关联
4. 同类采集器产品不断积累重复字段和脏数据

---

## 2. 当前问题

### 2.1 现状已经分裂成两套真相

当前代码里同时存在两种方向：

1. 产品治理写侧仍在把部分采集器父设备状态字段收口为短标识  
   例如 `signal_4g`、`battery_dump_energy`
2. 对象洞察、运行态落库与设备快照清理已经在向“全路径精确匹配”收口  
   例如 `S1_ZT_1.signal_4g`、`S1_ZT_1.battery_dump_energy`

这两者并存时，系统不会稳定，而是会持续反复出现：

1. 一处修成全路径，另一处又按短标识写回
2. 页面临时看起来正常，过一段时间又重新出现“系统繁忙 / 找不到字段 / 快照重复”
3. 测点下拉把正式字段和运行态快照混在一起，生成第二轮脏数据

### 2.2 根因不是单点，而是边界定义不统一

当前主要矛盾不是单个 bug，而是“父采集器状态字段的正式契约到底是什么”没有统一：

1. 采集器父产品 compare/apply 仍允许把父状态字段固化成短标识
2. 对象洞察配置保存已经要求按正式字段原始标识保存
3. 设备属性快照去重脚本已把“保留全路径、删除短标识”视为目标行为
4. 风险目录、风险绑定、规则配置直接依赖正式字段标识，但未统一上游来源

---

## 3. 设计目标

### 3.1 总目标

整个平台只保留一套正式字段标识符真相，所有治理、读侧、快照、风险闭环与后续测点属性都围绕同一标识符运转。

### 3.2 本轮目标

1. 明确 `identifier` 的唯一真相源
2. 统一采集器父产品状态字段的正式契约口径
3. 统一产品经营台、对象洞察、属性快照、测点下拉、风险目录和策略链路的读取口径
4. 为同类采集器产品提供历史数据清理与迁移方案

### 3.3 非目标

1. 不重写全部规范字段体系
2. 不改变裂缝、GNSS、深部位移、雨量计等子产品既有规范字段语义
3. 不在本轮引入新的平行合同草稿表
4. 不把显示中文名与机器标识符混用

---

## 4. 方案对比

### 方案 A：全路径作为唯一真相源

示例：

1. `S1_ZT_1.signal_4g`
2. `S1_ZT_1.battery_dump_energy`
3. `S1_ZT_1.temp`

优点：

1. 与真实上报结构一致
2. 能稳定区分多通道、多子设备、多父状态字段
3. 与当前设备属性去重方向一致
4. 能直接消除对象洞察、快照、测点下拉与风险绑定中的重复字段

缺点：

1. 需要迁移已存在的短标识正式字段和关联数据
2. 需要同步更新同类采集器产品的历史配置

### 方案 B：短标识作为唯一真相源

示例：

1. `signal_4g`
2. `battery_dump_energy`
3. `temp`

优点：

1. 表面改动范围较小

缺点：

1. 与真实上报结构不一致
2. 多通道 / 多父节点场景更容易冲突
3. 需要把当前对象洞察、设备快照、去重脚本、风险链路再次倒回短标识
4. 与已确认的数据清理方向相反

### 方案 C：双轨并存

即治理层保留短标识，运行态保留全路径。

缺点：

1. 会持续制造重复字段
2. 不能从根上消除脏数据
3. 后续测点属性、风险目录、策略配置仍会反复分裂

### 推荐结论

采用 **方案 A：全路径作为唯一真相源**。

---

## 5. 统一口径定义

### 5.1 统一原则

1. `identifier` 只表达机器契约，不表达展示语义
2. 中文名称统一由 `modelName`、对象洞察 `displayName`、风险目录 `riskMetricName` 承载
3. 运行态、治理态、风险态必须使用同一套正式字段 `identifier`

### 5.2 对采集器父产品的明确规定

对于采集器父产品自身运行状态字段，正式字段统一改为全路径标识，例如：

1. `S1_ZT_1.ext_power_volt`
2. `S1_ZT_1.solar_volt`
3. `S1_ZT_1.battery_dump_energy`
4. `S1_ZT_1.battery_volt`
5. `S1_ZT_1.supply_power`
6. `S1_ZT_1.consume_power`
7. `S1_ZT_1.temp`
8. `S1_ZT_1.humidity`
9. `S1_ZT_1.temp_out`
10. `S1_ZT_1.humidity_out`
11. `S1_ZT_1.lon`
12. `S1_ZT_1.lat`
13. `S1_ZT_1.signal_4g`
14. `S1_ZT_1.signal_nb`
15. `S1_ZT_1.signal_db`
16. `S1_ZT_1.sw_version`

这些字段属于父采集器产品自身，不下沉到子产品。

### 5.3 对子产品的明确规定

子产品的规范字段继续保持既有口径，不随本轮切换：

1. 裂缝 / 激光测距子产品继续使用 `value / sensor_state`
2. 深部位移子产品继续使用 `dispsX / dispsY / sensor_state`
3. GNSS 子产品继续使用既有规范字段

也就是说：

1. 父采集器产品的正式字段使用全路径父状态字段
2. 子产品的正式字段继续使用子产品规范标识
3. 两者不能再互相污染

---

## 6. 目标架构

### 6.1 真相源顺序

统一后的字段真相源顺序如下：

1. `iot_product_model.identifier`  
   正式合同字段唯一真相
2. `iot_product.metadata_json.objectInsight.customMetrics[].identifier`  
   必须直接引用正式字段
3. `iot_device_property.identifier`  
   latest 快照必须落到正式字段标识
4. `risk_metric_catalog.contract_identifier`
5. `risk_point_device.metric_identifier`
6. `rule_definition.metric_identifier`

### 6.2 允许的兼容范围

只允许以下兼容：

1. 空白裁剪
2. 大小写兼容
3. 显式 vendor mapping 规则映射

不再允许以下隐式兼容：

1. `S1_ZT_1.signal_4g -> signal_4g`
2. `S1_ZT_1.battery_dump_energy -> battery_dump_energy`
3. `S1_ZT_1.sensor_state.L1_LF_1 -> sensor_state`
4. 任意 dotted identifier 自动截成最后一段

---

## 7. 模块级设计

### 7.1 产品经营工作台 `/products`

#### compare/apply

1. 采集器父产品在 `状态数据 + 复合设备` 场景下，父状态叶子字段不再生成短标识候选
2. compare 结果中展示的正式字段标识直接使用全路径
3. apply 写入 `iot_product_model` 时，父状态字段只允许写全路径
4. 对旧短标识正式字段，迁移完成前仍可通过发布快照做只读兼容，但不得继续新增

#### 当前已生效字段

1. 当前已生效字段直接展示迁移后的全路径正式字段
2. 中文名称、对象洞察分组、趋势开关继续绑定在同一正式字段上

### 7.2 对象洞察台 `/insight`

1. `customMetrics[].identifier` 必须与正式字段完全一致
2. 设备属性快照按正式字段原始标识精确匹配，只保留大小写兼容
3. 趋势查询使用同一批正式字段标识
4. 前端内置命名注册表只能补显示名，不能再改写标识

### 7.3 latest 属性快照 `iot_device_property`

1. 运行时写入前，统一先根据已发布快照 / mapping 解析到正式字段标识
2. 若已解析到正式字段，则 latest 只保留正式字段标识
3. 已存在全路径与短标识同时出现的场景视为历史脏数据，统一清理短标识

### 7.4 测点下拉与后续测点属性

1. 设备测点下拉不再允许正式字段与 latest 属性双份并存
2. 若某标识已存在正式字段，latest 只作为值补充，不再额外产出第二条测点项
3. 测点属性体系沿用同一正式字段口径，避免后续新增第三套命名

### 7.5 风险目录、风险点绑定与策略

1. 风险目录 `contractIdentifier` 统一使用正式字段标识
2. 风险点绑定 `metricIdentifier` 统一使用正式字段标识
3. 阈值规则 `metricIdentifier` 统一使用正式字段标识
4. 旧短标识记录需要做一次迁移更新，否则风险链路仍会残留分裂口径

## 8. 数据迁移设计

### 8.1 迁移范围

本轮迁移范围不是单一产品，而是 **同类采集器产品**，至少包括与 `nf-collect-rtu-v1` 相同模式的父采集器产品。

### 8.2 迁移步骤

1. 扫描同类采集器产品中属于父采集器状态字段的短标识正式字段
2. 生成短标识到全路径的迁移映射
3. 更新 `iot_product_model.identifier`
4. 同步更新 `iot_product.metadata_json.objectInsight.customMetrics[].identifier`
5. 同步更新风险目录、风险点绑定、规则配置中的同类 `identifier`
6. 执行 `iot_device_property` 去重清理，删除已被全路径覆盖的短标识行

### 8.3 迁移原则

1. 保留全路径
2. 删除短标识重复项
3. 先迁正式合同，再迁风险链路，再清 latest 脏数据
4. 所有迁移都按产品范围与映射表精确执行，不做模糊全库替换

---

## 9. 风险与回滚

### 9.1 风险

1. 已发布批次的旧 resolver snapshot 可能仍含短标识别名
2. 风险目录或规则中若残留短标识，会导致部分旧配置无法命中最新快照
3. 前端若仍有本地短标识兜底逻辑，会把问题隐藏掉，导致迁移后难以及时暴露

### 9.2 回滚策略

1. 合同字段本身仍通过既有发布批次快照与回滚能力回退
2. 数据迁移脚本需按步骤留存执行清单和备份 SQL
3. 若中途发现某类产品映射不完整，应停止该类产品迁移，不继续扩散到下一批产品

---

## 10. 验收标准

### 10.1 产品经营工作台

1. 同类采集器产品的父状态字段正式字段全部为全路径
2. `当前已生效字段` 中不再出现与全路径等价的短标识父状态字段

### 10.2 对象洞察台

1. 设备属性快照不再同时出现 `signal_4g` 与 `S1_ZT_1.signal_4g`
2. 趋势查询按正式字段标识稳定命中
3. 内置显示名规则只改显示，不改 identifier

### 10.3 测点下拉与风险链路

1. 设备测点下拉不再出现同义双份字段
2. 风险目录、风险点绑定、规则配置都能用统一后的正式字段标识正常命中

### 10.4 数据清理

1. 同设备下若存在全路径与等价短标识 latest 属性，只保留全路径
2. 同类产品不再继续新增这类短标识脏数据

---

## 11. 实施顺序建议

1. 先收口 compare/apply，禁止继续产出新的短标识父状态字段
2. 再统一对象洞察配置保存与读侧校验
3. 再统一测点下拉与风险链路读取
4. 最后执行同类产品历史迁移与 latest 属性去重

## 13. 最终结论

本轮统一方案的核心不是“修一个对象洞察 bug”，而是把平台的正式字段口径从源头统一：

1. 产品经营工作台定义什么
2. 对象洞察台就看什么
3. latest 属性快照就存什么
4. 测点下拉就提供什么
5. 风险目录、绑定与策略就使用什么

统一后，`identifier` 将恢复为真正的跨模块契约，而不是每个模块各自解释的一段字符串。
