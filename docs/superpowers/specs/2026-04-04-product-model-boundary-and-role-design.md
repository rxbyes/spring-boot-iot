# 产品物模型在系统中的定位、边界与作用设计说明

> 日期：2026-04-04
> 适用范围：`spring-boot-iot` 当前主链路、产品定义中心、物模型治理、运行解释与下游业务消费
> 目标：统一团队对“物模型是什么、边界在哪里、为什么存在、代码里谁负责”的理解

## 1. 背景

当前仓库已经形成较明确的 IoT 五层治理顺序：`产品 -> 设备 -> 物模型 -> 属性/遥测 -> 消息日志`。

在实际协作中，团队仍然容易把以下几个对象混在一起：

1. 产品档案
2. 设备实例
3. 正式物模型
4. 运行期 latest 属性 / telemetry
5. 原始消息日志和样本证据

一旦混淆，后续就容易出现这些偏差：

1. 把某次设备上报直接当成正式字段标准。
2. 把物模型当成单纯的前端配置页，而不是正式契约层。
3. 让协议层、消息层或下游业务模块反向篡改产品正式模型。
4. 让 latest、telemetry、message log 与正式定义相互替代。

本说明的目标不是新增功能，而是把当前系统中已经存在的业务语义、架构边界与代码边界收口成统一口径。

## 2. 核心结论

`物模型是产品级正式契约层，用来把规范、报文和人工治理沉淀成平台认可的标准字段，并为运行解释和下游业务消费提供统一依据。`

这句话包含 4 个固定判断：

1. 它是`产品级`，不是设备级。
2. 它是`正式契约层`，不是运行值本身。
3. 它的输入是`证据与治理结果`，不是任意一次上报。
4. 它的输出是`平台标准字段定义`，供运行解释和业务消费复用。

## 3. 业务语义定位

### 3.1 物模型是什么

在 `spring-boot-iot` 中，物模型不是“设备当前报了什么”，而是“平台认可这一类产品应该按什么字段、什么名称、什么类型来理解数据”的正式定义。

它至少回答 3 个问题：

1. 这个产品有哪些正式字段、事件、服务。
2. 每个正式字段的 `identifier`、名称、数据类型、描述是什么。
3. 平台后续应当如何以统一口径消费这些字段。

因此，物模型本质上是设备世界与平台世界之间的统一词典和契约层：

1. 设备世界提供厂家报文、样本 JSON、运行证据。
2. 平台世界需要稳定的标准字段、标准名称和标准类型。
3. 物模型负责把两者连接起来。

### 3.2 物模型不是什么

物模型不是下面这些对象的别名：

1. 不是产品主档。
2. 不是某台设备实例。
3. 不是 latest 属性值。
4. 不是 telemetry 历史点位。
5. 不是原始 message log。
6. 不是某一页前端表单配置。

这几个对象的关系必须固定为：

1. `产品` 定义一类设备的接入身份与经营主体。
2. `设备` 定义具体资产实例。
3. `物模型` 定义平台认可的正式字段标准。
4. `属性/遥测` 定义设备运行值。
5. `消息日志` 定义原始证据与处理留痕。

### 3.3 物模型为什么必须存在

如果没有物模型，系统很快会退化成“谁报什么、平台就临时认什么”的状态，带来以下问题：

1. 同类产品字段名不统一，无法形成稳定业务语义。
2. 运行值与正式定义混淆，造成规则、风险、报表消费不稳定。
3. 厂家字段、现场样本和平台字段之间没有统一转换层。
4. 某一台设备的偶发上报会污染整类产品的正式标准。

物模型存在的根本原因，是把“证据”沉淀为“正式契约”，把“设备语言”翻译成“平台语言”。

## 4. 系统架构边界

### 4.1 在架构中的位置

物模型位于 IoT 主链路的中间位置，是一层正式解释层，而不是采集层或最终消费层。

它的典型位置可以表达为：

`规范证据 / 报文证据 / 人工治理 -> 物模型治理 -> 正式物模型 -> 运行解释 / 业务消费`

### 4.2 上游边界

物模型的上游只能提供`证据`和`候选`，不能直接替代正式模型。

当前上游主要有 3 类：

1. `规范证据`
   - 来自规范预设、产品治理预置标准、字段适用性判断。
2. `报文证据`
   - 来自真实上报、样本 JSON、消息日志、运行期属性快照。
3. `人工治理输入`
   - 来自人工补录的 `property / event / service` 候选。

这些上游共同回答的是“哪些字段可能应该进入正式模型”，但不能绕过 compare / apply 直接写成 `iot_product_model`。

### 4.3 中间边界

物模型治理的中间层职责非常单一：把上游证据转成产品正式契约。

因此它只负责以下事情：

1. 组织候选证据。
2. 比较候选与正式基线。
3. 决定 create / update / skip。
4. 把确认后的正式定义写入 `iot_product_model`。

它不负责以下事情：

1. 直接解析协议报文。
2. 直接保存 message log。
3. 直接代替 latest 或 telemetry 存储运行值。
4. 直接执行告警、事件、报表等业务逻辑。

### 4.4 下游边界

物模型的下游至少有两类：

1. `运行解释层`
   - 在 `PAYLOAD_APPLY`、latest 属性更新、telemetry 映射等环节消费正式字段定义。
2. `业务消费层`
   - 在风险点、规则、告警、事件、报表等场景消费标准化后的字段语义。

因此，下游可以依赖物模型，但不应反向定义物模型。

## 5. 代码落地边界

### 5.1 正式写入权归属

当前仓库中，正式物模型的写入权应固定收口到产品治理链路：

1. `ProductModelController`
2. `ProductModelService`
3. `ProductModelServiceImpl`
4. `ProductModelMapper`
5. `iot_product_model`

这条链路承担的职责包括：

1. 正式物模型列表查询。
2. 正式模型新增、编辑、删除。
3. 运行期候选提炼。
4. 手动样本提炼。
5. compare / apply 治理。
6. 候选确认并落正式模型。

统一规则是：`正式物模型只能由产品治理链路写入。`

### 5.2 运行期消费边界

当前代码里已经存在多个对物模型的“只读消费”入口，这些入口是合理的，但职责只能停留在解释和映射层：

1. `DevicePropertyMetadataServiceImpl`
   - 从 `iot_product_model` 读取 `property` 模型。
   - 为运行期属性补充正式名称、数据类型、兼容映射信息。
2. `DevicePayloadApplyStageHandler`
   - 在 `PAYLOAD_APPLY` 阶段依赖 `DevicePropertyMetadataService`。
   - 用正式物模型帮助 latest 属性更新保持统一字段语义。
3. `DeviceTelemetryMappingServiceImpl`
   - 从 `specsJson` 解析 telemetry 映射。
   - 让正式字段与 telemetry 指标映射关系稳定可复用。
4. `DeviceServiceImpl.listMetricOptions`
   - 先读取产品正式 property 模型，再补设备当前已有属性。
   - 体现“正式定义优先，运行值补充兜底”的顺序。
5. `DeviceMessageServiceImpl`
   - 在运行期字段解释和类型判断中读取 property 模型。

统一规则是：`运行期服务可以读物模型，但不能顺手改物模型。`

### 5.3 不得越界的模块

以下模块或层次不应持有正式物模型写入职责：

1. `spring-boot-iot-protocol`
   - 负责协议解码、Topic 解析、报文标准化。
   - 不负责定义产品正式字段标准。
2. `spring-boot-iot-message`
   - 负责 HTTP / MQTT 入口与 Pipeline 编排。
   - 可以调度消费物模型的 stage handler，但不应直接改写正式模型。
3. `latest 属性 / telemetry / message log`
   - 负责运行结果和证据留痕。
   - 可以产出候选和治理依据，但不能天然等于正式物模型。
4. `alarm / report / risk`
   - 应消费标准化字段，不应为了单一业务场景反向修改产品正式模型。

统一规则是：`证据层只能提建议，不能越权变成正式契约写入者。`

## 6. 物模型的核心用途和作用

为便于团队统一复述，物模型的作用建议固定为以下 6 条：

1. `统一字段语言`
   - 把厂家字段、样本字段和规范术语沉淀为统一的正式字段口径。
2. `承接证据治理`
   - 把规范证据、报文证据和人工输入转成正式契约。
3. `支撑运行解释`
   - 为 latest 属性更新、telemetry 映射和运行期字段解释提供依据。
4. `稳定业务消费`
   - 为风险点、规则、告警、事件、报表等下游能力提供统一字段语义。
5. `隔离设备个体波动`
   - 防止某台设备的偶发字段直接污染整类产品正式标准。
6. `形成产品级治理抓手`
   - 让产品不仅有档案和接入协议，还具备正式契约治理能力。

## 7. 团队统一话术

为了减少歧义，团队内部建议统一使用以下表达：

1. `物模型是产品级正式契约层，不是设备运行值。`
2. `消息日志是证据，latest/telemetry 是运行结果，物模型是正式定义。`
3. `规范、报文、人工输入都只能形成候选，不能直接绕过治理写正式模型。`
4. `正式物模型写权限收口在产品治理链路，其他模块只读不写。`
5. `下游业务应消费标准化字段，不应反向定制产品物模型。`

## 8. 结论

在 `spring-boot-iot` 当前基线中，物模型不是附属页面能力，而是 IoT 主链路中的正式契约层。

它的正确边界应始终保持为：

1. 业务上：定义产品级标准字段，而不是承载设备运行值。
2. 架构上：上接证据治理，下接运行解释和业务消费。
3. 代码上：写权限只在产品治理链路，其他模块只消费、不越界。

后续无论是继续扩展规范证据、报文 compare、字段适用性判断，还是把更多风险与报表能力建立在正式字段上，都应以这一边界为前提，而不是回退到“运行数据直接定义正式契约”的模式。
