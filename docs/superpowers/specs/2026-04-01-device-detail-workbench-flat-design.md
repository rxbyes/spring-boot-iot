# 设备资产中心详情抽屉扁平台账治理式改造设计

> 日期：2026-04-01
> 范围：`spring-boot-iot-ui`
> 页面：`/devices`
> 主题：把设备详情从“等宽小卡墙”改造成“扁平台账治理式”工作台

## 1. 背景

`/devices` 当前详情抽屉虽然已经完成第一轮扁平化，但仍保留明显的“卡片墙”问题：

- 中段 `身份与部署台账`、`运行与认证台账` 仍使用 `3` 列等宽网格。
- `设备编码`、`设备名称`、`部署位置`、时间字段这类长值被迫和短值共用同一宽度，容易换成多行。
- 区块说明文案偏长，进一步挤占了字段本身的展示空间。
- 视觉上更像“把字段塞进很多小卡片”，而不是稳定的账本工作区。

用户反馈的核心不是“想让页面更紧凑”，而是“不要压缩、不要挤压的感觉”。因此本轮不采用缩字体、缩 padding、继续堆小卡的方式，而是通过重排版式和字段优先级解决拥挤问题。

## 2. 目标

- 让详情抽屉整体更接近扁平 SaaS 工作台，而不是多段碎卡堆叠。
- 保留当前区块语义，但重做区块内部排版，让长值获得更宽、更自然的展示空间。
- 首屏先完成“资产判断 + 运行判断”，再进入完整台账核对。
- 已登记设备和未登记设备继续复用同一套详情语法，不再演化出第二套页面骨架。
- 同步把这套规则写回文档，防止以后回流到“等宽小卡墙”。

## 3. 非目标

- 不把设备详情改成多 Tab 工作台。
- 不新增第二套路由或详情抽屉。
- 不通过折叠、收起、隐藏主字段来换取表面上的清爽。
- 不引入新的私有主题色或偏营销风格的 hero 头图。

## 4. 设计决策

采用“扁平台账治理式”方案：借鉴现有产品工作台的扁平控制台语法，但不照搬 Tab 结构，而是把设备详情收口为“轻头部 + 双概览板 + 双主账本 + 底部补充区”的单页工作台。

### 4.1 总体结构

已登记设备详情抽屉调整为以下顺序：

1. `轻头部`
2. `资产状态与接入概况`
3. `首屏双概览板`
4. `身份与部署台账`
5. `运行与认证台账`
6. `关系与建档补充`
7. `扩展元数据快照`

未登记设备详情抽屉继续复用同一语法，但中段改为：

1. `轻头部`
2. `资产状态与接入概况`
3. `来源档案与失败摘要`
4. `最近载荷`

### 4.2 轻头部

轻头部只承担“对象定位 + 当前判断”，不再放长段说明。

保留内容：

- 设备名称或设备编码
- 一句当前判断
- 状态 pills：`已登记/未登记`、`在线/离线`、`已激活/未激活`、`启用/禁用`

去掉内容：

- 长段介绍性文案
- 与下方摘要带重复的字段说明

判断语气示例：

- 已登记设备：`资产信息完整，可继续核对部署、运行与认证台账。`
- 未登记设备：`当前仍是未登记上报，优先确认失败阶段与来源记录。`

### 4.3 资产状态与接入概况

顶部继续保留 `4` 格摘要带，但每格收口为“标签 + 主值 + 一句短提示”，不再承载长解释。

已登记设备固定为：

- `产品归属`
- `在线状态`
- `激活状态`
- `设备状态`

未登记设备固定为：

- `登记状态`
- `最近上报`
- `失败阶段`
- `失败摘要`

摘要带角色是“当前判断”，不是完整档案。它要更像工作台摘要条，而不是另一层小账本。

### 4.4 首屏双概览板

已登记设备在摘要带下增加两个更宽的概览板：

- 左侧：`资产概览`
- 右侧：`运行概览`

`资产概览` 负责优先表达：

- 设备编码
- 设备名称
- 产品归属
- 部署位置

`运行概览` 负责优先表达：

- 最近在线
- 最近离线
- 最近上报
- 更新时间

这两个板块不是 hero 卡，也不是第二套复杂图表，只是把最关键的长值和时间值提前，并给它们更宽、更稳的版面。

### 4.5 双主账本

保留 `身份与部署台账` 和 `运行与认证台账` 两块主账本，但内部不再使用 `3` 列等宽小卡，而改成 `2` 列混排账本。

#### 4.5.1 身份与部署台账

字段分层如下：

- 宽字段：
  - `设备编码`
  - `设备名称`
  - `产品归属`
  - `部署位置`
- 普通字段：
  - `设备 ID`
  - `节点类型`
  - `接入协议`
  - `固件版本`
  - `IP 地址`

宽字段统一跨两列，普通字段使用单列卡片。

#### 4.5.2 运行与认证台账

字段分层如下：

- 宽字段：
  - `最近在线`
  - `最近离线`
  - `最近上报`
- 普通字段：
  - `创建时间`
  - `更新时间`
  - `Client ID`
  - `用户名`
  - `密码`
  - `设备密钥`

其中 `密码`、`设备密钥` 继续遮罩显示，但视觉上不再和时间字段争抢同样的主位置。

### 4.6 底部补充区

`关系与建档补充` 和 `扩展元数据快照` 继续后置，但使用与主账本同一套扁平工作台语法。

`关系与建档补充` 保留：

- `父设备`
- `网关设备`
- `父设备主键`
- `网关主键`

`扩展元数据快照` 继续保留 `metadataJson` 的结构化预览，但整体视觉是“安静补充区”，不再像一张独立系统卡片。

### 4.7 未登记设备

未登记设备不新增第二套详情设计，而是在同一套语法下改写字段优先级。

`来源档案与失败摘要` 使用相同的 `2` 列混排账本：

- 宽字段：
  - `Topic`
  - `失败摘要`
- 普通字段：
  - `设备编码`
  - `产品标识`
  - `协议编码`
  - `来源记录`

`最近载荷` 继续放在补充区，保证错误上下文和原始报文仍可同屏查看，但不抢主账本首屏位置。

### 4.8 视觉语法

整体视觉采用“扁平台账治理式”：

- 保持浅底、白卡、轻描边、轻阴影的控制台语法。
- 区块标题更像工作台标题，不再配长说明段。
- 板块更宽、更稳，减少“很多等宽小卡并列”的感觉。
- 账本字段优先靠版面层级解决可读性，不靠一味缩字号或压间距。

参考图只借以下特征：

- 扁平工作台感
- 稳定的区块节奏
- 信息主次更清楚

明确不借以下特征：

- 多 Tab 页签结构
- 产品经营工作台那种跨视图切换骨架

### 4.9 响应式规则

- 主账本内部统一使用 `2` 列混排。
- 当抽屉可用宽度不足时，双概览板与双主账本比现状更早切为单列。
- 宽字段在双列阶段跨两列，在单列阶段自然回到单列，不额外制造占位空洞。
- 不允许为了勉强维持双栏而把文本挤压到更碎。

## 5. 组件边界

- 继续使用 `spring-boot-iot-ui/src/components/device/DeviceDetailWorkbench.vue` 承载设备详情布局。
- `DeviceWorkbenchView.vue` 继续只负责：
  - `StandardDetailDrawer`
  - 详情补数与错误态
  - 标题、副标题、状态 tags、底部动作
- 所有详情字段编排、主次关系和样式都留在 `DeviceDetailWorkbench`。

这样可以维持设备详情的独立边界，不把更多排版逻辑塞回超长的 `DeviceWorkbenchView.vue`。

## 6. 数据流与状态

- 列表摘要秒开、后台补数的现有策略保持不变。
- 本轮只改详情布局和字段排序，不改接口、不改字段来源。
- 详情补数中的 loading / error / empty 语义继续由 `StandardDetailDrawer` 承接，不额外新增页面私有状态壳。

## 7. 风险与控制

- 风险：把设备详情做得太像产品工作台，导致详情抽屉出现不必要的“经营工作台”感。
  - 控制：只借扁平工作台的节奏和视觉，不引入页签、经营指标或第二套工作区。

- 风险：为了追求“更宽”而让信息层级变得松散。
  - 控制：所有宽字段必须是长值或关键值，不能无差别放大所有字段。

- 风险：未登记设备又长出第二套排版语法。
  - 控制：继续共用同一组件和同一语法，只在字段映射上做差异。

## 9. 验证方式

按以下顺序执行：

1. 先修改 `DeviceDetailWorkbench` 组件测试，锁定：
   - 轻头部语气不回流长说明
   - 主账本采用宽窄混排
   - 未登记设备仍复用同一扁平语法
2. 再修改组件实现与样式。
3. 最后运行受影响验证：
   - `vitest` 聚焦组件与视图测试
   - `npm run component:guard`
   - `npm run list:guard`
   - `npm run style:guard`
