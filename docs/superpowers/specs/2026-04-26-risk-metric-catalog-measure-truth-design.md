# 风险指标目录切换到“设为监测数据”真相源设计

## 1. 背景

当前监测型产品的风险指标目录发布规则，仍然主要依赖 `productKey / scenarioCode / contractIdentifier` 的关键字和字段前缀猜测。这条链路已经带来两个持续性问题：

1. 它不能准确表达业务规则。像 `zhd-monitor-multi-displacement-v1` 这类多维设备，正式字段里虽然同时存在裂缝、倾角、加速度，但目录是否发布，取决于旧规则硬编码，而不是治理人员在契约字段页的明确选择。
2. 它无法兑现“手工移出目录”的治理语义。用户在契约字段页点击 `取消趋势展示` 后，当前系统并不会把该动作直接视为风险目录真相源，因此目录与页面认知可能继续分裂。

本次需要把风险指标目录的准入规则，从“规则猜测哪些字段像风险测点”，切换为“只有被明确治理为监测数据的正式字段，才允许进入风险目录”。

## 2. 目标与非目标

### 2.1 目标

1. 风险指标目录的唯一准入真相源切换为契约字段页的 `设为监测数据`。
2. 契约字段页的 `取消趋势展示` 成为风险目录的唯一移出真相源。
3. `设为状态事件`、`设为运行参数` 不进入风险目录真相源，只保留对象洞察分组语义。
4. 合同发布、契约字段治理动作、读侧自愈和审计脚本都复用同一套目录重建逻辑，不再保留产品白名单/关键字推断分叉。
5. 风险绑定、阈值策略、联动/预案继续只消费 `risk_metric_catalog`，不从正式字段全量旁路兜底。
6. 历史正式字段产品即使暂未形成 `releaseBatchId`，也要能按同一 truth source 重建目录，不能长期停留在旧目录残留。

### 2.2 非目标

1. 本轮不新增独立的“风险目录准入配置表”或平行治理入口。
2. 本轮不把“所有业务数据 property 自动进目录”设为默认行为。
3. 本轮不保留“多维设备只放裂缝量”“GNSS 只放总位移”这类产品级硬编码准入策略。
4. 本轮不新增 schema；若后续需要把“每次目录准入调整”沉淀为独立历史版本，再单开 schema 设计。

## 3. 核心决策

采用“**契约字段页监测数据选择 = 风险目录准入真相源**”方案。

唯一有效的目录准入动作只有两个：

1. `设为监测数据`
2. `取消趋势展示`

其中：

1. `设为监测数据` 表示该正式字段进入风险目录候选集合。
2. `取消趋势展示` 表示该正式字段退出风险目录候选集合。
3. `设为状态事件`
4. `设为运行参数`

后两者只维护对象洞察分组，不进入风险目录真相源，也不直接触发目录发布。

## 4. 真相模型与准入边界

### 4.1 风险目录准入条件

一个字段进入 `risk_metric_catalog`，必须同时满足：

1. 当前产品支持风险指标治理，属于监测型产品边界。
2. 字段属于当前已发布正式合同。
3. `modelType=property`。
4. 在产品 `metadataJson.objectInsight.customMetrics[]` 中存在对应正式字段标识。
5. 对应条目满足：
   - `group=measure`
   - `enabled=true`
   - `includeInTrend=true`

任何一项不满足，都不能进入风险目录。

### 4.2 明确排除项

下列对象一律不进入风险目录：

1. `event`
2. `service`
3. `group=status`
4. `group=runtime`
5. 未被明确执行 `设为监测数据` 的正式字段
6. 已执行 `取消趋势展示` 的正式字段

### 4.3 用户动作语义

契约字段页动作的业务语义固定为：

1. `设为监测数据`
   - 写入或更新 `metadataJson.objectInsight.customMetrics[]`
   - `group=measure`
   - `enabled=true`
   - `includeInTrend=true`
   - 触发风险目录重建
2. `取消趋势展示`
   - 移除对应 `customMetrics[]` 条目
   - 触发风险目录重建
3. `设为状态事件`
   - 只写对象洞察分组
   - 不触发进入风险目录
4. `设为运行参数`
   - 只写对象洞察分组
   - 不触发进入风险目录

如果某字段当前已在风险目录中，页面必须要求先执行 `取消趋势展示`，使其退出目录后，才能再改成状态事件或运行参数。

## 5. 同步链路设计

### 5.1 合同发布链路

合同发布后，仍由设备域生成 `releaseBatchId`、保存 `BEFORE_APPLY / AFTER_APPLY` 快照，并发出 `ProductContractReleasedEvent`。

Alarm 侧监听发布事件后，不再按 `productKey / scenario / identifier` 猜测可发布字段，而是执行：

1. 读取本次产品当前最新 `metadataJson.objectInsight.customMetrics[]`。
2. 提取其中满足 `measure + enabled + includeInTrend` 的正式字段标识。
3. 与本次发布批次中的正式 `property` 字段求交集。
4. 对交集字段执行 `upsert risk_metric_catalog`。
5. 对当前产品下已存在、但不在新交集中的目录行执行 `RETIRED`。

目录行的 `releaseBatchId` 继续写入本次真实发布批次。

若产品尚未形成正式发布批次，但已经存在历史正式字段，则同一套重建逻辑仍需可执行：此时读当前正式 `property` 字段现状与 `customMetrics[]` truth source 求交集后直接 upsert 目录，目录行允许暂时保留 `releaseBatchId=null`，直到后续首次正式发布再补齐批次追溯。

### 5.2 契约字段治理链路

用户在契约字段页执行以下动作后，必须触发同一套目录重建逻辑：

1. `设为监测数据`
2. `取消趋势展示`

若当前产品没有正式发布批次，上述动作也不能静默跳过；保存成功后仍需按“当前正式字段现状 + metadata truth source”立即重建目录。

重建时不允许按单字段增量 patch 风险目录，而是统一读取：

1. 当前产品最新正式发布批次
2. 当前批次正式字段快照
3. 当前产品最新 `metadataJson.objectInsight.customMetrics[]`

然后整批重算“应启用目录”和“应退役目录”。

这样可以保证：

1. 合同发布后的目录状态
2. 发布后的手工治理状态
3. 版本台账显示结果

都来自同一套逻辑。

### 5.3 读侧自愈与历史补账

`formal-metrics`、待治理推荐、审计脚本等旁路读侧，不再允许自己猜哪些字段应进目录。

它们只能复用同一个目录重建器，输入同样固定为：

1. 正式字段快照
2. 当前监测数据选择集
3. 当前产品最新正式 `releaseBatchId`

这样历史产品缺目录时，补出来的结果与发布链、页面治理链保持完全一致。

## 6. 组件改造边界

### 6.1 Alarm 模块

`spring-boot-iot-alarm` 负责：

1. 用新的真相源实现 `RiskMetricCatalogPublishRule`
2. 新增统一的目录重建协调器
3. 在合同发布、读侧自愈、审计/补账入口中复用同一重建逻辑
4. 保证 `risk_metric_catalog.releaseBatchId` 不再因为读侧自愈被写成 `null`

### 6.2 Device 模块

`spring-boot-iot-device` 继续负责：

1. 合同发布批次与快照真相
2. 产品 `metadataJson.objectInsight.customMetrics[]` 的写入与校验
3. 契约字段页动作复用 `PUT /api/device/product/{id}` 更新产品元数据

本轮不新增设备域平行配置表。

### 6.3 UI 模块

`spring-boot-iot-ui` 需要补齐：

1. 契约字段页对“风险目录真相源”的文案说明
2. 已在风险目录中的字段，`设为状态事件 / 设为运行参数` 的禁用或拦截提示
3. `取消趋势展示` 的目录退出文案

页面不新增第二套“发布风险目录”按钮；目录同步仍为治理动作后的自动结果。

## 7. 批次语义

在不扩 schema 的前提下，目录行继续挂在当前最新正式 `releaseBatchId` 下。

因此本轮固定采用以下语义：

1. `releaseBatchId` 表示“当前最新正式合同批次下，按当前监测数据选择生效的目录结果”。
2. 用户在发布后再次执行 `设为监测数据` 或 `取消趋势展示`，目录重建结果仍回写到当前最新正式批次名下。
3. 这保证版本台账、风险绑定、风险 replay 继续可用。
4. 但它不表达“每次监测数据治理动作的独立历史版本”。若后续需要该能力，必须单独设计 schema。

## 8. 存量数据治理

旧规则遗留数据需要按新口径重新审计：

1. 已在目录中，但当前不在 `measure + enabled + includeInTrend` 选择集中的字段，标记为 `CATALOG_RULE_DRIFT`，给出退役建议。
2. 当前已被明确 `设为监测数据`，但目录里缺失的字段，标记为 `MISSING_CATALOG_NEEDS_BACKFILL`，给出补齐建议。
3. 不再允许因为产品命名像“裂缝/GNSS/多维/雨量/泥位”就自动判定应进目录。

对 `CXH15522812` 而言：

1. 裂缝、倾角、加速度是否出现在正式绑定候选里，不再由产品白名单决定。
2. 哪些字段出现，只取决于这些正式字段是否在契约字段页被明确执行过 `设为监测数据`。

### 9.1 后端回归

至少补齐以下测试：

1. 契约字段页点击 `设为监测数据` 后触发目录同步链路

3. 已在目录中的字段，不能直接改成状态事件或运行参数

4. 重新跑监测型产品风险目录审计，确认没有“未设监测数据却仍残留在目录中”的旧数据

## 11. 结论

本次收口后，风险指标目录的业务语义将变得单一且可解释：

1. 想进入风险绑定、阈值策略、联动/预案闭环的正式字段，必须在契约字段页被明确标记为 `设为监测数据`。
2. 想退出风险闭环的字段，必须执行 `取消趋势展示`。
3. 系统不再替治理人员猜测“哪些字段像风险测点”。

这条规则同时适用于新发布产品、历史补账、读侧自愈和共享环境审计。
