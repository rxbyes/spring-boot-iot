# G5 归档批次摘要卡下钻设计

- 日期：2026-04-26
- 阶段：可观测证据链 G5
- 范围：`/system-log` 中 `归档批次台账` 的摘要卡交互下钻

## 1. 背景

G4 已经把归档批次异常摘要卡、`compareStatus / onlyAbnormal` 筛选和列表级异常高亮接到了 `/system-log`。当前运维已经可以看到“最近是否有异常批次、偏差总量多大”，但从“看到异常”到“落到具体批次”之间还要多做一步：要么手动改筛选，要么再去列表中找对应行。

G5 的目标不是再扩一层新能力，而是把已有读侧和现有抽屉串得更顺，让异常摘要卡真正承担“快速入口”的职责。

## 2. 目标

本轮要解决 2 件事：

1. 让前 3 张摘要卡成为可点击的快捷筛选入口。
2. 让 `最近异常批次` 卡可以直接把用户送到对应列表行，并自动打开详情抽屉。

## 3. 非目标

本轮明确不做这些事：

- 不新增后端接口。
- 不新增 URL query 状态。
- 不新增分页策略，不自动翻页，不自动扩大 `pageSize`。
- 不新增“跨页搜索最近异常批次”的特殊后端逻辑。
- 不调整 G3 的 `批次对比` 与 G2 的 `确认报告预览` 抽屉结构。

## 4. 方案对比

### 方案 A：纯页面本地交互

点击摘要卡后，直接修改当前页面筛选状态并刷新列表；点击 `最近异常批次` 时，刷新列表后尝试定位目标批次并自动打开详情抽屉。

- 优点：改动最小，风险最低，和现有 `/system-log` 交互模型一致。
- 缺点：刷新页面后不会保留“我是从哪张卡点进来的”。

### 方案 B：URL 同步交互

把 `compareStatus / onlyAbnormal / focusedBatchNo / autoOpenBatch` 写入路由 query。

- 优点：状态可分享、可刷新恢复。
- 缺点：需要给 `/system-log` 引入更重的路由状态模型，本轮收益不抵成本。

### 方案 C：混合 URL + 本地状态

只把 `focusedBatchNo` 放进 URL，其余仍用本地状态。

- 优点：比全量 URL 轻。
- 缺点：会形成两套状态规则，维护成本高。

### 结论

采用 **方案 A：纯页面本地交互**。

## 5. 交互设计

### 5.1 摘要卡点击规则

G4 的 4 张摘要卡在 G5 中分成两类：

1. **筛选型卡片**
   - `异常批次`
   - `执行偏差总量`
   - `剩余过期总量`

2. **定位型卡片**
   - `最近异常批次`

### 5.2 前 3 张卡片的行为

前 3 张卡片点击后，统一采用“**替换当前异常筛选**”口径，而不是叠加到现有异常筛选。

日期范围仍保留；被替换的仅是异常相关筛选条件。

#### `异常批次`

- 清空 `compareStatus`
- 设置 `onlyAbnormal=true`
- 保留 `dateFrom / dateTo`
- 刷新列表
- 不自动打开详情抽屉

#### `执行偏差总量`

- 设置 `compareStatus=DRIFTED`
- 设置 `onlyAbnormal=false`
- 保留 `dateFrom / dateTo`
- 刷新列表
- 不自动打开详情抽屉

#### `剩余过期总量`

- 保持不新增后端筛选参数
- 设置 `onlyAbnormal=true`
- 清空 `compareStatus`
- 保留 `dateFrom / dateTo`
- 刷新列表
- 列表层对 `remainingExpiredRows > 0` 的行使用更强的焦点样式

这个处理让“剩余过期”仍然是一个异常入口，但不把本轮范围扩大到新的服务端过滤规则。

### 5.3 `最近异常批次` 的行为

点击 `最近异常批次` 卡片后：

1. 读取 `overview.latestAbnormalBatch`
2. 若值为空，则卡片禁用，不触发动作
3. 否则：
   - 切到 `onlyAbnormal=true`
   - 清空 `compareStatus`
   - 保留 `dateFrom / dateTo`
   - 刷新列表
   - 刷新完成后在当前结果中查找该批次
   - 若查到，则定位到该行并自动打开详情抽屉
   - 若查不到，则仅展示页面内轻提示，不自动翻页、不自动改 page size

## 6. 页面状态设计

前端新增一层轻量页面状态，不改后端 contract。

### 6.1 新增本地状态

- `summarySelection`
  - `NONE`
  - `ABNORMAL`
  - `DRIFTED`
  - `REMAINING`
  - `LATEST_ABNORMAL`

- `focusedBatchNo`
  - 点击 `最近异常批次` 后短暂保存目标批次号

- `autoOpenFocusedBatch`
  - 布尔标记
  - 只用于“列表刷新完成后是否自动打开抽屉”

- `messageArchiveBatchFocusHint`
  - 页面级轻提示
  - 用于显示“最近异常批次不在当前结果中，请调整时间范围后重试”

### 6.2 状态清理规则

- 点击 `重置`：
  - 清空摘要卡激活态
  - 清空 `focusedBatchNo`
  - 清空 `autoOpenFocusedBatch`
  - 清空 `messageArchiveBatchFocusHint`

- 用户手动修改筛选后再点 `筛选`：
  - 以手动筛选结果为准
  - 若不再对应任何摘要卡口径，则清空 `summarySelection`

- 自动打开抽屉成功后：
  - 清空 `autoOpenFocusedBatch`
  - 保留当前筛选结果

## 7. 视觉与反馈

### 7.1 摘要卡激活态

被点击的摘要卡显示克制但明确的激活态：

- 更清晰的边框
- 稍强的背景层次
- 补一行短提示：
  - `当前已筛异常批次`
  - `当前已筛 DRIFTED`
  - `当前已筛剩余过期异常`
  - `当前定位最近异常批次`

其它卡片保持普通态，不整体降灰。

### 7.2 列表落点反馈

- 点击前 3 张卡：
  - 只刷新列表
  - 不自动滚动

- 点击 `最近异常批次`：
  - 刷新后滚动到目标行附近
  - 该行做一次短暂高亮
  - 然后自动打开详情抽屉

### 7.3 抽屉自动打开说明

不额外弹 toast。

详情抽屉顶部补一条轻提示：

- `已从最近异常批次定位`

这样既能解释“为什么抽屉自己开了”，又不会让页面噪音太大。

## 8. 失败与降级处理

### 8.1 `latestAbnormalBatch` 为空

- `最近异常批次` 卡禁用
- 不做空跳转

### 8.2 点击筛选型摘要卡后列表接口失败

- 保留原列表内容
- 摘要卡激活态回退到点击前
- 页面沿用现有台账错误提示

### 8.3 点击 `最近异常批次` 后当前 5 条结果中未命中

- 不自动翻页
- 不自动扩大 `pageSize`
- 不自动扩大时间窗口
- 页面内提示：
  - `最近异常批次不在当前结果中，请调整时间范围后重试`

### 8.4 自动打开抽屉时详情接口失败

- 列表定位保留
- 抽屉沿用当前 compare / preview 错误态
- 不再叠加第二层全局提示

## 11. 风险与控制

### 风险 1：摘要卡状态与手动筛选互相打架

控制方式：

- 手动筛选优先
- 摘要卡激活态只表达“当前筛选是否符合某张卡的语义”
- 不保留脱节的伪激活状态

### 风险 2：自动定位带来突兀跳动

控制方式：

- 仅 `最近异常批次` 卡触发滚动定位
- 其它卡不做滚动
- 只做一次短暂高亮，不做持续动画

### 风险 3：目标批次不在当前页导致误解

控制方式：

- 明确提示“当前结果中未命中”
- 不偷偷改查询窗口或分页结果

## 12. 结论

G5 采用“**摘要卡可筛选，最近异常批次可直开详情**”的混合下钻方案：

- 前 3 张卡负责把异常快速筛出来
- `最近异常批次` 负责一步把人送进具体批次
- 所有交互都留在当前页本地状态内完成

这样可以在不增加后端复杂度的前提下，把 `/system-log` 的归档批次异常排查路径继续压短一层。
