# 风险绑定正式测点目录化与设备编号搜索设计

**Date:** 2026-04-13  
**Status:** Approved in-session for spec drafting  
**Audience:** 风险对象中心 / 产品契约治理 / 前后端实现 / 测试验收 / 文档维护  
**Scope:** 在不新增一级路由、不改写 pending 治理主链的前提下，把 `/risk-point -> 风险绑定工作台 -> 维护绑定` 收口为“按设备编号快速定位 + 只允许绑定已发布到风险指标目录的正式字段”。

## 1. 背景

当前风险对象中心里的“风险绑定工作台 -> 维护绑定”已经承担以下职责：

1. 新增正式绑定。
2. 整机解绑。
3. 删除单测点绑定。
4. 更换单测点。

但当前测点来源仍复用通用设备接口 `GET /api/device/{deviceId}/metrics`。该接口的语义是“设备可见测点”，而不是“可进入正式风险闭环的测点”，因此它会把以下内容一起带到正式绑定下拉里：

1. 产品正式字段。
2. latest 属性表里观察到的设备属性。
3. 已存在治理语义、但并未进入风险指标目录的字段。

这与当前平台已确立的治理口径存在冲突：

1. 产品契约字段页已经把“正式字段”和“已发布到风险指标目录的字段”区分开来。
2. 风险对象、阈值策略、联动和预案应优先围绕风险指标目录收口，而不是继续直接消费所有正式字段。
3. 一线用户在设备较多时，仅靠滚动下拉逐条找设备编号，效率明显不足。

因此，本轮要把“维护绑定”进一步收紧成真正的正式闭环维护入口。

## 2. 目标

本轮只完成两项目标：

1. 在“新增正式绑定”中，设备选择支持输入设备编号快速定位。
2. 在“新增正式绑定 / 更换测点”中，测点只允许选择当前设备所属产品里**已发布到启用中风险指标目录**的正式字段。

## 3. 非目标

本轮明确不做：

1. 不新增一级路由、独立绑定页或第二套风险绑定系统。
2. 不改变“待治理转正”子视图的候选推荐链路。
3. 不把通用设备测点接口整体改写成目录专用接口。
4. 不自动清理或回滚历史正式绑定数据。
5. 不把风险治理规则搬回设备域通用接口默认语义。

## 4. 方案比较

### 4.1 方案 A：只改前端过滤

做法：

1. 设备框增加输入搜索。
2. 测点继续调用 `GET /api/device/{deviceId}/metrics`。
3. 前端按 `riskMetricId != null` 过滤后展示。

优点：

1. 改动最小。
2. 上线最快。

缺点：

1. 规则只存在于页面层，领域约束不牢。
2. 其它入口仍可继续误用全量设备字段。
3. 用户若绕过前端构造请求，后端仍可能接受非目录字段。

### 4.2 方案 B：直接收紧通用设备测点接口

做法：

1. 直接修改 `GET /api/device/{deviceId}/metrics` 的默认语义。
2. 让它只返回目录字段。

优点：

1. 接口层最统一。

缺点：

1. 会影响其它依赖该接口的页面与流程。
2. `待治理转正`、设备相关其它入口可能仍需要更宽口径。
3. 回归风险较大。

### 4.3 方案 C：新增风险绑定专用测点接口，推荐

做法：

1. 保留 `GET /api/device/{deviceId}/metrics` 作为通用设备字段接口。
2. 新增风险绑定专用读侧接口，只返回当前设备所属产品中、已发布到启用中 `risk_metric_catalog` 的正式目录字段。
3. `新增正式绑定` 与 `更换测点` 都改走该专用接口。
4. 写接口再补一层后端一致性校验，拒绝非目录字段写入正式绑定。

优点：

1. 语义最清晰。
2. 回归面最小。
3. 能把“只允许目录字段进入正式绑定”从 UI 规则升级为领域规则。

缺点：

1. 需要新增一个专用接口和对应测试。

### 4.4 选型

本轮采用 `方案 C：新增风险绑定专用测点接口`。

原因：

1. 既能保持设备域通用能力稳定，又能让风险域明确执行自己的正式闭环规则。
2. 能避免把 `待治理转正` 这种治理入口误伤成“只能看目录字段”。
3. 更符合“控制面与治理工作台围绕正式真相源收口”的既有方向。

## 5. 目标态行为

### 5.1 新增正式绑定

`/risk-point -> 风险绑定工作台 -> 维护绑定 -> 新增正式绑定` 调整为：

1. 设备下拉保持现有单选下拉形态，不新增独立搜索弹窗。
2. 设备下拉增加输入过滤能力，优先支持按 `deviceCode` 快速定位。
3. 选项文案继续显示为 `设备编号 - 设备名称`，便于一线逐条核对。
4. 选中设备后，测点下拉只加载“当前设备所属产品已发布到启用中风险指标目录”的正式字段。
5. 测点为空时，页面明确提示“该设备所属产品暂无可用于风险绑定的正式目录字段”。

### 5.2 更换测点

“更换测点”改为复用同一套专用目录接口：

1. 打开替换区时只加载目录字段。
2. 前端继续排除当前绑定自身和同设备下已绑定的其它测点。
3. 最终候选集固定为：

`目录字段` ∩ `当前设备有效字段` ∩ `当前未占用字段`

### 5.3 待治理转正

“待治理转正”保持现状，不跟本轮一起收紧：

1. 继续按 pending 推荐链路出候选。
2. 继续允许从治理证据里完成规范收口。
3. 不把它强行改造成“只能看已发布目录字段”的入口。

原因是“待治理转正”的职责是治理收口，而不是正式目录消费入口。

## 6. 接口与模块边界

### 6.1 前端

前端仍停留在现有页面和抽屉结构：

- 路由：`/risk-point`
- 工作台：`风险绑定工作台`
- 子视图：`维护绑定 / 待治理转正`

不新增路由，不拆二级页面。

### 6.2 后端新增接口

新增一个风险绑定专用读侧接口，语义应显式表达“正式绑定目录测点”。建议形态：

- `GET /api/risk-point/devices/{deviceId}/formal-metrics`

该接口返回最小字段集：

1. `identifier`
2. `name`
3. `dataType`
4. `riskMetricId`

返回条件固定为：

1. 当前设备可被当前登录人访问。
2. 设备所属产品存在启用中的 `risk_metric_catalog`。
3. 目录项与当前设备正式字段标识可对齐。
4. 只返回拥有有效 `riskMetricId` 的目录字段。

### 6.3 现有接口保留

以下接口继续保留原语义：

1. `GET /api/device/{deviceId}/metrics`
2. `GET /api/risk-point/bindable-devices/{riskPointId}`
3. `POST /api/risk-point/bind-device`
4. `POST /api/risk-point/bindings/{bindingId}/replace`

其中 `GET /api/device/{deviceId}/metrics` 不再承担正式风险绑定目录语义。

### 6.4 写侧一致性校验

`bind-device` 与 `replace` 的服务层增加同一条规则：

1. 若提交的 `metricIdentifier / riskMetricId` 不属于“当前设备所属产品启用中的目录字段”，则拒绝写入。
2. 拒绝逻辑必须在后端完成，不能只依赖前端下拉限制。

这样即使有人绕过前端直接构造请求，也不能把非目录字段写进 `risk_point_device`。

### 6.5 模块边界

模块职责保持如下：

1. `spring-boot-iot-device`
   - 继续承载通用设备字段查询。
2. `spring-boot-iot-alarm`
   - 承载风险绑定专用目录读侧接口。
   - 承载正式绑定写侧目录校验。
3. `spring-boot-iot-ui`
   - 承接设备编号搜索与正式目录测点下拉交互。

不把风险治理规则反向塞回设备域通用接口默认语义。

## 7. 数据流

正式绑定维护的数据流调整为：

1. 页面打开时，仍加载：
   - `bindable-devices`
   - `binding-groups`
2. 用户在设备框输入设备编号，本地过滤风险点可绑定设备列表。
3. 用户选中设备后，前端调用风险绑定专用目录接口。
4. 前端基于接口结果叠加“已绑定去重”得到最终候选。
5. 用户提交新增或替换。
6. 后端再次校验该字段是否属于设备所属产品的启用目录。
7. 校验通过后写入 `risk_point_device`；否则返回业务错误。

## 8. 异常与兼容策略

### 8.1 产品没有目录字段

若产品虽有正式字段，但当前没有任何启用中的目录字段：

1. 设备仍可选中。
2. 测点下拉为空。
3. 页面给出明确提示。
4. 不允许误绑到非目录字段。

### 8.2 历史绑定仍然存在

若历史正式绑定里的字段已不在当前目录中：

1. “当前正式绑定”列表仍正常展示历史数据。
2. 不做静默删除。
3. 但新增与更换时不再允许继续选择这类字段。

### 8.3 目录字段与设备字段无法对齐

若目录已发布，但当前设备无法定位到对应正式字段：

1. 风险绑定专用接口不返回该字段。
2. 页面按“无可绑定正式目录字段”处理。
3. 不回退到设备全量属性列表。

## 9. 详细交互要求

### 9.1 设备搜索

设备选择框要求：

1. 支持输入搜索。
2. 优先围绕 `deviceCode` 提升定位效率。
3. 不改成页面私有搜索弹窗。
4. 不新增第二套列表样式。

### 9.2 测点空态

测点空态要求：

1. 没选设备前：保持“请选择设备后再选测点”语义。
2. 设备已选但无目录字段：明确提示“当前暂无可绑定的正式目录字段”。
3. 不能再把“暂无候选”默认理解成系统异常。

### 9.3 更换测点弹层

更换测点区域要求：

1. 打开时加载目录字段。
2. 继续排除当前绑定自身与已占用字段。
3. 若无可替换字段，则保持空态而不是回退到全量字段。

## 10. 测试设计

### 10.1 前端测试

补到 `RiskPointBindingMaintenanceDrawer` 相关单测：

1. 设备选择框支持输入设备编号过滤。
2. 新增正式绑定只展示目录字段。
3. 更换测点只展示目录字段。
4. 非目录正式字段不再出现。
5. 目录为空时，测点下拉为空且不能提交。

### 10.2 后端测试

补到风险绑定专用接口与写侧校验：

1. 设备所属产品存在启用目录时，只返回目录字段。
2. 产品正式字段存在但未进目录时，不返回。
3. 手工提交未进目录字段时，`bind-device` 拒绝。
4. 手工提交未进目录字段时，`replace` 拒绝。
5. 历史已绑定读取不受影响。

优先在现有测试类补齐，不做无关重构。

## 12. 验收口径

在共享 `dev` 环境中，至少验证以下场景：

1. 打开 `/risk-point`，进入任一风险点的“风险绑定工作台 -> 维护绑定”。
2. 设备下拉可以通过输入设备编号快速锁定设备。
3. 选择一个已有目录发布产品的设备时，测点下拉只出现目录字段。
4. 提交新增正式绑定后，正式绑定列表刷新正常。
5. 选择一个虽有正式字段、但没有目录发布字段的设备时，测点下拉为空并给出明确提示。
6. “更换测点”同样只出现目录字段。
7. “待治理转正”子视图行为不回归。
8. 历史正式绑定展示不受影响。

## 13. 实施顺序

推荐按以下顺序落地：

1. 新增风险绑定专用目录测点接口。
2. 在写侧补目录一致性校验。
3. 前端把新增与替换测点改接新接口。
4. 前端补设备编号输入过滤。
5. 完成单测与文档同步。

## 14. 设计结论

本轮把风险对象中心里的“维护绑定”正式定义为：

**只维护正式风险闭环绑定，不再消费设备全量字段，只消费已发布到风险指标目录的正式字段。**

同时通过设备编号输入搜索，把一线维护动作从“翻下拉逐条找设备”收口成“按编号快速定位后，再只在目录字段中选择真正可闭环的正式测点”。
