# 风险点待治理业务测点收口设计

> 日期：2026-04-03
> 主题：按业务规范和真实上报证据，收口 `/api/risk-point/pending-bindings/*` 的候选测点与转正标准

## 1. 背景

当前风险点待治理转正链路已经具备“候选推荐 + 人工确认 + 正式转正”闭环，但推荐候选仍然偏向“把设备上报里能解析出的字段都列出来”：

- 候选合并了产品物模型、最新属性和消息日志叶子字段，能覆盖真实运行证据。
- 日志层已经排除了 `header / bodies / payloadBase64` 这类传输壳字段。
- 但当前仍会把大量设备状态参数、通信参数和运行维护字段作为风险点候选，例如：
  - `battery_dump_energy`
  - `signal_4g / signal_NB / signal_db`
  - `lat / lon`
  - `temp / humidity`
  - `sw_version`
- 这些字段虽然是“设备真实上报字段”，但并不等于“风险监测正式测点”。

用户已经明确补充了业务规范边界：

1. 风险点待治理转正必须优先遵循监测业务规范，而不是把所有运行参数都纳入正式测点。
2. 设备状态参数要按设备类型区分，只有规范明确要求时才允许进入候选。
3. 当上报同时出现规范字段和厂商原始字段时，候选要同时保留证据，但正式转正默认落规范字段名。

## 2. 目标

- 将风险点 pending 候选从“字段枚举”升级为“规范驱动的业务测点推荐”。
- 候选推荐只保留对风险监测有价值的正式测点，不再把设备状态参数默认暴露成正式候选。
- 规范字段与厂商原始字段允许同时被识别，但候选主标识统一收口到规范字段名。
- 转正写侧补一层兜底校验，防止通过手工构造请求把状态参数直接写进 `risk_point_device`。
- 变更范围只覆盖 `/api/risk-point/pending-bindings/{pendingId}/candidates` 与 `/promote`，不改产品物模型候选链路。

## 3. 非目标

- 本轮不改 `spring-boot-iot-device` 的产品物模型自动候选治理口径。
- 本轮不尝试建立完整的国家/行业标准编码库，不做超出当前规范截图范围的大而全抽象。
- 本轮不自动为所有 pending 做批量转正；系统只收紧候选和写入标准，实际转正仍由人工确认。

## 4. 已确认决策

1. 只收口风险点 pending 链路，不动产品物模型自动候选。
2. 设备状态参数按设备类型区分，只有规范明确要求时才允许进入候选。
3. 候选中同时保留规范字段和厂商原始字段证据，但正式转正默认落规范字段名。

## 5. 方案对比

### 5.1 方案 A：仅维护黑名单，继续用现有候选打分

- 在现有推荐逻辑上继续补黑名单。
- 只排除最明显的设备状态字段。

优点：

- 改动最小。

缺点：

- 无法回答“哪些字段是业务测点”。
- 无法稳定处理 `dispsX` 与 `L1_SW_1.dispsX`、`gpsTotalX` 与 `L1_GP_1.gpsTotalX` 这类规范字段/厂商字段并存场景。
- 会持续漏出业务上不该转正的字段。

### 5.2 方案 B：规范白名单 + 设备类型识别 + 字段归一

- 建立风险点待治理专用的“业务测点治理规则”。
- 先识别设备/测点上下文，再决定哪些字段属于规范测点。
- 同时支持规范字段和厂商原始字段，但候选主标识收口到规范字段。
- 写侧再做一层同规则校验。

优点：

- 与用户补充的业务规则一致。
- 能同时解决“候选过宽”和“手工绕过候选写脏数据”两个问题。
- 能把 `reasonSummary` 变成解释型推荐，而不是纯分数列表。

缺点：

- 需要引入一层规则对象和上下文判断。
- 需要补测试覆盖不同设备类型和别名归一。

### 5.3 方案 C：停止系统推荐，全部改成人工筛选

优点：

- 风险最低。

缺点：

- 治理效率显著下降。
- 与已经落地的“系统推荐 + 人工确认”方向不一致。

## 6. 选型

本轮采用方案 B。

## 7. 规则设计

### 7.1 规则入口

新增一层“风险点待治理业务测点治理规则”，只在 `spring-boot-iot-alarm` 内使用。

职责：

- 识别字段是否属于业务规范测点。
- 识别字段是否属于设备状态参数。
- 将厂商原始字段归一为规范字段。
- 输出候选解释信息，说明“为什么保留/为什么排除/原始证据来自哪些字段”。

### 7.2 规范测点集合

结合用户给出的规范截图，第一批纳入的核心正式测点包括但不限于：

- 裂缝：`value`
- GNSS：`gpsInitial`、`gpsTotalX`、`gpsTotalY`、`gpsTotalZ`
- 深部位移：`dispsX`、`dispsY`
- 加速度：`gX`、`gY`、`gZ`
- 倾角：`X`、`Y`、`Z`、`angle`、`AZI`
- 振动：`PLX`、`PLY`、`PLZ`、`value`、`SJX`、`SJY`、`SJZ`、`SJValue`
- 应力/土压力/雨量/气温/土壤温度/土壤含水率/水位/孔压/渗透压力/流速/沉降/气压等：
  - 在设备类型可确定的前提下允许 `value / totalValue / temp / x / y / z / speed`
- 地声/次声/声发射：`OSP`、`VSP`、`freq`、`wave`、`amplitude`、`energy`、`ringing`、`risetime`、`risecount`、`duration`、`arrivaltime`、`RMS`、`ASL`
- 雷达：`X`、`Y`、`Z`、`speed`

### 7.3 设备状态参数排除集合

以下字段按设备状态参数对待，默认不进入风险点正式候选：

- `ext_power_volt`
- `solar_volt`
- `battery_dump_energy`
- `battery_volt`
- `temp`
- `temp_out`
- `humidity`
- `humidity_out`
- `lon`
- `lat`
- `signal_4g`
- `signal_NB / signal_nb / singal_NB / singal_nb`
- `signal_db / singal_db`
- `sw_version`
- `sensor_state`
- `pa_state`
- `sound_state`
- 以及同类电源、通信、固件、定位、健康状态字段

但如果设备类型或规范明确要求某类 `temp / value / x / y / z` 属于监测量，则允许保留。例如：

- 地表水温、地下水温、孔隙水温的 `temp`
- 倾角的 `X / Y / Z`
- 标靶位移的 `x / y`
- 雷达的 `X / Y / Z`

### 7.4 设备类型识别

候选规则按以下顺序识别上下文：

1. 先看字段前缀是否带规范点位编码，如 `L1_SW_1.dispsX`
2. 再看设备名、产品名、产品键中的中文/英文语义，如“固定测斜仪（深部位移）”“GNSS”“倾角仪”“雨量计”
3. 如上下文仍不明确，则对 `value / temp / x / y / z` 这类歧义字段保持保守，不推荐进入正式候选

### 7.5 字段归一

字段归一规则：

- 如果字段本身就是规范名，直接使用规范名。
- 如果字段为“点位前缀 + 规范字段”，例如：
  - `L1_SW_1.dispsX`
  - `S1_ZT_1.gX`
  - `L1_QJ_2.X`
  则候选主标识统一收口到最后一段规范字段：
  - `dispsX`
  - `gX`
  - `X`
- 原始字段名不丢弃，而是写入候选说明，作为“原始证据字段”展示。

### 7.6 候选展示

候选主展示保持：

- `metricIdentifier`：规范字段名
- `metricName`：优先用业务中文名

候选解释通过 `reasonSummary` 追加：

- 当前识别到的监测类型
- 候选命中原因
- 原始证据字段别名
- 若被排除，则说明是设备状态参数或不在规范白名单

示例：

- `dispsX`
  - `深部位移规范测点；原始证据字段：L1_SW_1.dispsX`
- `temp`
  - 若设备是固定测斜仪，则排除，不进入候选
- `temp`
  - 若设备是地下水温监测设备，则保留，说明其属于温度监测规范字段

### 7.7 转正写侧兜底

`POST /api/risk-point/pending-bindings/{pendingId}/promote` 增加规范校验：

- 请求体里的 `metricIdentifier` 先做同样的归一化。
- 若归一后字段不属于当前设备上下文允许的业务测点，返回 `INVALID_METRIC`，不写正式绑定。
- 若字段属于业务测点，则正式 `risk_point_device.metric_identifier` 落规范字段名，而不是厂商原始字段名。

## 8. 落地范围

### 8.1 后端

- `RiskPointPendingRecommendationServiceImpl`
  - 在三路证据合并后，新增业务测点治理规则过滤和归一。
- `RiskPointPendingPromotionServiceImpl`
  - 在转正写入前按同一规则做归一和兜底校验。
- 必要时新增一到两个规则/值对象类承接治理逻辑，避免把规则硬塞进 service。

### 8.2 前端

- 当前页面无需新增大结构。
- 如继续复用 `reasonSummary`，前端只需照常显示即可，不强制新增字段。
- 若后续希望把“原始字段别名”单独做成标签，再作为下一轮增强处理。

## 9. 验收标准

1. 固定测斜仪（深部位移）候选中保留 `dispsX / dispsY`，排除 `battery_dump_energy / signal_4g / temp / humidity / lat / lon / sw_version`。
2. 当上报中出现 `L1_SW_1.dispsX` 时，候选主标识显示 `dispsX`，说明中保留原始字段证据。
3. 当用户手工提交 `battery_dump_energy` 到 promote 接口时，结果必须是 `INVALID_METRIC`。
4. 真实环境下现有 pending 候选列表明显减少状态参数噪音，正式绑定只写业务测点。
