# 产品定义中心统一经营工作台设计

> 日期：2026-03-30
> 范围：`spring-boot-iot-ui`
> 主题：把 `/products` 内的详情页、物模型页、查看设备页、编辑页重构为同一套经营简报式工作台抽屉
> 状态：已确认，进入实施

## 1. 背景

`产品定义中心` 当前已经完成列表台账、详情简报、物模型设计器、查看设备抽屉等多轮治理，但产品层的深度操作仍分散在四套独立入口里：

- `详情`
- `物模型`
- `查看设备`
- `编辑`

这些入口虽然都留在 `/products` 同一路由里，没有拆一级路由，但交互上仍是四套不同抽屉/弹层心智：

1. 用户进入后会丢失“我现在仍在同一产品经营上下文里”的连续感。
2. 详情、物模型、设备台账、编辑治理之间无法共享同一套经营摘要头部。
3. 同一产品的规模、活跃度、契约、设备、编辑治理被切散，页面气质更像工具集合而不是经营工作台。
4. 物模型设计器与设备列表已经完成视觉升级，但仍停留在“各自独立抽屉”阶段，未形成统一的产品经营舞台。

## 2. 目标

- 在 `/products` 内把 `详情 / 物模型 / 查看设备 / 编辑` 合并为同一套 `产品经营工作台抽屉`。
- 保持“不新增一级路由”的前提下，建立一个沉浸式、经营简报式、商务克制的统一工作台体验。
- 让四个入口共享同一套产品经营头部、同一条视图切换带和同一份产品上下文。
- 让首屏先看到规模、活跃和经营判断，再按视图进入物模型治理、关联设备、编辑治理。
- 保持现有真实接口、秒开详情、物模型候选提炼、关联设备查询和编辑保存能力，不回退功能。

## 3. 非目标

- 不新增 `/products/:id` 一类新路由。
- 不新增后端聚合接口。
- 不改变产品 CRUD、物模型 CRUD、候选提炼、设备分页的后端契约。
- 不把“新增产品”强行并入经营工作台；本轮统一的是产品既有对象的四个深度入口。
- 不新增第二套私有品牌色、营销式头图或英文 eyebrow。

## 4. 已确认设计结论

用户已确认采用以下方向：

- **形态**：经营简报式工作台
- **承载容器**：单一宽抽屉
- **路由策略**：继续保留 `/products`
- **内容组织**：四个真实业务视图，不做假页签

四个视图固定为：

1. `经营总览`
2. `物模型治理`
3. `关联设备`
4. `编辑治理`

入口映射固定为：

- `详情` -> 默认打开 `经营总览`
- `物模型` -> 默认打开 `物模型治理`
- `查看设备` -> 默认打开 `关联设备`
- `编辑` -> 默认打开 `编辑治理`

## 5. 总体结构

### 5.1 抽屉层

- 继续复用 `StandardDetailDrawer` 作为统一抽屉壳层。
- 抽屉宽度提升为更适合工作台阅读的沉浸式规格。
- 页眉不再按四种入口分别维护四套标题结构，而是统一表达当前产品的经营上下文。

### 5.2 工作台头部

统一头部固定包含：

- 产品名称
- `productKey`
- 协议编码 / 节点类型 / 数据格式
- 产品状态
- 核心经营指标
  - 关联设备数
  - 在线设备数
  - 30 日活跃
- 补充经营标签
  - 最近上报
  - 厂商
  - 更新时间

头部语气要求：

- 只保留中文标题、副标题和指标说明。
- 不再出现重复的英文 stage、hero、workspace eyebrow。
- 气质保持“精致现代、商务克制、高管简报式”，不是驾驶舱堆面板。

### 5.3 视图切换层

- 头部下方提供一组真实业务视图切换。
- 切换控件需复用现有控制台品牌 token，维持稳定的 hover / active 节奏。
- 切换后不离开当前抽屉，也不重建产品上下文。

### 5.4 主舞台层

每个视图都必须先给出一层“判断/摘要”，再进入主体内容：

- `经营总览`：延续现有 `ProductDetailWorkbench` 的经营摘要结构，作为总览主视图。
- `物模型治理`：先看候选提炼和正式模型治理摘要，再进入候选目录或正式模型列表。
- `关联设备`：先看总量/在线/离线与在线覆盖，再进入设备台账。
- `编辑治理`：先看编辑影响提示，再进入产品编辑表单。

## 6. 组件边界

推荐拆分如下：

- `ProductBusinessWorkbenchDrawer.vue`
  - 统一抽屉壳层、头部、视图切换、事件转发
- `ProductDetailWorkbench.vue`
  - 继续承接 `经营总览`
- `ProductModelDesignerWorkspace.vue`
  - 从原 `ProductModelDesignerDrawer.vue` 中抽出可嵌入的治理视图
- `ProductDeviceListWorkspace.vue`
  - 从原 `DeviceListDrawer.vue` 中抽出可嵌入的关联设备视图
- `ProductEditWorkspace.vue`
  - 承接产品编辑治理表单与编辑提醒

允许保留原 `ProductModelDesignerDrawer.vue`、`DeviceListDrawer.vue` 作为兼容包装层，但 `/products` 页面不再直接拼装四套独立抽屉。

## 7. 数据与交互规则

- 统一工作台打开时仍优先使用列表摘要秒开。
- 如缺完整详情，继续后台补齐产品详情。
- `物模型治理` 首次进入时加载物模型数据，后续视图切换复用已加载结果。
- `关联设备` 首次进入时加载设备分页，后续视图切换复用结果并支持主动刷新。
- `编辑治理` 使用当前产品详情初始化表单；保存成功后：
  - 更新列表缓存
  - 更新工作台头部信息
  - 如仍满足当前筛选条件，局部刷新当前列表数据

## 8. 测试目标

本轮至少覆盖：

1. 四个入口都打开同一个统一工作台组件。
2. `详情 / 物模型 / 查看设备 / 编辑` 会以不同默认视图进入同一工作台。
3. `/products` 页面不再直接挂载四套独立抽屉壳。
4. 工作台头部会在不同视图下持续保留同一产品经营摘要。
5. 编辑保存后能同步更新列表与工作台当前产品上下文。

## 10. 风险与控制

- 风险：重构把现有已验证的物模型治理和设备列表能力打散。
  - 控制：先以测试锁定四个入口共用同一抽屉，再抽内层工作区，不先动后端契约。
- 风险：统一工作台后内容过满，退化为“多块白卡并排”。
  - 控制：每个视图先有判断层，再进入明细；不堆第二套 Hero 概况卡。
- 风险：编辑视图与新增产品表单重复，后续再分叉。
  - 控制：优先抽离可复用的产品表单区块或共用字段结构，避免同域两套表单长期漂移。
