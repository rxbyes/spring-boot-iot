# 治理控制面集中调度与领域工作台执行设计

## 1. 目标

把平台治理相关动作正式收口为两层职责：

- 治理控制面负责统一发现、聚合、排序、分派、复盘和审计治理任务。
- 产品定义中心、风险对象中心、阈值策略等领域工作台继续负责各自真相源的专业编辑与执行。

首批落地范围只覆盖两个高频入口：

- `待发布合同`
- `待绑定风险点`

## 2. 当前问题

当前 `/governance-task` 已经能统一查看 `待治理产品 / 待发布合同 / 待绑定风险点 / 待补阈值策略 / 待补联动预案 / 待运营复盘` 六类待办，但仍然停留在“读任务、改状态、做复盘”的阶段。

实际执行仍需要人工再去判断：

- `待发布合同` 应该回到 `/products` 的哪个工作区。
- `待绑定风险点` 应该回到 `/risk-point` 的哪个执行入口。

这会带来三个长期问题：

1. 控制面只有“任务台”而没有“调度台”语义。
2. 领域工作台与控制面之间缺少标准上下文协议。
3. 后续做智能化治理时，AI 只能给出提示，不能稳定把人送到正确执行位置。

## 3. 设计原则

### 3.1 控制面负责调度，不吞掉领域真相源

控制面不直接改写产品契约字段、风险点正式绑定等领域数据；控制面只负责：

- 统一生成工作项
- 聚合上下文
- 给出执行入口
- 记录审计和处理状态

### 3.2 领域页负责执行，不重复承载全局调度

产品定义中心继续负责：

- 产品主数据
- 契约字段 compare/apply
- 合同发布批次
- 风险指标目录相关读取与影响分析

风险对象中心继续负责：

- 风险点档案
- 正式绑定维护
- pending 治理转正

### 3.3 控制面到领域页必须使用显式上下文跳转

后续所有控制面调度动作都应使用“显式 query 上下文 + 领域页自动打开对应工作台”的方式，而不是只把用户送回列表页后再手工点击。

## 4. 目标态交互

### 4.1 待发布合同

用户从 `/governance-task` 看到 `待发布合同` 后，点击执行动作，系统直接跳转到：

- `/products`
- 自动聚焦对应 `productId`
- 自动打开产品经营工作台
- 自动切到 `契约字段`

### 4.2 待绑定风险点

用户从 `/governance-task` 看到 `待绑定风险点` 后，点击执行动作，系统直接跳转到：

- `/risk-point`
- 自动聚焦对应 `riskPointId`，若缺失则回退到 `keyword`
- 自动打开 `待治理转正` 或 `维护绑定` 对应入口

首批默认优先打开 `待治理转正`。原因是 `待绑定风险点` 的主语义是 pending 治理缺口，优先收口 pending 比直接进入正式绑定更符合控制面“缺口处理”语义。

## 5. 首批上下文协议

### 5.1 控制面到产品页

控制面跳转产品页时，统一使用：

- `openProductId`
- `workbenchView=models`
- `governanceSource=task`
- `workItemCode`

### 5.2 控制面到风险对象页

控制面跳转风险对象页时，统一使用：

- `openRiskPointId`
- `bindingAction=pending-promotion`
- `keyword`
- `governanceSource=task`
- `workItemCode`

如果 work item 暂时缺少 `riskPointId`，允许先按 `keyword` 回退；但后续工作项模型应优先补齐强标识，减少模糊查询依赖。

## 6. 第一批改造范围

首批只做四类变更：

1. `/governance-task` 增加“去处理”分流动作。
2. `/products` 支持路由上下文自动打开产品经营工作台指定页签。
3. `/risk-point` 支持路由上下文自动打开 pending 转正入口。
4. 文档明确“控制面调度、领域页执行”的责任边界。

## 7. 非目标

本轮不做：

- 新增独立“风险绑定工作台”一级路由
- 改写 `risk_point_device`、`iot_product_model` 等领域写接口
- 把领域执行逻辑搬进 `/governance-task`
- 做 AI 自动执行或自动审批

## 8. 后续演进

在首批标准跳转稳定后，再继续推进：

1. 治理任务台支持按 work item 类型展示更精确的执行按钮文案。
2. 风险对象中心把 `维护绑定 / 待治理转正` 收口为统一风险绑定工作台。
3. 控制面统一输出 AI 建议、证据摘要、置信度、影响面和回滚提示。
4. 控制面形成 `发现 -> 分派 -> 执行 -> 审批 -> 回滚 -> 复盘` 闭环。
