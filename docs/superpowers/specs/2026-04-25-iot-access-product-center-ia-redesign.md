# 接入智维与产品定义中心信息架构重设计

> 日期：2026-04-25
> 状态：已在会话内确认方案，待用户审阅书面 spec
> 适用仓库：`spring-boot-iot`
> 适用模块：`spring-boot-iot-ui`、菜单初始化数据、相关文档与验收口径
> 目标：重新划清 `接入智维`、`无代码接入台`、`协议治理工作台` 与 `产品定义中心` 的职责边界，消除“一级菜单抽离后仍依赖产品上下文导致页面报错”的结构性问题。

## 1. 背景

当前前端已经形成以下稳定事实：

1. `接入智维` 作为一级工作台，已经汇聚 `无代码接入台`、`产品定义中心`、`协议治理工作台`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台` 与 `数据校验台`。
2. `/device-onboarding` 已经落地为“接入案例 + 当前步骤 + 阻塞原因 + 下一步动作”的编排页，并且会继续跳转到 `/protocol-governance` 与 `/products/:productId/contracts`。
3. `/products` 已经固定为产品定义中心主入口，产品详情子路由已经拆为 `/products/:productId/overview`、`/devices`、`/contracts`、`/mapping-rules`、`/releases` 五段。
4. `/protocol-governance` 已经形成协议族、解密档案、协议模板的共享治理工作台。

与此同时，也已经暴露出新的结构性问题：

1. 接入智维中“提炼出来的一级页面”里，有一部分展示和交互仍然直接依赖产品上下文。
2. 这些页面被当作平级入口展示，但内部组件仍默认要求 `productId`、产品总览摘要、发布批次或正式字段上下文存在。
3. 结果是：菜单结构表达的是“平级工作台”，实际数据依赖却是“编排页 -> 产品上下文执行页”，导致页面报错、职责混叠和用户认知混乱。

本轮设计不再围绕“还能不能继续提炼更多一级菜单”展开，而是要回答一个更根本的问题：哪些页面本质上适合独立，哪些页面必须留在产品上下文里。

## 2. 用户确认后的约束

本轮已在会话中确认以下设计取舍：

1. 采用本次会话里已经确认的推荐取舍：`产品定义中心` 继续作为所有产品上下文执行面的主入口；对应下文方案比较中的 `4.2 方案 B`。
2. `无代码接入台` 保留一级菜单，但定位为“编排与分诊页”，不再直接承接完整产品治理执行面。
3. `协议治理工作台` 保留一级菜单，因为它治理的是跨产品共享协议资产。
4. 不再继续把 `契约字段`、`映射规则`、`版本台账` 一类强依赖 `productId` 的页面，抽成无上下文一级执行页。
5. 若必须从接入智维直接进入产品执行面，必须通过显式跳转进入 `/products/:productId/*`，而不是在接入智维内部直接挂载产品治理主体组件。

额外假设：

- 在用户未额外要求驾驶舱化改造的前提下，`/products` 首屏继续保持“产品列表主入口 + 工作台进入动作增强”的形态，不新增独立的产品总控首页。

## 3. 目标与非目标

### 3.1 本轮目标

1. 让用户在菜单上就能理解“谁是编排页，谁是执行页，谁是共享治理页”。
2. 让所有带产品上下文的治理动作统一回到 `产品定义中心 -> 产品工作台` 内完成。
3. 让 `无代码接入台` 能直接回答“当前卡在哪一步、下一步该去哪一页处理”，而不是自己尝试承载全部治理内容。
4. 消除因缺少 `productId` 或产品摘要上下文而引发的前端报错和空白页。
5. 保持现有五大工作台一级结构不被推翻，只做边界收口和职责重排。

### 3.2 本轮非目标

1. 不重做五大工作台一级导航框架。
2. 不重构 `协议治理工作台` 的领域模型或审批链。
3. 不把 `无代码接入台` 升级为新的产品治理平台。
4. 不新增第二套产品治理真相、快照体系或独立路由体系。
5. 不把 `产品定义中心` 改造成完全脱离列表的运营驾驶舱。

## 4. 方案比较

### 4.1 方案 A：继续抽离一级菜单，但补一层产品选择壳

做法：

1. 保持当前被提炼出去的入口形态。
2. 对每个依赖产品的页面，新增“先选产品/补上下文”的壳层，再进入实际执行页。

优点：

1. 一级菜单看起来更扁平，直达感更强。
2. 不需要明显收回已经暴露出去的页面语义。

缺点：

1. 产品选择会在多个页面重复出现，体验割裂。
2. 页面平级关系仍然与真实数据依赖不一致。
3. 实现复杂度明显升高，需要为多个页面补充上下文选择、缓存和回跳逻辑。

结论：

- 不采用。

### 4.2 方案 B：按“编排 / 共享治理 / 产品上下文执行”重新分层，推荐

做法：

1. `无代码接入台` 只负责接入案例、当前步骤、阻塞原因、下一步动作和快捷跳转。
2. `协议治理工作台` 继续负责协议族、解密档案、协议模板等共享治理资产。
3. `产品定义中心` 继续承接所有依赖 `productId` 的执行面，例如 `contracts`、`mapping-rules`、`releases`。
4. 接入智维中的提炼页只保留真正独立的工作台；依赖产品上下文的内容只保留入口和摘要，不直接挂完整执行组件。

优点：

1. 与现有代码和业务真相最一致。
2. 可以一次性解决当前报错和职责错位。
3. 用户认知最稳定：先分诊，再跳到正确的执行页。

缺点：

1. 产品相关能力不会被进一步“铺平”为一堆一级工作台。
2. 需要重新梳理部分菜单文案、卡片说明和跳转逻辑。

结论：

- 采用本方案。

### 4.3 方案 C：把提炼出去的页面大范围收回产品定义中心

做法：

1. 只保留很少的一级入口。
2. 大部分接入智维中与产品有关的入口都回收到 `/products`。

优点：

1. 结构最稳。
2. 上下文最统一。

缺点：

1. 会把 `接入智维` 做得过轻，弱化接入主链路的价值。
2. 不符合当前已经形成的五大工作台分工。

结论：

- 不采用。

## 5. 设计结论

本轮固定采用“编排 / 共享治理 / 产品上下文执行”三层分工：

1. `无代码接入台`：编排层
2. `协议治理工作台`：共享治理层
3. `产品定义中心 -> 产品工作台`：产品上下文执行层

该结论对应的核心规则只有一条：

> 凡是需要 `productId`、产品总览、正式字段、映射规则台账、发布批次或产品设备关系才能成立的页面，都必须留在 `/products/:productId/*` 体系中执行，不再作为无上下文一级执行页存在。

## 6. 菜单与路由边界设计

### 6.1 一级菜单语义

`接入智维` 下继续保留以下主入口：

1. `无代码接入台`
2. `产品定义中心`
3. `协议治理工作台`
4. `设备资产中心`
5. `链路验证中心`
6. `异常观测台`
7. `链路追踪台`
8. `数据校验台`

但这些入口不再被视为同一层职责的“并列执行页”，而是按以下语义理解：

1. `无代码接入台`：回答“现在去哪”
2. `协议治理工作台`：回答“共享协议资产怎么配”
3. `产品定义中心`：回答“和某个产品有关的治理动作去哪做”

### 6.2 产品执行路由真相

以下路由继续作为产品执行真相，不再从语义上外提为独立工作台：

1. `/products/:productId/overview`
2. `/products/:productId/devices`
3. `/products/:productId/contracts`
4. `/products/:productId/mapping-rules`
5. `/products/:productId/releases`

这些路由继续挂在 `产品定义中心` 下，并保留“动态详情工作区页面、隐藏但可授权”的权限表达。

### 6.3 禁止事项

本轮实施后，以下设计不再允许继续扩大：

1. 在接入智维一级菜单下新增“契约字段工作台”“映射规则工作台”“版本台账工作台”一类无上下文页面。
2. 在 `/device-onboarding` 内直接挂载 `ProductModelDesignerWorkspace` 一类产品执行主体组件。
3. 通过复制产品工作区组件的方式，再造一套接入智维内部的产品治理页面。

## 7. 页面职责重设计

### 7.1 无代码接入台

`/device-onboarding` 固定收口为“接入编排页”，页面职责限定为：

1. 维护接入案例与模板包。
2. 展示当前步骤、阻塞状态和失败原因。
3. 回显已绑定的协议资产、产品、发布批次和验收设备编码。
4. 针对不同阻塞状态，提供明确的下一步动作跳转。
5. 在案例维度触发验收或进入结果页。

它不再承接以下职责：

1. 不直接编辑产品契约字段。
2. 不直接编辑映射规则台账。
3. 不直接展示产品发布台账主体内容。

如果某条案例已经绑定产品，只允许展示“关联产品摘要”和“进入产品工作台”的动作，例如：

1. 进入产品总览
2. 进入契约字段
3. 进入映射规则
4. 进入版本台账

### 7.2 产品定义中心

`/products` 继续作为产品治理主入口，但页面文案和行为要更明确表达它不仅是列表页，而是“产品上下文执行中心的入口”。

首屏策略固定为：

1. 继续保留产品列表、搜索、增删改、导出等主档动作。
2. 强化“进入产品工作台”的动作语义。
3. 治理提示、待发布提醒、对象洞察关联提示等，继续把用户导向具体产品详情子路由。

不新增一套脱离产品列表的“产品总控首页”。

### 7.3 产品工作台

`/products/:productId/*` 固定承接以下职责：

1. `overview`：产品档案、活跃度和总览摘要
2. `devices`：关联设备清单与最近上报
3. `contracts`：样本输入、识别结果、本次生效与当前已生效字段
4. `mapping-rules`：映射规则、运行态名称/单位治理与相关台账
5. `releases`：发布批次、回滚试算和跨批次差异

这些页面是“进入产品后执行治理动作”的唯一正式入口。

### 7.4 协议治理工作台

`/protocol-governance` 继续保持独立。它治理的是共享协议资产，因此不要求先进入某个产品。

但页面说明要继续强调：

1. 协议族、解密档案、协议模板在这里治理。
2. 厂商字段映射规则、产品合同发布与版本台账仍在产品工作台内处理。

## 8. 上下文与跳转规则

### 8.1 编排页到执行页

`无代码接入台` 中的下一步动作，按以下规则跳转：

1. 缺协议族 / 解密档案 / 模板：跳 `/protocol-governance`
2. 已有协议资产但未绑定产品：跳 `/products`
3. 已绑定产品但未发布合同：跳 `/products/:productId/contracts`
4. 已存在发布批次但缺验收设备编码：继续停留案例页并提示补齐
5. 已触发验收且存在 `runId`：跳 `/automation-results?runId=...`

### 8.2 产品上下文缺失时的行为

对于任何产品执行路由，若出现以下情况：

1. `productId` 缺失
2. `productId` 非法
3. 产品详情加载失败或产品不存在

页面不应再只停留在“无法打开工作区”的原始错误文案上，而应统一降级为：

1. 明确说明缺少有效产品上下文
2. 提供返回 `/products` 的主动作
3. 若存在来源页信息，可保留返回来源入口的次动作

实现上可通过路由守卫、页面壳层兜底或统一上下文守卫组件达成，但不改变产品详情子路由作为真相入口的事实。

### 8.3 被提炼页面的处理原则

如果当前存在已经提炼出去、但内部仍依赖产品上下文的页面，本轮统一按以下三类处理：

1. **真正独立**：保留为一级页面
2. **只需摘要与跳转**：降级为入口页或摘要页
3. **完全依赖产品执行上下文**：回收到 `/products/:productId/*`

## 9. 菜单与初始化数据影响

本轮不推翻已有五大工作台和菜单种子基线，但要调整表达方式：

1. 保留 `/device-onboarding`、`/products`、`/protocol-governance` 的菜单节点。
2. 保留 `/products/:productId/*` 作为隐藏但可授权的深链页面节点。
3. 不新增与 `contracts`、`mapping-rules`、`releases` 对应的一级菜单页面节点。
4. `sectionWorkspaces.ts` 中 `接入智维` 卡片描述要改为更清晰的分工表达：
   - `无代码接入台`：接入编排
   - `产品定义中心`：产品上下文治理
   - `协议治理工作台`：共享协议治理

如果后续发现菜单初始化数据中已经出现“无上下文产品执行页”的独立节点，应在实施中回收或隐藏，而不是继续补前端适配。

## 10. 错误处理与兼容策略

1. 现有旧深链如果最终仍能映射到 `/products/:productId/*`，则继续兼容跳转，不额外新增第二套路由。
2. 对于浏览器直接打开的失效入口，优先跳回 `/products` 或对应编排页，而不是保留原地报错。
3. 对 `无代码接入台` 中没有绑定产品但用户尝试进入产品执行页的情况，直接引导去 `/products` 先建立或选择产品，不做隐式猜测。

## 11. 验证

本轮实施后，至少需要覆盖以下验证：

1. 菜单与工作台总览中，`无代码接入台 / 产品定义中心 / 协议治理工作台` 的文案与职责一致。
2. 从 `无代码接入台` 进入产品相关动作时，均通过显式跳转进入 `/products` 或 `/products/:productId/*`。
3. 不存在接入智维内部直接渲染产品执行主体组件但缺少产品上下文的报错路径。
4. 打开无效产品详情路由时，页面提供友好降级和返回动作，而不是停留在原始技术错误文案。
5. 菜单初始化数据与权限树中，产品详情子路由仍可授权，但不会作为新的一级菜单挤入侧边栏。

## 12. 文档同步

实施阶段至少同步更新：

1. `docs/02-业务功能与流程说明.md`
2. `docs/19-第四阶段交付边界与复验进展.md`
3. `docs/21-业务功能清单与验收标准.md`
4. `README.md` 与 `AGENTS.md`：检查是否需要同步当前信息架构口径；若一级工作台结构未变、只调整职责边界，可记录为无需修改

## 13. 风险与约束

1. 如果只改菜单文案、不改页面职责和跳转，当前报错不会真正消失。
2. 如果为了“菜单扁平”继续把产品执行页往外提，会持续放大上下文依赖问题。
3. 如果把 `无代码接入台` 做得过重，后续会和 `产品定义中心`、`协议治理工作台` 形成新的职责重叠。
4. 当前工作区存在外部 `.pyc` 修改，本轮实施只纳入本任务文件，不回滚无关变更。

## 14. 实施建议顺序

推荐实施顺序固定为：

1. 先调整菜单、卡片文案和页面说明，明确三层职责
2. 再处理 `无代码接入台` 中所有产品相关动作的跳转边界
3. 再补产品上下文缺失时的统一降级与回跳体验
4. 最后再同步菜单 seed、权限基线和文档说明

这样可以先把认知边界收紧，再消掉具体报错点，最后再做初始化数据与文档收口。
