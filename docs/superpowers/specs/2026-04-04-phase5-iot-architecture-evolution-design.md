# Phase 5 IoT 架构演进设计说明

> 日期：2026-04-04
> 主题：面向“采集器挂载多子设备 + 物模型治理 + 高并发接入”的下一阶段架构演进
> 目标：在保持当前模块化单体基线和真实环境验收口径不破坏的前提下，把系统从“设备接入平台”升级为“关系驱动的 IoT 接入治理平台”

## 1. 背景

当前 `spring-boot-iot` 已具备长期稳定的接入主链路：

`INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> TELEMETRY_PERSIST -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE`

当前平台已能处理以下两类设备形态：

1. 单设备多能力一体机
   - 同一设备同时上报多个逻辑测点，如倾角、加速度、裂缝。
2. 采集器挂载多个子设备
   - 父设备作为采集型遥测终端或采集中枢上报多个逻辑通道数据。

最近已确认的典型场景包括：

1. `SK11E80D1307426AZ`
   - 单设备多能力一体机，不拆父子设备。
2. `SK00EA0D1307986`
   - 采集型遥测终端，作为中枢上报 `L1_LF_1 ~ L1_LF_9` 九组裂缝子设备数据。
3. 深部位移基准站
   - 父设备按 `L1_SW_n` 拆多个真实子设备。

当前系统已经迈出第一步：

1. 协议层可以按 `iot.device.sub-device-mappings` 把父报文拆成多个 `childMessages`。
2. 物模型治理已支持：
   - 规范证据
   - 运行期证据
   - 手动补录证据
   - compare / apply

但当前仍存在三个架构级问题：

1. 父子关系主要仍由协议适配层临时承担，缺少正式“设备关系模型”。
2. 物模型治理对“单设备一体机”和“采集器挂子设备”缺少统一的关系感知过滤。
3. 高并发下，latest、状态更新、风险分发和自动候选提炼会被放大，容易让脏数据与热点写入一起恶化。

## 2. 设计目标

本轮设计目标不是简单增加几种设备适配，而是为 Phase 5 定义一个可持续扩展的接入与治理底座。

核心目标：

1. 支持“父采集器 + 多子设备”成为正式一等建模模式，而不是设备特判。
2. 支持单设备多能力和父子挂载两种形态长期共存。
3. 把协议层、设备主数据、物模型治理、风险绑定、报表统计统一到同一套关系语义上。
4. 在不破坏当前模块化单体基线的前提下，为高并发和易接入留出架构扩展点。

非目标：

1. 本轮不引入全量微服务拆分。
2. 本轮不改变 `spring-boot-iot-admin` 作为唯一启动模块的基线。
3. 本轮不把持久化逻辑下沉到协议适配层。
4. 本轮不直接把所有治理流程自动化为“自动入正式模型”。

## 3. 核心判断

### 3.1 当前系统的真实瓶颈

当前瓶颈不是“协议写得不够多”，而是“设备关系没有成为正式模型”。

表现为：

1. 协议层已经理解父子关系，但这种理解停留在运行期拆分。
2. 物模型治理仍主要围绕字段名、样本和 compare 行，不真正理解“这个字段属于父设备还是子设备”。
3. 风险绑定和规则判级虽然已经开始偏向正式测点，但接入与治理仍有把父设备逻辑点直接暴露给业务消费的风险。

### 3.2 必须从“字段驱动”升级为“关系驱动”

下一阶段应把平台升级为：

1. 关系驱动
   - 先识别父设备、逻辑通道、子设备、状态归属，再做拆分、治理、绑定。
2. 模板驱动
   - 厂家接入依赖协议模板、关系模板和字段归一模板，而不是持续写分支逻辑。
3. 治理驱动
   - 原始证据、候选证据、正式模型分层隔离，避免脏数据直接进入正式 compare 和 apply。

## 4. 目标架构

建议将系统内部目标架构明确拆为三面：

1. 控制面
   - 产品、设备、设备关系、协议模板、关系模板、物模型正式基线、规则和报表配置。
2. 数据面
   - 上报接入、解码、归一、父子拆分、latest、telemetry、状态更新、风险分发。
3. 治理面
   - 原始证据、候选证据、正式模型、人工审核与 apply。

### 4.1 控制面

控制面负责“定义规则”，不是“消费高吞吐数据”。

主要对象：

1. 产品
2. 设备
3. 设备关系
4. 协议模板
5. 关系模板
6. 正式物模型
7. 风险绑定与规则配置

控制面要求：

1. 强一致
2. 可审计
3. 可回放
4. 变更频率低于数据面

### 4.2 数据面

数据面负责“执行规则”。

目标链路建议抽象为：

1. 接收原始报文
2. 解码
3. 字段归一
4. 关系路由
5. 父子拆分
6. latest 写入
7. telemetry 写入
8. 状态写入
9. 风险分发

数据面要求：

1. 幂等
2. 可分区
3. 可批量
4. 可异步
5. 支持热点隔离

### 4.3 治理面

治理面负责“把运行期观测变成正式平台契约”，不是直接消费原始字段。

建议分三层：

1. `raw evidence`
   - 原始 message log、latest 叶子字段、真实上报样本。
2. `candidate`
   - 经过关系过滤、别名归一、证据打分后的候选字段。
3. `formal`
   - 人工确认并 apply 到 `iot_product_model` 的正式模型。

治理面要求：

1. 原始证据可追溯
2. 候选可解释
3. 正式模型唯一
4. 候选与正式解耦

## 5. 设备关系主数据模型

下一阶段最关键的新增基础设施不是协议，而是“设备关系主数据”。

### 5.1 为什么要有正式关系模型

当前 `iot.device.sub-device-mappings` 适合作为运行期短期权威，但不适合作为长期架构中心，因为它：

1. 缺少审计信息
2. 缺少业务语义
3. 缺少产品角色信息
4. 难以被治理界面、规则引擎和报表统一消费

### 5.2 建议最小关系实体

建议引入正式关系模型，至少表达以下语义：

1. `parentDeviceCode`
2. `logicalChannelCode`
3. `childDeviceCode`
4. `childProductKey`
5. `relationType`
   - `collector_child`
   - `gateway_child`
   - `single_device_multi_metric`
6. `canonicalizationStrategy`
   - 例如：
     - `L1_LF_* -> value`
     - `L1_SW_* -> dispsX/dispsY`
7. `statusMirrorStrategy`
   - 例如：
     - `S1_ZT_1.sensor_state.L1_LF_n -> child.sensor_state`
8. `enabled`
9. `version`

### 5.3 设备关系的责任边界

设备关系模型负责回答以下问题：

1. 这个逻辑点属于父设备自身，还是属于子设备？
2. 父报文里的状态要不要镜像到子设备？
3. 子设备的正式主监测字段应该叫原始逻辑码，还是 canonical 字段？
4. 风险绑定和报表应该绑定父设备还是子设备？

## 6. 三层标识模型

为了避免“协议字段名”和“业务测点名”互相污染，建议把物模型标识拆成三层。

### 6.1 原始标识

`rawIdentifier`

示例：

1. `L1_LF_1`
2. `L1_SW_1.dispsX`
3. `singal_NB`

用途：

1. 保存厂家原始报文字段痕迹
2. 便于协议回放和证据展示

### 6.2 规范标识

`canonicalIdentifier`

示例：

1. `value`
2. `dispsX`
3. `signal_NB`

用途：

1. 成为子产品正式字段
2. 成为治理 compare 的标准主键

### 6.3 业务指标

`businessMetric`

示例：

1. 裂缝张开度
2. 顺滑方向累计变形量
3. 4G 信号强度

用途：

1. 面向业务人员展示
2. 面向规则、报表、风险点绑定展示

### 6.4 三层分离的收益

收益：

1. 协议变更不等于正式模型变更
2. 厂家字段漂移不直接污染业务语义
3. compare 行可以同时展示“原始字段”和“正式字段”

## 7. 协议模板引擎

### 7.1 当前问题

当前系统已经有可工作的父子拆分逻辑，但仍偏“设备场景特判”。

继续这样演进的风险：

1. 新增一种采集器设备就要新增一组逻辑分支。
2. 协议层会逐渐承担越来越多业务语义。
3. 回归成本会线性上升。

### 7.2 目标形态

建议把协议适配拆成三层模板：

1. 协议模板
   - 负责解码、解密、原始结构识别。
2. 关系模板
   - 负责判断父报文如何拆子设备。
3. 字段模板
   - 负责字段 canonicalize 和状态镜像。

### 7.3 建议模板能力

每个模板至少支持：

1. 逻辑通道匹配
2. 结构匹配
   - `timestamp -> scalar`
   - `timestamp -> object`
3. canonical 字段映射
4. 父状态镜像策略
5. 父字段保留/剔除策略

示例：

1. 裂缝子设备模板
   - `L1_LF_* -> child.value`
   - `S1_ZT_1.sensor_state.L1_LF_* -> child.sensor_state`
2. 深部位移子设备模板
   - `L1_SW_* -> child.dispsX / child.dispsY`
   - 可选镜像 `sensor_state`

## 8. 物模型治理流水线

### 8.1 当前问题

当前物模型治理已经有 compare / apply，但候选提炼仍以“字段被观测到”为主要入口。

这会导致：

1. 父采集器字段和子设备字段混在一起
2. 厂家原始逻辑码直接成为 compare 行
3. `other` 样本和偶发脏字段进入候选池

### 8.2 建议治理三层区

建议将治理流水线正式定义为：

1. `raw zone`
   - 所有原始报文证据、message log、latest 原始叶子字段。
2. `candidate zone`
   - 关系过滤后、字段归一后、打分后的候选。
3. `formal zone`
   - 正式 `iot_product_model`。

### 8.3 Candidate 过滤规则

建议新增统一候选过滤器，供手动提炼与自动提炼共用：

1. 关系过滤
   - 父产品只保留状态字段
   - 子产品只保留子设备正式监测字段和必要子状态
2. 别名归一
   - 例如：
     - `singal_NB -> signal_NB`
     - `singal_db -> signal_db`
3. 样本分层
   - `other` 样本默认 `needsReview`
4. 证据阈值
   - 只出现一次且无 message log 证据的字段默认降权
5. 可疑命名
   - `codex_verify_*`
   - 未知前缀
   - 临时调试字段

### 8.4 Compare 的目标状态

compare 行不再只是“发现了哪些字段”，而是要明确表达：

1. 来源于父设备还是子设备
2. 来源于原始字段还是 canonical 字段
3. 是否命中规范预设
4. 是否属于正式父产品或正式子产品

## 9. 高并发演进方向

### 9.1 架构原则

下一阶段不建议先做“全量微服务”，而应先优化高频热路径。

原则：

1. 控制面继续单体化
2. 数据面内部先异步化、批量化、分区化

### 9.2 建议的高并发治理点

1. latest 合并写
   - 同一设备或同一子设备短时间内多条写入合并。
2. telemetry 批量写
   - 同批上报按设备或分区批量 flush。
3. 状态去抖
   - 父状态和子状态按时间窗去抖。
4. 风险分发异步化
   - 属性持久化成功后异步投递风险判级。
5. 幂等键
   - 原始消息指纹 + 设备 + 时间窗。

### 9.3 长期目标

若后续上报规模继续上升，数据面可进一步演进为内部事件流：

1. `decoded-message`
2. `normalized-message`
3. `child-routed-message`
4. `latest-write-task`
5. `telemetry-write-task`
6. `risk-dispatch-task`

但这一步是长期目标，不作为第一阶段必做项。

## 10. 易接入设备演进方向

理想的设备接入不应持续依赖研发逐台读报文、手工补 if/else。

建议目标能力：

1. 上传样本 JSON 或真实报文
2. 系统识别：
   - 单设备一体机
   - 采集器挂子设备
   - 网关代子设备
3. 系统匹配：
   - 关系模板
   - 规范预设
   - 字段别名
4. 生成：
   - 父产品候选
   - 子产品候选
   - 关系候选
5. 人工 review 后 apply

这样“接入新厂家”的主过程将从“研发开发流程”升级为“治理工作台流程”。

## 11. 模块边界建议

### 11.1 `spring-boot-iot-device`

负责：

1. 设备主数据
2. 设备关系主数据
3. 产品正式模型
4. candidate / formal 治理流程
5. 风险绑定消费口径

### 11.2 `spring-boot-iot-protocol`

负责：

1. 解码
2. 解密
3. 模板执行
4. 字段归一
5. 父子拆分

不负责：

1. 正式模型治理判定
2. 业务规则
3. 持久化

### 11.3 `spring-boot-iot-message`

负责：

1. 入口编排
2. Pipeline 调度
3. 上下行桥接

### 11.4 `spring-boot-iot-telemetry`

负责：

1. latest
2. 历史 telemetry
3. 聚合
4. 批量写与读路由

### 11.5 `spring-boot-iot-alarm` / `spring-boot-iot-report`

统一消费：

1. canonical metric
2. 子设备正式测点

避免继续直接消费父设备原始逻辑点。

## 12. Phase 5 分阶段实施顺序

### 12.1 第一阶段：关系主数据化

目标：

1. 把父子关系从环境配置升级为正式主数据。
2. 明确父产品、子产品和关系模板的基本结构。

产出：

1. 关系实体设计
2. 设备关系管理接口
3. 运行期读取关系主数据

### 12.2 第二阶段：协议模板化

目标：

1. 将裂缝、深部位移等父子拆分逻辑抽成模板。
2. 保留当前固定 Pipeline，不改对外契约。

产出：

1. 模板执行框架
2. 裂缝模板
3. 深部位移模板

### 12.3 第三阶段：治理流水线化

目标：

1. 引入 raw / candidate / formal 分层。
2. 手动提炼与自动提炼共用同一候选过滤器。

产出：

1. candidate 过滤器
2. compare 增强
3. 父产品 / 子产品治理视角区分

### 12.4 第四阶段：数据面高并发优化

目标：

1. latest 合并写
2. telemetry 批量写
3. 状态去抖
4. 风险异步分发

产出：

1. 幂等策略
2. 批量 flush 策略
3. 热点隔离策略

## 13. 风险与约束

### 13.1 风险

1. 若关系主数据没有先落地，后续模板和治理仍会继续特判。
2. 若继续扩大自动提炼而不先做过滤，候选脏数据会越来越多。
3. 若过早全量微服务化，会在复杂度上先失败，而不是在吞吐上先成功。

### 13.2 约束

1. 不破坏当前模块化单体边界。
2. 不破坏当前真实环境验收主链路。
3. 不把业务判断塞回协议持久化层。
4. 不把“自动候选”直接等价成“正式模型”。

## 14. 结论

Phase 5 的核心不应定义为“再接更多设备”，而应定义为：

`把 spring-boot-iot 从设备接入平台升级为关系驱动的 IoT 接入治理平台。`

这个升级的关键不在于继续补多少协议，而在于先补：

1. 设备关系主数据
2. 三层标识模型
3. 协议模板引擎
4. 候选治理流水线

只有这四层建立起来，扩展性、高并发、易接入和治理质量才能同时提升，而不是分别打补丁。
