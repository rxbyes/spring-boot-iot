# `/system-log` 异常排查分组可展开主表设计

## 1. 背景

当前 `/system-log` 的 `异常排查` 已经从旧版摘要条切到“异常概览 + 关联异常明细”的双层联动结构，数据关系是清楚的，但页面感受仍然像两块内容上下拼接：

1. 用户先看到一张异常簇概览表。
2. 再看到另一张明细表。
3. 虽然两者联动，但视觉上仍然像“第一部分”和“第二部分”。

这次优化的目标不是继续压缩间距或改标题，而是把“判断哪类异常最值得看”与“进入真实异常记录排查”合成同一张主表。

## 2. 目标与非目标

### 2.1 目标

1. 把 `异常概览` 与 `关联异常明细` 合并成单一主表。
2. 保留异常簇聚合视角，避免回到“一上来就是海量明细”的旧问题。
3. 保留现有排查能力：详情、证据、追踪、更多、分页、筛选。
4. 保持 `观测热点 / 归档治理` 页签不受影响。

### 2.2 非目标

1. 不重做 `/audit-log` 业务审计模式。
2. 不新增新的异常真相表或新的详情接口体系。
3. 不把系统模式三个页签合并成一张超级表。

## 3. 结论

采用“分组可展开主表”方案：

1. 主表一行代表一个异常簇，而不是一条异常日志。
2. 用户展开某一行后，在该行下方直接看到该异常簇对应的真实异常明细。
3. 页面成功态不再出现独立的第二张明细表。

这意味着 `异常排查` 的主工作区从“双表联动”收口为“单表分组下钻”。

## 4. 信息结构

### 4.1 主表层

主表继续使用异常簇聚合结果 `clusterRows`，一行代表以下维度组合：

1. `operationModule`
2. `exceptionClass`
3. `errorCode`

主表列调整为：

1. `展开`
2. `异常簇`
   - 主标题：`operationModule`
   - 副信息：`exceptionClass`
   - 标签：`errorCode`
3. `发生次数`
4. `影响 Trace`
5. `影响设备`
6. `最近发生时间`
7. `最近入口`
8. `最近异常摘要`

### 4.2 展开层

展开后的内容不再是第二块完整工作台，而是该异常簇的行内详情区：

1. 顶部显示当前异常簇摘要。
2. 中间显示该异常簇对应的真实异常明细表。
3. 底部显示该异常簇明细分页。

同一时刻只允许一个异常簇展开，避免页面变成长串折叠面板。

## 5. 交互设计

### 5.1 默认状态

1. 页面加载异常簇后，默认自动展开第一条异常簇。
2. 自动请求该异常簇的明细列表。
3. 若当前筛选变更后原展开项仍存在，优先保持原展开项。
4. 若原展开项不存在，则回退为展开新的第一条。

### 5.2 展开与切换

1. 点击异常簇行的展开控件，展开该行明细。
2. 展开新行时自动收起旧行。
3. 收起当前展开行后，主表回到只展示聚合视图。

### 5.3 筛选与分页

1. 顶部筛选仍然作用于异常簇查询与异常明细查询。
2. 行内分页只作用于当前展开异常簇的明细。
3. 切换异常簇时，明细分页回到第一页。

## 6. 回退与异常态

### 6.1 概览为空

当 `clusterRows` 为空时：

1. 主表区域显示“当前筛选条件下没有异常簇”。
2. 不展示旧的残留明细。

### 6.2 概览接口失败

当异常簇接口失败时：

1. 顶部显示 inline warning。
2. 工作区降级为旧式异常明细表，继续允许用户排查全部异常明细。
3. 提示文案保持“异常概览加载失败，已回退为全部异常明细”。

这个降级保留当前已经存在的可用性兜底，不因为合并视图让系统更脆弱。

## 7. 前端实现边界

### 7.1 `AuditLogView.vue`

继续负责：

1. 筛选状态
2. `clusterRows / selectedClusterKey / detailClusterMode`
3. 当前展开异常簇的明细请求
4. 降级到全部异常明细的异常处理

### 7.2 `AuditLogErrorTabPanel.vue`

从“双区块面板”调整为“主表 + 行内展开详情”：

1. 正常态渲染一张异常簇主表。
2. 通过展开行承接真实异常明细。
3. 仅在异常簇接口失败时回退为独立明细表。

## 8. 后端与接口

本次不新增后端接口，继续复用现有实现：

1. `GET /api/system/audit-log/system-error/clusters/page`
2. `GET /api/system/audit-log/page`

后端已经提供的聚合字段足以支撑主表，不需要额外改真相层。

3. 收起展开行时主表仅保留聚合层。

## 10. 预期结果

完成后，异常观测台中的异常排查会从“上概览、下明细”的两段式结构，变成“一张按异常簇组织、可直接展开真实记录”的主表：

1. 首屏更像一个完整工作台，而不是两块拼接内容。
2. 用户仍然可以先判断异常簇，再进入真实日志。
3. 详情、证据、追踪等排查动作保持原样可用。
