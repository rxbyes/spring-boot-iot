# 产品契约客户视角工作台进阶设计

**Date:** 2026-04-05  
**Status:** Approved in-session for spec drafting  
**Audience:** 产品定义中心 / 前端实现 / 设备中心 / 文档维护  
**Scope:** 在不新增页面、不改后端 compare/apply 契约、不改变 `/products` 路由结构的前提下，把现有产品经营页、契约字段工作区与物模型双证据治理抽屉升级为更贴近客户决策语境的工作台。

## 1. 背景

当前 `/products` 已具备产品经营抽屉、契约字段工作区和物模型双证据治理抽屉，后端也已经补齐：

1. 规范证据优先治理与通用双证据治理并存。
2. compare 结果已经能承载 runtime 证据、正式模型基线和协议模板解释。
3. apply 已能回传会话级治理回执。

但从客户视角看，当前页面仍存在几个明显问题：

1. 首屏更像“治理流程介绍”，没有先回答“当前契约是否健康”“下一步该做什么”。
2. 关键术语仍以研发语境为主，例如 `compare`、`apply`、`规范预设`、`双证据治理`。
3. 对比结果优先按技术结构组织，而不是按客户动作和风险组织。
4. 协议模板证据虽已很强，但仍偏技术说明，没有转化为“为什么可信”的客户解释。
5. apply 前后缺少足够明确的结果感，客户不容易判断“确认应用”会发生什么。

因此本轮目标不是继续堆叠功能，而是把现有能力重排成“客户能看懂、敢决策、愿意确认”的契约经营工作台。

## 2. 设计目标

本轮服务以下目标：

1. 让客户进入产品经营页后，第一眼看到的是契约状态、风险和建议动作，而不是内部流程术语。
2. 让契约字段工作区从“流程说明页”升级为“待核对字段 / 正式字段”的双视角工作区。
3. 让治理抽屉遵循清晰的三步流：选择核对范围、查看待处理字段、确认生效契约。
4. 让 compare 行优先回答“这条字段该怎么处理、为什么这样建议”。
5. 让协议模板证据默认以业务化解释呈现，技术细节按需折叠展开。
6. 保持现有 API、compare/apply 主流程、组件体系和 `/products` 路由不变。

## 3. 非目标

本轮明确不做：

1. 不新增产品治理独立页面或新路由。
2. 不调整后端 compare/apply 数据契约。
3. 不新增契约健康统计接口或新的数据库持久化结构。
4. 不把属性 / 事件 / 服务的基础能力从现有治理链路中拆出。
5. 不改为营销页或偏宣传型视觉风格。

## 4. 方案对比与选型

### 4.1 方案 A：仅优化文案

做法：

1. 保持现有页面结构不变。
2. 仅替换标题、副标题、按钮文案和空态文案。

优点：

1. 改动最小。
2. 回归风险低。

缺点：

1. 客户仍需自己理解复杂流程。
2. 对比结果和治理抽屉的信息负担没有实质下降。

### 4.2 方案 B：信息架构升级，推荐

做法：

1. 保持当前组件和路由边界不变。
2. 重新组织产品经营页头部、契约字段工作区、治理抽屉和 compare 行的阅读顺序。
3. 用客户动作和风险分组替代纯技术分组。

优点：

1. 改动集中在前端，风险可控。
2. 能直接消费已有 compare、模板证据和 apply 回执数据。
3. 客户感知提升明显。

缺点：

1. 需要联动更新多个组件和测试。

### 4.3 方案 C：新增契约健康总台

做法：

1. 新增经营驾驶舱式页面。
2. 再将现有治理抽屉挂成二级工作台。

优点：

1. 最终呈现可能更完整。

缺点：

1. 范围扩张明显。
2. 容易触达路由、工作台编排和新接口诉求。
3. 不适合当前这一轮快速进阶。

### 4.4 选型

本轮采用 `方案 B：信息架构升级`。

原因：

1. 现有后端能力已经足够支撑客户化表达。
2. 不必额外扩散到新页面或新接口。
3. 更符合当前前端治理规则中“复用现有工作台结构、避免第二套页面语法”的要求。

## 5. 总体设计

### 5.1 产品经营页头部升级为“经营总览 + 契约状态”

落点：

1. `ProductBusinessWorkbenchDrawer.vue`

设计：

1. 顶部继续保留产品名称、产品 key、协议、节点类型和数据格式。
2. 首屏重点从“核心规模”转为“当前契约状态 + 下一步建议”。
3. 头部摘要要优先回答：
   - 当前是否已有正式契约
   - 当前是否已有运行设备可供核对
   - 当前更适合补齐契约还是先维护正式字段
4. 关联设备总量继续保留，但降级为辅助经营指标，不再独占首屏唯一强视觉。

建议文案方向：

1. 有运行设备时：`当前已有运行设备，可继续补齐并核对契约字段。`
2. 有设备但无在线覆盖时：`当前已有接入设备，建议优先核对契约基线。`
3. 无设备时：`产品档案已建立，待首批设备接入后再补齐契约字段。`

### 5.2 契约字段工作区改成“双视角”

落点：

1. `ProductModelDesignerWorkspace.vue`

设计：

1. 当前 `证据比对 / 正式字段` 语义改为：
   - `待核对字段`
   - `正式字段`
2. `待核对字段` 首屏先给出状态卡，不再先展示长流程说明。
3. 状态卡建议固定为：
   - 已正式生效
   - 可直接纳入
   - 待人工确认
   - 继续观察
4. 主 CTA 根据状态动态切换：
   - 有运行设备但还未开始核对：`开始补齐契约`
   - 已有 compare 结果或正在处理：`继续核对字段`
   - 没有运行设备：`先查看正式字段`
5. `正式字段` 视图继续保留当前类型切换和卡片结构，但标题和说明改为结果语境，例如：
   - 标题：`当前已生效契约`
   - 说明：`这里只展示已经正式生效的字段契约，新增或修订仍需回到待核对字段完成确认。`

### 5.3 治理抽屉改成三步式决策流

落点：

1. `ProductModelDesignerDrawer.vue`

设计：

1. 抽屉整体节奏改成：
   - 第一步：选择核对范围
   - 第二步：核对字段结果
   - 第三步：确认生效
2. 第一步保留现有能力，但客户语言化：
   - `规范优先核对`
   - `通用核对`
   - `样本 JSON 辅助核对`
   - `自动提炼：开启/关闭`
   - `人工补充字段`
3. 第二步不再只是“对比工作区”，而是“待你处理的字段清单”。
4. 第三步强化 apply 前结果感：
   - 本次将新增多少项
   - 本次将修订多少项
   - 哪些字段将在确认后正式生效
5. 底部主按钮文案从 `确认应用` 改成 `确认并生效契约`。
6. apply 后回执保留，但语言收口为“结果确认卡”，强调：
   - 本次已生效多少字段
   - 哪些字段已纳入
   - 若命中模板证据，展示自动识别来源摘要

### 5.4 compare 行按客户动作分组

落点：

1. `ProductModelGovernanceCompareTable.vue`

设计：

1. compare 行的首层组织从“先按属性 / 事件 / 服务切换”调整为“先按动作分组”：
   - 可直接纳入
   - 建议人工确认
   - 继续观察
   - 存在冲突
2. 属性 / 事件 / 服务继续保留，但下沉为分组内筛选维度，而不是首屏主导航。
3. 每一行要优先给出：
   - 字段名
   - 中文业务名
   - 当前建议动作
   - 一句原因说明
4. compareStatus 和 riskFlags 继续存在，但前端优先翻译成客户能理解的结论和风险提示。

建议风险话术：

1. `名称或定义可能不一致，建议人工确认`
2. `当前只有单侧证据，建议继续观察`
3. `该字段已存在正式契约，本轮不建议重复纳入`
4. `该字段更像状态字段，不建议按主监测值处理`

### 5.5 模板证据升级为“可信解释卡”

落点：

1. `ProductModelGovernanceCompareTable.vue`
2. `ProductModelDesignerDrawer.vue` 的 apply 回执区

设计：

1. 默认先展示业务化解释：
   - 自动识别来源
   - 子设备样本依据
   - 归一后的正式口径
2. 技术细节折叠展示：
   - 模板编码
   - 逻辑通道
   - canonicalization 策略
   - 状态镜像
   - 父字段剔除
   - 重解码失败次数
3. 示例表达：
   - `该字段来自裂缝模板自动识别`
   - `已在子设备 202018143 上观测到同类上报`
   - `系统已按 LF_VALUE 统一到正式字段口径`
4. 无模板证据时不展示空白解释区，也不补“暂无模板”类干扰文案。

### 5.6 决策语义保持稳定，但文案更接近经营动作

前端按钮和标签建议如下：

1. `纳入新增`：保持不变
2. `纳入修订`：保持不变
3. `继续观察`：保持不变
4. `人工裁决`：前端文案调整为 `稍后确认`
5. `忽略`：保持不变

原因：

1. 保留现有核心动作可降低理解迁移成本。
2. 仅把最技术化的 `人工裁决` 收口为客户更自然的动作表达。

## 6. 组件与实现落点

本轮实现只调整以下文件：

1. `spring-boot-iot-ui/src/components/product/ProductBusinessWorkbenchDrawer.vue`
2. `spring-boot-iot-ui/src/components/product/ProductModelDesignerWorkspace.vue`
3. `spring-boot-iot-ui/src/components/product/ProductModelDesignerDrawer.vue`
4. `spring-boot-iot-ui/src/components/product/ProductModelGovernanceCompareTable.vue`

实现约束：

1. 不新增页面。
2. 不改路由。
3. 不新增页面私有第二套壳层或页签工作台。
4. 优先复用 `StandardDetailDrawer`、`StandardButton`、`StandardDrawerFooter` 及既有品牌 token。
5. 保持“浅背景 + 白卡 + 轻描边 + 控制台工作台”视觉语法，不回退到重营销式风格。

## 7. 数据与接口约束

本轮不调整任何后端接口。

前端直接复用现有数据：

1. 产品基础档案信息
2. 正式模型列表
3. compare summary
4. compare rows
5. runtimeCandidate / manualCandidate / formalModel
6. protocolTemplateEvidence
7. applyResult / appliedItems

前端新增的状态卡和客户化提示，均应由现有 compare / formalModel 数据推导得出，不向后端追加统计接口。

## 8. 测试策略

至少同步更新以下测试：

1. `ProductBusinessWorkbenchDrawer.test.ts`
   - 校验经营页头部出现契约状态和建议动作，不再是单一规模展示。
2. `ProductModelDesignerWorkspace.test.ts`
   - 校验 `待核对字段 / 正式字段` 新语义。
   - 校验状态卡、主 CTA 和 warning 产品空态。
3. `ProductModelDesignerDrawer.test.ts`
   - 校验三步流文案。
   - 校验“确认并生效契约”语义。
   - 校验 apply 回执的新客户化表达。
4. `ProductModelGovernanceCompareTable.test.ts`
   - 校验动作分组。
   - 校验风险话术翻译。
   - 校验模板证据业务解释和技术依据折叠。
   - 校验无模板证据时不出现空解释块。

## 10. 验收标准

本轮完成后，应满足：

1. 客户进入 `/products` 的产品经营页后，能在首屏看到契约状态和下一步建议。
2. 契约字段工作区默认首先表达“待核对字段”和“正式字段”两类客户视角。
3. 治理抽屉能按“三步流”表达核对范围、字段结果和确认生效，不再只是长页面堆叠。
4. compare 行能先回答“如何处理、为什么这样建议”。
5. 模板证据默认呈现业务化可信解释，技术细节按需展开。
6. 不新增页面，不改后端 compare/apply 契约，不破坏现有治理链路。
