# 语义契约治理与风险闭环一体化架构设计

> 日期：2026-04-06
> 状态：已在会话内确认设计方向，可进入实施计划阶段
> 适用范围：`spring-boot-iot-device`、`spring-boot-iot-protocol`、`spring-boot-iot-alarm`、`spring-boot-iot-report`、`spring-boot-iot-admin`、`/products`、风险对象与风险运营相关工作台
> 目标：在保持当前模块化单体与真实环境验收基线不破坏的前提下，把平台从“页面驱动的字段治理 + 风险配置串联”升级为“语义契约治理域 + 风险运营闭环域”协同的一体化平台

## 1. 背景

当前仓库已经形成以下稳定事实：

1. `spring-boot-iot` 仍是模块化单体，唯一启动模块为 `spring-boot-iot-admin`。
2. 设备接入主链路已经稳定为：
   - `INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> TELEMETRY_PERSIST -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE`
3. `/products` 已形成“手动样本提炼 -> compare/apply -> 写入 iot_product_model”的产品物模型治理入口。
4. `iot_device_relation` 已成为“父设备逻辑通道 -> 子设备”关系主数据入口，并已支撑 legacy `$dp` 场景下的 `collector_child + LF_VALUE + SENSOR_STATE` 关系归一。
5. 风险平台已经具备：
   - 风险点
   - 阈值策略
   - 联动编排
   - 应急预案
   - 告警/事件
   - 实时监测、GIS、对象洞察、运营分析
6. 当前风险点与 pending 治理链路已经开始围绕“正式测点”做收口，但接入治理、合同发布和风险消费之间仍未形成稳定分层。

当前体系的根本问题不是“缺少更多页面”或“缺少更多协议特判”，而是以下三类语义仍然混在一起：

1. 行业/业务规范字段
2. 厂商真实上报字段与逻辑通道
3. 风险闭环真正消费的指标语义

其结果是：

1. `iot_product_model` 同时承担了“字段治理结果”“正式合同”“风险消费入口”的多重职责。
2. 厂商字段名、`$dp` 逻辑通道名、正式字段名之间缺少稳定桥梁，风险闭环仍有被原始字段漂移拖着跑的风险。
3. 平台虽然已经能形成“风险点 -> 策略 -> 联动 -> 预案 -> 告警/事件”的链条，但该链条缺少一个稳定的“风险指标目录”作为消费边界。
4. 当前治理能力更像页面级工作流，不足以支撑未来“允许做配置、不允许改代码”的长期目标。

## 2. 已确认决策

本次会话已经确认以下架构级决策：

1. 第一阶段先做“治理优先”，不把“新厂商零代码接入”作为本期硬验收。
2. 正式字段标准以行业/业务规范为最高真相源，厂商字段只作为映射别名和证据。
3. 第一阶段样板范围固定为：
   - 裂缝直报
   - 采集终端挂载裂缝子设备
4. 风险点、阈值策略、联动编排、应急预案、告警运营以后只绑定治理后生效的规范 `identifier`，不再绑定厂商原始字段。
5. 长期目标是“允许做配置，不允许改代码接入新厂商”，但该能力必须建立在稳定治理体系之上。
6. 当前总体架构采用“方案 B：语义契约中台”，但在本项目里进一步升级为：
   - 双核心域
   - 一座桥
   - 一条控制面
   - 一层运行底座

## 3. 设计目标

本设计服务以下目标：

1. 建立一个稳定的“语义契约治理域”，使行业规范、厂商证据和正式合同不再混写。
2. 建立一个稳定的“风险运营闭环域”，使风险点到运营分析的主链只依赖正式风险指标。
3. 在两大核心域之间引入显式“风险指标语义层”，避免风险平台直接消费产品合同的全部细节。
4. 补齐治理与运营控制面，使版本、审批、发布、回滚、漂移告警、真实环境复验具备平台级入口。
5. 保持现有模块化单体边界与真实环境验收口径不破坏，不回退到 H2 或并行替代链路。
6. 为后续“配置化接入”预留扩展点，但不在第一阶段过早建设一整套零代码接入平台。

## 4. 非目标

本轮明确不做以下事情：

1. 不把项目拆成微服务。
2. 不改变 `spring-boot-iot-admin` 作为唯一启动模块的基线。
3. 不把治理逻辑重新堆回单个页面或单个 service。
4. 不让风险点直接绑定厂商原始字段或 compare 临时候选。
5. 不在第一阶段一次性纳管全部 `$dp` 地灾类型。
6. 不把“协议配置化平台”作为第一阶段主交付。
7. 不通过新增 H2 验收路径或伪造数据链路替代真实环境验收。

## 5. 总体架构

### 5.1 顶层结构

目标架构不是单域扩张，而是以下五部分协同：

1. `核心域 A：语义契约治理域`
2. `核心域 B：风险运营闭环域`
3. `桥梁层：风险指标语义层`
4. `控制面：治理与运营控制面`
5. `运行底座：接入链路、证据沉淀、可观测与审计`

### 5.2 核心域 A：语义契约治理域

语义契约治理域负责回答：

1. 哪些字段是行业/业务规范承认的正式语义。
2. 厂商真实上报字段与逻辑通道如何映射到正式语义。
3. 哪些字段已经发布为产品正式合同。

该域内部拆为四层：

1. `规范语义层`
   - 行业/业务规范字段库
   - 命名规则
   - 单位、精度、监测内容编码、监测类型编码
   - 是否允许进入风险闭环
2. `运行证据层`
   - `$dp` 原始报文
   - latest
   - 消息日志
   - 逻辑通道
   - 父子设备关系
   - 协议模板证据
3. `治理决策层`
   - 差异比对
   - 命名推荐
   - 冲突裁决
   - 缺失识别
   - 版本治理
4. `合同发布层`
   - 产品级正式合同发布
   - 发布批次、版本、回滚
   - 正式落点为 `iot_product_model`

### 5.3 核心域 B：风险运营闭环域

风险运营闭环域负责回答：

1. 哪些正式语义被纳入风险对象。
2. 它们如何形成阈值策略、联动编排与预案匹配。
3. 如何进入告警、事件、实时监测、GIS、对象洞察与运营分析。

该域内部拆为四层：

1. `风险对象层`
   - 风险点
   - 对象档案
   - 设备/测点正式绑定
2. `策略决策层`
   - 阈值策略
   - 判级策略
   - 联动编排
   - 应急预案
3. `事件闭环层`
   - 告警
   - 事件
   - 工单/处置
   - 反馈/关闭
4. `运营呈现层`
   - 实时监测
   - GIS 态势图
   - 对象洞察
   - 告警运营
   - 运营分析

### 5.4 桥梁层：风险指标语义层

这是本设计中不可省略的一层。

风险平台不应直接把 `iot_product_model` 里的所有合同字段当成风险消费对象，而应通过“风险指标目录”二次筛选和收口。该层负责定义：

1. 哪个正式字段允许进入风险闭环。
2. 它属于哪类风险指标。
3. 它的量纲、方向性、阈值类型、趋势分析方式是什么。
4. 它是否适合：
   - 风险点绑定
   - 阈值策略
   - GIS
   - 对象洞察
   - 运营分析

目标主链路因此固定为：

`规范库 -> 契约治理 -> 合同发布(iot_product_model) -> 风险指标目录 -> 风险点绑定 -> 阈值策略 -> 联动编排 -> 应急预案 -> 告警/事件 -> 实时监测/GIS/对象洞察/运营分析`

### 5.5 控制面

控制面负责跨域治理，不属于单个业务对象，但必须成为平台级能力。其职责包括：

1. 治理任务
2. 发布批次
3. 审批记录
4. 影响分析
5. 版本对比
6. 漂移问题
7. 回滚记录
8. 审计留痕
9. 覆盖率与治理经营指标

### 5.6 运行底座

运行底座继续复用现有真实接入链路，但要加强对治理和风险两大核心域的证据支撑。职责包括：

1. Topic 路由
2. 解密
3. 解码
4. 关系归一
5. latest / telemetry / message-log 证据沉淀
6. `traceId / sessionId` 复盘
7. 漂移与缺口可观测

## 6. 核心对象模型

### 6.1 规范字段对象

规范字段对象是平台的最高真相源，不依赖任何厂商字段命名。至少定义：

1. 规范 `identifier`
2. 中文名称
3. 单位
4. 精度
5. 监测内容编码
6. 监测类型编码
7. 设备族/产品族
8. 是否允许进入风险闭环
9. 是否允许趋势分析
10. 默认阈值语义
11. 展示域建议
    - 实时监测
    - GIS
    - 对象洞察
    - 运营分析

### 6.2 厂商证据与映射对象

该对象承接所有运行事实，回答“厂商真实报文里出现了什么，以及它和规范字段的关系”。至少定义：

1. 原始字段名
2. 逻辑通道编码
3. 父设备/子设备归属
4. 样例值
5. 推断值类型
6. 出现频次
7. 最后出现时间
8. 来源表/来源链路
9. 建议映射到的规范字段
10. 冲突状态与歧义说明

### 6.3 已发布合同对象

已发布合同对象是产品级正式合同，继续保留 `iot_product_model` 作为存储入口，但其职责必须收缩为“发布结果”。它不再承担：

1. 规范字段库
2. 厂商字段证据仓
3. 风险指标目录

它只承载：

1. 某产品下已发布的正式规范字段
2. 字段配置
3. 字段排序与说明
4. 当前正式版本状态

### 6.4 风险指标目录对象

风险指标目录对象是合同与风险闭环之间的桥梁层，至少定义：

1. 来源合同字段
2. 风险指标编码
3. 风险指标中文名
4. 风险类别
5. 阈值方向
6. 默认判级语义
7. 是否可用于对象洞察
8. 是否可用于 GIS
9. 是否可用于运营分析

### 6.5 风险闭环对象

风险闭环对象继续由现有风险域主体承接，包括：

1. `risk_point`
2. `risk_point_device`
3. `rule_definition`
4. `linkage_rule`
5. `emergency_plan`
6. 告警/事件/工单相关对象

关键变化是：这些对象以后只绑定风险指标目录，不再直接绑定厂商原始字段或 compare 临时候选。

### 6.6 控制面对象

控制面建议显式抽象以下对象：

1. 治理任务
2. 合同发布批次
3. 风险指标发布批次
4. 审批记录
5. 影响分析结果
6. 漂移问题单
7. 回滚记录

## 7. 现有表与推荐新对象

### 7.1 建议继续复用

1. `iot_product_model`
   - 保留为发布结果表
2. `iot_device_relation`
   - 继续做关系主数据
3. `risk_point / risk_point_device`
   - 继续做风险对象与绑定主体
4. `rule_definition / linkage_rule / emergency_plan`
   - 继续做策略与预案主体
5. `iot_device_property / iot_device_metric_latest / iot_device_message_log`
   - 继续做运行证据来源

### 7.2 建议新增或显式抽象

在不破坏现有模块化边界的前提下，建议新增或显式抽象以下对象：

1. 规范字段库
2. 厂商字段证据/映射
3. 风险指标目录
4. 治理发布/审批/回滚控制对象
5. 漂移问题与缺口对象

## 8. 模块边界

### 8.1 `spring-boot-iot-protocol`

只负责：

1. Topic 路由
2. 解密
3. 解码
4. 逻辑通道识别
5. 父子归一
6. 协议模板执行
7. 输出标准化证据

不得承担：

1. 正式合同发布
2. 风险指标判级规则
3. 风险绑定策略

### 8.2 `spring-boot-iot-device`

负责：

1. 产品
2. 设备
3. 设备关系
4. 规范字段库
5. 厂商证据与映射
6. 合同发布
7. 治理工作台

不得承担：

1. 风险点判级逻辑
2. 联动编排逻辑
3. 预案匹配逻辑

### 8.3 `spring-boot-iot-alarm`

负责：

1. 风险指标目录
2. 风险点
3. 阈值策略
4. 联动编排
5. 应急预案
6. 告警/事件/工单
7. 风险运营消费

不得反过来决定：

1. 产品合同命名
2. 厂商字段归一规则
3. 产品正式合同发布

### 8.4 `spring-boot-iot-report`

继续只负责基于正式风险指标和闭环结果做统计分析。

### 8.5 `spring-boot-iot-admin`

负责：

1. 控制面装配
2. 可观测与审计装配
3. 发布/回滚配套入口

## 9. 第一阶段实施范围

### 9.1 范围

第一阶段只做样板切片，固定覆盖：

1. 裂缝直报
2. 采集终端挂载裂缝子设备

### 9.2 第一阶段必须完成

1. 建立裂缝样板的规范字段库
2. 把 `$dp` 运行证据沉淀成“厂商字段证据/映射”
3. 让 `iot_product_model` 明确退位为“已发布合同”
4. 在风险域引入“风险指标目录”桥梁层
5. 风险点绑定改为绑定风险指标，而不是原始字段
6. 阈值策略、联动编排、预案、告警运营统一消费风险指标目录
7. 补齐发布、审计、漂移与回滚最小控制面

### 9.3 第一阶段刻意不做

1. 不纳管全部 `$dp` 类型
2. 不先做零代码接入平台
3. 不先解决所有厂商协议族
4. 不把 GIS/运营分析内部所有表现细节和字段治理耦在一起开发

## 10. 三条主链路

### 10.1 接入主链路

`Topic / 解密 / 解码 / 关系归一 / 标准化证据沉淀`

### 10.2 治理主链路

`规范库对齐 / 差异比对 / 人工裁决 / 合同发布`

### 10.3 风险主链路

`风险点绑定 / 阈值策略 / 联动编排 / 应急预案 / 告警运营`

## 11. 控制面与运维要求

### 11.1 发布控制

治理结果不得无痕写入正式合同。必须具备：

1. 发布批次
2. 操作人
3. 发布时间
4. 影响对象
5. 审批记录
6. 回滚依据

### 11.2 错误处理

以下问题不得只停留在控制台日志中：

1. 字段漂移
2. 映射冲突
3. 风险指标缺失
4. 合同发布失败
5. 风险闭环断链

建议统一形成：

1. 结构化日志
2. 可观测告警
3. 可追踪的问题记录

### 11.3 回滚

必须支持：

1. 合同发布版本快照
2. 风险指标发布版本快照
3. 按产品/按指标/按策略局部回滚

### 11.4 管理者视角指标

控制面应能直接回答：

1. 有多少产品完成规范治理
2. 哪些产品仍停留在原始字段阶段
3. 哪些风险点还未绑定正式风险指标
4. 哪些阈值/联动/预案覆盖率不足
5. 新厂商接入卡在哪一环

## 12. 验证与验收

### 12.1 验证策略

必须继续以真实环境为准，分层验证：

1. 样板报文回放验证
2. 合同发布验证
3. 风险点绑定验证
4. 阈值/联动/预案链路验证
5. 实时监测/GIS/对象洞察/运营分析消费验证

### 12.2 第一阶段验收线

第一阶段通过标准固定为：

1. 裂缝直报和采集终端裂缝子设备都能从 `$dp` 真实报文沉淀出稳定证据。
2. 规范字段、厂商字段、逻辑通道、正式合同四层角色边界清晰。
3. `iot_product_model` 只保存已发布合同。
4. 风险点不再直接绑定厂商原始字段。
5. 风险点 -> 阈值策略 -> 联动编排 -> 应急预案 -> 告警运营主链可只依赖正式风险指标跑通。
6. 全链路变更具备审计、可回滚，并能通过 `application-dev.yml` 真实环境复验。

## 13. 方案价值

该方案相对现状的核心收益是：

1. 把“页面驱动治理”升级为“平台驱动治理”。
2. 把“产品正式合同”和“风险消费语义”从直接耦合改为桥接耦合。
3. 让厂商字段漂移不再直接冲击风险闭环。
4. 为未来“允许做配置、不允许改代码”的接入模式提前打好边界和对象模型。
5. 让管理者、一线用户、维护人员、开发者都拥有稳定的系统视角，而不是继续围绕单页工作台做局部拼接。

## 14. 选型结论

本轮正式采用：

1. `双核心域`
   - 语义契约治理域
   - 风险运营闭环域
2. `一座桥`
   - 风险指标语义层
3. `一条控制面`
   - 治理与运营控制面
4. `一层运行底座`
   - 接入链路、证据沉淀、可观测与审计

后续实施必须遵守以下原则：

1. 不再把规范库、证据、发布结果继续混写到 `iot_product_model`。
2. 不再让风险点直接绑定厂商原始字段。
3. 不再把厂商特例持续堆进单个 service 的 if/else。
4. 第一阶段只做裂缝样板切片，跑通方法论后再扩展到 GNSS、深部位移及更多厂商族。
