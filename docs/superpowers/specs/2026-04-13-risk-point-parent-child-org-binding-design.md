# 风险点绑定父子组织放开设计

**Date:** 2026-04-13  
**Status:** Approved in-session for execution  
**Audience:** 风险对象中心 / 权限治理 / 后端实现 / 测试验收 / 文档维护  
**Scope:** 在不放宽现有租户与数据权限边界的前提下，把风险点正式绑定从“设备组织必须与风险点组织完全相同”调整为“非超级管理员支持同租户内父子组织绑定，超级管理员保留跨租户/跨组织绑定能力”。

## 1. 背景

当前风险绑定工作台的候选设备和正式绑定写入都依赖 `RiskPointServiceImpl` 中的同组织硬校验：

1. 设备候选列表只保留 `riskPoint.orgId == device.orgId`。
2. 正式绑定写入时再次要求 `riskPoint.orgId == device.orgId`。

这会导致一个具体问题：

1. 设备资产中心里能查到集团组织下的设备。
2. 风险点归属在分公司或子机构下。
3. 用户明明有当前租户内的可见权限，却因为“必须完全同组织”被排除在风险绑定候选之外。

用户已经明确新的业务边界：

1. 支持父子组织绑定。
2. 超级管理员仍可跨部门、跨组织、跨租户。
3. 非超级管理员无论是管理员还是普通用户，都必须继续严格遵守同租户和既有数据权限控制。

## 2. 目标

本轮只完成以下目标：

1. 风险绑定候选设备支持“同租户内、父子组织可绑定”。
2. 正式绑定写入与设备级正式绑定写入执行同一条父子组织规则。
3. 超级管理员继续保留跨租户、跨组织绑定能力。

## 3. 非目标

本轮明确不做：

1. 不修改设备资产中心、组织树、角色数据范围定义。
2. 不放宽非超级管理员的设备可见范围。
3. 不允许兄弟组织、平级跨部门在无父子关系时直接绑定。
4. 不把“设备未归属组织”视为可绑定场景。
5. 不新增接口、不新增页面参数、不新增数据库结构。

## 4. 方案比较

### 4.1 方案 A：仅把同组织改成父子组织，推荐

做法：

1. 风险点和设备继续先走现有租户/组织数据权限校验。
2. 非超级管理员在正式绑定阶段额外要求：
   - 风险点与设备属于同一租户。
   - 风险点组织与设备组织满足“同节点或父子链路”。
3. 超级管理员跳过租户与组织层级限制，但仍拒绝未归属组织的设备。

优点：

1. 正好覆盖当前真实业务诉求。
2. 不会把兄弟组织和平级跨部门一起放开。
3. 不影响设备资产中心的查询权限模型。

缺点：

1. 需要在服务层补组织链路判断。

### 4.2 方案 B：只允许父组织设备绑定到子组织风险点

优点：

1. 放开范围最小。

缺点：

1. 与“支持父子组织绑定”的口径不一致。
2. 会留下“子组织设备是否可以绑定父组织风险点”的再次争议。

### 4.3 方案 C：只要当前账号可见就允许绑定

优点：

1. 实现最简单。

缺点：

1. 会把兄弟组织、平级跨部门、其他非父子关系场景一并放开。
2. 与用户追加的权限边界冲突。

### 4.4 选型

采用 **方案 A**。

## 5. 目标态规则

### 5.1 超级管理员

1. 候选设备列表不再受组织层级和租户一致性限制。
2. 正式绑定写入不再受组织层级和租户一致性限制。
3. 但设备仍必须存在有效 `orgId`，未归属组织设备继续拒绝绑定。

### 5.2 非超级管理员

1. 风险点本身必须位于当前账号现有数据权限范围内。
2. 设备本身必须位于当前账号现有数据权限范围内。
3. 风险点与设备必须属于同一租户。
4. 风险点组织与设备组织必须满足以下任一条件：
   - 同一组织；
   - 风险点组织是设备组织的祖先；
   - 设备组织是风险点组织的祖先。
5. 兄弟组织、跨支系组织、租户不一致、无组织归属都必须拒绝。

### 5.3 候选设备列表

`GET /api/risk-point/bindable-devices/{riskPointId}` 继续复用原接口路径，但业务语义改为：

1. 先按当前账号现有设备可见范围拿设备候选。
2. 继续排除已被其他风险点占用的设备。
3. 非超级管理员只保留“同租户 + 父子组织链路”的设备。
4. 超级管理员跳过第 3 条限制。

### 5.4 正式绑定写入

以下两条写链路统一执行同一套校验：

1. `RiskPointServiceImpl.bindDeviceAndReturn(...)`
2. `RiskPointServiceImpl.bindDeviceCapabilityAndReturn(...)`

统一结果：

1. 非超级管理员必须同租户且满足父子组织链路。
2. 超级管理员允许跨租户、跨组织。
3. 一设备多风险点占用限制保持不变。

## 6. 实现设计

### 6.1 后端改动位置

核心改动只落在：

1. `spring-boot-iot-alarm/src/main/java/com/ghlzm/iot/alarm/service/impl/RiskPointServiceImpl.java`
2. `spring-boot-iot-alarm/src/test/java/com/ghlzm/iot/alarm/service/impl/RiskPointServiceImplTest.java`

### 6.2 组织链路判断

`RiskPointServiceImpl` 新增组织关系辅助逻辑：

1. 通过 `PermissionService.getDataPermissionContext(currentUserId)` 判断是否超级管理员。
2. 非超级管理员绑定时执行同租户校验。
3. 通过 `OrganizationService.getById(...)` 向上追溯 `parentId`，判断两个组织是否存在祖先/后代关系。
4. 使用当前方法内的小缓存避免重复读取同一个组织节点。

### 6.3 一致性要求

为了避免“列表能看到、提交又失败”或“列表看不到、后端其实能绑”的分裂状态：

1. 候选设备列表和正式绑定写入必须共用同一条组织层级语义。
2. 错误消息需要明确区分：
   - 未归属组织；
   - 不同租户；
   - 非父子组织。

## 7. 测试策略

本轮至少补齐以下定向用例：

1. 非超级管理员在同租户下可以绑定父组织设备到子组织风险点。
2. 非超级管理员在同租户下可以绑定子组织设备到父组织风险点。
3. 非超级管理员在同租户下不能绑定兄弟组织设备。
4. 非超级管理员不能跨租户绑定。
5. 超级管理员可以跨租户绑定。
6. 候选设备列表能保留父子组织设备，继续排除兄弟组织设备和已占用设备。
