# IoT Access Diagnostic Flow Enhancement Design

**Date:** 2026-03-28
**Status:** Draft for review
**Scope:** `接入智维` 现有紧凑工作台方案之上的第一阶段进阶增强，聚焦 `流程联动 + 诊断辅助`

## 1. 背景

当前 `接入智维总览`、`产品定义中心`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 已完成第一轮 UI 收口，已经解决了以下问题：

1. 去掉了重复母版说明、能力墙和无效大卡片。
2. 页面主战区重新回到台账、验证、追踪和校验结果本身。
3. 总览与子页的职责边界已经清楚：
   - 总览负责“先去哪”
   - 子页负责“在这里怎么做”
4. 配色、按钮、列表和抽屉语法已经重新回到系统既有 token 体系。

但当前仍有明显的第二阶段缺口：

1. **跨页联动弱。** 用户从 `/reporting` 跳到 `/message-trace`、`/system-log`、失败归档时，仍需要重新理解上下文。
2. **诊断只展示数据，不直接给结论。** 页面能看，但还不够“会带路”。
3. **最近一次排查状态不可持续。** 查询条件、最近 session、上一跳入口还没有被组织成连续工作流。
4. **资产治理与诊断工作流还没有打通。** 例如链路验证发现身份异常后，页面还不能自然把用户带回产品或设备治理。

结论：

- 第一阶段已经完成“页面结构收口”。
- 第二阶段应该把重点从“单页更整洁”升级为“跨页更顺、判断更快、动作更少”。

## 2. 目标

本轮进阶增强只做第一阶段子项目：`流程联动 + 诊断辅助`。

目标如下：

1. 让 `链路验证中心 -> 链路追踪台 -> 异常观测台 -> 失败归档/数据校验` 形成连续排查路线。
2. 在现有紧凑状态条基础上，给每个诊断页补一层“当前判断 + 下一步动作”。
3. 减少跨页重复输入，优先复用当前 `deviceCode`、`traceId`、`productKey`、`topic`、`sessionId`。
4. 保持现有配色方案和共享组件体系，不重新发明第二套设计系统。
5. 为下一阶段“资产治理评分 + 批量治理”预留接口和语义入口，但本轮不直接实现评分系统。

## 3. 非目标

本轮明确不做：

1. 不重做当前 7 个页面的整体视觉母版。
2. 不新增接入智维一级路由或第二个总览中心。
3. 不引入大模型解释、聊天式 Copilot 或自由文本 AI 总结。
4. 不在本轮同步实现“产品治理分 / 设备治理分 / 批量修复建议”。
5. 不为了诊断辅助强行新增复杂后端聚合接口；优先使用现有接口和前端推导。

## 4. 设计原则

### 4.1 先连续，再聪明

优先解决“上一页到下一页的上下文不断裂”，再解决“系统是否足够聪明地给建议”。

### 4.2 结论必须贴着主战区

诊断结论不能再做成浮在上面的独立说明墙，必须贴着当前列表、结果区或右侧诊断区出现。

### 4.3 只做规则化辅助，不做黑盒判断

页面提示应来自可解释规则，例如：

- `traceId` 缺失
- MQTT 回流等待超时
- session 已存在但时间线为空
- 请求通道与 Topic 不匹配
- 设备能查到，但产品契约信息不完整

用户必须能看懂“为什么给出这个判断”。

### 4.4 继续保持紧凑

所有增强都必须服从当前已确定的语法：

- 顶部：一句判断 + 一行状态 + 1~2 个动作
- 中部：主战区
- 下部：必要的辅战区或最近恢复区

不能因为“增加诊断能力”就重新长回一堆解释卡。

## 5. 总体方案

本轮方案分为两块：

1. **流程联动层**
2. **诊断辅助层**

二者共享同一套轻量上下文模型。

### 5.1 共享上下文模型

前端建立统一的 `DiagnosticContext` 概念，承接跨页传递的最小排查上下文：

- `sourcePage`
- `deviceCode`
- `traceId`
- `productKey`
- `topic`
- `sessionId`
- `transportMode`
- `reportStatus`
- `capturedAt`

设计原则：

1. **高价值字段进路由 query。**
   - 例如 `deviceCode`、`traceId`、`productKey`
2. **辅助字段进 sessionStorage。**
   - 例如 `sourcePage`、`reportStatus`、较长的说明文本
3. **页面打开后优先恢复上下文。**
   - 先秒开，再允许用户手动修正

### 5.2 统一“继续排查”动作

所有诊断页在顶部状态条或工具条内只保留强相关动作，例如：

- `继续链路追踪`
- `查看异常观测`
- `回看失败归档`
- `核对数据校验`
- `返回设备资产`

动作规则：

1. 只有当前页面能构成明确下一跳时才展示。
2. 下一跳必须默认带上当前上下文。
3. 不允许重新堆一整组“模块目录式按钮”。

## 6. 流程联动设计

### 6.1 链路验证中心 `/reporting`

定位不变，仍是入口型诊断页。

本轮增强：

1. 在发送成功后，把当前上下文写入统一 `DiagnosticContext`。
2. 若是 HTTP：
   - 直接把 `traceId` 作为主联动键
3. 若是 MQTT：
   - 先以 `sessionId` 为主键
   - 绑定 trace 后自动补齐上下文
4. 诊断区新增“下一步建议”块：
   - 继续链路追踪
   - 查看异常观测
   - 打开数据校验

规则化结论示例：

- 设备身份已就绪，可直接发送
- MQTT 已发布，等待消费回流
- 已拿到 trace，可进入链路追踪
- session 已存在但时间线为空，优先查看异常观测

### 6.2 链路追踪台 `/message-trace`

定位不变，仍是追踪台账 + 最近会话辅战区。

本轮增强：

1. 顶部状态条优先展示“当前是从哪里进入的”。
2. 当存在来自 `/reporting` 的上下文时，默认带入筛选条件。
3. 最近会话区新增“恢复为当前排查上下文”的语义，不只是填表。
4. 表格行和详情抽屉内都给出下一步判断：
   - 去异常观测台
   - 去失败归档
   - 去数据校验台

规则化结论示例：

- trace 已完整，可继续查 system_error
- trace 缺失，但 session 最近仍活跃，优先恢复最近会话
- 时间线过期，仅保留消息日志，建议转异常观测或失败归档

### 6.3 异常观测台 `/system-log`

定位仍是 `system_error` 专用诊断台账。

本轮增强：

1. 当从 `链路验证中心` 或 `链路追踪台` 跳入时，优先恢复对应筛选条件。
2. 顶部状态条显示当前排查主键：
   - `traceId`
   - `deviceCode`
   - `productKey`
3. 工具条中保留反向跳转：
   - 返回链路追踪
   - 回看失败归档
4. 列表行和详情抽屉中增加“返回上一步”语义入口。

规则化结论示例：

- 当前异常集中在 MQTT consumer
- 当前异常与某设备编码高度相关
- 当前异常与某 traceId 直接相关
- 当前无异常命中，建议回到链路追踪或失败归档

### 6.4 数据校验台 `/file-debug`

定位仍是单设备校验页。

本轮增强：

1. 若从 `/reporting` 或 `/message-trace` 带入设备编码，则自动恢复当前设备。
2. 顶部状态条显示本次校验与哪一次排查相关。
3. 若文件快照和固件聚合都为空，给出回退建议：
   - 回链路追踪确认是否真实收到文件类消息
   - 回异常观测查看消费异常

规则化结论示例：

- 当前设备有快照，无聚合，优先核对 C.4
- 当前设备无快照无聚合，优先确认是否收到文件类消息
- 当前设备校验结果完整，可结束本轮排查

### 6.5 失败归档

失败归档继续依附 `/message-trace` 同路由双模式。

本轮增强：

1. 从异常观测台和链路追踪台跳入时，优先保留同一组上下文。
2. 失败详情中的契约快照区补“回产品定义中心 / 设备资产中心”的治理跳转。
3. 强化失败归档的角色定位：
   - 它不是另一个列表页
   - 它是“主链路失败后的证据页”

## 7. 诊断辅助设计

### 7.1 统一结论模型

前端建立统一 `DiagnosticFinding` 结构：

- `kind`
- `level`
- `title`
- `summary`
- `reason`
- `nextActionLabel`
- `nextActionTarget`

本轮只做规则化结论，不做自由文本总结。

### 7.2 结论分类

建议固定为 6 类：

1. `identity`
2. `contract`
3. `correlation`
4. `timeline`
5. `runtime`
6. `validation`

### 7.3 结论展示位置

统一规则：

1. **页级结论**
   - 放顶部状态条的 meta 区
2. **结果级结论**
   - 放主战区标题附近
3. **对象级结论**
   - 放行操作或详情抽屉中

禁止：

- 在主列表上方再插一层厚重“诊断报告卡”
- 在每个页面同时出现 3 套不同结论容器

## 8. 资产治理的预留接口

本轮虽然不直接做治理评分，但必须给第二阶段留钩子。

需要预留的两个动作方向：

1. 从失败归档 / 异常观测跳到 `产品定义中心`
   - 目标：看产品契约、协议、物模型是否完整
2. 从失败归档 / 异常观测跳到 `设备资产中心`
   - 目标：看设备是否已建档、是否在线、是否父子拓扑异常

这样第二阶段在做“治理评分 + 批量治理”时，不需要重做前一轮联动骨架。

## 9. 实施顺序建议

建议拆为 3 个小阶段：

### 9.1 Phase A：上下文联动

先做：

- `DiagnosticContext`
- 路由 query + sessionStorage 恢复
- 各页“继续排查”动作

这是收益最高、风险最低的一层。

### 9.2 Phase B：规则化诊断结论

再做：

- `DiagnosticFinding`
- 各页页级/结果级判断
- 统一的下一步建议

这一层会让接入智维从“好用”升级到“会带路”。

### 9.3 Phase C：治理回跳预留

最后做：

- 从失败归档 / 异常观测跳到产品、设备治理页
- 在治理页接收来源上下文

这样可以为下一份 spec 的“治理评分与批量治理”打底。

## 10. 验收标准

满足以下条件即可判定本轮达标：

1. 用户从 `/reporting` 进入 `/message-trace`、`/system-log`、`/file-debug` 时，不需要重复填写核心上下文。
2. 所有诊断页顶部都能给出一句当前判断或明确状态。
3. 至少三类典型场景有明确的下一步建议：
   - MQTT 等待回流
   - 时间线过期
   - system_error 命中/未命中
4. 页面结构仍保持当前紧凑语法，没有重新长回大卡片和重复说明。
5. 所有联动动作都继续复用现有系统配色、共享按钮和共享列表骨架。

## 11. 本轮后的下一份规划

本 spec 完成后，下一份独立 spec 再进入：

- `产品定义中心 / 设备资产中心` 的治理评分
- 批量治理入口
- 待处理队列
- 从诊断页回到治理页后的“带问题进入治理”

该部分应作为第二子项目单独设计，不与本轮“流程联动 + 诊断辅助”混做。
