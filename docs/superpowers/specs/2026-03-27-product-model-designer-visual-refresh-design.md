# Product Model Designer Visual Refresh Design

**Date:** 2026-03-27
**Status:** Draft for review
**Scope:** `/products` 内产品物模型设计器抽屉的布局重构、视觉强化、响应式收口与交互秩序优化

## 1. 背景

当前产品物模型设计器已经具备“候选提炼 + 正式模型”双模式能力，但页面仍存在以下问题：

1. 首屏层级偏平，主信息、主操作和模式切换没有形成明确焦点。
2. 候选提炼模式采用“三栏并排 + 普通卡片堆叠”，阅读节奏松散，目录、候选卡和确认面板之间缺少主次关系。
3. 正式模型模式相对简化，和候选提炼模式不在同一套视觉语言下，切换后有“进入另一张页面”的割裂感。
4. 卡片信息结构不够细，标识、说明、证据、治理动作等语义混在一起，导致页面“不整洁、不协调”。
5. 当前视觉虽然没有明显脱离主系统，但“工作台感”和“设计器感”都不够强，无法支撑大工作区抽屉的场景。

## 2. 目标

本轮设计优化目标：

1. 将物模型设计器升级为“大工作台抽屉”，形成更强的页面舞台感。
2. 保持与主系统现有品牌配色一致，继续使用品牌橙、云蓝和蓝灰正文体系，不引入新私有主色。
3. 让候选提炼模式成为首屏主视觉焦点，同时保证正式模型模式也拥有完整、稳定的结构秩序。
4. 强化卡片结构，让用户一眼区分“身份信息、指标、证据、说明、动作”。
5. 统一候选提炼卡与正式模型卡的视觉骨架，降低模式切换的割裂感。
6. 在大尺寸桌面抽屉内提供更充足的横向空间，同时保证平板和移动端可自然降级。

## 3. 非目标

1. 本轮不改变候选提炼与正式模型的业务语义。
2. 本轮不新增一级路由，仍继续复用 `/products` 内同一抽屉。
3. 本轮不引入新的 UI 框架或新的全局主题体系。
4. 本轮不重构产品物模型表单抽屉的字段规则，只同步必要的视觉对齐。

## 4. 设计方向结论

在“控制台运营台风格”“编辑器工作台风格”“视觉更强的展示型风格”三种方向中，本轮采用：

- **主方向：视觉更强的展示型风格**
- **约束：保留治理台信息密度和工作台可扫描性**
- **推荐布局：双模式舞台 + 统一资产卡**

原因：

1. 用户明确希望页面更整洁、更协调，同时接受更强的视觉表现。
2. 纯展示型瀑布布局会削弱治理效率，不适合大量候选和结构化证据。
3. “双模式舞台 + 统一资产卡”可以兼顾视觉张力和治理秩序，适合作为长期可维护方案。

## 5. 总体结构

设计器抽屉统一升级为四层结构：

1. **舞台头部层**
2. **总览指标层**
3. **主体工作区层**
4. **底部操作与附属抽屉层**

### 5.1 抽屉尺寸

- 当前 `StandardDetailDrawer` 默认 `size` 为 `48rem`。
- 本组件本轮显式改为 `72rem` 工作台宽度，不再使用默认宽度。

设计原则：

1. 候选提炼模式必须允许目录、候选主列表、确认面板三者同时可读。
2. 正式模型模式必须允许“类型概览条 + 模型卡列表”在一屏内形成完整秩序。

## 6. 舞台头部层

舞台头部是本轮最重要的视觉入口，不再只是普通 hero 卡，而是承担：

- 页面身份
- 产品身份
- 模式说明
- 主操作入口

### 6.1 结构

头部拆成左右两区：

#### 左区：产品与设计器身份

- 小眉标：`产品物模型设计器`
- 主标题：根据当前模式动态表达
  - 候选提炼：`基于真实上报提炼产品契约`
  - 正式模型：`统一维护产品正式物模型`
- 副标题：说明“不新增一级路由，继续复用产品定义中心”
- 产品身份带：
  - 产品名称
  - `productKey`
  - 协议编码
  - 节点类型

#### 右区：操作台

- 模式切换器
- 主操作按钮
  - 候选提炼模式：`重新提炼`
  - 正式模型模式：`新增属性模型 / 新增事件模型 / 新增服务模型`
- 保留一条轻提示：
  - 候选提炼模式：说明“仅提炼真实证据”
  - 正式模型模式：说明“维护已确认契约”

### 6.2 视觉

头部采用比普通详情面板更强的舞台感：

1. 继续使用品牌橙和云蓝的弱化 radial / linear 混合底纹。
2. 产品身份带采用浅底 pill / small card 形式，形成“资产标识条”。
3. 模式切换器不再像普通 tab，而是做成更完整的 segmented control。
4. 主按钮和模式切换器要形成一个紧凑、整洁的操作组。

## 7. 总览指标层

当前四张几乎同权重的 summary card 将调整为“1 主 + 3 辅”的节奏。

### 7.1 布局

- 左侧一张主引导卡，占比更大。
- 右侧三张紧凑指标卡，对应：
  - 属性
  - 事件
  - 服务

### 7.2 主引导卡内容

候选提炼模式下：

- 标题：`真实证据概览`
- 主文案：`先提炼，再确认，再沉淀为正式物模型`
- 副文案：强调属性来自 `iot_device_property` / `iot_device_message_log`，事件和服务需要真实证据
- 辅助信息：
  - 最近提炼时间
  - 当前待确认数量

正式模型模式下：

- 标题：`正式契约总览`
- 主文案：`按属性、事件、服务治理已确认物模型`
- 副文案：强调正式模型是产品契约权威来源
- 辅助信息：
  - 当前类型数量
  - 当前主操作提示

### 7.3 紧凑指标卡内容

每张卡统一包含：

- 类型标签
- 数量
- 一行说明
- 弱强调状态

候选提炼模式：

- 属性：候选数 / 原始证据数
- 事件：候选数 / 事件提示
- 服务：候选数 / 服务提示

正式模型模式：

- 属性：正式属性模型数
- 事件：正式事件模型数
- 服务：正式服务模型数

## 8. 主体工作区

### 8.1 候选提炼模式

继续保留三段结构，但要重构主次关系。

#### 8.1.1 左侧目录侧栏

职责：

- 仅负责分类切换
- 显示每类候选数量和待确认数量
- 作为轻量导航，不承担复杂说明

结构：

- 顶部：`候选目录`
- 中部：分类导航卡
- 底部：一块轻量规则提示

视觉：

- 做成更窄、更轻的导航面板
- 激活项具有更明显的背景和左侧高亮条
- 数字用更强的 badge 或 compact metric 表达

#### 8.1.2 中央候选主列表

这是候选提炼模式的主舞台。

每张候选卡统一拆成四段：

1. **标题区**
   - 模型名称
   - 标识符
   - 采纳复选框
   - 类型 / 分组 / 状态 badge

2. **摘要区**
   - 数据类型
   - 置信度
   - 最近上报时间
   - 来源表

3. **编辑区**
   - 建议名称输入
   - 中文备注输入
   - 将置信度信息与证据统计做成独立证据卡

4. **说明区**
   - 完整备注
   - 边界和规范说明

优化原则：

1. 候选卡要更像“可治理对象卡”，而不是简单表单块。
2. 标题区和编辑区之间必须有清晰分隔。
3. 证据和来源信息集中收口，不再散落在多处。

#### 8.1.3 右侧确认面板

确认面板从普通信息堆叠升级为“治理摘要面板”。

分成三块：

1. **写入摘要**
   - 拟新增
   - 待人工确认
   - 冲突回执

2. **规则提示**
   - 事件提示
   - 服务提示
   - 若无真实证据，明确展示空态说明

3. **主操作**
   - 确认写入按钮
   - 如果无选中项，按钮置灰并保持视觉占位

要求：

1. 指标区采用更规整的小卡栅格，而不是普通三块堆叠。
2. 提示说明保持轻量，不要让右侧变成长文栏。

### 8.2 正式模型模式

正式模型模式不再只是“类型切换条 + 列表卡”，而是也进入同一套“舞台工作区”语言。

#### 8.2.1 顶部类型概览条

类型切换从普通 chip 升级为“分类资产条”。

每个类型块包含：

- 类型名
- 数量
- 一行治理说明

激活项表现：

- 更明显的背景层次
- 更明确的激活状态
- 与候选模式目录项保持同宗语法

#### 8.2.2 模型主列表

正式模型卡统一拆成四段：

1. **标题区**
   - 模型名称
   - 标识符
   - 必填 / 选填 tag

2. **摘要标签区**
   - 排序
   - 数据类型 / 事件类型 / 服务输入输出摘要

3. **说明区**
   - 描述
   - 如果无描述，显示当前类型的空说明引导

4. **动作区**
   - 编辑
   - 删除

要求：

1. 与候选卡共享标题区和摘要区的视觉骨架。
2. 正式模型卡比候选卡更克制，避免两边都一样重。

#### 8.2.3 空状态

空状态必须分模式表达：

- 候选模式：`当前目录暂无候选`
- 正式模型模式：`暂无物模型`

空态视觉统一到同一张轻面板中，并带一句引导说明：

- 候选模式引导用户切换类型或重新提炼
- 正式模型模式引导用户新增当前类型模型

## 9. 配色与视觉语法

本轮继续完全匹配主系统配色，不引入新私有主色。

### 9.1 配色原则

1. 主操作、激活态、重要强调继续使用品牌橙系 `--brand`
2. 次级说明、辅助提示继续复用云蓝 / 蓝灰体系
3. 卡片背景继续保持白卡 + 浅冷底色 + 轻描边 + 轻阴影
4. 不新增高饱和紫色、深蓝大底或营销化渐变头

### 9.2 视觉强化方式

允许增强的部分：

1. 头部舞台底纹
2. 主引导卡背景层次
3. 激活态 segmented / type card / nav item
4. 候选卡标题区与证据区的分层

不允许增强的部分：

1. 整页大面积高饱和渐变
2. 大量写死颜色字面量
3. 单页引入与主系统冲突的视觉情绪

### 9.3 设计 token 约束

优先使用：

- `--brand`
- `--accent`
- `--panel-border`
- `--text-heading`
- `--text-caption`
- `--text-caption-2`
- `--shadow-*`
- `--radius-*`

若确实缺少 token，再补 `tokens.css`，不要在组件内散写一组新颜色常量。

## 10. 响应式策略

### 10.1 桌面端

- 保持大工作台抽屉
- 候选提炼模式采用三列：
  - 窄导航
  - 宽主列表
  - 中等确认面板

### 10.2 平板端

- 总览指标层改为两行
- 主体工作区改为“主列表优先”，确认面板下移
- 导航仍保留，但宽度收窄

### 10.3 移动端

- 头部纵向堆叠
- 模式切换和主操作整行排布
- 总览指标改为单列
- 主体工作区改为单列堆叠
- 候选卡内部编辑区改为单列
- 正式模型类型切换改为单列卡片

要求：

1. 移动端不能出现难以阅读的三列压缩。
2. 主操作和模式切换在窄屏仍然清晰可点击。

## 11. 与现有共享组件的关系

本轮继续复用：

- `StandardDetailDrawer`
- `StandardFormDrawer`
- `StandardDrawerFooter`
- `StandardButton`
- `StandardActionLink`
- `StandardRowActions`

允许在本组件内部新增页面私有包装层，但不新增第二套共享抽屉体系。

边界：

1. 若 `StandardDetailDrawer` 仅需调整 `size`，直接在本组件传参，不修改共享抽屉实现。
2. 本轮不抽离新的共享 detail-panel 语法，只在当前组件内完成布局优化。

## 12. 文案与信息语义

本轮文案要求：

1. 保持中文一致性，不引入无必要中英混排。
2. 强调“真实证据、候选提炼、正式契约、边界规范”等治理语义。
3. 正式模型模式不再显得像备用页面，而是产品契约的权威维护区。

## 13. 测试与验收

### 13.1 自动化验证

至少覆盖：

1. 既有 `ProductModelDesignerDrawer` 单测继续通过。
2. 候选提炼接口失败时仍能降级到正式模型模式的回归测试继续通过。
3. `ProductWorkbenchView` 相关视图测试继续通过。
4. 前端 `build` 通过。

### 13.2 人工验收重点

1. `/products` 打开物模型设计器后，首屏舞台感明显增强。
2. 候选提炼模式下，目录、主列表、确认面板的主次关系清晰。
3. 正式模型模式下，分类概览和模型卡的节奏明显优于当前版本。
4. 配色与主系统一致，没有产生“另一个产品”的感觉。
5. 窄屏下不会出现三列挤压导致的布局失衡。

## 14. 实施范围

本轮实施只做以下内容：

1. 调整设计器抽屉尺寸
2. 重构舞台头部层
3. 重构总览指标层
4. 重构候选提炼主体布局
5. 重构正式模型主体布局
6. 收口候选卡与正式模型卡的视觉骨架
7. 更新前端治理文档与业务文档中关于该页面结构的描述

本轮不做：

1. 新增复杂动画
2. 重写表单抽屉交互
3. 改动候选提炼与正式模型的业务规则
