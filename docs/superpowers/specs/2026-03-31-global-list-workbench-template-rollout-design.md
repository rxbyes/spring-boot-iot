# 全站列表工作台模板统一与分批迁移设计

> 日期：2026-03-31
> 范围：`spring-boot-iot-ui`
> 主题：把 `产品定义中心` 已验证的列表工作台语法上升为全站模板，并按业务域分批迁移所有带列表的模块
> 状态：设计已确认，待书面 spec 审阅

## 1. 背景

近期 `产品定义中心` 已率先形成一套较稳定的高频台账工作台语法：

- `StandardPageShell + StandardWorkbenchPanel` 的页级节奏基本稳定
- 查询区、已生效筛选条、工具条、结果区和分页区已经形成清晰层级
- 桌面表格与移动卡片拥有同一阅读顺序
- 行内操作已收口为共享动作语法，并接入自适应操作列宽度

但仓库当前仍存在一个更大的问题：带列表的模块已经广泛分布在接入侧、风险域、平台治理和特殊工作台中，尽管大部分页面已经部分接入共享组件，但视觉密度、列表节奏和操作区语法仍存在明显分叉：

- 同类列表页仍在使用不同的间距、字号和表头亮度
- 同类工具条在元信息密度、动作收口和按钮节奏上还不完全一致
- 同类操作列和移动卡片仍有页面私有语法残留
- 如果只按单页微调推进，列表体系会一直停留在“局部统一、全站分叉”的状态

因此，本轮目标不再是单页优化，而是把 `产品定义中心` 已验证的列表工作台语法，上升为全站可复用的“高频台账模板”，并分批迁移所有带列表的模块。

## 2. 目标

- 把 `产品定义中心` 的列表工作台语法提炼为全站模板，而不是继续做页面局部微调。
- 统一全站带列表模块的页壳节奏、筛选区、工具条、结果区、分页区和操作区设计语言。
- 统一间距、字号、操作内容功能间距、按钮尺寸、移动卡片密度和操作列宽度策略。
- 保留不同业务域的字段差异、状态差异和业务动作差异，不把所有列表页做成一个模板复制品。
- 形成“共享模板 + 分批迁移 + 持续文档治理”的长期机制，而不是一次性的视觉清理。

## 3. 非目标

- 不把所有页面一次性重构为同一大组件。
- 不在本轮统一详情抽屉、表单抽屉、导入/替换等业务流程。
- 不修改 API、权限编码、分页协议和后端字段口径。
- 不覆盖纯表单页、GIS 主舞台、驾驶舱型首页和非台账主视图。
- 不把所有工作台正文都强行改造成列表页语法，只处理真正属于列表主工作台的部分。

## 4. 用户已确认方向

本轮已经在对话中确认以下方向：

- 把范围从 `设备资产中心` 扩大到“所有带列表的模块”。
- 采用“总模板 + 分批实施”的路线，而不是一次性全站混改。
- 优先级固定为：
  - 接入侧高频台账
  - 风险与处置台账
  - 平台治理台账
  - 剩余特殊工作台列表区
- 实施时统一的是“列表工作台语法”，不是把整个页面连业务流程一起重构。

## 5. 全站模板边界

### 5.1 全站统一的内容

- 页级骨架：
  - `StandardPageShell`
  - `StandardWorkbenchPanel`
- 列表主流程：
  - 筛选区
  - 已生效筛选条
  - 工具条
  - 结果区
  - 分页区
- 视觉节奏：
  - 标题与说明层级
  - 区块间距
  - 表头亮度
  - 结果区底板
  - `loading / empty / list` 语法
- 列表交互：
  - 桌面表格主视图
  - 移动卡片补充视图
  - 自适应断点
- 行内操作：
  - 共享动作语法
  - 操作列自适应宽度
  - 双功能紧凑间距
  - 多功能统一收口到 `更多操作`
- 字体与密度：
  - 标题字号
  - 字段字号
  - meta 标签
  - 按钮尺寸
  - 操作区按钮间距

### 5.2 明确不统一的内容

- 各业务域字段口径
- 各业务域状态标签语义
- 详情抽屉与表单抽屉结构
- 导入、替换、派发、关闭等业务流程
- 非列表型主舞台

### 5.3 统一原则

全站带列表页面进入同一套工作台系统，但不被压成同一页面复制品。

允许保留：

- 字段数量不同
- 状态标签不同
- 筛选项数量不同
- 操作项数量不同

不再允许：

- 同类列表页使用两套不同的间距和字号
- 同类工具条出现不同的动作节奏
- 同类操作列出现不同的留白和宽度策略
- 同类移动卡片出现不同的标题、字段和 meta 语法

## 6. 分批实施范围

### 6.1 第一批：接入侧高频台账

第一批作为模板来源和首批验收对象，优先覆盖高频、强验收和强对比页面：

- `ProductWorkbenchView`
- `DeviceWorkbenchView`
- `MessageTraceView`
- `AuditLogView`
- 失败归档相关列表组件
- `链路验证中心`、`数据校验台` 中真正属于台账的列表区

目标：

- 把最核心的“间距、字号、操作列间距、列表节奏”统一到稳定基线
- 让接入侧列表成为后续批次的模板来源

### 6.2 第二批：风险与处置台账

第二批覆盖结构相近、状态更复杂的风险域列表页：

- `AlarmCenterView`
- `EventDisposalView`
- `RiskPointView`
- `RuleDefinitionView`
- `LinkageRuleView`
- `EmergencyPlanView`

目标：

- 验证模板对复杂状态表达与治理动作的适配能力
- 统一风险域的列表节奏，不改业务流程

### 6.3 第三批：平台治理台账

第三批覆盖标准 CRUD 和治理型列表页：

- `OrganizationView`
- `UserView`
- `RoleView`
- `RegionView`
- `DictView`
- `ChannelView`
- `MenuView`
- `HelpDocView`
- `InAppMessageView`
- `AuditLogView` 中治理态列表

目标：

- 批量消灭页面私有列表骨架
- 让平台治理域进入同一套列表模板系统

### 6.4 第四批：特殊工作台列表区

第四批只处理真正属于列表区的部分，不强行覆盖整个页面：

- `RealTimeMonitoringView` 中的台账区
- `DeviceInsightView` 中的列表区
- `AutomationTestCenterView` 中的列表区
- 报表、复盘和其他工作台中的清单区

目标：

- 只把“属于列表工作台”的部分纳入模板
- 避免正文主舞台和列表语法互相污染

## 7. 实施规则

### 7.1 先改共享层，再改页面层

每一批都先收口共享语法，再落具体页面。

共享层负责：

- 页壳节奏
- 列表骨架
- 结果区底板
- `loading / empty / list` 语法
- 表格和移动卡片密度
- 操作列间距和宽度策略

页面层负责：

- 字段
- 筛选项
- 状态口径
- 业务动作
- 详情/表单入口

### 7.2 只改列表主工作台

每一批只处理：

- 筛选区
- 已生效筛选条
- 工具条
- 列表结果区
- 分页
- 桌面操作列
- 移动卡片操作区

本轮不处理：

- 详情抽屉结构
- 表单抽屉结构
- 新增/编辑/导入/替换流程
- API、权限、路由和后端字段

### 7.3 统一规则优先级

所有带列表页面统一按以下优先级处理：

1. 共享组件与共享 token
2. 共享全局列表样式
3. 页面业务差异
4. 页面 scoped CSS 的极少量补充

禁止再次回到“先写页面私有样式，再考虑能否抽共享”的旧路径。

### 7.4 每批都做来源页和目标页联动回归

以第一批为例：

- 来源页：`ProductWorkbenchView`
- 目标页：`DeviceWorkbenchView`、`MessageTraceView` 等

改完目标页之后，必须回看来源页，避免模板来源被后续迁移反向打散。

## 8. 验收口径与完成定义

### 8.1 单批验收

每一批完成时，至少满足：

- 该批页面进入同一套列表工作台语法
- 桌面表格与移动卡片的字号、间距、操作区节奏进入统一规范
- 操作列右侧无明显大块空白
- 工具条元信息与动作区进入统一节奏
- 页面不再保留与来源页明显分叉的结果区底板和 loading 骨架
- 业务动作、字段口径和状态语义不发生回退

### 8.2 全站完成定义

全站完成不是“所有页面像同一个页面”，而是：

- 全站带列表模块都进入同一套列表工作台系统
- 不同业务域仍保留自己的字段与动作表达
- 列表页不再因为页面来源不同而出现两套间距、字号和操作语法
- 新增列表页有明确模板可继承，不再从页面私有样式起步

### 8.3 自适应要求

所有纳入模板的页面继续支持自适应模式：

- 桌面端以表格为主视图
- 移动端以卡片为补充视图
- `720px` 左右断点切换稳定
- 不新增手动“表格/卡片视图”切换
- 操作列与卡片操作区在窄屏下不出现裁切和错乱

## 9. 回归与验证

### 9.1 验收顺序

推荐固定验收顺序：

1. 第一批接入侧
2. 第二批风险域
3. 第三批平台治理
4. 第四批特殊工作台

每一批完成后都要独立回归，不把全站风险一次性叠加到最后。

### 9.2 工程验证

每一批实现完成后至少执行：

- 受影响组件/视图测试
- `npm run component:guard`
- `npm run list:guard`
- `npm run style:guard`
- `npm run build`

如某批次改动了共享样式或共享动作组件，应优先补受影响测试，再执行守卫命令。

## 10. 文档同步要求

本轮属于平台级模板治理，不是单页优化，因此必须同步更新：

- `docs/06-前端开发与CSS规范.md`
- `docs/15-前端优化与治理计划.md`
- `docs/08-变更记录与技术债清单.md`

文档需要明确记录：

- `产品定义中心` 已成为全站列表模板的主要来源页之一
- 带列表模块按业务域分批迁移，而不是单页散改
- 后续新增列表页必须优先继承共享模板，而不是从 scoped CSS 开始写第二套列表语法

## 11. 风险与控制

### 11.1 风险

- 一次性全站混改导致回归面过大
- 为了统一而过度复制来源页，导致业务域差异被压平
- 共享层改动过重，反向影响已验收页面
- 特殊工作台正文与列表区语法互相污染

### 11.2 控制

- 严格按四批推进，不做一把梭全站混改
- 统一的是列表工作台语法，不统一业务域字段与流程
- 每一批都执行来源页和目标页联动回归
- 只把真正属于列表工作台的内容纳入模板，不强行覆盖整个页面
