# 采集器与子设备状态/监测数据边界设计

## 1. 背景

当前平台已经支持采集器下挂多个子设备的复合建模，典型场景包括：

- 南方测绘激光测距：`L1_LF_* -> value / sensor_state`
- 南方测绘深部位移：`L1_SW_* -> dispsX / dispsY / sensor_state`

现有能力已经明确：

- 采集器原始报文可以同时包含采集器自身状态、子设备业务数据、子设备状态数据
- 复合设备治理会按 `iot_device_relation` 中的 `canonicalization_strategy / status_mirror_strategy` 把子设备数据归一到子产品
- 采集器自身 `temp / humidity / signal_4g` 等字段不会进入子产品 compare

当前待定问题不是“能不能解析”，而是“正式物模型与正式展示边界怎么定义”。

核心矛盾：

- 一线希望在采集器页面看到全部子设备状态和监测值
- 架构上又必须避免父设备和子设备双份建模、双份存储、双份风险绑定

## 2. 目标

本设计要解决以下问题：

1. 采集器产品是否应重复挂载子设备监测值和子设备状态
2. 单台设备与复合设备如何共用一套正式物模型口径
3. 风险、对象洞察、趋势、告警应绑定到父设备还是子设备
4. 页面如何既满足“采集器总览全部子状态”的诉求，又不破坏单一真相源

## 3. 设计原则

### 3.1 单一真相源

每个正式监测指标、正式状态指标只能归属于一个正式设备主体。

- 子设备监测值归子设备
- 子设备健康状态归子设备
- 采集器自身运行状态归采集器

不允许同一正式指标同时在父设备和子设备产品中各维护一份。

### 3.2 接入层与治理层分离

采集器原始报文可以很复杂，但正式物模型必须收敛。

- 接入层保留原始报文结构
- 治理层只保留正式业务语义
- 展示层允许做聚合视图，但不反向推导正式物模型重复建模

### 3.3 父页聚合不等于父设备拥有

“采集器详情页展示所有子设备状态/监测值”是读侧聚合能力，不代表这些字段属于采集器正式物模型。

### 3.4 单台与复合统一口径

同一类传感器，不因部署方式不同而定义两套正式物模型。

- 单台深部位移设备：`dispsX / dispsY / sensor_state`
- 复合场景下的深部位移子设备：仍然是 `dispsX / dispsY / sensor_state`

## 4. 推荐方案

推荐采用：

**接入层保留原始报文，治理层严格父子拆分，读侧按采集器聚合展示。**

即：

- 存储与正式物模型按“父子分离”设计
- 页面与运维视图按“采集器聚合”展示

这是当前平台最稳妥、后续可扩展性最好的方案。

## 5. 正式建模边界

### 5.1 采集器产品

采集器产品只承载采集器自身字段，例如：

- `temp`
- `humidity`
- `signal_4g`
- `battery`
- `online_state`
- 其他采集器运行态、通讯态、自检态

采集器产品不承载：

- 激光子设备 `value`
- 深部位移子设备 `dispsX`
- 深部位移子设备 `dispsY`
- 子设备 `sensor_state`

### 5.2 子设备产品

子设备产品承载该传感器自身正式监测数据和正式状态。

激光测距子设备：

- `value`
- `sensor_state`

深部位移子设备：

- `dispsX`
- `dispsY`
- `sensor_state`

### 5.3 单台设备

若同类设备存在单台部署模式，则正式物模型与子设备模式完全一致。

例如单台深部位移设备仍定义为：

- `dispsX`
- `dispsY`
- `sensor_state`

若某设备当前只收到业务值、尚未收到状态报文，不补父设备字段，不新增替代状态字段；只允许 latest 暂时缺少 `sensor_state`。

## 6. 状态语义拆分

建议把“状态”拆成两类，不混用：

### 6.1 传感器正式状态

正式物模型字段：

- `sensor_state`

含义：

- 传感器自身健康态
- 由设备协议显式上报
- 可进入治理、对象洞察、告警、风险辅助分析

### 6.2 上游链路状态

非正式传感器业务字段，建议做派生状态或运维态：

- `collector_online`
- `collector_child_link_state`
- `upstream_comm_state`

含义：

- 子设备是否可经由采集器被访问
- 采集器离线导致的不可达
- 网络或转发链路异常

注意：

- 采集器离线不等于传感器自身故障
- 不应用上游链路状态覆盖 `sensor_state`

## 7. 读写架构分层

### 7.1 写侧

写侧按正式设备主体落库：

1. 原始报文先进入消息日志
2. 协议层按 `iot_device_relation` 识别 `parent -> logicalChannel -> child`
3. 按 `canonicalization_strategy / status_mirror_strategy` 归一
4. 子设备业务值与子设备状态落到子设备 latest / telemetry
5. 采集器自身状态落到采集器 latest / telemetry

### 7.2 读侧

读侧允许额外提供采集器聚合视图：

- 采集器下子设备总数
- 在线子设备数
- 异常子设备数
- 每个逻辑通道最近值
- 每个逻辑通道最近状态
- 子设备健康分布

但这些聚合结果：

- 不写回采集器正式物模型
- 不参与采集器产品合同发布
- 不作为采集器风险指标目录来源

## 8. 风险与对象洞察边界

### 8.1 风险绑定

正式风险指标只绑定真正监测主体：

- 激光测距风险指标绑定到激光子设备
- 深部位移风险指标绑定到深部位移子设备
- 采集器只绑定采集器自身运行风险或通讯风险（如果业务确有需要）

### 8.2 对象洞察

对象洞察正式趋势项也按正式设备主体读取：

- 子设备趋势看子设备
- 采集器趋势看采集器

采集器总览页若需要“全部子设备状态总览”，应走聚合面板，不应把子设备指标加入采集器 `objectInsight.customMetrics[]`。

## 9. 场景落地建议

### 9.1 激光测距

推荐口径：

- 采集器产品：`temp / humidity / signal_4g / ...`
- 激光子设备产品：`value / sensor_state`
- 风险目录：当前继续只发布 `value`
- 采集器页：增加“子设备状态总览/测距总览”聚合卡

### 9.2 深部位移

推荐口径：

- 采集器产品：`temp / humidity / signal_4g / ...`
- 深部位移子设备产品：`dispsX / dispsY / sensor_state`
- 风险目录：是否纳入由业务规则单独定义，但主体仍是子设备
- 单台深部位移设备：沿用同一正式物模型，不因没有采集器而改字段

## 10. 不推荐方案

### 10.1 父设备重复挂子设备业务值

问题：

- 同一指标双份真相
- latest 可能不一致
- 趋势查询口径冲突
- 风险绑定会重复或混乱

### 10.2 父设备重复挂子设备状态

问题：

- 容易把“采集器链路状态”误当“传感器健康状态”
- 告警与对象洞察语义会漂移
- 单台设备与复合设备口径不一致

### 10.3 页面方便驱动物模型重复建模

问题：

- 读模型和写模型耦合
- 后续每新增一类采集器都要重复设计一遍

## 11. 决策结论

正式决策建议如下：

1. 采集器产品不重复承载子设备监测数据
2. 采集器产品不重复承载子设备正式状态 `sensor_state`
3. 子设备产品必须完整承载“监测数据 + 传感器正式状态”
4. 单台设备与复合子设备共用一套正式物模型
5. 采集器页面允许聚合展示子设备状态和最新监测值，但这属于读侧聚合，不属于采集器正式物模型
6. 风险、对象洞察、趋势、告警默认绑定到真正监测主体，不绑定到采集器镜像字段

## 12. 对当前平台的影响

若按本设计推进，当前平台后续应保持以下收口方向：

- `iot_device_relation` 继续作为父子边界权威来源
- 产品定义中心继续禁止把采集器自身状态字段带入子产品
- 后续不要再把子设备 `sensor_state` 回写为采集器正式字段
- 前端若要增强采集器页，优先补“聚合视图接口/查询模型”，不要扩采集器合同字段

## 13. 实施优先级建议

优先级建议如下：

1. 先固化产品建模规范与页面展示规范
2. 再补采集器聚合视图的读侧接口
3. 最后按场景补采集器总览卡、子设备状态总览卡、异常分布卡

不建议反过来先加采集器重复字段，再回头治理。
