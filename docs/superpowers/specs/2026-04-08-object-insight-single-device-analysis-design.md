# 对象洞察台单设备分析重构设计

> 日期：2026-04-08
> 主题：把 `/insight` 收口为“单设备对象洞察台”，通过设备资产中心跳转进入风险运营板块，并以 `TDengine` 时序数据作为趋势主数据源

## 1. 背景

当前对象洞察台已经从早期“纯设备洞察页”切到“风险监测详情为主、设备补充数据为辅”的实现，但仍存在以下问题：

- `/insight` 直开时会自动带入默认设备编码，搜索框不是真空态，不符合“用户主动输入设备编号后再综合分析”的使用口径。
- 设备资产中心虽然已经支持“更多 -> 洞察”跳转，但对象洞察页的页面身份仍偏向“风险监测绑定对象”，与用户希望的“单设备分析页”不一致。
- 当前趋势图仍允许在无风险详情时从消息日志里反解析数值字段，这种口径不满足客户对趋势真实性和时序连续性的要求。
- 监测值、状态值、电量等不同语义的指标还没有形成稳定分组，无法面向不同设备类型做差异化表达。
- 现有页面结构更像研发排障页，客户视角下“先看设备当前状态，再看趋势与文字结论”的阅读路径不够明确。

用户补充后的硬约束如下：

1. 设备资产中心跳转进来时，必须带选中的设备编号到对象洞察台；对象洞察直开时搜索框默认空。
2. 趋势主数据必须来自 `TDengine` 时序数据库，不能来源于消息日志。
3. 折线图必须支持 `近一天 / 近一周 / 近一月 / 近一季度 / 近一年` 时间维度，默认展示最近一周。
4. 缺失时间桶必须补 `0`，拒绝折线图出现断裂画面。
5. 对象洞察页只分析单台设备，不表达风险点整体分析。
6. 不同设备能力展示不同指标，首批以 `SK00EB0D1308313` 泥水位设备为样板，重点展示：
   - 泥水位高程
   - 传感器在线状态
   - 剩余电量
7. 后续扩展位应表达为“系统自定义参数”，例如相对湿度、4G 信号等；本轮优先建设扩展结构，不同步建设可视化配置后台，但允许复用既有设备 `metadataJson` 做轻量配置覆盖。

## 2. 目标

- 把对象洞察台稳定收口为“单设备分析页”，而不是“风险点整体分析页”。
- 统一支持两种进入方式：
  - 设备资产中心跳转进来，自动带 `deviceCode` 并加载
  - 风险运营内直开 `/insight`，搜索框默认空，等待用户输入
- 趋势查询统一改为基于 `TDengine` 的时序查询，页面不再使用消息日志构造主趋势。
- 趋势区域按“监测数据 / 状态数据”两大类分开展示，形成稳定的客户阅读路径。
- 前端采用“设备能力注册表 + 模板化渲染”的方式，为 `预警型 / 监测型 / 采集型` 和指定设备样板提供差异化展示。
- 页面视觉改为“设备画像式工作台”，优先服务客户解读，而不是研发排障。

## 3. 非目标

- 本轮不建设“对象洞察指标配置后台”。
- 本轮不把对象洞察台扩展成风险点列表页或整体监测总览页。
- 本轮不把消息日志作为趋势主数据源；日志仅保留为审计和兜底证据。
- 本轮不引入第二条对象洞察路由。
- 本轮不建设独立的对象洞察配置表、配置后台或新路由；轻量扩展优先采用“前端注册表 + 设备 `metadataJson.objectInsight.customMetrics[]` 覆盖”的方式承接“系统自定义参数”扩展位。

## 4. 方案比较

### 4.1 方案 A：前端继续拼装现有接口

- 继续使用 `risk-monitoring + device + properties + message-logs`
- 趋势缺失时仍从日志中反解

优点：

- 改动小，交付快。

缺点：

- 违反“趋势只能来自 TDengine”的新约束。
- 并发切换设备时容易出现旧请求覆盖新请求。
- 扩展 `battery_dump_energy / humidity / signal_4g` 等状态项时，页面逻辑会越来越散。

### 4.2 方案 B：新增对象洞察专用后端聚合接口

- 后端直接返回“单设备画像 + 趋势 + 文案 + 分组”

优点：

- 前端实现最简。
- 并发控制和数据一致性最好。

缺点：

- 后端会承载大量展示知识，耦合过重。
- 后续每增加一个展示字段，都需要同时改接口和前端。

### 4.3 方案 C：前端能力注册表 + 后端通用时序批量查询

- 风险上下文继续使用 `risk-monitoring`
- 设备主档与属性快照继续使用 `device` 接口
- 新增一个通用 `TDengine` 时序批量查询接口，一次按设备和多个指标拉历史序列
- 前端通过能力注册表决定展示哪些指标、如何分组、如何生成分析文案

优点：

- 扩展性最好，后续增加指标多数只需补配置。
- 并发更稳，同一设备一次拉取批量趋势，减少多请求竞争。
- 模块边界清晰：`telemetry` 只负责时序查询，`risk-monitoring` 只负责风险上下文，`ui` 负责展示编排。
- 符合本轮“先做可扩展结构，后续再加配置入口”的目标。

缺点：

- 比方案 A 多一层后端改动。

## 5. 选型

本轮采用方案 C。

理由：

- 它同时满足 `TDengine` 主数据源、页面扩展性、并发安全和后续可配置化演进四个核心要求。
- 它不会把展示编排固化进后端聚合接口，能避免对象洞察页成为难以维护的“专用大接口消费者”。
- 它允许本轮先以 `SK00EB0D1308313` 样板打通模板机制，后续只扩配置，不推翻架构。

## 6. 信息架构

### 6.1 页面主语义

对象洞察台的主身份固定为“单台设备”。

页面内仍可展示风险点、绑定测点和风险等级，但这些信息是该设备的“风险上下文”，不是页面主身份。

### 6.2 页面进入规则

#### 6.2.1 直开 `/insight`

- 搜索框默认空。
- 页面不自动请求。
- 首屏展示空态引导：提示输入设备编码后再开始综合分析。

#### 6.2.2 从设备资产中心跳转

- `DeviceWorkbenchView` 的“更多 -> 洞察”保持现有入口，但保证始终传入 `deviceCode`
- 对象洞察台收到 `deviceCode` 后自动加载
- 搜索框同步显示该设备编码

### 6.3 页面布局

页面采用“设备画像式工作台”结构，而不是仪表盘网格式。

固定分区如下：

1. 顶部搜索与时间维度区
   - 设备编码搜索框
   - 查询 / 重置
   - 时间档位切换：`近一天 / 近一周 / 近一月 / 近一季度 / 近一年`
2. 设备画像区
   - 设备名称、设备编码、产品名称、设备类型、在线状态、最新上报时间
   - 当前风险态势作为补充徽标
3. 基础档案区
   - 设备基础档案
   - 风险上下文档案
4. 核心指标卡区
   - 当前监测值
   - 当前状态值
   - 当前关键状态项（如电量）
   - 当前风险态势
5. 趋势区
   - 左侧：监测数据趋势
   - 右侧：状态数据趋势
6. 图文分析区
   - 分析摘要
   - 风险上下文
   - 扩展指标位
7. 底部证据区
   - 设备属性快照
   - 必要时保留日志与治理缺口提示

### 6.3.1 基础档案信息范围

基础档案区必须作为独立正文分区保留，不能只折叠到顶部设备画像摘要中。

建议拆成两个卡片：

1. `设备基础档案`
   - 设备名称
   - 设备编码
   - 产品名称
   - 设备类型
   - 所属机构
   - 所在区域
   - 部署位置
   - 接入协议
   - 固件版本
   - 最近在线时间
   - 最近上报时间
2. `风险上下文档案`
   - 关联风险点名称
   - 风险点编码
   - 当前主监测项
   - 当前风险态势
   - 当前绑定状态
   - 风险等级来源说明

设计要求：

- 基础档案使用中文业务名称，不显示内部 identifier。
- 当设备尚未纳入风险监测时，`风险上下文档案` 仍保留卡片，但内容明确表达“当前未纳管”。
- 客户进入页面后，应先能从基础档案确认“这是什么设备、装在哪里、归属哪个风险上下文”，再进入趋势判断。

### 6.4 客户视角布局要求

- 首屏必须先回答“这台设备是谁、现在怎样”。
- 基础档案必须作为独立内容块稳定存在，不能被压缩成页头摘要。
- 趋势区必须一眼看出“监测值”和“状态值”是两类数据。
- 文字分析紧跟折线图，避免客户看完图不知道结论。
- 核心指标、趋势图标题、分析文案统一使用中文业务名称，不直接向客户暴露 `L4_NW_1`、`S1_ZT_1.sensor_state.L4_NW_1` 这类内部标识。
- 不再使用研发感过强的诊断页表达，不在首屏堆叠日志和技术说明。

## 7. 数据架构

### 7.1 现有数据源保留职责

- `GET /api/device/code/{deviceCode}`
  - 设备主档、产品、协议、安装位置、最近在线与最近上报
- `GET /api/device/{deviceCode}/properties`
  - 当前最新属性快照，用于当前值补充、状态卡和证据表格
- `GET /api/risk-monitoring/realtime/list?deviceCode=...`
  - 当前设备命中的风险绑定列表
- `GET /api/risk-monitoring/realtime/{bindingId}`
  - 当前主绑定对应的风险上下文、风险等级、当前测点信息

### 7.2 新增数据源：TDengine 批量时序查询

本轮新增一个面向对象洞察的通用批量查询接口，建议放在 `spring-boot-iot-telemetry`：

- `POST /api/telemetry/history/batch`

请求语义：

- 输入：`deviceId`、指标列表、时间范围、聚合粒度、是否补零
- 输出：按指标返回对齐后的时间桶序列

建议请求体：

```json
{
  "deviceId": 1001,
  "identifiers": [
    "L4_NW_1",
    "S1_ZT_1.sensor_state.L4_NW_1",
    "S1_ZT_1.battery_dump_energy"
  ],
  "rangeCode": "7d",
  "bucket": "day",
  "fillPolicy": "ZERO"
}
```

建议返回体：

```json
{
  "deviceId": 1001,
  "rangeCode": "7d",
  "bucket": "day",
  "points": [
    {
      "identifier": "L4_NW_1",
      "displayName": "泥水位高程",
      "seriesType": "measure",
      "buckets": [
        { "time": "2026-04-01 00:00:00", "value": 0 },
        { "time": "2026-04-02 00:00:00", "value": 1.28 }
      ]
    }
  ]
}
```

### 7.3 时间维度与补零规则

时间档位固定为：

- `1d`
- `7d`
- `30d`
- `90d`
- `365d`

建议桶粒度：

- `1d` -> 小时桶
- `7d` -> 天桶
- `30d` -> 天桶
- `90d` -> 周桶
- `365d` -> 月桶

补零规则：

- 所有时间桶先按目标范围完整生成
- 未查询到值的桶统一补 `0`
- 前端图表只消费对齐后的完整桶，不自行再做缺口拼接

这样可以保证：

- 折线连续
- 多序列共用统一时间轴
- 不把“无数据”展示成断裂感

### 7.4 消息日志口径调整

对象洞察台中的消息日志不再参与主趋势构建。

日志只保留两类用途：

- 作为调试或证据补充
- 当属性快照和时序查询都不可用时，给出“暂无趋势数据”的诚实提示

## 8. 设备能力注册表设计

### 8.1 目标

通过一份前端注册表驱动不同设备的：

- 主类型
- 关键指标
- 趋势分组
- 文案模板
- 系统自定义参数扩展位

### 8.2 匹配层级

注册表采用两层匹配：

1. 精确设备模板
   - 例如 `deviceCode = SK00EB0D1308313`
2. 通用设备类型模板
   - `预警型`
   - `监测型`
   - `采集型`

匹配优先级：

- 设备精确模板优先
- 未命中时回退到设备类型模板
- 再未命中时回退到通用模板

### 8.3 注册表结构

建议字段：

```ts
interface InsightCapabilityProfile {
  key: string;
  match: {
    deviceCodes?: string[];
    productNameKeywords?: string[];
    objectTypes?: Array<'warning' | 'detect' | 'collect'>;
  };
  heroMetrics: string[];
  displayNameMap: Record<string, string>;
  trendGroups: Array<{
    key: 'measure' | 'status';
    title: string;
    identifiers: string[];
  }>;
  narrativeBlocks: Array<{
    key: string;
    title: string;
    identifiers: string[];
    template: string;
  }>;
  extensionParameters: Array<{
    parameterKey: string;
    identifier: string;
    displayName: string;
  }>;
}
```

说明：

- `identifier` 继续作为系统内部指标标识，用于查询 `TDengine`、属性快照和风险上下文。
- `displayNameMap` 和 `displayName` 负责客户展示文案，页面层不得直接显示英文标识。
- `extensionParameters` 语义固定为“系统自定义参数”，不是任意裸字段列表。
- 运行期若设备 `metadataJson.objectInsight.customMetrics[]` 存在显式配置，前端应在注册表之上继续合并设备级覆盖，用于补充趋势指标、显示名称和分析模板。

### 8.4 泥水位设备样板

`SK00EB0D1308313` 首批模板固定为：

- `heroMetrics`
  - `L4_NW_1` -> `泥水位高程`
  - `S1_ZT_1.sensor_state.L4_NW_1` -> `传感器在线状态`
  - `S1_ZT_1.battery_dump_energy` -> `剩余电量`
- `trendGroups`
  - `监测数据`
    - `L4_NW_1` -> `泥水位高程`
  - `状态数据`
    - `S1_ZT_1.sensor_state.L4_NW_1` -> `传感器在线状态`
    - `S1_ZT_1.battery_dump_energy` -> `剩余电量`
- `extensionParameters`
  - `humidity` -> `S1_ZT_1.humidity` -> `相对湿度`
  - `signal_4g` -> `S1_ZT_1.signal_4g` -> `4G 信号强度`

### 8.5 后续扩展策略

当后续要增加相对湿度、4G 信号或其他状态项时：

- 优先补注册表配置；如需对单台设备做覆盖，可补 `metadataJson.objectInsight.customMetrics[]`
- 若只是增加展示字段，不改后端接口语义
- 若新增字段也需要趋势，则把 identifier 加入批量时序查询列表
- 若未来需要配置后台，则后台产出的结构应以“系统自定义参数”模型为主，并与 `metadataJson.objectInsight.customMetrics[]` / 前端注册表保持同型

## 9. 并发与稳定性设计

### 9.1 页面加载入口

统一使用一个入口函数：

- `loadInsight(deviceCode, rangeCode, source)`

其中 `source` 用于区分：

- `direct`
- `device-workbench`
- `route-change`
- `time-range-change`

### 9.2 防串页策略

- 每次加载生成 `requestVersion`
- 所有异步结果落地前必须校验当前版本号
- 旧版本请求即使后返回，也不得覆盖最新设备页面

如果项目现有请求层支持取消，则可进一步使用 `AbortController`

### 9.3 批量化原则

- 时间范围切换只触发一次批量时序查询
- 不按卡片逐个请求趋势
- 风险上下文与设备主档并行请求，但最终统一落到同一次页面状态提交

### 9.4 失败处理

- 设备不存在：提示设备编码无效，不渲染洞察内容
- 风险绑定缺失：继续展示单设备洞察，但风险态势区显示“未纳入风险监测”
- 时序查询失败：页面保留设备画像与属性快照，趋势区显示“TDengine 趋势查询失败”
- 单个指标无数据：该指标补 `0`，并在图例或摘要中注明“当前范围内无有效上报”

## 10. 页面视觉设计

### 10.1 视觉方向

继续复用项目现有 `StandardPageShell + StandardWorkbenchPanel + PanelCard`，但对象洞察正文采用客户导向的“设备画像式工作台”。

原则：

- 强标题、弱说明
- 首屏卡片数量克制
- 趋势图面积要充足，避免做成小卡预览
- 图表与文字说明要同屏呼应
- 状态类折线使用更温和的强调色，监测值折线使用主品牌色

### 10.2 趋势区视觉合同

- 监测数据折线图在左，状态数据折线图在右
- 两个图共享相同的时间切换器
- 默认展示最近一周
- 图表下方紧跟对应文字结论
- 缺失值补零后，折线连续，但 tooltip 仍应允许提示“补零桶”
- 折线图不使用平滑曲线效果，统一采用明确折线表达，避免弱化数据拐点

### 10.3 移动端适配

- 桌面端：双列趋势布局
- 平板和移动端：趋势图改为上下堆叠
- 顶部指标卡从四列降为两列或一列

## 11. 测试与验证

### 11.1 前端测试

- `/insight` 直开时搜索框为空，页面不自动请求
- 从设备资产中心跳转时自动带入 `deviceCode`
- 时间切换触发新的批量时序查询
- 不同设备模板命中不同趋势分组
- `SK00EB0D1308313` 模板正确展示三类核心指标，且页面显示中文名称
- 缺失桶补零后，图表序列长度与时间轴长度一致
- 快速切换设备时，旧请求结果不会覆盖新页面
