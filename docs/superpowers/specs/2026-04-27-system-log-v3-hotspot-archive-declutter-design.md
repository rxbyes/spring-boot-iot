# `/system-log` V3 减负优化设计（观测热点 + 归档治理）

## 1. 背景

`/system-log` 当前已经形成 `异常排查 / 观测热点 / 归档治理` 三张平级页签，并且都具备真实可用的数据链路与证据能力。  
但从使用体验上看，`异常排查` 已经开始收口为工作台，而 `观测热点` 与 `归档治理` 仍然存在明显的“信息堆砌”问题：

1. **首屏主角不清楚**
   - `观测热点` 同时在讲热点、样本、趋势、调度任务
   - `归档治理` 同时在讲筛选、摘要、焦点、批次台账、治理信号
2. **信息层级过平**
   - 很多信息都是“重要的”，结果等于都不够重要
   - 用户很难判断“这一屏我先看哪里”
3. **首屏认知负荷偏高**
   - 用户还没做选择，页面已经把多个分析层一股脑摊出来
4. **视觉上像模块拼装，不像连续工作流**
   - 页面虽然不缺信息，但缺“先做什么、再做什么”的顺序感

因此，这一轮优化的目标不再是补更多能力，而是把 `观测热点` 和 `归档治理` 从“信息墙”收成“工作台”。

## 2. 范围判断与拆分

这次需求涉及两个相对独立的前端子系统：

1. `观测热点`
2. `归档治理`

它们共享同一套设计原则，但实现边界和测试面并不完全重叠。  
因此本轮**不建议写成一份直接落代码的大杂烩实现**，而是拆成两个连续切片：

1. **子项目 A：观测热点减负**
2. **子项目 B：归档治理减负**

两者共用一份 V3 体验设计，但后续 implementation plan 与代码执行应拆成两个切片，避免一次性大改带来回归风险。

## 3. 目标与非目标

### 3.1 目标

1. 降低 `观测热点` 与 `归档治理` 的首屏认知负荷
2. 明确每张页签的“主对象”
3. 把复杂信息收进联动区、详情抽屉或次级区域，而不是全部摊在首屏
4. 统一 `/system-log` 的页签语法：`焦点条 -> 主对象列表 -> 联动下钻区/详情`
5. 保留现有证据、追踪、对比、报告能力，不为减负牺牲排查深度

### 3.2 非目标

1. 本轮不新增后端 API
2. 本轮不修改 `/audit-log` 审计中心
3. 本轮不重做 `异常排查`
4. 本轮不把详情抽屉改成新路由页
5. 本轮不追求三张页签一次性全部大换血

## 4. 方案对比

### 4.1 方案 A：只做视觉轻美化

做法：
1. 保留现有结构
2. 微调间距、颜色、标题、边框

优点：
1. 改动最小
2. 风险最低

缺点：
1. 信息层级问题不解决
2. 用户仍然会感到“看起来多、看起来重”

### 4.2 方案 B：主对象优先 + 渐进展开，我推荐

做法：
1. 每个页签首屏只保留一个主对象列表
2. 细节内容进入联动区、次级分段或抽屉
3. 通过选中态和焦点条保持上下文

优点：
1. 首屏更轻
2. 逻辑更像工作台
3. 最符合你当前对“不要堆信息”的诉求

缺点：
1. 需要改动现有页签组件结构
2. 需要补对应测试

### 4.3 方案 C：把所有页面都折成纯表格

做法：
1. 强行用表格承载所有层级
2. 把卡片/焦点区全部去掉

优点：
1. 结构最统一
2. 实现相对直接

缺点：
1. 热点与治理的上下文表达会变弱
2. 页面会变得像台账，而不是诊断工作区

### 4.4 推荐

采用 **方案 B：主对象优先 + 渐进展开**。

原因：
1. 它同时解决“美观性”和“理解成本”
2. 不靠砍功能来减负，而是靠重排信息层级
3. 最适合复用到 `观测热点` 和 `归档治理` 两块独立页面

## 5. 统一设计原则

### 5.1 只允许一个首屏主角

每个页签的第一屏只能有一个主对象：

1. `观测热点` 的主角是：**热点对象**
2. `归档治理` 的主角是：**归档批次**

其他内容必须退后：

1. 变成联动区
2. 变成次级信息条
3. 变成抽屉内容

### 5.2 渐进展开，而不是同屏并列

用户先完成第一步判断，再看第二层信息：

1. 先选热点，再看样本/趋势/相关任务
2. 先选批次，再看治理对比/报告/异常原因

### 5.3 说明文案做减法

页面上不再堆太多解释句。  
首屏文案只做两件事：

1. 告诉用户当前看的对象是什么
2. 告诉用户下一步点哪里

### 5.4 高密细节收进深层

像这些高密治理/诊断字段：

1. dry-run 偏差
2. preview availability
3. trace 细粒度上下文
4. 归档行数/删除行数/候选行数的完整组合

应该尽量进入：

1. 行展开
2. 详情抽屉
3. 联动区

而不是首屏全量铺开。

## 6. 子项目 A：观测热点减负

### 6.1 当前问题

当前 [AuditLogHotspotTabPanel.vue](E:/idea/ghatg/spring-boot-iot/spring-boot-iot-ui/src/components/auditLog/AuditLogHotspotTabPanel.vue) 同时展示：

1. 慢点 Top
2. 当前焦点条
3. 最近样本
4. 趋势观察
5. 调度任务台账

这使得页面缺少明确的阅读顺序。

### 6.2 V3 目标结构

`观测热点` 改成三段结构：

1. **热点焦点条**
   - 当前选中的热点名称
   - 目标对象
   - 最新 traceId
   - 最新发生时间
   - 当前趋势窗口

2. **热点主表**
   - 一行一个热点对象
   - 首屏只回答“哪个热点值得优先看”

3. **联动下钻区**
   - 默认展示 `最近样本`
   - 次级切换展示 `趋势`
   - `相关任务` 作为第三视图或次级区块

### 6.3 主表字段建议

热点主表建议保留：

1. 热点名称
2. 目标对象
3. 最近发生时间
4. 最大耗时
5. 平均耗时
6. 总次数
7. 操作

不再在首屏主表里直接摊开太多附加信息。

### 6.4 下钻区建议

默认下钻视图：**最近样本**

原因：
1. 用户先要判断“这是不是一个真实值得追的热点”
2. 最近样本比趋势图更适合作为第一眼排查入口

下钻区采用分段切换：

1. `最近样本`
2. `趋势`
3. `相关任务`

其中 `相关任务` 用来承接当前独立铺开的调度任务台账，避免它继续抢主角。

### 6.5 调度任务降级原则

调度任务台账不删除，但降级成热点的伴随信息：

1. 不再作为第二大块完整平铺
2. 默认只在选中热点后进入相关任务视图或次级折叠区
3. 仍保留 `证据` 动作

### 6.6 成功标准

用户进入 `观测热点` 后的自然顺序应该变成：

1. 先看哪个热点最异常
2. 选中一个热点
3. 先看最近样本
4. 再切趋势或相关任务

而不是首屏同时被五块信息抢注意力。

## 7. 子项目 B：归档治理减负

### 7.1 当前问题

当前 [AuditLogArchiveTabPanel.vue](E:/idea/ghatg/spring-boot-iot/spring-boot-iot-ui/src/components/auditLog/AuditLogArchiveTabPanel.vue) 同时存在：

1. 高密筛选器
2. 焦点条
3. overview 指标卡
4. focus hint
5. 批次主表
6. 主表行内又承载大量治理数值

结果是页面有“信息都重要，所以都显得很重”的问题。

### 7.2 V3 目标结构

`归档治理` 改成：

1. **精简治理摘要**
   - 异常批次数
   - 执行偏差总量
   - 剩余过期总量

2. **批次主表**
   - 一行一个归档批次
   - 首屏以批次判断为主

3. **详情抽屉**
   - 承接批次对比
   - 承接确认报告预览
   - 承接更细的治理数值和异常原因

### 7.3 摘要层建议

overview cards 从“并列很多关注点”收成“少量核心指标”：

1. 异常批次数
2. 执行偏差总量
3. 剩余过期总量

`最近异常批次` 不建议继续作为普通并列卡。  
更适合做成：

1. 一条置顶焦点提示
2. 或一个可点击的轻量胶囊入口

因为它更像“当前优先处理对象”，不是常规统计项。

### 7.4 主表字段建议

归档主表首屏建议保留：

1. 批次号
2. 执行状态
3. 对比结论
4. 进度摘要（确认 / 归档 / 删除）
5. 风险信号
6. 创建时间
7. 操作

不建议继续在首屏一行里摊太多治理细节。

### 7.5 深层承接建议

以下内容应优先进入详情抽屉或行展开，而不是首屏主表：

1. candidateRows
2. deltaDryRunVsDeleted
3. remainingExpiredRows 的完整辅助解释
4. report preview availability 的细粒度差异
5. 多路径报告来源

### 7.6 成功标准

用户进入 `归档治理` 后的自然顺序应该变成：

1. 先判断今天治理是否异常
2. 再锁定某个批次
3. 最后进详情抽屉看对比和报告

而不是先被大量摘要和高密批次信息压住。

## 9. 执行顺序

推荐执行顺序：

1. **第一刀：观测热点减负**
   - 收主表
   - 收下钻区
   - 把调度任务降级为相关任务

2. **第二刀：归档治理减负**
   - 精简摘要层
   - 压缩主表信息密度
   - 让详情抽屉承接复杂治理信息

原因：

1. `观测热点` 的收益最直接
2. `归档治理` 改动的治理语义更重，放第二刀更稳

## 10. 结论

这轮优化的核心不是再加更多功能，而是把 `观测热点` 和 `归档治理` 从“信息堆砌”收成“主对象优先、渐进展开”的工作台。

最终目标是：

1. `观测热点` 先看热点，再看样本/趋势/相关任务
2. `归档治理` 先看治理状态，再锁定批次，再进详情
3. 两张页签都减少首屏负担，增强主次关系和视觉节奏

这比继续在当前页面上叠更多块内容，更符合你现在对体验和美观性的真实诉求。
