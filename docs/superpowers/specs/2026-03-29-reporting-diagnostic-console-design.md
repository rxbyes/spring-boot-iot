# Reporting Diagnostic Console Refresh Design

**Date:** 2026-03-29
**Status:** Draft for review
**Scope:** `链路验证中心 /reporting` 在既有 `message-flow` 真实环境基线之上的诊断主控台升级，聚焦 `模拟验证 + 结果复盘 + 最近记录` 三块能力的重新组织

## 1. 背景

当前 `/reporting` 已经具备以下能力：

1. 设备编码反查接入契约。
2. HTTP / MQTT 双通道模拟发送。
3. JSON / XML / 原始文本输入与明文帧预演。
4. 基于 `sessionId / traceId` 的 `message-flow` 时间线复盘。
5. 最近提交恢复。
6. 跳转 `链路追踪台 / 异常观测台 / 数据校验台` 的诊断联动。

当前实现也已经符合既有文档基线：

1. 路由仍为 `/reporting`。
2. 页面仍属于 `接入智维` 诊断专台。
3. `message-flow` 单次复盘继续依赖：
   - `POST /api/message/http/report`
   - `POST /api/message/mqtt/report/publish`
   - `GET /api/device/message-flow/session/{sessionId}`
   - `GET /api/device/message-flow/recent`
4. 前端现有主文件为 `spring-boot-iot-ui/src/views/ReportWorkbenchView.vue`，已经在同页承接了模拟、复盘、最近提交和诊断联动。

但当前页面仍存在 4 个核心缺口：

1. **首屏仍偏“发送工具页”，不是“诊断主控台”。**
   - 用户进入页面后先看到的是输入与发送，而不是本轮验证当前是成功、等待中还是失败。
2. **结果复盘有数据，但缺少统一结论层。**
   - 页面能展示 timeline、response、payload 预演，但用户需要自己拼结论。
3. **最近记录仍偏“恢复入口”，还不是诊断清单。**
   - 记录本身没有稳定表达“本条结论是什么、卡在哪一段、下一步去哪”。
4. **前端状态判断散在单页面大文件中。**
   - 现有 `/reporting` 虽然功能完整，但 `ReportWorkbenchView.vue` 规模过大，后续继续增强会放大维护成本。

结论：

- 当前页面已经解决“能不能发、能不能看 timeline”的问题。
- 下一步要解决的是“用户能不能 5 秒内看懂当前验证状态，并被带去正确下一步”。

## 2. 目标

本轮设计目标如下：

1. 把 `/reporting` 从“模拟发送工具页”升级为“诊断主控台”。
2. 保留 `模拟验证 / 结果复盘 / 最近记录` 三块业务语义，但让 `结果复盘` 成为首屏主角。
3. 在页面顶部新增共享 `诊断状态头`，让用户 5 秒内看懂当前是：
   - 尚未验证
   - 验证成功
   - 等待回流
   - 验证失败
   - 复盘受限
4. 把 `最近记录` 从“最近 session 恢复列表”升级为“诊断清单”，每条记录优先表达结论、卡点和下一步动作。
5. 保持现有 `DiagnosticContext + query` 跨页联动协议，不为本轮重新发明第二套上下文传递机制。
6. 视觉方向采用“2026 扁平商业风”，但继续服从现有浅色控制台品牌基线，不新增私有主题。
7. 以前端重构为主，后端只做 `message-flow recent` 最小必要增强，不改接入主链路。

## 3. 非目标

本轮明确不做：

1. 不新增 `/reporting` 的第二条静态路由。
2. 不新建独立 observability 诊断页或总览页。
3. 不改动 `message-flow` 固定 Pipeline、Redis key 口径或真实环境 TTL 策略。
4. 不改动当前 `/reporting -> /message-trace -> /system-log -> /file-debug` 的 query 联动协议。
5. 不为 `最近记录` 新增重型分页台账接口、数据库表或多维聚合服务。
6. 不把页面升级成大面积营销化视觉、深色风格或私有主题色实验页。
7. 不在本轮引入 AI 文本总结、聊天式助手或黑盒诊断结论。

## 4. 用户已确认的设计结论

本轮设计已经在对话中得到以下明确确认：

1. 页面主气质选择：`诊断台优先`。
2. `最近记录` 的角色选择：`诊断清单`，不是单纯恢复入口。
3. 首屏最优先表达的信息：`这次验证现在是成功、等待中还是失败`。
4. 实现范围策略：`前端主导 + 少量后端补强`。
5. 版式方向选择：采用可视化草图中的 `A. 结果主控台`。
6. 最终混合方案：`A 为主，吸收 C 的真页签切换感`。

一句话结论：

`/reporting` 改成“保留真页签的诊断主控台”，由共享状态头统领三块能力，默认进入结果复盘，模拟验证退到效率轨，最近记录升级为诊断清单。

## 5. 设计原则

### 5.1 先给判断，再给证据

用户首屏先看到“当前状态是什么”，再看 timeline、response、payload 和帧预演等证据。

### 5.2 真页签保留，但主次重排

页面继续保留真页签，不回退到单大长页；但三个页签不再是三个互相平级的内容岛，而是同一条诊断链路的三个视角：

1. `模拟验证` 负责发起。
2. `结果复盘` 负责判断。
3. `最近记录` 负责回看和继续处理。

### 5.3 只做规则化诊断，不做黑盒判断

所有状态与动作建议必须能解释来源，例如：

1. `correlationPending=true 且 traceId 为空`
2. `session.status=FAILED`
3. `timelineAvailable=true 但 session 查询为空`
4. `trace 已过期，仅保留消息日志`

### 5.4 风格更现代，但不脱离控制台基线

本轮的“2026 互联网风格”解释为：

1. 更扁平
2. 更商业
3. 更稳定
4. 更强调版面主次
5. 更少说明墙和装饰性色块

不是：

1. 新主题
2. 新配色
3. 大面积高饱和渐变
4. 深色炫技面板

## 6. 总体方案

### 6.1 页面级结构

`/reporting` 页面重新组织为：

1. 面包屑
2. 业务标题 `链路验证中心`
3. 真页签：`结果复盘 / 模拟验证 / 最近记录`
4. 共享 `诊断状态头`
5. 当前页签主内容区

与现状相比，核心变化有 4 点：

1. 真页签仍在，但展示顺序调整为 `结果复盘 -> 模拟验证 -> 最近记录`。
2. 默认激活页签改为 `结果复盘`。
3. `诊断状态头` 固定在真页签和主内容区之间，三个页签共用。
4. `发送成功` 后自动跳入 `结果复盘`，不再停留在输入页让用户自己找结果。

### 6.2 三块能力的角色定位

#### `结果复盘`

页面默认主舞台，负责：

1. 给出本轮验证状态。
2. 展示 timeline、trace/session、response、payload 证据。
3. 给出下一步动作。

#### `模拟验证`

高效操作轨，负责：

1. 设备查询
2. 发送配置
3. Payload 编辑
4. 发送动作

它保留完整功能，但不再承担首页判断权。

#### `最近记录`

诊断清单，负责：

1. 快速看懂最近几条验证各自结论。
2. 看卡点和失败阶段。
3. 直接继续处理，而不是只恢复 session。

## 7. 共享诊断状态头设计

### 7.1 结构

共享状态头采用一体化扁平板块，而不是多个并列概况卡，固定分为 3 个信息槽：

1. `当前状态`
   - 大字结论：`尚未验证 / 验证成功 / 等待回流 / 验证失败 / 复盘受限`
   - 一句短说明
2. `当前卡点`
   - 失败阶段、等待原因或当前限制说明
3. `下一步动作`
   - 最多 3 个动作按钮

状态头下方增加一条轻量上下文带，统一展示：

- `deviceCode`
- `transportMode`
- `topic`
- `sessionId`
- `traceId`

### 7.2 动作规则

动作不允许回到“模块目录式按钮墙”，只展示与当前状态直接相关的 1~3 个动作。

示例：

1. `验证成功`
   - `继续链路追踪`
   - `打开数据校验`
2. `等待回流`
   - `刷新状态`
   - `查看异常观测`
3. `验证失败`
   - `继续链路追踪`
   - `查看异常观测`
   - `回看最近记录`
4. `尚未验证`
   - `前往模拟验证`
5. `复盘受限`
   - `查看异常观测`
   - `回看最近记录`

## 8. 状态模型与判定规则

### 8.1 页面统一状态枚举

前端统一收口为 5 个用户可感知状态：

1. `idle`
   - 对外文案：`尚未验证`
2. `validated`
   - 对外文案：`验证成功`
3. `pending`
   - 对外文案：`等待回流`
4. `failed`
   - 对外文案：`验证失败`
5. `degraded`
   - 对外文案：`复盘受限`

其中 `degraded` 专门用于区分“本轮业务不一定失败，但当前无法完整复盘”的状态，避免把环境/证据问题误判成业务失败。

### 8.2 判定优先级

为了避免页面出现前后矛盾，统一采用以下判断顺序：

1. 若当前无 `sessionId / traceId / restored recent session`，判定为 `idle`。
2. 若 `messageFlowSession.status=FAILED` 或 timeline 最终失败，判定为 `failed`。
3. 若 `correlationPending=true` 且 `traceId` 为空，判定为 `pending`。
4. 若出现以下任一情况，判定为 `degraded`：
   - timeline 查询异常
   - timeline 缺失
   - HTTP 返回 `timelineAvailable=true` 但后续 `session` 查询为空
   - trace/timeline 已过期
5. 若 timeline 存在且链路可完整复盘，判定为 `validated`。

### 8.3 现有前端可直接派生的结论

以下判断不需要额外后端支持，前端即可稳定完成：

1. 当前五类总体状态
2. 状态头主结论文案
3. `结果复盘` 顶部摘要
4. `等待回流`、`时间线缺失`、`Redis 异常` 的区别化文案
5. `最近记录` 的基础结论标签与轻量分组

## 9. 三个真页签的内容设计

### 9.1 结果复盘

`结果复盘` 是默认首屏，也是本轮主舞台。

内容顺序固定为：

1. 共享 `诊断状态头`
2. `复盘结论区`
   - 一句话总结本轮验证当前状态
3. `上下文证据区`
   - `traceId / sessionId / transport / topic / deviceCode / protocolCode`
4. `主时间线区`
   - 继续复用 `StandardTraceTimeline`
5. `结果证据区`
   - 实际发送内容
   - 最近一次响应
   - 明文帧预演
6. `下一步动作区`

特殊状态处理：

1. MQTT pending 时，主时间线区上方先展示“等待回流状态面板”，明确：
   - 正在等待什么
   - 已等待多久
   - 超时后优先去哪
2. timeline 缺失或 Redis 异常时，不允许只剩普通空态；仍要保留结论面板，清晰说明当前属于 `复盘受限`。

### 9.2 模拟验证

`模拟验证` 继续保留完整能力，但视觉降级为效率轨页签。

结构改为 3 段：

1. `设备身份段`
   - 设备编码查询
   - 产品 Key / 协议编码 / clientId / 设备名称只读反显
2. `发送配置段`
   - 传输方式
   - 上报模式
   - Topic
   - tenantId
3. `Payload 编辑段`
   - 模板
   - 格式化
   - payload 编辑
   - 明文帧类型 3 文件流输入
   - 一键补全
   - 发送动作

交互规则：

1. `模拟验证` 页签顶部仍显示共享 `诊断状态头`，因此用户不会因切回发送页而丢失当前诊断上下文。
2. 发送按钮区域保留简短发送前判断，但缩成一句短摘要，不再与结果复盘争主次。
3. 发送成功后自动切回 `结果复盘`。

### 9.3 最近记录

`最近记录` 不再是“最近 session 恢复卡片”，而是“诊断清单”。

每条记录的固定信息顺序为：

1. `结论标签`
   - `成功 / 等待回流 / 失败 / 时间线缺失 / 已过期`
2. `一句话诊断`
3. `基础上下文`
   - 设备
   - 通道
   - Topic
   - 时间
4. `当前卡点`
   - 失败阶段或等待原因
5. `动作`
   - 恢复为当前复盘上下文
   - 打开链路追踪
   - 查看异常观测
   - 打开数据校验

最近记录只保留轻量分组，不做重型筛选台账：

1. `全部`
2. `失败`
3. `等待中`
4. `成功`

## 10. 最小后端补强

### 10.1 增强原则

本轮后端只做“让 `最近记录` 更像诊断清单”的最小增强，不新增第二套聚合接口。

现有接口与归属保持不变：

- Controller：`spring-boot-iot-device/.../DeviceMessageFlowController`
- VO：`spring-boot-iot-device/.../MessageFlowRecentSessionVO`
- 路径：`GET /api/device/message-flow/recent`

### 10.2 建议新增的可选字段

在 `MessageFlowRecentSessionVO` 中新增以下可选字段：

1. `verdict`
   - `validated / pending / failed / degraded`
2. `timelineStatus`
   - `available / missing / expired / pending`
3. `failureStage`
   - 最近一次明确失败阶段；非失败场景为空
4. `diagnosticReason`
   - 面向前端直出的短结论，例如：
     - `等待 trace 绑定`
     - `时间线已过期`
     - `协议解析失败`
5. `recommendedAction`
   - `message-trace / system-log / file-debug / reporting`
6. `recommendedActionLabel`
   - 用户可见动作文案

### 10.3 为什么只补这些字段

这 6 个字段已经足够让前端把 recent 记录渲染成诊断清单：

1. 能直接展示结论。
2. 能显示失败阶段或等待原因。
3. 能给出下一步动作。
4. 不需要每条 recent 再触发额外详情查询。

同时又不会把 `/recent` 变成第二个超大详情接口。

## 11. 前端实现拆分建议

本轮不建议继续把所有增强堆回 `ReportWorkbenchView.vue`。建议拆为以下结构：

### 11.1 页面内组件

1. `ReportingStatusHeader`
   - 统一承接状态结论、卡点、动作和上下文带
2. `ReportingReplayPanel`
   - 承接结果复盘主舞台
3. `ReportingSimulatePanel`
   - 承接设备查询、发送配置、payload 编辑和发送动作
4. `ReportingRecentDiagnosisList`
   - 承接最近记录诊断清单

### 11.2 composable / 纯逻辑模块

1. `useReportingDiagnosis`
   - 统一归并页面状态模型、文案和动作建议
2. `useReportingRecentDiagnosis`
   - 把 recent 返回体映射成诊断清单项
3. 保留现有：
   - `reportWorkbenchState.ts`
   - `reportPayloadFrame.ts`

### 11.3 组件拆分目标

拆分后应满足：

1. 状态判断逻辑只有一套权威实现。
2. 结果复盘与最近记录不再各自复制 verdict 判定。
3. 视觉调整主要发生在组件层，不继续加重主页面文件。

## 12. 数据流与交互闭环

### 12.1 发送闭环

1. 用户在 `模拟验证` 页签查询设备并发送。
2. 发送成功后写入统一 `当前诊断上下文`。
3. 页面自动切换到 `结果复盘`。
4. `结果复盘` 基于 `sessionId / traceId` 拉取 timeline，并实时刷新 `诊断状态头`。
5. recent 列表同步刷新。

### 12.2 最近记录闭环

1. `最近记录` 拉取 `/api/device/message-flow/recent`。
2. 前端把每条 recent 记录映射成诊断清单项。
3. 点击“恢复”后：
   - 设置当前诊断上下文
   - 切回 `结果复盘`
   - 以该 session 为主键继续拉取 timeline

### 12.3 跨页联动闭环

跨页仍复用现有 `DiagnosticContext + buildDiagnosticRouteQuery + persistDiagnosticContext`：

1. `链路追踪`
2. `异常观测`
3. `数据校验`

本轮不重新发明第二套联动协议。

## 13. 错误态与降级态设计

本轮必须明确区分以下 5 类状态，避免用户误判：

1. `等待回流`
   - 是处理中，不是失败
2. `时间线缺失`
   - 是复盘受限，不等于发送失败
3. `Redis / message-flow 查询异常`
   - 是环境或依赖阻塞，不等于业务失败
4. `发送失败`
   - 才是真正的提交失败
5. `时间线过期`
   - 是证据过期，不是链路未执行

对应要求：

1. 文案必须不同。
2. 颜色语义必须不同。
3. 动作建议必须不同。
4. 不允许把上述 5 类情况都折叠成“暂无处理时间线”这一种模糊空态。

## 14. 视觉语言

### 14.1 总体方向

视觉方向定义为：

`扁平商业风 + 2026 成熟互联网诊断控制台感`

具体含义：

1. 结果优先于装饰。
2. 信息密度高，但版面有主次。
3. 说明文字更短。
4. 卡片更少、层次更清楚。

### 14.2 保持不变的部分

1. 继续使用现有 `brand / accent / panel-border / text-*` token。
2. 继续使用浅背景、白卡、蓝灰正文、品牌橙主动作色。
3. 不新增页面私有主题变量。

### 14.3 本轮应强化的部分

1. `诊断状态头` 做成一体化扁平板块，而不是三张独立 KPI 卡。
2. 真页签更轻，像业务 mode switch，而不是厚 pill。
3. `结果复盘` 的 timeline 和结论区成为视觉主角。
4. `最近记录` 更像诊断列表，而不是高而松散的大卡片流。
5. 长说明收口成短句，避免继续堆“解释墙”。

## 15. 测试与验收建议

### 15.1 前端定向测试

至少覆盖以下内容：

1. `useReportingDiagnosis`
   - `idle`
   - `validated`
   - `pending`
   - `failed`
   - `degraded`
2. `useReportingRecentDiagnosis`
   - success
   - pending
   - failureStage
   - timeline missing
   - expired
3. 页签行为
   - 默认进入 `结果复盘`
   - 发送成功后自动切换到 `结果复盘`
   - 切换页签不丢当前上下文
4. 状态头动作
   - 不同 verdict 下动作数量和目标正确

### 15.2 后端定向测试

若补 `MessageFlowRecentSessionVO` 字段，至少覆盖：

1. failed session 的 `failureStage / diagnosticReason`
2. pending session 的 `verdict / timelineStatus`
3. trace 缺失但 correlation pending 超时场景
4. recent 记录在 timeline 存在、缺失、过期三种情况下的差异化返回

### 15.3 真实环境验收重点

实现阶段应继续以真实环境为准，重点核对：

1. HTTP 发送后是否自动切换到 `结果复盘`
2. MQTT pending 是否稳定显示 `等待回流`
3. trace 绑定成功后状态头是否从 `等待回流` 切换为 `验证成功`
4. recent 记录是否能稳定表达 `成功 / 等待 / 失败 / 受限`
5. 跳转 `链路追踪 / 异常观测 / 数据校验` 时上下文是否不断裂
