# 风险对象中心组织归属与自动编号设计

> 日期：2026-04-01
> 主题：优化 `/risk-point` 风险对象中心，新增独立组织归属字段、风险点编号自动生成，并修复新增/编辑/绑定设备/删除主链路

## 1. 背景

当前风险对象中心已经具备风险点主档和设备绑定的基础页面、接口与数据库表，但存在三类直接影响验收的问题：

- 风险点编号仍要求人工录入，新增时缺少统一生成规则，容易出现命名不一致和重复冲突。
- 风险点表单里的“区域”仍是手工输入的纯文本，未复用平台治理中的组织主数据，也无法承接后续“按组织范围看风险点”的权限治理。
- 风险对象中心的数据列表虽然展示了新增、编辑、绑定设备、删除等动作，但当前链路存在“点击后不生效或失败无感知”的体验问题，难以在真实环境下稳定验收。

结合当前仓库与文档现状，可以确认以下事实：

- `risk_point` 当前只有 `region_id`、`region_name`，没有独立的组织归属字段。
- `sys_organization` 已经是平台治理里的独立主数据，并且已有树形接口可供下拉选择复用。
- `risk_point` 继续承担风险对象主档语义，后续数据权限若按组织范围落地，需要有明确的组织挂点，不能继续把组织语义混入 `region`。
- 风险对象中心当前页面 `spring-boot-iot-ui/src/views/RiskPointView.vue` 仍把编号与区域都做成用户手填字段，同时多个失败分支只记录 `console.error`，没有统一的用户提示。

本轮目标是在不扩大到完整组织数据权限改造的前提下，先把风险点主档模型、页面录入方式和核心动作链路收口稳定。

## 2. 目标

- 风险点主档新增独立“所属组织”字段，作为后续组织范围权限治理的承接点。
- 风险点编号改为新增时由后端自动生成，保存成功后固定不变，不再要求人工录入。
- 风险对象中心主页面把“区域”主语义收口为“所属组织”，列表和抽屉都统一展示组织归属。
- 新增、编辑、绑定设备、删除四条链路恢复为可感知、可提示、可回归的稳定操作。
- 历史风险点允许暂时没有组织归属，但新增数据和编辑后的数据必须补齐所属组织。
- 为下一轮组织范围数据权限预留查询和数据模型挂点，但本轮不直接实现按登录人组织范围过滤。

## 3. 非目标

- 本轮不实施完整的组织范围数据权限，不区分“本组织 / 下级组织 / 跨组织角色 / 超级管理员”差异规则。
- 本轮不移除 `risk_point.region_id`、`risk_point.region_name`，也不把 GIS、风险监测等地理语义同步切到组织字段。
- 本轮不对历史风险点执行批量组织回填或强制阻断历史数据编辑。
- 本轮不扩展新的风险点导入、批量编辑或批量绑定设备能力。
- 本轮不重做风险对象中心的整体布局，只在现有共享列表/抽屉合同内修正字段和交互。

## 4. 已确认约束

本轮设计基于以下已确认决策：

1. 风险点新增独立组织归属字段，组织语义不复用现有 `region_id`、`region_name`。
2. 风险点编号在“新增”时自动生成，保存后固定不变；后续编辑名称、组织、风险等级时不回写历史编号。
3. 本轮只为后续组织权限做数据模型和查询承接，不直接落地登录人组织范围过滤。
4. 风险对象中心主列表和抽屉只展示“所属组织”，不在本轮主界面同时展示地理区域。
5. 历史风险点不做组织字段自动回填；若旧数据缺少组织归属，则先允许展示，并提示“未配置组织”。

## 5. 方案对比

### 5.1 方案 A：继续复用 `region` 表达组织

- 不新增字段，直接把当前页面里的“区域”改名为“所属组织”。
- 前端下拉选择组织树，但仍把结果写回 `region_id`、`region_name`。
- 编号自动生成和动作修复同步完成。

优点：

- 数据库改动最少。
- 页面改造较快。

缺点：

- 把“组织范围”与“地理区域”混成同一语义，后续 GIS、区域筛选、组织权限都会冲突。
- 文档和数据库含义不一致，长期维护成本高。

### 5.2 方案 B：新增独立组织归属字段并修复主链路

- 在 `risk_point` 新增 `org_id`、`org_name`。
- 保留现有 `region_id`、`region_name` 供地理语义继续使用。
- 风险点编号由后端按“风险点名称 + 所属组织 + 风险等级”自动生成。
- 风险对象中心主列表、抽屉、接口和文档统一切到“所属组织”语义。
- 同步修复新增、编辑、绑定设备、删除动作。

优点：

- 组织范围与地理区域语义清晰分离。
- 便于后续继续做组织权限，不会返工。
- 本轮范围仍聚焦在风险点域，可控。

缺点：

- 涉及数据库、后端、前端和文档的联动修改。

### 5.3 方案 C：在方案 B 基础上直接做组织权限

- 除方案 B 外，同轮把风险点列表与详情查询收口到“登录人只看本组织及下级组织”。

优点：

- 组织归属和数据可见性一步到位。

缺点：

- 会扩大到鉴权上下文、组织树范围解析、角色差异处理与更多真实环境回归。
- 与本轮“先修复主链路”的目标相比改动面过大。

## 6. 选型

本轮采用方案 B。

原因：

- 用户已明确接受“新增独立组织字段、自动编号只在新增时生成、权限延后、页面主视图只展示组织、历史数据不强制回填”的边界。
- 当前最紧迫的问题是风险对象中心主链路不可稳定使用，而不是一次性完成权限体系。
- 方案 B 既解决本轮录入、展示、交互问题，又为后续权限治理留下干净的数据模型。

## 7. 数据模型设计

### 7.1 `risk_point`

新增字段：

- `org_id BIGINT DEFAULT NULL COMMENT '所属组织ID'`
- `org_name VARCHAR(128) DEFAULT NULL COMMENT '所属组织名称'`

保留字段：

- `region_id`
- `region_name`

约束口径：

- `org_id`、`org_name` 本轮允许历史存量为空。
- 新增风险点时必须写入 `org_id`、`org_name`。
- 编辑风险点时若目标记录缺少组织归属，提交前必须补齐。
- `tenant_id + risk_point_code` 唯一索引继续保留，编号唯一性仍按租户维度约束。

### 7.2 文档口径

- `risk_point.org_id`、`risk_point.org_name` 表达组织归属。
- `risk_point.region_id`、`risk_point.region_name` 继续表达区域/地理语义。
- 风险对象中心本轮主界面默认使用组织归属，不再把 `region` 当作主维护字段。

## 8. 后端设计

### 8.1 职责边界

- `spring-boot-iot-alarm`
  - 负责风险点主档、自动编号生成、风险点 CRUD、设备绑定/解绑和风险点分页查询。
- `spring-boot-iot-system`
  - 继续提供组织树/组织列表查询能力，不承接风险点业务规则。

### 8.2 风险点新增

新增时服务层处理顺序：

1. 校验 `riskPointName`、`orgId`、`riskLevel` 必填。
2. 根据 `orgId` 查询组织，确认组织存在且状态可用。
3. 以组织主数据回填 `orgName`，不信任前端直传文案。
4. 根据“风险点名称 + 所属组织 + 风险等级”生成候选基础码。
5. 在当前租户下检测是否冲突，冲突则追加递增序号，直到得到唯一编号。
6. 写入 `riskPointCode`、`orgId`、`orgName` 及其他业务字段。

### 8.3 风险点编辑

- 编辑时必须先查原记录。
- `riskPointCode` 不再按名称/组织/等级重新计算，沿用历史值。
- 若编辑请求带来新的 `orgId`，后端重新校验组织并回填最新 `orgName`。
- 若目标风险点不存在或已删除，返回明确 `BizException`。

### 8.4 风险点删除

- 继续沿用逻辑删除。
- 若删除前发现风险点不存在，返回明确提示。
- 本轮不额外设计“有绑定设备禁止删除”的强规则；若实现过程中确认真实业务需要此保护，应在服务层返回明确错误信息，而不是静默失败。

### 8.5 设备绑定

绑定时服务层继续负责：

- 校验风险点存在。
- 校验设备存在。
- 校验 `metricIdentifier` 非空。
- 拦截重复绑定。
- 对失败原因使用明确 `BizException`，供前端提示。

### 8.6 风险点查询

本轮查询层变化：

- 分页与列表返回结果中增加 `orgId`、`orgName`。
- 本轮不直接按登录人组织范围过滤。
- 但查询结构需要保留后续扩展点，便于下一轮在服务层追加组织范围过滤条件。

## 9. 风险点编号生成规则

### 9.1 输入因子

- `riskPointName`
- `orgName`
- `riskLevel`

### 9.2 生成原则

- 由后端统一生成，前端不再参与正式编码算法。
- 编码需要可读，但不强求把原始中文完全可逆映射到编码文本。
- 生成前统一进行规范化处理，例如去掉多余空格、统一分隔符、转成稳定的码段表达。
- 基础码不足以保证唯一时，追加同租户范围内递增尾号。

### 9.3 生命周期

- 仅新增时生成。
- 保存成功后固定。
- 编辑风险点名称、组织、风险等级时不改编号。

### 9.4 前后端分工

- 后端：唯一正式生成者。
- 前端：新增表单只显示“保存后自动生成”提示，不提供手输入口。

## 10. 前端设计

### 10.1 主列表

风险对象中心主列表调整为：

- 继续展示 `riskPointCode`、`riskPointName`、`riskLevel`、`responsiblePhone`、`status`、`createTime`。
- 原“区域”列改为“所属组织”，展示 `orgName`。
- 若 `orgName` 为空，统一显示固定文案“未配置组织”，帮助识别存量数据。

### 10.2 新增/编辑抽屉

字段口径：

- `风险点编号`
  - 改为只读字段。
  - 新增时显示“保存后自动生成”。
  - 编辑时显示当前编号。
- `风险点名称`
  - 仍为可编辑必填字段。
- `所属组织`
  - 替代当前手输区域输入框。
  - 数据来源改为平台治理组织树接口，页面按树形下拉选择组织。
- `风险等级`
  - 保持必选。
- `负责人电话`、`状态`、`描述`
  - 保持现有录入逻辑。

交互口径：

- 新增时不允许人工修改编号。
- 编辑时只展示历史编号，不允许覆盖。
- 若目标记录缺少组织归属，则编辑提交前必须补选组织。

### 10.3 绑定设备抽屉

- 打开抽屉时先加载设备选项。
- 选择设备后联动加载测点选项。
- 提交前校验风险点、设备、测点均有效。
- 对“设备列表加载失败”“测点加载失败”“绑定失败”统一给出可见提示。

### 10.4 行内动作恢复

当前页面需要明确修复以下体验问题：

- 新增按钮点击后必须稳定打开抽屉。
- 编辑按钮点击后必须能带出当前记录，并允许提交。
- 绑定设备按钮点击后必须加载设备与测点流程。
- 删除按钮确认后必须给出成功或失败反馈。

实现原则：

- 页面不能只保留 `console.error`。
- 所有失败分支都需要有用户可见的错误提示。
- 成功后要刷新列表并保证抽屉正确关闭/重置。

## 11. 错误处理设计

### 11.1 表单层

- 风险点名称必填。
- 所属组织必填。
- 风险等级必填。
- 绑定设备时设备和测点必填。

### 11.2 服务层

统一使用 `BizException` 表达业务错误，例如：

- 所属组织不存在
- 所属组织已停用
- 风险点不存在
- 风险点编号冲突
- 设备不存在
- 测点不能为空
- 设备已绑定到该风险点

### 11.3 页面反馈层

- 新增、编辑、绑定设备、删除失败都必须弹出明确提示。
- 加载组织、设备、测点失败都必须弹出明确提示。
- 不允许只在控制台打印而无用户反馈。

## 12. 存量数据策略

- 本轮不做历史风险点 `org_id`、`org_name` 的批量回填。
- 已存在且组织为空的风险点允许继续在列表中展示。
- 历史记录编辑时要求补齐组织。
- 历史记录若未编辑，可继续保留现状，避免错误把区域映射成组织。

## 13. 测试与验收设计

### 13.1 后端验证

至少覆盖以下逻辑：

- 新增风险点时自动生成编号。
- 同租户下基础码冲突时能生成唯一尾号。
- 编辑风险点不会改写历史编号。
- 新增/编辑时组织校验生效。
- 重复绑定设备被正确拦截。

### 13.2 前端验证

至少补齐一组风险对象中心页面回归，覆盖：

- 新增按钮能打开抽屉。
- 编辑、绑定设备、删除行内动作能触发对应流程。
- 抽屉字段从“区域手输 + 编号手输”切换为“组织选择 + 编号只读”。
- 失败时会显示用户可见提示，而不是静默。

### 13.3 真实环境验收

本轮真实环境最小验收链路：

1. 新增风险点，编号自动生成成功。
2. 编辑风险点，不改历史编号。
3. 为风险点绑定设备与测点成功。
4. 删除一条可删除风险点成功。

若真实环境组织数据、设备数据或权限访问受阻，需要明确记录为环境阻塞，不回退到废弃验收路径。

## 15. 风险与缓解

### 15.1 组织与区域语义混用风险

缓解：

- 在数据库和文档上显式分离 `org_*` 与 `region_*`。
- 页面主口径统一使用“所属组织”。

### 15.2 历史数据缺少组织导致页面异常

缓解：

- 允许空值展示。
- 用明确占位提示识别存量数据。
- 编辑时再要求补齐。

### 15.3 自动编号规则不稳定

缓解：

- 编码生成集中在后端。
- 只在新增时生成一次，避免因编辑导致历史编号漂移。

### 15.4 按钮“看似无反应”再次回流

缓解：

- 页面所有操作失败分支补齐显式提示。
- 前后端各补一轮最小回归测试。

## 16. 实施边界总结

本轮实施应聚焦三件事：

1. 给风险点主档补齐组织归属数据模型。
2. 把风险点编号改成新增时由后端自动生成且终身固定。
3. 修复风险对象中心新增、编辑、绑定设备、删除四条主链路的可用性与可感知反馈。

本轮不应扩大到：

- 组织范围数据权限
- 地理区域语义重构
- 历史数据批量清洗
- 风险对象中心的大规模页面重构
