# 接入智维自动化测试专项设计

> 日期：2026-04-05
> 范围：`config/automation`、`scripts/auto`、`scripts/run-business-function-smoke.ps1`、`spring-boot-iot-ui`、`docs/05`、`docs/21`
> 主题：为接入智维 6 个现有入口建立一套最小可交付的研发专项自动化基线
> 状态：设计已确认，已完成自检，待用户审阅

## 1. 背景

当前 `spring-boot-iot` 已经形成两条与自动化相关的基础链路：

1. `接入智维` 负责承接设备接入、产品设备建档、链路验证、异常排查与数据校验。
2. `质量工场` 负责承接研发自动化资产、统一执行编排和结果证据复盘。

其中，`接入智维` 当前的核心入口已经明确收敛为 6 个现有页面：

1. `/products` 产品定义中心
2. `/devices` 设备资产中心
3. `/reporting` 链路验证中心
4. `/system-log` 异常观测台
5. `/message-trace` 链路追踪台
6. `/file-debug` 数据校验台

同时，仓库已经具备一套可复用的自动化底座：

1. `config/automation/acceptance-registry.json` 统一注册表
2. `config/automation/sample-web-smoke-plan.json` 浏览器配置驱动计划
3. `scripts/auto/run-acceptance-registry.mjs` 统一执行入口
4. `scripts/run-business-function-smoke.ps1` 业务/API 冒烟脚本
5. `scripts/run-message-flow-acceptance.py` 消息链路回归
6. `质量工场 -> 研发工场 / 执行中心 / 结果与基线中心` 的结果沉淀与复盘入口

但目前这套底座还没有形成一份明确的 **接入智维专项自动化计划**。现状问题主要有四个：

1. 现有自动化场景对接入智维已有部分覆盖，但仍是“通用场景散点挂接”，不是“接入智维专项分组”。
2. `/products`、`/devices`、`/reporting`、`/message-trace` 已有较多回归能力，但 `/system-log`、`/file-debug` 在专项层面没有形成同一口径的首轮目标。
3. 当前没有一份清晰文档说明：接入智维首轮自动化到底测哪些主链路、不测哪些深水区、由谁维护、如何验收。
4. 若直接追求全量覆盖，容易把“最小可交付专项”做成一轮范围失控的大工程。

因此，本轮设计目标不是再造第二套自动化平台，也不是一次做全，而是先为接入智维建立一套 **最小可交付、研发可持续维护、结果可统一复盘** 的专项自动化基线。

## 2. 目标

1. 只针对 `接入智维` 的 6 个现有入口，建立首轮专项自动化计划。
2. 以“主链路回归优先”为原则，先覆盖页面可进入、关键查询、核心动作和结果核对。
3. 复用现有 `browserPlan / apiSmoke / messageFlow / acceptance-registry` 底座，不新增第二套执行模型。
4. 让接入智维专项结果继续沉淀到现有 `registry-run-*.json / .md` 和结果与基线中心。
5. 明确首轮纳入范围、首轮不纳入范围、角色分工和完成定义。

## 3. 非目标

1. 本轮不扩展到风险运营、风险策略、平台治理或质量工场其他模块。
2. 本轮不把接入智维总览、失败归档兼容模式或未登记诊断衍生路径额外纳入专项范围。
3. 本轮不追求全按钮、全筛选、全权限、全视觉和全异常分支自动化。
4. 本轮不新增新的后端调度服务、数据库归档表或第二套结果中心。
5. 本轮不把研发专项计划直接包装成业务验收台消费入口。

## 4. 已确认决策

对话中已经明确确认以下边界：

1. 本次专项计划只覆盖 6 个现有入口，不扩项。
2. 计划深度选择“最小可交付计划”，不是完整全量计划。
3. 采用“主链路优先、分层补齐”的方案，不做页面全量平铺，也不只做接口层。
4. 第一阶段交付重点是：
   - 每页至少 1 条稳定主用例
   - 3 条跨入口联动链路
   - 统一执行与统一结果沉淀
5. 业务、产品、项目经理不是这份专项计划的直接使用者；首轮主要服务研发、测试和运维协同排障。

## 5. 方案对比与决策

### 5.1 方案 A：主链路优先、分层补齐

结构：

1. 把 6 个入口拆为 3 层：
   - 资产层：`/products`、`/devices`
   - 验证层：`/reporting`
   - 诊断层：`/system-log`、`/message-trace`、`/file-debug`
2. 每个入口只纳入“可进入、可查询、核心动作可执行、结果可核对”的首轮断点
3. 统一复用现有执行器和结果体系

优势：

1. 能最快形成可交付专项基线
2. 与现有质量工场和注册表体系兼容最好
3. 易于后续继续做 Phase 2 深化

代价：

1. 首轮不会覆盖全部深水区和全部异常分支
2. 需要明确文档边界，防止被误解为“全量自动化已完成”

### 5.2 方案 B：页面全量平铺覆盖

结构：

1. 对 6 个入口逐页列出所有按钮、筛选、抽屉、异常提示、导出和详情
2. 第一轮直接规划全量页面自动化

问题：

1. 首轮工作量过大
2. 易把“可交付基线”和“长期增强项”混在一起
3. 专项落地节奏慢，难以形成第一版稳定产物

### 5.3 方案 C：接口优先、页面从简

结构：

1. 重点补 API smoke / contract / message-flow
2. 页面只保留导航和最小到达性校验

问题：

1. 对“接入智维”这种强工作台属性模块来说，页面主价值覆盖不足
2. 很难回答“接入智维页面是否能作为一线诊断与联调用台稳定使用”

### 5.4 最终决策

采用 **方案 A：主链路优先、分层补齐**。

即：

1. 先形成接入智维专项基线
2. 每页只锁高价值主用例
3. 跨入口联动优先覆盖主排查链路
4. 结果统一沉淀，不新增第二套自动化体系

## 6. 目标范围与覆盖矩阵

### 6.1 目标范围

本轮专项只覆盖以下 6 个入口：

1. `/products`
2. `/devices`
3. `/reporting`
4. `/system-log`
5. `/message-trace`
6. `/file-debug`

### 6.2 覆盖矩阵

| 入口 | 首轮自动化目标 | 推荐自动化方式 | 首轮纳入 | 首轮暂不纳入 |
|---|---|---|---|---|
| `/products` 产品定义中心 | 保证产品台账主链路可回归 | `browserPlan + apiSmoke` | 页面可进入、快速查询可用、新增产品成功、列表可回显、详情或治理入口可打开 | 全量物模型治理 `compare/apply`、导出、全部筛选组合、全部行级动作 |
| `/devices` 设备资产中心 | 保证设备主档与未登记入口可回归 | `browserPlan + apiSmoke` | 页面可进入、主档查询可用、新增设备成功、详情可打开、未登记名单切换可用 | 批量导入、设备更换、批量删除、全量拓扑治理、全部数据权限分支 |
| `/reporting` 链路验证中心 | 保证接入验证主链路可回归 | `browserPlan + apiSmoke + messageFlow` | 设备编码反查、HTTP 上报或 MQTT 模拟上报、结果状态回显、时间线或最近记录最小复盘 | 所有 payload 编码变体、所有协议分支、所有密文/二进制细节、全部异常文案 |
| `/system-log` 异常观测台 | 保证异常排查入口可回归 | `browserPlan + apiSmoke` | 页面可进入、列表查询可用、按关键字段筛选、详情可打开、可跳转链路追踪 | 删除、导出、全部组合筛选、统计口径全量校验 |
| `/message-trace` 链路追踪台 | 保证 Trace 排障主链路可回归 | `browserPlan + messageFlow` | 按 TraceId 或设备编码检索、详情可打开、Payload 对照可见、时间线降级语义可见 | 全部过滤器组合、复制细节、全部 recent session 辅助能力 |
| `/file-debug` 数据校验台 | 保证最小校验链路可回归 | `browserPlan` | 页面可进入、输入校验参数、返回原始响应、异常反馈可见 | 深层格式比对、复杂多场景样本、跨设备批量校验 |

### 6.3 覆盖矩阵背后的判断

1. `产品 -> 设备 -> 上报` 是接入智维的最小业务闭环，因此应作为第一优先级。
2. `异常观测 -> 链路追踪 -> 数据校验` 是诊断闭环，因此首轮以“能到达、能打开、能回看”为核心，不做穷举式页面细节覆盖。
3. 首轮应尽量复用现有成熟能力，不为单个专项再造工具链或结果格式。

## 7. 实施阶段规划

本轮设计建议分为两个阶段，但只把 `Phase 1` 作为当前交付目标；`Phase 2` 仅保留为后续增强储备。

### 7.1 Phase 1：最小可交付自动化基线

#### 7.1.1 自动化目录建模

在统一注册表中为接入智维新增专项分组，不再只靠零散通用场景覆盖。

推荐分组方式：

1. 资产组：`products`、`devices`
2. 验证组：`reporting`
3. 诊断组：`system-log`、`message-trace`、`file-debug`

每组都应明确：

1. 场景编码
2. 页面或链路目标
3. 执行器类型
4. 证据类型
5. 阻断级别

#### 7.1.2 六个入口一页一条主用例

`/products`

1. 产品查询
2. 产品新增
3. 详情或治理入口可打开

`/devices`

1. 设备查询
2. 设备新增
3. 详情可打开或未登记名单可切换

`/reporting`

1. 设备编码反查
2. HTTP 上报或 MQTT 模拟上报至少跑通一条
3. 最小结果复盘可见

`/system-log`

1. 列表可加载
2. 关键字段筛选可用
3. 详情可打开

`/message-trace`

1. 按 TraceId 或设备编码检索
2. 详情可打开
3. Payload 或时间线最小信息可见

`/file-debug`

1. 页面可进入
2. 提交一次校验
3. 原始响应或错误反馈可见

#### 7.1.3 三条跨入口联动链路

1. `产品 -> 设备`
   - 目标：证明产品建档与设备建档主链路未断
2. `上报 -> 链路追踪`
   - 目标：证明上报结果和链路追踪证据可以串联
3. `异常观测 -> 链路追踪`
   - 目标：证明异常排查和 Trace 复盘可以串联

#### 7.1.4 结果沉淀与回归闭环

1. 所有场景必须进入 `registry-run-*.json / .md`
2. 必须能够从执行中心触发，并在结果与基线中心查看
3. 失败结果至少要回答：
   - 哪个入口失败
   - 失败在哪个主动作
   - 属于页面失败、接口失败还是环境阻塞

### 7.2 Phase 2：增强覆盖

以下内容保留为后续增强储备，不并入本轮完成定义：

1. `/products` 物模型治理 `compare/apply` 深水区
2. `/devices` 批量导入、设备更换、批量删除
3. `/reporting` 多协议、多编码和更多异常分支
4. `/system-log` 全量统计、删除、导出
5. `/message-trace` recent session、复杂筛选、复制细节
6. `/file-debug` 多样本、多格式、批量校验
7. 更细的权限、组织范围、租户范围回归
8. 更细的视觉基线和页面断言

## 8. 执行方式与角色分工

### 8.1 执行方式

本专项不建议新起一套自动化链路，执行方式统一复用现有质量工场与注册表体系。

统一执行原则：

1. 页面主动作优先走 `browserPlan`
2. 关键接口成功/失败口径补 `apiSmoke`
3. 上报到追踪的链路证据补 `messageFlow`
4. 所有专项运行统一收口到 `run-acceptance-registry.mjs`
5. 所有结果统一进入 `registry-run-*.json / .md` 与结果与基线中心

按入口推荐执行器：

| 入口 | 推荐执行器 |
|---|---|
| `/products` | `browserPlan` 为主，必要时补 `apiSmoke` |
| `/devices` | `browserPlan` 为主，必要时补 `apiSmoke` |
| `/reporting` | `browserPlan + apiSmoke + messageFlow` |
| `/system-log` | `browserPlan + apiSmoke` |
| `/message-trace` | `browserPlan + messageFlow` |
| `/file-debug` | `browserPlan` |

### 8.2 角色分工

开发人员：

1. 负责专项场景建模、脚本维护、断言维护和结果判读规则维护
2. 负责把新页面和新主动作纳入注册表或浏览器计划
3. 负责区分功能缺陷、环境阻塞和断言失效

测试人员：

1. 负责审核覆盖矩阵是否满足提测与回归要求
2. 负责补充关键失败样本和复验清单
3. 负责对专项运行结果做复核，但不主导底层页面脚本实现

运维人员：

1. 不作为专项自动化的主要维护者
2. 负责共享环境、MQTT、Redis、数据库等依赖可用性协助
3. 在 `reporting -> message-trace`、`system-log -> message-trace` 排障链路中提供环境支持

产品 / 业务 / 项目经理：

1. 不参与专项脚本维护
2. 只确认专项覆盖哪些主链路、哪些结果算通过、哪些结果算阻塞
3. 若后续需要业务消费入口，应由研发选择稳定场景下沉到业务验收台

### 8.3 维护边界

`接入智维自动化测试专项` 的定位是：

1. 研发/测试使用的专项回归计划
2. 运维可辅助使用的排障回归基线
3. 不是直接给业务人员操作的验收工作台

后续若其中部分场景足够稳定，才应考虑下沉到 `/business-acceptance` 作为业务消费包，而不是直接让业务角色进入研发专项编排入口。

## 9. 验收标准与里程碑

### 9.1 验收标准

验收标准 1：范围验收

1. 只覆盖以下 6 个入口：
   - `/products`
   - `/devices`
   - `/reporting`
   - `/system-log`
   - `/message-trace`
   - `/file-debug`

验收标准 2：自动化纳管验收

1. 6 个入口均纳入统一自动化目录
2. 每个入口至少 1 条稳定主用例
3. 至少 3 条跨入口联动链路纳入专项计划

验收标准 3：执行与结果验收

1. 场景可通过统一注册表执行
2. 结果能落到 `registry-run-*.json / .md`
3. 失败能在结果与基线中心复盘，而不是只剩零散日志

验收标准 4：维护与文档验收

1. 研发、测试、运维职责边界明确
2. 文档明确首轮纳入范围、暂不纳入范围和后续增强范围
3. 新增入口或新主动作时，知道该补到哪类场景、哪类执行器、哪类结果口径

### 9.2 里程碑建议

`M1：专项建模完成`

1. 完成分组方式、场景命名、执行器映射和覆盖矩阵

`M2：六个入口主用例落地`

1. 6 个入口各完成 1 条主用例

`M3：跨入口联动链路落地`

1. 完成 `产品 -> 设备`
2. 完成 `上报 -> 链路追踪`
3. 完成 `异常观测 -> 链路追踪`

`M4：结果沉淀与交付收口`

1. 执行入口、结果复盘和文档口径全部打通

## 10. 风险与约束

本轮落地的最大风险不是技术难度，而是范围失控。需要重点防止以下问题：

1. 把“最小可交付专项”做成“大而全全量自动化工程”
2. 为接入智维单独发明第二套执行器、第二套结果格式
3. 把业务验收台和研发专项计划混成同一个入口
4. 把环境阻塞、共享样本问题误判成专项自动化能力缺陷

同时，仍需接受以下现实约束：

1. 当前结果中心主要依赖 `logs/acceptance` 台账，而不是集中式 CI 归档
2. 当前已有场景更偏接入主链路，不是所有诊断页都已有成熟脚本可直接复用
3. 首轮应优先复用现有 `sample-web-smoke-plan` 与统一注册表能力，不宜同时引入大量新 DSL 或新抽象

## 11. 最终建议落地结构

若按本设计推进，建议后续文档与实施顺序如下：

1. 当前先保留本设计稿：
   - `docs/superpowers/specs/2026-04-05-iot-access-automation-design.md`
2. 用户确认设计稿后，再生成实施计划：
   - `docs/superpowers/plans/2026-04-05-iot-access-automation-implementation-plan.md`
3. 实施完成后，再把长期有效结论回写到权威文档：
   - `docs/05-自动化测试与质量保障.md`
   - `docs/21-业务功能清单与验收标准.md`
   - 必要时补 `docs/08-变更记录与技术债清单.md`

## 12. 一句话定义

为接入智维 6 个现有入口建立一套最小可交付的研发专项自动化基线，优先覆盖主链路回归、跨入口联动和结果复盘，不追求首轮全量覆盖，不新增第二套自动化体系。
