# 对象洞察趋势指标历史残留标识温和治理设计

## 背景

在产品定义中心的契约字段工作区，用户将“当前已生效字段”设为对象洞察趋势指标时，后端会校验 `metadataJson.objectInsight.customMetrics[].identifier` 必须命中“已发布合同标识符”。

在 `zhd-monitor-multi-displacement-v1` 等产品上，历史配置中存在短标识残留（例如 `gX`），而当前发布口径已是全路径（例如 `L1_JS_1.gX`）。用户本次操作虽然选择了已生效字段，但因为旧数据混入同一次保存 payload，触发如下报错：

- `对象洞察指标必须使用已发布合同标识符: gX`

这属于“存量残留导致当前写入失败”的治理体验问题，不应继续要求用户手工清理每个历史脏项。

## 目标

1. 在不放宽“已发布标识符”治理边界的前提下，消除历史残留导致的保存阻断。
2. 允许“当前值覆盖历史值”：用户本次保存时，旧短标识自动替换为当前发布标识。
3. 保持现有读写主链路、模块边界和发布合同口径不变。

## 非目标

1. 不引入新的对象洞察配置表或平行草稿表。
2. 不改变发布合同的 canonical 规则。
3. 不做读侧长期双轨兼容（短标识和全路径长期并存）。

## 方案选型

### 方案 A：只做一次性清理

- 优点：实现最快。
- 缺点：后续仍可能因新残留或回滚数据再次触发同类问题。

### 方案 B：一次性清理 + 保存时自动归一兜底（推荐）

- 优点：当前问题立即可解，且可持续防回流。
- 优点：改动集中在产品 metadata 归一化链路，影响面小。
- 缺点：需要新增少量单测和兜底分支。

### 方案 C：读写长期双兼容

- 优点：短期最“宽松”。
- 缺点：会削弱“已发布合同标识符”为唯一治理口径，长期数据再次分叉。

## 推荐方案（B）设计

### 1. 一次性历史清理

#### 范围

- 表：`iot_product`
- 字段：`metadata_json.objectInsight.customMetrics[].identifier`
- 首批目标产品：`zhd-monitor-multi-displacement-v1`（可参数化扩展到更多产品）

#### 规则

1. 短标识可唯一映射到当前发布标识时，自动替换。
2. 无法映射时标记为“待人工处理”，不盲改。
3. 存在多候选冲突时标记“冲突”，不自动落库。
4. 对同一指标的旧短标识和新全路径并存时，仅保留新全路径（当前值覆盖历史值）。

#### 执行模式

1. `dry-run`：只生成报告，不改库。
2. `apply`：按规则落库。

#### 输出物

1. 变更前备份清单（productId/productKey/旧 metadataJson）。
2. 变更结果清单（成功替换数、冲突数、人工处理数）。

### 2. 保存链路温和兜底（核心）

改动入口：`ProductServiceImpl.normalizeObjectInsightIdentifier(...)`

当前链路已具备以下能力：

1. 已发布快照解析（`MetricIdentifierResolver.resolveForGovernance`）。
2. `formalIdentifierMap` 的大小写归一。

本次新增一个“唯一后缀匹配兜底”分支：

1. 当 resolver 无法将输入标识符解析为已发布标识符时，尝试在 `snapshot.publishedIdentifiers()` 中做“后缀严格匹配”。
2. 仅当匹配结果唯一时，自动替换为该全路径发布标识符。
3. 匹配结果为 0 或大于 1 时，继续抛出现有 BizException，不放宽规则。

这样可将历史 `gX` 温和归一为 `L1_JS_1.gX`，同时避免误映射。

### 3. 覆盖历史值语义

利用现有 `deduplicateObjectInsightMetrics(...)` 与归一化链路，保证：

1. 本次保存后的 `customMetrics` 中只保留最终发布口径标识符。
2. 同名旧标识残留不会在下一次保存中再次参与校验。

## 错误处理与风险控制

### 错误处理

1. 无唯一映射：保持现有错误提示并附原始标识符，要求人工确认。
2. 映射冲突：拒绝自动替换，避免 silent corruption。
3. 非文本或空标识：沿用现有校验逻辑。

### 风险与缓解

1. 风险：后缀匹配误伤。
   - 缓解：仅“唯一命中”才替换；其余全部失败返回。
2. 风险：一次性清理误更新。
   - 缓解：强制 `dry-run -> 人工确认 -> apply`，并保留备份。
3. 风险：旧快照与当前发布口径不一致。
   - 缓解：继续优先依赖 `PublishedProductContractSnapshot`，不绕开发布真相。

## 测试设计

### 后端单测

在 `ProductServiceImplTest` 增加/调整用例：

1. 输入短标识 `gX`，发布集合包含 `L1_JS_1.gX`，且唯一后缀命中 -> 保存成功并归一为 `L1_JS_1.gX`。
2. 输入短标识无法命中任何发布标识 -> 维持原报错。
3. 输入短标识命中多个发布标识（冲突）-> 维持原报错。
4. 同时存在旧短标识和新全路径 -> 保存后仅保留全路径。

### 脚本验证

1. `dry-run` 不改库，仅输出报告。
2. `apply` 后重复执行 `dry-run`，应显示“无可迁移项”或仅剩人工项。

## 验收口径

1. 在 `zhd-monitor-multi-displacement-v1` 上设置 `L1_JS_1.gX` 趋势指标时，不再因历史 `gX` 阻断保存。
2. 保存后产品 metadata 中趋势指标标识统一为发布口径全路径。
3. 对无法唯一映射的残留项，系统仍明确阻断并提示人工处理。

1. 先交付清理脚本并完成 `dry-run` 报告。
2. 落保存兜底逻辑与单测。
3. 在目标产品执行 `apply` 清理并复测趋势配置保存。
4. 原位更新文档并记录变更证据。

## 结论

采用“历史清理 + 保存时唯一映射兜底”的最小增量方案，可在保持治理边界不变的前提下，实现用户感知上的温和处理：当前操作优先成功、旧残留自动收口、冲突场景仍安全阻断。
