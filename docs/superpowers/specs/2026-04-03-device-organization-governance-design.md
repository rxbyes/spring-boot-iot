# 设备归属机构与风险点唯一绑定治理设计

> 日期：2026-04-03
> 范围：`spring-boot-iot-device`、`spring-boot-iot-alarm`、`spring-boot-iot-system`、`spring-boot-iot-ui`、`sql/init.sql`、`sql/init-data.sql`
> 主题：把设备资产中心从“仅租户侧治理”升级为“设备归属机构落表 + 风险点唯一绑定 + 机构范围治理”的最小可用闭环
> 状态：设计已确认，待书面 spec 审阅

## 1. 背景

当前系统治理链路已经补齐了 `用户 -> 主机构 -> 角色 -> dataScopeType` 的第一轮身份模型，但设备资产中心仍停留在“按 `tenantId` 收口”的阶段，尚未形成真实的机构归属闭环。

结合当前仓库和文档，可以确认以下事实：

- `iot_device` 当前只有 `tenant_id`、`product_id` 等设备主档字段，没有稳定的 `org_id`、`org_name`。
- 设备新增、编辑、批量导入、设备更换当前都不会落设备归属机构，因此 `ORG / ORG_AND_CHILDREN` 无法在设备域真实执行。
- 风险对象中心 `risk_point` 已经拥有 `org_id / org_name`，并已经作为第一批组织范围治理入口。
- 当前风险点设备绑定链路没有把“一台设备只能绑定一个风险点”固化成强规则，设备候选清单也没有明确排除已绑定设备。
- 用户已明确要求：设备归属应由当前登录人的主机构决定；历史设备若 `org_id` 为空，应优先从已绑定风险点的所属机构反推；一台设备不能同时绑定多个风险点。

因此，本轮需要把“设备归属机构”正式落到 `iot_device`，并把风险点绑定唯一性一起纳入治理，避免后续设备资产中心和风险对象中心继续各自维护不同的机构事实来源。

## 2. 目标

- 在 `iot_device` 上落下稳定的 `org_id / org_name` 设备归属字段。
- 新增、批量新增、编辑、设备更换生成新设备时，统一自动继承当前登录人的主机构，不让前端手工选择归属机构。
- 历史 `iot_device.org_id` 为空的设备，优先按已绑定风险点的 `org_id / org_name` 回填。
- 把设备资产中心从“租户侧治理”升级为“租户 + 机构范围治理”。
- 把风险点绑定收口为“一设备一风险点”，并让已绑定设备不再出现在可绑定设备候选列表中。
- 保持现有页面风格和共享工作台语法，不新增新的视觉方案或第二套设备治理页面。

## 3. 非目标

- 本轮不引入“设备多机构归属”或“设备-机构关系表”。
- 本轮不在设备域伪造 `SELF=按责任人用户过滤` 的能力；设备域没有独立责任人字段时，不额外制造用户级权限语义。
- 本轮不顺手把同一套机构范围治理继续扩到告警、事件、报表等其他业务查询。
- 本轮不重做设备资产中心和风险对象中心的整体布局，只在现有共享页面合同内补齐字段、校验和提示。
- 本轮不把无风险点绑定、且未经过新治理链路重写的历史设备强行归到某个默认机构。

## 4. 已确认约束

对话中已经确认以下硬规则：

1. 设备归属由**当前登录人的主机构**决定，不由前端显式选择。
2. 历史设备 `org_id` 为空时，应优先从“所属风险点的所属机构”反推。
3. 一台设备只能绑定在一个风险点上，不能同时绑定多个风险点。
4. 风险点绑定后，该设备不能再次出现在“未绑定设备”下拉列表中。
5. 没有绑定风险点的历史设备，不应被伪造机构归属，也不应出现在风险点绑定清单中作为“空机构设备”。
6. 设备与风险点归属机构必须一致；机构不一致时，绑定请求直接拒绝。

## 5. 方案对比

### 5.1 方案 A：直接在 `iot_device` 落 `org_id / org_name`

- 设备主档直接保存归属机构。
- 写入侧统一从当前登录人的主机构自动回填。
- 查询侧直接用 `tenant + org_id + dataScopeType` 收口。
- 风险点绑定唯一性和设备候选过滤同步在服务层执行。

优点：

- 设备中心、风险对象中心和权限模型共享同一份机构事实来源。
- 查询、导出、详情、诊断链路都能直接复用设备主档归属，不需要每次回查风险点。
- 与当前“主机构 + 数据范围”身份链路天然对齐。

缺点：

- 需要改表结构、服务逻辑、前端展示和历史数据回填。

### 5.2 方案 B：新增设备-机构映射表

- 不改 `iot_device` 主表，单独维护设备和机构的映射关系。
- 查询时再通过映射表拼机构范围。

优点：

- 看起来更“可扩展”，以后支持多机构设备更方便。

缺点：

- 当前用户已经明确不做多机构。
- 会把本轮最小可用闭环扩大成关系建模工程，复杂度过高。

### 5.3 方案 C：只在风险点绑定层限制设备唯一性，不改设备表

- 设备归属仍不落库。
- 仅要求一台设备只能被一个风险点绑定。

优点：

- 改动面最小。

缺点：

- 设备资产中心依然无法进入机构范围治理，治理链路仍然不闭环。

## 6. 选型

本轮采用方案 A。

原因：

- 用户当前目标是“先把系统治理做成可用闭环，再把用户-机构-权限模型落到治理链路里”。
- 设备归属如果不落在 `iot_device`，设备资产中心始终无法成为机构范围治理的一部分。
- 当前项目仍坚持“单主机构、单风险点绑定”的最小模型，没有必要提前引入更重的关系建模。

## 7. 核心业务规则

### 7.1 设备归属规则

- `iot_device.org_id / org_name` 作为设备正式归属机构字段。
- 前端不提供设备归属机构编辑控件。
- 新增、批量新增、编辑、设备更换生成新设备时，后端统一取当前登录人的主机构回填。
- 当前登录人若没有主机构，则拒绝设备写入。

### 7.2 历史设备回填规则

- 仅处理 `iot_device.org_id IS NULL` 的历史设备。
- 若设备存在有效风险点绑定，则使用该风险点的 `org_id / org_name` 回填。
- 若设备没有风险点绑定，则不伪造机构，不自动归到默认机构。
- 历史无归属设备只有在重新进入新增/编辑/替换等写入链路后，才会按当前登录人的主机构补齐归属。

### 7.3 风险点绑定规则

- 一台设备同一时刻只能绑定一个风险点。
- 绑定请求必须校验：
  - 设备是否已被其他风险点绑定
  - 设备机构是否与风险点机构一致
- 已绑定设备不得再出现在“可绑定设备”候选下拉中。
- 编辑风险点时，当前风险点已绑定的设备允许回显，但不得切换到其他风险点已占用设备。

### 7.4 设备域数据范围规则

- 设备域从“仅租户侧治理”升级为“租户 + 主机构/机构子树治理”。
- 建议规则：
  - `ALL / TENANT`：当前租户全部设备
  - `ORG_AND_CHILDREN`：当前主机构及子树设备
  - `ORG`：当前主机构设备
  - `SELF`：设备域当前等价收敛为 `ORG`

原因：

- 设备当前没有稳定的“责任人用户”字段，不应伪造用户维度过滤。

## 8. 数据模型设计

### 8.1 `iot_device`

新增字段：

- `org_id BIGINT DEFAULT NULL COMMENT '所属机构ID'`
- `org_name VARCHAR(128) DEFAULT NULL COMMENT '所属机构名称'`

新增索引建议：

- 至少补一条以 `tenant_id + org_id + deleted` 为前缀的索引，供设备分页、设备选项和详情过滤复用。

### 8.2 `risk_point_device`

本轮不重做表结构语义，但需要把“一设备一风险点”提升为正式约束。

约束方式：

- 服务层显式校验同一设备是否已有有效绑定。
- 如果当前表结构允许表示“有效/失效”绑定，则唯一性应只作用于有效绑定记录。
- 若现有表结构没有状态字段，则至少保证“未解绑状态下 device_id 唯一”。

### 8.3 文档口径

- 设备归属机构的正式事实来源改为 `iot_device.org_id / org_name`。
- 风险点绑定继续是设备与风险点的业务关系，不再承担设备机构事实来源。

## 9. 后端设计

### 9.1 设备写入链路

涉及范围：

- `DeviceAddDTO`
- `DeviceServiceImpl`
- 设备批量导入
- 设备更换
- 设备编辑

处理规则：

1. 从 `PermissionService#getDataPermissionContext(currentUserId)` 读取当前登录人上下文。
2. 校验当前登录人存在 `orgId`。
3. 回查 `sys_organization` 获取组织名称。
4. 将 `org_id / org_name` 回填到 `iot_device`。
5. 设备编辑时同样重新按当前登录人主机构覆盖归属，保持“设备归属由维护人主机构决定”的单一规则。

### 9.2 设备查询链路

需要升级到机构范围治理的设备接口包括但不限于：

- `GET /api/device/{id}`
- `GET /api/device/code/{deviceCode}`
- `GET /api/device/page`
- `GET /api/device/list`
- `GET /api/device/{deviceId}/metrics`
- `GET /api/device/{deviceCode}/properties`
- `GET /api/device/{deviceCode}/message-logs`
- `GET /api/device/message-trace/page`
- `GET /api/device/message-trace/stats`
- `GET /api/device/access-error/page`
- `GET /api/device/access-error/stats`
- `GET /api/device/access-error/{id}`

过滤表达：

- 先按当前登录人的 `tenantId` 收口。
- 若非超管，再按 `org_id` 与 `dataScopeType` 继续收口。
- 设备相关诊断链路如通过 `device_id / device_code` 反查设备主档，也必须沿用设备机构范围判断，不允许只停留在消息日志表的租户过滤。

### 9.3 风险点绑定链路

涉及范围：

- 风险点新增绑定
- 风险点编辑绑定
- 风险点设备候选查询

新增校验：

1. 设备已被其他风险点绑定时，拒绝绑定。
2. 设备 `org_id` 为空时，拒绝绑定。
3. 设备 `org_id` 与风险点 `org_id` 不一致时，拒绝绑定。

错误提示应明确，不使用静默失败或“系统繁忙”兜底。

### 9.4 历史数据回填

本轮需要补一条真实环境可执行的数据回填路径：

1. 先检查是否存在“一设备多风险点有效绑定”的脏数据。
2. 如存在，立即中断迁移并输出冲突设备清单。
3. 如不存在，则按 `risk_point_device -> risk_point.org_id / org_name` 反写 `iot_device.org_id / org_name`。
4. 仅处理 `iot_device.org_id IS NULL` 记录。

## 10. 前端设计

### 10.1 设备资产中心

维持现有页面风格和共享工作台语法，不引入新视觉方案。

调整内容：

- 新增/编辑/更换设备抽屉不新增“所属机构”可编辑字段。
- 在设备详情抽屉中补充“所属机构”只读展示。
- 设备列表建议新增“所属机构”列，直观体现归属是否落表成功。
- 页面不新增“所属机构”手工筛选控件，继续由后端按数据范围自动收口。

### 10.2 风险对象中心

调整内容：

- 绑定设备下拉只返回“当前可见范围内且未被其他风险点绑定”的设备。
- 编辑风险点时，当前已绑定设备允许回显。
- 若设备被其他风险点占用或机构不一致，前端直接展示明确业务错误，不做静默兜底。

## 11. 错误处理

### 11.1 设备写入

以下场景直接返回 `BizException`：

- 当前账号未绑定主机构
- 当前账号主机构不存在或已不可用
- 设备写入时无法解析当前登录人的机构上下文

建议提示：

- `当前账号未绑定主机构，禁止维护设备归属`

### 11.2 风险点绑定

以下场景直接拒绝：

- 设备已绑定其他风险点
- 设备归属机构为空
- 设备归属机构与风险点归属机构不一致
- 同一设备存在多条有效绑定脏数据

### 11.3 历史迁移

- 若发现一设备多风险点有效绑定，则迁移中止，不自动选第一条。
- 若风险点机构为空，则该设备不参与回填。

## 12. 测试策略

### 12.1 后端单测

至少覆盖：

- 设备新增时自动继承当前登录人主机构
- 设备编辑时自动补齐或覆盖 `org_id / org_name`
- 设备分页/详情/设备选项按 `ORG / ORG_AND_CHILDREN / ALL` 收口
- 风险点绑定时同一设备重复绑定被拒绝
- 风险点绑定时机构不一致被拒绝
- 风险点设备候选查询不会返回已绑定设备
- 历史数据回填前的冲突检查能正确拦截“一设备多风险点绑定”

### 12.2 前端测试

至少覆盖：

- 设备抽屉只读展示所属机构，不出现机构编辑控件
- 设备详情能看到 `orgName`
- 风险点绑定设备选择器不会出现已绑定设备

### 12.3 聚合验证

- `spring-boot-iot-device` 定向测试
- `spring-boot-iot-ui` 相关视图测试
- `mvn -s .mvn/settings.xml -pl spring-boot-iot-admin -am clean package -DskipTests`

### 12.4 真实环境验收

继续基于 `spring-boot-iot-admin/src/main/resources/application-dev.yml`。

至少验证：

1. 新增设备后自动落库当前登录人主机构
2. 风险点绑定后，该设备无法被其他风险点再次选择
3. 不同机构账号查看 `/devices` 时，数据已经按机构范围收口

## 13. 实施顺序

1. 为 `iot_device` 增加 `org_id / org_name` 与索引，并补真实环境 schema 对齐脚本。
2. 补设备实体、VO、前端类型和设备写入链路的机构回填逻辑。
3. 把设备分页、详情、选项、属性、消息日志、链路追踪、失败归档升级到机构范围过滤。
4. 收口风险点绑定唯一性和设备候选下拉。
5. 补前端展示、测试、文档和真实环境验收。

## 14. 范围控制

本轮只交付以下内容：

- 设备归属机构落表
- 风险点唯一绑定
- 设备资产中心进入机构范围治理闭环

本轮明确不做：

- 设备多机构关系
- 设备域用户级 `SELF` 权限语义
- 告警、事件、报表的同批扩面治理
