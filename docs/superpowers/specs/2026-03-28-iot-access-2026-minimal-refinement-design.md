# IoT Access 2026 Minimal Refinement Design

**Date:** 2026-03-28
**Status:** Draft for review
**Scope:** `接入智维` 6 个核心页面在既有两层导航结构上的第二轮视觉精修，方向为“静稳控制台基线 + 适度吸收 2026 现代 SaaS 留白”

## 1. 背景

`接入智维` 在 2026-03-28 已完成第一轮结构收口，当前页面骨架已经稳定为：

1. 单主列表页：`面包屑 + 主标题 + 筛选区 + 工具条 + 单主列表`
2. 真页签诊断页：`面包屑 + 真页签 + 业务标题 + 查询区 + 单主内容区`

这一轮结构调整已经解决了以下问题：

1. 三层重复导航被收口为两层结构。
2. `产品定义中心`、`设备资产中心`、`异常观测台` 等单主列表页不再保留中间伪导航层。
3. `链路追踪台`、`数据校验台` 等诊断页已去掉右上角跨页功能菜单。
4. 页面重新回到“筛选 -> 操作 -> 查看结果”的主线。

但用户继续明确提出第二轮诉求：

1. **继续精修，但不改当前系统配色。**
2. **参考 2026 年互联网产品设计风格，整体要更简约大方。**
3. **布局不变。**
4. **全站同类页面统一，不做单页私有花样。**

结论：

- 第一轮的核心任务是“结构收口”。
- 第二轮的核心任务是“气质升级”。
- 本轮要解决的不是页面层级再重做，而是让既有结构看起来更稳、更轻、更现代。

## 2. 目标

本轮精修目标如下：

1. 在不改变当前系统主配色的前提下，提升 `接入智维` 页面的一致性和现代感。
2. 让 `产品定义中心`、`设备资产中心`、`链路验证中心`、`异常观测台`、`链路追踪台`、`数据校验台` 六个页面形成统一的视觉层级。
3. 用更克制的方式提升 2026 年互联网 B 端产品常见的“轻、稳、简洁”气质。
4. 强化标题、筛选区、查询按钮、工具条、表格表头、操作列之间的主次关系。
5. 继续优先复用现有共享组件和 design token，不为单页重新发明新主题或新布局。

## 3. 非目标

本轮明确不做：

1. 不改变既有两层导航结构，不回退到三层导航。
2. 不新增一级路由、分组页或伪页签。
3. 不改变当前品牌橙、蓝灰正文、浅灰控制台背景这套主配色。
4. 不引入新的营销式视觉、大面积高饱和渐变或脱离系统基线的主题色。
5. 不重做页面业务语义、接口行为或抽屉逻辑。
6. 不为追求“互联网感”而牺牲现有台账页的可读性和治理效率。

## 4. 设计结论

### 4.1 主方向

本轮采用用户确认的推荐方案：

- **主方向：静稳控制台**
- **补充约束：适度吸收现代 SaaS 的留白和呼吸感**

一句话结论：

`接入智维` 页面继续保持当前控制台主配色与两层结构，但在标题层级、筛选密度、工具条轻重、表头亮度、操作区节奏上向 2026 年极简互联网控制台靠拢。

### 4.2 为什么不是纯“现代 SaaS”方案

本轮不直接采用更轻、更飘的纯 SaaS 方向，原因如下：

1. 当前系统是设备接入与诊断工作台，信息密度高，列表和排障仍然是主战区。
2. 纯 SaaS 化会让页面更“好看”，但可能削弱工具页和台账页的扫描效率。
3. 用户明确要求保持当前系统配色与布局不变，因此更适合做“现代化的控制台”，而不是“重主题化的互联网后台”。

### 4.3 最终视觉原则

本轮最终采用以下视觉原则：

1. **颜色不变，层级重排。**
2. **边框更轻，留白更稳。**
3. **标题更聚焦，说明更短。**
4. **统计更轻，主操作更明确。**
5. **表格更安静，操作更统一。**

## 5. 视觉语言规则

### 5.1 配色规则

继续严格使用现有 token 体系：

1. 主品牌色仍为 `brand / primary` 橙色体系。
2. 正文与表格仍使用蓝灰文字体系。
3. 页面背景、卡片背景、表头背景继续使用浅灰和白色控制台体系。
4. 不新增暖灰、米色、紫色或其他私有页面主题。

允许的变化只有：

1. 调整同色系透明度与明度。
2. 调整边框强弱。
3. 调整阴影强弱。
4. 调整表头、工具条和轻提示的背景亮度。

### 5.2 字体层级规则

统一建立以下层级：

1. **面包屑**：最轻，仅做路径定位。
2. **页面主标题**：最强，承担页面身份。
3. **筛选标签 / 工具条统计**：次级信息，不抢主操作。
4. **表头**：低对比、稳定、可扫描。
5. **操作链接**：保持品牌橙，作为局部行为锚点。

特别要求：

1. `设备台账` 与 `产品定义中心` 必须保持同一标题级别和字重。
2. 诊断页标题与列表页标题属于同一等级，不再因为页面类型不同出现明显尺寸漂移。

### 5.3 空间与密度规则

本轮密度策略为“紧凑但不压迫”：

1. 查询区输入框与按钮保持统一高度，轻微抬高到更现代的控制台密度。
2. 标题区与筛选区之间保留稳定留白，不再堆额外说明文案。
3. 工具条与表格之间维持更清晰的节奏断点。
4. 表格行高、表头留白和操作列间距统一复用共享组件语法。

### 5.4 组件气质规则

共享组件的气质收口为：

1. **更薄的边框**
2. **更软的卡片阴影**
3. **更淡的表头底色**
4. **更明确的主按钮**
5. **更轻的统计信息**

不允许：

1. 页面私自新增显眼色块
2. 页面私自新增大标题说明区
3. 页面私自新增第二套圆角、阴影或按钮语法

## 6. 共享组件设计

本轮优先通过共享组件收口视觉，而不是先在页面内散改。

### 6.1 `StandardWorkbenchPanel`

职责：

1. 统一标题与头部留白
2. 统一筛选、工具条、结果区的节奏
3. 统一卡片边框和阴影的轻量化表达

设计调整：

1. 主标题更稳，使用更清晰的字体层级和 line-height。
2. 头部、筛选区、工具条、分页区的垂直节奏进一步统一。
3. 面板边框和卡片阴影更克制，减少“厚重卡片感”。

### 6.2 `StandardListFilterHeader`

职责：

1. 统一筛选输入区的网格、按钮和展开逻辑
2. 统一“查询 / 重置 / 主操作”在视觉上的主次关系

设计调整：

1. 输入区高度、内边距与圆角统一到更现代的控制台密度。
2. 查询按钮作为视觉锚点继续保留，但不通过额外颜色扩张存在感。
3. `展开更多筛选` 与辅助提示继续保留，但权重进一步降低。

### 6.3 `StandardTableToolbar`

职责：

1. 承接统计信息和轻操作
2. 作为主列表上方唯一的摘要区

设计调整：

1. 左侧统计信息从“次主角”降级为“辅助信息”。
2. 右侧导出、批量、刷新等动作保持清晰，但避免挤压主列表阅读。
3. 整体背景更轻、边界更清楚。

### 6.4 `IotAccessTabWorkspace`

职责：

1. 承接真业务页签
2. 维持诊断页在同一路由下的真实业务切换

设计调整：

1. 页签继续是线性业务切换，而不是 pill。
2. 激活态保留品牌橙，但下划线、更细的边界和更轻的 idle 文字让页签更现代。
3. 页签与下面业务标题之间保留更自然的呼吸感。

## 7. 页面逐页设计

### 7.1 产品定义中心

保持：

1. 面包屑
2. `产品定义中心` 标题
3. 筛选区
4. 工具条
5. 单主列表区

精修重点：

1. 标题更稳，副说明继续保持最小化。
2. 查询区输入、按钮、快速搜索标签做同一组节奏收口。
3. 工具条统计信息弱化，让新增产品、批量操作和导出动作更清晰。
4. 表格表头更轻，列表主体更安静。

### 7.2 设备资产中心

保持：

1. 面包屑
2. `设备台账` 标题
3. 筛选区
4. 工具条
5. 单主列表区

精修重点：

1. `设备台账` 标题视觉等级与 `产品定义中心` 对齐。
2. 由于筛选项较多，查询区要优先通过网格和按钮秩序体现“简约”，而不是通过减少信息量实现简约。
3. 统计项较多时，工具条要弱化统计、强化高频动作。
4. 注册状态、在线状态、设备状态等标签不额外放大，避免列表显得杂。

### 7.3 异常观测台

保持：

1. 面包屑
2. `异常台账` 或系统当前对应主标题
3. 筛选区
4. 工具条
5. 单主列表区

精修重点：

1. 维持日志/异常台账的冷静感，不放大“异常”语义颜色。
2. 统计与摘要信息继续放在工具条，不回到顶部状态区。
3. 列表操作和 TraceId / 设备定位信息保持统一节奏。

### 7.4 链路验证中心

保持：

1. 面包屑
2. 真页签
3. 业务标题
4. 配置 / 查询区
5. 主工作区

精修重点：

1. 左右工作面板在不改布局的前提下，视觉上更像一套完整工作区。
2. `设备查询`、`模拟配置`、`Payload 编辑` 等 section 之间的间距更现代、更利落。
3. 诊断侧的信息密度不变，但说明文案更短，卡片关系更清楚。

### 7.5 链路追踪台

保持：

1. 面包屑
2. `链路追踪 / 失败归档` 真页签
3. 业务标题
4. 筛选区
5. 列表结果区

精修重点：

1. 页签更轻，列表标题更稳。
2. 工具条统计轻量化，避免追踪量和失败摘要抢走主线。
3. `详情 / 观测 / 回跳治理` 一类动作继续统一品牌橙语义。
4. 右上角跨页功能菜单保持删除状态，不回流。

### 7.6 数据校验台

保持：

1. 面包屑
2. 真页签
3. `设备校验` 业务标题
4. 设备编码查询区
5. 校验结果主内容区

精修重点：

1. 查询区更轻、更整洁。
2. 双列结果区在卡片边框、标题和正文层次上更统一。
3. 原始响应与校验结果虽然属于不同页签，但整体语法继续和链路追踪台保持一致。
4. `链路追踪台` 右上角按钮继续移除，不回流到本页。

## 8. 实现范围与边界

### 8.1 本轮实现范围

本轮实现范围限定为：

1. 共享组件视觉层级精修
2. 六个目标页面的样式对齐和少量结构微调
3. 必要的测试更新
4. 必要的文档同步

### 8.2 本轮不应触碰的边界

实现时不得：

1. 改动页面业务流程
2. 改动诊断联动 query 逻辑
3. 新增页面私有主题变量
4. 引入页面私有大块说明卡
5. 恢复三层导航或右上角功能菜单

## 9. 错误处理与回退策略

本轮主要是视觉与结构微调，不涉及新后端依赖。

错误处理原则：

1. 若某个共享组件收口后导致个别页面密度异常，应优先通过共享 token 或受控变体修正，不回退到页面私有样式分叉。
2. 若诊断页页签、工具条或列表在小屏下出现换行挤压，应继续通过共享组件响应式布局修正，而不是在页面内加临时补丁。
3. 若某个精修点会破坏现有列表 guard 或样式 guard，优先调共享实现，确保门禁可以稳定表达新基线。

## 10. 验证策略

本轮验证分为四层：

1. **组件测试**
   - 更新共享组件相关测试，确保头部、工具条、真页签结构不回退。

2. **页面回归测试**
   - `ProductWorkbenchView`
   - `DeviceWorkbenchView`
   - `AuditLogView`
   - `ReportWorkbenchView`
   - `MessageTraceView`
   - `FilePayloadDebugView`

3. **静态门禁**
   - `npm run component:guard`
   - `npm run list:guard`
   - `npm run style:guard`

4. **构建验证**
   - `npm run build`

验证结论必须回答两个问题：

1. 六个页面是否仍保持既定两层结构与真实页签语义。
2. 新的视觉精修是否没有引入新的页面私有样式分叉。

若本轮实现落地，需要同步检查并按实际改动更新：

1. `README.md`
2. `docs/02-业务功能与流程说明.md`
3. `docs/06-前端开发与CSS规范.md`

3. `AGENTS.md` 只在协作规则或全局基线变化时更新；若无需修改，要在交付说明中明确“已检查，无需修改”。
