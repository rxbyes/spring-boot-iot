# 产品物模型规范预设适用性分流设计

**Date:** 2026-04-04  
**Status:** Approved in-session for spec drafting  
**Audience:** 设备中心 / 产品治理 / 前后端实现 / 文档维护  
**Scope:** 修正 `/products -> 物模型双证据治理` 中“规范证据优先”预设对所有产品一刀切展示的问题，使 `zhd-warning-sound-light-alarm-v1` 等预警型产品不再误展示“倾角 / 加速度 / 裂缝一体机”规范字段，并在无适用预设时平滑退化到通用双证据治理。

## 1. 背景

当前 `/products -> 查看 -> 工作台 -> 物模型治理 -> 双证据治理` 已经完成“规范证据 + 报文证据 + 正式基线”的第一版升级，并默认把规范治理入口展示为首批内置预设 `landslide-integrated-tilt-accel-crack-v1`。

现状问题在于：

1. 前端规范预设目录当前只有一套固定预设：`倾角 / 加速度 / 裂缝一体机`。
2. 进入双证据治理抽屉时，当前实现不会根据产品类型、产品族或 `productKey` 判断该预设是否适用。
3. 因此，`zhd-warning-sound-light-alarm-v1` 这类“中海达 预警型 声光报警器”产品，也会看到监测型一体机的倾角 / 加速度 / 裂缝字段卡片。
4. 这会误导治理人员，以为该产品应套用监测型规范字段，违背现有业务事实和上层技术规范。

本轮目标不是立即给“预警型声光报警器”补一套新规范预设，而是先解决“错误预设被展示”的问题，并保证用户还能继续治理该产品。

## 2. 目标

本设计服务以下目标：

1. 让规范预设展示从“全产品固定一套”升级为“按产品判定是否适用”。
2. 让 `zhd-warning-sound-light-alarm-v1` 这类预警型产品不再展示监测型一体机规范字段。
3. 在当前产品没有适用规范预设时，仍保留“规范证据优先”入口，但明确提示“暂无适用规范预设”。
4. 用户在没有适用预设的情况下，仍能顺畅进入通用双证据治理，不被卡死在规范模式空态中。
5. 为后续新增 `预警型声光报警器` 等专属规范预设保留稳定扩展位，而不重做整个治理链路。

## 3. 非目标

本轮明确不做以下事情：

1. 不立即补一套“预警型声光报警器”正式规范预设。
2. 不引入预设管理后台、数据库表或新的模板持久化体系。
3. 不修改当前 compare / apply 核心语义。
4. 不改变“正式模型唯一落点仍是 `iot_product_model`”这一基线。
5. 不把产品分类体系扩展成新的主数据治理项目。

## 4. 方案对比与选型

### 4.1 方案 A：无适用预设时直接隐藏规范治理入口

表现：

1. 对没有适用预设的产品，直接不展示 `规范证据优先`。
2. 默认只保留 `通用双证据`。

优点：

1. 表现最直接。
2. 用户不会误点到错误预设。

缺点：

1. 会让不同产品的治理入口结构不一致。
2. 用户难以理解“为什么有的产品有规范治理，有的产品没有”。
3. 不利于后续平滑扩展新预设。

### 4.2 方案 B：保留规范治理入口，但在无适用预设时显示空态并提供退化动作

表现：

1. 模式切换仍保留 `规范证据优先 / 通用双证据`。
2. 当前产品没有适用预设时，规范模式显示“暂无适用规范预设”空态卡。
3. 用户可一键切到 `通用双证据`，或继续点击 compare 后自动退化。

优点：

1. 产品治理入口结构保持一致。
2. 明确表达“规范治理能力存在，但当前产品暂无适用模板”。
3. 最利于未来补新预设。

缺点：

1. 需要前后端各补一层适用性判断。

### 4.3 方案 C：立刻为预警型产品补专属规范预设

表现：

1. `zhd-warning-sound-light-alarm-v1` 直接切到一套新的“预警型声光报警器”规范字段。

优点：

1. 一步到位。

缺点：

1. 当前任务缺少完整、稳定的预警型规范字段清单。
2. 容易为了修复误展示而临时发明一套不稳定预设。

### 4.4 选型

本轮采用 `方案 B：保留规范治理入口，但在无适用预设时显示空态并提供退化动作`。

理由：

1. 最符合当前用户诉求：“这类设备不应该出现倾角 / 加速度 / 裂缝一体机的信息”。
2. 不会把用户赶回旧入口，也不会破坏现有治理流程结构。
3. 可以把“当前产品有没有适用预设”与“系统是否支持规范治理”这两个概念清晰分开。

## 5. 已确认决策

本次会话已明确确认以下设计决策：

1. `zhd-warning-sound-light-alarm-v1` 不应展示 `倾角 / 加速度 / 裂缝一体机` 规范字段。
2. 对这类产品，不采用“直接隐藏规范治理入口”的方式。
3. 当前产品没有适用预设时，应保留规范治理入口，但显示“暂无适用规范预设”。
4. 用户应可从该空态平滑切到 `通用双证据`。
5. 当前轮次不补“预警型声光报警器”专属规范预设，只补预设适用性分流。

## 6. 设计概览

本轮把规范治理链路拆成两层判断：

1. **治理模式层**
   - 仍保留 `normative` 与 `generic` 两种模式。
2. **预设适用性层**
   - 仅在 `normative` 模式下生效。
   - 判断当前产品是否命中任何规范预设。

当产品命中预设时：

1. 正常展示预设标题、说明和字段勾选卡片。
2. compare 请求按规范模式发送。

当产品未命中预设时：

1. 规范模式下只显示空态卡，不显示错误字段卡片。
2. compare 行为自动退化为 `generic`。
3. 用户仍可主动切换到 `通用双证据`。

## 7. 预设匹配规则

### 7.1 基本原则

预设适用性不靠用户手工指定，也不靠数据库配置；先使用前后端同源的静态匹配规则。

每个规范预设除现有元数据外，还应额外表达“适用产品规则”，至少支持：

1. `productKey` 明确白名单
2. `productKey` 前缀匹配
3. 产品名关键字匹配

### 7.2 首批规则

首批仅固化以下结论：

1. `landslide-integrated-tilt-accel-crack-v1`
   - 只适用于监测型一体机产品族
   - 不适用于 `zhd-warning-sound-light-alarm-v1`
2. `zhd-warning-sound-light-alarm-v1`
   - 当前不命中任何规范预设

### 7.3 解析结果模型

前后端都需要具备统一语义的解析结果：

1. 当前产品命中的预设编码
2. 若未命中，返回“无适用预设”状态
3. 不允许把“不适用”默认为“使用首个预设”

## 8. 前端设计

### 8.1 抽屉入口表现

进入双证据治理抽屉时：

1. 模式切换仍保留：
   - `规范证据优先`
   - `通用双证据`
2. 默认仍先停在 `规范证据优先`。

### 8.2 命中预设时

若当前产品命中规范预设：

1. 展示预设标题、副说明、辅助说明。
2. 展示字段勾选卡片。
3. 默认勾选由该预设自身定义。

### 8.3 无适用预设时

若当前产品未命中任何规范预设：

1. 不显示任何监测型字段卡片。
2. 改为显示一张空态卡，文案固定表达：
   - 标题：`当前产品暂无适用规范预设`
   - 说明：`该产品未命中监测型规范字段模板，请改用通用双证据治理，或待专属规范预设补齐后再使用规范治理。`
3. 空态卡提供按钮：
   - `切换到通用双证据`

### 8.4 compare 按钮行为

若当前产品未命中预设，用户仍点击“生成对比结果”：

1. 前端给出轻提示，说明当前产品暂无适用规范预设，系统将改按通用双证据执行 compare。
2. compare 请求实际按 `generic` 发送。

该行为优先于“阻止操作”，目的是不让治理流程被空态卡住。

## 9. 后端设计

### 9.1 规范预设注册表增强

当前 `ProductModelNormativePresetRegistry` 除字段定义与别名映射外，还要支持：

1. 根据产品信息解析适用预设
2. 验证“某预设是否适用于当前产品”

输入至少包含：

1. `productKey`
2. `productName`

输出语义至少包含：

1. 命中的 `presetCode`
2. 未命中时的空结果

### 9.2 compare 请求保护

后端需要补一层防御，防止前端误传：

1. 当 `governanceMode=normative` 且 `normativePresetCode` 不适用于当前产品时：
   - 不允许继续按该预设产出 compare 结果
2. 处理策略采用“安全退化”：
   - 返回明确业务提示，或
   - 按 `generic` 模式执行 compare

本轮推荐后端采用“显式拒绝 + 前端退化”的职责分工：

1. 前端优先自行退化，减少误请求。
2. 后端保留拒绝保护，避免错误预设继续产出错误字段。

## 10. 数据与接口语义

本轮不新增数据库表。

接口层的关键变化是“什么时候允许前端发送规范模式”：

1. 只有当前产品命中有效预设时，前端才发送：
   - `governanceMode: normative`
   - `normativePresetCode`
   - `selectedNormativeIdentifiers`
2. 当前产品没有适用预设时，前端直接发送：
   - `governanceMode: generic`
3. 后端不得把“缺省预设”当成所有产品的兜底预设。

## 11. 错误处理

### 11.1 前端错误处理

1. 预设解析失败或当前产品无适用预设时，不展示错误字段卡片。
2. compare 自动退化时应给轻提示，不弹阻断式错误。

### 11.2 后端错误处理

1. 若前端误传不适用预设，后端应返回明确业务错误，而不是继续产出监测型字段。
2. 错误信息应表达“当前产品不适用该规范预设”，而不是泛化成系统异常。

## 12. 测试与验收

### 12.1 前端测试

至少补以下测试：

1. 监测型产品进入规范治理时，仍正常展示 `倾角 / 加速度 / 裂缝一体机` 预设。
2. `zhd-warning-sound-light-alarm-v1` 进入规范治理时，显示“暂无适用规范预设”空态，不显示监测型字段卡片。
3. 无适用预设时点击“切换到通用双证据”，模式正确切换。
4. 无适用预设时直接点击 compare，实际发送 `generic` 请求。

### 12.2 后端测试

至少补以下测试：

1. 监测型产品能正确解析出 `landslide-integrated-tilt-accel-crack-v1`。
2. `zhd-warning-sound-light-alarm-v1` 解析结果为空。
3. 不适用产品误传监测型预设时，后端拒绝继续按该预设生成 compare。

### 12.3 人工验收

在 `/products` 中验收以下路径：

1. 进入监测型产品的双证据治理，确认规范字段仍正常出现。
2. 进入 `zhd-warning-sound-light-alarm-v1`，确认不再出现倾角 / 加速度 / 裂缝字段。
3. 在该产品上点击“切换到通用双证据”或直接 compare，确认治理流程不中断。

## 14. 实施边界

本设计只解决“规范预设适用性分流”和“无适用预设时的降级表现”。

后续如果要继续推进：

1. 为 `预警型声光报警器` 定义专属规范预设
2. 引入更多产品族预设
3. 将预设匹配规则从静态代码迁移到更可配置的治理资产

这些都应作为下一轮独立 spec，不并入本轮。
