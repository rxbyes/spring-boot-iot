# 风险对象中心正式绑定可视化与维护设计

> 日期：2026-04-04
> 范围：`spring-boot-iot-alarm`、`spring-boot-iot-ui`、`docs/02-业务功能与流程说明.md`、`docs/03-接口规范与接口清单.md`、`docs/15-前端优化与治理计划.md`、`docs/21-业务功能清单与验收标准.md`
> 主题：把风险对象中心从“能绑定、能待治理转正”升级为“能看清正式绑定、能按设备和测点维护正式关系”的最小可用闭环
> 状态：设计已确认，待书面 spec 审阅

## 1. 背景

当前风险对象中心已经具备三条基础能力：

- 风险点主档增删改查与列表分页。
- 风险点与设备测点的正式绑定写入。
- 风险点待治理记录分页、候选推荐、人工转正与忽略。

但结合当前页面、接口和表结构，可以确认仍有一个明显缺口：

- 风险点列表页当前只展示风险点主档，不展示“已绑定多少台设备、多少个正式测点、还有多少待治理记录”。
- 页面当前有“绑定设备”抽屉和“待治理转正”抽屉，但没有“正式绑定明细”的统一查看与维护入口。
- 现有解绑接口是按 `riskPointId + deviceId` 删除绑定，会把该设备在该风险点下的所有正式测点一起删掉。
- `risk_point_device` 的真实粒度不是“风险点-设备”，而是“风险点-设备-测点”；同一设备在同一风险点下可以绑定多个正式测点。

因此，如果继续把需求写成“风险点能修改设备、设备测点能删除修改”，就会把“设备主档维护”和“绑定关系维护”混为一谈，并且容易把“整机解绑”和“单测点解绑”做成同一个动作，造成错误的数据语义。

## 2. 目标

- 在风险对象中心列表中，清晰展示每个风险点当前的正式绑定概况。
- 在风险点维度，按设备分组查看当前已生效的正式绑定测点，而不是只看待治理记录。
- 明确拆分“设备级关系动作”和“测点级绑定动作”，避免操作语义含混。
- 保持待治理转正为独立治理入口，不与正式绑定维护混做一个抽屉。
- 延续现有共享工作台、抽屉和列表模式，不新增第二套路由或私有布局体系。

## 3. 非目标

- 本轮不在风险对象中心直接编辑设备主档信息；设备名称、产品、组织等主数据仍归设备资产中心维护。
- 本轮不在风险对象中心直接编辑产品物模型或协议测点定义。
- 本轮不重做 `risk_point_device` 表结构，也不引入新的绑定关系表。
- 本轮不把待治理推荐逻辑和正式绑定维护逻辑合并为同一个流程。
- 本轮不放开“一设备一风险点”的既有约束。

## 4. 原始诉求的合理化重写

用户原始诉求：

1. 能看到绑定关联的设备。
2. 能看到每个设备转正的测点。
3. 风险点支持解绑和修改设备。
4. 设备的测点支持修改删除。

重写后的正式需求口径：

1. 风险对象中心列表应显示绑定概览，包括正式绑定设备数、正式绑定测点数、待治理记录数。
2. 风险点应提供“正式绑定明细”视图，按设备分组展示当前已生效的正式绑定测点，并可区分是否来自待治理转正留痕。
3. 风险点页面应支持维护绑定关系，而不是修改设备主档；设备级动作收口为“新增设备绑定关系”和“整机解绑当前设备”。
4. 测点级动作应收口为“新增正式测点绑定”“删除单个正式测点绑定”“替换当前正式测点绑定”，而不是直接修改设备物模型。

额外说明：

- “修改设备”在本轮不定义为直接改设备档案，而是通过“解绑错误设备 + 新增正确设备绑定”完成。
- “修改测点”在本轮不定义为随意改历史绑定行内容，而是通过“替换绑定测点”完成，语义上仍属于绑定关系调整。

## 5. 方案对比

### 5.1 方案 A：在风险点列表直接展开所有设备和测点

- 列表行内直接展开全部正式绑定明细。
- 用户无需再进入抽屉即可看到所有设备和测点。

优点：

- 信息最直接。

缺点：

- 风险点一旦绑定设备和测点较多，列表会迅速变重。
- 与当前共享工作台的“列表承载摘要、抽屉承载详情”的模式不一致。
- 后续设备级和测点级动作会把主列表操作区撑得过于复杂。

### 5.2 方案 B：列表只展示绑定摘要，详情在“正式绑定维护抽屉”中完成

- 列表新增绑定概览列。
- 行级新增“正式绑定维护”入口。
- 在抽屉内按设备分组查看正式绑定，并执行设备级和测点级动作。
- 待治理转正仍保留独立抽屉。

优点：

- 与现有共享页面模式最一致。
- 主列表保持清爽，复杂动作下沉到抽屉。
- 能明确分离“正式绑定维护”和“待治理转正”。

缺点：

- 需要补一组更清晰的绑定详情和维护接口。

### 5.3 方案 C：新增单独“绑定维护”页面或页签

- 风险点列表只保留跳转入口。
- 绑定维护进入独立页面或同路由页签。

优点：

- 信息承载能力最强。

缺点：

- 超出当前页面结构最小改动范围。
- 与“接入智维单主列表页统一两层结构”的治理约束不一致。
- 当前问题规模不足以支撑新页面。

## 6. 选型

采用方案 B。

原因：

- 当前项目已经存在风险点列表、绑定抽屉、待治理转正抽屉，最稳妥的做法是在既有页面语义上补齐“正式绑定维护抽屉”。
- 用户核心诉求不是新增一个管理系统，而是把“看不清、改不准”的正式绑定关系做清楚。
- 方案 B 能同时满足“列表看摘要”和“详情做维护”，且对现有页面治理规则最友好。

## 7. 核心业务规则

### 7.1 正式绑定的真实粒度

- `risk_point_device` 的粒度定义为：`风险点 + 设备 + 测点`。
- 同一风险点下，同一设备可以存在多条正式绑定，每条绑定对应一个 `metricIdentifier`。
- 同一风险点下，同一设备、同一测点不可重复绑定。

### 7.2 设备级动作规则

- 设备级动作仅代表“维护风险点与设备的关系”，不代表“编辑设备主档”。
- 整机解绑的含义是：删除该风险点下该设备的全部正式测点绑定。
- 风险点页面不直接提供设备主档编辑入口。
- 如果发现绑定了错误设备，应通过“整机解绑错误设备 + 新增正确设备绑定”完成修正。

### 7.3 测点级动作规则

- 新增测点：在风险点与设备关系不变的前提下，为该设备新增一个正式测点绑定。
- 删除测点：只删除单条正式绑定，不影响同设备下其他正式测点。
- 替换测点：本质是“退役旧绑定 + 建立新绑定”的事务动作，不允许把历史绑定行当作任意可改的文本记录。

### 7.4 待治理规则

- 待治理转正仍然是正式绑定的重要来源，但不是正式绑定明细的唯一来源。
- 正式绑定明细页展示的是“当前已生效的正式绑定结果”，待治理抽屉展示的是“尚待人工治理的记录”。
- 若同一风险点同时存在正式绑定和待治理残留，应在正式绑定维护抽屉中提供提示，但不合并操作区。

## 8. 页面设计

### 8.1 列表层

风险对象中心列表新增一列“绑定概览”，建议展示：

- `X 台设备`
- `Y 个测点`
- `待治理 Z 条`

展示要求：

- 优先展示摘要，不在主列表直接展开所有设备与测点。
- 若无正式绑定，则显示“未绑定”。
- 若无正式绑定但存在待治理记录，应显示“待治理 Z 条”提示，提醒进入治理。

行级动作建议调整为：

- `编辑`
- `维护绑定`
- `待治理转正`
- `删除`

说明：

- 现有“绑定设备”动作升级为“维护绑定”，语义从单次新增扩成正式绑定关系维护入口。

### 8.2 正式绑定维护抽屉

抽屉目标：统一承载风险点当前正式绑定关系的查看与维护。

建议结构：

1. 顶部摘要区
   - 风险点名称、编号、所属组织。
   - 正式绑定设备数、正式绑定测点数、待治理记录数。

2. 设备分组区
   - 每个设备一张分组卡片。
   - 卡片头展示：设备编码、设备名称、当前正式测点数。
   - 设备级动作：`新增测点`、`整机解绑`。

3. 测点明细区
   - 每个设备卡片下展示测点列表。
   - 每条测点显示：测点名称、测点标识符、来源提示。
   - 测点级动作：`替换测点`、`删除测点`。

4. 治理提示区
   - 若当前风险点仍存在 `pending` 记录，则显示“仍有 N 条待治理记录，建议进入待治理转正抽屉继续处理”。

### 8.3 设备与测点动作的交互语义

#### 新增测点

- 从当前设备可用测点中选择一个尚未正式绑定的测点。
- 仅新增单条正式绑定。
- 复用现有“绑定风险点与设备测点”的后端写侧能力。

#### 整机解绑

- 明确提示“将删除该设备在当前风险点下的全部正式测点绑定”。
- 二次确认后执行。
- 不影响设备主档，也不影响其他风险点，因为系统已禁止一设备多风险点。

#### 删除测点

- 仅删除当前 `bindingId` 对应的一条正式绑定。
- 二次确认后执行。
- 删除后若该设备仍有其他正式测点，则设备卡片继续保留。
- 删除后若该设备已无任何正式测点，则设备卡片从正式绑定列表消失。

#### 替换测点

- 选择新的测点后，以事务方式完成“新增新测点绑定 + 删除旧测点绑定”。
- 旧绑定 ID 不复用。
- 若新测点已存在正式绑定，则直接拒绝，避免重复。

### 8.4 不在本轮风险点页实现的动作

- 不直接编辑设备名称、设备所属产品、设备组织等设备主数据。
- 不直接编辑产品物模型中的测点定义。
- 不提供“自动把设备 A 替换成设备 B 并自动迁移全部测点”的重型向导；这类需求存在测点不兼容风险，后续若业务确认需要，再单开设计。

## 9. 后端设计

### 9.1 保留并复用的现有接口

- `GET /api/risk-point/bound-devices/{riskPointId}`
- `POST /api/risk-point/bind-device`
- `POST /api/risk-point/unbind-device`
- `GET /api/risk-point/pending-bindings`
- `GET /api/risk-point/pending-bindings/{pendingId}/candidates`
- `POST /api/risk-point/pending-bindings/{pendingId}/promote`
- `POST /api/risk-point/pending-bindings/{pendingId}/ignore`

语义重定义：

- `unbind-device` 仅在前端作为“整机解绑”使用，不再误用于单测点删除。

### 9.2 新增接口建议

#### 绑定摘要批量查询

`GET /api/risk-point/binding-summaries?riskPointIds=1,2,3`

返回建议：

- `riskPointId`
- `boundDeviceCount`
- `boundMetricCount`
- `pendingBindingCount`

用途：

- 避免风险点分页列表逐行调用详情接口形成 N+1 请求。

#### 单测点删除

`POST /api/risk-point/bindings/{bindingId}/remove`

处理规则：

- 仅删除该 `bindingId` 对应的正式绑定。
- 必须校验该绑定所属风险点在当前登录人的数据权限范围内。

#### 单测点替换

`POST /api/risk-point/bindings/{bindingId}/replace`

请求体建议：

- `metricIdentifier`
- `metricName`

处理规则：

- 读取旧绑定，确认当前登录人可访问其所属风险点。
- 校验新测点在同 `riskPointId + deviceId` 下尚未绑定。
- 在一个事务内完成“新增新绑定、删除旧绑定”。

### 9.3 VO 建议

建议新增正式绑定维护视图对象，而不是直接把实体原样透给页面。

建议对象：

- `RiskPointBindingSummaryVO`
- `RiskPointBindingMetricVO`
- `RiskPointBindingDeviceGroupVO`

设备分组对象建议字段：

- `deviceId`
- `deviceCode`
- `deviceName`
- `metricCount`
- `metrics`

测点对象建议字段：

- `bindingId`
- `metricIdentifier`
- `metricName`
- `bindingSource`
- `createTime`

说明：

- `bindingSource` 可先提供最小口径，例如 `MANUAL / PENDING_PROMOTION / UNKNOWN`。
- 若当前无法稳定回填来源，也可先返回 `UNKNOWN`，但字段应预留。

## 10. 前端设计

### 10.1 页面组织

- 保持 `/risk-point` 单页主列表结构不变。
- 不新增页面私有 tab，不新增二级轻路由。
- 继续复用现有共享组件：`StandardWorkbenchPanel`、`StandardFormDrawer`、`StandardDrawerFooter`、`StandardTableTextColumn`、`confirmAction`。

### 10.2 页面状态

新增前端状态建议：

- 风险点分页列表的绑定摘要缓存。
- 正式绑定维护抽屉的设备分组列表。
- 当前选中设备的新增测点 / 替换测点临时表单。

### 10.3 交互优先级

优先级建议如下：

1. 列表摘要可见。
2. 正式绑定明细可见。
3. 整机解绑可用。
4. 单测点删除可用。
5. 新增测点可用。
6. 替换测点可用。

原因：

- 前两项解决“看不清”。
- 第三和第四项解决“删不准”。
- 第五和第六项再解决“改不准”。

## 11. 验收标准

### 11.1 功能验收

- 风险点分页列表可以看到每条风险点的绑定概览摘要。
- 进入“维护绑定”后，可以看到按设备分组的正式绑定测点。
- 整机解绑只在设备级入口出现，删除后该设备下全部正式测点消失。
- 删除测点只影响当前单条绑定，不影响同设备其他测点。
- 新增测点后，正式绑定列表立即出现新绑定。
- 替换测点后，旧测点不再显示，新测点显示为正式绑定。
- 待治理转正入口继续独立存在，且不会与正式绑定列表混淆。

### 11.2 规则验收

- 仍然保持一设备只能绑定一个风险点。
- 仍然保持同一风险点下，同设备同测点不能重复绑定。
- 风险点页不能直接修改设备主档信息。

### 11.3 真实环境验收

- 验收仍使用 `spring-boot-iot-admin/src/main/resources/application-dev.yml` 或覆盖它的环境变量。
- 不回退到 H2 验收路径。
- 若真实库缺少 pending 相关表或正式绑定新增接口依赖字段，必须明确报告环境阻塞，不得回退到伪环境。

## 13. 风险与后续观察点

- 若后续业务坚持要“直接修改设备”，需要先确认它究竟是要改设备主档，还是要改风险点绑定关系，不能继续混写。
- 若后续业务坚持要“一键替换设备并自动迁移全部测点”，需要新增针对测点兼容性的独立设计，不适合在本轮最小闭环里顺手塞入。
- 若后续要把“绑定来源”作为稳定展示口径，可能还需要补齐正式绑定来源回填规则或历史补录逻辑。
