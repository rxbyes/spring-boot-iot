# 产品经营工作台扁平化精修设计

> 日期：2026-03-30
> 范围：`spring-boot-iot-ui`
> 主题：把 `/products` 中“进入工作台”后的四个内页统一升级为扁平化、克制、可验收的产品经营工作台
> 状态：设计已确认，待 spec review 后进入实现计划

## 1. 背景

`产品定义中心` 已经完成第一轮统一经营工作台重构：

- `详情 / 物模型治理 / 关联设备 / 编辑治理` 已收口到同一抽屉
- 列表行内动作已压缩为 `进入工作台 + 删除`
- `/products` 页面不再并行挂载多套对象抽屉

但当前工作台内页仍未达到验收标准。主要问题不是“功能缺失”，而是“视觉语言未完全收口”：

1. 工作台头部和三个子页仍残留明显的 `hero + 渐变摘要卡 + 多组提示卡` 语法。
2. `经营总览`、`物模型治理`、`关联设备`、`编辑治理` 四页虽然都在同一抽屉里，但视觉结构还没有完全拉齐。
3. 部分页面首屏仍被厚重摘要区占据，真正主战区进入不够快。
4. 扁平化程度不足，整体更像“拼在同一抽屉里的多块页面”，而不是“统一语法的产品经营工作台”。

## 2. 目标

- 在不改变既有功能范围的前提下，把产品经营工作台四个内页统一升级为扁平化语法。
- 保留“经营简报式”的阅读主次，但压平装饰层、渐变层和多余提示层。
- 让四个内页共享同一套工作台骨架、间距节奏和信息层级。
- 确保 `经营总览` 仍保留设备规模主舞台，不退回普通档案详情抽屉。
- 确保 `关联设备`、`编辑治理` 的信息密度适当提升，不做成空泛简报页。

## 3. 非目标

- 不新增路由，不拆出 `/products/:id`。
- 不新增后端聚合接口。
- 不改变产品、设备、物模型相关后端契约。
- 不把统一工作台再次拆回多套独立抽屉。
- 不引入第二套品牌色、重营销风格或英文 eyebrow。

## 4. 已确认方向

用户已确认采用：

- **总骨架**：`A. 扁平经营简报式`
- **局部补强**：`关联设备` 与 `编辑治理` 吸收少量 `B. 扁平台账治理式` 的信息密度

落地解释固定为：

- `经营总览` 继续承担主舞台，不回退到台账页
- `物模型治理` 保留治理感，但整体压平，不再保留厚 hero 和多层提示卡
- `关联设备` 提高信息密度，让设备台账更早进入第一阅读层
- `编辑治理` 提高操作效率，让表单区更快成为主内容

## 5. 总体结构

### 5.1 统一工作台头部

头部统一压缩为“轻头部 + 真页签”结构，只保留：

- 产品名称
- 一句当前状态判断
- `productKey / protocolCode / nodeType / dataFormat`
- 四个真实视图切换：
  - `经营总览`
  - `物模型治理`
  - `关联设备`
  - `编辑治理`

头部移除或弱化的元素：

- 厚重 hero 背景
- 明显渐变大片
- 重复 eyebrow
- 第二组概况卡
- 与正文重复的提示卡

### 5.2 统一正文语法

四个内页都统一采用：

1. `区块标题`
2. `扁平主板块`
3. `必要的次级板块`

统一规则：

- 首屏尽量不出现两层以上同权重摘要卡
- 主舞台只保留一个阅读焦点
- 提示类信息优先内联化、短句化
- 表格或清单类主战区要尽快进入第一阅读层

## 6. 各视图设计

### 6.1 经营总览

#### 保留内容

- 设备规模主舞台
- 在线覆盖 / 30 日活跃 / 平均在线时长
- 当前判断
- 趋势判断
- 接入契约摘要
- 产品档案摘要
- 维护治理建议

#### 结构重排

总览页固定改为四层：

1. **主舞台**
   - 只突出 `关联设备总量`
   - 配一条经营判断短句
2. **辅助指标列**
   - `在线覆盖`
   - `30 日活跃`
   - `平均在线时长`
3. **双板块摘要**
   - `当前判断 + 趋势判断`
   - `契约摘要 + 档案摘要`
4. **治理板块**
   - `维护规则`
   - `变更前确认`

#### 设计约束

- 只允许一个主焦点，不再平铺 4 到 6 张同权重指标卡
- `趋势提示`、`接入提示`、`当前建议` 合并为更短的业务判断板块
- 契约与档案下沉到二级阅读层

### 6.2 物模型治理

#### 保留内容

- 候选提炼
- 正式模型
- 模式切换
- 候选目录
- 候选摘要
- 完整治理入口

#### 结构重排

物模型页固定改为三层：

1. **轻头部**
   - 标题
   - 模式切换
   - 一句治理说明
2. **候选/正式摘要条**
   - 属性候选
   - 待人工确认
   - 正式模型数量
   - 最近提炼时间
3. **主工作区**
   - 左：候选目录或类型目录
   - 中：候选卡列表或正式模型列表
   - 右：轻摘要轨道或治理动作

#### 设计约束

- 去掉明显 hero-stage 的视觉压迫感
- 候选卡片统一压平，不再通过大面积渐变制造层次
- `候选提炼` 与 `正式模型` 必须共享同一套页面语法，不做成两套风格

### 6.3 关联设备

#### 保留内容

- 设备总数
- 在线设备
- 离线设备
- 在线比例
- 关联设备台账
- 行内 `查看`

#### 结构重排

关联设备页固定改为两层半：

1. **紧凑指标带**
   - 3 到 4 个概览指标
2. **一句台账说明**
   - 放在表格标题区
3. **设备台账主区**
   - 表格尽快进入首屏

#### 设计约束

- 不再保留占高度较大的说明型概览卡
- 表格是主战区，指标摘要只做补充
- 允许比总览页更“实”，但不能退回厚重控制台语法

### 6.4 编辑治理

#### 保留内容

- 产品档案摘要
- 编辑影响提示
- 基础档案
- 接入基线
- 补充说明
- 保存 / 取消

#### 结构重排

编辑页固定改为三层：

1. **档案摘要条**
   - `产品 Key`
   - `协议编码`
   - `节点与状态`
2. **内联治理提示**
   - 一条 `StandardInlineState` 风格提示
3. **表单主区**
   - `基础档案`
   - `接入基线`
   - `补充说明`

#### 设计约束

- 去掉“编辑 hero + 编辑影响提示卡”双层重头部
- 表单必须更早进入主视觉层
- 说明和提醒改短句化，不再用两张并列说明卡占首屏

## 7. 视觉与样式规范

### 7.1 扁平化基线

- 白色板块 + 轻描边 + 轻阴影
- 渐变仅允许极弱背景辅助，不可成为主视觉主体
- 首屏避免连续多层圆角大卡片叠加
- 统一使用现有品牌橙、蓝灰正文和浅灰底板

### 7.2 统一节奏

- 区块间距统一收紧
- 头部与正文之间的距离缩短
- 卡片、列表、表单块采用同一档位圆角与内边距
- 同层级标题、正文、辅助文案字号保持统一

### 7.3 提示表达

优先级从高到低固定为：

1. `StandardInlineState`
2. 区块内单句说明
3. 必要时的轻 notice

禁止回流：

- 大块双色提示卡
- 多张并列说明墙
- 正常状态下的厚重 danger/warning 展示

## 8. 交互与状态

- 视图切换继续保活，不反复销毁已加载工作区
- Loading / Empty / Error 状态统一改为轻量表达
- 编辑保存成功后继续回写：
  - 当前工作台头部
  - `/products` 当前列表结果
- 物模型、设备、编辑页内部不新增跨页跳转菜单

## 10. 测试目标

本轮实现至少验证：

1. 统一工作台头部仍只保留一套产品上下文
2. `经营总览` 仍保留设备规模主舞台，不退回平铺摘要卡
3. `物模型治理` 的候选/正式模式共享同一扁平化语法
4. `关联设备` 的首屏以设备台账为主，不再被厚概览区挤压
5. `编辑治理` 的表单更早进入首屏，影响提示改为内联表达
6. `/products` 的统一工作台回写逻辑不回退

## 12. 风险与控制

- 风险：为了扁平化而把总览页做得过空
  - 控制：保留主舞台，不删主信息，只压平装饰层
- 风险：设备页和编辑页因追求简报感而牺牲操作效率
  - 控制：两页明确吸收少量 `B` 的密度，提高主战区优先级
- 风险：四页各自调样式，最后仍不统一
  - 控制：先改统一工作台壳层，再按同一板式收四个内页
