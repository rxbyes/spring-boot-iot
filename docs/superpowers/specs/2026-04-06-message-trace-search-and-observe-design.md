# 链路追踪台快速搜索与观测动作收口设计

> 日期：2026-04-06
> 范围：`spring-boot-iot-ui/src/views/MessageTraceView.vue`、`spring-boot-iot-device/src/main/java/com/ghlzm/iot/device/controller/DeviceMessageLogController.java`、`spring-boot-iot-device/src/main/java/com/ghlzm/iot/device/dto/DeviceMessageTraceQuery.java`、`spring-boot-iot-device/src/main/java/com/ghlzm/iot/device/service/impl/DeviceMessageServiceImpl.java`、`docs/03-接口规范与接口清单.md`、`docs/06-前端开发与CSS规范.md`、`docs/08-变更记录与技术债清单.md`、`docs/11-可观测性、日志追踪与消息通知治理.md`、`docs/15-前端优化与治理计划.md`
> 主题：修复链路追踪台消息类型筛选失效问题，收口“观测”动作语义，并把 TraceId / 设备编码 / 产品标识提炼为统一快速搜索
> 状态：设计已确认，待写入实现计划

## 1. 背景

`/message-trace` 当前已经收口为同一路由下的双模式工作台：

1. `链路追踪`
2. `失败归档`

其中 `链路追踪` 的职责是按 `TraceId`、设备编码、产品标识、消息类型和 Topic 复盘消息日志与固定 Pipeline；`失败归档` 的职责是承接 MQTT / `$dp` 接入失败样本，并继续联动异常观测台与链路追踪台。

但当前链路追踪台仍存在三类问题：

1. `消息类型` 筛选不生效。
2. `链路追踪` 列表中的正常链路也会显示 `观测` 动作，造成“正常复盘”和“异常排障”语义混杂。
3. `TraceId / 设备编码 / 产品标识` 仍拆成多个入口，没有提炼成统一快速搜索，首屏交互成本偏高。

本次设计目标不是扩一套新诊断能力，而是在现有 `/message-trace` 双模式结构中，把筛选、动作语义和前后端查询口径一起收稳。

## 2. 目标

1. 修复链路追踪台 `消息类型` 筛选失效问题。
2. 把 `TraceId / 设备编码 / 产品标识` 收口为一个统一快速搜索入口，提升首屏定位效率。
3. 让 `链路追踪` 与 `失败归档` 的行内动作语义重新与模式职责对齐。
4. 保持现有双模式路由结构和跨页诊断上下文协议稳定，不新增第二条静态路由，也不引入新的诊断页头结构。

## 3. 非目标

1. 本次不重做 `/message-trace` 详情抽屉。
2. 本次不新增新的诊断模式或新的跨页工作台。
3. 本次不扩展 `message-flow` 时间线字段或失败归档详情字段。
4. 本次不把 `链路追踪` 改造成全文模糊搜索台账。

## 4. 方案对比

### 4.1 方案 A：按模式语义硬收口

做法：

1. `链路追踪` 模式只保留 `详情`。
2. `失败归档` 模式继续保留 `详情 / 追踪 / 观测`。
3. `链路追踪` 顶部改为一个统一快速搜索框，命中 `TraceId / 设备编码 / 产品标识`。
4. `消息类型`、`Topic` 收到“更多筛选”。
5. 前后端统一补 `keyword` 查询参数和消息类型兼容映射。

优势：

1. 语义最清晰，正常链路只负责复盘，异常样本才进入异常观测。
2. 与当前 `/message-trace` 双模式设计天然一致。
3. 能同时解决首屏交互、动作误导和消息类型筛选失效。

代价：

1. 需要前后端同时改查询契约。

### 4.2 方案 B：链路追踪里按行状态条件显示 `观测`

做法：

1. 正常链路只显示 `详情`。
2. 只有异常链路才显示 `观测`。

问题：

1. 需要先定义并持续维护“异常链路”的判定口径。
2. 容易出现“为什么这条有观测、那条没有”的解释成本。
3. 当前消息日志记录本身并没有一条稳定、面向前端的异常布尔标记可直接复用。

### 4.3 方案 C：只做最小修补

做法：

1. 仅修复 `消息类型` 筛选。
2. 保留现有多输入框首屏结构。
3. 保留 `观测` 常驻。

问题：

1. 无法解决正常链路和异常排障动作混杂的问题。
2. 首屏仍然过重，用户还要先判断该填哪个框。
3. 只修接口、不修信息架构，验收反馈仍会反复出现。

### 4.4 最终决策

采用方案 A。

即：

1. 用模式语义而不是行文案猜测来定义动作可见性。
2. 用统一 `keyword` 承接标识型定位。
3. 用真实主链路消息类型值替换前端失真的旧枚举。

## 5. 最终设计

### 5.1 链路追踪首屏筛选结构

`链路追踪` 模式的顶部筛选区改为“一个快速搜索 + 更多筛选”结构。

#### 5.1.1 快速搜索

快速搜索框固定承接以下三个高频定位口径：

1. `TraceId`
2. `设备编码`
3. `产品标识`

占位文案改为：

`快速搜索（TraceId / 设备编码 / 产品标识）`

用户输入后，页面不再要求先判断应该填写哪个字段，而是统一通过一个 `keyword` 参数发给后端做标识型检索。

#### 5.1.2 更多筛选

以下条件收口到“更多筛选”：

1. `消息类型`
2. `Topic`

保留原则：

1. 高频定位先走快速搜索。
2. 精细排查再展开更多筛选。
3. 不再把 `设备编码` 和 `产品标识` 作为首屏独立输入框与快速搜索重复出现。

#### 5.1.3 筛选结果提示

快速搜索继续保留单独的紧凑标签：

`快速搜索：<keyword>`

该标签作为快速搜索的唯一显式提示来源，不在 `StandardAppliedFiltersBar` 里重复渲染同一条 `keyword` 标签，避免出现两份重复提示。

其余高级筛选条件仍通过 `StandardAppliedFiltersBar` 展示和移除。

### 5.2 行内动作语义收口

#### 5.2.1 链路追踪模式

`链路追踪` 模式下，行内动作固定只保留：

1. `详情`

不再显示：

1. `观测`

这里采用“直接隐藏”而不是“可见但禁用”，原因是正常链路本身不需要异常观测动作；如果动作不该被使用，就不应继续占据用户视线。

桌面表格和移动卡片都按同一规则收口，避免双端语义不一致。

#### 5.2.2 失败归档模式

`失败归档` 模式继续保留：

1. `详情`
2. `追踪`
3. `观测`

因为失败归档中的每一行天然都是异常样本，需要继续联动：

1. 链路追踪台
2. 异常观测台

#### 5.2.3 列宽与共享动作列

`链路追踪` 模式移除 `观测` 后，操作列宽度同步收口为单动作档位，不再按旧双动作宽度保留多余空白。

该收口仍需继续复用共享 `StandardWorkbenchRowActions` 与列宽工具，不新增页面私有动作列样式。

### 5.3 查询契约与后端口径

#### 5.3.1 新增 `keyword`

为以下接口新增统一查询参数 `keyword`：

1. `GET /api/device/message-trace/page`
2. `GET /api/device/message-trace/stats`

`keyword` 的语义是“标识型快速定位”，固定命中以下字段之一：

1. `trace_id`
2. `device_code`
3. `product_key`

后端查询逻辑使用同组 `OR` 命中，而不是把一个关键词同时塞进三个字段做 `AND` 过滤。

示意规则：

1. `trace_id = keyword`
2. 或 `device_code = keyword`
3. 或 `product_key = keyword`

不采用全文 `%LIKE%` 模糊搜索，原因如下：

1. 这三类字段本身都是强标识。
2. 当前链路追踪表是日志型台账，优先保持索引可用和结果可解释。
3. 用户的真实需求是“用一个标识定位一条链路”，而不是全文检索长文本。

#### 5.3.2 现有独立字段兼容策略

现有接口参数继续保留：

1. `traceId`
2. `deviceCode`
3. `productKey`
4. `messageType`
5. `topic`

兼容原则：

1. 老入口、老链接、跨页诊断上下文仍可继续传 `traceId/deviceCode/productKey/topic`。
2. `keyword` 只用于本轮新的快速搜索，不替代现有跨页诊断 query 协议。
3. 当 `keyword` 与独立字段同时出现时，后端按“独立字段 + keyword”共同生效；其中独立字段是显式约束，`keyword` 是额外的快速定位补充。

这样可以避免为了一个前端交互优化，反向牵动整个诊断链路的 query 协议。

#### 5.3.3 `messageType` 真实值对齐

当前主链路的 MQTT Topic 解析和协议落库已经以真实消息类型值为准，首批稳定值包括：

1. `property`
2. `event`
3. `status`
4. `reply`
5. `service`（如后续链路存在该值）

因此，`链路追踪` 模式中的前端消息类型下拉改为按真实值展示：

1. `property` -> `属性上报`
2. `event` -> `事件上报`
3. `status` -> `状态上报`
4. `reply` -> `命令回执`
5. `service` -> `服务调用`

不再继续以 `report / online / offline` 作为当前主筛选选项，因为这些值与当前主链路真实落库口径已经分叉，是导致“消息类型搜索不生效”的核心原因之一。

#### 5.3.4 历史别名兼容

为兼容旧入口和历史参数，后端对 `messageType` 至少保留以下归一：

1. `report` -> `property`

其余值仍按原值透传查询，不直接报错。

这样可以保证：

1. 老前端或老链接即使仍传 `report`，也不会直接查空。
2. 新前端统一只展示真实主链路消息类型值。

### 5.4 前端状态与上下文恢复

#### 5.4.1 页面内状态

`MessageTraceView` 的页面内查询状态新增 `keyword` 字段，作为快速搜索的唯一内部承载字段。

原先由 `quickSearchKeyword <-> traceId` 的单向映射逻辑改为：

1. `quickSearchKeyword <-> keyword`

#### 5.4.2 跨页上下文

本次不改造跨页诊断上下文协议。

仍然沿用已有的：

1. `traceId`
2. `deviceCode`
3. `productKey`
4. `topic`

原因是：

1. 其他诊断页当前已经稳定消费这套字段。
2. 本轮“快速搜索统一化”主要是链路追踪页内交互优化，不值得为了一个输入框去扩大跨页协议改造面。

因此：

1. 其他页面跳入 `/message-trace` 时，仍然优先按显式 `traceId/deviceCode/productKey/topic` 恢复筛选。
2. 用户在链路追踪页内自行输入的 `keyword`，主要服务本页查询，不要求向其他页面扩散成新的协议字段。

### 5.5 验收与测试设计

#### 5.5.1 前端回归

至少补齐以下前端用例：

3. `链路追踪` 模式下不再渲染 `观测` 按钮。
4. `失败归档` 模式继续渲染 `详情 / 追踪 / 观测`。
5. `消息类型` 下拉选 `property/status/reply/event` 后，请求参数正确透传。
6. 桌面表格和移动卡片的动作可见性保持一致。

#### 5.5.2 后端回归

至少补齐以下后端用例：

1. `keyword` 能按 `trace_id / device_code / product_key` 生成同组 `OR` 查询。
2. `stats` 查询复用与分页一致的 `keyword` 口径。
3. `messageType=report` 时会归一到 `property`。
4. 显式 `messageType=property/status/reply/event/service` 时按真实值过滤。

#### 5.5.3 页面验收口径

本轮页面验收至少覆盖：

1. 输入一条 `TraceId` 能命中对应链路。
2. 输入一条设备编码能命中对应链路。
3. 输入一条产品标识能命中对应链路。
4. `消息类型=状态上报` 能筛到 `status` 类型消息。
5. `链路追踪` 行内动作只剩 `详情`。
6. 切到 `失败归档` 后，`观测` 仍可正常使用。

## 7. 假设

1. 当前消息日志真实主链路的消息类型仍以 `property / event / status / reply / service` 为主。
2. `链路追踪` 的主要用户目标仍是“先定位链路，再看详情”，而不是在首屏做全文探索。
3. 失败排障入口继续由 `失败归档` 与 `异常观测台` 承担，`链路追踪` 不再额外承担“异常入口集合页”的职责。
