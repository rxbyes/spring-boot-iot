# 产品定义中心详情工作台设计

> 日期：2026-03-29
> 范围：`spring-boot-iot-ui`
> 主题：把 `/products` 产品定义详情从常规右侧详情抽屉升级为更宽的沉浸式“经营简报式”详情工作台
> 状态：设计已在对话中确认，待书面 spec 审阅

## 1. 背景

当前 `产品定义中心` 详情抽屉已经完成第一轮结构治理，具备以下正确基线：

- 保持 `StandardDetailDrawer` 统一交互。
- 保持“秒开 + 后台补数 + 本地短 TTL 缓存”的详情体验。
- 详情内容已经拆成 `产品汇总 / 接入契约 / 产品档案 / 设备活跃度 / 维护与治理` 五块。
- 详情页已经纳入浅色控制台风格，不再是原始字段堆叠。

但当前详情页仍存在 4 个明显问题：

1. 首屏焦点不够集中，多个区块权重接近，打开后没有“一眼先看什么”的明确答案。
2. 设备规模与活跃度虽然已接入真实数据，但仍更像“卡片集合”，不像一个有主次关系的经营摘要。
3. 接入契约、产品档案、治理建议都在争夺首屏注意力，页面读起来更像“整理过的字段详情”，而不是“商务简报型详情页”。
4. 当前 `42rem` 常规抽屉宽度不够支撑更强的主舞台结构，无法充分表达“规模主舞台 + 副指标列 + 下沉核对区”的秩序。

## 2. 目标

- 让详情页第一眼先看到 `设备规模 + 活跃表现`，形成明确主焦点。
- 在保持现有品牌橙、蓝灰正文、浅灰控制台底色不变的前提下，把详情升级为更精致现代的商务工作台。
- 把页面气质从“普通详情抽屉”提升为“高管简报式沉浸详情工作台”。
- 继续保留真实业务内容和现有交互能力，只重排信息结构与视觉层级。
- 确保新版详情页与当前 `接入智维` 的“静稳控制台基线 + 少量现代 SaaS 留白”风格一致。

## 3. 非目标

- 不新增一级路由，不把详情改成独立页面。
- 不修改 `GET /api/device/product/{id}` 接口或新增后端聚合接口。
- 不改变现有产品详情字段口径、活跃度口径和治理文案语义。
- 不回退 `秒开 + 后台补数` 基线，不重新引入红色补数提示。
- 不为单页发明新的私有主题色、营销式渐变头或第二套按钮体系。

## 4. 设计结论

本轮采用用户确认的推荐方案：

- **主方向：经营简报式工作台**
- **信息优先级：设备规模与活跃表现优先**
- **视觉气质：精致现代，但保持商务克制**
- **交互形态：更宽的沉浸式详情工作台，不新增一级路由**

选择该方向的原因：

1. 它最符合“高管简报式”而不是“运营看板式”的阅读节奏。
2. 它允许首屏只保留一个最强焦点，避免继续把详情做成多块同权重白卡。
3. 它既能保留现有控制台配色和共享抽屉规范，又能把 `/products` 详情显著拉向更现代的 B 端工作台气质。

## 5. 总体结构

### 5.1 抽屉定位

- 继续复用 `StandardDetailDrawer`。
- 从当前常规详情抽屉升级为更宽的沉浸式工作台抽屉。
- 本轮详情工作台宽度固定从当前 `42rem` 提升到 `60rem`，使桌面端能够稳定容纳“主舞台 + 副指标列 + 下沉区块”的布局。

### 5.2 信息层次

新版详情统一重组为 4 层：

1. **Hero Stage**
2. **Trend Stage**
3. **Contract & Archive Stage**
4. **Governance Stage**

### 5.3 Hero Stage

首屏改为“主舞台 + 副指标列”结构：

- 左侧主舞台：
  - 关联设备总量主数字
  - 一句经营判断
  - 一段短摘要，解释在线率、活跃表现与当前状态
- 右侧副指标列：
  - 在线设备
  - 30 日活跃
  - 平均在线时长

设计原则：

- 首屏只能有一个最强视觉中心，即设备规模主舞台。
- 副指标只做补充，不与主数字争抢视觉权重。
- 状态、生命周期、轻标签等信息保留，但降级为辅助信息，不再平铺成同层级摘要。

### 5.4 Trend Stage

当前“设备活跃度”从 5 张并列数字卡升级为“趋势主区 + 次级摘要”：

- 左侧：活跃趋势主区，强调近期活跃表现和变化判断。
- 右侧：今日活跃、7 日活跃、最长在线时长等轻量摘要。

设计原则：

- 活跃度是首屏主焦点的延续，不再作为普通插入区块处理。
- 趋势区优先表达“走势和判断”，而不是继续把所有字段做成同权重统计卡。

### 5.5 Contract & Archive Stage

接入契约与产品档案整体下沉为第三阅读层：

- `接入契约` 保留协议编码、节点类型、数据格式等核对字段。
- `产品档案` 保留厂商、产品编号、产品 Key、创建时间、产品说明。

设计原则：

- 这些内容依然重要，但不再占据首屏主焦点。
- 视觉上采用更安静的核对区语法，适合真正需要时再看，而不是一进详情就抢注意力。

### 5.6 Governance Stage

治理区作为页面收尾：

- 当前建议
- 维护规则
- 变更前确认

设计原则：

- 结尾更像“经营决策建议”，而不是“补充说明集合”。
- 保留当前治理语义，但压缩成更简短、更好读的短句和清单。

## 6. 视觉语言

### 6.1 字体层级

本轮建立 5 级稳定字体梯度：

1. `Display`：首屏主数字与一级经营判断
2. `Section`：二级区块标题
3. `Metric`：副指标数值
4. `Body`：解释性短句
5. `Caption`：标签、提示与辅助说明

设计要求：

- 不再把主要数字、区块标题和说明文案都挤在 `12px ~ 14px` 的单一层次里。
- `Display` 只用于首屏主舞台，避免多个大数字同时出现。
- 说明文案统一改为短句，不再堆大段解释墙。

### 6.2 配色规则

继续严格使用现有控制台 token，不新增第二套品牌色：

- 标题与关键数字：深蓝灰
- 正文与辅助说明：蓝灰
- 主语义强调：品牌橙
- 核对区背景：浅灰控制台底

控制规则：

- 橙色只允许用于主舞台、趋势线、主按钮和极少量状态强调。
- 契约、档案、说明、标签全部回到更安静的蓝灰中性色层。
- 不再把多个区块同时做成浅橙大板块，避免页面发热、发满、发廉价。

### 6.3 材质规则

详情内部建立 3 种卡片材质等级：

1. **主舞台卡**：采用极轻的橙蓝混合底纹，用于承接规模和经营判断。
2. **安静信息卡**：用于契约、档案、副指标，背景更白、边框更轻。
3. **治理提示卡**：只保留极淡提醒，不做强告警式色块。

设计原则：

- 页面不再是一排同权重白卡。
- 主次关系主要通过材质、字号和留白建立，而不是靠更重的颜色堆叠。

## 7. 组件映射与实现边界

### 7.1 组件映射

- 外层抽屉：继续使用 `StandardDetailDrawer`
- 头部动作：继续使用现有 `header-actions`
- 底部动作：继续使用 `StandardDrawerFooter`
- 数据来源：继续使用现有产品详情接口和缓存逻辑

### 7.2 页面实现边界

首选实现位置：

- [ProductWorkbenchView.vue](/Users/rxbyes/Downloads/rxbyes/idea/spring-boot-iot/spring-boot-iot-ui/src/views/ProductWorkbenchView.vue)

允许的实现方式：

- 在当前页面内重组详情模板与局部样式。
- 如需拆分结构，只允许抽离产品详情私有片段组件，不新增共享通用组件。

本轮不做：

- 为这次设计新建一整套通用共享组件。
- 把详情工作台拆成新的路由页或新的页面壳。

## 8. 交互与数据规则

- 继续保持详情“秒开”，列表摘要仍作为首屏预填数据。
- 抽屉打开后异步补齐完整详情，补数失败时默认静默失败，不显示打断式错误提示。
- 现有详情缓存策略继续保留，仍以 `id + updateTime` 作为复用和补数判断依据。
- `todayActiveCount / sevenDaysActiveCount / thirtyDaysActiveCount / avgOnlineDuration / maxOnlineDuration` 继续作为活跃度展示基础字段，不新增后端字段依赖。
- 头部高频动作固定保留 `编辑`。
- 底部动作固定保留 `关闭 / 查看设备`。
- `物模型设计器` 不进入本轮详情工作台头部动作，继续留在列表行操作语义中。

## 9. 响应式与可用性

- 桌面端：采用双列和“主舞台 + 副列”结构，充分利用更宽抽屉空间。
- 平板端（约 `<= 1024px`）：副指标列、趋势摘要与档案区统一折叠为单列。
- 移动端或窄屏：所有板块按阅读顺序改为纵向堆叠，不保留双列结构。
- 保持现有 `StandardDetailDrawer` 的 loading / empty / error 语义，不新增另一套状态表达。

## 11. 验证方式

### 11.1 视觉验证

- 打开详情后，是否第一眼先看到设备规模与活跃表现。
- 接入契约和产品档案是否明显退居第二阅读层。
- 页面整体是否更像商务简报，而不是“多张白卡的字段集合”。

### 11.2 交互验证

- 点击详情仍然秒开。
- 后台补数不会打断用户。
- 更宽抽屉下头部动作、主舞台和副指标区不会互相挤压。
- 窄屏下能够自然折叠，不出现内容横向压缩和漂移。

### 11.3 工程验证

- `npm run build`
- `npm run test`
- 如样式或共享 token 调整触及守卫，再执行 `npm run style:guard`

## 12. 假设与风险

### 12.1 假设

- 当前产品详情接口持续返回活跃度字段，且现有缓存策略可继续复用。
- 本轮只做前端详情结构与视觉升级，不依赖后端新增聚合能力。
- 现有 `StandardDetailDrawer` 能承接更宽尺寸而不破坏全局抽屉行为。

### 12.2 风险与控制

- 风险：更宽工作台抽屉如果控制不好，容易滑向“营销式大板块”或“驾驶舱过载”。
- 控制：保留现有控制台配色，不新增私有主题色；首屏只保留一个主焦点；橙色只做主语义强调。
