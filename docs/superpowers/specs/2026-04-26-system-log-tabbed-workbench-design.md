# `/system-log` 异常观测台选项卡收口设计

## 1. 背景

`/system-log` 当前已经承接了三类真实能力：

1. **异常排查**：系统异常筛选、异常台账、详情抽屉、证据抽屉、回链路追踪。
2. **观测热点**：性能慢点 Top、慢点明细/趋势下钻、调度任务台账。
3. **归档治理**：归档批次台账、异常摘要、批次 compare、确认报告预览。

随着 G2-G5 的能力持续叠加，这一页已经具备足够强的可观测与治理能力，但页面首屏也明显变重：

1. 三类任务在一张长页里顺序堆叠，进入页面时不容易判断“我这次是来查异常、看热点，还是核治理批次”。
2. 不同区块同时存在独立 loading、筛选和摘要语义，用户会感知到“功能都在，但层次有点复杂”。
3. `AuditLogView.vue` 同时承担审计中心与异常观测台两种模式，系统模式如果继续横向加能力，维护成本会持续上升。

用户已经明确了本轮方向：

1. **三者平级，但分成清晰选项卡。**
2. 顶部保留一条**轻量总览带**。
3. 默认进入 **异常排查**。
4. 三个选项卡**各自记住自己的筛选状态**。
5. 本轮**只作用于 `/system-log`**，不改变 `/audit-log` 审计中心模式。

## 2. 目标与非目标

### 2.1 目标

1. 把 `/system-log` 的系统模式重排为“轻量总览带 + 三张平级选项卡 + 共享抽屉”。
2. 让 `异常排查 / 观测热点 / 归档治理` 各自只承载自己的主任务，不再同屏顺序堆叠。
3. 保留现有深挖链路，包括：
   - 异常详情
   - 证据抽屉
   - 慢点明细/趋势下钻
   - 归档批次详情抽屉中的 compare + report preview
4. 让系统模式的 UI 结构真实降复杂，而不是只换个排版。
5. 在实现层顺手把系统模式拆成更清晰的内部 panel，避免 `AuditLogView.vue` 继续膨胀。

### 2.2 非目标

1. 本轮不改 `/audit-log` 的审计中心交互、结构或文案。
2. 本轮不新增 `/system-log` 子路由，也不拆成 `errors / hotspots / archives` 独立路由。
3. 本轮不新增新的可观测后端接口，不重写 G2-G5 已完成的归档批次读侧语义。
4. 本轮不新增新的跨页菜单或页头跳转墙。
5. 本轮不把深挖能力改造成独立详情页，仍以抽屉与同页下钻为主。

## 3. 方案对比与推荐

### 3.1 方案 A：单页内真选项卡分仓

做法：

1. 继续使用 `/system-log` 当前路由。
2. 在系统模式内增加轻量总览带和三张真正的 tab panel。
3. `AuditLogView.vue` 只负责页头、总览带、当前 tab 和共享抽屉。
4. 各 tab 的正文区域拆成独立内部 panel 组件。

优点：

1. 与现有入口连续，不改变路由心智。
2. 能真实把三类任务拆开，而不是继续长页堆叠。
3. 风险和改动范围都可控。

缺点：

1. 系统模式仍然共享一条页面级状态树，需要清楚划分 tab 责任。
2. 需要补一层内部组件拆分。

### 3.2 方案 B：伪选项卡 + 同页滚动分区

做法：

1. 顶部看起来像 tab。
2. 点击后只是滚动到同页不同 section。
3. 区块仍全部同时存在。

优点：

1. 表面改动小。
2. 实现速度快。

缺点：

1. 本质上没有降低复杂度。
2. loading、摘要、筛选仍会同时压在一张长页里。
3. 容易把“结构重排”做成“导航样式调整”。

### 3.3 方案 C：子路由页签化

做法：

1. 继续以 `/system-log` 为入口，但拆出子路由或 query 深链：
   - `/system-log?tab=errors`
   - `/system-log?tab=hotspots`
   - `/system-log?tab=archives`
2. 每个子页可以更强隔离自己的状态与文档语义。

优点：

1. 结构最彻底。
2. 长期可扩展性最好。

缺点：

1. 本轮明显超范围。
2. 需要一起调整路由、上下文恢复、深链与文档。
3. 已经从“页面收口”变成“路由级 IA 改造”。

### 3.4 推荐

采用 **方案 A：单页内真选项卡分仓**。

原因：

1. 它最符合用户这轮真实诉求：页面太复杂，需要分仓，但不需要大改入口。
2. 它能真正把三类任务拆开，同时保留当前 `/system-log` 的诊断页定位。
3. 它允许本轮顺手完成组件边界收口，为后续继续演进留出空间。

## 4. 页面信息架构

### 4.1 总体骨架

系统模式下的 `/system-log` 固定调整为：

1. **轻量页头**
   - 标题：`异常观测台`
   - 描述：继续回答“当前节点：后台异常核对”与下一步去向
2. **轻量总览带**
   - 不做厚卡片，不回到第二层驾驶舱概况墙
   - 用一行紧凑摘要项回答“哪一类任务现在更值得先看”
3. **三张平级选项卡**
   - `异常排查`
   - `观测热点`
   - `归档治理`
4. **共享抽屉层**
   - 异常详情
   - 证据抽屉
   - 归档批次详情

### 4.2 页头约束

页头继续沿用诊断页轻量语法：

1. 不新增第二条页内面包屑。
2. 不回流 `链路追踪台 / 数据校验台` 等跨页头部按钮。
3. 说明文案继续强调本页职责与下一步，而不是重讲整棵诊断树。

### 4.3 轻量总览带

总览带放在页头下、tab 上方，采用**紧凑摘要项/胶囊条**而不是厚卡片。

建议固定 4 个口径：

1. `异常`
2. `慢点`
3. `调度`
4. `异常批次`

交互规则：

1. 点击 `异常`：切到 `异常排查`
2. 点击 `慢点`：切到 `观测热点`
3. 点击 `调度`：切到 `观测热点`
4. 点击 `异常批次`：切到 `归档治理`

职责边界：

1. 总览带只负责方向判断与快捷切换。
2. 不承载详细筛选。
3. 不替代 G4/G5 已有的归档治理摘要卡。

### 4.4 选项卡

固定 3 张平级 tab：

1. `异常排查`
2. `观测热点`
3. `归档治理`

默认进入：

1. 首次进入 `/system-log` 时默认选中 `异常排查`

本轮不做：

1. URL query 持久化当前 tab
2. 路由级子页签

## 5. 各选项卡内容边界

### 5.1 `异常排查`

这是默认首屏，也是系统模式的主工作区。

保留：

1. 当前异常筛选区
2. 异常台账列表
3. 行级 `证据 / 追踪`
4. 异常详情抽屉
5. 证据抽屉

不保留：

1. `性能慢点 Top`
2. `调度任务台账`
3. `归档批次台账`

补充约束：

1. 当前异常统计 tag 可继续保留，但保持异常语义，不外溢到其它 tab。
2. “更多操作”继续只属于这张主工作区，不在其它 tab 复制。

### 5.2 `观测热点`

这一张回答“系统现在慢在哪、调度最近怎么样”。

保留：

1. `性能慢点 Top`
2. 慢点明细下钻
3. 慢点趋势下钻
4. `调度任务台账`

不保留：

1. 异常主列表
2. 归档批次台账

布局建议：

1. 先展示 `性能慢点 Top`
2. 再展示 `调度任务台账`
3. 慢点明细与趋势仍然跟随热点区域展开，不单独升级成新页

### 5.3 `归档治理`

这一张回答“日志治理有没有按确认结果落地”。

保留：

1. 当前归档治理异常摘要卡
2. `归档批次台账`
3. `批次号 / 状态 / 对比结论 / 创建日期 / 仅看异常`
4. 批次详情抽屉中的 `批次对比 + 确认报告预览`

不保留：

1. 异常主列表
2. 慢点 / 调度区块

补充约束：

1. G4/G5 的摘要卡下钻逻辑继续留在这张 tab 内，不搬到页头总览带。
2. 这张 tab 可以保持比另外两张更高的信息密度，因为它本身是治理台账。

## 6. 状态记忆与切换行为

### 6.1 每个 tab 各自记住状态

系统模式下为 3 个 tab 各自维护独立本地页面状态：

1. `errorTabState`
2. `hotspotTabState`
3. `archiveTabState`

各自记住自己的筛选、分页与局部 UI 状态，不互相污染。

建议的记忆范围：

#### `errorTabState`

1. 快速搜索关键词
2. 异常模块
3. 结果
4. 请求通道
5. 高级筛选
6. 分页

#### `hotspotTabState`

1. 当前选中的慢点热点
2. 当前趋势窗口（如 `24h / 7d`）
3. 调度台账分页
4. 当前展开的明细/趋势局部状态

#### `archiveTabState`

1. 批次号
2. 状态
3. 对比结论
4. 时间范围
5. `onlyAbnormal`
6. 当前 focus/hint
7. 分页

### 6.2 切换行为

1. 首次进入 `/system-log`
   - 默认进入 `异常排查`
   - 其它 tab 先不抢加载
2. 切换到某个 tab
   - 首次进入该 tab 时懒加载
   - 已经加载过则优先复用已有结果
3. `刷新列表`
   - 只刷新当前 tab
   - 不联动刷新另外两张
4. 手工修改某个 tab 的筛选
   - 只影响该 tab
   - 不清空其它 tab 已保存状态

### 6.3 降级规则

1. 某个 tab 接口失败
   - 错误只显示在当前 tab
   - 不影响其它 tab 使用
2. 某个 tab 正在加载
   - loading 只覆盖当前 tab
   - 不再出现整页多块区域一起转圈的感知
3. 关闭任何抽屉
   - 不清空当前 tab 状态
   - 不切换当前 tab

## 7. 组件拆分建议

### 7.1 视图边界

推荐在系统模式内新增 3 个内部 panel 组件：

1. `AuditLogErrorTabPanel`
2. `AuditLogHotspotTabPanel`
3. `AuditLogArchiveTabPanel`

`AuditLogView.vue` 继续负责：

1. `/audit-log` 与 `/system-log` 模式分流
2. 系统模式页头
3. 轻量总览带
4. 当前 tab key
5. 共享抽屉编排
6. 与诊断上下文恢复相关的页面级入口

各 panel 负责：

1. 本域筛选
2. 本域列表/区块渲染
3. 本域 loading / empty / error
4. 本域事件上抛

### 7.2 推荐目录

推荐落在现有审计/可观测相关组件目录下，例如：

1. `spring-boot-iot-ui/src/components/auditLog/AuditLogErrorTabPanel.vue`
2. `spring-boot-iot-ui/src/components/auditLog/AuditLogHotspotTabPanel.vue`
3. `spring-boot-iot-ui/src/components/auditLog/AuditLogArchiveTabPanel.vue`

若实现中发现总览带本身值得复用，可再增加：

4. `AuditLogSystemOverviewStrip.vue`

### 7.3 共享抽屉不下放

以下抽屉继续保留在页面级，而不是拆到单个 tab 内：

1. `AuditLogDetailDrawer`
2. 证据抽屉
3. 归档批次详情抽屉

原因：

1. 它们本质上是系统模式的共享证据层。
2. 继续放在 page-level，能避免重复注册与状态串扰。

## 8. 数据与事件流

### 8.1 不新增后端接口

本轮只做前端信息架构与组件分仓，不新增新的系统模式后端接口。

继续复用现有读侧：

1. 异常台账相关接口
2. `spans/slow-summary`
3. `spans/slow-trends`
4. `scheduled-tasks/page`
5. `message-archive-batches/page`
6. `message-archive-batches/overview`
7. `message-archive-batches/compare`
8. `message-archive-batches/report-preview`

### 8.2 页面级协调

页面级状态机需要承担两件事：

1. 当前 tab 与总览带切换
2. 共享抽屉的打开/关闭协调

其它查询与展示逻辑尽量下沉到 panel 内部，避免 `AuditLogView.vue` 再继续横向增长。

### 8.3 诊断上下文恢复

本轮继续保留系统模式已有诊断上下文恢复能力，但落点固定为：

1. 从链路追踪台或链路验证中心带入的异常语义上下文，默认落在 `异常排查`
2. 不因为本轮新增 tab，而改变现有异常排查入口的默认承接逻辑

## 9. 测试与验收

### 9.1 前端测试

至少覆盖：

1. `/system-log` 系统模式默认进入 `异常排查`
2. 轻量总览带渲染并支持切换到对应 tab
3. `异常排查 / 观测热点 / 归档治理` 只渲染各自负责的区块
4. tab 之间各自记住自己的筛选/分页状态
5. `刷新列表` 只刷新当前 tab
6. 某个 tab 报错时不影响其它 tab
7. 现有异常详情、证据抽屉、归档批次详情抽屉仍可正常打开
8. `/audit-log` 审计中心模式结构不受影响

### 9.2 浏览器验收

需要在本地浏览器至少复核：

1. `/system-log`
   - 默认 tab
   - tab 切换
   - 每个 tab 的 loading / empty / error 表现
   - 共享抽屉是否仍正常
2. `/audit-log`
   - 审计中心模式未被误伤

### 9.3 构建验收

至少运行：

1. `AuditLogView` 相关前端测试
2. 新增 panel 组件测试
3. `npm --prefix spring-boot-iot-ui run build`

## 11. 推荐结论

本轮 `/system-log` 采用以下固定方案：

1. 保持一条路由，不拆子路由
2. 只重构系统模式，不动审计中心模式
3. 顶部保留轻量总览带，但不做厚概况卡
4. 主体改为 `异常排查 / 观测热点 / 归档治理` 三张平级选项卡
5. 默认进入 `异常排查`
6. 三个选项卡各自记住自己的筛选和分页状态
7. 深挖证据能力继续留在共享抽屉层
8. 实现时把系统模式拆成独立 panel，完成 UI 收口与组件边界收口
