# 风险绑定正式绑定多测点选择设计

> 日期：2026-04-25
> 适用范围：`/risk-point` 风险绑定工作台、列表页旧“新增正式绑定”抽屉、`/api/risk-point` 正式绑定写侧

## 1. 背景

当前风险绑定正式绑定链路已经收口为“只允许绑定已发布到启用中 `risk_metric_catalog` 的正式目录字段”，这解决了旧页面曾经混入设备全量属性的问题。

但现状仍有两个明显缺口：

1. 正式绑定一次只能选择一个测点，无法满足同一台监测型设备在一个风险点下批量补齐多个正式测点的使用场景。
2. 当前前后端契约仍是单个 `metricIdentifier` 模型，导致即便页面改成多选，也会拆成多次请求、多次审批和多次写入，无法形成一致的审批与审计语义。

本次目标是在不改变“只允许正式目录字段进入正式绑定”这一边界的前提下，把“新增正式绑定”升级为一次选择多个测点并一次提交。

## 2. 目标

### 2.1 用户目标

- 风险绑定工作台中的“新增正式绑定”支持为同一台监测型设备一次选择多个正式测点。
- 列表页旧“新增正式绑定”抽屉与风险绑定工作台保持一致，不再保留一套单测点旧语义。
- 提交时只产生一次领域命令和一次审批语义，而不是前端循环发起多次单测点绑定。

### 2.2 系统目标

- 批量正式绑定继续只消费 `formal-metrics` 专用接口返回的目录字段。
- 批量写侧继续显式记录 `riskMetricId / metricIdentifier / metricName`，避免退回仅靠原始字段名运行。
- 批量审批、去重、回执和审计都保持确定性，不因局部失败产生“部分已生效、部分还待审批”的混乱状态。

## 3. 非目标

- 本次不把“更换测点”升级为批量替换；更换测点仍保持单条绑定单条替换。
- 本次不扩展视频设备、预警设备的绑定模型；设备级正式绑定继续保持现状。
- 本次不放宽目录约束，不允许未发布到 `risk_metric_catalog` 的正式字段或运行态字段进入正式绑定。
- 本次不把待治理转正与正式绑定合并为同一个接口，只复用其“批量 metrics[]”思路。

## 4. 现状问题

### 4.1 前端现状

- 列表页旧抽屉 `RiskPointView.vue` 当前只有一个 `bindForm.metricIdentifier`，提交时调用一次 `bindDevice(...)`。
- 风险绑定工作台 `RiskPointBindingMaintenanceDrawer.vue` 当前只有一个 `addForm.metricIdentifier`，新增正式绑定时也只调用一次 `bindDevice(...)`。
- 页面虽然已经改成只读 `formal-metrics`，但交互仍然是单选。

### 4.2 后端现状

- `POST /api/risk-point/bind-device` 当前请求体使用 `RiskPointDevice`，只承接单个 `riskMetricId / metricIdentifier / metricName`。
- `submitBindDevice` / `bindDevice` / 审批快照写入链路都是单绑定语义。
- `risk_point_device` 真相表按“一条绑定对应一条测点绑定记录”建模，没有问题；问题在于领域命令仍是单条。

### 4.3 业务影响

- 同一台设备要补齐多个正式测点时，用户需要重复开单和重复审批。
- 若前端自行循环提交，会形成多张审批单、多次成功提示和不一致的失败处理。
- 当前产品已经出现“正式目录为空”和“需要一次补多个测点”两个问题叠加，进一步放大了正式绑定流程阻力。

## 5. 方案比较

### 方案 A：前端多选，逐条调用现有单测点接口

优点：

- 后端改动小。

缺点：

- 一次操作会拆成多次审批、多次审计、多次成功或失败提示。
- 一旦中间一条失败，页面需要自行拼接部分成功语义，难以维持一致性。
- 与待治理转正现有 `metrics[]` 批量模型不一致。

### 方案 B：新增批量正式绑定请求，推荐

优点：

- 一次提交、一张审批单、一批写入，语义清晰。
- 可与待治理转正的批量模型保持一致。
- 更容易给出原子性结果和确定性审计。

缺点：

- 需要同步调整前后端契约、审批快照和测试。

### 方案 C：保留旧接口，再新增批量接口，仅工作台升级

优点：

- 可减轻旧页面改造压力。

缺点：

- `/risk-point` 列表页旧抽屉与风险绑定工作台语义再度分叉。
- 后续维护成本高，容易重新出现“一个入口支持多选、一个入口只支持单选”的体验断层。

结论：采用方案 B，并同步把旧抽屉与工作台都切到同一批量绑定语义。

## 6. 目标行为

### 6.1 页面行为

- 设备选择保持单选，仍支持按 `deviceCode` 输入过滤。
- 测点选择改为多选，候选仅来自 `GET /api/risk-point/devices/{deviceId}/formal-metrics`。
- 当设备没有任何正式目录字段时，继续显示“当前设备所属产品暂无可用于风险绑定的正式目录字段”。
- 当用户选择多个测点后，提交按钮一次提交全部已选测点。
- 提交成功后，绑定结果分组区立即显示新增的多条正式绑定记录。

### 6.2 提示与回执

- 未选设备：提示“请先选择设备”。
- 已选设备但未选任何测点：提示“请至少选择一个测点”。
- 直接生效：提示“已新增 N 个正式测点绑定”。
- 待审批：提示“已提交 N 个测点绑定申请，当前状态：待审批”。

### 6.3 去重与冲突

- 页面侧不展示同一设备下已绑定的测点作为候选。
- 服务端再次校验：同一风险点、同一设备、同一测点不得重复绑定。
- 批量请求体内若重复选择同一 `metricIdentifier`，服务端统一拒绝并返回明确业务提示。

## 7. 接口设计

### 7.1 新增批量正式绑定请求体

新增批量请求模型，建议命名为 `RiskPointBatchBindDeviceRequest`：

- `riskPointId`
- `deviceId`
- `deviceCode`
- `deviceName`
- `metrics[]`

其中 `metrics[]` 每项结构为：

- `riskMetricId`
- `metricIdentifier`
- `metricName`

### 7.2 接口策略

保留现有 `POST /api/risk-point/bind-device` 路径，但请求体从“单测点模型”升级为“设备 + metrics[] 批量模型”。

原因：

- 前端已有调用点较集中，保留路径可减少路由层扩散修改。
- 领域语义仍然是“为风险点绑定设备的正式测点”，只是从单条升级为批量。

### 7.3 原子性

批量正式绑定采用“全部校验通过后一次提交”的原子性策略：

- 若任一测点不合法、重复、越权或不在正式目录内，则整次请求失败，不做部分落库。
- 命中审批策略时，整批测点共享一张审批单；审批通过后整批生效。

## 8. 后端设计

### 8.1 控制层

- `RiskPointController.bindDevice(...)` 改为接收新的批量请求 DTO。
- 返回值继续保持 `GovernanceSubmissionResultVO`，但补充批量语义摘要字段，例如绑定条数。

### 8.2 服务层

在 `RiskPointBindingMaintenanceService` / `RiskPointBindingMaintenanceServiceImpl` 中新增批量正式绑定主方法，职责如下：

1. 校验风险点、设备和权限。
2. 校验 `metrics[]` 非空。
3. 对 `metrics[]` 进行去重。
4. 用 `listFormalBindingMetricOptions(deviceId, currentUserId)` 校验每个测点都属于当前设备可绑定正式目录。
5. 校验当前风险点下未重复绑定同一设备同一测点。
6. 命中审批时，生成一份包含全部测点的快照；不命中时，一次性写入全部正式绑定记录。

### 8.3 写库模型

`risk_point_device` 表保持不变，仍是一条记录对应一个正式测点绑定。批量正式绑定只是一次领域命令写入多条记录，不新增平行真相表。

### 8.4 审批快照

批量审批快照中需完整记录：

- 风险点信息
- 设备信息
- `metrics[]`
- 每个测点的 `riskMetricId / metricIdentifier / metricName`

这样审批落地和审计留痕都能恢复整批请求真相。

## 9. 前端设计

### 9.1 风险绑定工作台

`RiskPointBindingMaintenanceDrawer.vue`：

- `addForm.metricIdentifier` 改为 `addForm.metricIdentifiers: string[]`
- 测点选择 `el-select` 改为 `multiple`
- 提交前根据选中的 identifiers 组装 `metrics[]`
- 成功提示切换为批量语义

### 9.2 列表页旧抽屉

`RiskPointView.vue`：

- `bindForm.metricIdentifier` 改为 `bindForm.metricIdentifiers: string[]`
- 测点下拉改为多选
- 调用同一批量正式绑定接口
- 空目录提示与设备搜索保持现有行为

### 9.3 一致性要求

- 两个入口必须复用同一接口语义与同一提示口径。
- 不允许工作台支持多测点而旧抽屉继续停留单测点。

## 10. 测试与验证

### 10.1 后端

至少补齐以下测试：

- 批量正式绑定成功，直接生效时一次写入多条记录。
- 批量正式绑定命中审批时只生成一张审批单。
- 批量中包含目录外字段时整批拒绝。
- 批量中包含重复测点时整批拒绝。
- 批量中包含已绑定测点时整批拒绝。

### 10.2 前端

至少补齐以下测试：

- 正式绑定测点选择框支持多选。
- 旧抽屉与工作台都按 `metrics[]` 提交。
- 空目录场景继续显示空提示。
- 已绑定测点不会再次出现在候选中。
- 成功提示显示批量条数。

### 10.3 环境复核

真实环境至少复核：

1. 选择同一设备的多个正式目录测点并一次提交。
2. 审批关闭时直接生效并生成多条绑定记录。
3. 审批开启时只生成一张审批单。
4. 已无正式目录字段的产品仍为空且提示明确。

## 11. 风险与缓解

### 风险 1：旧审批快照反序列化兼容

缓解：

- 新批量请求体使用新的 DTO 和新的快照结构，审批执行器对旧单测点快照保留兼容解析。

### 风险 2：列表页旧抽屉与工作台再次分叉

缓解：

- 本次明确要求两个入口同时升级，不接受只改一个入口。

### 风险 3：用户误把大量测点一次全绑

缓解：

- 保持候选只来自正式目录。
- 页面继续过滤已绑定项。
- 后端整批去重与整批校验。

## 12. 结论

本次正式绑定多测点选择采用“单设备 + 多测点 + 一次领域命令 + 一张审批单 + 多条正式绑定记录”的批量模型。

这样既保留了“只允许正式风险目录字段进入正式绑定”的治理边界，又能把同一台监测型设备的一次补齐动作收口为完整、可审计、可回归的一次操作。
