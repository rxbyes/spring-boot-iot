# 对象洞察台切换到风险监测实时详情设计

> 日期：2026-04-01
> 主题：将 `/insight` 从“纯设备洞察页”收口为“风险监测对象洞察页”，主口径切换到 `risk-monitoring` 实时详情接口

## 1. 背景

当前对象洞察台 `spring-boot-iot-ui/src/views/DeviceInsightView.vue` 仍以设备接口为主：

- `GET /api/device/code/{deviceCode}`
- `GET /api/device/{deviceCode}/properties`
- `GET /api/device/{deviceCode}/message-logs`

这导致页面里的“当前风险等级”“趋势预览”“监测对象主身份”都只能在前端根据设备在线状态、属性条数、日志条数和少量数值信号做轻量推断。该口径对“设备洞察”成立，但对“风险运营对象洞察”不够准确，尤其存在以下问题：

- 风险等级并非来自风险监测绑定与风险点主数据，而是前端临时评分。
- 趋势图只会从消息日志里反解析 `properties` 数值字段，不知道当前绑定测点，也不知道当前对象到底属于哪类监测对象。
- 页面主身份仍偏“设备”，没有真正进入“风险点 / 监测对象 / 测点”语义。
- 首屏信息过多，除核心指标外还叠加了“当前建议动作”“一线建议”“运维建议”“研发建议”等次级内容，弱化了风险运营页的判断焦点。

本轮目标是在不破坏现有 `/insight` 路由入口的前提下，把对象洞察台主口径切到风险监测实时详情，并按新的展示优先级收口页面内容。

## 2. 目标

- 对象洞察台的主身份从“设备”切换为“风险监测绑定对象”。
- 当前风险等级、趋势和监测对象上下文统一来自 `risk-monitoring` 实时详情接口。
- 页面保留用户熟悉的 `/insight` 路由和“按设备编码进入”的现有入口，不强制同轮改造所有上游跳转。
- 页面信息结构按以下优先级固定：
  - 第一优先级：当前风险等级、在线状态、属性快照数、消息日志数。
  - 第二优先级：基础档案、研判依据、属性趋势预览。
  - 第三优先级：关键监测指标、设备属性快照、风险分析草稿。
- 删除本轮不需要的内容，避免对象洞察台继续承担“通用建议中心”的角色。

## 3. 非目标

- 不新增新的对象洞察路由。
- 不把对象洞察台改造成独立的风险监测列表页。
- 不在本轮引入新的后端聚合接口，优先复用现有 `riskMonitoring` 和 IoT 设备接口。
- 不在本轮保留“当前建议动作”“一线建议”“运维建议”“研发建议”“消息日志与审计回看”独立大卡。
- 不在本轮重做风险监测详情抽屉；仅复用其已有数据口径。

## 4. 方案对比

### 4.1 方案 A：仍以 `deviceCode` 为唯一入口，页内先查绑定再拉详情

- 页面接收 `deviceCode`。
- 先调用 `GET /api/risk-monitoring/realtime/list?deviceCode=...`。
- 再取一条绑定调用 `GET /api/risk-monitoring/realtime/{bindingId}`。

优点：

- 基本不动现有入口链路。
- 改造集中在对象洞察页。

缺点：

- 同一设备存在多条风险监测绑定时，页面必须额外约定默认展示哪一条。
- 页面已经进入“监测对象”语义，但 URL 仍只表达“设备”。

### 4.2 方案 B：统一改为 `bindingId` 主入口

- 页面以 `bindingId` 为唯一查询参数。
- 所有上游入口都改为传 `bindingId`。

优点：

- 风险运营语义最准确。
- 页面状态最清晰，没有多绑定歧义。

缺点：

- 会牵动实时监测台、GIS、告警、事件、设备页等多个入口。
- 本轮改动面过大，不利于快速收口。

### 4.3 方案 C：双入口并存，优先 `bindingId`，回退 `deviceCode`

- `/insight` 同时接受 `bindingId` 和 `deviceCode`。
- 有 `bindingId` 时直接拉 `risk-monitoring` 详情。
- 仅有 `deviceCode` 时，先查绑定列表，再自动解析默认绑定。
- 若命中多条绑定，页面在主工作台内提供轻量绑定切换，不新增新路由。

优点：

- 保留现有设备入口兼容性。
- 允许风险运营页后续逐步改成直接传 `bindingId`。
- 同时解决多绑定设备的显示歧义。

缺点：

- 页面内部状态稍复杂，需要同时处理“绑定解析中”和“详情加载中”。

## 5. 选型

本轮采用方案 C。

原因：

- 用户已明确要求“同步切到 `risk-monitoring` 实时详情接口，让风险等级、趋势和监测对象口径都来自风险监测绑定数据”。
- 当前仓库里大量入口仍按 `deviceCode` 跳转到 `/insight`，如果直接切到方案 B，会把对象洞察页改造扩散到多个页面。
- 方案 C 允许对象洞察台先完成语义切正，再逐步推动上游入口收口到 `bindingId`。

## 6. 数据来源设计

### 6.1 风险监测主数据

主数据改为：

- `GET /api/risk-monitoring/realtime/list?deviceCode=...`
- `GET /api/risk-monitoring/realtime/{bindingId}`

对象洞察台中的以下内容统一由风险监测详情驱动：

- 当前风险等级
- 在线状态
- 监测对象主身份
- 测点名称 / 标识符
- 当前值 / 最新上报
- 趋势点数据
- 最近告警数 / 最近事件数
- 区域、风险点、地址等监测对象上下文

### 6.2 IoT 设备补充数据

以下数据仍继续复用现有 IoT 接口：

- `GET /api/device/code/{deviceCode}`
- `GET /api/device/{deviceCode}/properties`
- `GET /api/device/{deviceCode}/message-logs`

这些数据只承担补充角色：

- 属性快照数
- 消息日志数
- 设备属性快照表
- 关键监测指标卡片
- 研判依据中的补充异常信号
- 风险分析草稿中的辅助描述

### 6.3 多绑定设备的默认解析规则

当只传 `deviceCode` 且命中多条绑定时，页面按以下优先级默认选中一条：

1. `latestReportTime` 最新的绑定优先。
2. 若 `latestReportTime` 相同，则 `onlineStatus=1` 优先。
3. 若仍相同，则风险等级更高的绑定优先。
4. 若仍无法区分，则取列表首条。

同时，页面在筛选区下方增加一条轻量绑定切换区，仅在“同一设备存在多条绑定”时显示，供用户切换监测对象，不新增二级页签或独立弹窗。

## 7. 页面结构设计

页面继续保持：

- `StandardPageShell`
- 单一 `StandardWorkbenchPanel`
- `StandardListFilterHeader`
- `StandardInlineState`

但正文结构收口如下。

### 7.1 第一屏：核心指标

首屏只保留四个核心指标：

- 当前风险等级
- 在线状态
- 属性快照数
- 消息日志数

处理规则：

- “当前风险等级”直接取 `riskMonitoring.detail.riskLevel`，不再使用前端自算分值作为主值。
- “在线状态”直接取 `riskMonitoring.detail.onlineStatus`。
- “属性快照数”来自 `properties.length`。
- “消息日志数”来自 `logs.length`。

原来的 `risk-banner` 和“当前建议动作”卡片删除，不再保留额外的风险评分、角色动作和三连跳转按钮。

### 7.2 第二层：主体判断区

主体判断区固定为三块：

1. 基础档案
2. 研判依据
3. 属性趋势预览

其中：

- 基础档案优先展示监测对象语义字段：风险点、风险点编码、设备名称、设备编码、产品名称、测点名称、测点标识符、区域、地址、最新上报。
- 若 `device` 补充信息存在，再展示固件版本、协议编码、最近在线时间等设备补充字段。
- 研判依据优先来自真实风险监测状态：
  - 风险等级
  - 监测状态
  - 在线状态
  - 最新上报时效
  - 当前值与告警/事件活跃度
  - 属性与日志补充信号

### 7.3 属性趋势预览

趋势图不再仅根据消息日志自由挑 4 个字段，而是按“监测对象类型”只显示核心指标。

对象类型推断规则：

- 优先根据 `metricIdentifier / metricName / riskPointName / productName` 的关键字推断。
- 推断结果落三类：
  - 检测型
  - 预警型
  - 采集型
- 若无法命中，回退为“通用型”，直接按当前绑定测点趋势显示。

趋势数据来源：

- 优先使用 `riskMonitoring.detail.trendPoints`。
- 不再以 `PropertyTrendPanel` 当前“从日志里扫任意 4 条数字属性”的逻辑作为主实现。

趋势展示规则：

- 使用折线图。
- 主序列固定为当前绑定测点。
- 旁侧摘要卡展示“最新值 / 最小值 / 最大值 / 近 24h 点数”。
- 若只有一个点，显示静态趋势说明，不强行渲染误导性的波动线。

## 9. 关键模块设计

### 9.1 `DeviceInsightView.vue`

职责调整为：

- 负责查询参数解析：`bindingId` 优先，`deviceCode` 回退。
- 负责绑定列表解析与选中绑定状态。
- 负责并行加载：
  - 风险监测详情
  - 设备详情
  - 设备属性
  - 消息日志
- 负责生成页面级计算数据：
  - 核心指标
  - 基础档案字段
  - 研判依据
  - 风险分析草稿

### 9.2 趋势组件

现有 `PropertyTrendPanel.vue` 更适合“从日志中泛化展示属性趋势”，不适合作为风险监测对象的核心趋势卡。

本轮建议：

- 将其改造为可接收明确的风险监测趋势数据和对象类型。
- 或新增一个对象洞察专用趋势组件，仅服务 `/insight`。

选型偏向新增轻量专用组件，理由是避免把“任意属性趋势”与“风险监测测点趋势”混成一个抽象。

### 9.3 研判依据生成器

前端保留轻量组合逻辑，但输入口径改变：

- 主依据来自 `riskMonitoring.detail`。
- 次依据来自 `properties` 和 `logs`。
- 输出 3 到 6 条研判依据卡片。

优先级示例：

- 风险等级高于普通级
- 当前监测状态为告警 / 无数据 / 离线
- 最新上报超时
- 当前值异常或近期有活跃告警
- 属性快照缺失
- 日志缺失

## 10. 交互设计

- 查询入口默认仍是设备编码输入框。
- 当存在 `bindingId` 时，输入框仍显示当前设备编码，但页面以绑定对象为准。
- 多绑定切换仅在命中多条绑定时出现，采用轻量下拉或切换按钮组，不引入页内 Tab 工作区。
- 刷新按钮继续沿用共享主按钮风格。
- 重置按钮恢复到初始查询参数。

## 12. 验证思路

前端实现阶段至少覆盖：

- `bindingId` 直接进入对象洞察台。
- `deviceCode` 进入并自动解析绑定。
- 单设备多绑定时默认绑定选择正确。
- 切换绑定后，风险等级、基础档案、趋势和研判依据同步更新。
- 删除内容不再出现在页面。
- 趋势图在只有单点、无点、多点三种情况下表现正确。
- `npm run test`
- `npm run build`

## 13. 风险与缓解

- 风险：设备未绑定风险点时，对象洞察台会出现“设备存在，但无风险监测对象”的空态。
  - 缓解：空态需明确提示“当前设备未纳入风险监测绑定”，并引导去风险对象中心或实时监测台确认绑定。

- 风险：同一设备多绑定会让对象身份不明确。
  - 缓解：提供默认解析规则和轻量绑定切换。

- 风险：现有趋势组件抽象不匹配风险监测趋势。
  - 缓解：避免强行复用旧泛化抽象，优先用专用趋势卡。

## 14. 实施前确认结论

本设计确认以下假设：

- `/insight` 本轮继续保留为风险运营下的对象洞察入口，不新增新路由。
- 风险等级、趋势和监测对象主身份必须以 `risk-monitoring` 实时详情为准。
- `deviceCode` 仍作为兼容入口保留。
- 同一设备多绑定时，默认取“最新上报优先”的绑定，并允许页内切换。
- 页面应聚焦洞察和判断，不再承载角色动作建议中心。
