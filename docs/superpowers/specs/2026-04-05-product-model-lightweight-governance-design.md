# 产品物模型极简治理设计

**Date:** 2026-04-05  
**Status:** Approved in-session for execution, refined with compact workbench B3 follow-up  
**Audience:** 产品定义中心 / 前端实现 / 文档维护  
**Scope:** 在不新增路由、不改后端 compare/apply 契约的前提下，把 `/products` 的契约字段核对抽屉重构为客户视角的一屏式物模型提炼工具。

## 1. 背景

当前契约字段核对抽屉虽然能力完整，但对客户来说过重：

1. 大量篇幅用于解释流程，而不是直接展示结果。
2. `第一步 / 第二步 / 第三步` 结构有强流程感，阅读成本高。
3. `规范优先 / 通用核对 / 样本类型 / 人工补录 / 自动提炼` 同时出现，决策复杂。
4. 页面在“告诉客户系统怎么想”，而不是“告诉客户识别出了什么、要不要生效”。

用户本轮明确要求：

1. 模型提炼尽量直接采用上报数据格式，自动完成。
2. 删除无用的提取入口和大段说明文字。
3. 用通俗、直接、客户能看懂的方式展现。
4. 删除“第一步：选择核对范围”到“第三步：确认生效”的全部介绍性内容。

## 2. 设计目标

本轮目标：

1. 把抽屉收口为“贴数据 -> 自动提炼 -> 选择生效”。
2. 首屏直接呈现上报样本输入和提炼结果，不再先讲流程。
3. 只保留客户会用到的动作：粘贴样本、补齐关系、自动提炼、选择是否生效。
4. 识别到父设备样本时，自动按子产品字段归一，不要求客户理解后端术语。
5. compare 结果卡片化、结果化，不再以三段证据叙事为主。
6. 在极简基础上继续把工作台做紧凑，接近设备资产中心详情页的阅读密度和信息节奏。

## 3. 非目标

本轮不做：

1. 不改后端 compare/apply 接口。
2. 不新增独立页面或二级路由。
3. 不引入新的数据库结构。
4. 不继续扩展规范优先治理的表达。
5. 不在本轮解决真实环境里 runtime compare 的慢链路，只通过前端默认行为规避。

## 4. 选型

### 方案 A：极简一屏式治理，推荐

做法：

1. 保留产品身份区和少量结果数字。
2. 删除“三步流”、长说明、概览说明段落。
3. 默认直接展示样本输入区和“自动提炼字段”按钮。
4. 生成后直接展示字段结果列表，每条字段只提供最小必要信息与生效动作。
5. 人工补录下沉为低优先级或折叠入口，不再默认占据主版面。

优点：

1. 最符合客户“简单提炼、少文字”的目标。
2. 现有 compare/apply 能力足够承接。
3. 前端改造集中，风险可控。

缺点：

1. 会弱化规范优先治理的显式入口。
2. 需要重构现有抽屉布局和部分测试。

### 方案 B：保留现有结构，只删文案

不采用，原因：

1. 仍然保留重流程感。
2. 客户还是会面对过多按钮和概念。

本轮采用 `方案 A`。

### 方案 A2：在方案 A 上继续做 B3 紧凑进化，推荐

做法：

1. 保留一屏式提炼主线，不再新增页签或副流程。
2. 结果区顶部增加一条轻量分组条：`可直接生效 / 待确认 / 继续观察 / 存在差异`。
3. 下方继续保留单列结果卡，不改成重型看板或多栏宫格。
4. 每张卡强化 `样例值 / 来源 / 是否已有正式字段 / 当前动作` 四类信息。
5. 抽屉整体密度向设备资产中心详情页靠拢：减少留白、缩短说明、统一成紧凑台账语法。

优点：

1. 兼顾“看得懂”和“处理得快”。
2. 不会把页面重新做回重型治理平台。
3. 更符合客户希望的紧凑详情页式工作台节奏。

缺点：

1. 需要继续调整抽屉头部、结果卡和底部确认区的布局密度。

本轮 follow-up 采用 `方案 A2 / B3`。

## 5. 最终设计

### 5.1 抽屉头部

保留：

1. 产品名称
2. productKey / 协议 / 节点类型
3. 三个简短统计：已生效、本次识别、待生效

删除：

1. `契约核对工作台`
2. 长副标题
3. 研发味过强的状态解释

### 5.2 主工作区

主工作区改成两个连续区块：

1. `上报数据`
2. `识别结果`

其中：

1. `规范优先核对 / 通用核对` 默认移除出主视图。
2. `业务数据 / 状态数据 / 其他数据` 默认移除出主视图。
3. `第一步 / 第二步 / 第三步` 全部移除。
4. `本轮核对概览` 这类解释型段落移除。

### 5.3 上报数据区

直接保留：

1. JSON 输入框
2. `自动提炼字段` 主按钮

识别到父设备关系场景时，再显示最少必要控件：

1. 父设备编码
2. 关系预设选择
3. `一键补齐设备关系`

但不再展示大段说明，只保留一条短提示：

1. `已识别为父设备样本，将自动归一到子产品字段`

### 5.4 自动提炼行为

默认行为：

1. 普通样本直接按当前产品提炼。
2. 父设备样本自动尝试关系归一。
3. 关系归一场景下默认关闭 runtime 自动提炼，避免真实环境 compare 卡慢链路。

### 5.5 结果列表

每条字段卡片只显示：

1. 中文名
2. identifier
3. 数据类型
4. 一个样例值
5. 当前动作按钮

不再默认展示：

1. 规范依据 / 运行依据 / 当前正式契约 三列说明
2. 风险说明大段文本
3. 技术依据折叠细节

如需保留证据，只保留一行短标签，例如：

1. `来自上报样本`
2. `来自父设备归一`
3. `正式模型已存在`

### 5.6 结果区 B3 紧凑进化

结果区采用“轻分组 + 紧凑单列卡”：

1. 顶部先给出一条轻量结果分组条：
   - `可直接生效`
   - `待确认`
   - `继续观察`
   - `存在差异`
2. 分组条只回答“先处理哪一堆”，不承担大段解释。
3. 下方仍使用单列结果卡，避免重新变成多栏看板。
4. 每张卡第一行固定展示：
   - 中文名
   - identifier
   - 类型
   - 样例值
5. 每张卡第二行固定展示来源标签：
   - `来自上报样本`
   - `来自父设备归一`
   - `来自运行数据`
   - `正式模型已存在`
6. 如果命中协议模板，只追加模板名、子设备编号、canonical 策略等紧凑摘要，不再展开成长解释墙。
7. 风险提示保留，但改成短句或短 badge，不再长段解释。

### 5.7 工作台密度

整体工作台向设备资产中心详情页靠拢：

1. 头部只保留产品名、关键信息行和 3 张紧凑结果卡。
2. 区块标题只保留短标题，不再带长副说明。
3. `上报数据` 区中的模式切换、运行期开关和主按钮要尽量压到同一工作台语法中。
4. 关系归一表单使用紧凑双列表单语法，不再单独长成说明面板。
5. `本次生效`、`本次结果` 与 `当前已生效字段` 都保持台账卡式紧凑密度，减少视觉噪音。

### 5.8 生效区

底部只保留：

1. `关闭`
2. `确认并生效`

同时在按钮上方展示一句短结果汇总，例如：

1. `本次将新增 3 个字段`

并进一步增强为：

1. 常驻显示 `已选 X 项`
2. 能直接回答 `新增 Y 项 / 修订 Z 项`
3. 让底部更像紧凑详情页操作条，而不是单独一块说明区

## 6. 验收口径

本轮完成后，页面需要满足：

1. 看不到 `第一步 / 第二步 / 第三步` 标题和对应说明。
2. 看不到大段“这里先回答哪些字段已经明确……”之类的解释性文案。
3. 客户进入后第一眼就是输入数据和自动提炼。
4. 关系归一场景能够继续正常补齐设备关系并产出子产品字段。
5. 页面用词以“上报数据 / 自动提炼 / 识别结果 / 确认生效”为主，不再强调 compare/apply/治理术语。
6. 结果区能一眼看出哪些字段可直接生效，哪些需要确认。
7. 工作台整体密度接近设备资产中心详情页，不再出现大留白、大说明块和重复引导。
