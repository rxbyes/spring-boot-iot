## 3. 真实环境启动
### 3.1 后端启动
统一命令：
```bash
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev
```

若当前工作区未提供 `.mvn/settings.xml`，直接去掉 `-s .mvn/settings.xml`，改为 plain `mvn` 即可。

若本机 `9999` 已被其他进程占用，可临时改用：
```bash
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev -Dspring-boot.run.arguments=--server.port=10099
```
此时需同步把 `curl` 基准地址、前端 `VITE_PROXY_TARGET` 或冒烟脚本 `-BaseUrl` 改到对应端口，并在验收记录里注明“本机端口冲突”环境阻塞。

可选脚本：
```powershell
powershell -ExecutionPolicy Bypass -File scripts/start-backend-acceptance.ps1
```
该脚本会先执行全量构建，再用 `dev` profile 启动后端；若仓库不存在 `.mvn/settings.xml`，脚本会自动回退到 plain `mvn`。

若需直接改跑隔离端口，可使用：
```powershell
powershell -ExecutionPolicy Bypass -File scripts/start-backend-acceptance.ps1 -Port 10099
```

- 也可先设置 `IOT_BACKEND_ACCEPTANCE_PORT=10099`，再沿用原脚本入口。
- 脚本当前会在构建成功后把 `spring-boot-iot-admin/target/*.jar` 复制到 `logs/backend-runtime/` 下的运行副本后再启动，避免上一次验收中的 `target` 产物锁住后续 `clean package`。

若本轮需要对刚修改过的上游模块做严格真实环境取证，推荐改用：

```bash
mvn -pl spring-boot-iot-admin -am clean package -DskipTests
java -jar spring-boot-iot-admin/target/spring-boot-iot-admin-1.0.0-SNAPSHOT.jar --spring.profiles.active=dev
```

- 该方式更容易避免 `spring-boot:run` 仍引用旧依赖类，尤其适用于 `device`、`alarm`、`framework` 等跨模块改动后的接口 / 数据库复验。

## 4. HTTP 主链路真实环境验收
### 步骤 1：创建产品
```bash
curl -X POST http://localhost:9999/api/device/product/add \
  -H "Content-Type: application/json" \
  -d '{
    "productKey":"accept-http-product-01",
    "productName":"验收产品-HTTP-01",
    "protocolCode":"mqtt-json",
    "nodeType":1,
    "dataFormat":"JSON"
  }'
```

### 步骤 2：创建设备
```bash
curl -X POST http://localhost:9999/api/device/add \
  -H "Content-Type: application/json" \
  -d '{
    "productKey":"accept-http-product-01",
    "deviceName":"验收设备-HTTP-01",
    "deviceCode":"accept-http-device-01",
    "deviceSecret":"123456",
    "clientId":"accept-http-device-01",
    "username":"accept-http-device-01",
    "password":"123456"
  }'
```

### 步骤 3：发送 HTTP 上报
```bash
curl -X POST http://localhost:9999/api/message/http/report \
  -H "Content-Type: application/json" \
  -d '{
    "protocolCode":"mqtt-json",
    "productKey":"accept-http-product-01",
    "deviceCode":"accept-http-device-01",
    "payload":"{\"messageType\":\"property\",\"properties\":{\"temperature\":26.5,\"humidity\":68}}",
    "topic":"/sys/accept-http-product-01/accept-http-device-01/thing/property/post",
    "clientId":"accept-http-device-01",
    "tenantId":"1"
  }'
```

说明：
- 响应体当前应返回 `sessionId`、`traceId`、`status`、`timelineAvailable`、`correlationPending`。
- 记录本次返回的 `sessionId / traceId`，后续用于校验 `/api/device/message-flow/session/{sessionId}` 和 `/trace/{traceId}`。

页面验收补充：
- 打开 `/reporting`，输入 `accept-http-device-01` 后点击“查询设备”。
- 确认 `productKey`、`protocolCode`、`clientId` 已只读反显，`tenantId` 默认 `1`，`topic` 默认 `$dp`。
- 明文模式下输入合法 JSON 或 XML，点击格式化，确认诊断区能展示“实际发送内容”预演；非法 JSON 不允许发送。
- 发送成功后，页面应立即展示本次 HTTP 提交对应的“处理时间线”面板，阶段顺序固定为 `INGRESS -> TOPIC_ROUTE -> PROTOCOL_DECODE -> DEVICE_CONTRACT -> MESSAGE_LOG -> PAYLOAD_APPLY -> TELEMETRY_PERSIST -> DEVICE_STATE -> RISK_DISPATCH -> COMPLETE`。

### 步骤 4：查询验证
```bash
curl http://localhost:9999/api/device/accept-http-device-01/properties
curl http://localhost:9999/api/device/accept-http-device-01/message-logs
curl http://localhost:9999/api/device/code/accept-http-device-01
curl -H "Authorization: Bearer <token>" http://localhost:9999/api/device/message-flow/session/<HTTP_SESSION_ID>
curl -H "Authorization: Bearer <token>" http://localhost:9999/api/device/message-flow/trace/<HTTP_TRACE_ID>
curl -H "Authorization: Bearer <token>" http://localhost:9999/api/device/message-flow/ops/overview
curl -H "Authorization: Bearer <token>" "http://localhost:9999/api/device/message-flow/recent?size=10&deviceCode=accept-http-device-01"
```

可选接口校验：
- 使用 `GET /api/device/code/{deviceCode}` 返回的真实 `id`，再调用 `GET /api/telemetry/latest?deviceId=<DEVICE_ID>`，确认返回 `storageType=tdengine`、真实 `properties` 和最新 `traceId`。

TDengine 核查示例：
```sql
-- 若产品已配置 legacy 映射，优先核对对应 stable，例如 s1_zt_1 / l1_qj_1
SELECT ts, rd, id, temp, humidity
FROM s1_zt_1
WHERE device_sn = 'accept-http-device-01'
ORDER BY ts DESC
LIMIT 10;

-- 未映射指标或兼容回退场景，再核对通用表
SELECT ts, reported_at, device_code, metric_code, trace_id, value_text
FROM iot_device_telemetry_point
WHERE device_code = 'accept-http-device-01'
ORDER BY ts DESC
LIMIT 10;
```

说明：
- 默认 `legacy-compatible` 模式下，若产品已配置 `specsJson.tdengineLegacy`，标准化属性会优先进入 legacy stable；只有未映射指标才会继续进入 `iot_device_telemetry_point`。
- `reported_at` 保留设备真实采集时间；`ts` 当前作为 `iot_device_telemetry_point` 的唯一行键时间，按全局单调递增方式生成，不再等于 `reported_at + 属性序号偏移`。
- 若要排查“同一设备、同一采集时刻的 property/status 组合是否互相覆盖”，必须同时按 `device_code + reported_at + trace_id` 核对通用回退表，并确认最新查询按 `reported_at DESC, ts DESC` 取值。
- 2026-03-24 共享 dev 复验证据：
  - `SK00E60D1306931` 在 `reported_at=2026-03-24T07:18:03.000Z` 下同时保留 `status trace=c4bfcfa4ba9349b68ab0eac87287e25f` 的 17 行和 `property trace=d74178cd34f64c0197a913bbf82af152` 的 2 行。
  - `SK00EA0D1307992` 在 `reported_at=2026-03-24T07:17:38.000Z` 下同时保留 `status trace=6544c3bbb7504ea38bd8c7a34136b400` 的 19 行和 `property trace=90bfb1b8b3f74be0b16d4a772db81b0e` 的 3 行。

### 通过标准
- 接口返回 `code = 200`
- `properties` 返回最新属性值
- `message-logs` 返回对应 topic 和 payload
- `iot_message_log` 有新增记录
- 若产品已配置 legacy 映射，TDengine 对应 stable 应新增该设备 `device_sn` 的最新一行；若存在未映射指标，`iot_device_telemetry_point` 还应新增对应设备、指标编码和 `trace_id` 的时序点
- 同一条双指标属性消息在 legacy stable 场景下一般表现为 1 行宽表更新；若两项都是未映射指标，则仍应在 `iot_device_telemetry_point` 查到 2 行
- `iot_device.online_status = 1`
- `last_online_time` 与 `last_report_time` 已刷新
- `/reporting` 同页已展示完整 timeline，且包含 `traceId / handlerClass / handlerMethod / costMs`
- timeline 中必须出现 `TELEMETRY_PERSIST`，且属性消息在 `tdengine` 模式下应看到 `persistedPointCount > 0`
- 若已调用 `GET /api/telemetry/latest`，返回值不得再是硬编码示例，且 `properties` 应与本次上报一致
- `GET /api/device/message-flow/session/{sessionId}` 应立即返回 timeline，且 `traceId` 与提交结果一致
- `GET /api/device/message-flow/ops/overview` 应返回非空 `stageMetrics`
- `GET /api/device/message-flow/recent` 应能查到本次 HTTP session，且 `timelineAvailable=true`

## 5. MQTT 标准 Topic 真实环境验收
### 5.0 链路验证中心 MQTT 模拟验收

标准 Topic 示例：

```bash
curl -X POST http://localhost:9999/api/message/mqtt/report/publish \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "protocolCode":"mqtt-json",
    "productKey":"accept-http-product-01",
    "deviceCode":"accept-http-device-01",
    "topic":"/sys/accept-http-product-01/accept-http-device-01/thing/property/post",
    "payload":"{\"messageType\":\"property\",\"properties\":{\"temperature\":28.1,\"humidity\":59}}"
  }'
```

说明：
- `/reporting` 页面里的 MQTT 模拟上报即调用该接口。
- 若 payload 需要发送明文二进制帧，可附带 `payloadEncoding=ISO-8859-1`。
- 该接口会把原始字节发布到 Broker，再由现有 MQTT consumer 回流进入主链路；若客户端未连接、设备不存在、产品不匹配、目标设备未配置协议或协议不匹配，应返回业务错误。
- 当前默认启用 MQTT consumer `cluster-singleton`：多节点共享同一 Redis/MySQL/TDengine 环境时，只允许 1 个 leader 节点订阅 Broker；但 `/api/message/mqtt/report/publish` 当前无论打到 leader 还是 standby，都会使用独立 publisher 客户端发起模拟上行。
- 返回成功后应至少拿到 `sessionId`、`status=PUBLISHED`、`timelineAvailable=false`、`correlationPending=true`，并允许 `/reporting` 按该 `sessionId` 轮询处理状态。

### 步骤 1：启动后端并确保 MQTT 已连接
建议附加唯一 `clientId`：
```bash
IOT_MQTT_CLIENT_ID=accept-mqtt-consumer-001 \
mvn -s .mvn/settings.xml -pl spring-boot-iot-admin spring-boot:run -Dspring-boot.run.profiles=dev
```

若共享环境同时起了多个后端节点，先核对：
- `GET /actuator/health` 中恰好 1 个节点满足 `mqttConsumer.details.leadershipMode=LEADER`、`consumerActive=true`
- 其他节点应为 `leadershipMode=STANDBY`、`consumerActive=false`、`connected=false`
- 允许在 leader 或 standby 节点调用 `/api/message/mqtt/report/publish`，但最终消费时间线仍必须由 leader 节点生成并回填到同一个 `sessionId`

### 步骤 2：使用 MQTTX 建立连接
- Host：`8.130.107.120`
- Port：`1883`
- Username：`emqx`
- Password：读取 `application-dev.yml` 中 `iot.mqtt.password`
- Client ID：如 `mqttx-accept-001`

### 步骤 3：发送标准 Topic 消息
Topic：
```text
/sys/accept-http-product-01/accept-http-device-01/thing/property/post
```

Payload：
```json
{"messageType":"property","properties":{"temperature":28.1,"humidity":59}}
```

### 通过标准
- 应用日志可见 MQTT 收到消息并进入统一主链路
- `GET /api/device/accept-http-device-01/properties` 可见最新值
- `GET /api/device/accept-http-device-01/message-logs` 返回标准 topic 记录
- 若产品已配置 legacy 映射，TDengine 对应 stable 应可查询到同一设备 `device_sn` 的最新宽表行；若存在未映射指标，`iot_device_telemetry_point` 也可查询到同一设备、同一 `traceId` 的回退时序点
- `iot_device.online_status = 1`
- Redis 会话 `iot:device:session:accept-http-device-01` 刷新 `lastSeenTime`
- `/reporting` 页面先显示“等待消费回流”，随后在同一 `sessionId` 绑定 trace 后展示完整 timeline，且 timeline 中出现 `TELEMETRY_PERSIST`
- `GET /api/device/message-flow/session/{sessionId}` 在 `session-match-window-seconds` 窗口内应由 `correlationPending=true` 过渡到 `traceId!=null`
- MQTT 关联命中后，同一条 timeline 的顶层 `sessionId` 与 `INGRESS.summary.sessionId` 必须保持一致，统一回填为 publish 返回的 `sessionId`
- 同一次 `/api/message/mqtt/report/publish` 最终只允许绑定 1 个 `traceId`；不得再出现“同一 `sessionId` 绑定两个不同 `traceId`”的重复消费现象
- `GET /api/device/<deviceCode>/message-logs` 对同一次 MQTT publish 只应新增 1 条 MQTT 消息日志；若 payload 含 2 个未映射指标，则 `iot_device_telemetry_point` 只应新增同一 `traceId` 下的 2 个点位，不得额外出现第二个 MQTT `traceId`
- `GET /api/device/message-flow/recent` 应能查到本次 MQTT session
- `GET /api/device/message-flow/ops/overview` 在单实例直连消费场景下应能看到 `published`、`matched`
- 若当前共享 `dev` 环境实际拓扑为“本机实例负责 publish，leader consumer 实例负责 MQTT 回流匹配”，则当前实例的 `correlationCounts` 只看到 `published` 也可判定通过；此时应同时满足：
  - `GET /api/device/message-flow/session/{sessionId}` 已绑定 `traceId`
  - `GET /api/device/message-flow/trace/{traceId}` 可查
  - `GET /api/device/message-flow/recent` 可查到同一 MQTT session
  - 验收记录中明确备注“matched 计数发生在其他 consumer runtime，本机 ops overview 仅保留 publish 侧计数”
- 打开 `/message-trace`，定位同一条 `traceId` 的任一日志记录，详情抽屉应展示同一条 trace 的处理时间线；若超出 TTL，则必须明确显示“时间线已过期，仅保留消息日志”

### 5.1 时间线过期降级校验

前置条件：
- 准备 1 条已超过 `iot.observability.message-flow.ttl-hours` 的历史 `traceId`。

验证步骤：

通过标准：

### 13.2 截图基线与视觉回归

- 基线目录默认落在 `config/automation/baselines/<planSlug>/<scenarioKey>/<baselineName>.png`，适合按计划、场景、断言名称分层维护。
- `assertScreenshot` 支持两类目标：`page` 用于整页基线，`locator` 用于表格、图表、卡片等局部区域基线。
- 若未找到基线且未开启 `--update-baseline`，执行器会把该步骤标记为失败，并在报告中提示缺失基线。
- 若截图断言步骤额外声明 `optional: true`，则执行器会把该步记为 `optional_failed` 并继续输出 manifest / diff 建议；场景结果与 CLI 退出码仍按阻断范围单独判定。
- 若存在基线，执行器会输出 baseline / actual / diff 三类图片路径，并在 Markdown 报告中汇总视觉断言通过数、失败数、刷新数与缺失数。
- 每轮执行还会生成 visual manifest JSON、diff 图片索引页和失败截图明细页，便于测试、产品、设计协同查看视觉差异。
- 推荐将关键页面首屏、复杂表格、图表看板、弹窗结果页纳入视觉断言，和文本/API 断言形成互补证据链。

### 13.3 视觉基线治理命令

- `scripts/auto/manage-visual-baselines.mjs` 支持读取两类输入：配置驱动执行输出的 visual manifest JSON，或完整的 browser results JSON。
- 默认 `--mode=audit` 会按筛选条件输出治理 JSON 与 Markdown 报告，便于先做离线审计，再决定是否提升基线。
- `--status=passed,mismatch,missing,updated` 支持按视觉断言状态过滤；若不传且使用 `--mode=promote`，默认治理 `missing,mismatch`。
- `--scenario=...` 与 `--label=...` 可进一步按场景、步骤名称缩小治理范围，适合对单页或单组件做定向基线维护。
- 仅在明确评审通过后才执行 `--mode=promote --apply`，脚本会把 actual 图片复制为 baseline，并输出本轮提升记录。
- 前端工作目录已补充快捷命令：`cd spring-boot-iot-ui && npm run acceptance:browser:baseline:manage -- --input=../logs/acceptance/config-browser-visual-manifest-<timestamp>.json`
