产品定义中心与风险闭环的架构路线

目标：既能治理当前 $dp 真实上报，又为未来“只配配置、不改代码接入新厂商”打地基。

方案 A

工作台增强型

继续围绕 compare/apply -> iot_product_model 演进，把样本、日志、latest 证据和命名推荐塞进现有治理页。

优点

  • 改动最小，最快落地
  • 对当前裂缝样板够用
  • 风险最低

缺点

  • 仍是页面驱动，不是平台能力
  • 多厂商扩展会持续堆规则
  • 难支撑后续免代码接入
方案 B

语义契约中台

新增“规范语义层”，把行业规范、厂商字段、逻辑通道、正式物模型、风险闭环拆成清晰的五层，并把 iot_product_model 降为“已发布合同”。

推荐分层

1. 规范库: 行业/业务标准字段

2. 证据层: $dp / latest / 日志 / 关系映射

3. 治理层: 差异、命名、冲突、版本

4. 发布层: 发布到 iot_product_model

5. 闭环层: 风险点 / 阈值 / 联动 / 预案 / 运营

优点

  • 长期演进最稳
  • 治理与接入解耦
  • 风险闭环只依赖正式字段

缺点

  • 需要较大重构
  • 第一阶段要先补对象模型
  • 短期实现复杂度更高
方案 C

接入配置平台优先

优先把 Topic、解密、映射、父子关系全部配置化，目标是新厂商先接得进来，治理以后再补。

优点

  • 最贴近免代码接入终局
  • 对商务扩张想象力大
  • 配置化价值直观

缺点

  • 会把脏语义更快引入系统
  • 正式字段标准会继续漂移
  • 风险闭环稳定性最差
