第三段：模块边界与第一阶段重构路径

目标：按现有模块化单体边界实施，不破坏现有模块职责，同时把未来扩展点提前抽出来。

保留的核心模块职责

spring-boot-iot-protocol

只负责 Topic、解密、解码、父子归一、协议模板执行，输出标准化证据。

spring-boot-iot-device

只负责产品、设备、设备关系、合同发布、治理工作台，不直接承载风险判级规则。

spring-boot-iot-alarm

只负责风险指标目录、风险点、阈值、联动、预案、告警、事件、运营消费。

spring-boot-iot-report

只负责基于正式风险指标和闭环结果做分析报表。

第一阶段建议新增的子能力

device 内新增

规范字段库、厂商字段证据/映射、合同发布批次与差异治理服务。

alarm 内新增

风险指标目录层，让风险点绑定不再直接面向产品合同字段。

admin / observability 配套

治理任务、发布审计、漂移告警、回滚记录。

第一阶段推荐重构顺序

Phase 1：先把“裂缝直报 + 采集终端挂裂缝子设备”做成样板切片。

1. 建规范字段库与字段分类规则

2. 把 `$dp` 运行证据沉淀为“厂商字段证据/映射”

3. 让 iot_product_model 只发布正式合同

4. 在 alarm 域新增“风险指标目录”桥梁层

5. 风险点绑定改绑风险指标，而不是直接绑原始字段

6. 阈值/联动/预案/运营都统一消费风险指标目录

必须避免的反模式

  • 继续把厂商字段归一规则堆进 ProductModelServiceImpl
  • 让风险点直接绑定 iot_product_model.identifier 而没有风险指标桥梁
  • 让 spring-boot-iot-alarm 反过来决定产品合同命名
  • 把“规范库、证据、发布结果”继续混存到同一张表

第一阶段刻意不做

  • 不立刻追求全品类、全厂商一次纳管
  • 不先做零代码接入平台
  • 不把风险 GIS/运营分析细节和字段治理耦在一起开发
  • 不为短期兼容继续放大页面层 compare 逻辑
