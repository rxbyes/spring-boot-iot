第 3 节：后台能力边界与分阶段落地

这一节回答两个问题：业务验收台和现有研发工场怎么共存，以及第一阶段到后续智能化应该怎么落地，避免一次做得太重。

推荐边界
业务验收台负责什么
展示预置验收包
收集环境 / 账号模板 / 模块范围
发起一次运行
展示业务结论与失败模块
展开失败步骤 / 接口 / 页面动作
业务验收台不负责什么
不维护页面盘点
不编辑场景模板
不编排执行计划
不配置阻断规则和执行器
不开放给业务人员修改验收包
与现有体系的关系
分层结构

业务验收台

消费层。面向验收人员、产品、项目经理，只负责运行预置验收包和看结论。

研发工场

生产层。面向研发/测试，继续维护页面盘点、模板、计划和交付资产。

执行中心 + 结果中心

共享底座。业务验收台和研发工场都复用同一套执行器、运行结果和证据台账。

后台能力最小新增

第一阶段新增

  • 预置验收包定义模型
  • 业务验收运行入口
  • 运行参数模型：环境、账号模板、模块范围
  • 模块级聚合结论
  • 失败明细视图：模块/场景/步骤/接口/页面动作

继续复用

  • 现有 browserPlan / apiSmoke / messageFlow / riskDrill 执行器
  • 现有统一验收注册表
  • 现有结果与基线中心台账
  • 现有 logs/acceptance 证据产物
三阶段路线
Phase 1

一键业务验收

交付“业务验收台 + 预置验收包 + 轻配置 + 模块级结论 + 失败明细展开”。

Phase 2

智能判错与建议

自动把失败归因为前端、后端、环境、数据准备，并给出建议处理人。

Phase 3

AI 验收助理

支持业务语言触发验收、推荐验收包组合、生成复测建议。

推荐优先级
先做什么，不先做什么
先做：业务入口、预置验收包、轻配置、业务结论、失败展开
后做：智能归因、建议处理人、复测建议
最后做：自然语言生成验收、AI 编排

本节结论

A

落地顺序合理

按这套 Phase 1 到 Phase 3 路线继续写完整设计稿。

B

路线要调整

业务验收台方向不变，但阶段划分或后台边界还要调整。
