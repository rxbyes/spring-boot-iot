研发工场拆分方案

目标前提：以研发为主角色，优先服务“自动化资产编排”，再把执行、结果和交付协作从属挂接出去。

A / 轻拆分
质量工场总览
自动化资产中心
执行中心
结果与基线中心
研发快捷入口

A / 轻量职责分层

保留现有 3 个专项页，只新增一个“研发工场总览”，把研发常用入口和默认操作重新编排。

优点

  • 改动最小，落地最快。
  • 现有路由和菜单基本不动。

风险

  • 资产中心内部仍然偏大。
  • 只是“重排入口”，不是彻底拆解。
B / 推荐
研发工场总览
页面盘点台
场景模板台
计划编排台
交付打包台
转执行中心
转结果中心

推荐 / 研发工场独立分组

新增“研发工场”作为主入口，把资产编排拆成 4 个研发子模块；执行中心和结果中心继续作为跨角色共享模块存在。

优点

  • 最贴合你当前“研发优先 + 资产编排优先”的方向。
  • 拆分后每页语义单一，后续更容易继续扩展。
  • 不会把执行和结果逻辑重复建设一套。

风险

  • 需要新增一组研发菜单和路由。
  • 资产中心现有组件要重新切片。
C / 全角色矩阵
研发工场
测试工场
验收工场
巡检工场
各自再挂资产 / 执行 / 结果视图

C / 全角色工场矩阵

按角色彻底拆成研发、测试、验收、巡检 4 套工场，每套都有自己的总览和专属流程。

优点

  • 角色边界最清晰。
  • 每类人看到的默认动作最强。

风险

  • 路由、菜单、权限、文档都会成倍增加。
  • 极易出现重复页面和双份规则口径。

我当前的建议

先选 B。它能把研发资产编排从现在的大页里真正拆出来，同时继续复用现有执行中心和结果中心，不会马上把质量工场做成 4 套重复体系。
